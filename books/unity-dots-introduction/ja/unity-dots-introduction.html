Part I. はじめに

Chapter 1. 環境構築 — Unity 6000.5 + Entities 6.5.0

1. 概要

Unity 6000.4+ から、Entities は Core Package となった — エディター内に同梱されており、Package Manager 経由で追加する必要はない。本ガイドではエディターのインストール、DOTS 対応プロジェクトの作成、そして Entities が有効になっていることの確認まで順に説明する。

プロジェクト環境の参考情報： Unity 6000.5.0b2 · Entities 6.5.0 · Collections / Mathematics / Entities Graphics（すべて Core Package）。

2. Unity 6000.5+ のインストール

  1. Unity Hub を開く。まだ持っていない場合は先に Unity Hub をインストールする（https://unity.com/download）。
  2. Hub で Installs → Install Editor に進む。
  3. 6000.5.x バージョン（b系ベータ以降）を選択する。Entities 6.5 は 6000.5+ でのみ利用可能だ。
  4. モジュール：デフォルトのままでよい。ECS 自体に追加モジュールは不要だ。

"DOTS" モジュールを選択する必要はない — Entities は 6000.4 からエディターに組み込まれている。

3. 新規プロジェクトの作成

Hub → Projects → New project から：

フィールド 値
Editor Version 6000.5.x
Template Universal 3D（または 3D (Built-in)、HDRP — どれでも可）
Project Name 任意

Create project をクリックする。エディターが空のシーンで開く。

6000.4+ では専用の「DOTS テンプレート」はもはや不要だ。どのテンプレートでも Entities ワークフローをホストできる。

4. Core Package が有効であることの確認

Window → Package Manager を開き、スコープを In Project に切り替える。以下のパッケージが 「Built-in」 とマークされているはずだ：

  • Entities
  • Collections
  • Mathematics
  • Entities Graphics

ファイルで確認したい場合は Packages/manifest.json を確認する：エディターに同梱されているため com.unity.entities エントリは存在しない。

次に ECS ウィンドウを開く：

  • Window → Entities → Hierarchy — 実行時の Entity を表示する。
  • Window → Entities → Systems — 登録されている System を表示する。
  • Window → Entities → Archetypes — チャンクレイアウトを表示する。

3つのメニューがすべて存在すれば、Entities は正しく接続されている。

5. スモークテスト — SubScene の作成

  1. Hierarchy で右クリック → New Sub Scene → Empty Scene…
  2. MainSubScene.unity と名前を付けて保存する。
  3. 新しい SubScene の中で任意のプリミティブを追加する（3D Object → Cube）。
  4. シーンを保存する。SubScene を閉じると（黄色のアイコン）、またはビルド時に Cube は Entity としてベイクされる。
  5. Play を押す。Window → Entities → Hierarchy を開くと Cube Entity が表示されるはずだ。スモークテストはパスだ。

SubScene はオーサリングのエントリポイントであり、その中の GameObject は保存時とビルド時に Entities へベイクされる。以降の 03_Hello DOTS — First Entity.md のチュートリアルは、これを前提に進む。

6. トラブルシューティング

症状 原因 / 対処法
Window → Entities メニューが見当たらない エディターが 6000.4 より古い — Entities はまだ Core Package ではない。エディターをアップグレードすること。
Baker<T> / IComponentData が見つからない アセンブリ定義ファイルが Unity.Entities を除外している。参照として追加するか、簡単なテストのために asmdef を削除すること。
SubScene が GameObject のままでベイクされない SubScene は 閉じた状態（黄色のアイコン）でないとベイク済み Entity が表示されない。Hierarchy のチェックボックスで開閉すること。
Play モードで Entities Hierarchy ウィンドウに何も表示されない SubScene が現在のシーンに含まれていること、および Window → Entities → Hierarchy の「World」ドロップダウンが DefaultGameObjectInjectionWorld に設定されていることを確認すること。
ビルドエラー：com.unity.entities not found manifest.json に 1.x プロジェクトの com.unity.entities エントリが残っている — 削除すること。Migration/02_Package Manager → Core Package.md を参照。

Chapter 2. コアパッケージ解説

1. 概要

Entities パッケージ 6.4.0 のリリース（Unity Editor 6000.4 に同梱）により、主要な DOTS パッケージは Package Manager からエディター自体に Core Package として移行した。パッケージの変更履歴における正確な記述は次のとおりだ：

"Package is now a core package embedded in Unity."

この変更は com.unity.entities、com.unity.collections、com.unity.mathematics、com.unity.entities.graphics、および（対応する 6.5 リリースで）Netcode for Entities に影響した。インストール、バージョン管理、追跡方法が変わるが、これは 配布 の変更であり、API の変更ではない。

本ページでは以下を説明する：

  • 現在 Core Package となっているパッケージ（そうでないものも含む）。
  • バージョン番号の決まり方。
  • manifest.json と packages-lock.json の変化。
  • この移行が行われた理由。

2. Core Package とは何か

Core Package とは エディターの内部に同梱される パッケージだ。Package Manager 経由で追加する必要はなく、エディターとは独立してバージョンを変更することはできず、リリースサイクルもエディターに従う。

特性 Package Manager パッケージ Core Package
インストール手順 UPM 経由で追加 不要 — エディターに同梱
バージョンの決定者 ユーザー（manifest.json で） Unity（エディターバージョンごと）
リリースサイクル UPM 独立 エディターリリースに従う
manifest.json に掲載 あり なし
packages-lock.json に掲載 あり あり（builtin としてマーク）
Package Manager に表示 「In Project」の下 「In Project」の下に Built-in タグ付きで表示

注記：Job System と Burst は UPM パッケージではなく、エンジンの一部だった。6000.4 の変更は、その上にある DOTS パッケージ に関するものだ。

4. 移行後のバージョン番号

Core Package のバージョンは エディターに合わせて 付番される：

エディター Core Entities
Unity 6000.4 Entities 6.4
Unity 6000.5 Entities 6.5

6000.4+ にまだ移行できないプロジェクト向けに、旧来の 1.x ラインは引き続き存在する。2つのトラックは独立しており、6.x へのアップグレードはエディターのアップグレードによって行う。

Entities の移行ノートは Changelog/Entities 1.4 → 6.5 Key Changes.md、Netcode 固有の変更点は Changelog/Netcode for Entities 1.4 → 6.5 Key Changes.md を参照のこと。

5. プロジェクトファイルへの影響

manifest.json

com.unity.entities、com.unity.collections、com.unity.mathematics、com.unity.entities.graphics のエントリは不要だ。1.x プロジェクトをアップグレードする場合はそれらを削除する — Migration/02_Package Manager → Core Package.md を参照のこと。

packages-lock.json

Core Package はバージョン URL の代わりに "source": "builtin" として表示される。

アセンブリ定義（.asmdef）

.asmdef ファイルは引き続きアセンブリ名（例：Unity.Entities、Unity.Collections）を参照する。変わるのはパッケージの配布レイヤーのみで、ここは変更不要だ。

6. 移行の理由

公表されている目的は、DOTS の更新を独立したパッケージスケジュールではなくエディターリリースに連携させることで、ECS チームがより速く、より段階的にフィーチャーを提供できるようにすることだ。パッケージがオプションだった間はエンジンが ECS に内部依存することも不便だったが、それも解消される。

7. トラブルシューティング

症状 原因 / 対処法
Package Manager に旧バージョンで「Unity Registry」の Entities が表示される エディターが 6000.4 より古い。エディターをアップグレードするか、1.x ラインに留まること。
com.unity.entities を削除した後に manifest.json の解決エラーが出る packages-lock.json を削除し、エディターに再生成させること。
サンプルやアセットストアのパッケージが com.unity.entities@1.x を対象にしている 6000.5+ では 1.x API はほぼ廃止されている。Migration フォルダーで対応する API を確認すること。
2つの Editor インストールで、表示される Entities バージョンが異なる 予期した動作だ。バージョンはエディターインストールごとに異なる。

Chapter 3. Hello DOTS — はじめての Entity

1. 作るもの

ECS システムによって Y 軸を中心に回転するキューブだ。一度の作業ですべてのコアコンセプトに触れることになる：SubScene、オーサリング → baker、IComponentData、ISystem、そして Entities Hierarchy / Systems ウィンドウ。最後にメインスレッドのシステムを IJobEntity に置き換えて、並列処理がどのように入るかを確認する。

前提条件：[01_Environment Setup.md](01_Environment Setup (Unity 6000.5 + Entities 6.5.0).md) — SubScene と Entities メニューが確認済みの Unity 6000.5 プロジェクト。

2. SubScene の作成

  1. 空のシーンを開く。
  2. Hierarchy で右クリック → New Sub Scene → Empty Scene…
  3. Assets/Scenes/ 内に HelloDots.unity として保存する。
  4. 新しい SubScene の中で右クリック → 3D Object → Cube。名前を RotatingCube に変更する。
  5. シーンを保存する（Ctrl/⌘ + S）。Hierarchy でその名前の横のチェックボックスをクリックして SubScene を閉じる — アイコンが黄色になり、中身が Entity としてベイクされていることを示す。

以降、RotatingCube のコンポーネントに変更を加えると、SubScene を開閉した際に自動的に再ベイクが実行される。

3. コンポーネントの定義 — RotationSpeed

Assets/Scripts/RotationSpeed.cs を作成する：

using Unity.Entities;

public struct RotationSpeed : IComponentData
{
    public float RadiansPerSecond;
}

注記：

  • アンマネージド struct 上の IComponentData がデフォルトの選択だ — チャンク内に直接配置され、Burst フレンドリーだ。
  • フィールドはブリッタブルに保つこと（プリミティブ、Unity.Mathematics 型、他のアンマネージド構造体）。

4. オーサリング + baker の定義

Assets/Scripts/RotationSpeedAuthoring.cs を作成する：

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class RotationSpeedAuthoring : MonoBehaviour
{
    public float DegreesPerSecond = 90f;

    class Baker : Baker<RotationSpeedAuthoring>
    {
        public override void Bake(RotationSpeedAuthoring authoring)
        {
            // Dynamic — this entity will have its transform written each frame.
            var entity = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(entity, new RotationSpeed
            {
                RadiansPerSecond = math.radians(authoring.DegreesPerSecond)
            });
        }
    }
}

オーサリングをキューブにアタッチする

  1. SubScene 内で RotatingCube を選択する。
  2. Inspector → Add Component → Rotation Speed Authoring。
  3. DegreesPerSecond = 90 のままにする。

SubScene を閉じた瞬間（黄色のアイコン）、baker が実行され、キューブは RotationSpeed コンポーネントを持つ Entity に変換される。

ベイクの確認

  • Window → Entities → Hierarchy — RotatingCube が表示されているはずだ。
  • クリックすると右の Inspector にコンポーネントが表示される — RotationSpeed { RadiansPerSecond: ~1.5708 }（π/2）を探すこと。

5. システムの記述 — RotationSystem

Assets/Scripts/RotationSystem.cs を作成する：

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<RotationSpeed>>())
        {
            transform.ValueRW = transform.ValueRO.RotateY(speed.ValueRO.RadiansPerSecond * dt);
        }
    }
}

注記：

  • partial struct + ISystem は アンマネージドシステム の形式だ。Burst コンパイルが可能で、GC オーバーヘッドなしで動作する。
  • SystemAPI.Query<RefRW<A>, RefRO<B>>() は現代のイテレーション API だ — 廃止された Entities.ForEach の代替となる。
  • LocalTransform.RotateY(radians) は回転済みのコピーを返す。transform.ValueRW = … で書き戻す。

Play を押す

キューブが回転する。Window → Entities → Systems でシミュレーショングループ配下に RotationSystem がフレームタイミング付きで表示されているはずだ。

6. 作ったものを確認する

Play モード中に役立つウィンドウ：

ウィンドウ 表示内容
Entities Hierarchy Entity のライブ一覧。任意の Entity をクリックするとそのチャンクとコンポーネントが表示される。
Systems 実行中のすべてのシステム（SystemGroup でグループ化）。System ごとのタイミングが確認できる。
Archetypes コンポーネントシグネチャでグループ化されたチャンク。断片化を検出するのに役立つ。

以下を確認すること：

  1. RotatingCube Entity が存在し、LocalTransform + RotationSpeed を持っている。
  2. RotationSystem が SimulationSystemGroup 配下に存在し、フレーム時間を蓄積している。

7. IJobEntity に切り替える（並列処理）

並列処理がどのように組み込まれるかを確認するために、RotationSystem.cs をジョブベースのバージョンに置き換える：

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotateJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in RotationSpeed speed)
    {
        transform = transform.RotateY(speed.RadiansPerSecond * DeltaTime);
    }
}

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new RotateJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

注記：

  • IJobEntity.Execute(ref A, in B) はソース生成によりチャンクレベルのジョブに変換される。パラメーターが触るコンポーネントを指定する。
  • .ScheduleParallel() はチャンクをワーカースレッドに分散させる。キューブが1つではスピードアップは確認できないが、コードの形は変更なしに数千の Entity にスケールする。
  • システムの依存関係は自動的に処理される — ISystem.OnUpdate から ScheduleParallel() を呼ぶと state.Dependency が自動的に結合される。

ScheduleParallel() で「LocalTransform への競合状態」例外が発生する 別のシステムが同フレーム内で順序付けなしに LocalTransform に書き込んでいる。[UpdateBefore]/[UpdateAfter] を追加するか、共有 EntityCommandBuffer を使用すること。

Part II. DOTS ワークフロー

Chapter 4. Baker パターン & SubScene

1. 概要

Entity は Hierarchy で直接オーサリングできない。代わりに SubScene 内で GameObject をオーサリングし、SubScene を閉じるかビルドする際に Baker がそれらを Entity に変換する。

Figure: Baker と SubScene のパイプライン。

2. SubScene のライフサイクル

SubScene には視覚的に3つの状態がある：

Hierarchy のアイコン 状態 意味
⬜（白いチェックボックス） Open（開） GameObject が編集可能な状態；まだ GameObject のまま、ベイクなし。
🟨（黄色のチェックボックス） Closed（閉） Baker が実行済み；中身は Entity としてのみ存在する。
⏳ Baking（ベイク中） インクリメンタルベイクが進行中（一時的）。

SubScene の作成

  • Hierarchy → 右クリック → New Sub Scene → Empty Scene… — .unity アセットを保存し、親 GameObject に SubScene コンポーネントを追加する。
  • SubScene 名の横のチェックボックスで開閉する。閉じるとベイクが実行される。

インクリメンタルベイク

SubScene が開いている状態で GameObject を編集すると、変更されたオーサリングに触れた Baker だけが再実行される。このため Bake() では重い処理を避けること。

3. Baker<T> API

Baker は1つのオーサリング MonoBehaviour を1つ以上の Entity + コンポーネントにマップする。

public class EnemyAuthoring : MonoBehaviour
{
    public float MaxHealth = 100f;
    public GameObject ProjectilePrefab;

    class Baker : Baker<EnemyAuthoring>
    {
        public override void Bake(EnemyAuthoring authoring)
        {
            // Primary entity for this GameObject.
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Health { Value = authoring.MaxHealth });

            // Bake a GameObject reference into an Entity reference.
            AddComponent(self, new ProjectilePrefabRef
            {
                Value = GetEntity(authoring.ProjectilePrefab, TransformUsageFlags.Dynamic)
            });
        }
    }
}

主要 API：

API 用途
GetEntity(TransformUsageFlags) 現在のオーサリング GameObject の Entity。
GetEntity(Object, TransformUsageFlags) 別の GameObject/Prefab に対応する Entity（Prefab Entity として登録する）。
CreateAdditionalEntity(TransformUsageFlags) 同じオーサリングにリンクされた第2の Entity（サブパーツなど）。
AddComponent<T>(entity, value) コンポーネントデータをアタッチする。
AddBuffer<T>(entity) / SetBuffer<T> DynamicBuffer<T> をアタッチする。
AddSharedComponent<T> ISharedComponentData をアタッチする。
DependsOn(asset) アセット依存関係を宣言し、アセットが変更されたときに Baker を再実行させる。

4. TransformUsageFlags

TransformUsageFlags で要求したときのみ、ベイクは LocalTransform / LocalToWorld / Parent を追加する。不要なトランスフォームコンポーネントはチャンクを無駄に大きくするため、まだ機能する最も狭いフラグを選ぶこと。

フラグ 追加されるもの 典型的な用途
None なし 純粋なデータ Entity（設定、Singleton、ロジックのみ）。
ManualOverride なし；トランスフォームは自分で管理する カスタムトランスフォームレイアウト。
Renderable LocalToWorld 決して動かない静的メッシュ。
Dynamic LocalTransform、LocalToWorld（親子関係がある場合は Parent も追加） System によって移動・回転させられるもの。
WorldSpace LocalTransform、LocalToWorld；親子関係を防ぐ 階層を無視する Entity。
NonUniformScale PostTransformMatrix を追加 非均一スケール（DOTS では稀）。

System によって移動・回転させるものには Dynamic が安全なデフォルトだ。

5. Baking System（上級）

Baker は単一のオーサリングからコンポーネントを生成する。Baking System はすべての Baker が実行された後に動作し、ベイク済みの World 全体を見られる — クロス Entity のポストプロセス（参照の解決、派生データの計算）に有用だ。

[WorldSystemFilter(WorldSystemFilterFlags.BakingSystem)]
public partial struct EnemyWaveLinkSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Runs during bake only. Query entities tagged earlier by Bakers
        // and wire them together.
    }
}

Baker が必要なすべての情報を把握できない場合（カウント、クロス参照など）に使用する。決定論的に保つこと — ベイクは毎回同じ結果を生成しなければならない。

7. トラブルシューティング

症状 原因 / 対処法
Entity が表示されるがコンポーネントが欠けている Baker が実行されていない（SubScene が閉じた後にオーサリングが追加された）— SubScene を再度開閉して再ベイクすること。
GetEntity(prefab, ...) が Entity.Null を返す 参照先の GameObject が Prefab アセットでないか、Baker からアクセスできないシーンオブジェクトだ。Prefab であるか同じ SubScene 内にあることを確認すること。
エディター内で Baker が毎フレーム実行される Bake() に非決定論的なロジック（Random、Time.time など）がある。Baker はオーサリングの純粋な関数でなければならない。
Entity に LocalToWorld はあるが LocalTransform がない TransformUsageFlags.Renderable が使用されている。Entity が移動する場合は Dynamic に切り替えること。
オーサリングフィールドの変更が再ベイクされない Baker がそのフィールドを読んでいないか、外部アセットに対して DependsOn を呼んでいない。
ベイクが遅い Bake() 内で重い処理をしている。Entity をまたぐロジックは Baking System か、ランタイムの一回限りの初期化処理に移すこと。

Chapter 5. Spawner サンプル

1. 概要

Spawner は他の Entity を生成する単一の Entity だ — 起動時、タイマー、またはイベントで生成する。本ページではエンドツーエンドで1つ構築し、以下を解説する：

  • SubScene 内での Spawner のオーサリング。
  • Prefab Entity 参照の保存（GameObject Prefab からベイク）。
  • 構造変更が安全に行われるよう EntityCommandBuffer 経由でランタイムにインスタンス化する方法。

先に 01_Baker Pattern & SubScene.md を読んでおくと理解しやすい。

2. データモデル

using Unity.Entities;
using Unity.Mathematics;

public struct Spawner : IComponentData
{
    public Entity  Prefab;       // baked prefab entity
    public float3  SpawnPoint;
    public float   Interval;
    public float   TimeLeft;
    public int     RemainingCount;  // stop when 0
}

注目すべき2つの設計選択：

  • Prefab は GameObject ではなく Entity だ。Baker が GameObject Prefab を Prefab Entity に変換するため、ランタイムは Entity のみを扱う。
  • TimeLeft と RemainingCount は Spawner Entity 上に置くことで、System をステートレスに保つ。

3. オーサリング + Baker

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float      Interval = 0.5f;
    public int        Count    = 100;

    class Baker : Baker<SpawnerAuthoring>
    {
        public override void Bake(SpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Spawner
            {
                Prefab        = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                SpawnPoint    = authoring.transform.position,
                Interval      = authoring.Interval,
                TimeLeft      = 0f,            // fires immediately on first update
                RemainingCount = authoring.Count
            });
        }
    }
}

SubScene 内の空の GameObject に SpawnerAuthoring をドロップし、Prefab を割り当て、カウントを設定する。SubScene を閉じると — Spawner が Entity になる。

4. ランタイムシステム

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnerSystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<Spawner>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Use the EndSimulation ECB so structural changes flush at a safe point.
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged);

        foreach (var (spawnerRW, spawnerEntity) in
                 SystemAPI.Query<RefRW<Spawner>>().WithEntityAccess())
        {
            ref var spawner = ref spawnerRW.ValueRW;

            if (spawner.RemainingCount <= 0)
            {
                ecb.DestroyEntity(spawnerEntity);   // done — remove the spawner
                continue;
            }

            spawner.TimeLeft -= dt;
            if (spawner.TimeLeft > 0f) continue;

            spawner.TimeLeft = spawner.Interval;
            spawner.RemainingCount--;

            var instance = ecb.Instantiate(spawner.Prefab);
            ecb.SetComponent(instance, LocalTransform.FromPosition(spawner.SpawnPoint));
        }
    }
}

重要なポイント：

  • RequireForUpdate<Spawner>() は Spawner が存在しない場合にシステムを丸ごとスキップする。
  • EndSimulationEntityCommandBufferSystem Singleton は共有 ECB を提供し、フレーム終了時にフラッシュされる — ループ中に EntityManager 経由で直接変更するよりも安全だ。
  • ecb.Instantiate(prefab) は 遅延 Entity を作成する；ecb.SetComponent(instance, …) はそれに対するコンポーネント書き込みをスケジュールする。instance ハンドルはこの ECB のプレイバック内でのみ有効だ。14_EntityCommandBuffer · Deferred Entity.md を参照のこと。

5. スケールアップ — 並列 Spawner

多数の Spawner から1フレームで数千体スポーンする必要がある場合は、並列 ECB ライターを使った IJobEntity に変換する：

[BurstCompile]
public partial struct SpawnJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, entity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

System 側は：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

new SpawnJob { DeltaTime = SystemAPI.Time.DeltaTime, ECB = ecb }
    .ScheduleParallel();

chunkIndex ソートキーが並列プレイバックを決定論的に保つ鍵だ。15_ParallelWriter · Deterministic Playback.md を参照のこと。

6. トラブルシューティング

症状 原因 / 対処法
何もスポーンされない Spawner Entity が存在しない — SubScene を開いて SpawnerAuthoring がアタッチされていること、SubScene が閉じていること（黄色のアイコン）を確認すること。
Prefab がワールド原点にスポーンされる ECB で LocalTransform が設定されていない。ecb.SetComponent(instance, LocalTransform.FromPosition(...)) を追加するか、Prefab に目的の位置のトランスフォームを持たせること。
Prefab が表示されるが動かない Prefab が Dynamic ではなく TransformUsageFlags.Renderable でベイクされている。
SystemAPI.GetSingleton<...ECBS.Singleton>() が例外を投げる ECB System が登録されていない — 非常に削ぎ落とされた World で発生する。state.EntityManager + 手動 ECB にフォールバックするか、Bootstrap で ECB System を登録すること。
Spawner が停止しない RemainingCount が spawnerRW.ValueRW ではなくキャッシュされた Spawner に対してデクリメントされている。常に RefRW.ValueRW 経由で書き込むこと。
遅延インスタンスに対する ecb.SetComponent が無視される Instantiate と SetComponent で異なる ECB を使用している。両方の呼び出しは同じ ECB インスタンスを共有しなければならない。

Chapter 6. ECS コアコンセプト

1. 5つの名詞

Entities ECS は5つの名詞に集約される：

Figure: ECS コアの名詞。

この図を内面化することが、既存の DOTS コードを読む上でほぼすべてだ。

2. Entity

Entity は単なる 64 ビットのハンドルだ：struct Entity { int Index; int Version; }。データもメソッドも持たず、識別子以外の役割はない — すべてのデータはコンポーネントに置かれる。

Entity e = state.EntityManager.CreateEntity();   // fresh, no components
state.EntityManager.AddComponent<Health>(e);     // now it has data
state.EntityManager.DestroyEntity(e);            // handle becomes stale

Version フィールドは古い参照を無効化する：同じ Index で破棄・再生成すると、古い Entity 値はもはや一致しない。

3. Component

Component は純粋なデータコンテナだ。振る舞いを持たない。コードは System に置かれる。

public struct Health : IComponentData
{
    public float Value;
    public float Max;
}

コンポーネントにはいくつかのバリアントがある（アンマネージド、マネージド、バッファ、共有、クリーンアップ、タグ、チャンク、Enableable）。05_Component Types.md を参照のこと。

4. Archetype

Archetype は複数の Entity が共有するコンポーネント型の一意なセットだ。

  • { LocalTransform, Health, Enemy } を持つ Entity は Archetype A に属する。
  • { LocalTransform, Health, Enemy, Burning } を持つ Entity は Archetype B に属する — Burning を追加することでコンポーネントセットが変わったため、別の Archetype だ。

Archetype 間の移動（コンポーネントの追加/削除）は 構造変更 であり、Entity を物理的に新しいチャンクに移動させる。13_Structural Change & Safety.md を参照のこと。

5. Chunk

同じ Archetype の Entity は 16 KB チャンクに詰め込まれる。チャンク内では各コンポーネント型が連続配列を形成する — キャッシュ局所性に優れる。

事実 含意
チャンクは 16 KB 各コンポーネントがスライスを消費する；Archetype のコンポーネントが多いほど（または大きいほど）、チャンあたりの Entity 数が減る。
データはコンポーネントごとに列指向（SoA）に配置される IJobChunk は1コンポーネント配列ずつイテレートでき、Burst でベクトル化できる。
コンポーネントの追加/削除で Entity が移動する 構造変更。クエリイテレーターはイテレート中に Archetype が安定していることを前提とする。
チャンクはコンポーネントの変更バージョンを追跡する System は前回の実行以降に変更されていないチャンクをスキップできる（EntityQuery.SetChangedVersionFilter）。

ライブのチャンクレイアウトは Window → Entities → Archetypes で確認できる。

6. System

System は各フレームに実行されるロジックの断片で、通常は Entity のクエリに対して処理を行う。

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var health in SystemAPI.Query<RefRW<Health>>())
        {
            health.ValueRW.Value = math.min(health.ValueRW.Value + dt, health.ValueRW.Max);
        }
    }
}

System は自動的に登録され、SystemGroup 内に存在する。08_System — ISystem vs SystemBase.md および 09_System Group & Update Order.md を参照のこと。

7. World

World は以下を所有するコンテナだ：

  • EntityManager（Entity の作成、破棄、クエリ）。
  • 一連の System とその SystemGroup。
  • チャンク自体（EntityManager 経由）。

デフォルトのランタイム World は DefaultGameObjectInjectionWorld だ。ほとんどのゲームは本番環境で1つの World を持つ；マルチ World セットアップは Netcode（サーバー/クライアント）やエディターツールで登場する。Netcode の World 分割については 16_Netcode Client-Server World & Bootstrap.md を参照のこと。

World world = World.DefaultGameObjectInjectionWorld;
EntityManager em = world.EntityManager;

Entity e = em.CreateEntity(typeof(Health));
em.SetComponentData(e, new Health { Value = 100, Max = 100 });

8. EntityManager

EntityManager は構造変更のためのメインスレッド API だ：作成、破棄、コンポーネントの追加/削除、Prefab Entity のインスタンス化、World のコピーなど。EntityManager 経由の構造変更は即時に行われ、実行中のクエリイテレーターやジョブ参照を無効化する可能性がある。

ジョブからやイテレーション中の遅延構造変更には EntityCommandBuffer を使用すること — 14_EntityCommandBuffer · Deferred Entity.md を参照のこと。

9. まとめ

World
└── EntityManager
    └── Archetype { LocalTransform, Health, Enemy }
        └── Chunk  (up to ~N entities)
            └── Entity { Index, Version }
                ├── LocalTransform
                ├── Health
                └── Enemy (tag)

{ LocalTransform, Health } をクエリする System は、Archetype が両方を含むすべてのチャンクに触れる。これには上記の Archetype だけでなく、Enemy と追加コンポーネントを持つ他の Archetype も含まれる。

10. トラブルシューティング

症状 原因 / 対処法
Entity の検索が古いデータを返す Entity が破棄・再生成された — Version がもはや一致しない。Entity を再解決すること。
クエリが何もマッチしないが、Entity に明らかにコンポーネントがある コンポーネントが別の World で追加されたか、Enableable コンポーネントが現在無効になっている。
Archetype の数が爆発する ユニークなコンポーネントの組み合わせが多すぎる（高カーディナリティのエンティティ別タグによく見られる）。共有コンポーネントや Enableable フラグで統合すること。
チャンクに 1〜2 体の Entity しかいない Archetype が大きすぎる — Entity に不要なコンポーネントを削除するか、複数の Entity/Archetype に分割すること。
イテレーション中に InvalidOperationException: … invalidated … クエリの途中で構造変更が発生した。ECB を使うか、ループ後に変更をまとめること。

Chapter 7. Entity 参照 — Entity · EntityPrefabReference · UnityObjectRef

1. 概要

DOTS コードは少数の参照型を使用する。すべて「何かを指す」ため混同しやすいが、オーサリングからランタイムへのパイプラインの異なる箇所に位置する。

型 名前空間 参照先 典型的な用途
Entity Unity.Entities 1つの ECS World 内の Entity。 ランタイムの関係性とコンポーネントフィールド。
EntityPrefabReference Unity.Entities.Serialization 後でロード可能なベイク済み Entity Prefab アセット。 ストリーミングとコンテンツワークフロー。
UnityObjectRef<T> Unity.Entities アンマネージドラッパー経由の UnityEngine.Object アセット/オブジェクト。 ECS データからメッシュ、マテリアル、テクスチャ、オーディオクリップ、オーサリングアセットを参照する。

本ページは DOTS / Entities の参照型のみを対象とする。エンジン全体の UnityEngine.Object 識別子 API はこのマニュアルの対象外だ。

2. Entity

Entity はランタイム ECS のハンドルだ。1つの World 内のコンポーネントデータの行を識別するアンマネージドな値だ。

using Unity.Entities;

public struct Target : IComponentData
{
    public Entity Value;
}

重要なルール：

  • Entity 値はそれを作成した World 内でのみ意味を持つ。
  • Entity を破棄すると古いハンドルが無効化される。
  • Entity インデックスは再利用される；保存済みハンドルを直接クエリしていない場合は参照前に EntityManager.Exists(entity) で確認すること。
  • 生の Entity 値を安定した ID としてディスクに保存したり独立した World 間で送ったりしないこと。
if (state.EntityManager.Exists(target.Value))
{
    var transform = state.EntityManager.GetComponentData<LocalTransform>(target.Value);
}

3. ベイク時の Entity 参照

Baker がオーサリングオブジェクトを ECS データに変換する際は、参照先のオーサリングオブジェクトや Prefab を Entity 参照に変換するために GetEntity を使用する。

using Unity.Entities;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
}

public struct Spawner : IComponentData
{
    public Entity Prefab;
}

public class SpawnerBaker : Baker<SpawnerAuthoring>
{
    public override void Bake(SpawnerAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new Spawner
        {
            Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic)
        });
    }
}

結果はランタイム ECS の参照だ。マネージドな GameObject 参照ではない。

4. EntityPrefabReference

EntityPrefabReference は、現在のシーンに直接 Entity 参照として埋め込む代わりに、Entities コンテンツパイプラインを通じてロードすべき Prefab コンテンツ向けだ。

using Unity.Entities;
using Unity.Entities.Serialization;
using UnityEngine;

public class ProjectileLibraryAuthoring : MonoBehaviour
{
    public GameObject ProjectilePrefab;
}

public struct ProjectileLibrary : IComponentData
{
    public EntityPrefabReference ProjectilePrefab;
}

public class ProjectileLibraryBaker : Baker<ProjectileLibraryAuthoring>
{
    public override void Bake(ProjectileLibraryAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new ProjectileLibrary
        {
            ProjectilePrefab = new EntityPrefabReference(authoring.ProjectilePrefab)
        });
    }
}

Prefab がコンテンツ管理やストリーミングワークフローに属する場合は EntityPrefabReference を使用する。Prefab が同じ SubScene にベイクされシーンとともに利用可能な場合は、通常の Entity Prefab 参照を使用する。

5. UnityObjectRef<T>

UnityObjectRef<T> を使うと、アンマネージドな ECS データがコンポーネントをマネージドコンポーネントにすることなく UnityEngine.Object を参照できる。

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

オーサリングデータからベイクする：

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

.Value の参照解除はメインスレッドまたはマネージドなプレゼンテーションコードでのみ行うこと：

Mesh mesh = visual.Mesh.Value;
Material material = visual.Material.Value;

ルール：

  • 保存されるラッパーはアンマネージドで IComponentData 内に置ける。
  • 参照先のオブジェクトは依然としてマネージド Unity オブジェクトであり、.Value は Burst ジョブ内で使える操作ではない。
  • 参照解除時に参照先オブジェクトがロードされていなければならない。

6. 適切な参照の選択

必要なもの 使用する型
同じ World 内の別の Entity を指す Entity
同じ SubScene にベイクされた Prefab を保存する GetEntity(authoring.Prefab, ...) からの Entity Prefab 参照
コンテンツストリーミング/遅延ロード向けの Prefab を保存する EntityPrefabReference
アンマネージド ECS データにメッシュ、マテリアル、テクスチャ、オーディオクリップ、ScriptableObject 参照を保存する UnityObjectRef<T>

7. トラブルシューティング

症状 原因 / 対処法
保存した Entity がもはや存在しない Entity が破棄されたか別の World に属している。再解決するか EntityManager.Exists で確認すること。
ベイク後に Prefab 参照が Entity.Null になる Baker が Prefab に対して GetEntity を呼んでいないか、Prefab がベイク済みシーン/コンテンツワークフローに含まれていない。
UnityObjectRef<T>.Value が null を返す 参照先の Unity オブジェクトがロードされていないか破棄されている。参照解除前にガードし、アセットを再ロード/再解決すること。

セーブファイルに生の Entity 値が保存されている 生の Entity ハンドルは World ローカルで安定していない。代わりにゲーム固有の ID またはコンテンツキーを保存すること。

Chapter 8. Component 型

1. 概要

ECS における「コンポーネント」とは Entity にアタッチされるデータコンテナだ。Entities はストレージ、ライフタイム、コストのプロファイルが異なるいくつかの種類を提供する。適切なものを選ぶことが ECS System 設計の半分だ。

簡易マップ：

インターフェース / 形式 ストレージ 変更時に構造変更が発生するか 典型的な用途
IComponentData（アンマネージド struct） チャンク内、SoA いいえ（値変更時） デフォルトのコンポーネント
IComponentData（マネージド class） サイドテーブル いいえ 参照型（メッシュ、GameObject 参照）
IBufferElementData チャンク内 + ヒープオーバーフロー いいえ Entity ごとの動的配列
ISharedComponentData チャンクごと はい（Entity を新しいチャンクに移動） 設定を共有する Entity のグループ化
ICleanupComponentData チャンク内 通常 Entity 破棄クリーンアッププロトコル
ICleanupSharedComponentData チャンクごと 通常 共有リソースクリーンアッププロトコル
タグ（空の IComponentData） チャンク内でゼロサイズ いいえ フィルターマーカー、データなし
チャンクコンポーネント チャンクごと はい チャンクごとに一度計算されるメタデータ
Enableable（IEnableableComponent） チャンク内 + ビットマスク いいえ Entity ごとに振る舞いを ON/OFF する

本ページの残りでは各行を詳しく説明する。

2. アンマネージド IComponentData

デフォルト。密にパックされた SoA データとしてチャンク内に直接置かれる。Burst と相性がよい。

public struct Velocity : IComponentData
{
    public float3 Value;
}

ルール：

  • struct でなければならない — 参照フィールド不可（class、string、List<T> は不可）。
  • 許可されるフィールド型：プリミティブ、Unity.Mathematics、他のアンマネージド構造体、Entity、FixedString*、BlobAssetReference<T>、UnityObjectRef<T>。
  • SystemAPI.Query から RefRO<T> で読み取り、RefRW<T> で書き込む。

3. マネージド IComponentData

Entity でインデックスされたサイドテーブルに置かれる。参照型を扱えるが、GC アロケーションのコストがかかり、そのコンポーネントに対する Burst 互換性が失われる。

public class EnemyAI : IComponentData
{
    public AIModel Brain;      // a managed class
    public List<Entity> Allies;
}

マネージド参照が本当に必要な場合（メッシュ、テクスチャ、オーサリングされた ScriptableObject）に使用する。UnityEngine.Object への弱参照が必要なだけなら、アンマネージドコンポーネント内の UnityObjectRef<T> を優先すること。

4. IBufferElementData — 動的バッファ

DynamicBuffer<T> は Entity ごとにリサイズ可能な配列だ。

public struct Waypoint : IBufferElementData
{
    public float3 Position;
}

var buffer = state.EntityManager.AddBuffer<Waypoint>(entity);
buffer.Add(new Waypoint { Position = new float3(1, 0, 0) });

ストレージ：InternalBufferCapacity 要素数まではチャンク内に置かれ、それを超えるとヒープにあふれる。キャパシティは [InternalBufferCapacity(N)] で調整する。

5. ISharedComponentData — グループ化

共有コンポーネントは チャンクをグループ化する値 を持つ。同じ ISharedComponentData 値を持つ2つの Entity は同じチャンクに入る；異なる値を持つと異なるチャンクが生成される。

public struct RenderMeshGroup : ISharedComponentData
{
    public int MaterialID;
}

結果：

  • 値を変更すると Entity が別のチャンクに移動する → 構造変更、コストが高い。
  • 頻繁に変わらないがグループによって異なるデータに最適（マテリアルセット、チームID、ベイク領域）。
  • 型が参照を含む場合、値はマネージドテーブルに保存される（ISharedComponentData はマネージドにできる）。

6. クリーンアップコンポーネント — ICleanupComponentData

Entity に「破棄」されたときに、専用システムがそれらを処理するまでクリーンアップコンポーネントだけを残して生き続けるようにマークする。

public struct ConnectionCleanup : ICleanupComponentData
{
    public int ConnectionHandle;
}

// DestroyEntity(e) removes all normal components but leaves ConnectionCleanup
// behind so you can close the socket in a dedicated system, then destroy the
// cleanup component explicitly.

兄弟の ICleanupSharedComponentData は同じプロトコルを共有リソースに適用する。

ユースケース：ネイティブリソースの解放、外部システムからの登録解除、フレームをまたぐ段階的な破棄。

7. タグコンポーネント

タグは空の IComponentData だ：

public struct Enemy : IComponentData {}

チャンク内でゼロバイト、他のコンポーネントと同様にクエリ可能だ。「X の一種である」マーカーとして理想的だ。タグを使いすぎる（数百のユニークなもの）と Archetype が断片化する — 高カーディナリティの状態には Enableable コンポーネントや共有コンポーネントを使用すること。

8. チャンクコンポーネント

チャンクコンポーネントは Entity ごとではなく チャンクごとに1つの値 を持つ。

state.EntityManager.AddChunkComponentData<RegionBounds>(query, new RegionBounds { ... });

典型的な用途：チャンク内のすべての Entity に適用される事前計算データ（バウンディング、リージョン ID、パーティションキー）。チャンクコンポーネントを変更すると構造変更が発生する。

9. Enableable コンポーネント（IEnableableComponent）

詳細は専用ページ 06_Enableable Component.md を参照のこと。要約：構造変更なしで Entity ごとに ON/OFF できるコンポーネントで、チャンクごとのビットマスクを使用する。

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

state.EntityManager.SetComponentEnabled<Stunned>(entity, false);  // no structural change

クエリは無効化されている Enableable コンポーネントを持つ Entity を自動的にスキップする — システムから Entity を一時的に除外する安価でよく使われる方法だ。

10. 判断ガイド

やりたいこと 使用するもの
Entity ごとの数値データを保存する アンマネージド IComponentData
Entity ごとにマネージド参照を保持する マネージド IComponentData または UnityObjectRef<T>
Entity ごとに可変長リストを保存する IBufferElementData
バッチ処理のために Entity をグループ化する ISharedComponentData
フレームごとにコンポーネントを ON/OFF する Enableable コンポーネント
型をマークするだけでよい タグ（空の IComponentData）
チャンク全体のデータを事前計算する チャンクコンポーネント
Entity が破棄されたときにコードを実行する クリーンアップコンポーネント

11. トラブルシューティング

症状 原因 / 対処法
InvalidOperationException: The type T must be unmanaged アンマネージドが必要な箇所でマネージド型を使用した。コンポーネントまたは API を切り替えること（例：ComponentLookup<T> vs ManagedAPI.GetComponent<T>）。
共有コンポーネント値の設定でカクつく 各変更は構造変更だ。ECB でバッチするか、この値は稀にしか変更しないと割り切ること。
DynamicBuffer<T> がヒープにあふれる [InternalBufferCapacity(N)] が使用量に対して小さすぎる。増やすかサイズを制限すること。
DestroyEntity 後に Entity が再出現する クリーンアップコンポーネントがまだアタッチされている。クリーンアップ完了後に明示的に削除すること。
Archetype が多すぎる タグの爆発だ。Entity ごとのユニークな「状態」を Enableable コンポーネントに変換すること。
タグコンポーネントがクエリをフィルタリングしない タグコンポーネントはデータを持たないため、SystemAPI.Query<...> タプルで RefRW<Tag> / RefRO<Tag> を要求できない。代わりに .WithAll<Tag>() / .WithNone<Tag>() を使用すること。

Chapter 9. Enableable Component

1. Enableable コンポーネントが存在する理由

コンポーネントの追加・削除は 構造変更 だ：Entity が別のチャンクに移動し、クエリは再解決が必要になり、参照が無効化される可能性がある。これを Entity ごとにフレームごとに行う場合（例：「Stunned」の ON/OFF）、そのコストがフレーム全体を支配してしまう。

Enableable コンポーネント は構造変更なしで振る舞いの ON/OFF を切り替える方法を提供する。コンポーネントは常にチャンク上に存在しており、チャンクごとのビットマスク内の1ビットが、クエリの観点から各 Entity がそれを「持っているか」を決定する。

2. インターフェース

IComponentData と IEnableableComponent の両方を実装する：

using Unity.Entities;

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

IBufferElementData も Enableable にできる。

3. 切り替え

System から：

// Turn Stunned off without removing the component
state.EntityManager.SetComponentEnabled<Stunned>(entity, false);

// From an ECB (deferred, safe mid-iteration)
ecb.SetComponentEnabled<Stunned>(entity, true);

// From a job with a ComponentLookup
var stunnedLookup = SystemAPI.GetComponentLookup<Stunned>();
stunnedLookup.SetComponentEnabled(entity, false);

現在の状態を読み取る：

bool isStunned = state.EntityManager.IsComponentEnabled<Stunned>(entity);

3つの方法はいずれもビットマスクを操作し、構造変更は発生させない。

4. クエリの動作

SystemAPI.Query<...> および EntityQuery ベースのイテレーションは、Enableable コンポーネントが現在無効になっている Entity をスキップする — デフォルトでは、リストされたすべてのコンポーネントが有効な Entity のみが表示される。

foreach (var stunned in SystemAPI.Query<RefRW<Stunned>>())
{
    // only fires for entities where Stunned is enabled
}

無効な Entity も確認する必要がある場合はオーバーライドする：

foreach (var (stunnedRW, entity) in SystemAPI
             .Query<RefRW<Stunned>>()
             .WithEntityAccess()
             .WithOptions(EntityQueryOptions.IgnoreComponentEnabledState))
{
    // fires for ALL entities with the component, enabled or not
}

5. サンプル — 時間経過で解除されるスタン

単一の IJobEntity は同じコンポーネントについて ref Stunned（RefRW<T>）と EnabledRefRW<Stunned> の両方を受け取ることはできない — Entities は1つのジョブで同じコンポーネントを両方の形式でラップすることを明示的に禁止している。有効な2つのパターン：

5.1 値を読み書きし、有効ビットを ECB 経由で切り替える

[BurstCompile]
public partial struct StunTickJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Stunned stunned)
    {
        stunned.Duration -= DeltaTime;
        if (stunned.Duration <= 0f)
            ECB.SetComponentEnabled<Stunned>(chunkIndex, entity, false);
    }
}

プレイバックは次の ECB バウンダリ（例：EndSimulationECBS）で行われる — 14_EntityCommandBuffer · Deferred Entity.md を参照のこと。構造変更なし；既知のポイントでビットマスクが反転するだけだ。

5.2 コンポーネントの値が不要な場合に有効ビットを直接切り替える

[BurstCompile]
public partial struct StunDisableJob : IJobEntity
{
    void Execute(EnabledRefRW<Stunned> stunnedEnabled)
    {
        stunnedEnabled.ValueRW = false;   // no structural change
    }
}

EnabledRefRW<T> がこのジョブでの唯一の Stunned 参照なのでコンパイルできる。System が純粋なスイッチャーである場合にこの形式を使用する。

1回のパスで値を読み取りかつ有効ビットを同じ Entity で切り替えたい場合は、メインスレッドから EntityManager.SetComponentEnabled<T>(entity, false) で行うか、state.Dependency でチェーンした2つのジョブに分割すること。

6. Enableable vs 追加/削除の使い分け

状況 選択
高頻度の切り替え（フレームごと、イベントごと） Enableable
常に存在するわけではない状態を追加する（可変フィールドを持つ Entity ごとの修飾子） Enableable
低頻度の Archetype 変更（例：恒久的な能力の追加） 通常どおりコンポーネントを追加/削除する
多くのシステムを均一にフィルタリングする Enableable（クエリが自動的に拾い上げる）

一般ルール：「このコンポーネントを持っているか」ビットが Entity の生涯に複数回反転するなら Enableable にする。

7. 他の機能との相互作用

機能 動作
WithChangeFilter<T> 有効ビットの切り替えもコンポーネントの変更バージョンをバンプする。
ICleanupComponentData クリーンアップコンポーネントは Enableable にもなれない。
Netcode for Entities Enableable コンポーネントは Ghost 同期できるが、有効/無効の切り替えはオーソリティ側を通す必要がある。
シリアライゼーション / World エクスポート 有効ビットは保持される。

8. トラブルシューティング

症状 原因 / 対処法
クエリに「OFF」にすべき Entity が含まれる コンポーネントが IEnableableComponent を実装していないか、WithOptions(EntityQueryOptions.IgnoreComponentEnabledState) が設定されている。
SetComponentEnabled<T> が「type T does not support being enableable」と例外を投げる struct が IComponentData のみを実装している — IEnableableComponent を追加すること。
EnabledRefRW<T> パラメーターがソースジェネレーターに拒否される ジョブまたはシステムに partial がないか、コンポーネントが Enableable でない。
ソースジェネレーターエラー「cannot have both RefRW and EnabledRefRW for the same component」 ロジックを分割すること — ジョブごとに ref T または EnabledRefRW<T> のどちらかを使い、同じコンポーネントで両方を使わない。§5 の2つのパターンを参照。
変更フィルターが誤って発火する Enabled の切り替えは WithChangeFilter の目的では変更としてカウントされる。予期した動作だ — 切り替えではなく EnabledRefRO の読み取りと組み合わせること。
ECB の SetComponentEnabled が無視される それを作成した ECB がフラッシュされる前に遅延 Entity に対して呼び出した。両方が同じ ECB サイクル内になければならない。

Chapter 10. Singleton Component

1. Singleton とは何か（そして何でないか）

Entities における Singleton は、World 内のちょうど1つの Entity に存在することが期待されるコンポーネントに過ぎない。特別な ISingleton インターフェースはない — 任意の IComponentData が慣例によって Singleton になり、SystemAPI が「ちょうど1つ」のルールをアサートする使いやすいアクセサーを提供する。

典型的な用途：ゲーム状態、グローバル設定、時間/ティックソース、ネットワーク状態、中央 ECB 登録。

public struct GameState : IComponentData
{
    public int   Score;
    public float TimeScale;
    public bool  Paused;
}

2. アクセサー

任意の ISystem / SystemBase 内で：

// Reads
GameState gs      = SystemAPI.GetSingleton<GameState>();
RefRW<GameState> gsRW  = SystemAPI.GetSingletonRW<GameState>();
bool has          = SystemAPI.HasSingleton<GameState>();
bool got          = SystemAPI.TryGetSingleton<GameState>(out var gs2);

// Writes
SystemAPI.SetSingleton(new GameState { Score = 10 });

// Entity handle, if you need it
Entity e          = SystemAPI.GetSingletonEntity<GameState>();

// Buffers work too
DynamicBuffer<Waypoint> buf = SystemAPI.GetSingletonBuffer<Waypoint>();

Get* バリアントは Singleton が0個または2個以上存在する場合に例外を投げる。TryGet* はその場合に false を返す。RW バリアントはコンポーネントを変更済みとしてマークし、変更追跡が機能する。

3. Singleton の作成

よく使われる2つのパターン：

3.1 SubScene からベイクする

public class GameStateAuthoring : MonoBehaviour
{
    public float InitialTimeScale = 1f;

    class Baker : Baker<GameStateAuthoring>
    {
        public override void Bake(GameStateAuthoring authoring)
        {
            var e = GetEntity(TransformUsageFlags.None);
            AddComponent(e, new GameState
            {
                Score     = 0,
                TimeScale = authoring.InitialTimeScale,
                Paused    = false
            });
        }
    }
}

SubScene 内の1つの GameObject に GameStateAuthoring をアタッチする；Baker がちょうど1つの Entity を Singleton として生成する。

3.2 System 起動時に作成する

public void OnCreate(ref SystemState state)
{
    var e = state.EntityManager.CreateEntity(typeof(GameState));
    state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
}

オーサリングに置くべきでないデータ（サーバーから注入される設定、手続き生成など）に使用する。

4. Singleton に System をゲートする

Singleton が存在した後にのみ意味を持つ System がある場合：

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<GameState>();
}

RequireForUpdate<T>() は、コンポーネントが存在しないときに OnUpdate を丸ごとスキップするよう System に指示する。OnUpdate 内の防御的なヌルチェックを避けられる。

5. 読み取り専用 vs 読み書きのパフォーマンス

  • GetSingleton<T>() — コピーを読み取る。変更追跡なし。
  • GetSingletonRW<T>() — チャンクを指す RefRW<T> を返す。.ValueRW に代入するとコンポーネントが変更済みとしてマークされる。これは他の System が WithChangeFilter<T> を使用する場合に必要だ。
  • SetSingleton — 完全な書き込み、常にバージョンをバンプする。

読み取りが多い System では GetSingleton を優先し、実際に変更する場合のみ GetSingletonRW に切り替えること。

6. 複数の候補 — 「Singleton が2つ以上ある」トラップ

GetSingleton<T> は InvalidOperationException: expected exactly one entity with component T, found N を投げる。2つの Entity が同じ Singleton コンポーネントを持った場合に起きる。よくある原因：

  • SubScene 内の2つの GameObject の両方にオーサリングスクリプトがアタッチされている。
  • テストフィクスチャがランタイムで Singleton を作成し、かつ SubScene もそれをベイクする。
  • System が HasSingleton でガードせずに OnCreate で Singleton を作成する。

防御的パターン：

public void OnCreate(ref SystemState state)
{
    if (!SystemAPI.HasSingleton<GameState>())
    {
        var e = state.EntityManager.CreateEntity(typeof(GameState));
        state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
    }
}

7. バッファ Singleton

DynamicBuffer<T> も Singleton になれる：

public struct QueuedEvent : IBufferElementData
{
    public int Type;
    public int Payload;
}

// Access:
DynamicBuffer<QueuedEvent> events = SystemAPI.GetSingletonBuffer<QueuedEvent>();
events.Add(new QueuedEvent { Type = 1 });

グローバルイベントキューやコマンドストリームに有用だ。

8. トラブルシューティング

症状 原因 / 対処法
InvalidOperationException: expected exactly one entity with component T, found 0 Singleton Entity が作成されていない。baker を追加するか、HasSingleton ガード付きで OnCreate で作成すること。
... found 2 2つの Entity がコンポーネントを持っている。オーサリングで重複を排除するか、起動時にクリーンアップすること。
GetSingletonRW がコンポーネントを変更済みとしてマークしていないようだ .ValueRO に書き込んだか、RefRW を無視した。変更追跡をトリガーするのは .ValueRW のみだ。
Singleton が作成される前に System が実行される RequireForUpdate<T> を使用するか、作成者が先に実行されるよう System を順序付けること（[UpdateBefore]）。
順序付けなしで2つの System から Singleton が変更される [UpdateAfter]/[UpdateBefore] を追加するかロジックをマージすること。Singleton の競合状態は特にデバッグが難しい。
Netcode：クライアントとサーバーで Singleton の値が乖離する Singleton を ghost 同期するか、同期済み入力から決定論的に再構築すること。

Chapter 11. System — ISystem vs SystemBase

1. 概要

Entities には System を宣言する2つの方法がある：ISystem（アンマネージド struct）と SystemBase（マネージド class）だ。SystemBase が必要な特定の理由がない限り ISystem を選ぶこと。

ISystem SystemBase
宣言形式 partial struct : ISystem partial class : SystemBase
メモリ アンマネージド、スタックアロケート可 マネージド、GC 管理
Burst コンパイル可 可（[BurstCompile] を追加） 不可
マネージドフィールド 不可 可
マネージド API アクセス EntityManager + ManagedAPI 経由 直接
デフォルトの選択 ✅ 必要な場合のみ

2. ISystem — デフォルト

using Unity.Burst;
using Unity.Entities;

[BurstCompile]
public partial struct EnemyAISystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        // Called once when the system is created.
        state.RequireForUpdate<EnemySpawnConfig>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var config = SystemAPI.GetSingleton<EnemySpawnConfig>();
        // ... logic
    }

    [BurstCompile]
    public void OnDestroy(ref SystemState state)
    {
        // Called once when the system is destroyed.
    }
}

必須の要素：

  • struct は partial でなければならない — ソースジェネレーターが隠れたラッパーを追加する。
  • 3つのメソッド（OnCreate、OnUpdate、OnDestroy）はすべてオプション — 不要なものは省略できる。
  • struct と Burst コンパイルしたい各メソッドに [BurstCompile] を付ける。

3. SystemBase — マネージド System が避けられない場合

using Unity.Entities;

public partial class PlayerInputSystem : SystemBase
{
    private InputActions _input;   // managed field — the reason SystemBase exists

    protected override void OnCreate()
    {
        _input = new InputActions();
        _input.Enable();
    }

    protected override void OnUpdate()
    {
        var move = _input.Gameplay.Move.ReadValue<Vector2>();
        // ... dispatch to components
    }

    protected override void OnDestroy() => _input.Dispose();
}

SystemBase を使うのは次のような場合だ：

  • System と共に存在しなければならないマネージドフィールドを保持する場合（Input System オブジェクト、UI Toolkit ハンドル、マネージドサービスなど）。
  • EntityManager 経由のマーシャリングなしに OnUpdate 内で直接マネージド API を呼び出す必要がある場合。

SystemBase のすべては Burst なしでメインスレッドで実行される。

4. System のライフサイクル

両方のフレーバーが同じフック名を共有する：

Figure: System ライフサイクル。

フック 発火タイミング 典型的な用途
OnCreate 一度、System 作成時 クエリのキャッシュ、RequireForUpdate<T>、ネイティブコンテナの確保
OnStartRunning クエリが空 → 非空に変わったとき セッションごとの状態のリセット
OnUpdate 親 SystemGroup の毎フレーム メインの処理
OnStopRunning クエリが非空 → 空に変わったとき バッファされた状態のフラッシュ
OnDestroy 一度、World 終了時 ネイティブ/マネージドリソースの解放

RequireForUpdate<T>()

必要なコンポーネントや Singleton が存在しない場合に OnUpdate を完全にスキップする：

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<EnemySpawnConfig>();
    state.RequireForUpdate<GameRunning>();
}

これは System を「設定が存在する」「ゲームが開始済み」といった条件で実行制御する標準的な方法だ。

5. 2つの選択

経験則：

  1. ISystem から始める。 Burst フレンドリーで GC フリーだ。
  2. マネージドフィールドが必要な場合のみ SystemBase に切り替える。 その場合でも分割を検討すること：マネージドな処理を結果をコンポーネントに押し込む薄い SystemBase に保ち、重いロジックはそのコンポーネントを読む ISystem で実行する。
  3. ISystem struct にマネージドフィールドを追加して混在させない — コンパイルエラーになる。上記の分割がイディオマティックな回避策だ。

分割の例：

// Managed — bridges Input System into components.
public partial class InputBridgeSystem : SystemBase
{
    private InputActions _input;
    // ... fills a PlayerInput singleton each frame
}

// Unmanaged + Burst — reads the PlayerInput singleton and drives gameplay.
[BurstCompile]
public partial struct PlayerMoveSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) { /* ... */ }
}

6. SystemState クイックリファレンス

ISystem 内では ref SystemState state が World へのウィンドウだ。

メンバー 用途
state.EntityManager メインスレッドの構造変更。
state.World 所有している World。
state.Dependency 現在のジョブ依存関係ハンドル — スケジュールしたジョブと結合する。
state.GetEntityQuery(...) キャッシュされたクエリの構築または取得。
state.RequireForUpdate<T>() コンポーネントの存在に System をゲートする。
state.Enabled false にすると System を破棄せずにスキップする。

SystemBase では、これらはクラス自体のメンバーだ（例：this.EntityManager）。

7. トラブルシューティング

症状 原因 / 対処法
error CS0246: The type or namespace name 'Entities' could not be found using Unity.Entities; がない。
OnUpdate が一度も発火しない RequireForUpdate<T> がまだ存在しないコンポーネントでゲートしている — そのコンポーネントが他で作成されていることを確認すること。
コンパイルエラー：「partial struct must be declared partial」 ISystem はソースジェネレーターが struct を拡張できるよう partial が必要だ。

System が Window → Entities → Systems に表示されない partial がない、デフォルト World 外にある、または [DisableAutoCreation] で除外されている。

Chapter 12. System Group & 更新順序

1. 概要

すべての System は SystemGroup 内に存在する。グループは各フレームに子がいつ実行されるかを決定し、順序付け属性がグループ内のシーケンスを決定する。グループ分けを適切に行うことで、フレーム遅延の書き込み、競合状態、「なぜこの Entity がジッターするのか」というバグを避けられる。

2. 組み込みグループ

Figure: System Group と更新順序。

グループ 実行タイミング 典型的な内容
InitializationSystemGroup フレームごとに一度、シミュレーション前 一回限りの初期化処理、ブートストラップタスク
SimulationSystemGroup フレームごとに一度 ほとんどのゲームプレイロジック
FixedStepSimulationSystemGroup 固定タイムステップ（デフォルト 60 Hz）；フレームごとに 0、1、または N 回実行 物理演算、決定論的ステップロジック
PresentationSystemGroup フレームごとに一度、シミュレーション後 レンダリング準備、補間
BeginSimulationEntityCommandBufferSystem Simulation の最初 ECB プレイバックの順序付け（早期）
EndSimulationEntityCommandBufferSystem Simulation の最後 ECB プレイバックの順序付け（後期）

Netcode for Entities はさらに多くのグループを追加する（PredictedSimulationSystemGroup、GhostSendSystem など）— 上記のグループ内にネストされる。16_Netcode Client-Server World & Bootstrap.md および 19_Netcode Prediction & Rollback.md を参照のこと。

3. グループへの System の配置

using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial struct EnemyAISystem : ISystem { /* ... */ }

[UpdateInGroup] を省略すると、System はデフォルトで SimulationSystemGroup になる。

ネスト

グループは他のグループ内に置ける：

[UpdateInGroup(typeof(SimulationSystemGroup), OrderFirst = true)]
public partial class GameplaySystemGroup : ComponentSystemGroup { }

[UpdateInGroup(typeof(GameplaySystemGroup))]
public partial struct MovementSystem : ISystem { /* ... */ }

ComponentSystemGroup はカスタムグループの基底クラスだ。System とコンテナの両方として機能する。

4. グループ内の順序付け

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateAfter(typeof(EnemyAISystem))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct EnemyPathfindingSystem : ISystem { /* ... */ }
属性 意味
[UpdateBefore(typeof(X))] X の前に実行（両方が同じグループ内にある必要がある）。
[UpdateAfter(typeof(X))] X の後に実行。
OrderFirst = true（[UpdateInGroup] で） グループ内でできるだけ早く実行。
OrderLast = true できるだけ遅く実行。

ヒント：OrderFirst/Last よりも [UpdateAfter] チェーンを優先すること — コードレビューで読みやすく、リファクタリングにも耐える。

5. [RequireMatchingQueriesForUpdate]

System の各クエリに少なくとも1体の Entity がマッチしない限り OnUpdate をスキップするよう指示する、新しい属性（Entities 1.x+）だ：

[RequireMatchingQueriesForUpdate]
public partial struct EnemyAISystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Only runs when there is at least one entity matching every query the system uses.
    }
}

特定のコンポーネント/Singleton を必要とする state.RequireForUpdate<T>() と比べると、[RequireMatchingQueriesForUpdate] はより暗黙的だ — System が最終的に持つクエリに基づいてキーが決まる。

6. カスタムグループ

作成する理由：

  • 一貫したゲームプレイフェーズ（例：「アニメーション前にダメージを適用」）。
  • 異なるティックレート（例：30 Hz のカスタム固定ステップ）。
  • 機能トグル — グループ全体を有効/無効にする。
using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial class DamagePipelineGroup : ComponentSystemGroup
{
    // You can override OnUpdate to customise how the group ticks.
}

[UpdateInGroup(typeof(DamagePipelineGroup), OrderFirst = true)]
public partial struct ApplyDamageSystem : ISystem { /* ... */ }

[UpdateInGroup(typeof(DamagePipelineGroup))]
[UpdateAfter(typeof(ApplyDamageSystem))]
public partial struct DeathSystem : ISystem { /* ... */ }

カスタムティックレート

public partial class SlowTickGroup : ComponentSystemGroup
{
    private float _accum;
    protected override void OnUpdate()
    {
        _accum += (float)SystemAPI.Time.DeltaTime;
        if (_accum < 0.1f) return;      // 10 Hz
        _accum = 0f;
        base.OnUpdate();                // tick children
    }
}

7. 固定 vs 可変タイムステップ

決定論が重要な場合（物理演算、ネットワーク予測、決定論的再生）は、System を FixedStepSimulationSystemGroup に配置する：

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial struct PhysicsStepSystem : ISystem { /* ... */ }

注意点：

  • FixedStepSimulationSystemGroup はリアルタイムに追いつくために1フレームに複数回（または0回）実行される可能性がある。
  • SystemAPI.Time.DeltaTime を使用すること — このグループ内では固定ステップの dt だ。
  • 入力を読み取ったりプレゼンテーションを生成するものは SimulationSystemGroup / PresentationSystemGroup に留めること。

8. ランタイムで順序を確認する

Window → Entities → Systems では以下が表示される：

  • すべてのグループとその子、実際の実行順序で。
  • System ごとのタイミング。
  • 有効/無効の状態（ランタイムで切り替える System）。

属性で宣言した順序と実際の実行順序が一致しない場合、ほぼ必ず循環依存または [UpdateInGroup] の指定漏れが原因だ。

9. トラブルシューティング

症状 原因 / 対処法
System が間違ったフレームフェーズで実行される [UpdateInGroup] がないか誤っている。デフォルトは SimulationSystemGroup だ。
[UpdateAfter] が無視される 対象 System が別のグループにある。両方が同じグループを共有しなければならない。
起動時に循環依存エラーが発生する [UpdateBefore] / [UpdateAfter] のチェーンが循環を形成している。Systems ウィンドウがサイクルを出力する。
System が予期せず1フレームに複数回実行される FixedStepSimulationSystemGroup に配置した — そのグループはキャッチアップする。不要なら SimulationSystemGroup に移すこと。
カスタムグループが永遠に実行される OnUpdate オーバーライドで base.OnUpdate() を呼んでいない。
System は実行されるが何もしない RequireForUpdate<T> が満たされていないか、すべてのクエリが空だ。Window → Entities → Query を確認すること。

Chapter 13. JobSystem & Burst

1. 概要

DOTS のパフォーマンスは、Entities より前から存在し今もそれを支える2つのエンジン機能に依存している：

  • Job System — 依存グラフを使ってワーカースレッドに作業をスケジュールする。
  • Burst Compiler — C# ジョブをタイトなネイティブコード（SIMD、GC なし、仮想ディスパッチなし）にコンパイルする。

どちらも Entities 固有の概念ではない。IJobEntity と IJobChunk は IJob + Burst の薄いレイヤーだ。ベースレイヤーを理解することで、上位レベルのパターンが明確に読めるようになる。

2. Job System — コアな型

インターフェース 形式 使いどき
IJob void Execute() 一回限りのシングルスレッド作業。
IJobParallelFor void Execute(int index) 0 から N まで並列ループ。
IJobParallelForBatch void Execute(int start, int count) 並列バッチでループ（インデックスごとよりキャッシュフレンドリー）。
IJobChunk void Execute(ArchetypeChunk chunk, int chunkIndex, bool useEnabledMask, in v128 enabledMask) ECS チャンクを並列でイテレートする。
IJobEntity void Execute([component params]) ECS Entity イテレーション — ソース生成によりチャンクジョブに変換される。

これらはすべて struct であり、すべて Burst をサポートする。

最小の IJobParallelFor サンプル

using Unity.Burst;
using Unity.Collections;
using Unity.Jobs;

[BurstCompile]
struct ScaleJob : IJobParallelFor
{
    public NativeArray<float> Data;
    public float Factor;

    public void Execute(int index) => Data[index] *= Factor;
}

// Scheduling:
var handle = new ScaleJob { Data = arr, Factor = 2f }.Schedule(arr.Length, 64);
handle.Complete();

注記：

  • フィールドはブリッタブルでなければならない。
  • 複数のジョブから同じ NativeArray への書き込みアクセスには依存関係チェーンが必要だ（後述）。

3. スケジューリングと依存関係

JobHandle a = jobA.Schedule();
JobHandle b = jobB.Schedule(a);              // b runs after a
JobHandle c = jobC.Schedule(JobHandle.CombineDependencies(a, b));
c.Complete();

ルール：

  • すべての Schedule() は JobHandle を返す。新しいジョブが共有データを読み書きする場合は前のハンドルを依存関係として渡すこと。
  • メインスレッドが結果を読む前に最終ハンドルで .Complete() を忘れないこと（Safety System が強制する）。
  • ISystem では state.Dependency がシステムレベルのハンドルだ；.Schedule(state.Dependency) を使うか IJobEntity.ScheduleParallel(...) のようなソース生成ヘルパーに依存すると、スケジュールされたジョブが自動的に結合される。

ISystem 内

[BurstCompile]
public void OnUpdate(ref SystemState state)
{
    var job = new MyJob { Dt = SystemAPI.Time.DeltaTime };
    state.Dependency = job.ScheduleParallel(state.Dependency);
}

明示的な依存関係なしの IJobEntity.ScheduleParallel() は state.Dependency と暗黙的に結合される。複数のジョブをチェーンする場合は明示的に記述すること。

4. Burst — 何であり、なぜ必要か

Burst は C# のサブセット（[BurstCompile] でマークされたメソッド、GC アロケーションなし、ブリッタブルなデータ）を LLVM 経由で高度最適化されたネイティブコードにコンパイルする。

Burst がうまくコンパイルするもの

  • [BurstCompile] タグが付いた struct IJob* / ISystem メソッド。
  • Unity.Mathematics 型（float3、quaternion、int4 など）の計算。
  • NativeArray<T>、NativeList<T>、DynamicBuffer<T>、ArchetypeChunk 列配列に対するタイトなループ。

Burst がコンパイルできないもの

  • マネージド参照（class、string、List<T>、アンマネージドでないもの）。
  • 例外（キャッチはできるが、一般的には避けること）。
  • インターフェースを通じた動的ディスパッチ（明示的なジェネリック特殊化は問題なし）。
  • System.IO / Reflection / 数学・プリミティブ以外のほとんどの BCL。

Burst の有効化

using Unity.Burst;

[BurstCompile]
public partial struct MyJob : IJobEntity
{
    // attribute on the struct marks the type for Burst
    // methods need [BurstCompile] too if you want each compiled
    [BurstCompile]
    void Execute(ref Velocity v) { /* ... */ }
}

System の場合：

[BurstCompile]
public partial struct MySystem : ISystem
{
    [BurstCompile] public void OnCreate(ref SystemState state) { }
    [BurstCompile] public void OnUpdate(ref SystemState state) { }
}

型への属性だけでは不十分 — コンパイルしたい各メソッドにも [BurstCompile] が必要だ。

Burst Inspector

Window → Analysis → Burst Inspector はメソッドごとの LLVM IR、アセンブリ、ベクトル使用状況を表示する。「なぜ速くないのか」という疑問があるときに使用する。

5. ネイティブコンテナ

Burst と Safety System はマネージドコレクションの代わりに ネイティブコンテナ を期待する：

マネージド ネイティブ相当品
T[] NativeArray<T>
List<T> NativeList<T>
Dictionary<K,V> NativeHashMap<K,V>、NativeParallelHashMap<K,V>
HashSet<T> NativeHashSet<T>、NativeParallelHashSet<T>
Queue<T> NativeQueue<T>
string FixedString32Bytes / FixedString64Bytes / など

アロケーション：

var arr = new NativeArray<int>(1024, Allocator.TempJob);
// ...
arr.Dispose();

ライフタイムが重要だ：Burst + Safety はコンテナが Dispose されていること、ジョブのライフタイム後にアクセスされていないことを検証する。

6. アロケーター

アロケーター ライフタイム 典型的な用途
Allocator.Temp 現在のフレーム内、メインスレッドのみ OnUpdate 内のスクラッチコンテナ
Allocator.TempJob 1つのジョブをまたぐ、≤4 フレーム ジョブに渡されるコンテナ
Allocator.Persistent 手動の Dispose() が必要 フレームをまたいで存在する System 所有のコンテナ
state.WorldUpdateAllocator System グループ更新の終了時にクリアされる System 内の一時データに推奨

System 内では state.WorldUpdateAllocator（またはその Rewindable アロケーター）を優先すること — 手動の Dispose() 管理を避けられる。

7. Safety System

「Jobs > Leak Detection」と「Jobs > Safety Checks」が ON のエディター環境では、Unity は以下を検証する：

  • 2つのジョブが依存関係エッジなしに同じネイティブコンテナに書き込まない。
  • メインスレッドの読み取りが実行中の書き込みと競合しない。
  • ネイティブコンテナが Dispose() 後またはアロケーターのライフタイム後にアクセスされない。

Safety エラーは「job A and job B both access X」という明確なメッセージで例外として表面化する。開発中に Safety チェックを無効にしないこと — パフォーマンス上のメリットは小さく、デバッグコストは高い。

8. トラブルシューティング

症状 原因 / 対処法
Burst 警告：「method not burst compiled」 メソッドに [BurstCompile] がないか、Burst がサポートされていない型を検出した（Burst Inspector で失敗している行を確認すること）。
InvalidOperationException: … writes to … without declaring dependency 依存関係エッジがない — 前の JobHandle を渡すか、state.Dependency チェーンを使用すること。
NativeArray に ObjectDisposedException ジョブ完了前に Dispose した。arr.Dispose(handle) を使うか先にハンドルを待つこと。
スケジュールされたジョブが完了しない チェーンで .Complete() が欠けているか、循環依存がある。
[BurstCompile] を追加した後にパフォーマンスが低下する メソッドがドメインリロードごとに再コンパイルされている；Burst キャッシュを確認すること。または Burst 非互換の型が混入した — メソッドが暗黙的に IL にフォールバックしている。
NativeHashMap の列挙が例外を投げる 列挙は並行書き込みとの相性が悪い — 代わりに NativeArray<K> のコピーを列挙すること。

Chapter 14. IJobEntity · SystemAPI.Query

1. 概要

Entities 6.5 でのコンポーネントイテレーションの 95% をカバーする2つの API がある：

API 実行場所 使いどき
SystemAPI.Query<...>() メインスレッド、OnUpdate 内 処理が軽い（数百 Entity 程度）場合、またはメインスレッド API が必要な場合。
IJobEntity ワーカースレッド、Schedule / ScheduleParallel 経由 処理が重いまたは並列処理の恩恵を受ける場合。

これらはレガシーの Entities.ForEach（Entities 1.4 で非推奨としてマーク；6.5 では非推奨警告付きでまだコンパイルできる；Unity の変更履歴では将来のメジャーリリースで削除予定と記載）の推奨代替だ。

2. SystemAPI.Query<T>

メインスレッドでマッチする Entity をイテレートする。コンポーネントは RefRO<T> / RefRW<T> でアクセスする。

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRW.Max);
        }
    }
}

ルール：

  • System struct は partial でなければならない。
  • 読み取り専用には RefRO<T>、読み書きには RefRW<T>。RefRO 経由では書き込まないこと。
  • ID が必要な場合はタプルに Entity を追加する：SystemAPI.Query<RefRW<Health>>().WithEntityAccess()。

フィルター

foreach (var health in SystemAPI.Query<RefRW<Health>>()
             .WithAll<Enemy>()
             .WithNone<Dead>()
             .WithAny<Burning, Poisoned>())
{
    // only entities that are Enemy AND not Dead AND (Burning OR Poisoned)
}
  • WithAll<T>() — リストされたコンポーネントをすべて持つこと。
  • WithNone<T>() — それらのいずれも持たないこと。
  • WithAny<T...>() — それらのうち少なくとも1つを持つこと。
  • WithChangeFilter<T>() — 前回実行以降にコンポーネント T が変化したチャンクのみ。
  • WithEntityAccess() — Entity ID も返す。

3. IJobEntity

ソース生成による並列イテレーション。Execute のシグネチャで、アクセスするコンポーネントを宣言する。

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

シグネチャのルール：

  • void Execute(...) — パラメーターはコンポーネントに 1:1 でマップされる。
  • ref Component — 読み書き。in Component — 読み取り専用（書き込み不可）。
  • Entity entity パラメーターは許可される — ソースジェネレーターが埋める。
  • ジョブの struct は partial でなければならない。

スケジュールバリアント

呼び出し スレッディング
.Run() メインスレッド、返す前に完了。デバッグや軽い作業に有用。
.Schedule() 単一ワーカースレッド。
.ScheduleParallel() チャンクをワーカースレッドに分散させる。

3つすべてが ISystem.OnUpdate 内で state.Dependency を自動的に更新する。

ジョブ内フィルター

new HealthRegenJob { DeltaTime = dt }
    .ScheduleParallel(SystemAPI.QueryBuilder()
        .WithAll<Enemy, Regen>()
        .WithNone<Dead>()
        .Build());

またはジョブ struct 自体の [WithAll]/[WithNone]/[WithAny] 属性経由で。

ジョブ内での Entity アクセス

void Execute(Entity entity, ref Health health) { /* ... */ }

チャンクインデックス / [ChunkIndexInQuery]

決定論的なプレイバックのために ParallelWriter ECB と組み合わせる場合に有用だ — チャンクインデックスを sortKey として渡す。15_ParallelWriter · Deterministic Playback.md を参照のこと。

void Execute([ChunkIndexInQuery] int chunkIndex, in Health health, Entity entity)
{
    if (health.Value <= 0)
        ecb.DestroyEntity(chunkIndex, entity);
}

状況 選択
数十〜数百 Entity、軽い計算 SystemAPI.Query
数千以上の Entity、独立した Entity ごとの作業 IJobEntity.ScheduleParallel
チャンクごとの作業（バウンディング、集計、空間分割） IJobChunk — 12_IJobEntity vs IJobChunk.md を参照
ループ内でマネージド API が必要 SystemBase からの SystemAPI.Query
遅延構造変更が必要 どちらの形式でも可、ただし EntityCommandBuffer.ParallelWriter 経由で書き込む — 14_EntityCommandBuffer · Deferred Entity.md を参照

5. 注意点

  • ジョブ内で this やマネージドフィールドをキャプチャしないこと。 ソースジェネレーターが試みた場合にエラーを出す。
  • RefRW<T>.ValueRW は変更追跡を引き起こす。 ValueRW 経由で読み取って書き込まなくても、チャンクがダーティになる。
  • WithChangeFilter<T> はチャンクレベルだ。 チャンク内の任意の Entity が T に書き込めば、チャンク全体が変更済みと見なされる。
  • Aspect は廃止済み（ただし削除されていない）。 IAspect グループ化インターフェースは 1.4 で非推奨になり、6.5 API リファレンスでは引き続き有効な Execute パラメーター種別として記載されている。新しいコードはプレーンなコンポーネントパラメーターを使うこと；既存の Aspect は 6.5 でもコンパイルできる。

6. トラブルシューティング

症状 原因 / 対処法
コンパイルエラー：IJobEntity must be partial ジョブ struct に partial を追加すること。
ScheduleParallel で競合状態エラーが発生する 2つのジョブが順序エッジなしに同じコンポーネントに書き込んでいる。state.Dependency が正しく結合されていることを確認するか、System 間に [UpdateAfter] を追加すること。
ジョブが Burst で実行されない ジョブ struct に [BurstCompile] がないか、Burst 非互換の型（マネージド参照、FixedString なしの string）がキャプチャされている。
SystemAPI.Query が何も返さない フィルターの組み合わせが空だ。Window → Entities → Query でクエリが実際に何にマッチするかを確認すること。
RefRO 経由での書き込み（サイレントバグ） 現在のコンパイラでは refRO.ValueRW = … でエラーになる。古い source-gen バージョンですり抜ける場合は RefRW に切り替えること。
SystemAPI.Query と手動で構築した EntityQuery で Entity 数が一致しない フィルターセットが異なる。SystemAPI.Query はデフォルトで Enableable コンポーネントフィルタリングを含む；EntityQueryBuilder は明示的な設定が必要だ。

Chapter 15. IJobEntity vs IJobChunk

1. 短い答え

  • IJobEntity：各 Entity に同じことをする場合。
  • IJobChunk：単純な Entity ごとのマップではないチャンクごとの処理をする場合。

IJobEntity は IJobChunk の上に生成される — 同じイテレーション機構で動作する。IJobChunk はただチャンク全体をコードに公開するだけだ。

2. IJobEntity — Entity ごとの使いやすさ

[BurstCompile]
public partial struct DamageTickJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Burn burn)
    {
        health.Value -= burn.DamagePerSecond * DeltaTime;
    }
}

強み：

  • 1つのメソッド、1つの Entity ビュー。書きやすく、読みやすい。
  • ソース生成の配管：Entity、[ChunkIndexInQuery]、[EntityIndexInQuery] パラメーターが自動で埋められる。
  • struct の [WithAll] / [WithNone] / [WithAny] 属性、または .Schedule* に渡す QueryBuilder でフィルタリング。

弱み：

  • Execute 内でクロス Entity の可視性がない。追加の状態なしに「このチャンクを合計する」ことは簡単にできない。
  • Execute 内からチャンクレベルのメタデータ（変更バージョン、共有コンポーネント値）にアクセスできない。

3. IJobChunk — 完全なチャンク制御

[BurstCompile]
public partial struct HealthAggregateJob : IJobChunk
{
    [ReadOnly] public ComponentTypeHandle<Health> HealthHandle;
    public NativeArray<float> PerChunkTotal;  // size = chunk count

    public void Execute(in ArchetypeChunk chunk,
                        int unfilteredChunkIndex,
                        bool useEnabledMask,
                        in v128 chunkEnabledMask)
    {
        var healths = chunk.GetNativeArray(ref HealthHandle);
        float total = 0f;
        for (int i = 0; i < chunk.Count; i++)
            total += healths[i].Value;

        PerChunkTotal[unfilteredChunkIndex] = total;
    }
}

強み：

  • チャンク配列への直接アクセス：chunk.GetNativeArray(ref handle)、chunk.GetBufferAccessor(ref bufferHandle)。
  • 条件付き処理のための chunk.Has<T>()、chunk.GetSharedComponent<T>(...)。
  • チャンクごとの有効マスクと変更バージョンを確認できる。
  • リダクション、空間分割、バッチ ECB 記録に最適だ。

コスト：

  • ボイラープレートが多い：System で ComponentTypeHandle<T> を宣言し、毎フレーム handle.Update(ref state) で更新する。
  • 内側の for ループを自分で管理する — 間違える箇所が多い。

System でのハンドル設定

[BurstCompile]
public partial struct HealthAggregateSystem : ISystem
{
    private EntityQuery _query;
    private ComponentTypeHandle<Health> _healthHandle;

    public void OnCreate(ref SystemState state)
    {
        _query        = state.GetEntityQuery(ComponentType.ReadOnly<Health>());
        _healthHandle = state.GetComponentTypeHandle<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthHandle.Update(ref state);

        var totals = CollectionHelper.CreateNativeArray<float>(
            _query.CalculateChunkCount(), state.WorldUpdateAllocator);

        state.Dependency = new HealthAggregateJob
        {
            HealthHandle   = _healthHandle,
            PerChunkTotal  = totals
        }.ScheduleParallel(_query, state.Dependency);
    }
}

handle.Update(ref state) はキャッシュされた型ハンドルを毎フレーム更新する — これを忘れると Burst が古いインデックスにヒットする。

4. 有効マスク（useEnabledMask）

IEnableableComponent を含むクエリでは、Execute は useEnabledMask = true と有効ビットの v128 マスクを受け取る。マスクを尊重しないと無効な Entity を処理してしまう。

ヘルパー：

var enumerator = new ChunkEntityEnumerator(useEnabledMask, chunkEnabledMask, chunk.Count);
while (enumerator.NextEntityIndex(out int i))
{
    // i is the index of the next enabled entity in the chunk
}

IJobEntity はマスクを自動的に適用する — チャンクレベルの可視性が必要でない限り IJobEntity を優先するもう一つの理由だ。

5. 判断ガイド

状況 選択
ヘルスのティック、位置の移動、入力の適用 — 純粋な Entity ごとの計算 IJobEntity
チャンク内すべての Entity の値を合計する IJobChunk
Entity ごとではなくチャンクごとに1つの ECB コマンドを記録する IJobChunk
ループ内で共有コンポーネントの値にアクセスする IJobChunk
チャンクの変更バージョンが必要 IJobChunk
各 Entity の DynamicBuffer<T> をイテレートする どちらでも可；IJobEntity の方がきれい
Entity ごとのロジックだが Entity のチャンクインデックスが必要 [ChunkIndexInQuery] int chunkIndex を使った IJobEntity
スケジューリングコストが重要で処理が軽い ベンチマークを取ること — Entity 数よりチャンク数が少ない方が勝る。IJobChunk は数千チャンクで通常勝つ。

6. 相互運用 — 両方を組み合わせる

よくあるパターン：IJobChunk でチャンクごとのデータを収集し、IJobEntity で Entity ごとの書き込みをし、依存関係でチェーンする。

var gather = new HealthAggregateJob { ... }
    .ScheduleParallel(query, state.Dependency);

var act = new HealJob { PerChunkTotal = totals }
    .ScheduleParallel(gather);

state.Dependency = act;

7. トラブルシューティング

症状 原因 / 対処法
IJobChunk.Execute がゴミ値を読む スケジュール前に handle.Update(ref state) を忘れたか、間違ったクエリが使用された。
ComponentTypeHandle<T> が見つからない using Unity.Entities; がないか、ハンドルが間違ったアクセスモードで宣言されている。読み取り専用には GetComponentTypeHandle<T>(true) を使用すること。
有効な Entity がスキップされる マスクが尊重されていない — 示したように ChunkEntityEnumerator を使用すること。IJobEntity はこれを避けられる。
ジョブが2つのチャンクから同じ NativeArray に書き込む チャンクごとのスロットを用意する（HealthAggregateJob のように）か、チャンクインデックスをキーとした NativeParallelHashMap を使用すること。
query 引数なしで IJobChunk をスケジュールする ScheduleParallel(query, ...) はクエリを明示的に必要とする。IJobEntity.ScheduleParallel() は属性/ソース生成から拾い上げる。
v128 型が見つからない using Unity.Burst.Intrinsics; がない。

Chapter 16. 構造変更 & 安全性

1. 構造変更とは何か

構造変更とは、Entity をチャンク間で移動させたりチャンクのインベントリを変更したりする可能性のある操作だ。以下のすべてが構造的だ：

操作 構造的である理由
CreateEntity / DestroyEntity チャンクスロットを確保または解放する。
AddComponent<T> / RemoveComponent<T> Entity の Archetype が変わるため、物理的に別のチャンクに移動する。
SetSharedComponent<T> 共有コンポーネントの値が異なる → 異なるチャンク（値でグループ化される）。
AddChunkComponent<T> / RemoveChunkComponent<T> チャンク自体のメタデータを変更する。
SetName(entity, ...) EntityManager の Entity メタデータに触れる。
MoveEntitiesFrom World 間で Entity をコピーする。
Instantiate(prefab) 新しい Entity を確保する（Archetype を事前に埋める可能性がある）。

SetComponentData / RefRW<T>.ValueRW 経由でコンポーネントの値を書き込むことは構造変更ではない — コンポーネントは既にチャンク内に存在しており、バイトを変更するだけだ。

2. 構造変更が危険な理由

Archetype が変わると、チャンク内の位置ベースの参照を保持しているものはすべて無効になる：

  • 実行中のクエリイテレーター（SystemAPI.Query<...>）が Entity をスキップしたり再訪したりする可能性がある。
  • 影響を受けた Entity の ComponentLookup<T> エントリが無効になる。
  • ジョブ内の ArchetypeChunk ハンドルが古いチャンクを指す。
  • 並行して実行している他のシステムが矛盾した状態を観測する。

Safety System はエディターでこれらのほとんどを InvalidOperationException で捕捉する。ビルドではクラッシュまたは破損した Entity になる。

3. 経験則

同じ System の更新内でクエリをイテレートしながら構造変更を行わないこと。

推奨順に2つの準拠パターン：

  1. EntityCommandBuffer を使用する。 イテレーション中に操作を記録し、後でプレイバックする。14_EntityCommandBuffer · Deferred Entity.md を参照のこと。
  2. NativeList<Entity> にイテレートしてからループ後に変更する。 小さなケースにはシンプル；ECB オーバーヘッドなし。

パターン 2 の例：

var toKill = new NativeList<Entity>(state.WorldUpdateAllocator);

foreach (var (health, e) in SystemAPI
             .Query<RefRO<Health>>()
             .WithEntityAccess())
{
    if (health.ValueRO.Value <= 0f)
        toKill.Add(e);
}

state.EntityManager.DestroyEntity(toKill.AsArray());

4. Enableable コンポーネント — 逃げ道

コンポーネントを毎フレーム追加・削除していることに気づいたら、ほぼ確実に代わりに Enableable コンポーネントにすべきだ — 有効ビットの切り替えは構造変更ではない。06_Enableable Component.md を参照のこと。

5. ジョブと構造変更

構造変更は メインスレッドの操作だ。ジョブ内では実行できない。

ジョブから変更するイディオマティックな方法：

アプローチ 使いどき
ジョブ内で記録する EntityCommandBuffer.ParallelWriter、メインスレッドでプレイバック 並列コードからの Entity ごとの構造変更。
IJob または IJobChunk で記録する EntityCommandBuffer（シングルスレッド）、メインスレッドでプレイバック バッチ / チャンクスコープの変更。
ジョブ内で EnabledRefRW<T> を使った Enableable コンポーネントの切り替え 構造変更なしの Entity ごとの「有効化/無効化」。

ECB は操作をコマンドとしてキャプチャする；プレイバックはメインスレッド上の既知のポイント、通常は EntityCommandBufferSystem のバウンダリで行われる。

6. 構造変更と読み取りの組み合わせ

2つのシステムが連続して実行され、システム A が実行する構造変更にシステム B が依存する場合は、順序付けを明示的にする：

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct SpawnSystem : ISystem { /* uses ECB */ }

ECB System のプレイバックが A と B の間に位置する場合（例：適切な [UpdateAfter] チェーンを持つ EndSimulationECBS）、B が読む前に構造変更が反映される。

7. コストモデル

単一の変更に対する大まかなランキング：

SetComponentData (value only)          ~ 1x
SetComponentEnabled (enableable bit)   ~ 1x (per-chunk bitmask write)
AddComponent (structural)              ~ 10x   (entity moves chunk)
SetSharedComponent (structural)        ~ 10x
DestroyEntity (structural)             ~ 10x
Per-frame add + remove pattern         pathological

桁数的なもので；正確な数値は Archetype サイズ、チャンク占有率、アロケーター負荷による。

8. 構造変更の乱用を診断する

症状：

  • Window → Entities → Systems でロジックに対して不釣り合いなタイミングを示す System がある。
  • Profiler マーカー EntityManager.SetArchetype または Move Entity がフレームを支配している。
  • Archetype の数が時間とともに増加する。

対策：

  • AddComponent / RemoveComponent の呼び出しを確認する — 毎フレーム実行されているものはないか？ Enableable コンポーネントに置き換えること。
  • 共有コンポーネントの値セットを確認する — 誤って Entity ごとのキーになっているものはあるか？
  • Archetypes ウィンドウで1体だけの Entity しかないチャンクを確認する；それは過細粒度なコンポーネントセットによる断片化だ。

9. トラブルシューティング

症状 原因 / 対処法
InvalidOperationException: EntityCommandBuffer has already been played back プレイバック後に ECB に記録している。各 ECB は単一利用だ — 毎フレーム ECBS から新しいものを取得すること。
foreach 中に InvalidOperationException: … invalidated … ループ内で構造変更が発生している。ECB に移すかバッファ→適用パターンを使うこと。
Entity がサイレントに消える DestroyEntity が発生したが他のシステムが古い Entity ハンドルを保持している。EntityManager.Exists でハンドルを確認すること。
追加/削除パターンがフレーム時間を圧迫する Enableable コンポーネントに置き換えること。
RemoveComponent で「cleanup component present」エラーが発生する Entity にクリーンアップコンポーネントがある。先にクリーンアップを処理すること（05_Component Types.md を参照）。
共有コンポーネントの値セットが爆発する（数百の異なる値） 値が細かすぎる。バケット化するか（例：最も近い N に丸める）、可変部分を通常のコンポーネントとして保存すること。

Chapter 17. EntityCommandBuffer · 遅延 Entity

1. ECB が存在する理由

EntityCommandBuffer（ECB）は構造変更 — 作成、破棄、追加、削除、設定 — を記録し、後でメインスレッド上の安全なポイントで再生する。以下の場所から World を変更できるようにする：

  • クエリの foreach 内（直接の EntityManager 呼び出しがイテレーターを無効化してしまう場所）。
  • ジョブ内（メインスレッドの構造変更が許可されていない場所）。
  • 多くのチャンクにまたがる並列ジョブ内（ParallelWriter 経由）。

「なぜこれが危険なのか」の背景については 13_Structural Change & Safety.md を参照のこと。

2. ECB のライフサイクル

Figure: EntityCommandBuffer のライフサイクル。

ECB は 単一利用 だ：プレイバック後は Dispose される。毎フレーム適切な EntityCommandBufferSystem に新しいものを要求する。

3. 組み込み ECB システム

システム プレイバックタイミング 典型的な用途
BeginInitializationEntityCommandBufferSystem Initialization グループの開始 前フレームからの早期 World セットアップ
EndInitializationEntityCommandBufferSystem Initialization グループの終了 —
BeginSimulationEntityCommandBufferSystem Simulation の開始 ゲームプレイが実行される前に保留中の変更を適用する
EndSimulationEntityCommandBufferSystem Simulation の終了 ゲームプレイが引き起こすスポーン/破棄のデフォルト選択
BeginPresentationEntityCommandBufferSystem Presentation の開始 稀；ビジュアルのみのセットアップ

適切なものを選ぶことで他のシステムが変更をいつ観測するかが決まる。SimulationSystemGroup でバレットをスポーンして同フレームに移動させたい場合は、次フレームの BeginSimulation に記録する — 次フレームで構わなければ EndSimulation に記録する。

4. ECB の取得

任意の ISystem.OnUpdate 内で：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

.Singleton ネスト型は、各組み込み ECB システム用にソースジェネレーターが公開する System ごとのトークンだ。CreateCommandBuffer はそのシステムがティックしたときにプレイバックされる新しい ECB を返す。

並列ジョブの場合：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

並列プレイバックの契約については 15_ParallelWriter · Deterministic Playback.md を参照のこと。

5. 記録 — Entity の作成

Entity e = ecb.CreateEntity();
ecb.AddComponent(e, new Health { Value = 100 });
ecb.AddComponent(e, LocalTransform.FromPosition(0, 0, 0));

または Prefab Entity からインスタンス化する：

Entity inst = ecb.Instantiate(prefab);
ecb.SetComponent(inst, LocalTransform.FromPosition(spawnPos));

CreateEntity と Instantiate の両方が 遅延 Entity を返す（§6 を参照）。

6. 遅延 Entity

ecb.CreateEntity() または ecb.Instantiate() が返す Entity はまだ存在しない。負のインデックスを持つプレースホルダーだ。できること：

  • 同じ ECB 上の後続の ECB 呼び出しに渡す — AddComponent、SetComponent、AppendToBuffer。
  • ecb.SetComponent(other, new ReferenceTo { Entity = inst }) で別のコンポーネントに保存する。

できないこと：

  • ECB がプレイバックするまで EntityManager 経由で読み書きする。
  • 異なる ECB 間で渡す（プレースホルダーは所有する ECB 内でのみ解決される）。

プレイバック時に ECB システムはプレースホルダーが使用されたすべての箇所を実際の Entity ハンドルに置換する。

7. 記録 — 構造変更

ecb.AddComponent(entity, new Stunned { Duration = 2f });
ecb.RemoveComponent<Stunned>(entity);
ecb.SetComponent(entity, new Health { Value = 50 });

ecb.SetComponentEnabled<Stunned>(entity, false);

var buf = ecb.AddBuffer<Waypoint>(entity);
buf.Add(new Waypoint { Position = new float3(1, 0, 0) });
ecb.AppendToBuffer(entity, new Waypoint { Position = new float3(2, 0, 0) });

ecb.DestroyEntity(entity);

これらはすべてキューに入れられたコマンドだ；プレイバックまで何も反映されない。

8. IJobEntity からの ECB

[BurstCompile]
public partial struct KillOnHealthZeroJob : IJobEntity
{
    public EntityCommandBuffer ECB;

    void Execute(Entity entity, in Health health)
    {
        if (health.Value <= 0)
            ECB.DestroyEntity(entity);
    }
}

// In OnUpdate:
var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

new KillOnHealthZeroJob { ECB = ecb }.Schedule();

.ScheduleParallel() では .AsParallelWriter() に切り替え、sortKey を渡すこと — 次のページで解説する。

9. スタンドアロン（手動）ECB

自分でプレイバックを制御したい場合がある — 例：単一の OnUpdate 内：

var ecb = new EntityCommandBuffer(Allocator.Temp);
// ... record ...
ecb.Playback(state.EntityManager);
ecb.Dispose();

自己完結した一回限りの変更パターンに使用する。システム境界をまたぐものやジョブで実行されるものには上記の組み込み ECBS を使用すること。

10. 決定論性

ECB コマンドはシングルライター ECB では記録順でプレイバックされる。ParallelWriter ではプレイバック順序は提供する sortKey に依存する — 15_ParallelWriter · Deterministic Playback.md を参照のこと。

Netcode / リプレイシステムでは決定論的なプレイバックは必須だ。常に安定したソートキー（チャンクインデックス、クエリ内 Entity インデックス）を渡し、非決定論的な順序で実行されるメインスレッドのコードパスに記録することを避けること。

11. トラブルシューティング

症状 原因 / 対処法
InvalidOperationException: ECB has already been played back フレームをまたいで再利用している。CreateCommandBuffer 経由で OnUpdate ごとに新しい ECB を取得すること。
プレイバック後に遅延 Entity ハンドルが Entity.Null になる 2つの異なる ECB をまたいで参照した。作成と後続の設定を同じ ECB に置くこと。
スポーンされた Entity が今フレームに表示されない EndSimulation ECBS に記録した — プレイバックはフレーム終了時だ。ほとんどのシステムより前に観測するには次フレームの BeginSimulation に切り替えること。
ecb.SetComponent で「component does not exist」が発生する 対象 Entity がまだコンポーネントを持っていない（先に AddComponent する）か、プレイバックが行われていない。
ECBS が管理する ECB に Dispose() を呼んだ CreateCommandBuffer からの ECB に Dispose() を呼ばないこと — ECBS がライフタイムを管理する。スタンドアロン（new EntityCommandBuffer(Allocator.Temp)）のみ Dispose すること。
並列 ECB のプレイバックがコマンドを並べ替える sortKey が決定論的でない。[ChunkIndexInQuery] または安定したクエリ提供インデックスを使用すること。

Chapter 18. ParallelWriter · 決定論的再生

1. なぜ別の ParallelWriter があるのか

EntityCommandBuffer はシングルライターだ。2つのスレッドが同じ ECB に記録すると内部チェーンで競合が発生する。EntityCommandBuffer.ParallelWriter は各呼び出し元に スレッドローカルチェーン を与え、コマンドごとに ソートキー を要求することでこれを解決する。

プレイバック時に ECB システムは：

  1. すべてのスレッドローカルチェーンをフラット化する。
  2. コマンドを sortKey でソートする（安定ソート）。
  3. メインスレッドでそれらを実行する。

ソートキーが決定論的である限り、結果は決定論的だ。

2. ParallelWriter の取得

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

AsParallelWriter() は struct EntityCommandBuffer.ParallelWriter を返す。IJobEntity、IJobChunk、または IJobParallelFor にコピーして渡せる。

3. sortKey の契約

すべての並列 ECB 呼び出しは最初の引数として sortKey を取る：

ecb.Instantiate(sortKey, prefab);
ecb.SetComponent(sortKey, entity, value);
ecb.DestroyEntity(sortKey, entity);
ecb.AddComponent(sortKey, entity, component);

良いソートキーのルール：

  • 決定論的。 同じ入力 → 同じキー、スケジューラーが作業をどのように分配しても。
  • 十分にユニーク。 同じ Entity に対する2つのコマンドは、プレイバック順序を曖昧にしたい場合を除いて衝突すべきでない。
  • 安価。 既に持っている int であること、ハッシュではない。

実際のところ、最もよく使われる選択は：

ソース 取得方法 使いどき
IJobEntity の [ChunkIndexInQuery] ソースジェネレーターが埋める 99% のケース
IJobChunk.Execute の unfilteredChunkIndex API が提供する IJobChunk ジョブ
IJobEntity の [EntityIndexInQuery] ソースジェネレーター Entity ごとのコマンドを個別に順序付ける必要がある場合
手動で計算した決定論的インデックス 自分で計算する 稀；カスタムジョブの形状

4. 完全なサンプル — 並列スポーン

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnParallelJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity spawnerEntity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, spawnerEntity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

[BurstCompile]
public partial struct SpawnerParallelSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged)
            .AsParallelWriter();

        new SpawnParallelJob
        {
            DeltaTime = SystemAPI.Time.DeltaTime,
            ECB       = ecb
        }.ScheduleParallel();
    }
}

chunkIndex を Instantiate と後続の SetComponent の両方に渡すことで、2つのコマンドが同じ位置にソートされ、Instantiate からの遅延 Entity 参照が正しいターゲットに解決される。

5. 決定論性 — 保証されること、されないこと

保証される： 同じ ECB の内容であれば、どのワーカースレッドがどのコマンドを記録したかに関わらず、プレイバックは同一の結果状態を生成する。

自動的には保証されないこと：

  • Entity ID。 前のコマンドが異なった場合、2回の実行で異なる Index 値を持つ Entity が生成される可能性がある。
  • システム間のチャンク順序。 システム A が並列で Entity を生成し、システム B が ChunkIndexInQuery の順序でそれらを読む場合、イテレーション順序は依然として Archetype の作成履歴に依存する可能性がある。

実行をまたぐビット完全な決定論性が必要な場合（リプレイ、Netcode ロールバック）は以下を組み合わせること：

  1. 決定論的なソートキーを使った並列 ECB。
  2. FixedStepSimulationSystemGroup 内の固定ステップシステム。
  3. Entity.Index への依存なし；自分で管理するゲームプレイ ID でキーを付ける。

6. ソートキーのパターン

6.1 チャンクごと（デフォルト）

[ChunkIndexInQuery] int chunkIndex を使用する。チャンク内のすべての Entity が同一のコマンドを生成する場合（例：チャンク全体がスタン状態になるなど）、衝突は問題ない。

6.2 Entity ごと

厳密な Entity ごとの順序付けが必要な場合は [EntityIndexInQuery] int entityIndex を使用する（例：Entity 順でプレイバックしなければならない Entity ごとのダメージイベントを記録する場合）。

6.3 カスタムの決定論的インデックス

ごく稀に、入力から自分でフラットインデックスを構築する — 例：int sortKey = baseOffset + localIndex;。一般的な2つが合わない場合のみ行うこと；カスタムインデックスは決定論性バグの頻繁な原因だ。

7. 避けるべきパターン

  • sortKey として JobsUtility.ThreadIndex を使用する。 決定論的でない。
  • キーの構築に Random を使用する。 決定論的でない。
  • ECB を混在させる。 2つの異なる ECB に記録すると全体の順序が失われる。
  • 複数のシステムから System ごとの ECB に記録する。 1つの ECB、1人のレコーダー（または1つの ECB をラップする1つの ParallelWriter）。

8. トラブルシューティング

症状 原因 / 対処法
遅延 Entity が間違ったターゲットに解決される Instantiate と後続の SetComponent で異なるソートキーを使用した。同じにすること。
コマンドが実行ごとに異なる順序で発火する ソートキーが決定論的でない。[ChunkIndexInQuery] に切り替えること。
「JobEntity must have partial」コンパイルエラー ジョブ struct に partial を追加すること。
並列ジョブ内の ECB.SetComponent で「wrong thread」が発生する 並列ジョブ内で非並列 EntityCommandBuffer を使用している。先に .AsParallelWriter() を呼んで並列 API を使用すること。
Netcode のリプレイがデシンクする どこかの並列 ECB が決定論的なソートキーを使用していないか、IJobEntity が非決定論的な値をキャプチャしている。チェーンを監査すること。
Dispose 済みの ECB に AsParallelWriter() を呼ぶ ECB は既にプレイバック済みだ — このフレームで新しいものを取得すること。

Chapter 19. Netcode クライアント・サーバー World & Bootstrap

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概要

Netcode for Entities は クライアント・サーバーモデル を使用し、クライアントとサーバーのロジックを異なる ECS World に分離する。

World 役割
Server World 権威あるシミュレーション。
Client World ローカルプレイヤーのシミュレーション、予測、補間、プレゼンテーション。
Thin Client World 開発と負荷テストに使用されるダミークライアント。

クライアントデバイスがサーバーもホストする場合、プロジェクトは クライアントホスト型サーバー 構成で動作している。

2. World の種類とシステムフィルタリング

2.1 WorldSystemFilter

3. Bootstrap プロセス

ClientServerBootstrap は実行時に Server World と Client World を作成する。

3.1 デフォルト動作

デフォルトの Bootstrap は起動時に Server World と Client World の両方を作成する。自動的には接続しないため、リッスンと接続のタイミングはゲームが制御する。

3.2 カスタム Bootstrap

using Unity.NetCode;

    {
        // Create only the default local world.
        CreateLocalWorld(defaultWorldName);

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

public class TickRateAuthoring : MonoBehaviour
{
    public int SimulationTickRate = 60;
    public int NetworkTickRate = 30;

    class Baker : Baker<TickRateAuthoring>
    {
        public override void Bake(TickRateAuthoring authoring)
        {
            var entity = GetEntity(TransformUsageFlags.None);

                SimulationTickRate = authoring.SimulationTickRate,

NetworkTickRate を SimulationTickRate より低く設定すると帯域幅を節約できる。その設定では、GhostSendSystem の負荷がラウンドロビンスケジューリングで複数のシミュレーションティックに分散される。

5. サーバーとクライアントの更新フロー

Server World
└─ Fixed timestep, 60 Hz default
   └─ SimulationSystemGroup
      ├─ GhostSendSystem
      └─ CommandReceiveSystem

Client World
├─ Dynamic timestep
│  └─ SimulationSystemGroup
│     ├─ GhostReceiveSystem
│     └─ CommandSendSystem
├─ Fixed timestep, same as the server
│  └─ PredictedSimulationSystemGroup
│     └─ Prediction code and rollback loop
└─ PresentationSystemGroup
   └─ Rendering and interpolation

サーバーは決定論的な権威あるシミュレーションのために固定タイムステップを使用する。クライアントは通常のシミュレーションとプレゼンテーションに動的タイムステップを使用し、予測コードは固定予測ループで実行される。

6. World マイグレーション

World の切り替え時に接続状態を維持する必要がある場合は DriverMigrationSystem を使用する。

using Unity.Entities;
using Unity.NetCode;

public World MigrateServerWorld(World sourceWorld)
{
    DriverMigrationSystem migrationSystem = null;
    foreach (var world in World.All)
    {
        if ((migrationSystem = world.GetExistingSystemManaged<DriverMigrationSystem>()) != null)
            break;
    }

    var ticket = migrationSystem.StoreWorld(sourceWorld);
    sourceWorld.Dispose();

    var newWorld = ClientServerBootstrap.CreateServerWorld("NewServerWorld");
    var newMigrationSystem = newWorld.GetExistingSystemManaged<DriverMigrationSystem>();
    newMigrationSystem.LoadWorld(ticket);
    return newWorld;
}

Chapter 20. Netcode ネットワーク接続 & 承認

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概要

Netcode for Entities はすべてのネットワーク接続を Entity として表現する。接続 Entity は NetworkStreamConnection を持ち、接続が終了すると Entity が破棄される。

両サイドが NetworkStreamInGame を追加するまで、接続は「ゲーム内」とは見なされない。このコンポーネントがなければ、サーバーはスナップショットを送信せず、クライアントはコマンドを送信しない。

2. コア接続コンポーネント

コンポーネント 意味
NetworkStreamConnection トランスポートハンドルを保存する。
NetworkId 承認後にサーバーが割り当てるユニーク ID。
NetworkStreamInGame スナップショットとコマンドデータの交換に必要。
CommandTarget プレイヤーコマンドを受け取る Entity；AutoCommandTarget を使用しない限りゲームコードが管理する。

5.3 Approval RPC

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;

public struct ApprovalRpc : IApprovalRpcCommand
{
    public FixedString512Bytes Token;
}

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation |
    WorldSystemFilterFlags.ThinClientSimulation)]
public partial struct ClientApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (connection, entity) in
            SystemAPI.Query<RefRW<NetworkStreamConnection>>()
                .WithNone<NetworkId>()
                .WithNone<ApprovalSent>()
                .WithEntityAccess())
        {
            var rpcEntity = ecb.CreateEntity();
            ecb.AddComponent(rpcEntity, new ApprovalRpc { Token = "MY_TOKEN" });
            ecb.AddComponent<SendRpcCommandRequest>(rpcEntity);
            ecb.AddComponent<ApprovalSent>(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ServerApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, rpc, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ApprovalRpc>>()
                .WithEntityAccess())
        {
            var connEntity = req.ValueRO.SourceConnection;

            if (rpc.ValueRO.Token.Equals("MY_TOKEN"))
                ecb.AddComponent<ConnectionApproved>(connEntity);
            else
                ecb.AddComponent<NetworkStreamRequestDisconnect>(connEntity);

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

Approval のタイムアウトは ClientServerTickRate.HandshakeApprovalTimeoutMS で制御する。デフォルトは 5000 ms だ。

Chapter 21. Netcode Ghost Snapshot & Synchronization

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Overview

Ghost はサーバーでシミュレートされ、クライアントへレプリケートされるネットワーク Entity だ。サーバーは定期的に Ghost の状態を Snapshot として送信し、各クライアントは補間または予測によってその Snapshot を適用する。

メカニズム 用途
Ghost 信頼性のないパケット上で最終的な一貫性を実現し、組み込みのレプリケーション最適化を提供する。
RPC ワンショットの信頼性のあるイベント。

2. Ghost vs RPC

Ghost を使う場合 RPC を使う場合
クライアント近傍の空間 Entity の状態。 ハイレベルなゲームフローイベント。
クライアント予測が必要なデータ。 チャットや設定変更などのワンショットコマンド。
切断/再接続後も存在し続けるべき状態。 永続的なレプリケート状態を持たないイベント。

3. Ghost authoring

レプリケートするべき Prefab に GhostAuthoringComponent を追加する。

プロパティ 意味
Name Ghost の識別子。
Importance 帯域幅が制限されている場合の Snapshot 優先度。
Supported Ghost Mode All、Interpolated、または Predicted。
Default Ghost Mode Interpolated、Predicted、または OwnerPredicted。
Optimization Mode デフォルトは Dynamic。ほとんど変化しないオブジェクトには Static。
MaxSendRate Hz 単位の最大レプリケーション レート。NetworkTickRate が上限となる。
UseSingleBaseline 1.5.0 系で追加されたシングルベースライン差分圧縮オプション。

3.1 Ghost modes

モード 意味
Interpolated クライアントがサーバー Snapshot 間を補間する。ビジュアルは滑らかだが遅延する。
Predicted クライアントが先行してシミュレートし、入力遅延を低減する。CPU コストが高い。
OwnerPredicted 所有クライアントは予測し、他のクライアントは補間する。

4. Serialization attributes

4.1 GhostField

シリアライズするべき Component フィールドに GhostField を使用する。

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

public struct MyComponent : IComponentData
{
    [GhostField(Quantization = 1000, Smoothing = SmoothingAction.Interpolate)]
    public float3 Position;

    [GhostField(Quantization = 100)]
    public float Health;

    [GhostField]
    public int Score;

    [GhostField(SendData = false)]
    public float LocalOnly;
}
GhostField プロパティ 意味
Quantization float を整数に変換する乗数。1000 で小数点以下3桁を保持する。
Smoothing Clamp、Interpolate、または InterpolateAndExtrapolate。
MaxSmoothingDistance テレポート時に補間を無効にする距離しきい値。
Composite 構造体を1つの変更ビットとして扱う。
SendData false にするとフィールドをシリアライズから除外する。
SubType 特殊なシリアライズルールを選択する。

7. Snapshot system

7.1 Partial snapshots

Snapshot が MTU を超える場合、Netcode は重要度の高い Ghost を優先して送信する。残りの Ghost は後のティックで送信されるため、ワールド状態は段階的にストリーミングされる。

7.2 NetworkTickRate

NetworkTickRate が SimulationTickRate より低い場合、GhostSendSystem の処理は複数のシミュレーションティックに分散される。

SimulationTickRate = 60, NetworkTickRate = 30
Tick 1: GhostSendSystem sends a snapshot
Tick 2: GhostSendSystem skips
Tick 3: GhostSendSystem sends a snapshot
...

8. Ghost prefab registration workflow

1. Create a Ghost prefab
   ├─ Add GhostAuthoringComponent
   └─ Add the required authoring components

2. Place or reference it from a SubScene
   ├─ Put the prefab inside the SubScene, or
   └─ Let GhostCollection register the prefab

3. Bake
   ├─ Baker analyzes GhostField attributes
   └─ Source Generator emits serialization code

4. Runtime
   ├─ Server instantiates the Ghost entity
   ├─ GhostSendSystem sends snapshots
   └─ Client automatically spawns the Ghost

8.1 Prespawned Ghosts

SubScene 内に配置された Ghost は Prespawned Ghosts になる。

  • SubScene のロード時にサーバーとクライアントが自動的に同じ Entity を対応付ける。
  • 個別のスポーンメッセージは不要で、Snapshot 同期で十分だ。
  • 1.6.1+ 系では Prespawn Ghost の ID が Host Migration をまたいで保持される。

建物や地形などの静的なワールドオブジェクトは、通常 Static Optimization Mode の Prespawned Ghosts として設定するのが最善だ。

Chapter 22. Netcode Prediction & Rollback

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Overview

クライアント予測は、オーソリティサーバーの結果が届く前に、クライアントで同じシミュレーションコードを実行する。これにより入力遅延が低減され、プレイヤー所有の移動が即座に反応するように感じられる。

予測には実際の CPU コストがかかる。権威ある Snapshot が届くと、クライアントは古いティックにロールバックし、現在の予測ティックまで再シミュレーションを行う。

2. Core components and systems

Component / system 意味
PredictedGhost クライアントの予測 Ghost と、サーバーのすべての Ghost に追加される。
Simulate 現在の予測ティックでシミュレートすべき Entity に設定される Enableable タグ。
PredictedSimulationSystemGroup 予測ループを実行する固定タイムステップグループ。
NetworkTime サーバーティック・予測ティック・関連するタイミングデータを保持する。

3. Client prediction flow

1. Receive a server snapshot
   └─ Apply the latest authoritative state to predicted entities

2. Find the oldest applied tick
   └─ Use the oldest tick across affected predicted entities

3. Roll back and re-simulate
   └─ PredictedSimulationSystemGroup repeats from oldest tick to target tick
   └─ TimeData and NetworkTime update for each tick

4. Render
   └─ Present the predicted result

レイテンシが高いほどコストが増大する。300 ms の接続では、ティックレートと予測設定によっておよそ 20 フレーム以上の再シミュレーションが必要になる場合がある。

4. Writing prediction systems

4.1 Use the Simulate tag

予測ループの開始時に、Netcode は予測 Ghost の Simulate を無効化し、現在のティックで実行すべき Entity にのみ有効にする。

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct MovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 5f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<Simulate>())
        {
            var move = new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical);
            transform.ValueRW.Position += move * speed * dt;
        }
    }
}

予測クエリには常に .WithAll<Simulate>() を含める。これがないと、すでに完了したティックの Entity を再シミュレートしてしまう。

4.2 Server-side behavior

サーバーでは予測ループがサーバーティックごとに1回だけ実行される。

  • TimeData には通常のサーバーティックデータが含まれる。
  • Simulate は常に有効だ。
  • 同じシステムコードがクライアントとサーバーの両方で動作する。

5. Remote-player prediction

別のプレイヤーのコマンドデータがシリアライズされていれば、そのリモートプレイヤーも予測できる。

6. Prediction smoothing

Prediction smoothing は、サーバーの結果が予測結果と異なる場合に、目に見える補正を低減する。

using Unity.NetCode;
using Unity.Transforms;

GhostPredictionSmoothing.RegisterSmoothingAction<LocalTransform>(
    state.EntityManager,
    CustomSmoothing.Action);

using Unity.Burst;
using Unity.Collections.LowLevel.Unsafe;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[BurstCompile]
public static unsafe class CustomSmoothing
{
    public static readonly PortableFunctionPointer<
        GhostPredictionSmoothing.SmoothingActionDelegate> Action =
        new(SmoothingAction);

    [BurstCompile(DisableDirectCall = true)]
    private static void SmoothingAction(
        void* currentData, void* previousData, void* userData)
    {
        ref var current = ref UnsafeUtility.AsRef<LocalTransform>(currentData);
        ref var backup = ref UnsafeUtility.AsRef<LocalTransform>(previousData);

        var dist = math.distance(current.Position, backup.Position);
        if (dist > 0)
        {
            current.Position = math.lerp(backup.Position, current.Position, 0.5f);
        }
    }
}

7. Prediction switching

予測はコストが高いため、Ghost は Predicted と Interpolated を動的に切り替えられる。

7.1 Switch prerequisites

  • Supported Ghost Modes が All でなければならない。
  • 現在のモードが OwnerPredicted であってはならない。
  • Ghost がすでに切り替え中であってはならない。

7.2 Queueing switches

using Unity.NetCode;

ref var switchQueues = ref SystemAPI
    .GetSingletonRW<GhostPredictionSwitchingQueues>()
    .ValueRW;

switchQueues.ConvertToPredictedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 1f,
});

switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 0f,
});

Interpolated と Predicted の Ghost は異なるタイムライン上で動作し、通常はおよそ 2 ping 間隔だけ離れている。SwitchPredictionSmoothing は設定された時間にわたって位置と回転を補間し、目に見えるテレポートを回避する。

Chapter 24. Netcode RPC

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Overview

RPC はクライアントとサーバー間の 信頼性のあるワンショットメッセージだ。RPC のシリアライズ・デシリアライズコードは自動生成され、RPC の処理は ECS / Job System フレーム内に収まる。

RPC は Ghost レプリケーションやコマンドストリームの代替ではない。高頻度な状態ではなく、離散的なゲームフローイベントに使用する。

2. RPC vs Ghost vs Command

RPC Ghost Command
トランスポート挙動 信頼性あり、順序保証あり。 レプリケーション最適化付きの非信頼性。 冗長送信付きの非信頼性。
方向 双方向。 サーバー → クライアント。 クライアント → サーバー。
頻度 ワンショット。 必要に応じてネットワークティックごと。 シミュレーションティックごと。
用途 ゲームフロー・ロビー・チャット。 レプリケートされた Entity の状態。 プレイヤー入力。
飛行中の制限 あり。信頼性パイプラインの制限。 なし。 なし。

RPC は順序保証かつ信頼性があるため、頻繁な RPC トラフィックは飛行中のパケットの後ろで停滞することがある。

Chapter 25. Netcode Ghost Optimization · Importance · Relevancy

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Overview

大規模なマルチプレイヤーワールドでは、すべての Ghost をすべてのクライアントに毎ティック送信することはできない。Netcode for Entities は Optimization Mode・Importance Scaling・Relevancy・Ghost ごとの送信レート制御を使用して、CPU コストと帯域幅を削減する。

基本ルールはシンプルだ。サーバーは Snapshot バジェットを、特定の接続に今まさに重要な Ghost に費やすべきである。

2. Optimization Mode

Optimization Mode は GhostAuthoringComponent で設定する。

モード 意味 適した用途
Dynamic デフォルト。データが変化する場合もしない場合も Snapshot サイズを最適化する。 通常の移動する Ghost。
Static Ghost が変化していない場合はデータの再送を回避する。 建物・地形・静的なプロップ。

1.11.0 系で Static Ghost の最適化が追加され、変化のない Static Ghost に対する GhostUpdateSystem の処理時間を大幅に削減できる。

3. Snapshot-size limits

Snapshot バジェットはいくつかのレベルで制限できる。

設定 意味
NetworkStreamSnapshotTargetSize 接続ごとのソフトバイトターゲット。
GhostSendSystemData.MaxSendEntities Snapshot ごとの最大 Entity 数。
GhostSendSystemData.MaxSendChunks Snapshot ごとの最大チャンク数。

バジェットが制限されている場合、重要度の高い Ghost が優先して送信される。

4. Importance Scaling

Importance Scaling はサーバーで Ghost ごとの送信優先度を計算する。

4.1 Inputs

GhostImportance は3種類のデータを組み合わせる。

データ 意味
接続ごとのデータ プレイヤー位置などの接続固有のデータ。
Singleton データ タイルサイズなどのグローバル設定。
チャンクごとの共有データ チャンクレベルのスケーリングパラメータ。

4.2 Distance-based importance

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

var gridSingleton = state.EntityManager.CreateSingleton(
    new GhostDistanceData
    {
        TileSize = new int3(50, 50, 256),

5. Ghost Relevancy

Ghost Relevancy により、サーバーは接続ごとに特定の Ghost を非表示または表示できる。

モード 意味
Disabled デフォルト。フィルタリングなし。すべての Ghost が候補になる。
SetIsRelevant リストされた Ghost のみがその接続にレプリケートされる。
SetIsIrrelevant リストされた Ghost はその接続から除外される。

5.1 Use cases

シナリオ 実装のアイデア
距離カリング 遠い Ghost を Irrelevant にする。
フォグ オブ ウォー 視野外の Entity を非表示にする。
クライアント固有の Ghost Ghost を1つのクライアントにのみ Relevant としてマークする。
チームフィルタリング 敵のみの内部データを非表示にする。

5.2 Relevancy example

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
[UpdateInGroup(typeof(GhostSimulationSystemGroup))]
[UpdateBefore(typeof(GhostSendSystem))]
public partial struct RelevancySystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var relevancy = SystemAPI.GetSingletonRW<GhostRelevancy>();

        relevancy.ValueRW.GhostRelevancyMode = GhostRelevancyMode.SetIsRelevant;
        relevancy.ValueRW.GhostRelevancySet.Clear();

        foreach (var (ghostInstance, transform) in
            SystemAPI.Query<RefRO<GhostInstance>, RefRO<LocalTransform>>())
        {
            foreach (var (connId, connPos) in
                SystemAPI.Query<RefRO<NetworkId>, RefRO<GhostConnectionPosition>>())
            {
                float dist = math.distance(
                    transform.ValueRO.Position,
                    connPos.ValueRO.Position);

                if (dist < 100f)
                {
                    relevancy.ValueRW.GhostRelevancySet.TryAdd(
                        new RelevantGhostForConnection(
                            connId.ValueRO.Value,
                            ghostInstance.ValueRO.ghostId),
                        1);
                }
            }
        }
    }
}

この簡略化された例は O(Ghost × 接続) だ。大規模なワールドでは、距離ベースの Importance パーティショニングを優先し、フォグ オブ ウォーやチーム可視性などパーティショニングでは表現できないルールにのみ Relevancy を使用する。

5.3 DefaultRelevancyQuery

DefaultRelevancyQuery は、グローバルゲーム状態の Ghost や UI 関連のネットワーク状態など、すべての接続に常に Relevant な Ghost チャンクを定義する。

5.4 Relevancy plus importance

1.6.1+ 系では PrioChunks.isRelevant を通じて Ghost Relevancy と Importance Scaling を組み合わせることができ、より高速な Relevancy パスが使用できる。

20 フレーム以上の予測再シミュレーションが必要な高レイテンシのケースでは次の対処を行う。

  • メインスレッドへの集約の方が低コストな場合は、PhysicsStep Singleton を通じてマルチスレッド物理を無効にする。
  • 繰り返す物理ティックのバッチ処理に MaxPredictionStepBatchSizeRepeatedTick を使用する。

7. MaxIterateChunks

GhostSendSystemData.MaxIterateChunks は1ティックで処理するチャンク数を制限する。

設定 意味
MaxIterateChunks ティックあたりのシリアライズ CPU コストを制限する。
MaxSendChunks Snapshot に含まれるチャンク数を制限して帯域幅を抑える。

これらの設定は異なる問題を解決する。MaxIterateChunks は CPU 処理を分散させ、MaxSendChunks は Snapshot サイズを制約する。

8. MaxSendRate

GhostAuthoringComponent.MaxSendRate は Ghost の送信頻度を Hz 単位で制限する。

NetworkTickRate = 60, MaxSendRate = 10
→ This Ghost can be included once every 6 network ticks.

頻度が低いまたは重要度の低い Ghost にはこの設定を使用する。

9. UseSingleBaseline

GhostAuthoringComponent.UseSingleBaseline = true は差分圧縮に1つのベースラインを使用する。CPU コストを削減できるが、帯域幅が増加する可能性がある。

10. Optimization checklist

  • ほとんど変化しない Ghost には Static Optimization Mode を使用する。
  • NetworkStreamSnapshotTargetSize を設定して接続ごとの帯域幅を制限する。
  • 距離ベースの Importance Scaling を実装する。
  • パーティショニングでは表現できないルールにのみ Relevancy を適用する。
  • 遠距離の Ghost を Interpolated モードに切り替える。
  • 低頻度の Ghost には MaxSendRate を使用する。
  • 変化しない GhostField データへの書き込みアクセスを避ける。
  • Importance Visualizer を使ってランタイムの優先度を確認する。

Chapter 26. Netcode Physics Integration & Lag Compensation

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Overview

Netcode for Entities はネットワーク物理シミュレーションのために Unity Physics と統合されている。物理の挙動は Ghost が Interpolated か Predicted かによって異なる。

予測物理を実行するには、シーンに少なくとも1つの Predicted Ghost が必要だ。

2. Interpolated vs Predicted physics

2.1 Interpolated Ghosts on clients

  • 位置と回転はサーバーの Snapshot から取得される。
  • Simulate が無効化されているため、物理オブジェクトはキネマティックとして振る舞う。
  • PhysicsVelocity は無視される。

2.2 Predicted Ghosts on clients

  • 物理は予測ループ内で実行され、レンダリングフレームごとに複数回実行される場合がある。
  • PhysicsSystemGroup は PredictedFixedStepSimulationSystemGroup 内で実行される。
  • クライアントは予測物理についてサーバーと同じシミュレーション結果を生成することが期待される。
Netcode initialization
├─ PhysicsSystemGroup moves under PredictedFixedStepSimulationSystemGroup
└─ FixedStepSimulationSystemGroup moves under PredictedFixedStepSimulationSystemGroup

3. Predicted physics performance

予測物理はロールバックが物理を繰り返し実行するためコストが高い。

3.1 Server batching

using Unity.NetCode;

ClientServerTickRate tickRate = default;
tickRate.MaxSimulationStepBatchSize = 4;
tickRate.MaxSimulationStepsPerFrame = 4;

3.2 Client batching

using Unity.NetCode;

ClientTickRate clientRate = default;
clientRate.MaxPredictionStepBatchSizeFirstTimeTick = 2;
clientRate.MaxPredictionStepBatchSizeRepeatedTick = 4;

バッチ処理は予測ミスのリスクを高める可能性があるため、デフォルトではなくパフォーマンストレードオフとして使用する。

3.3 Quantization

物理 Ghost のデフォルトの Transform / Velocity Quantization は 1000 だ。

高い Quantization 低い Quantization
より精密なシミュレーション。 低い帯域幅。
高い帯域幅。 より多くの精度損失。

  3. クライアント専用の物理オブジェクトに同じ値で PhysicsWorldIndex を追加する。

        state.World.GetExistingSystemManaged<PredictedPhysicsConfigSystem>()

}

Part III. Optimization and Debugging

Chapter 28. Chunk Layout & TypeManager

1. The 16 KB chunk

すべての Archetype は Entity を 16 KB のチャンクに格納する。チャンクには次のものが含まれる。

  1. 小さなヘッダ（メタデータ — Archetype ID・バージョン番号・Enableable ビットマスク・変更バージョン）。
  2. 連続した SoA 領域: Component タイプごとに1カラム、各カラムは値の配列。
  3. 16 KB 境界へのパディング。

チャンク容量 — 収まる Entity の数 — は次の計算式から導かれる。

capacity ≈ (16 KB − header) / sum(sizeof(component_i))

少数の小さな Component を持つ密なArchetypeは数百の Entity を保持する。多数の大きな Component を持つ太ったArchetypeはわずかしか保持できない。占有率が低いチャンクは、より多くの Entity を保持できたはずのバイトを無駄にする。

2. TypeManager — the component registry

TypeManager はワールド起動時にすべての Component・バッファ・System を登録し、それぞれに安定した 型インデックスを割り当てる。レジストリからランタイムは次の情報を読み取る。

フィールド 用途
TypeIndex クエリとチャンクメタデータで使用されるパック済み識別子。
SizeInChunk チャンク内でのこの Component の Entity あたりのバイト数。
Alignment チャンク内のカラムアライメント。
ElementSize バッファ要素のサイズ（IBufferElementData 用）。
HasBlobReferences / HasEntityReferences シリアライズおよび移動操作中の検証を有効にする。

TypeManager を直接操作することはほとんどない。重要なのは、新しい Component タイプを追加・名前変更・アセンブリ間で移動すると型レジストリの再ビルドがトリガーされることで、これが大規模プロジェクトでの起動時間の主な原因だ。

Entities 1.4 では不要なアセンブリスキャンをスキップすることで大規模プロジェクトでの TypeManager.Initialize が約2倍高速化されたと報告された。6.5 ではその最適化が継承されている。

3. Inspecting chunk layout — the Archetypes window

Window → Entities → Archetypes にはワールド内のすべての Archetype が一覧表示される。各行の情報は次の通りだ。

  • Entities — ライブカウント。
  • Chunks — 使用中のチャンク数。
  • Entities / Chunk — 平均占有率。
  • Capacity — チャンクあたりの理論上の最大値。

注意すべき読み方:

読み方 解釈
1 Entity / チャンクの Archetype が多数ある 過度に細分化された tag による Archetype の急増。統合する。
1つの Archetype に Entity 数が多く高占有率 健全な状態 — 敵・発射物・弾丸など。
Entities / Chunk ≪ Capacity フラグメンテーション — 構造変更チャーンにより Entity が Archetype 間を移動している。
Capacity = 1 Archetype が大きすぎる（Component が大きすぎる）。大きすぎるマネージド / 共有 Component を確認する。

4. Designing archetype-friendly components

影響の大きい順のヒューリスティック:

  1. アンマネージド構造体はアンマネージドのままにする。 単一のマネージド IComponentData は Entity ごとのサイドテーブルストレージを強制し、Burst を無効にする。
  2. 構造体をパディングしない。 float3 を float の後ではなく前に置く — アライメントのホールが SizeInChunk を膨らませる。
  3. コールドなフィールドを分離する。 1つの System しか読まないフィールドはサブコンポーネントに移動する。未使用のカラムも Entity あたりのバイト数を消費する。
  4. フレームごとの add/remove より Enableable Component を優先する。 13_Structural Change & Safety.md を参照 — ビットのトグルは Archetype の再形成より有利だ。
  5. 共有 Component の値セットを制限する。 新しい値ごとに新しいチャンクバケットが生まれる。
  6. タグ Component はゼロバイトだが Archetype を追加する。 すべての Entity をユニークなマーカーでタグ付けしない。

5. Chunk version numbers

各 Component カラムは 変更バージョンを追跡する — これは System が RefRW<T>.ValueRW・SetComponentData・ECB の SetComponent を通じて書き込むたびにインクリメントされる 32 ビットカウンタだ。

System は最後の実行からバージョンが変わっていないチャンクをスキップできる。

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>());
query.SetChangedVersionFilter(typeof(Health));

変更フィルターでクエリをイテレートすると、System が最後に実行されてからバンプされたチャンクのみが訪問される。これが「Health 変更時に UI を更新する」などのパターンを低コストにする方法だ。

注意点:

  • フィルターは Entity レベルではなく チャンクレベルだ。1つの Entity の書き込みがチャンク全体をダーティにする。
  • RefRW<T>.ValueRW による読み取り（実際に変更しない場合でも）はバージョンをバンプする。
  • Enableable Component のトグルもバージョンをバンプする。

6. BlobAsset storage

BlobAssetReference<T> は大きな読み取り専用データ（メッシュ・LOD テーブル・ナビゲーションフィールド）をチャンクの外部に保持する。チャンにはポインタのみが格納される。これにより、同じベイクされたデータを多くの Entity が共有する場合にチャンク容量が膨らむのを防ぐ。

実践的なヒント: あるタイプのすべての Entity で同一の FixedList や DynamicBuffer に支配された Component がある場合、大きなペイロードをリーンな Component から参照される Blob アセットに格納する。

7. Examples

7.1 A fat archetype to avoid

public struct EnemyState : IComponentData
{
    public float3 Position;
    public quaternion Rotation;
    public float Health;
    public float HealthMax;
    public float Stamina;
    public float StaminaMax;
    public int AIState;
    public float3 LastKnownPlayerPos;
    public float TargetDistance;
    public Entity Target;
    public FixedList512Bytes<float3> PathBuffer;   // 512 bytes per entity!
}

PathBuffer フィールドだけで Entity あたり 512 バイトを消費する。16 KB チャンクあたりの容量は低い 2 桁に落ちる。分割する:

public struct EnemyStats : IComponentData
{
    public float Health, HealthMax, Stamina, StaminaMax;
    public int   AIState;
    public Entity Target;
}

public struct EnemyPathPoint : IBufferElementData
{
    public float3 Value;
}

これで DynamicBuffer は独自のストレージに配置され、チャンク密度が回復する。

7.2 Change filter for cheap UI updates

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>(),
                                 ComponentType.ReadWrite<HealthBarLink>());
query.SetChangedVersionFilter(typeof(Health));

state.Dependency = new UpdateHealthBarJob { /* ... */ }
    .ScheduleParallel(query, state.Dependency);

Health が変化したチャンクのみが訪問される。

Chapter 29. Systems · Entity Inspector · Query Window

1. Overview

よく使う3つの Entities ツールウィンドウ:

ウィンドウ 開く方法 答える問いかけ

Systems Window → Entities → Systems 「どの System が、どんな順序で、どのくらいの時間実行されているか？」

4つ目の Archetypes（Window → Entities → Archetypes）は 01_Chunk Layout & TypeManager.md で扱う。

2. Entities Hierarchy

GameObject ではなく Entity に対応したライブの Hierarchy として機能する。

World picker (top of window)

DefaultGameObjectInjectionWorld（シングルプレイヤー）・ClientWorld/ServerWorld（Netcode）・テストフィクスチャの World を切り替える。World を変更すると、そのワールドの Entity に対してツリーが再実行される。

Tree view

  • 各行が Entity。子行は Parent / LinkedEntityGroup でリンクされた Entity だ。
  • 右カラムのカウント: Component 数・子 Entity 数。
  • 行をクリックすると Inspector にその Component が表示される。

Search

  • c:Health — Health Component を持つ Entity にフィルタリングする。
  • n:Enemy — 名前フィルタ。
  • w:Server — World フィルタ。
  • 組み合わせ: c:Health c:Enemy n:boss。

Inspector

Entity の Inspector には次が表示される。

  • Components セクション — 各 IComponentData と現在の値。Play モードでライブ編集が可能だ。
  • Relationships — 他の Entity へのリンク（Parent・Child・LinkedEntityGroup）。
  • Enabled state — IEnableableComponent ごとのトグル。

ライブ編集は ライトスルーだ: フィールドを変更すると、そのチャンクの変更バージョンが即座にバンプされる。

3. Systems

Systems ウィンドウには、スケジューラが実際にティックする順序で、SystemGroup ごとにグループ化された登録済み System がすべてリストされる。

Columns to read first

カラム 意味
Name System の型。
Namespace 存在する場所（多くの場合、パッケージとユーザーコードを識別する）。
Time (ms) OnUpdate で費やされたメインスレッド時間。約毎フレーム更新される。
Queries System がアクセスしたクエリ数。
Entity Count プライマリクエリにマッチした Entity 数。

What to look for

読み方 解釈
System が 0 ms だが緑色 RequireForUpdate<T> または [RequireMatchingQueriesForUpdate] によりスキップされている。正常な状態だ。
System が常に高い ms を示す 実際の処理が行われている。詳細は Profiler で確認する。
System の Entity 数が -- を示す マッチするクエリがない（またはクエリがまだビルドされていない）。
2つの System が予期した順序と異なる [UpdateBefore] / [UpdateAfter] のチェーンが間違っているか、異なるグループにある。

Enable / disable

各 System にはチェックボックスがある — チェックを外すと再コンパイルなしに Play モードでスキップできる。A/Bテストに便利だ。

System detail pane

System を選択するとサイドペインが開き、次の内容が表示される。

  • Queries — 各クエリの Component セット（All/Any/None/Disabled/Present/Absent）。
  • Dependencies — この System が [UpdateBefore]/[UpdateAfter] する System。実際のスケジュールに解決される。
  • Type Info — クラスおよび各メソッドに [BurstCompile] が付いているかどうか。

1.4+ / 6.x では Queries タブに Disabled / Present / Absent / None の Component 状態が表示されるようになり、Enableable クエリが曖昧に見えなくなった。

4. Query window

「X にマッチするのは誰か？」という問いかけのための専用ツールだ。

使い方:

  1. プラスボタンで All Component を追加する。
  2. 必要に応じて Any / None を追加する。
  3. ウィンドウの左パネルに マッチする Entity、右パネルに このクエリを実行する System が一覧表示される。

ユースケース:

  • 「どの System が Health を読んでいるか？」→ All に Health を追加する。右側に表示されるすべての System がそれに触れている。
  • 「この Entity が System のクエリにマッチしないのはなぜか？」→ System のクエリを貼り付け、Hierarchy ウィンドウの Entity と Component の存在を比較する。
  • 「Prefab がこのクエリに混入していないか？」→ Prefab は結果リストでアイコンで区別される（1.4+ で追加）。

Query ウィンドウではクエリを保存してセッション中に再利用することもできる。

5. Journaling — the last-resort diagnostic

Window → Analysis → Entities Journaling（Project Settings → Entities → Journaling で有効化）は、すべての構造変更・Component の書き込み・System の更新をリングバッファに記録する。再生ビュー:

  • Operations — タイムスタンプ・ソース System・影響を受けた Entity を含む、記録された各変更。
  • Entities — 誰がいつ触れたかの Entity ごとのタイムライン。

Journaling にはランタイムコストがある — バグを調査する場合にのみ有効にし、通常の開発では無効にする。

6. Field workflow — debugging a missing update

よくあるバグ:「System が Entity を処理したが Component が変化しなかった。」調査手順:

  1. Entities Hierarchy → Entity を見つけ、Component を持っていることを確認する。
  2. Inspector → Play 前に値を記録する。Play を押して変化を見る。
  3. 値が変化しない場合: Systems ウィンドウ → 関連 System を見つけ、Time (ms) を確認する。0 なら、スキップされている。
  4. 時間 > 0 だが値がまだ変化しない場合: Query ウィンドウ → System のクエリと Entity の Component を貼り付け、Entity がマッチすることを確認する。
  5. まだ解決しない場合: 1〜2 フレームの Journaling を有効にする。その Component に対する他の System からの書き込みがないか確認する。

7. Troubleshooting

症状 原因 / 対処
Play モードで Entities Hierarchy が空 World ピッカーが間違った World に設定されている（例: Client のみ、Server なし）。
ライブフィールドが更新されない ドメインリロード後にキャッシュされた Entity を表示している。再選択する。
Systems ウィンドウが 0 ms と表示されるが確実に実行されている 実際の処理が ScheduleParallel にある場合、Profiler ではチャンク Job のマーカー下に表示されることがある。その場合、メインスレッドの OnUpdate は実際に 0 だ — Profiler で Job のタイミングを確認する。

Entity と GameObject が同期していないように見える SubScene が開いている（白アイコン）— 閉じる（黄アイコン）と Baker が実行され Entity が表示される。

Chapter 30. Profiler · Bottleneck Analysis

1. Overview

Unity Profiler（Window → Analysis → Profiler）は DOTS で「なぜこのフレームが遅いのか」という問いに答えるための主要ツールだ。Entities は十分なマーカーを出力するため、特定の System・Job・チャンクに時間を帰属できる — 見る場所を知っていれば。

このページではプロファイルの読み方・典型的な DOTS のボトルネックパターン・その対処方法を説明する。

2. Before you profile

  1. Deep Profile は控えめに使用する。 特定のマネージドパスを追跡するのに役立つが、タイミングを歪める — パフォーマンス数値には Deep Profile を信頼しない。
  2. 「Record in Editor & Standalone」を有効にすることで Editor フレームだけでなくビルドもキャプチャできる。
  3. リリースパフォーマンスを測定する際は Profiler をプレイヤーにターゲット設定する（Editor ではなく）。Editor のオーバーヘッドは大きい。
  4. 読む前に数秒プレイする。最初の数フレームにはドメインリロードと JIT 処理が含まれる。

3. Reading the CPU Usage module

DOTS フレーム下のトップレベルマーカー:

PlayerLoop
├── Initialization.*
├── Update.ScriptRunBehaviourUpdate              (MonoBehaviour.Update)
├── SimulationSystemGroup                        ← DOTS simulation lives here
│   ├── SystemA.OnUpdate
│   ├── SystemB.OnUpdate
│   └── EndSimulationEntityCommandBufferSystem.Playback
├── FixedStepSimulationSystemGroup
├── PresentationSystemGroup
├── Render
└── WaitForTargetFPS

SystemA.OnUpdate マーカーをドリルダウンするとサブコール — ScheduleParallel・Job 完了待ち・EntityManager の呼び出し — が見える。

4. Where the time actually goes — common shapes

4.1 Main-thread OnUpdate dominant

単一の SystemX.OnUpdate が 10 ms 以上を占める場合、処理がメインスレッドで実行されている。確認ポイント:

  • IJobEntity.ScheduleParallel にすべきタイトループで SystemAPI.Query を使っていないか？
  • マネージドフィールドによってメインスレッドに強制された SystemBase ではないか？
  • EntityManager.AddComponent/RemoveComponent が Entity ごとに呼ばれていないか？

4.2 Job timings

Job は Timeline ビュー（または Hierarchy ビューの「Workers」下）に表示される。確認ポイント:

  • あるジョブレーンが完全に飽和している一方、他が遊んでいる → 処理がチャンク間に分散していない。
  • Job を待つメインスレッド（JobHandle.Complete() マーカー）→ .Complete() を早く呼びすぎている。フレームの後半に移動する。
  • 巨大な Burst 関数名 → 正常。シンボルは完全なジェネリックインスタンス化だ。

4.3 EntityManager hotspots

次の名前のマーカーを確認する:

  • EntityManager.SetArchetype / Move Entity — 構造変更チャーン。13_Structural Change & Safety.md を参照。
  • EntityCommandBufferSystem.OnUpdate (Playback) — 高コストな ECB プレイバック。通常、フレームごとに多くのコマンドが記録されすぎていることを意味する。
  • TypeManager.Initialize — 起動時にのみ表示される。フレームごとに表示される場合、何かが World を再ロードしている。

4.4 GC.Alloc markers

シミュレーション内のどんな GC.Alloc も DOTS におけるリグレッションだ。ソース:

  • OnUpdate 内でアロケーションする SystemBase。
  • System 内でマネージドコレクションを foreach でイテレートする。
  • ボクシング — 例: 構造体での object.ToString()。

Burst System は Profiler のメモリモジュールでゼロアロケーションを示すはずだ。

5. Profiler modules to pin

  • CPU Usage — マーカーのタイムラインと階層。
  • Memory — 割り当てられたヒープ。GC.Used Memory の増加を監視する。
  • Entities（利用可能な場合）— System ごとのタイミングを時系列で表示する DOTS 専用モジュール。

Entities Profiler モジュールは 1.8.0 / 6.x で導入され、タイムライン上で Systems ウィンドウに合わせられる。複数フレームにわたるスパイクが見られる場合、これが最も素早く帰属できる方法だ。

ネットワークプロジェクトでは、Ghost Snapshot・予測ティック・Netcode 固有の帯域幅を確認するために ../DOTS Workflows/24_Netcode Profiler & Debugging.md と組み合わせて使用する。

6. Common bottlenecks and fixes

パターン Profiler での症状 対処
Entity ごとの Entities.ForEach（レガシー） 小さなサンプルが多い、メインスレッドが重い OnUpdate IJobEntity.ScheduleParallel に移行する。
フレームごとの AddComponent/RemoveComponent Move Entity マーカーが支配的 状態を Enableable Component に変換する。
ECB の過負荷 ...ECBS.Playback が ms を占める コマンドをバッチ処理する（Entity ごとではなくチャンクごとに1回）か、バッチサイズを大きくする。
共有 Component の値が爆発する SetSharedComponent が支配的 値をバケット化する（より少ない個別キーに量子化する）。
.Complete() をフレーム中に呼ぶ メインスレッドの JobHandle.Complete 待機が長い 完了をフレームの末尾まで延期するか、メインスレッドがまだ結果を必要としないよう再構成する。
Burst フォールバック ホットメソッドがアンマングルされた C# 名を示す キャプチャされた非 blittable 型が IL にフォールバックした。Burst Inspector で確認する。
Archetype のフラグメンテーション 小さなチャンクが多数。EntityManager.*Query マーカーが高い Archetype が細かすぎる — タグを統合するか Enableable Component を使用する。
Bake 時のリグレッション Play 中に Bake マーカーが表示される ドメインリロードが Baking を再トリガーした。通常 Editor のみ。ビルドで確認する。

7. Frame budget worked example

16.6 ms バジェット（60 Hz）を想定する。Render を除外した後:

  • SimulationSystemGroup: 6 ms
    • EnemyAISystem.OnUpdate: 2.1 ms（Burst、4 ワーカーで並列 Job）
    • MovementSystem.OnUpdate: 1.4 ms（Burst、並列）
    • EndSimulationECBS.Playback: 1.8 ms ← 要注意
    • その他合計: 0.7 ms

少数の構造変更のはずの ECB プレイバックに 1.8 ms かかっている場合、通常これが最大のレバーだ。コマンド数を確認する:

  1. 1フレームの Entities Journaling → System ごとの Instantiate/AddComponent/DestroyEntity の回数。
  2. 毎フレーム数百のコマンドを記録する System がある場合、バッチ処理する（Entity ごとではなくチャンクごとに1コマンド、またはチャンクインデックスソートキーで並列レコーダーに切り替える）。

8. Standalone Profiling

実際のビルドの場合:

  1. File → Build Profiles → Development Build + Autoconnect Profiler。
  2. プレイヤーが実行され、Profiler がリモートプロセスからキャプチャする。
  3. 結果は、プレイヤーが体験するものにより正直だ。Editor での測定はコードとは無関係な理由で 2〜5 倍遅くなることがある。

9. Troubleshooting

症状 原因 / 対処
プロファイルに DOTS マーカーが表示されない Profiler モジュールがフィルタリングしている — モジュール設定で「Scripts」と「Jobs」を有効にする。

Chapter 31. Managed Object Reference Audit

1. Overview

マネージドされた Unity オブジェクト参照は、ECS System を高速パスから誤って引き離す最も簡単な方法の1つだ。この監査は、Component データ・System・レンダリンググルーが Burst・Job・クリーンな SubScene Baking をブロックする方法で UnityEngine.Object 参照を保持している箇所を見つけるのに役立つ。

このページはコア ECS 移行後、またはプロファイラートレースでデータ指向コードが期待される場所にマネージドな処理が表示された場合に使用する。

2. What to search for

パターン 重要な理由
class .*: IComponentData マネージドコンポーネントデータ。有効だが通常ホットパスには適さない。
UnityEngine.Object マネージドされた Unity オブジェクト参照の広範なマーカー。
GameObject、Transform、Component ランタイムのシーンオブジェクト参照は通常シミュレーションデータには属さない。
Mesh、Material、Texture、AudioClip 多くの場合 UnityObjectRef<T> またはプレゼンテーション側でキャッシュされたデータにするべきだ。
Resources.Load ランタイムのアセット検索。Baking / コンテンツシステムで処理する方が良い場合が多い。
ScriptableObject 有用な Authoring ソースだが、ランタイム Job はベイクされた値や Blob を読み取るべきだ。

監査の範囲を ECS に面したコードに限定する。MonoBehaviour のみの UI やエディタツールは DOTS パターンに強制する必要はない。

3. Triage rules

発見事項 推奨アクション
ホットシミュレーションクエリにマネージド Component がある アンマネージド IComponentData・BlobAssetReference<T>・UnityObjectRef<T> に変換する。
プレゼンテーションのみで使用されるマネージド Component Burst / Job シミュレーションパスから分離されているなら維持する。
毎フレーム読まれる ScriptableObject 必要な値を Component または Blob アセットにベイクする。
ECS データにメッシュ / マテリアル参照がある UnityObjectRef<Mesh> / UnityObjectRef<Material> を使用し、プレゼンテーションコードで逆参照する。
Component データに GameObject / Transform が格納されている ECS データに置き換えるか、参照をハイブリッドプレゼンテーションブリッジに移動する。

4. Audit example

// Suspicious: managed component data used as per-entity visual state.
public class EnemyVisual : IComponentData
{
    public Material Material;
}

推奨:

using Unity.Entities;
using UnityEngine;

public struct EnemyVisual : IComponentData
{
    public UnityObjectRef<Material> Material;
}

その後、メインスレッドのプレゼンテーション System で逆参照するか、レンダラーレジストリに解決済みマテリアルをキャッシュする。

5. Profiler signs

Profiler の症状 確認事項
シミュレーション System が毎フレームアロケートしている OnUpdate 内のマネージドオブジェクト検索または LINQ / コレクションのアロケーション。
データのみになるべき System で Burst が無効 クエリパス内のマネージドコンポーネントまたはマネージド API アクセス。
メインスレッドに小さなレンダリング検索が多数ある Entity ごとのループの外で UnityObjectRef<T>.Value の結果をキャッシュする。
SubScene が予想より頻繁に再ベイクされる Authoring スクリプトまたはオブジェクト参照が不要な依存関係を作成している。

6. Validation

  1. §2 の検索パターンを再実行する。
  2. ホットシミュレーション System がアンマネージド Component のみをクエリしていることを確認する。
  3. Burst でコンパイルされるべき System / Job の Burst Inspector を確認する。
  4. 代表的なシーンをプロファイルし、ホットパスからマネージドアロケーションがなくなったことを確認する。
  5. クリーンな Editor 起動から Play モードに入り、SubScene Baking が参照を解決できることを確認する。

7. Troubleshooting

症状 原因 / 対処
変換した Component が Burst でクエリできない 残りのフィールドがマネージドだ。ネストされた構造体も確認する。
UnityObjectRef<T>.Value が Play モードで null を返す 参照されたアセットがベイクまたはロードされていない。Baker と SubScene / コンテンツワークフローを確認する。
変換後もビジュアル System が遅い 毎フレーム Entity ごとに逆参照している。共有アセットでキャッシュするか、コンパクトなルックアップキーをベイクする。
ScriptableObject の変更がランタイムデータに反映されない 値がベイクされた。SubScene を再インポート / 再ベイクするか、適切な Baking 依存関係を追加する。
Editor 専用の参照がプレイヤービルドで失敗する エディタ API を #if UNITY_EDITOR の後ろに移動し、ランタイムセーフなデータをベイクする。

Part IV. Migration

Chapter 32. Entities 1.x → 6.5 Overview

1. What you're signing up for

Entities 1.x から 6.5 へのアップグレードは、主にパッケージ配布形態の移行に3つの DOTS 移行が重なるものだ：

  1. Editor / パッケージ配布のアップグレード。 Unity 6000.4+（com.unity.entities 6.4）または 6000.5+（Entities 6.5）に移行する。Entities は Editor に組み込まれた Core Package になる。
  2. ECS データのクリーンアップ。 ECS に面したコードのマネージドされた Unity オブジェクト参照を監査し、ホットパスのコンポーネントデータをアンマネージドコンポーネント・ベイクされた値・BlobAssetReference<T>・UnityObjectRef<T> に移行する。
  3. ECS API 移行。 Entities 1.4 で廃止としてマークされたレガシー API — Entities.ForEach・IAspect — は 6.5 でも廃止のままだ。また ComponentLookup.GetRefRWOptional → TryGetRefRW の移行も必要だ。

各移行はこのフォルダ内の個別のサブドキュメントで行う。このページは マップだ — いつ・どれを・どの順序で行うか。

命名の注意: Entities パッケージバージョン（1.4 / 6.4 / 6.5）は Unity Editor バージョン（6000.3 / 6000.4 / 6000.5）とは異なる。6.4 以降は慣例的に一致するが、別個のリリースノートで管理される。このドキュメントでは2つの軸として扱う。Changelog ページ（../Changelog/Entities 1.4 → 6.5 Key Changes.md）も同様だ。

2. Recommended order

図: Entities 1.x から 6.5 への移行順序。

この順序の理由:

  • Editor を最初に — Editor が 6.x API を認識するまで、他のものがその API に対してコンパイルできない。
  • Package Manager → Core Package を次に — これなしには、重複 / 競合するパッケージ解決が発生する。
  • マネージド参照のクリーンアップをイテレーションリファクタの前に — System と Job を書き直す前にデータモデルを明確にする。
  • foreach / IAspect を最後に — これらは ECS 内部のもの。コードベースの残りをブロックしない。

3. What stays the same

驚くほど多くのものが変わらない — 移行が見た目よりも恐ろしくない理由だ。

変わらないもの 備考
IComponentData、IBufferElementData、ISharedComponentData 同じインターフェース、同じセマンティクス。
ISystem / SystemBase 同じライフサイクルフック、同じアトリビュート。
EntityManager API サーフェス CreateEntity・AddComponent など — 同一。
SystemAPI.Query 1.x で導入され、中心的なイテレーション API として維持される。
IJobEntity、IJobChunk 同じ形状。IJobEntity が推奨デフォルト。
EntityCommandBuffer + ECBS 同一。
Burst、Jobs、Collections、Mathematics 同一。
Netcode for Entities のコンセプト（Ghost・RPC・Prediction） 同じアーキテクチャ。パッケージの配布方法が異なるだけだ。

1.x のコードがすでに IJobEntity と SystemAPI.Query（レガシーの Entities.ForEach ではない）を使用しているなら、移行のステップ 4 と 5 はほぼ消えるはずだ。

5. Backup strategy

開始前に必須の作業:

  1. git でブランチを作成する。 説明的な名前を付ける（migrate/entities-6.5）。main で移行しない。
  2. 移行前の状態の manifest.json と packages-lock.json をコミットする。 解決が壊れた場合の revert のソースオブトゥルースだ。
  3. 移行前のコミットにタグを付ける。 git tag pre-entities-6.5 で revert が簡単になる。
  4. 準備ができるまで古い Editor バージョンでプロジェクトを開いたままにする。 6000.5 で一度開くと ProjectVersion.txt が変更され、移行パスが強制される。

6. Per-step checklists

6.1 Editor upgrade

  • Hub 経由で Unity 6000.5.x をインストールする。
  • まだプロジェクトを開かない。先に保留中の作業をコミットする。
  • 新しい Editor でプロジェクトを開く — ProjectVersion.txt のアップグレードを提案される。承認する。
  • 最初のインポートが完了するのを待つ。廃止 API や欠落パッケージに関する警告が出るはずだ。それが次のステップで修正する内容だ。

6.2 Package cleanup → 02_Package Manager → Core Package.md

  • manifest.json から com.unity.entities・com.unity.collections・com.unity.mathematics・com.unity.entities.graphics を削除する。
  • packages-lock.json を削除し、Editor に再生成させる。
  • プロジェクトが Netcode for Entities を使用している場合はそれも削除する（6.5 は Core Package バージョン）。

6.3 Managed object references → 03_Managed Object References → UnityObjectRef.md

  • ECS に面したコードのマネージドされた Unity オブジェクトフィールドを監査する。
  • 可能な場合、ホットパスのコンポーネントデータをアンマネージド ECS データに変換する。
  • Unity アセット / オブジェクトへの ECS 側の参照には UnityObjectRef<T> を使用する。
  • 大きな不変ゲームプレイデータには BlobAssetReference<T> を使用する。

6.4 Legacy ForEach → 04_foreach → IJobEntity.md

  • Entities.ForEach / Job.WithCode を使用している System については、IJobEntity + SystemAPI.Query に移行する。

6.5 Aspect removal → 05_IAspect Removal.md

  • Aspect メソッドを RefRW<T> / RefRO<T> パラメータを使用した呼び出し元にインライン化する。
  • IAspect 構造体を削除する。

7. Testing after migration

移行完了を宣言する前に実行するスモークテスト:

  1. 代表的な SubScene での Play モード。 Entity が存在し・System がティックし・起動時に何もスローしないことを確認する。
  2. スタンドアロンプレイヤーをビルドする。 ビルドのみの不具合（通常はシリアライズ）がここで現れる。
  3. 既存のゲームプレイ回帰テストを実行する。 パフォーマンスが重要な場合は、移行前のベースライン Profiler トレースをキャプチャして比較する。
  4. セーブ / ロードを確認する。 生の Entity ハンドルは World ローカルで安定していない。ゲーム所有の ID・コンテンツキー・Authored データを保存する。
  5. ターゲットプラットフォームで実行する。 プラットフォーム固有のバックエンド（IL2CPP の特性、AOT）がここで現れる場合がある。

8. When to roll back

クリーンなロールバックのトリガー:

  • Editor がプロジェクトをまったく開けない（まれ。通常 ProjectVersion.txt の不一致）。
  • packages-lock.json の解決が意味不明なグラフを生成し、削除しても修正されない。
  • 依存している重要なパッケージ（サードパーティ）がまだ 6.5 と互換性がない。

手順: git checkout pre-entities-6.5、古い Unity バージョンを再インストール、旧バージョンでゲームが動作することを確認する。

部分的に完了したが行き詰まった移行はクリーンに中止し、タグから再開する — 半移行のコードをリリースしない。

9. Troubleshooting

症状 原因 / 対処

manifest クリーンアップ後、「Unity.Entities assembly not found」コンパイルエラー Editor が 6000.4 より古い — Core Package が利用できない。Editor をアップグレードするか、UPM パッケージを追加し直す。

Chapter 33. Package Manager → Core Package

1. What changes

Unity 6000.4+ では、Entities（およびそのコンパニオン DOTS パッケージ）は Package Manager 経由でのインストールが不要になる。これらは Core Package として Editor 内に同梱される。プロジェクトファイルは3箇所で変わる:

  • Packages/manifest.json — com.unity.entities* のエントリが削除される。
  • Packages/packages-lock.json — 解決されたグラフが縮小する。影響を受けるパッケージは "source": "builtin" と表示される。
  • Package Manager ウィンドウ — パッケージが「In Project」に Built-in とラベル付きで表示される。

C# コードは変わらない。アセンブリ名（Unity.Entities・Unity.Collections など）は同じで、.asmdef 参照も変更なしだ。

2. Which packages move

6000.4+ で Core Package に移行:

UPM id（移行前） 移行後
com.unity.entities Core Package、Editor に同梱
com.unity.collections Core Package
com.unity.mathematics Core Package
com.unity.entities.graphics Core Package
com.unity.netcode（Netcode for Entities） 6.5 で Core Package

Package Manager のまま:

  • com.unity.physics（Unity Physics）
  • com.unity.charactercontroller（Character Controller）

エンジンに組み込み（変更なし）:

  • Jobs system
  • Burst（com.unity.burst は一部の設定では UPM エントリとして公開されているが、6000.4+ では Editor バージョンに紐付けられている。manifest はそのままにしておくのが最も安全だ）。

Packages/manifest.json を開く。次のような内容だ:

     ...
   }
 }

com.unity.physics とその他 UPM のまま残すものはそのままにする。

ファイルを保存する。

4. Rebuilding packages-lock.json

2つのオプション:

4.1 Let the Editor regenerate it

Editor を閉じ、Packages/packages-lock.json を削除し、再度開く。Unity が再生成する。これが最もクリーンなオプションだ。

4.2 Edit in place

packages-lock.json を開き、上記5つのパッケージに一致するエントリを削除する。Editor は次回起動時に調整する。複雑なロックファイルの場合は若干高速だが、エラーが起きやすい。

いずれの方法でも最終的なロックファイルには次のようなエントリが含まれる:

"com.unity.entities": {
  "version": "6.5.0",
  "depth": 0,
  "source": "builtin",
  "dependencies": {}
}

"source": "builtin" がパッケージが Core Package になった証だ。

5. Verifying the result

  1. Package Manager ウィンドウ（Window → Package Manager → 「In Project」）— Entities・Collections・Mathematics・Entities Graphics（使用している場合は Netcode for Entities も）が Built-in タグ付きで表示されるはずだ。
  2. Editor メニュー — Window → Entities → Hierarchy/Systems/Archetypes が存在して動作する。
  3. コンパイル — Play を押す。欠落アセンブリを参照するコンパイルエラーなしに Play モードに入るはずだ。

Chapter 34. Managed Object References → UnityObjectRef

1. Overview

古い ECS コードを移行する際の一般的な問題は、マネージドオブジェクト参照がランタイムのコンポーネントデータに漏れ込むことだ。Entities 6.5 では、可能な限りゲームプレイコンポーネントをアンマネージドのままにし、ECS データがメッシュ・マテリアル・テクスチャ・オーディオクリップ・Authored ScriptableObject などの Unity オブジェクトを指す必要がある場合は UnityObjectRef<T> を使用する。

このページの範囲は DOTS コンポーネントの設計だ。エンジン全体の Unity オブジェクト識別子 API はカバーしない。

2. What to look for

マネージドされた Unity オブジェクトを直接格納しているコンポーネントを探す:

パターン 確認すべき理由
IComponentData を実装する class コンポーネント マネージドコンポーネントは許可されるが、より低速でメインスレッドに制限される。
コンポーネントデータ内の UnityEngine.Object フィールド 通常、マネージドコンポーネントまたは無効なアンマネージドコンポーネント設計を示す。
GameObject・Transform・Material・Mesh・Texture・AudioClip フィールド これが Authoring・プレゼンテーション・UnityObjectRef<T> のどれに属するか判断する。
Entity ごとに Resources.Load を呼ぶランタイム System アセット解決を Baking またはセントラルコンテンツシステムに移動する。

3. Before and after

Before — managed component

using Unity.Entities;
using UnityEngine;

public class ProjectileVisual : IComponentData
{
    public Mesh Mesh;
    public Material Material;
}

これは機能するが、Component をマネージドにしてしまう。マネージドコンポーネントは Burst / Job の柔軟性を低下させ、Entity ごとのゲームプレイデータには通常不適切だ。

After — unmanaged component with UnityObjectRef<T>

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

参照を一度ベイクする:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

4. Runtime access pattern

シミュレーション Job は純粋にデータ指向に保つ。マネージドされた Unity API が許可される場所でのみ Unity オブジェクトを逆参照する。

using Unity.Entities;
using UnityEngine;

public partial class ProjectileVisualUploadSystem : SystemBase
{
    protected override void OnUpdate()
    {
        foreach (var visual in SystemAPI.Query<RefRO<ProjectileVisual>>())
        {
            Mesh mesh = visual.ValueRO.Mesh.Value;
            Material material = visual.ValueRO.Material.Value;

            if (mesh == null || material == null)
                continue;

            // Upload to a rendering registry, material property block cache, etc.
        }
    }
}

高頻度のレンダリングでは、Entity ごとにフレームごとに逆参照することを避ける。コンパクトな ECS 値をキーとするプレゼンテーション側のレジストリに解決済みオブジェクトをキャッシュする。

5. When not to use UnityObjectRef<T>

ケース より良い選択
純粋なゲームプレイデータ プリミティブ / 数学 / Blob データを直接 IComponentData に格納する。
大きな読み取り専用テーブル BlobAssetReference<T>。
同じ SubScene にベイクされた Entity Prefab Entity Prefab 参照。
コンテンツワークフロー経由でストリームされる Prefab EntityPrefabReference。
Entity ごとの複雑なマネージド挙動 シミュレーションパスの外に保つか、マネージドプレゼンテーション System に分離する。

6. Migration checklist

  • Unity オブジェクト参照を格納するためだけに使用されているマネージドコンポーネントデータを見つける。
  • アンマネージドコンポーネントが適切な場所では、それらのフィールドを UnityObjectRef<T> に置き換える。
  • オブジェクトの割り当てを Baker に移動する。
  • Burst Job からは .Value の逆参照を除外する。
  • 多くの Entity が同じアセットを共有する場合は逆参照済みオブジェクトをキャッシュする。
  • ランタイムでの ScriptableObject の読み取りの代わりに、大きな不変ゲームプレイデータには BlobAssetReference<T> を使用する。

7. Troubleshooting

症状 原因 / 対処
Component が Burst Job で使用できない まだマネージドだ。Unity オブジェクトフィールドを UnityObjectRef<T> に変換するか、プレゼンテーションデータを分離する。
.Value が null 参照されたアセットがロードされていないか破棄された。Authoring の参照とコンテンツロードパスを確認する。
System が毎フレームアロケートする Entity ごとにアセットを解決またはロードしている。参照をベイクし、解決済みマネージドオブジェクトをキャッシュする。
ゲームプレイ System がランタイムで ScriptableObject に依存している 必要な値を Component データまたは Blob アセットにベイクする。
ビルドがアセットをストリップする アセットが Baking / コンテンツワークフロー経由で参照されており、ランタイムの文字列パスだけで参照されていないことを確認する。

Chapter 35. foreach → IJobEntity Migration

1. Status — obsolete, not yet removed

レガシーなコンポーネントイテレーション API は Entities 1.4+ / 6.5 で obsolete 扱いだが、まだコンパイルはできる：

レガシー 6.5 でのステータス 代替
Entities.ForEach 1.4 以降 廃止。使用するたびに廃止警告が出る。Unity の Changelog では将来のメジャーリリースで削除予定とされている SystemAPI.Query<...>（メインスレッド）または IJobEntity（Job）
Job.WithCode ForEach とともに 廃止 プレーンな IJob 構造体、または OnUpdate にインライン
IJobForEach（1.0 以前） 1.4 よりずっと前に削除済み IJobEntity

ここに来るのは、1.x プロジェクトを 6000.5+ で開いてコンパイラが大量に警告を出しているか、将来のメジャーパッケージリリースで API が削除される前にこれらの警告をクリアしたいためだ。パターンを理解すればリファクタリングは機械的に行える。

2. The two replacement APIs

  • SystemAPI.Query<T, U, ...> — OnUpdate 内のメインスレッドイテレーション。シンプルで Job オーバーヘッドがなく、小さいワークロードやマネージド API が必要な場合に理想的だ。
  • IJobEntity — ソース生成された並列 Job。マルチスレッドな処理を行う準備ができていれば直接移行できる。これが DOTS の強みだ。

背景: ../DOTS Workflows/11_IJobEntity · SystemAPI.Query.md。

3. Port — Entities.ForEach → SystemAPI.Query

3.1 Before

public partial class HealthRegenSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float dt = Time.DeltaTime;

        Entities
            .WithAll<Alive>()
            .WithNone<Dead>()
            .ForEach((ref Health health, in Regen regen) =>
            {
                health.Value = math.min(health.Value + regen.Rate * dt, health.Max);
            })
            .ScheduleParallel();
    }
}

3.2 After — SystemAPI.Query (main thread)

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>()
                          .WithAll<Alive>()
                          .WithNone<Dead>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRO.Max);
        }
    }
}

形状の変化に関する注意点:

  • SystemBase → ISystem（より軽量で Burst 化できる）。古いコードがマネージドフィールドをキャプチャしている場合は SystemBase を維持する必要があるかもしれない。
  • ref Health → RefRW<Health> タプル要素。.ValueRO で読み取り、.ValueRW で書き込む。
  • in Regen → RefRO<Regen>。
  • Time.DeltaTime → SystemAPI.Time.DeltaTime。
  • フィルター（WithAll/WithNone）は Query ビルダーに移動する。

4. Port — Entities.ForEach.ScheduleParallel → IJobEntity

ForEach が並列スケジュールされていた場合、Job 形式は直接 1:1 で移行でき、並列性を維持する。

4.1 After — IJobEntity

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();

}

フィルターは Job 構造体にアトリビュートとして記述する:

[WithAll(typeof(Alive))]
[WithNone(typeof(Dead))]
public partial struct HealthRegenJob : IJobEntity { ... }

またはスケジュール時にクエリを渡す:

var q = SystemAPI.QueryBuilder()
    .WithAll<Health, Regen, Alive>()
    .WithNone<Dead>()
    .Build();
new HealthRegenJob { DeltaTime = dt }.ScheduleParallel(q);

5. Captures and their replacements

レガシーの Entities.ForEach では .WithReadOnly・.WithDisposeOnCompletion などでローカル変数をキャプチャできた。IJobEntity では、キャプチャは Job 構造体のフィールドだ — Burst は何が渡されているかを正確に把握する。

// BEFORE
var buffer = new NativeArray<int>(count, Allocator.TempJob);
Entities.ForEach((ref Foo f) => { /* use buffer */ })
    .WithDisposeOnCompletion(buffer)
    .ScheduleParallel();

// AFTER
[BurstCompile]
partial struct FooJob : IJobEntity
{
    [ReadOnly] public NativeArray<int> Buffer;

    void Execute(ref Foo f) { /* use Buffer */ }
}

var buffer = new NativeArray<int>(count, Allocator.TempJob);
state.Dependency = new FooJob { Buffer = buffer }.ScheduleParallel(state.Dependency);
buffer.Dispose(state.Dependency);   // chained dispose replaces WithDisposeOnCompletion

アトリビュートの置き換え:

  • .WithReadOnly(x) → [ReadOnly] public NativeArray<T> X;
  • .WithDisposeOnCompletion(x) → スケジュール後の x.Dispose(handle)。
  • .WithoutBurst() → Job 構造体の [WithoutBurst]（ただし、そもそも WithoutBurst が必要になった原因を取り除けないか強く検討すること）。
  • .WithEntityAccess() → Execute に Entity entity パラメータを追加する。

6. When a SystemBase is still required

元の System に真のマネージドフィールド（Input System・UI Toolkit・外部サービス）があった場合、機械的に ISystem に変換できない。2つのオプションがある:

  1. SystemBase のままにするし、OnUpdate 内で SystemAPI.Query を使用する。シングルスレッドだ。
  2. 分割する — マネージドワールドをブリッジする SystemBase と Burst 処理を行う ISystem に分ける。../DOTS Workflows/08_System — ISystem vs SystemBase.md を参照。

7. Validation

System を移行した後:

  1. 挙動のユニットテストを行う。 移行でロジックは変わらないはずだ。出力が異なる場合は、リファクタリングでバグを導入した。
  2. プロファイルする。 並列 ForEach と IJobEntity.ScheduleParallel は同等の処理で 5% 以内のはずだ。大きなリグレッションは Burst が有効でないことを意味する — Burst Inspector を確認する。
  3. 古い System ファイルを削除する。 #if LEGACY のフォールバックを残さないようにする — それは腐敗する。

Chapter 36. IAspect Deprecation — Migrating Off Aspects

1. Status — deprecated, not yet removed

IAspect は Entities 1.4 で廃止としてマークされ、6.x 系でも廃止のままだ。まだコンパイルでき・動作し・6.5 の API リファレンスでは有効な IJobEntity.Execute パラメータ種類としてリストされている — ただしコンパイラは廃止警告を出し、Unity は将来のメジャーリリースで削除されると述べている。新しいコードでは使用すべきでない。既存のコードは機会があれば移行する。

IAspect は System が一連の Component に対する「ビュー」を宣言し、Entities.ForEach または IJobEntity で単一の引数として使用できるように導入された。実際にはそのコストを回収できないソース生成の層と並列型システムを追加してしまった:

  • Aspect を書くことで Component パラメータリストが重複した。
  • デバッガが生成されたラッパーコードをステップ実行した。
  • IJobEntity と SystemAPI.Query の RefRW<T> / RefRO<T> は、追加の型なしに同じ「Component をバンドルとして渡す」エルゴノミクスを実現する。

API サーフェスはまだ残っているため、Editor のアップグレードでコードは壊れない — この移行は警告を解消し、2.0 に備えるためのものだ。

2. Mechanical refactor

2.1 Starting point

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public readonly partial struct MovementAspect : IAspect
{
    public readonly RefRW<LocalTransform> Transform;
    public readonly RefRO<Velocity>       Velocity;
    public readonly RefRO<Speed>          Speed;

    public void Step(float dt)
    {
        Transform.ValueRW.Position += Velocity.ValueRO.Direction * Speed.ValueRO.Value * dt;
    }
}

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var m in SystemAPI.Query<MovementAspect>())
            m.Step(dt);
    }
}

2.2 After — direct component parameters

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var (transform, velocity, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<Velocity>, RefRO<Speed>>())
        {
            transform.ValueRW.Position += velocity.ValueRO.Direction * speed.ValueRO.Value * dt;
        }
    }
}

2つの変更点:

  1. Aspect 型がなくなった。そのフィールドが SystemAPI.Query のタプル要素になる。
  2. Aspect に存在したメソッド（Step）がインライン化された。インライン化するには大きすぎる場合は、同じパラメータを取る 静的ヘルパーに抽出する — これで Burst フレンドリーを保てる:
static void Step(ref LocalTransform t, in Velocity v, in Speed s, float dt)
{
    t.Position += v.Direction * s.Value * dt;
}

呼び出し方:

Step(ref transform.ValueRW, velocity.ValueRO, speed.ValueRO, dt);

3. Aspects in IJobEntity

3.1 Before

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;
    void Execute(MovementAspect aspect) => aspect.Step(DeltaTime);
}

3.2 After — components as Execute parameters

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in Velocity velocity, in Speed speed)
    {
        transform.Position += velocity.Direction * speed.Value * DeltaTime;
    }
}

IJobEntity は RefRW<T> / RefRO<T> ではなく、プレーンな ref / in パラメータを使用することに注意する。（ジェネレーターが内部でコンポーネントをチャンク配列アクセスにラップする。）

4. Aspects that wrapped a DynamicBuffer

// BEFORE
public readonly partial struct PathAspect : IAspect
{
    public readonly DynamicBuffer<Waypoint> Path;
    public void Advance(int n) => Path.RemoveRange(0, n);
}

// AFTER — access buffer directly
foreach (var (path, e) in SystemAPI.Query<DynamicBuffer<Waypoint>>().WithEntityAccess())
{
    path.RemoveRange(0, 1);
}

Job では DynamicBuffer<T> は有効な Execute パラメータだ — 追加のものは不要だ。

5. Cross-entity aspects

Aspect が「別の Entity の Component を覗き見る」ために ComponentLookup<T> をラップすることがあった。代替:

// BEFORE
public readonly partial struct AITargetAspect : IAspect
{
    public readonly RefRO<Target> Target;
    [ReadOnly] public readonly ComponentLookup<Health> HealthLookup;
    public float TargetHealth =>
        HealthLookup.TryGetComponent(Target.ValueRO.Entity, out var h) ? h.Value : 0f;
}

// AFTER — declare the lookup on the system / job directly
public partial struct AISystem : ISystem
{
    private ComponentLookup<Health> _healthLookup;

    public void OnCreate(ref SystemState state)
    {
        _healthLookup = state.GetComponentLookup<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthLookup.Update(ref state);

        foreach (var (target, e) in SystemAPI
                     .Query<RefRO<Target>>()
                     .WithEntityAccess())
        {
            float hp = _healthLookup.TryGetComponent(target.ValueRO.Entity, out var h)
                ? h.Value : 0f;
            // ...
        }
    }
}

このパターンは Lookup が Entity 間で共有されるため、Aspect バージョンより軽量でもある。

6. When an aspect was complex

Aspect に多くのフィールドがあり・状態をカプセル化し・非自明な継承に参加していた場合、それは単なる移行ではなく本格的なリファクタリングを検討すべき兆候だ。多くの場合、God Aspect を持つ1つの System よりも、Component を通じてデータを受け渡す2つの System の方がコードが読みやすくなる。

9. Supporting concepts

用語 内容
Burst Job と [BurstCompile] でマークされた構造体のための Unity のネイティブコンパイラ。ネイティブ SIMD コードを生成する。
Change version System がチャンク内の Component カラムに書き込むたびにインクリメントされる 32 ビットカウンタ。SetChangedVersionFilter で変化のないチャンクをスキップできる。
SoA (Structure of Arrays) 各フィールドが独自の配列になるストレージレイアウト — スカラーイテレーションでキャッシュフレンドリー。チャンクは SoA を使用する。
RefRW<T> / RefRO<T> SystemAPI.Query で読み書き / 読み取り専用の意図を持って Component にアクセスするためのラッパー型。
EnabledRefRW<T> / EnabledRefRO<T> 構造変更をトリガーせずに Enableable Component のトグルビットを読み書きするためのラッパー。
TypeManager 各 Component に安定した型インデックスを割り当てるランタイムレジストリ。起動時に再ビルドされる。大規模プロジェクトではこれが感じられる。
BlobAssetReference<T> チャンク外に格納された大きな読み取り専用データ Blob へのポインタ。チャンク容量の肥大化を防ぐ。
