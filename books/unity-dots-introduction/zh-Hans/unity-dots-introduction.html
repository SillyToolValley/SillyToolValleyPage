第一部分. 入门

第 1 章. 环境搭建 — Unity 6000.5 + Entities 6.5.0

1. 概述

从 Unity 6000.4+ 起，Entities 已成为 Core Package——它随编辑器一同发布，无需通过 Package Manager 添加。本指南将引导您安装编辑器、创建支持 DOTS 的项目，并确认 Entities 已正常启用。

项目环境参考：Unity 6000.5.0b2 · Entities 6.5.0 · Collections / Mathematics / Entities Graphics（均为 Core Package）。

2. 安装 Unity 6000.5+

  1. 打开 Unity Hub。如果尚未安装，请先安装 Unity Hub（https://unity.com/download）。
  2. 在 Hub 中，前往 Installs → Install Editor。
  3. 选择任意 6000.5.x 版本（b 系列测试版或更新版本）。Entities 6.5 仅在 6000.5+ 上可用。
  4. 模块：保持默认选项即可。ECS 本身不需要额外模块。

您无需选择任何"DOTS"模块——从 6000.4 起，Entities 已内置于编辑器中。

3. 创建新项目

在 Hub → Projects → New project 中：

字段 值
Editor Version 6000.5.x
Template Universal 3D（或 3D (Built-in)、HDRP——均可使用）
Project Name 任意名称

点击 Create project。编辑器将在空场景中打开。

在 6000.4+ 上，不再需要专用的"DOTS 模板"。任何模板均可承载 Entities 工作流。

4. 验证核心包已激活

打开 Window → Package Manager，将范围切换为 In Project。您应当看到以下包被标记为 "Built-in"：

  • Entities
  • Collections
  • Mathematics
  • Entities Graphics

如果您希望通过文件验证，请检查 Packages/manifest.json：其中不存在 com.unity.entities 条目，因为它随编辑器一同发布。

接下来，打开 ECS 相关窗口：

  • Window → Entities → Hierarchy——在运行时显示 Entity 列表。
  • Window → Entities → Systems——显示已注册的 System。
  • Window → Entities → Archetypes——显示 Chunk 布局。

如果这三个菜单均存在，说明 Entities 已正确配置。

5. 冒烟测试——创建 SubScene

  1. 在 Hierarchy 中，右键单击 → New Sub Scene → Empty Scene…
  2. 将其命名为 MainSubScene.unity 并保存。
  3. 在新建的 SubScene 中，添加任意基元体（3D Object → Cube）。
  4. 保存场景。当 SubScene 被关闭（显示黄色图标）或在构建时，Cube 将被 Bake 为 Entity。
  5. 按下 Play。打开 Window → Entities → Hierarchy——您应当看到一个 Cube Entity。冒烟测试通过。

SubScene 是 Authoring 的入口点：其中的 GameObject 在保存时和构建时将被 Bake 为 Entity。后续教程详见 03_Hello DOTS — First Entity.md。

第 2 章. 核心包详解

1. 概述

随着 Entities 包 6.4.0 的发布（随 Unity Editor 6000.4 一同推出），主要 DOTS 包从 Package Manager 迁移到编辑器内部，成为 Core Package。包变更日志中的原文表述为：

"Package is now a core package embedded in Unity."

此变更影响了 com.unity.entities、com.unity.collections、com.unity.mathematics、com.unity.entities.graphics，以及（在其对应的 6.5 版本发布时）Netcode for Entities。这一变更影响了安装、版本管理和追踪方式——但这是一次分发方式的变更，而非 API 变更。

本页将说明：

  • 哪些包现在是 Core Package（以及哪些不是）。
  • 其版本号从何而来。
  • manifest.json 和 packages-lock.json 的变化。
  • 此次变更的原因。

2. 什么是 Core Package？

Core Package 是随编辑器内部一同发布的包。您无需通过 Package Manager 添加它，也无法独立于编辑器更改其版本，其发布节奏与编辑器保持一致。

特性 Package Manager 包 Core Package
安装步骤 通过 UPM 添加 无需安装——随编辑器一同提供
版本选择方 您，在 manifest.json 中 Unity，按编辑器版本决定
发布节奏 UPM 独立发布 随编辑器发布
出现在 manifest.json 中 是 否
出现在 packages-lock.json 中 是 是，标记为 builtin
在 Package Manager 中显示 在"In Project"下 在"In Project"下，标记为 Built-in

3. Unity 6000.5+ 上的 DOTS 包映射

包 在 Unity 6000.5+ 上 用途
Entities Core Package（6.5） ECS 运行时：Entity、Component、System、World
Collections Core Package 原生容器类型（NativeList<T>、NativeHashMap<K,V> 等）

注意：Job System 和 Burst 从未是 UPM 包——它们是引擎的一部分。6000.4 中的变化专指构建于其之上的 DOTS 包。

4. 过渡后的版本号

Core Package 版本号与编辑器版本对齐：

旧的 1.x 系列仍然适用于尚无法迁移到 6000.4+ 的项目。两个版本系列是独立的；通过升级编辑器来升级至 6.x。

关于 Entities 过渡说明，请参阅 Changelog/Entities 1.4 → 6.5 Key Changes.md；关于 Netcode 专项变更，请参阅 Changelog/Netcode for Entities 1.4 → 6.5 Key Changes.md。

5. 对项目文件的影响

manifest.json

无需填写 com.unity.entities、com.unity.collections、com.unity.mathematics 或 com.unity.entities.graphics 条目。如果您升级 1.x 项目，请将其删除——详见 Migration/02_Package Manager → Core Package.md。

packages-lock.json

Core Package 将以 "source": "builtin" 形式出现，而非版本 URL。

程序集定义（.asmdef）

您的 .asmdef 文件仍然引用程序集名称（例如 Unity.Entities、Unity.Collections）。此处无需任何变更——只有包的分发层发生了改变。

6. 此次变更的原因

其既定目标是：通过将 DOTS 更新与编辑器发布周期绑定（而非使用独立的包发布计划），让 ECS 团队能够更快速、更渐进地交付功能。同时，这也使引擎能够在内部依赖 ECS，而在包为可选状态时，这种依赖关系较为尴尬。

第 3 章. Hello DOTS — 第一个 Entity

1. 您将构建的内容

一个绕 Y 轴旋转的 Cube，由 ECS System 驱动。在一次练习中，您将接触到所有核心概念：SubScene、Authoring → Baker、IComponentData、ISystem，以及 Entities Hierarchy / Systems 窗口。最后，您将把主线程 System 替换为 IJobEntity，以了解如何引入并行处理。

前提条件：[01_Environment Setup.md](01_Environment Setup (Unity 6000.5 + Entities 6.5.0).md)——一个已确认 SubScene 和 Entities 菜单可用的 Unity 6000.5 项目。

2. 创建 SubScene

  1. 打开您的空场景。
  2. 在 Hierarchy 中，右键单击 → New Sub Scene → Empty Scene…
  3. 将其保存为 Assets/Scenes/ 目录下的 HelloDots.unity。
  4. 在新建的 SubScene 中，右键单击 → 3D Object → Cube。将其重命名为 RotatingCube。
  5. 保存场景（Ctrl/⌘ + S）。通过单击 Hierarchy 中 SubScene 名称旁的复选框来关闭它——图标变为黄色，表示其内容正在被 Bake 为 Entity。

从此刻起，对 RotatingCube 组件的修改，将在您重新打开/关闭 SubScene 时自动触发重新 Bake。

3. 定义 Component——RotationSpeed

创建 Assets/Scripts/RotationSpeed.cs：

using Unity.Entities;

public struct RotationSpeed : IComponentData
{
    public float RadiansPerSecond;
}

说明：

  • 在非托管 struct 上实现 IComponentData 是默认选择——它直接存储在 Chunk 中，且对 Burst 友好。
  • 字段应保持可位块传输（blittable）（基础类型、Unity.Mathematics 类型、其他非托管结构体）。

4. 定义 Authoring + Baker

创建 Assets/Scripts/RotationSpeedAuthoring.cs：

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class RotationSpeedAuthoring : MonoBehaviour
{
    public float DegreesPerSecond = 90f;

    class Baker : Baker<RotationSpeedAuthoring>
    {
        public override void Bake(RotationSpeedAuthoring authoring)
        {
            // Dynamic — this entity will have its transform written each frame.
            var entity = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(entity, new RotationSpeed
            {
                RadiansPerSecond = math.radians(authoring.DegreesPerSecond)
            });
        }
    }
}

将 Authoring 附加到 Cube 上

  1. 在 SubScene 中选中 RotatingCube。
  2. Inspector → Add Component → Rotation Speed Authoring。
  3. 保持 DegreesPerSecond = 90。

SubScene 关闭（显示黄色图标）后，Baker 立即运行，Cube 将被转化为带有 RotationSpeed Component 的 Entity。

验证 Bake 结果

  • Window → Entities → Hierarchy——您应当看到 RotatingCube 列于其中。
  • 单击它。右侧 Inspector 显示其 Component——找到 RotationSpeed { RadiansPerSecond: ~1.5708 }（π/2）。

5. 编写 System——RotationSystem

创建 Assets/Scripts/RotationSystem.cs：

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<RotationSpeed>>())
        {
            transform.ValueRW = transform.ValueRO.RotateY(speed.ValueRO.RadiansPerSecond * dt);
        }
    }
}

说明：

  • partial struct + ISystem 是非托管 System 的形式。它可被 Burst 编译，运行时无 GC 开销。
  • SystemAPI.Query<RefRW<A>, RefRO<B>>() 是现代迭代 API——它取代了已废弃的 Entities.ForEach。
  • LocalTransform.RotateY(radians) 返回旋转后的副本；我们通过 transform.ValueRW = … 将其写回。

按下 Play

Cube 开始旋转。在 Window → Entities → Systems 中，您可以看到 RotationSystem 列于仿真组下方，并附有每帧耗时。

6. 检视您构建的内容

进入 Play 模式后的实用窗口：

窗口 显示内容
Entities Hierarchy Entity 实时列表。单击任意 Entity 可查看其 Chunk 及 Component。
Systems 所有正在运行的 System，按其 SystemGroup 分组。此处可查看每个 System 的耗时。
Archetypes 按 Component 签名分组的 Chunk。可用于发现碎片化问题。

请花时间确认：

  1. RotatingCube Entity 存在，且拥有 LocalTransform + RotationSpeed。
  2. RotationSystem 出现在 SimulationSystemGroup 下方，并累计帧耗时。

7. 切换为 IJobEntity（并行化）

将 RotationSystem.cs 替换为基于 Job 的版本，以了解如何接入并行处理：

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotateJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in RotationSpeed speed)
    {
        transform = transform.RotateY(speed.RadiansPerSecond * DeltaTime);
    }
}

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new RotateJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

说明：

  • IJobEntity.Execute(ref A, in B) 通过源代码生成转化为 Chunk 级别的 Job。参数即为您操作的 Component。
  • .ScheduleParallel() 将 Chunk 分发到工作线程。对于单个 Cube，您不会看到速度提升，但此代码形式可以在不修改的情况下扩展至数千个 Entity。
  • System 的依赖关系由框架自动处理——当您从 ISystem.OnUpdate 中调用 ScheduleParallel() 时，state.Dependency 会自动合并。

第二部分. DOTS 工作流

第 4 章. Baker 模式 & SubScene

1. 概述

Entity 无法直接在 Hierarchy 中 Authoring。您需要在 SubScene 中 Authoring GameObject，由 Baker 在 SubScene 关闭或构建时将其转换为 Entity。

Figure: Baker and SubScene Pipeline.

2. SubScene 生命周期

SubScene 有三种可见状态：

Hierarchy 图标 状态 含义
⬜（白色复选框） 打开 其 GameObject 可编辑；仍为 GameObject，未进行 Baking。
🟨（黄色复选框） 关闭 Baker 已运行；其内容仅以 Entity 形式存在。
⏳ Baking 中 增量 Bake 进行中（短暂状态）。

创建 SubScene

  • Hierarchy → 右键单击 → New Sub Scene → Empty Scene…——保存一个 .unity 资产，并在父 GameObject 上添加 SubScene Component。
  • 通过 SubScene 名称旁的复选框来开关。关闭时触发 Bake。

增量 Baking

当 SubScene 处于打开状态时，如果您编辑某个 GameObject，只有接触到被修改 Authoring 的 Baker 才会重新运行。因此请避免在 Bake() 中执行耗时操作。

3. Baker<T> API

Baker 将一个 Authoring MonoBehaviour 映射到一个或多个 Entity 及其 Component。

public class EnemyAuthoring : MonoBehaviour
{
    public float MaxHealth = 100f;
    public GameObject ProjectilePrefab;

    class Baker : Baker<EnemyAuthoring>
    {
        public override void Bake(EnemyAuthoring authoring)
        {
            // Primary entity for this GameObject.
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Health { Value = authoring.MaxHealth });

            // Bake a GameObject reference into an Entity reference.
            AddComponent(self, new ProjectilePrefabRef
            {
                Value = GetEntity(authoring.ProjectilePrefab, TransformUsageFlags.Dynamic)
            });
        }
    }
}

关键 API：

API 用途
GetEntity(TransformUsageFlags) 当前 Authoring GameObject 对应的 Entity。
GetEntity(Object, TransformUsageFlags) 对应另一个 GameObject/Prefab 的 Entity（注册一个 Prefab Entity）。
CreateAdditionalEntity(TransformUsageFlags) 链接到同一 Authoring 的第二个 Entity（例如子部件）。
AddComponent<T>(entity, value) 附加 Component 数据。
AddBuffer<T>(entity) / SetBuffer<T> 附加 DynamicBuffer<T>。
AddSharedComponent<T> 附加 ISharedComponentData。
DependsOn(asset) 声明资产依赖，以便资产变更时 Baker 重新运行。

4. TransformUsageFlags

Baking 仅在您通过 TransformUsageFlags 明确请求时，才会添加 LocalTransform / LocalToWorld / Parent。不必要的 Transform Component 会无谓地增大 Chunk，因此请选择能够满足需求的最小标志。

标志 添加内容 典型用途
None 无 纯数据 Entity（配置、Singleton、纯逻辑）。
ManualOverride 无；由您自行管理 Transform 自定义 Transform 布局。
Renderable LocalToWorld 从不移动的静态网格。
Dynamic LocalTransform、LocalToWorld（若有父级则附加 Parent） 任何会移动或由 System 写入的对象。
WorldSpace LocalTransform、LocalToWorld；阻止父子关系 忽略层级的 Entity。
NonUniformScale 附加 PostTransformMatrix 非均匀缩放（在 DOTS 中不常见）。

Dynamic 是所有将由 System 移动或旋转的对象的安全默认选项。

5. Baking System（高级）

Baker 从单个 Authoring 生成 Component。Baking System 在所有 Baker 完成后运行，并能看到完整的 Baked World——适用于跨 Entity 的后处理（解析引用、计算派生数据）。

[WorldSystemFilter(WorldSystemFilterFlags.BakingSystem)]
public partial struct EnemyWaveLinkSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Runs during bake only. Query entities tagged earlier by Bakers
        // and wire them together.
    }
}

当 Baker 并不掌握所有所需信息时（例如数量统计、交叉引用），请使用此方式。保持其确定性——Baking 每次运行必须产生相同的结果。

Chapter 5. Spawner 示例

1. 概述

Spawner 是一个能够创建更多 Entity 的单一 Entity——可在启动时、定时触发或响应事件时创建。本节将完整构建一个 Spawner，涵盖：

  • 在 SubScene 中编写 Spawner 的 Authoring。
  • 存储预制体 Entity 引用（从 GameObject Prefab Bake 而来）。
  • 通过 EntityCommandBuffer 在运行时实例化，确保结构性变更安全执行。

建议先阅读 01_Baker Pattern & SubScene.md。

2. 数据模型

using Unity.Entities;
using Unity.Mathematics;

public struct Spawner : IComponentData
{
    public Entity  Prefab;       // baked prefab entity
    public float3  SpawnPoint;
    public float   Interval;
    public float   TimeLeft;
    public int     RemainingCount;  // stop when 0
}

有两点设计决策值得关注：

  • Prefab 是 Entity，而非 GameObject。Baker 会将 GameObject Prefab 转换为预制体 Entity；运行时只看到 Entity。
  • TimeLeft 和 RemainingCount 存储在 Spawner Entity 上，使 System 保持无状态。

3. Authoring + Baker

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float      Interval = 0.5f;
    public int        Count    = 100;

    class Baker : Baker<SpawnerAuthoring>
    {
        public override void Bake(SpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Spawner
            {
                Prefab        = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                SpawnPoint    = authoring.transform.position,
                Interval      = authoring.Interval,
                TimeLeft      = 0f,            // fires immediately on first update
                RemainingCount = authoring.Count
            });
        }
    }
}

将 SpawnerAuthoring 挂载到 SubScene 中的空 GameObject 上，指定 Prefab 并设置数量。关闭 SubScene——Spawner 此后即成为一个 Entity。

4. 运行时 System

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnerSystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<Spawner>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Use the EndSimulation ECB so structural changes flush at a safe point.
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged);

        foreach (var (spawnerRW, spawnerEntity) in
                 SystemAPI.Query<RefRW<Spawner>>().WithEntityAccess())
        {
            ref var spawner = ref spawnerRW.ValueRW;

            if (spawner.RemainingCount <= 0)
            {
                ecb.DestroyEntity(spawnerEntity);   // done — remove the spawner
                continue;
            }

            spawner.TimeLeft -= dt;
            if (spawner.TimeLeft > 0f) continue;

            spawner.TimeLeft = spawner.Interval;
            spawner.RemainingCount--;

            var instance = ecb.Instantiate(spawner.Prefab);
            ecb.SetComponent(instance, LocalTransform.FromPosition(spawner.SpawnPoint));
        }
    }
}

要点：

  • RequireForUpdate<Spawner>() 使 System 在没有 Spawner 时完全跳过。
  • EndSimulationEntityCommandBufferSystem Singleton 提供一个共享 ECB，在帧末刷新——比在循环中途通过 EntityManager 直接修改更安全。
  • ecb.Instantiate(prefab) 创建一个延迟 Entity；ecb.SetComponent(instance, …) 调度对它的 Component 写入。instance 句柄仅在此 ECB 的回放期间有效。参见 14_EntityCommandBuffer · Deferred Entity.md。

5. 规模扩展——并行 Spawner

如需每帧从大量 Spawner 生成数千个 Entity，请转换为带并行 ECB Writer 的 IJobEntity：

[BurstCompile]
public partial struct SpawnJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, entity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

在 System 中：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

new SpawnJob { DeltaTime = SystemAPI.Time.DeltaTime, ECB = ecb }
    .ScheduleParallel();

chunkIndex 排序键是并行回放保持确定性的关键。参见 15_ParallelWriter · Deterministic Playback.md。

6. 故障排查

症状 原因 / 解决方法
什么都没有生成 Spawner Entity 不存在——打开 SubScene，确认 SpawnerAuthoring 已挂载且 SubScene 已关闭（黄色图标）。
Prefab 在世界原点生成 ECB 从未设置 LocalTransform。请添加 ecb.SetComponent(instance, LocalTransform.FromPosition(...))，或确保 Prefab 已有所需位置的 Transform。
Prefab 可见但不移动 Prefab Bake 时使用了 TransformUsageFlags.Renderable 而非 Dynamic。
SystemAPI.GetSingleton<...ECBS.Singleton>() 抛出异常 ECB System 未注册——发生在精简的 World 中。回退到 state.EntityManager + 手动 ECB，或在 Bootstrap 中注册 ECB System。
Spawner 永不停止 RemainingCount 在缓存的 Spawner 副本上递减，而非通过 spawnerRW.ValueRW。请务必通过 RefRW.ValueRW 写入。
ecb.SetComponent 对延迟实例无效 Instantiate 和 SetComponent 使用了不同的 ECB。两次调用必须共享同一 ECB 实例。

Chapter 6. ECS 核心概念

1. 五个核心名词

Entities ECS 可归纳为五个核心名词：

Figure: ECS Core Nouns.

理解这张图，是读懂现有 DOTS 代码的大半功夫。

2. Entity

Entity 只是一个 64 位句柄：struct Entity { int Index; int Version; }。它本身没有数据和方法，仅具备身份标识——所有数据都存储在 Component 中。

Entity e = state.EntityManager.CreateEntity();   // fresh, no components
state.EntityManager.AddComponent<Health>(e);     // now it has data
state.EntityManager.DestroyEntity(e);            // handle becomes stale

Version 字段会使旧引用失效：若在相同 Index 处销毁并重建 Entity，旧的 Entity 值将不再匹配。

3. Component

Component 是纯数据容器，不承载任何行为。逻辑代码存放于 System 中。

public struct Health : IComponentData
{
    public float Value;
    public float Max;
}

Component 有多种变体（非托管、托管、Buffer、共享、清理、Tag、Chunk、可启用）。参见 05_Component Types.md。

4. Archetype

Archetype 是若干 Entity 共有的唯一 Component 类型集合。

  • 拥有 { LocalTransform, Health, Enemy } 的 Entity 属于 Archetype A。
  • 拥有 { LocalTransform, Health, Enemy, Burning } 的 Entity 属于 Archetype B——因为添加 Burning 改变了 Component 集合，所以是不同的 Archetype。

在 Archetype 之间移动（添加/移除 Component）是一次结构性变更，Entity 会被物理移动到新的 Chunk 中。参见 13_Structural Change & Safety.md。

5. Chunk

同一 Archetype 的 Entity 被紧密打包进 16 KB 的 Chunk 中。在 Chunk 内部，每种 Component 类型形成一个连续数组——极利于缓存局部性。

事实 影响
Chunk 大小为 16 KB 每个 Component 占用一段空间；Archetype 中 Component 越多（或越大），每个 Chunk 能容纳的 Entity 就越少。
数据按 Component 列式（SoA）存储 IJobChunk 可逐个迭代 Component 数组，并通过 Burst 进行向量化处理。
添加/移除 Component 会移动 Entity 这是结构性变更。Query 迭代器假设迭代期间 Archetype 保持稳定。
Chunk 跟踪 Component 变更版本 System 可跳过自上次运行以来未发生变更的 Chunk（EntityQuery.SetChangedVersionFilter）。

您可在 Window → Entities → Archetypes 中查看实时 Chunk 布局。

6. System

System 是每帧运行的逻辑代码，通常对一组 Entity 的 Query 进行处理。

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var health in SystemAPI.Query<RefRW<Health>>())
        {
            health.ValueRW.Value = math.min(health.ValueRW.Value + dt, health.ValueRW.Max);
        }
    }
}

System 会自动注册并存在于 SystemGroup 中。参见 08_System — ISystem vs SystemBase.md 和 09_System Group & Update Order.md。

7. World

World 是一个容器，拥有：

  • 一个 EntityManager（创建、销毁、查询 Entity）。
  • 一组 System 及其 SystemGroup。
  • Chunk 本身（通过 EntityManager）。

默认运行时 World 为 DefaultGameObjectInjectionWorld。大多数游戏在生产环境中只有一个 World；多 World 设置出现在 Netcode（服务器 / 客户端）或 Editor 工具中。参见 16_Netcode Client-Server World & Bootstrap.md，了解 Netcode 的 World 拆分。

World world = World.DefaultGameObjectInjectionWorld;
EntityManager em = world.EntityManager;

Entity e = em.CreateEntity(typeof(Health));
em.SetComponentData(e, new Health { Value = 100, Max = 100 });

8. EntityManager

EntityManager 是主线程的结构性变更 API：创建、销毁、添加/移除 Component、实例化预制体 Entity、复制 World 等。通过 EntityManager 进行的结构性变更是即时的，可能使任何正在进行的 Query 迭代器或 Job 引用失效。

对于来自 Job 或迭代中途的延迟结构性变更，请使用 EntityCommandBuffer——参见 14_EntityCommandBuffer · Deferred Entity.md。

9. 整体结构

World
└── EntityManager
    └── Archetype { LocalTransform, Health, Enemy }
        └── Chunk  (up to ~N entities)
            └── Entity { Index, Version }
                ├── LocalTransform
                ├── Health
                └── Enemy (tag)

查询 { LocalTransform, Health } 的 System 会访问所有 Archetype 同时包含这两者的 Chunk。这包括上述 Archetype——以及任何同时包含 Enemy 和其他 Component 的 Archetype。

10. 故障排查

症状 原因 / 解决方法
Entity 查找返回过时数据 该 Entity 已被销毁并重建——Version 不再匹配。重新解析该 Entity。
Query 无匹配，但 Entity 明明有该 Component Component 是在不同 World 中添加的，或该 Component 是可启用的且当前处于禁用状态。
Archetype 数量激增 唯一 Component 组合过多（常见于高基数的按 Entity Tag）。请改用共享 Component 或可启用标志进行整合。
Chunk 中只有 1-2 个 Entity Archetype 过大——移除 Entity 不需要的 Component，或拆分为多个 Entity/Archetype。
迭代时出现 InvalidOperationException: … invalidated … Query 迭代中途发生了结构性变更。请使用 ECB，或在循环结束后批量处理变更。

Chapter 7. Entity 引用 — Entity · EntityPrefabReference · UnityObjectRef

1. 概述

DOTS 代码使用一小组引用类型。它们很容易混淆，因为都指向"某个对象"，但它们处于从 Authoring 到运行时流程的不同阶段。

类型 命名空间 引用对象 典型用途
Entity Unity.Entities 一个 ECS World 内部的 Entity。 运行时关系和 Component 字段。
EntityPrefabReference Unity.Entities.Serialization 可稍后加载的已 Bake Entity 预制体资产。 流式传输和内容工作流。
UnityObjectRef<T> Unity.Entities 通过非托管包装器访问的 UnityEngine.Object 资产/对象。 在 ECS 数据中引用网格、材质、纹理、音频剪辑和已 Author 资产。

本节仅介绍 DOTS / Entities 引用类型。引擎级 UnityEngine.Object 标识符 API 不在本手册范围内。

2. Entity

Entity 是运行时 ECS 句柄。它是一个非托管值，用于标识一个 World 内部的 Component 数据行。

using Unity.Entities;

public struct Target : IComponentData
{
    public Entity Value;
}

重要规则：

  • Entity 值只在创建它的 World 内有意义。
  • 销毁 Entity 会使旧句柄失效。
  • Entity 索引可被复用；在解引用存储的句柄前，若并非刚刚查询所得，请先使用 EntityManager.Exists(entity) 检查。
  • 不要将原始 Entity 值保存到磁盘，或将其作为稳定 ID 跨独立 World 传递。
if (state.EntityManager.Exists(target.Value))
{
    var transform = state.EntityManager.GetComponentData<LocalTransform>(target.Value);
}

3. Baking 中的 Entity 引用

当 Baker 将 Authoring 对象转换为 ECS 数据时，使用 GetEntity 将被引用的 Authoring 对象或 Prefab 转换为 Entity 引用。

using Unity.Entities;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
}

public struct Spawner : IComponentData
{
    public Entity Prefab;
}

public class SpawnerBaker : Baker<SpawnerAuthoring>
{
    public override void Bake(SpawnerAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new Spawner
        {
            Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic)
        });
    }
}

结果是一个运行时 ECS 引用，而非托管的 GameObject 引用。

4. EntityPrefabReference

EntityPrefabReference 用于应通过 Entities 内容管线加载的 Prefab 内容，而非直接嵌入当前场景作为 Entity 引用。

using Unity.Entities;
using Unity.Entities.Serialization;
using UnityEngine;

public class ProjectileLibraryAuthoring : MonoBehaviour
{
    public GameObject ProjectilePrefab;
}

public struct ProjectileLibrary : IComponentData
{
    public EntityPrefabReference ProjectilePrefab;
}

public class ProjectileLibraryBaker : Baker<ProjectileLibraryAuthoring>
{
    public override void Bake(ProjectileLibraryAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new ProjectileLibrary
        {
            ProjectilePrefab = new EntityPrefabReference(authoring.ProjectilePrefab)
        });
    }
}

当 Prefab 属于内容管理或流式传输工作流时，使用 EntityPrefabReference。当 Prefab 已 Bake 到同一 SubScene 且随场景一并可用时，使用普通的 Entity 预制体引用。

5. UnityObjectRef<T>

UnityObjectRef<T> 允许非托管 ECS 数据引用 UnityEngine.Object，而无需将 Component 变为托管 Component。

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

从 Authoring 数据 Bake：

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

仅在主线程或托管的表现层代码中解引用 .Value：

Mesh mesh = visual.Mesh.Value;
Material material = visual.Material.Value;

规则：

  • 存储的包装器是非托管的，可以存放在 IComponentData 中。
  • 目标对象仍是托管的 Unity 对象；.Value 不能在 Burst Job 中使用。
  • 解引用时，被引用的对象必须仍处于加载状态。

6. 选择正确的引用类型

需求 使用
在同一 World 中从一个 Entity 指向另一个 Entity
存储已 Bake 到同一 SubScene 的 Prefab 通过 GetEntity(authoring.Prefab, ...) 得到的 Entity 预制体引用
存储用于内容流式传输 / 延迟加载的 Prefab EntityPrefabReference
在非托管 ECS 数据中存储网格、材质、纹理、音频剪辑或 ScriptableObject 引用 UnityObjectRef<T>
跨会话持久化 Entity 游戏自有的稳定 ID，而非原始 Entity

7. 故障排查

症状 原因 / 解决方法
存储的 Entity 不再存在 该 Entity 已被销毁或属于其他 World。重新解析它，或检查 EntityManager.Exists。
Baking 后预制体引用为 Entity.Null Baker 未对该 Prefab 调用 GetEntity，或该 Prefab 未包含在已 Bake 的场景/内容工作流中。
UnityObjectRef<T>.Value 返回 null 被引用的 Unity 对象未加载或已被销毁。对解引用添加守卫判断，并重新加载/解析资产。
Burst Job 无法使用网格/材质引用 UnityObjectRef<T> 的存储是非托管的，但 .Value 会访问托管的 Unity 对象。请将解引用移至主线程表现层代码中。
存档文件中存储了原始 Entity 值 原始 Entity 句柄是 World 本地的，不具备稳定性。请改为存储游戏自有 ID 或内容键。

Chapter 8. Component 类型

1. 概述

ECS 中的"Component"是附加在 Entity 上的任意数据容器。Entities 提供了多种风格，具有不同的存储方式、生命周期和性能开销。选择正确的类型是设计 ECS 系统工作量的一半。

快速参考表：

接口 / 形式 存储位置 修改时触发结构性变更？ 典型用途
IComponentData（非托管 struct） Chunk 内，SoA 否（值变更时） 默认 Component
IComponentData（托管 class） 附属表 否 引用类型（网格、GameObject 引用）
IBufferElementData Chunk 内 + 堆溢出 否 每 Entity 的动态数组
ISharedComponentData 每 Chunk 是（将 Entity 移至新 Chunk） 将共享配置的 Entity 分组
ICleanupComponentData Chunk 内 正常 Entity 销毁清理协议
ICleanupSharedComponentData 每 Chunk 正常 共享资源清理协议
Tag（空 IComponentData） Chunk 内零尺寸 否 过滤标记，无数据
Chunk Component 每 Chunk 是 每 Chunk 计算一次的元数据
可启用（IEnableableComponent） Chunk 内 + 位掩码 否 按 Entity 开关行为

本节其余部分将逐行展开说明。

2. 非托管 IComponentData

默认类型。以紧密排列的 SoA 数据直接存储在 Chunk 中。Burst 友好。

public struct Velocity : IComponentData
{
    public float3 Value;
}

规则：

  • 必须是 struct——不得包含引用字段（无 class、无 string、无 List<T>）。
  • 允许的字段类型：基本类型、Unity.Mathematics、其他非托管 struct、Entity、FixedString*、BlobAssetReference<T>、UnityObjectRef<T>。
  • 通过 SystemAPI.Query 的 RefRO<T> 读取，通过 RefRW<T> 写入。

3. 托管 IComponentData

存储在按 Entity 索引的附属表中。可使用引用类型，但会产生 GC 分配，且该 Component 不兼容 Burst。

public class EnemyAI : IComponentData
{
    public AIModel Brain;      // a managed class
    public List<Entity> Allies;
}

当确实需要托管引用（网格、纹理、已 Author 的 ScriptableObject）时使用。若只需对 UnityEngine.Object 的弱引用，请优先在非托管 Component 中使用 UnityObjectRef<T>。

4. IBufferElementData——动态 Buffer

DynamicBuffer<T> 是每 Entity 的可调整大小数组。

public struct Waypoint : IBufferElementData
{
    public float3 Position;
}

var buffer = state.EntityManager.AddBuffer<Waypoint>(entity);
buffer.Add(new Waypoint { Position = new float3(1, 0, 0) });

存储：最多 InternalBufferCapacity 个元素存储在 Chunk 内；超过后 Buffer 溢出至堆上。通过 [InternalBufferCapacity(N)] 调整容量。

5. ISharedComponentData——分组

共享 Component 携带一个对 Chunk 进行分组的值。具有相同 ISharedComponentData 值的两个 Entity 会落入同一个 Chunk；不同的值会产生不同的 Chunk。

public struct RenderMeshGroup : ISharedComponentData
{
    public int MaterialID;
}

影响：

  • 修改该值会将 Entity 移至另一个 Chunk → 结构性变更，开销较大。
  • 适合不常变更但按组差异化的数据（材质集、队伍 ID、Bake 区域）。
  • 若类型包含引用，值存储在托管表中（ISharedComponentData 可以是托管的）。

6. 清理 Component——ICleanupComponentData

为 Entity 打标记，使其"销毁"时仅保留清理 Component 继续存活，直到您的 System 处理完毕。

public struct ConnectionCleanup : ICleanupComponentData
{
    public int ConnectionHandle;
}

// DestroyEntity(e) removes all normal components but leaves ConnectionCleanup
// behind so you can close the socket in a dedicated system, then destroy the
// cleanup component explicitly.

兄弟接口 ICleanupSharedComponentData 将相同协议应用于共享资源。

使用场景：释放原生资源、从外部系统注销、跨帧分阶段销毁。

7. Tag Component

Tag 是一个空的 IComponentData：

public struct Enemy : IComponentData {}

在 Chunk 中占零字节，可像任何其他 Component 一样进行查询。适合作为"属于 X 类型"的标记。过度使用 Tag（数百个唯一 Tag）会碎片化 Archetype——对高基数状态请改用可启用 Component 或共享 Component。

8. Chunk Component

Chunk Component 每个 Chunk 携带一个值，而非每个 Entity。

state.EntityManager.AddChunkComponentData<RegionBounds>(query, new RegionBounds { ... });

典型用途：适用于 Chunk 内所有 Entity 的预计算数据（边界、区域 ID、分区键）。修改 Chunk Component 会引发结构性变更。

9. 可启用 Component（IEnableableComponent）

请参阅专题页面 06_Enableable Component.md 了解更深入的内容。摘要：一种可通过每 Chunk 位掩码逐 Entity 启用/禁用，且不会触发结构性变更的 Component。

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

state.EntityManager.SetComponentEnabled<Stunned>(entity, false);  // no structural change

Query 会自动跳过可启用 Component 当前处于禁用状态的 Entity——这是临时将 Entity 排除在 System 之外的一种廉价且常见的方式。

Chapter 9. 可启用 Component

1. 可启用 Component 存在的原因

添加或移除 Component 是一次结构性变更：Entity 会移动到不同的 Chunk，Query 必须重新解析，引用可能失效。如果每帧对每个 Entity 都执行此操作（例如反复开关"Stunned"），这部分开销会主导整帧。

可启用 Component 提供了一种在不触发结构性变更的情况下开关行为的方式。Component 始终存在于 Chunk 中；每 Chunk 掩码中的一个位决定了从 Query 的角度来看每个 Entity 是否"拥有"它。

2. 接口

同时实现 IComponentData 和 IEnableableComponent：

using Unity.Entities;

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

也可以让 IBufferElementData 实现可启用接口。

3. 切换开关

在 System 中：

// Turn Stunned off without removing the component
state.EntityManager.SetComponentEnabled<Stunned>(entity, false);

// From an ECB (deferred, safe mid-iteration)
ecb.SetComponentEnabled<Stunned>(entity, true);

// From a job with a ComponentLookup
var stunnedLookup = SystemAPI.GetComponentLookup<Stunned>();
stunnedLookup.SetComponentEnabled(entity, false);

读取当前状态：

bool isStunned = state.EntityManager.IsComponentEnabled<Stunned>(entity);

以上三种方式均作用于位掩码，均不触发结构性变更。

4. Query 行为

SystemAPI.Query<...> 和基于 EntityQuery 的迭代会跳过可启用 Component 当前处于禁用状态的 Entity——默认情况下，您只会看到所有列出 Component 均已启用的 Entity。

foreach (var stunned in SystemAPI.Query<RefRW<Stunned>>())
{
    // only fires for entities where Stunned is enabled
}

若需要同时看到已禁用的 Entity，可覆盖此行为：

foreach (var (stunnedRW, entity) in SystemAPI
             .Query<RefRW<Stunned>>()
             .WithEntityAccess()
             .WithOptions(EntityQueryOptions.IgnoreComponentEnabledState))
{
    // fires for ALL entities with the component, enabled or not
}

5. 示例——会过期的眩晕效果

单个 IJobEntity 不能对同一 Component 同时接收 ref Stunned（即 RefRW<T>）和 EnabledRefRW<Stunned>——Entities 明确禁止在一个 Job 中用两种形式包装同一 Component。有两种有效模式：

5.1 读写值，通过 ECB 切换启用位

[BurstCompile]
public partial struct StunTickJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Stunned stunned)
    {
        stunned.Duration -= DeltaTime;
        if (stunned.Duration <= 0f)
            ECB.SetComponentEnabled<Stunned>(chunkIndex, entity, false);
    }
}

回放在下一个 ECB 边界发生（例如 EndSimulationECBS）——参见 14_EntityCommandBuffer · Deferred Entity.md。无结构性变更；仅在已知时间点翻转位掩码。

5.2 不需要 Component 值时直接切换启用位

[BurstCompile]
public partial struct StunDisableJob : IJobEntity
{
    void Execute(EnabledRefRW<Stunned> stunnedEnabled)
    {
        stunnedEnabled.ValueRW = false;   // no structural change
    }
}

该 Job 中对 Stunned 的唯一引用是 EnabledRefRW<T>，因此可以编译通过。当 System 纯粹用于切换开关时，使用这种形式。

希望在一次遍历中既读取值又切换同一 Entity 的启用位？可从主线程通过 EntityManager.SetComponentEnabled<T>(entity, false) 实现，或将逻辑拆分为通过 state.Dependency 串联的两个 Job。

6. 何时选用可启用 Component 而非添加/移除

场景 选择
高频切换（每帧、每事件） 可启用 Component
添加并非始终存在的状态（具有可变字段的按 Entity 修饰符） 可启用 Component
低频 Archetype 变更（例如添加永久能力） 正常添加/移除 Component
统一过滤多个 System 可启用 Component（Query 自动拾取）

通用原则：如果某 Component 的"是否拥有"标志在 Entity 生命周期中多次翻转，请将其设为可启用的。

7. 与其他特性的交互

特性 行为
WithChangeFilter<T> 切换启用位同样会更新 Component 的变更版本。
ICleanupComponentData 清理 Component 不能同时是可启用的。
Netcode for Entities 可启用 Component 可以进行 Ghost 同步，但切换操作必须通过权威端进行。
序列化 / World 导出 启用位会被保留。

8. 故障排查

症状 原因 / 解决方法
Query 包含了本应"关闭"的 Entity Component 未实现 IEnableableComponent，或设置了 WithOptions(EntityQueryOptions.IgnoreComponentEnabledState)。
SetComponentEnabled<T> 抛出"类型 T 不支持启用" 该 struct 仅实现了 IComponentData——请添加 IEnableableComponent。
EnabledRefRW<T> 参数被源码生成器拒绝 Job 或 System 缺少 partial，或该 Component 不是可启用的。
源码生成器报错"同一 Component 不能同时有 RefRW 和 EnabledRefRW" 拆分逻辑——每个 Job 中对同一 Component 只使用 ref T 或 EnabledRefRW<T>，不能并存。参见 §5 的两种模式。
变更过滤器意外触发 切换 Enabled 会作为 WithChangeFilter 的变更被记录，这是预期行为——结合 EnabledRefRO 读取，而非反复切换。
ECB 的 SetComponentEnabled 被忽略 在创建该延迟 Entity 的 ECB 刷新之前就对其调用了此方法。两次操作必须在同一个 ECB 周期内。

Chapter 10. Singleton Component

1. Singleton 是什么（以及不是什么）

Entities 中的 Singleton 只是一个预期在 World 中恰好只有一个 Entity 拥有的 Component。没有专门的 ISingleton 接口——任何 IComponentData 按约定即可成为 Singleton，SystemAPI 提供了符合人体工程学的访问器，并断言"恰好一个"规则。

典型用途：游戏状态、全局配置、时间/Tick 源、网络状态、中央 ECB 注册。

public struct GameState : IComponentData
{
    public int   Score;
    public float TimeScale;
    public bool  Paused;
}

2. 访问器

在任意 ISystem / SystemBase 中：

// Reads
GameState gs      = SystemAPI.GetSingleton<GameState>();
RefRW<GameState> gsRW  = SystemAPI.GetSingletonRW<GameState>();
bool has          = SystemAPI.HasSingleton<GameState>();
bool got          = SystemAPI.TryGetSingleton<GameState>(out var gs2);

// Writes
SystemAPI.SetSingleton(new GameState { Score = 10 });

// Entity handle, if you need it
Entity e          = SystemAPI.GetSingletonEntity<GameState>();

// Buffers work too
DynamicBuffer<Waypoint> buf = SystemAPI.GetSingletonBuffer<Waypoint>();

Get* 变体在 Singleton 数量为零或超过一个时抛出异常。TryGet* 在这种情况下返回 false。RW 变体会将 Component 标记为已变更，用于变更跟踪。

3. 创建 Singleton

两种常见模式：

3.1 从 SubScene Bake

public class GameStateAuthoring : MonoBehaviour
{
    public float InitialTimeScale = 1f;

    class Baker : Baker<GameStateAuthoring>
    {
        public override void Bake(GameStateAuthoring authoring)
        {
            var e = GetEntity(TransformUsageFlags.None);
            AddComponent(e, new GameState
            {
                Score     = 0,
                TimeScale = authoring.InitialTimeScale,
                Paused    = false
            });
        }
    }
}

将 GameStateAuthoring 挂载到 SubScene 中的单个 GameObject 上；Baker 会生成恰好一个带 Singleton 的 Entity。

3.2 在 System 启动时创建

public void OnCreate(ref SystemState state)
{
    var e = state.EntityManager.CreateEntity(typeof(GameState));
    state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
}

适用于不应存在于 Authoring 中的数据（服务器注入配置、程序化生成数据等）。

4. 基于 Singleton 对 System 进行门控

若 System 仅在 Singleton 存在后才有意义：

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<GameState>();
}

RequireForUpdate<T>() 指示 System 在 Component 不存在时完全跳过 OnUpdate。避免在 OnUpdate 内部进行防御性空值检查。

5. 只读与读写性能

  • GetSingleton<T>()——读取副本，无变更跟踪。
  • GetSingletonRW<T>()——返回指向 Chunk 内部的 RefRW<T>。赋值 .ValueRW 会标记 Component 已变更，适用于其他 System 使用 WithChangeFilter<T> 的场景。
  • SetSingleton——完整写入，始终更新版本号。

在以读为主的 System 中优先使用 GetSingleton；仅在实际需要变更时切换到 GetSingletonRW。

6. 多个候选——"Singleton 超过一个"的陷阱

当两个 Entity 最终携带同一 Singleton Component 时，GetSingleton<T> 会抛出 InvalidOperationException: expected exactly one entity with component T, found N。常见原因：

  • SubScene 中两个 GameObject 都挂载了 Authoring 脚本。
  • 测试夹具在运行时创建了 Singleton，同时 SubScene 也对其进行了 Bake。
  • System 在 OnCreate 中创建 Singleton，但未通过 HasSingleton 进行守卫判断。

防御性写法：

public void OnCreate(ref SystemState state)
{
    if (!SystemAPI.HasSingleton<GameState>())
    {
        var e = state.EntityManager.CreateEntity(typeof(GameState));
        state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
    }
}

7. Buffer Singleton

DynamicBuffer<T> 也可以是 Singleton：

public struct QueuedEvent : IBufferElementData
{
    public int Type;
    public int Payload;
}

// Access:
DynamicBuffer<QueuedEvent> events = SystemAPI.GetSingletonBuffer<QueuedEvent>();
events.Add(new QueuedEvent { Type = 1 });

适用于全局事件队列和命令流。

8. 故障排查

症状 原因 / 解决方法
InvalidOperationException: expected exactly one entity with component T, found 0 Singleton Entity 从未创建。添加 Baker，或在 OnCreate 中通过 HasSingleton 守卫进行创建。
... found 2 两个 Entity 携带同一 Component。在 Authoring 中去重，或在启动时清理。
GetSingletonRW 似乎未将 Component 标记为已变更 写入了 .ValueRO 或忽略了 RefRW。只有 .ValueRW 才会触发变更跟踪。
System 在 Singleton 创建之前运行 使用 RequireForUpdate<T>，或通过 [UpdateBefore] 排序，使创建者 System 先运行。
Singleton 被两个无排序关系的 System 修改 添加 [UpdateAfter]/[UpdateBefore]，或合并逻辑。Singleton 上的竞争条件尤其难以调试。
Netcode：Singleton 值在客户端和服务器之间出现偏差 对 Singleton 进行 Ghost 同步，或从已同步的输入中确定性地重建它。

Chapter 11. System — ISystem 与 SystemBase

1. 概述

Entities 提供两种声明 System 的方式：ISystem（非托管 struct）和 SystemBase（托管 class）。除非您特别需要 SystemBase 提供的功能，否则请选择 ISystem。

ISystem SystemBase
声明方式 partial struct : ISystem partial class : SystemBase
内存 非托管，可栈分配 托管，GC 所有
可 Burst 编译 是（添加 [BurstCompile]） 否
托管字段 否 是
托管 API 访问 通过 EntityManager + ManagedAPI 直接访问

2. ISystem——默认选择

using Unity.Burst;
using Unity.Entities;

[BurstCompile]
public partial struct EnemyAISystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        // Called once when the system is created.
        state.RequireForUpdate<EnemySpawnConfig>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var config = SystemAPI.GetSingleton<EnemySpawnConfig>();
        // ... logic
    }

    [BurstCompile]
    public void OnDestroy(ref SystemState state)
    {
        // Called once when the system is destroyed.
    }
}

必要要素：

  • Struct 必须是 partial——源码生成器会添加隐藏的包装层。
  • 三个方法（OnCreate、OnUpdate、OnDestroy）均为可选——可省略不需要的方法。
  • 在 struct 上以及在每个需要 Burst 编译的方法上添加 [BurstCompile]。

3. SystemBase——托管 System 不可避免时

using Unity.Entities;

public partial class PlayerInputSystem : SystemBase
{
    private InputActions _input;   // managed field — the reason SystemBase exists

    protected override void OnCreate()
    {
        _input = new InputActions();
        _input.Enable();
    }

    protected override void OnUpdate()
    {
        var move = _input.Gameplay.Move.ReadValue<Vector2>();
        // ... dispatch to components
    }

    protected override void OnDestroy() => _input.Dispose();
}

在以下情况使用 SystemBase：

  • 需要持有一个必须与 System 共存的托管字段（Input System 对象、UI Toolkit 句柄、托管服务等）。
  • 需要在 OnUpdate 中直接调用托管 API，而无需通过 EntityManager 进行封装。

SystemBase 中的所有内容都在主线程上运行，不使用 Burst。

4. System 生命周期

两种形式共享相同的回调名称：

Figure: System Lifecycle.

回调 触发时机 典型用途
OnCreate System 创建时，调用一次 缓存 Query、RequireForUpdate<T>、分配原生容器
OnStartRunning Query 从空变为非空 重置每会话状态
OnUpdate 在父 SystemGroup 中每帧调用 主要工作
OnStopRunning Query 从非空变为空 刷新缓冲状态
OnDestroy World 拆除时，调用一次 释放原生/托管资源

RequireForUpdate<T>()

当所需 Component 或 Singleton 不存在时，完全跳过 OnUpdate：

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<EnemySpawnConfig>();
    state.RequireForUpdate<GameRunning>();
}

这是将 System 限制在"配置已存在"/"游戏已开始"条件下运行的标准方式。

5. 在两者之间做选择

经验法则：

  1. 从 ISystem 开始。 Burst 友好，无 GC。
  2. 仅在托管字段迫使您这样做时切换到 SystemBase。 即便如此，也应考虑拆分：将托管工作放在轻量 SystemBase 中并将结果推送到 Component 中，再用一个读取这些 Component 的 ISystem 运行繁重逻辑。
  3. 不要混用——不要向 ISystem struct 添加托管字段，这无法编译。上述拆分方式是惯用的变通方案。

拆分示例：

// Managed — bridges Input System into components.
public partial class InputBridgeSystem : SystemBase
{
    private InputActions _input;
    // ... fills a PlayerInput singleton each frame
}

// Unmanaged + Burst — reads the PlayerInput singleton and drives gameplay.
[BurstCompile]
public partial struct PlayerMoveSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) { /* ... */ }
}

6. SystemState 快速参考

在 ISystem 中，ref SystemState state 是您访问 World 的窗口。

成员 用途
state.EntityManager 主线程结构性变更。
state.World 所属 World。
state.Dependency 当前 Job 依赖句柄——与调度的 Job 合并使用。
state.GetEntityQuery(...) 构建或获取缓存的 Query。
state.RequireForUpdate<T>() 基于 Component 的存在性对 System 进行门控。
state.Enabled 设为 false 可跳过 System 而不销毁它。

在 SystemBase 中，这些是类本身的成员（例如 this.EntityManager）。

1. 概述

每个 System 都存在于一个 SystemGroup 中。Group 决定其子项每帧的运行时机，排序特性决定 Group 内部的执行顺序。正确的分组方式是避免帧延迟写入、竞争条件以及"为什么这个 Entity 会抖动"类型 Bug 的关键。

2. 内置 Group

BeginSimulationEntityCommandBufferSystem Simulation 最开始 有序 ECB 回放（早期）
EndSimulationEntityCommandBufferSystem Simulation 最末尾 有序 ECB 回放（晚期）

Netcode for Entities 还添加了更多 Group（PredictedSimulationSystemGroup、GhostSendSystem 等）——它们嵌套在上述 Group 内部。参见 16_Netcode Client-Server World & Bootstrap.md 和 19_Netcode Prediction & Rollback.md。

3. 将 System 放入 Group

using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial struct EnemyAISystem : ISystem { /* ... */ }

若省略 [UpdateInGroup]，System 默认归属于 SimulationSystemGroup。

嵌套

Group 可以嵌套在其他 Group 中：

[UpdateInGroup(typeof(SimulationSystemGroup), OrderFirst = true)]
public partial class GameplaySystemGroup : ComponentSystemGroup { }

[UpdateInGroup(typeof(GameplaySystemGroup))]
public partial struct MovementSystem : ISystem { /* ... */ }

ComponentSystemGroup 是自定义 Group 的基类。它们同时充当 System 和容器。

4. Group 内部排序

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateAfter(typeof(EnemyAISystem))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct EnemyPathfindingSystem : ISystem { /* ... */ }
特性 含义
[UpdateBefore(typeof(X))] 在 X 之前运行（两者必须在同一 Group 中）。
[UpdateAfter(typeof(X))] 在 X 之后运行。
OrderFirst = true（在 [UpdateInGroup] 上） 尽可能在 Group 中最早运行。
OrderLast = true 尽可能在 Group 中最晚运行。

提示：优先使用 [UpdateAfter] 链而非 OrderFirst/Last——代码审查时更易阅读，且在重构后仍能保持正确。

5. [RequireMatchingQueriesForUpdate]

这是一个较新的特性（Entities 1.x+），它指示 System 在每个 Query 至少有一个 Entity 匹配之前跳过 OnUpdate：

[RequireMatchingQueriesForUpdate]
public partial struct EnemyAISystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Only runs when there is at least one entity matching every query the system uses.
    }
}

与 state.RequireForUpdate<T>() 相比，后者要求特定 Component/Singleton 存在。[RequireMatchingQueriesForUpdate] 更隐式——它以 System 最终拥有的所有 Query 为基准。

6. 自定义 Group

创建自定义 Group 的理由：

  • 内聚的游戏阶段（例如"在动画之前应用伤害"）。
  • 不同的 Tick 频率（例如 30 Hz 的自定义固定步）。
  • 功能开关——整体启用/禁用该 Group。
using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial class DamagePipelineGroup : ComponentSystemGroup
{
    // You can override OnUpdate to customise how the group ticks.
}

[UpdateInGroup(typeof(DamagePipelineGroup), OrderFirst = true)]
public partial struct ApplyDamageSystem : ISystem { /* ... */ }

[UpdateInGroup(typeof(DamagePipelineGroup))]
[UpdateAfter(typeof(ApplyDamageSystem))]
public partial struct DeathSystem : ISystem { /* ... */ }

自定义 Tick 频率

public partial class SlowTickGroup : ComponentSystemGroup
{
    private float _accum;
    protected override void OnUpdate()
    {
        _accum += (float)SystemAPI.Time.DeltaTime;
        if (_accum < 0.1f) return;      // 10 Hz
        _accum = 0f;
        base.OnUpdate();                // tick children
    }
}

若确定性至关重要（物理、网络预测、确定性回放），请将 System 放入 FixedStepSimulationSystemGroup：

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial struct PhysicsStepSystem : ISystem { /* ... */ }

注意事项：

  • FixedStepSimulationSystemGroup 每帧可能运行多次（或零次）以追赶真实时间。
  • 使用 SystemAPI.Time.DeltaTime——在该 Group 内部它是固定步长的 dt。
  • 任何读取输入或产生表现层输出的内容应保留在 SimulationSystemGroup / PresentationSystemGroup 中。

症状 原因 / 解决方法

Chapter 13. Job System 与 Burst

1. 概述

DOTS 的性能依赖于两项在 Entities 之前就已存在、至今仍是其基础的引擎特性：

  • Job System——通过依赖图将工作调度到工作线程上。
  • Burst Compiler——将 C# Job 编译为紧凑的原生代码（SIMD、无 GC、无虚分发）。

这两者都不是 Entities 特有的概念。IJobEntity 和 IJobChunk 是 IJob + Burst 之上的薄层封装。理解基础层能让所有高层模式一目了然。

2. Job System——核心类型

接口 形式 适用场景
IJob void Execute() 一次性的单线程工作。
IJobParallelFor void Execute(int index) 从 0 到 N 并行循环。
IJobParallelForBatch void Execute(int start, int count) 并行批次循环（比按索引缓存更友好）。
IJobChunk void Execute(ArchetypeChunk chunk, int chunkIndex, bool useEnabledMask, in v128 enabledMask) 并行迭代 ECS Chunk。
IJobEntity void Execute([component params]) ECS Entity 迭代——源码生成后转化为 Chunk Job。

以上所有接口均为 struct，均支持 Burst。

IJobParallelFor 最简示例

using Unity.Burst;
using Unity.Collections;
using Unity.Jobs;

[BurstCompile]
struct ScaleJob : IJobParallelFor
{
    public NativeArray<float> Data;
    public float Factor;

    public void Execute(int index) => Data[index] *= Factor;
}

// Scheduling:
var handle = new ScaleJob { Data = arr, Factor = 2f }.Schedule(arr.Length, 64);
handle.Complete();

注意事项：

  • 字段必须可 Blit。
  • 多个 Job 对同一 NativeArray 的写访问需要依赖链（见下文）。

3. 调度与依赖

JobHandle a = jobA.Schedule();
JobHandle b = jobB.Schedule(a);              // b runs after a
JobHandle c = jobC.Schedule(JobHandle.CombineDependencies(a, b));
c.Complete();

规则：

  • 每次 Schedule() 都返回一个 JobHandle。若新 Job 读写共享数据，请将前一个句柄作为依赖传入。
  • 在主线程读取结果之前，务必对最终句柄调用 .Complete()（安全系统会强制执行此规则）。
  • 在 ISystem 中，state.Dependency 是系统级句柄；使用 .Schedule(state.Dependency) 或依赖源码生成辅助方法（如 IJobEntity.ScheduleParallel(...)）调度的 Job 会自动与其合并。

在 ISystem 内部

[BurstCompile]
public void OnUpdate(ref SystemState state)
{
    var job = new MyJob { Dt = SystemAPI.Time.DeltaTime };
    state.Dependency = job.ScheduleParallel(state.Dependency);
}

不带显式依赖的 IJobEntity.ScheduleParallel() 会隐式与 state.Dependency 合并。链式调用多个 Job 时请显式传入依赖。

4. Burst——是什么以及为什么

Burst 通过 LLVM 将 C# 的一个子集（标记了 [BurstCompile] 的方法，无 GC 分配，可 Blit 的数据）编译为高度优化的原生代码。

Burst 编译效果好的内容

  • 标记了 [BurstCompile] 的 struct IJob* / ISystem 方法。
  • 对 Unity.Mathematics 类型（float3、quaternion、int4 等）的数学运算。
  • 对 NativeArray<T>、NativeList<T>、DynamicBuffer<T>、ArchetypeChunk 列数组的紧密循环。

Burst 无法编译的内容

  • 托管引用（class、string、List<T>，任何非非托管的内容）。
  • 异常（可以捕获，但通常应避免）。
  • 通过接口的动态分发（显式泛型实例化是允许的）。
  • System.IO / Reflection / 基础类库中数学/基本类型之外的大部分内容。

启用 Burst

using Unity.Burst;

[BurstCompile]
public partial struct MyJob : IJobEntity
{
    // attribute on the struct marks the type for Burst
    // methods need [BurstCompile] too if you want each compiled
    [BurstCompile]
    void Execute(ref Velocity v) { /* ... */ }
}

对于 System：

[BurstCompile]
public partial struct MySystem : ISystem
{
    [BurstCompile] public void OnCreate(ref SystemState state) { }
    [BurstCompile] public void OnUpdate(ref SystemState state) { }
}

类型上的特性不够——每个需要编译的方法也需要 [BurstCompile]。

Burst Inspector

Window → Analysis → Burst Inspector 显示每个方法的 LLVM IR、汇编代码和向量使用情况。当"为什么这段代码不快"时，使用它来排查。

5. 原生容器

Burst 和安全系统期望使用原生容器而非托管集合：

托管类型 原生等效类型
T[] NativeArray<T>
List<T> NativeList<T>
Dictionary<K,V> NativeHashMap<K,V>、NativeParallelHashMap<K,V>
HashSet<T> NativeHashSet<T>、NativeParallelHashSet<T>
Queue<T> NativeQueue<T>
string FixedString32Bytes / FixedString64Bytes / 等

分配：

var arr = new NativeArray<int>(1024, Allocator.TempJob);
// ...
arr.Dispose();

生命周期至关重要：Burst + 安全系统会验证容器已被释放，且不会在 Job 生命周期结束后被访问。

6. 分配器

分配器 生命周期 典型用途
Allocator.Temp 当前帧内，仅主线程 OnUpdate 中的临时容器
Allocator.TempJob 跨单个 Job，≤4 帧 传递给 Job 的容器
Allocator.Persistent 需要手动 Dispose() 跨帧存活的 System 持有容器
state.WorldUpdateAllocator System Group 更新结束时清除 System 内每帧瞬态数据的首选

在 System 内部优先使用 state.WorldUpdateAllocator（或其 Rewindable 分配器）——它可避免手动 Dispose() 的管理负担。

7. 安全系统

在 Editor 中开启"Jobs > Leak Detection"和"Jobs > Safety Checks"后，Unity 会验证：

  • 两个 Job 不会在没有依赖边的情况下向同一原生容器写入。
  • 主线程读取不会与正在运行的写入发生竞争。
  • 原生容器不会在 Dispose() 后或超出其分配器生命周期后被访问。

安全错误会以异常形式出现，并附带清晰的"Job A 和 Job B 均访问了 X"消息。开发阶段请勿禁用安全检查——性能提升微乎其微，调试成本却极高。

8. 故障排查

症状 原因 / 解决方法
Burst 警告："method not burst compiled" 方法缺少 [BurstCompile]；或 Burst 发现了不支持的类型（在 Burst Inspector 中检查失败的行）。
InvalidOperationException: … writes to … without declaring dependency 缺少依赖边——传入前一个 JobHandle 或使用 state.Dependency 链式调用。
NativeArray 上出现 ObjectDisposedException 在 Job 完成前已释放。使用 arr.Dispose(handle) 或先等待句柄完成。
已调度的 Job 从未完成 链中缺少 .Complete()，或存在循环依赖。
添加 [BurstCompile] 后性能下降 方法在每次域重载时重新编译；检查 Burst 缓存。或者混入了 Burst 不兼容的类型——方法静默回退到 IL 执行。
NativeHashMap 枚举抛出异常 并发写入时枚举行为不稳定——改为枚举 NativeArray<K> 副本。
主线程等待 Job 时停滞 过早调用了 .Complete()。尽量将完成点延后至帧内尽可能晚的位置。

Chapter 14. IJobEntity · SystemAPI.Query

1. 概述

Entities 6.5 中有两套 API 覆盖了 95% 的 Component 迭代场景：

API 运行位置 适用场景
SystemAPI.Query<...>() 主线程，在 OnUpdate 内部 工作量微小（数百个 Entity），或需要主线程 API。
IJobEntity 工作线程，通过 Schedule / ScheduleParallel 工作量较大或受益于并行处理。

它们是遗留 Entities.ForEach 的推荐替代方案（在 Entities 1.4 中标记为过时；在 6.5 上仍可编译但会产生废弃警告；Unity 更新日志表示将在未来的主要版本中移除）。

2. SystemAPI.Query<T>

在主线程上迭代匹配的 Entity。Component 通过 RefRO<T> / RefRW<T> 访问。

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRW.Max);
        }
    }
}

规则：

  • System struct 必须是 partial。
  • 只读用 RefRO<T>，读写用 RefRW<T>。不要通过 RefRO 写入。
  • 若需要 Entity ID，将 Entity 添加到元组中：SystemAPI.Query<RefRW<Health>>().WithEntityAccess()。

过滤器

foreach (var health in SystemAPI.Query<RefRW<Health>>()
             .WithAll<Enemy>()
             .WithNone<Dead>()
             .WithAny<Burning, Poisoned>())
{
    // only entities that are Enemy AND not Dead AND (Burning OR Poisoned)
}
  • WithAll<T>()——必须拥有所有列出的 Component。
  • WithNone<T>()——不能拥有其中任何一个。
  • WithAny<T...>()——必须至少拥有其中之一。
  • WithChangeFilter<T>()——仅处理自上次运行以来 Component T 发生变更的 Chunk。
  • WithEntityAccess()——同时返回 Entity ID。

3. IJobEntity

源码生成的并行迭代。Execute 的签名声明了您要访问哪些 Component。

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

签名规则：

  • void Execute(...)——参数与 Component 一一对应。
  • ref Component——读写；in Component——只读（不允许写入）。
  • 允许将 Entity entity 作为参数——源码生成器会自动填充。
  • Job struct 必须是 partial。

调度变体

调用 线程模式
.Run() 主线程，返回前完成。适合调试或小型工作量。
.Schedule() 单个工作线程。
.ScheduleParallel() 将 Chunk 分布到多个工作线程上。

以上三者在 ISystem.OnUpdate 内部均会自动更新 state.Dependency。

Job 中的过滤器

new HealthRegenJob { DeltaTime = dt }
    .ScheduleParallel(SystemAPI.QueryBuilder()
        .WithAll<Enemy, Regen>()
        .WithNone<Dead>()
        .Build());

或通过 Job struct 本身上的 [WithAll]/[WithNone]/[WithAny] 特性指定。

Job 中的 Entity 访问

void Execute(Entity entity, ref Health health) { /* ... */ }

Chunk 索引 / [ChunkIndexInQuery]

与 ECB 的 ParallelWriter 配合用于确定性回放时很有用——将 Chunk 索引作为 sortKey 传入。参见 15_ParallelWriter · Deterministic Playback.md。

void Execute([ChunkIndexInQuery] int chunkIndex, in Health health, Entity entity)
{
    if (health.Value <= 0)
        ecb.DestroyEntity(chunkIndex, entity);
}

4. 如何选择

场景 选择
数十到数百个 Entity，运算微小 SystemAPI.Query
数千个以上 Entity，每 Entity 工作相互独立 IJobEntity.ScheduleParallel
每 Chunk 的工作（边界、求和、空间分区） IJobChunk——参见 12_IJobEntity vs IJobChunk.md
循环内部需要托管 API 在 SystemBase 中使用 SystemAPI.Query
需要延迟结构性变更 两种形式均可，但通过 EntityCommandBuffer.ParallelWriter 写入——参见 14_EntityCommandBuffer · Deferred Entity.md

5. 注意事项

  • 不要在 Job 中捕获 this 或托管字段。 源码生成器会报错。
  • RefRW<T>.ValueRW 会触发变更跟踪。 通过 ValueRW 读取但从不写入仍会将 Chunk 标记为脏。
  • WithChangeFilter<T> 是 Chunk 级别的。 如果 Chunk 中任何 Entity 写入了 T，整个 Chunk 都被视为已变更。
  • Aspect 已过时（但未删除）。 IAspect 分组接口在 1.4 中被废弃，在 6.5 的 API 参考中仍被列为有效的 Execute 参数类型。新代码应使用普通 Component 参数；现有 Aspect 在 6.5 上仍可编译。

6. 故障排查

症状 原因 / 解决方法
编译错误：IJobEntity must be partial 为 Job struct 添加 partial。
ScheduleParallel 抛出竞争条件错误 两个 Job 在没有排序边的情况下向同一 Component 写入。确保正确合并 state.Dependency，或在 System 之间添加 [UpdateAfter]。
Job 未在 Burst 下运行 Job struct 缺少 [BurstCompile]，或捕获了 Burst 不兼容的类型（托管引用、未使用 FixedString 的 string）。
SystemAPI.Query 返回空结果 过滤器组合为空。在 Window → Entities → Query 中检查 Query 的实际匹配内容。
通过 RefRO 写入（隐性 Bug） 编译器现在会对 refRO.ValueRW = … 报错。若旧版 source generator 未检出此问题，请改用 RefRW。
SystemAPI.Query 与手动构建的 EntityQuery Entity 数量不一致 过滤器集合不同。SystemAPI.Query 默认包含可启用 Component 过滤；EntityQueryBuilder 需要显式配置。

Chapter 15. IJobEntity 与 IJobChunk

1. 简短结论

  • IJobEntity——"对每个 Entity 执行相同操作"。
  • IJobChunk——"对每个 Chunk 执行并非简单的按 Entity 映射的操作"。

IJobEntity 是在 IJobChunk 之上生成的——它们运行在同一套迭代机制之上。IJobChunk 只是将完整的 Chunk 暴露给您的代码。

2. IJobEntity——以 Entity 为单位的人体工程学

[BurstCompile]
public partial struct DamageTickJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Burn burn)
    {
        health.Value -= burn.DamagePerSecond * DeltaTime;
    }
}

优势：

  • 一个方法，一个 Entity 视图。易于编写，易于阅读。
  • 源码生成的模板代码：Entity、[ChunkIndexInQuery]、[EntityIndexInQuery] 参数由源码生成器自动填充。
  • 通过 struct 上的 [WithAll] / [WithNone] / [WithAny] 特性，或通过传入 .Schedule* 的 QueryBuilder 指定过滤器。

弱点：

  • Execute 内部无法跨 Entity 可见。不易在不借助额外状态的情况下"对本 Chunk 求和"。
  • 在 Execute 内部无法访问 Chunk 级元数据（变更版本、共享 Component 值）。

3. IJobChunk——完整 Chunk 控制

[BurstCompile]
public partial struct HealthAggregateJob : IJobChunk
{
    [ReadOnly] public ComponentTypeHandle<Health> HealthHandle;
    public NativeArray<float> PerChunkTotal;  // size = chunk count

    public void Execute(in ArchetypeChunk chunk,
                        int unfilteredChunkIndex,
                        bool useEnabledMask,
                        in v128 chunkEnabledMask)
    {
        var healths = chunk.GetNativeArray(ref HealthHandle);
        float total = 0f;
        for (int i = 0; i < chunk.Count; i++)
            total += healths[i].Value;

        PerChunkTotal[unfilteredChunkIndex] = total;
    }
}

优势：

  • 直接访问 Chunk 数组：chunk.GetNativeArray(ref handle)、chunk.GetBufferAccessor(ref bufferHandle)。
  • 可通过 chunk.Has<T>()、chunk.GetSharedComponent<T>(...) 进行条件处理。
  • 可见每 Chunk 的启用掩码和变更版本。
  • 非常适合归约、空间分区、批量 ECB 记录。

代价：

  • 更多样板代码：需在 System 中声明 ComponentTypeHandle<T>，并在每帧通过 handle.Update(ref state) 更新它们。
  • 需要自己管理内部 for 循环——因此出错的地方更多。

在 System 中设置句柄

[BurstCompile]
public partial struct HealthAggregateSystem : ISystem
{
    private EntityQuery _query;
    private ComponentTypeHandle<Health> _healthHandle;

    public void OnCreate(ref SystemState state)
    {
        _query        = state.GetEntityQuery(ComponentType.ReadOnly<Health>());
        _healthHandle = state.GetComponentTypeHandle<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthHandle.Update(ref state);

        var totals = CollectionHelper.CreateNativeArray<float>(
            _query.CalculateChunkCount(), state.WorldUpdateAllocator);

        state.Dependency = new HealthAggregateJob
        {
            HealthHandle   = _healthHandle,
            PerChunkTotal  = totals
        }.ScheduleParallel(_query, state.Dependency);
    }
}

handle.Update(ref state) 每帧刷新缓存的类型句柄——遗漏此步骤，Burst 将命中过期索引。

4. 启用掩码（useEnabledMask）

对于包含 IEnableableComponent 的 Query，Execute 会收到 useEnabledMask = true 以及一个 v128 启用位掩码。您必须遵守该掩码，否则会处理到已禁用的 Entity。

辅助工具：

var enumerator = new ChunkEntityEnumerator(useEnabledMask, chunkEnabledMask, chunk.Count);
while (enumerator.NextEntityIndex(out int i))
{
    // i is the index of the next enabled entity in the chunk
}

IJobEntity 会自动为您应用掩码——这是另一个除非需要 Chunk 级可见性否则优先选择它的理由。

Chapter 16. 结构性变更与安全性

1. 什么算作结构性变更

结构性变更是任何可能在 Chunk 之间移动 Entity 或改变 Chunk 清单的操作。以下所有操作均属结构性：

操作 为何属于结构性变更
CreateEntity / DestroyEntity 分配或释放 Chunk 槽位。
AddComponent<T> / RemoveComponent<T> Entity 的 Archetype 变化，因此物理移动到不同的 Chunk。
SetSharedComponent<T> 不同的共享 Component 值 → 不同的 Chunk（按值分组）。
AddChunkComponent<T> / RemoveChunkComponent<T> 改变 Chunk 自身的元数据。
SetName(entity, ...) 访问 EntityManager 中的 Entity 元数据。
MoveEntitiesFrom 在 World 之间复制 Entity。
Instantiate(prefab) 分配新 Entity（可能预填充 Archetype）。

通过 SetComponentData / RefRW<T>.ValueRW 写入 Component 的值并不属于结构性变更——Component 已存在于 Chunk 中；您只是在改变字节内容。

2. 为何结构性变更是危险的

当 Archetype 发生变化时，任何持有 Chunk 内基于位置的引用的对象都会失效：

  • 正在进行的 Query 迭代器（SystemAPI.Query<...>）可能跳过或重复访问 Entity。
  • 受影响 Entity 的 ComponentLookup<T> 条目变为无效。
  • Job 中的 ArchetypeChunk 句柄指向旧 Chunk。
  • 并发运行的其他 System 观察到不一致的状态。

安全系统在 Editor 中会以 InvalidOperationException 的形式捕获大多数此类情况。在构建版本中，它们会变成崩溃或 Entity 损坏。

3. 基本原则

不要在同一 System 更新中迭代 Query 的同时执行结构性变更。

两种合规模式，按优先级排序：

  1. 使用 EntityCommandBuffer。 在迭代期间记录操作，之后回放。参见 14_EntityCommandBuffer · Deferred Entity.md。
  2. 先迭代到 NativeList<Entity>，循环结束后再变更。 对于小规模情况更简单；无 ECB 开销。

模式 2 示例：

var toKill = new NativeList<Entity>(state.WorldUpdateAllocator);

foreach (var (health, e) in SystemAPI
             .Query<RefRO<Health>>()
             .WithEntityAccess())
{
    if (health.ValueRO.Value <= 0f)
        toKill.Add(e);
}

state.EntityManager.DestroyEntity(toKill.AsArray());

4. 可启用 Component——逃生舱口

如果您发现自己每帧都在添加和移除同一个 Component，它几乎肯定应该改为可启用 Component——切换启用位不是结构性变更。参见 06_Enableable Component.md。

5. Job 与结构性变更

结构性变更是主线程操作。不能在 Job 内部执行。

从 Job 进行变更的惯用方式：

方式 适用场景
在 Job 中记录 EntityCommandBuffer.ParallelWriter，在主线程回放 来自并行代码的按 Entity 结构性变更。
在 IJob 或 IJobChunk 中记录单线程 EntityCommandBuffer，在主线程回放 批量 / Chunk 范围的变更。
在 Job 内通过 EnabledRefRW<T> 切换可启用 Component 无需结构的按 Entity"激活/停用"。

ECB 将操作记录为命令；回放在主线程上的已知时间点发生，通常在 EntityCommandBufferSystem 边界处。

6. 结合结构性变更与读取

若两个 System 相继运行，且 System A 执行了 System B 依赖的结构性变更，请明确排序：

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct SpawnSystem : ISystem { /* uses ECB */ }

当 ECB System 的回放位于 A 和 B 之间时（例如配合 [UpdateAfter] 链的 EndSimulationECBS），结构性变更会在 B 读取之前落地。

7. 开销模型

单次变更的粗略排名：

SetComponentData (value only)          ~ 1x
SetComponentEnabled (enableable bit)   ~ 1x (per-chunk bitmask write)
AddComponent (structural)              ~ 10x   (entity moves chunk)
SetSharedComponent (structural)        ~ 10x
DestroyEntity (structural)             ~ 10x
Per-frame add + remove pattern         pathological

数量级估算；精确数字取决于 Archetype 大小、Chunk 占用率和分配器压力。

8. 诊断结构性变更抖动

症状：

  • Window → Entities → Systems 显示某 System 的计时与其逻辑严重不符。
  • Profiler 标记 EntityManager.SetArchetype 或 Move Entity 占据了大量帧时间。
  • Archetype 数量随时间增长。

应对策略：

  • 审查 AddComponent / RemoveComponent 调用——是否有每帧执行的？改用可启用 Component。
  • 检查共享 Component 的值集合——是否存在意外的按 Entity 键？
  • 在 Archetypes 窗口中查找单 Entity 的 Chunk；那是 Component 集合粒度过细导致的碎片化。

9. 故障排查

症状 原因 / 解决方法
InvalidOperationException: EntityCommandBuffer has already been played back 在 ECB 回放后继续向其录制。每个 ECB 是一次性的——每帧通过 ECBS 获取新的 ECB。
InvalidOperationException: … invalidated … 出现在 foreach 中 循环内部发生了结构性变更。改用 ECB 或先缓冲再应用。
Entity 无声无息地消失 执行了 DestroyEntity，但其他 System 仍持有旧的 Entity 句柄。用 EntityManager.Exists 检查句柄。
添加/移除模式占用大量帧时间 改用可启用 Component。
RemoveComponent 触发"存在清理 Component"错误 该 Entity 有清理 Component。先处理清理工作（参见 05_Component Types.md）。
共享 Component 值集合激增（数百个不同值） 值的粒度太细。进行分桶处理（例如四舍五入到最近的 N），或将可变部分存储为常规 Component。

Chapter 17. EntityCommandBuffer · 延迟 Entity

1. ECB 存在的原因

EntityCommandBuffer（ECB）记录结构性变更——创建、销毁、添加、移除、设置——并在主线程的安全时间点稍后回放。它允许您从以下位置变更 World：

  • Query foreach 内部（直接调用 EntityManager 会使迭代器失效的场景）。
  • Job 内部（不允许主线程结构性变更的场景）。
  • 跨多个 Chunk 的并行 Job（通过 ParallelWriter）。

参见 13_Structural Change & Safety.md 了解"为什么这很危险"的背景。

2. ECB 生命周期

Figure: EntityCommandBuffer Lifecycle.

ECB 是一次性的：回放后即销毁。您需要每帧向正确的 EntityCommandBufferSystem 请求新的 ECB。

3. 内置 ECB System

System 回放时机 典型用途
BeginInitializationEntityCommandBufferSystem Initialization Group 开始 上一帧遗留的早期 World 设置
EndInitializationEntityCommandBufferSystem Initialization Group 结束 —
BeginSimulationEntityCommandBufferSystem Simulation 开始 在游戏逻辑运行前应用待处理变更
EndSimulationEntityCommandBufferSystem Simulation 结束 游戏逻辑驱动的生成/销毁的默认选择
BeginPresentationEntityCommandBufferSystem Presentation 开始 少见；仅视觉相关的设置

选择正确的 System 决定了其他 System 何时观察到您的变更。如果您在 SimulationSystemGroup 中生成一颗子弹，并希望它在同一帧移动，请录制到下一帧的 BeginSimulation——或者如果下一帧也可以，就录制到 EndSimulation。

4. 获取 ECB

在任意 ISystem.OnUpdate 内部：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

.Singleton 嵌套类型是源码生成器为每个内置 ECB System 公开的按 System 令牌。CreateCommandBuffer 返回一个新的 ECB，将在该 System 的 Tick 时回放。

对于并行 Job：

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

参见 15_ParallelWriter · Deterministic Playback.md 了解并行回放契约。

5. 录制——Entity 创建

Entity e = ecb.CreateEntity();
ecb.AddComponent(e, new Health { Value = 100 });
ecb.AddComponent(e, LocalTransform.FromPosition(0, 0, 0));

或从预制体 Entity 实例化：

Entity inst = ecb.Instantiate(prefab);
ecb.SetComponent(inst, LocalTransform.FromPosition(spawnPos));

CreateEntity 和 Instantiate 都返回一个延迟 Entity（见 §6）。

6. 延迟 Entity

由 ecb.CreateEntity() 或 ecb.Instantiate() 返回的 Entity 尚不存在。它具有负数的占位索引。您可以：

  • 将其传入同一 ECB 的后续调用——AddComponent、SetComponent、AppendToBuffer。
  • 通过 ecb.SetComponent(other, new ReferenceTo { Entity = inst }) 将其存储在另一个 Component 中。

您不能：

  • 在 ECB 回放之前通过 EntityManager 对其进行读写。
  • 在不同 ECB 之间传递它（占位符只能在所属 ECB 内解析）。

在回放时，ECB System 会在所有使用占位符的地方替换为真实的 Entity 句柄。

7. 录制——结构性变更

ecb.AddComponent(entity, new Stunned { Duration = 2f });
ecb.RemoveComponent<Stunned>(entity);
ecb.SetComponent(entity, new Health { Value = 50 });

ecb.SetComponentEnabled<Stunned>(entity, false);

var buf = ecb.AddBuffer<Waypoint>(entity);
buf.Add(new Waypoint { Position = new float3(1, 0, 0) });
ecb.AppendToBuffer(entity, new Waypoint { Position = new float3(2, 0, 0) });

ecb.DestroyEntity(entity);

以上所有操作均为排队命令；在回放之前均不会生效。

8. 在 IJobEntity 中使用 ECB

[BurstCompile]
public partial struct KillOnHealthZeroJob : IJobEntity
{
    public EntityCommandBuffer ECB;

    void Execute(Entity entity, in Health health)
    {
        if (health.Value <= 0)
            ECB.DestroyEntity(entity);
    }
}

// In OnUpdate:
var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

new KillOnHealthZeroJob { ECB = ecb }.Schedule();

对于 .ScheduleParallel()，请改用 .AsParallelWriter() 并传入 sortKey——下一节将详细介绍。

9. 独立（手动）ECB

有时您需要自行控制回放时机——例如在单个 OnUpdate 内部：

var ecb = new EntityCommandBuffer(Allocator.Temp);
// ... record ...
ecb.Playback(state.EntityManager);
ecb.Dispose();

适用于自包含的一次性变更模式。对于任何跨 System 边界或在 Job 中运行的情况，请使用上述内置 ECBS。

10. 确定性

在单写入者 ECB 上，ECB 命令按录制顺序回放。在 ParallelWriter 上，回放顺序取决于您提供的 sortKey——参见 15_ParallelWriter · Deterministic Playback.md。

对于 Netcode / 回放系统，确定性回放是不可妥协的。始终传入稳定的排序键（Chunk 索引、Query 内 Entity 索引），并避免在以非确定性顺序运行的主线程代码路径中录制。

11. 故障排查

症状 原因 / 解决方法
InvalidOperationException: ECB has already been played back 跨帧复用了 ECB。每帧通过 CreateCommandBuffer 获取新的 ECB。
回放后延迟 Entity 句柄为 Entity.Null 在两个不同的 ECB 中引用了它。创建和后续设置必须使用同一个 ECB。
生成的 Entity 本帧未出现 录制到了 EndSimulation ECBS——回放在帧末进行。改用 BeginSimulation，以便下帧大多数 System 运行前可见。
ecb.SetComponent 抛出"Component 不存在" 目标 Entity 尚未拥有该 Component（先调用 AddComponent）或回放尚未发生。
在 ECBS 管理的 ECB 上调用了 Dispose() 不要对 CreateCommandBuffer 返回的 ECB 调用 Dispose()——ECBS 负责其生命周期。只对独立（new EntityCommandBuffer(Allocator.Temp)）的 ECB 调用 Dispose。
并行 ECB 回放时命令顺序错乱 sortKey 不确定。改用 [ChunkIndexInQuery] 或 Query 提供的稳定索引。

Chapter 18. ParallelWriter · 确定性回放

1. 为何存在独立的并行写入器

EntityCommandBuffer 是单写入者的。两个线程向同一 ECB 录制会在其内部链上产生竞争。EntityCommandBuffer.ParallelWriter 通过为每个调用者提供一个线程本地链并要求每条命令提供一个排序键来解决这个问题。

在回放时，ECB System：

  1. 展平所有线程本地链。
  2. 按 sortKey 排序命令（稳定排序）。
  3. 在主线程上执行它们。

只要您的排序键是确定性的，结果就是确定性的。

2. 获取 ParallelWriter

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

AsParallelWriter() 返回一个 struct EntityCommandBuffer.ParallelWriter，可复制到 IJobEntity、IJobChunk 或 IJobParallelFor 中。

3. sortKey 契约

每次并行 ECB 调用都以 sortKey 作为第一个参数：

ecb.Instantiate(sortKey, prefab);
ecb.SetComponent(sortKey, entity, value);
ecb.DestroyEntity(sortKey, entity);
ecb.AddComponent(sortKey, entity, component);

好的排序键的规则：

  • 确定性。 相同输入 → 相同键，无论调度器如何分配工作。
  • 足够唯一。 对同一 Entity 的两条命令不应产生冲突，除非您希望回放顺序模糊。
  • 廉价。 使用您已经拥有的 int，而非哈希值。

实践中，首选方案为：

来源 获取方式 适用场景
IJobEntity 上的 [ChunkIndexInQuery] 源码生成器自动填充 99% 的情况
IJobChunk.Execute 中的 unfilteredChunkIndex API 提供 IJobChunk Job
IJobEntity 上的 [EntityIndexInQuery] 源码生成器 需要按 Entity 单独排序命令时
手动计算的确定性索引 自行计算 少见；自定义 Job 形式

4. 完整示例——并行 Spawn

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnParallelJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity spawnerEntity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, spawnerEntity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

[BurstCompile]
public partial struct SpawnerParallelSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged)
            .AsParallelWriter();

        new SpawnParallelJob
        {
            DeltaTime = SystemAPI.Time.DeltaTime,
            ECB       = ecb
        }.ScheduleParallel();
    }
}

将 chunkIndex 同时传给 Instantiate 和后续的 SetComponent，可确保两条命令排序到同一位置，从而使 Instantiate 中的延迟 Entity 引用解析到正确的目标。

5. 确定性——保证什么，不保证什么

保证的： 给定相同的 ECB 内容，无论哪个工作线程录制了哪条命令，回放都会产生相同的结果状态。

不自动保证的内容：

  • Entity ID。 如果之前的命令不同，两次运行可能产生具有不同 Index 值的 Entity。
  • 跨 System 的 Chunk 顺序。 如果 System A 并行产生 Entity，而 System B 按 ChunkIndexInQuery 顺序读取它们，迭代顺序仍可能依赖于 Archetype 的创建历史。

如果您需要跨运行的逐位确定性（回放、Netcode 回滚），请组合使用：

  1. 带确定性排序键的并行 ECB。
  2. 在 FixedStepSimulationSystemGroup 内的固定步长 System。
  3. 不依赖 Entity.Index；以您控制的游戏 ID 为键。

6. 排序键模式

6.1 按 Chunk（默认）

使用 [ChunkIndexInQuery] int chunkIndex。如果 Chunk 中的每个 Entity 生成相同的命令（例如整个 Chunk 被标记为眩晕），键冲突是可以接受的。

6.2 按 Entity

当需要严格的按 Entity 排序时（例如记录必须按 Entity 顺序回放的每 Entity 伤害事件），使用 [EntityIndexInQuery] int entityIndex。

6.3 自定义确定性索引

极少数情况下，您从输入自行构建扁平索引——例如 int sortKey = baseOffset + localIndex;。只在常用的两种方式不适用时才这样做；自定义索引是确定性 Bug 的常见来源。

7. 避免以下模式

  • 使用 JobsUtility.ThreadIndex 作为排序键。不具确定性。
  • 使用 Random 构建键。 不具确定性。
  • 混用多个 ECB。 录制到两个不同的 ECB 中会失去总排序。
  • 从多个 System 录制到同一 System 的 ECB。 一个 ECB，一个录制者（或一个包装同一 ECB 的并行写入者）。

8. 故障排查

症状 原因 / 解决方法
延迟 Entity 解析到错误目标 Instantiate 和后续 SetComponent 使用了不同的排序键。保持它们一致。
每次运行命令顺序不同 排序键不具确定性。改用 [ChunkIndexInQuery]。
"JobEntity must have partial"编译错误 为 Job struct 添加 partial。
ECB.SetComponent 在并行 Job 中抛出"线程错误" 在并行 Job 中使用了非并行 EntityCommandBuffer。先调用 .AsParallelWriter() 并使用并行 API。
Netcode 回放出现去同步 某处并行 ECB 未使用确定性排序键，或 IJobEntity 捕获了非确定性值。审查整个调用链。
在已释放的 ECB 上调用 AsParallelWriter() ECB 已被回放——本帧重新获取一个新的。

1. 概述

3. Bootstrap 流程

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

public class TickRateAuthoring : MonoBehaviour
{

    public int NetworkTickRate = 30;

    class Baker : Baker<TickRateAuthoring>
    {

        {

Server World
└─ Fixed timestep, 60 Hz default
   └─ SimulationSystemGroup
      ├─ GhostSendSystem

6. World 迁移

当 World 切换必须保留连接状态时，使用 DriverMigrationSystem。

using Unity.Entities;
using Unity.NetCode;

public World MigrateServerWorld(World sourceWorld)
{
    DriverMigrationSystem migrationSystem = null;
    foreach (var world in World.All)
    {
        if ((migrationSystem = world.GetExistingSystemManaged<DriverMigrationSystem>()) != null)
            break;
    }

    var ticket = migrationSystem.StoreWorld(sourceWorld);
    sourceWorld.Dispose();

    var newWorld = ClientServerBootstrap.CreateServerWorld("NewServerWorld");
    var newMigrationSystem = newWorld.GetExistingSystemManaged<DriverMigrationSystem>();
    newMigrationSystem.LoadWorld(ticket);
    return newWorld;
}

Chapter 20. Netcode 网络连接与审批

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概述

Netcode for Entities 将每个网络连接表示为一个 Entity。连接 Entity 携带 NetworkStreamConnection；连接结束时，该 Entity 被销毁。

在双方都添加 NetworkStreamInGame 之前，连接不被视为"在游戏中"。没有该 Component，服务器不会发送快照，客户端也不会发送命令。

2. 核心连接 Component

Component 含义
NetworkStreamConnection 存储传输层句柄。
NetworkId 服务器审批后分配的唯一 ID。
NetworkStreamInGame 交换快照和命令数据的必要条件。
CommandTarget 接收玩家命令的 Entity；由游戏代码维护，除非使用 AutoCommandTarget。
NetworkStreamRequestDisconnect 断开连接请求；不要直接调用驱动层断开连接。
NetworkStreamIsReconnected 在 1.6.1 版本线中添加的重连跟踪 Component。

4. 连接状态机

4.1 客户端状态

Connecting → Handshake → [Approval] → Connected → Disconnected
状态 含义
Connecting 调用 Connect 后触发一次。
Handshake 传输层连接已建立；协议握手待处理。
Approval 仅在启用连接审批时存在。
Connected 服务器已分配 NetworkId。
Disconnected 断开连接或超时；DisconnectReason 已设置。

4.2 服务器状态

Handshake → [Approval] → Connected → Disconnected
状态 含义
Handshake 客户端已接受；NetworkProtocolVersion 交换中。
Approval 仅在启用连接审批时存在。
Connected 连接已审批并分配 NetworkId。
Disconnected 客户端已断开连接。

5.3 审批 RPC

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;

public struct ApprovalRpc : IApprovalRpcCommand
{
    public FixedString512Bytes Token;
}

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation |
    WorldSystemFilterFlags.ThinClientSimulation)]
public partial struct ClientApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (connection, entity) in
            SystemAPI.Query<RefRW<NetworkStreamConnection>>()
                .WithNone<NetworkId>()
                .WithNone<ApprovalSent>()
                .WithEntityAccess())
        {
            var rpcEntity = ecb.CreateEntity();
            ecb.AddComponent(rpcEntity, new ApprovalRpc { Token = "MY_TOKEN" });
            ecb.AddComponent<SendRpcCommandRequest>(rpcEntity);
            ecb.AddComponent<ApprovalSent>(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ServerApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, rpc, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ApprovalRpc>>()
                .WithEntityAccess())
        {
            var connEntity = req.ValueRO.SourceConnection;

            if (rpc.ValueRO.Token.Equals("MY_TOKEN"))
                ecb.AddComponent<ConnectionApproved>(connEntity);
            else
                ecb.AddComponent<NetworkStreamRequestDisconnect>(connEntity);

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

审批超时由 ClientServerTickRate.HandshakeApprovalTimeoutMS 控制；默认值为 5000 毫秒。

Chapter 21. Netcode Ghost 快照与同步

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概述

Ghost 是在服务器上模拟并复制到客户端的网络 Entity。服务器定期将 Ghost 状态作为快照发送，每个客户端通过插值或预测应用这些快照。

机制 用途
Ghost 通过不可靠数据包实现最终一致性，内置复制优化。
RPC 一次性可靠事件。

2. Ghost 与 RPC 的对比

使用 Ghost 的场景 使用 RPC 的场景
客户端附近的空间 Entity 状态。 高层游戏流程事件。
需要客户端预测的数据。 一次性命令，如聊天或设置变更。
断线/重连后仍应存在的状态。 无持久复制状态的事件。

3. Ghost Authoring

在需要复制的 Prefab 上添加 GhostAuthoringComponent。

属性 含义
Name Ghost 标识符。
Importance 带宽受限时的快照优先级。
Supported Ghost Mode All、Interpolated 或 Predicted。
Default Ghost Mode Interpolated、Predicted 或 OwnerPredicted。
Optimization Mode 默认为 Dynamic，或对极少变更的对象使用 Static。
MaxSendRate 最大复制频率（Hz），受 NetworkTickRate 限制。
UseSingleBaseline 1.5.0 版本线新增的单基线增量压缩选项。

3.1 Ghost 模式

模式 含义
Interpolated 客户端在服务器快照之间插值。视觉效果流畅但有延迟。
Predicted 客户端提前模拟以降低输入延迟。CPU 开销较高。
OwnerPredicted 所有者客户端预测；其他客户端插值。

4. 序列化特性

4.1 GhostField

在应被序列化的 Component 字段上使用 GhostField。

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

Chapter 22. Netcode 预测与回滚

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概述

客户端预测会在权威服务器结果到达前，在客户端运行相同的模拟代码。这可以降低输入延迟，让玩家控制的移动显得即时响应。

预测有实际的 CPU 开销：当权威快照到达时，客户端可以回滚到较早的 Tick，并向前重新模拟到当前预测的 Tick。

2. 核心 Component 与 System

Component / System 含义
PredictedGhost 在客户端添加到预测 Ghost 上，以及在服务器上添加到所有 Ghost 上。
Simulate 可启用 Tag，设置在应为当前预测 Tick 进行模拟的 Entity 上。
PredictedSimulationSystemGroup 运行预测循环的固定时间步长 Group。
NetworkTime 保存服务器 Tick、预测 Tick 及相关计时数据。

3. 客户端预测流程

1. Receive a server snapshot
   └─ Apply the latest authoritative state to predicted entities

2. Find the oldest applied tick
   └─ Use the oldest tick across affected predicted entities

3. Roll back and re-simulate
   └─ PredictedSimulationSystemGroup repeats from oldest tick to target tick
   └─ TimeData and NetworkTime update for each tick

4. Render
   └─ Present the predicted result

高延迟会倍增开销。根据 Tick 频率和预测设置，300 毫秒延迟可能需要大约 20+ 帧的重新模拟。

4. 编写预测 System

4.1 使用 Simulate Tag

在预测循环开始时，Netcode 会禁用预测 Ghost 上的 Simulate，然后仅为应在当前 Tick 运行的 Entity 启用它。

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct MovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 5f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<Simulate>())
        {
            var move = new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical);
            transform.ValueRW.Position += move * speed * dt;
        }
    }
}

预测 Query 中始终包含 .WithAll<Simulate>()。没有它，System 可能会对已完成的 Tick 重新模拟 Entity。

4.2 服务器端行为

在服务器上，预测循环每个服务器 Tick 恰好运行一次：

  • TimeData 包含正常的服务器 Tick 数据。
  • Simulate 始终处于启用状态。
  • 相同的 System 代码可以同时在客户端和服务器上运行。

5. 远程玩家预测

如果另一个玩家的命令数据被序列化，该远程玩家也可以被预测。

6. 预测平滑

当服务器结果与预测结果不同时，预测平滑可减少可见的纠正跳变。

using Unity.NetCode;
using Unity.Transforms;

GhostPredictionSmoothing.RegisterSmoothingAction<LocalTransform>(
    state.EntityManager,
    CustomSmoothing.Action);

using Unity.Burst;
using Unity.Collections.LowLevel.Unsafe;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[BurstCompile]
public static unsafe class CustomSmoothing
{
    public static readonly PortableFunctionPointer<
        GhostPredictionSmoothing.SmoothingActionDelegate> Action =
        new(SmoothingAction);

    [BurstCompile(DisableDirectCall = true)]
    private static void SmoothingAction(
        void* currentData, void* previousData, void* userData)
    {
        ref var current = ref UnsafeUtility.AsRef<LocalTransform>(currentData);
        ref var backup = ref UnsafeUtility.AsRef<LocalTransform>(previousData);

        var dist = math.distance(current.Position, backup.Position);
        if (dist > 0)
        {
            current.Position = math.lerp(backup.Position, current.Position, 0.5f);
        }
    }
}

7. 预测切换

预测开销较高，因此 Ghost 可以在 Predicted 和 Interpolated 之间动态切换。

7.1 切换先决条件

  • Supported Ghost Modes 必须为 All。
  • 当前模式不得为 OwnerPredicted。
  • 该 Ghost 不能已处于切换中。

7.2 排队切换

using Unity.NetCode;

ref var switchQueues = ref SystemAPI
    .GetSingletonRW<GhostPredictionSwitchingQueues>()
    .ValueRW;

switchQueues.ConvertToPredictedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 1f,
});

switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 0f,
});

插值 Ghost 和预测 Ghost 运行在不同的时间线上，通常相差约两个 Ping 间隔。SwitchPredictionSmoothing 会在配置的持续时间内对位置和旋转进行插值，以避免可见的传送跳变。

1. 概述

3. IInputComponentData

3.1 定义输入

using Unity.NetCode;

3.2 在客户端收集输入

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[UpdateInGroup(typeof(GhostInputSystemGroup))]
public partial struct GatherInputSystem : ISystem
{

    {

        bool left = Input.GetKey(KeyCode.A);

            input.ValueRW = default;
            if (jump) input.ValueRW.Jump.Set();
            if (left) input.ValueRW.Horizontal -= 1;
            if (right) input.ValueRW.Horizontal += 1;
        }
    }
}

Chapter 24. Netcode RPC

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概述

RPC 是客户端与服务器之间的可靠单次消息。RPC 序列化和反序列化代码会自动生成，RPC 处理仍适配 ECS / Job System 帧循环。

RPC 不能替代 Ghost 复制或命令流。将其用于离散的游戏流程事件，而非高频状态。

2. RPC 与 Ghost 与命令的对比

RPC Ghost 命令
传输行为 可靠，有序。 不可靠，具备复制优化。 不可靠，具备冗余发送。
方向 双向。 服务器 → 客户端。 客户端 → 服务器。
频率 一次性。 按需每网络 Tick。 每模拟 Tick。
用途 游戏流程、大厅、聊天。 复制 Entity 状态。 玩家输入。
在途限制 有，可靠管线限制。 无。 无。

由于 RPC 是有序且可靠的，频繁的 RPC 流量可能在在途数据包后面排队阻塞。

Chapter 25. Netcode Ghost 优化 · 重要性 · 相关性

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 概述

大型多人游戏世界无法在每个 Tick 将所有 Ghost 发送给每个客户端。Netcode for Entities 使用优化模式、重要性缩放、相关性和每 Ghost 发送速率控制来降低 CPU 开销和带宽占用。

核心规则很简单：服务器应将快照预算花在当前对特定连接真正重要的 Ghost 上。

2. 优化模式

在 GhostAuthoringComponent 上设置优化模式。

模式 含义 适合场景
Dynamic 默认。在数据变更和不变更时均对快照大小进行优化。 普通会移动的 Ghost。
Static Ghost 未变更时避免重新发送数据。 建筑、地形、静态道具。

1.11.0 版本线添加了静态 Ghost 优化，可显著降低未变更静态 Ghost 的 GhostUpdateSystem 耗时。

3. 快照大小限制

快照预算可在多个层级进行限制。

设置 含义
NetworkStreamSnapshotTargetSize 每连接的软字节目标。
GhostSendSystemData.MaxSendEntities 每快照最大 Entity 数。
GhostSendSystemData.MaxSendChunks 每快照最大 Chunk 数。

当预算有限时，重要性更高的 Ghost 会被优先发送。

4. 重要性缩放

重要性缩放在服务器上计算每个 Ghost 的发送优先级。

4.1 输入

GhostImportance 结合了三类数据。

数据 含义
每连接数据 特定于连接的数据，如玩家位置。
Singleton 数据 全局设置，如瓦片大小。
每 Chunk 共享数据 Chunk 级缩放参数。

4.2 基于距离的重要性

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

var gridSingleton = state.EntityManager.CreateSingleton(
    new GhostDistanceData
    {
        TileSize = new int3(50, 50, 256),
        TileCenter = new int3(0, 0, 128),
        TileBorderWidth = new float3(1f, 1f, 1f),
    });

state.EntityManager.AddComponentData(gridSingleton, new GhostImportance
{
    ScaleImportanceFunction = GhostDistanceImportance.ScaleFunctionPointer,
    GhostConnectionComponentType = ComponentType.ReadOnly<GhostConnectionPosition>(),
    GhostImportanceDataType = ComponentType.ReadOnly<GhostDistanceData>(),
    GhostImportancePerChunkDataType = ComponentType.ReadOnly<GhostDistancePartitionShared>(),
});

来自 1.8.0 版本线的破坏性变更：GhostDistanceImportance 缩放函数不再将 baseImportance 乘以 1000。假设旧乘数的现有自定义重要性代码需要审查。

5. Ghost 相关性

Ghost 相关性允许服务器按连接隐藏或公开特定 Ghost。

模式 含义
Disabled 默认。不过滤；所有 Ghost 都是候选。
SetIsRelevant 仅列出的 Ghost 复制到该连接。
SetIsIrrelevant 列出的 Ghost 被排除在该连接之外。

5.1 使用场景

场景 实现思路
距离剔除 将远处的 Ghost 标记为不相关。
战争迷雾 隐藏视野之外的 Entity。

5.2 相关性示例

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
[UpdateInGroup(typeof(GhostSimulationSystemGroup))]
[UpdateBefore(typeof(GhostSendSystem))]
public partial struct RelevancySystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var relevancy = SystemAPI.GetSingletonRW<GhostRelevancy>();

        relevancy.ValueRW.GhostRelevancyMode = GhostRelevancyMode.SetIsRelevant;
        relevancy.ValueRW.GhostRelevancySet.Clear();

        foreach (var (ghostInstance, transform) in
            SystemAPI.Query<RefRO<GhostInstance>, RefRO<LocalTransform>>())
        {
            foreach (var (connId, connPos) in
                SystemAPI.Query<RefRO<NetworkId>, RefRO<GhostConnectionPosition>>())
            {
                float dist = math.distance(
                    transform.ValueRO.Position,
                    connPos.ValueRO.Position);

                if (dist < 100f)
                {
                    relevancy.ValueRW.GhostRelevancySet.TryAdd(
                        new RelevantGhostForConnection(
                            connId.ValueRO.Value,
                            ghostInstance.ValueRO.ghostId),
                        1);
                }
            }
        }
    }
}

此简化示例的时间复杂度为 O(Ghost × 连接数)。对于大型世界，优先使用基于距离的重要性分区，将相关性保留给分区无法表达的规则，如战争迷雾或队伍可见性。

1.6.1+ 版本线支持通过 PrioChunks.isRelevant 组合 Ghost 相关性和重要性缩放，启用更快的相关性路径。

Chapter 26. Netcode Physics Integration & Lag Compensation

1. 概述

Netcode for Entities 与 Unity Physics 集成，用于网络化物理模拟。物理行为取决于 Ghost 是 Interpolated 还是 Predicted。

场景中至少需要一个 Predicted Ghost，预测物理才能运行。

2. Interpolated 与 Predicted 物理

2.1 客户端上的 Interpolated Ghosts

  • 位置和旋转来自服务器快照。
  • Simulate 被禁用，物理对象表现为运动学。
  • PhysicsVelocity 被忽略。

2.2 客户端上的 Predicted Ghosts

  • 物理在预测循环内运行，每渲染帧可执行多次。
  • PhysicsSystemGroup 在 PredictedFixedStepSimulationSystemGroup 内运行。
  • 客户端应为预测物理产生与服务器相同的模拟结果。
Netcode initialization
├─ PhysicsSystemGroup moves under PredictedFixedStepSimulationSystemGroup
└─ FixedStepSimulationSystemGroup moves under PredictedFixedStepSimulationSystemGroup

3. 预测物理性能

Part III. Optimization and Debugging

Chapter 28. Chunk Layout & TypeManager

1. 16 KB Chunk

每个 archetype 将其 entity 存储在 16 KB chunk 中。chunk 包含：

  1. 一个小型头部（元数据——archetype id、版本号、启用位掩码、变化版本）。
  2. 一个连续的 SoA 区域：每个 component 类型一列，每列为值数组。
  3. 补齐至 16 KB 边界的填充字节。

Chunk 容量——可容纳多少 entity——由以下公式得出：

capacity ≈ (16 KB − header) / sum(sizeof(component_i))

具有少量小型 component 的紧凑 archetype 可容纳数百个 entity。具有许多大型 component 的臃肿 archetype 只能容纳少量 entity。占用率低的 chunk 浪费了本可容纳更多 entity 的字节。

2. TypeManager——component 注册表

TypeManager 在 World 启动时注册每个 component、buffer 和 System，并为每个分配稳定的类型索引。运行时从注册表中读取：

字段 用途
TypeIndex 查询和 chunk 元数据中使用的打包标识符。
SizeInChunk 此 component 在 chunk 中每个 entity 的字节数。
Alignment chunk 内的列对齐方式。
ElementSize buffer 元素大小（用于 IBufferElementData）。
HasBlobReferences / HasEntityReferences 在序列化和移动操作期间启用验证。

通常无需直接操作 TypeManager。重要的是：添加新 component 类型、重命名它们或在程序集间移动它们会触发类型注册表重建，这是大型项目中启动时间消耗的主要来源。

Entities 1.4 报告称，通过跳过不必要的程序集扫描，大型项目的 TypeManager.Initialize 速度提高了约 2 倍。该优化在 6.5 版本中得到继承。

3. 检查 chunk 布局——Archetypes 窗口

Window → Entities → Archetypes 列出 World 中的每个 archetype。每行显示：

  • Entities——活跃数量。
  • Chunks——正在使用的 chunk 数量。
  • Entities / Chunk——平均占用率。
  • Capacity——每个 chunk 的理论最大值。

需要关注的情况：

读数 解读
许多 archetype 每个 chunk 只有 1 个 entity 过细粒度的 Tag 导致 Archetype 数量激增。请进行合并。
一个 archetype 有大量 entity 且占用率高 健康状态——敌人、投射物、子弹。
Entities / Chunk 远小于 Capacity 碎片化——entity 不断转移到其他 archetype（结构性抖动）导致占用分散。
Capacity = 1 archetype 过宽（component 过大）。检查是否有过大的托管/共享 component。

4. 设计 archetype 友好的 component

按影响程度排序的启发式原则：

  1. 保持非托管结构体为非托管。 单个托管 IComponentData 会强制每个 entity 使用边表存储，并破坏 Burst。
  2. 不要填充结构体。 将 float3 放在 float 之前而非之后——对齐空洞会膨胀 SizeInChunk。
  3. 将冷字段分离出去。 如果只有一个 System 读取某个字段，将其移至兄弟 component。未使用的列仍然占用每个 entity 的字节。
  4. 优先使用可启用 component 而非每帧添加/删除。 参见 13_Structural Change & Safety.md——切换一个位远比重塑 archetype 代价低。
  5. 限制共享 component 的值集合。 每个新值都会产生一个新的 chunk 桶。
  6. 标签 component 占用零字节，但仍会增加一个 archetype。 不要给每个 entity 添加独一无二的标记。

5. Chunk 版本号

每个 component 列跟踪一个变化版本——一个 32 位计数器，当任何 System 通过 RefRW<T>.ValueRW、SetComponentData 或 ECB SetComponent 写入时递增。

System 可以跳过自上次运行以来版本未变化的 chunk：

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>());
query.SetChangedVersionFilter(typeof(Health));

使用变化过滤器迭代查询时，只有自 System 上次运行以来发生变化的 chunk 才会被访问。这是"Health 变化时更新 UI"等模式变得廉价的原因。

注意事项：

  • 过滤器是chunk 级别的，而非 entity 级别。一个 entity 的写入会污染整个 chunk。
  • 通过 RefRW<T>.ValueRW 读取（即使实际上没有修改）也会更新版本。
  • 切换可启用 component 同样会更新版本。

6. BlobAsset 存储

BlobAssetReference<T> 将大型只读数据（网格、LOD 表、导航场）存储在 chunk 外部。chunk 中只存储一个指针。当许多 entity 共享相同的烘焙数据时，这避免了 chunk 容量的膨胀。

实用技巧：如果某个 component 以 FixedList 或所有同类 entity 共享的 DynamicBuffer 为主，请将大型负载存储在 blob asset 中，并从精简的 component 中引用它。

{
    public float3 Position;
    public quaternion Rotation;
    public float Health;
    public float HealthMax;
    public float Stamina;
    public float StaminaMax;
    public int AIState;
    public float3 LastKnownPlayerPos;
    public float TargetDistance;
    public Entity Target;
    public FixedList512Bytes<float3> PathBuffer;   // 512 bytes per entity!
}

仅 PathBuffer 一个字段就占用每个 entity 512 字节。每个 16 KB chunk 的容量降至两位数低端。请拆分：

public struct EnemyStats : IComponentData
{
    public float Health, HealthMax, Stamina, StaminaMax;
    public int   AIState;
    public Entity Target;
}

public struct EnemyPathPoint : IBufferElementData
{
    public float3 Value;
}

现在动态 buffer 存储在其独立存储中，chunk 密度得到恢复。

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>(),
                                 ComponentType.ReadWrite<HealthBarLink>());
query.SetChangedVersionFilter(typeof(Health));

state.Dependency = new UpdateHealthBarJob { /* ... */ }
    .ScheduleParallel(query, state.Dependency);

只有 Health 发生变化的 chunk 才会被访问。

1. 概述

World 选择器（窗口顶部）

  • 每行是一个 entity；子行是通过 Parent / LinkedEntityGroup 关联的 entity。
  • 右列计数：component 数量、子项数量。
  • 点击一行 → 其 component 显示在 Inspector 中。

  • 组合使用：c:Health c:Enemy n:boss。

Inspector

entity 的 Inspector 显示：

  • Components 部分——每个 IComponentData 及其当前值。在 Play 模式下可实时编辑。

  • Enabled state——每个 IEnableableComponent 的切换状态。

实时编辑是直写的：更改字段会立即更新该 chunk 的变化版本。

3. Systems

Systems 窗口列出每个已注册的 System，按其 SystemGroup 分组，并按调度器实际执行的顺序排列。

优先阅读的列

列 含义

读数 解读

4. Query 窗口

专为"谁匹配 X？"问题而设计。

工作流程：

  1. 通过加号按钮添加 All component。
  2. 根据需要添加 Any / None。
  3. 窗口在左侧面板列出匹配的 entity，在右侧面板列出运行此查询的 System。

使用场景：

  • "哪些 System 读取 Health？"→ 将 Health 添加到 All；右侧列出的所有 System 都会触及它。
  • "为什么这个 entity 不匹配我的 System 查询？"→ 粘贴 System 的查询，与 Hierarchy 窗口中该 entity 的 component 存在情况进行比对。
  • "有没有 Prefab 漏入这个查询？"→ 在 1.4+ 中，结果列表中通过图标区分 Prefab。

Query 窗口还允许你保存查询以在会话期间复用。

5. Journaling——最后手段的诊断

Window → Analysis → Entities Journaling（通过 Project Settings → Entities → Journaling 启用）将每次结构性变更、component 写入和 System 更新记录到环形缓冲区中。回放视图：

  • Operations——每条记录的变更，包含时间戳、源 System 和受影响的 entity。
  • Entities——每个 entity 的时间轴，记录谁在何时操作了它。

Journaling 有运行时开销——仅在调查 bug 时开启，正常开发时关闭。

6. 字段工作流——调试缺失更新

常见 bug："我的 System 处理了 entity 但 component 没有变化。"排查步骤：

  1. Entities Hierarchy → 找到 entity，确认它拥有该 component。
  2. Inspector → Play 前记录该值。按下 Play 并观察它。
  3. 如果值没有变化：Systems 窗口 → 找到相关 System，检查 Time (ms)。如果为 0，则该 System 被跳过。
  4. 如果时间 > 0 但值仍不变化：Query 窗口 → 粘贴 System 的查询和该 entity 的 component。验证 entity 是否匹配。
  5. 仍未解决：启用 Journaling 1-2 帧。查找其他 System 对该 component 的写入是否覆盖了你的结果。

Chapter 30. Profiler · Bottleneck Analysis

1. 概述

Unity Profiler（Window → Analysis → Profiler）是 DOTS 中解答"为什么这帧很慢"的主要工具。Entities 发出足够多的标记，让你可以将时间归因于特定的 System、Job 和 chunk——前提是你知道该去哪里查找。

本页介绍如何读取 Profile、典型的 DOTS 瓶颈模式以及如何采取行动。

2. 分析前的准备

  1. 谨慎使用 Deep Profile。 它有助于追踪特定的托管路径，但会扭曲计时——永远不要依赖 Deep Profile 的性能数据。
  2. 启用"Record in Editor & Standalone"，以便同时捕获构建版本而非仅 Editor 帧。
  3. 测量发布性能时，将 Profiler 目标设置为 Player 而非 Editor。Editor 开销较大。
  4. 播放几秒后再读取数据；最初几帧包含 domain reload 和 JIT 工作。

3. 读取 CPU Usage 模块

DOTS 帧下的顶层标记：

PlayerLoop
├── Initialization.*
├── Update.ScriptRunBehaviourUpdate              (MonoBehaviour.Update)
├── SimulationSystemGroup                        ← DOTS simulation lives here
│   ├── SystemA.OnUpdate
│   ├── SystemB.OnUpdate
│   └── EndSimulationEntityCommandBufferSystem.Playback
├── FixedStepSimulationSystemGroup
├── PresentationSystemGroup
├── Render
└── WaitForTargetFPS

深入 SystemA.OnUpdate 标记查看其子调用——ScheduleParallel、job 完成等待、EntityManager 调用。

4. 时间实际去哪了——常见模式

4.1 主线程 OnUpdate 主导

如果单个 SystemX.OnUpdate 占用 10+ ms，则工作在主线程上运行。检查：

  • 是否在紧密循环中使用了本应是 IJobEntity.ScheduleParallel 的 SystemAPI.Query？
  • 是否有 SystemBase 因托管字段而被迫运行在主线程？
  • 是否在每个 entity 上调用了 EntityManager.AddComponent/RemoveComponent？

4.2 Job 计时

Job 显示在时间轴视图中（或在层次视图的"Workers"下）。关注：

  • 一个 Job 泳道完全饱和，而其他泳道空闲 → 工作没有分布到各个 chunk。
  • 主线程等待 Job（JobHandle.Complete() 标记）→ .Complete() 调用过早；将其推迟到帧的后面。
  • 巨大的 Burst 函数名 → 正常，符号是完整的泛型实例化。

4.3 EntityManager 热点

查找名为以下的标记：

  • EntityManager.SetArchetype / Move Entity——结构性变更抖动。参见 13_Structural Change & Safety.md。
  • EntityCommandBufferSystem.OnUpdate (Playback)——昂贵的 ECB 回放。通常意味着每帧记录了过多命令。
  • TypeManager.Initialize——仅在启动时出现；如果每帧出现，说明某些东西在重载 World。

4.4 GC.Alloc 标记

DOTS 模拟中的任何 GC.Alloc 都是回归问题。来源：

  • SystemBase 在其 OnUpdate 中分配内存。
  • System 内部对托管集合的 foreach。
  • 装箱——例如在结构体上调用 object.ToString()。

Burst System 在 Profiler 的 Memory 模块中应显示零分配。

5. 需要固定的 Profiler 模块

  • CPU Usage——标记的时间轴和层次视图。
  • Memory——已分配堆；观察 GC.Used Memory 是否上升。
  • Entities（如可用）——DOTS 专属模块，显示各 System 随时间的计时。

Entities Profiler 模块随 1.8.0 / 6.x 引入，与 Systems 窗口在时间轴上对齐。如果你看到多帧峰值，这是归因的最快方式。

对于网络项目，请与 ../DOTS Workflows/24_Netcode Profiler & Debugging.md 配合使用，以检查 Ghost 快照、预测 tick 和 Netcode 专属带宽。

6. 常见瓶颈与修复方案

模式 Profiler 中的现象 修复方案
逐 entity 的 Entities.ForEach（遗留） 主线程密集的 OnUpdate，有大量小样本 迁移到 IJobEntity.ScheduleParallel。
每帧 AddComponent/RemoveComponent Move Entity 标记占主导 将状态转换为可启用 component。
ECB 过载 ...ECBS.Playback 耗时 ms 批量命令（每个 chunk 一个而非每个 entity），或提高批处理大小。
共享 component 值爆炸 SetSharedComponent 占主导 对值进行分桶（量化为更少的不同键）。
帧中途调用 .Complete() 主线程上长时间的 JobHandle.Complete 等待 将完成推迟到帧末，或重构使主线程不再提前需要结果。
Burst 回退 热点方法显示未修饰的 C# 名称 捕获的非 blittable 类型导致回退到 IL。检查 Burst Inspector。
Archetype 碎片化 许多小 chunk；EntityManager.*Query 标记较高 Archetype 过于细粒度——合并标签 / 使用可启用 component。
Bake 时期回归 Play 期间出现 Bake 标记 Domain reload 重新触发了 baking。通常仅在 Editor 中；请在构建版本中确认。

7. 帧预算示例

假设 16.6 ms 预算（60 Hz）。排除 Render 后：

  • SimulationSystemGroup: 6 ms
    • EnemyAISystem.OnUpdate: 2.1 ms（Burst，4 个工作线程并行 Job）
    • MovementSystem.OnUpdate: 1.4 ms（Burst，并行）
    • EndSimulationECBS.Playback: 1.8 ms ← 可疑
    • 其余所有：0.7 ms

对于本应只有少量结构性变更的场景，1.8 ms 的 ECB 回放通常是最大的优化杠杆。检查命令数量：

  1. 对 1 帧启用 Entities Journaling → 统计每个 System 的 Instantiate/AddComponent/DestroyEntity 数量。
  2. 如果某 System 每帧记录数百条命令，请进行批处理（每个 chunk 一个命令，或使用带有 chunk 索引排序键的并行记录器）。

8. 独立版本分析

对于真实构建版本：

  1. File → Build Profiles → Development Build + Autoconnect Profiler。
  2. Player 运行，Profiler 从远程进程捕获。
  3. 结果更真实地反映玩家实际体验。Editor 测量值可能因与你代码无关的原因慢 2-5 倍。

Chapter 31. Managed Object Reference Audit

1. 概述

托管 Unity 对象引用是意外将 ECS System 从快速路径拉出的最简单方式之一。本审计有助于发现 component 数据、System 或渲染胶水代码中以阻碍 Burst、Job 或干净 SubScene Baking 方式持有 UnityEngine.Object 引用的位置。

在完成核心 ECS 迁移后，或每当 Profiler 追踪在预期数据导向代码的地方显示托管工作时，请使用此审计。

2. 搜索什么

模式 重要性
class .*: IComponentData 托管 component 数据；有效但通常不适合热路径。
UnityEngine.Object 托管 Unity 对象引用的广泛标记。
GameObject、Transform、Component 运行时场景对象引用通常不应出现在模拟数据中。
Mesh、Material、Texture、AudioClip 通常应改用 UnityObjectRef<T> 或表现层缓存数据。
Resources.Load 运行时资源查找；通常更适合由 baking/内容系统处理。
ScriptableObject 有用的 Authoring 来源，但运行时 Job 应读取烘焙值或 blob。

将审计范围限定在面向 ECS 的代码。仅 MonoBehaviour 的 UI 或编辑器工具无需强制转换为 DOTS 模式。

3. 分类规则

发现 可能的行动
热模拟查询中的托管 component 转换为非托管 IComponentData、BlobAssetReference<T> 或 UnityObjectRef<T>。
仅由表现层使用的托管 component 如果与 Burst/Job 模拟路径隔离，则保留。
每帧读取 ScriptableObject 将所需值烘焙到 component 或 blob asset 中。
ECS 数据中的 Mesh/Material 引用 使用 UnityObjectRef<Mesh> / UnityObjectRef<Material>，并在表现代码中解引用。
component 数据中存储了 GameObject/Transform 替换为 ECS 数据，或将引用移至混合表现桥接器。

4. 审计示例

// Suspicious: managed component data used as per-entity visual state.
public class EnemyVisual : IComponentData
{
    public Material Material;
}

推荐做法：

using Unity.Entities;
using UnityEngine;

public struct EnemyVisual : IComponentData
{
    public UnityObjectRef<Material> Material;
}

然后在主线程表现 System 中解引用，或在渲染器注册表中缓存解析后的 material。

5. Profiler 信号

Profiler 现象 检查内容
模拟 System 每帧都在分配内存 OnUpdate 中存在托管对象查找或 LINQ/集合分配。
本应纯数据的 System 的 Burst 被禁用 查询路径中存在托管 component 或托管 API 访问。
主线程有许多小型渲染查找 在逐 entity 循环外缓存 UnityObjectRef<T>.Value 的结果。
SubScene 重新 Bake 的频率超出预期 Authoring 脚本或对象引用产生了不必要的依赖。

6. 验证

  1. 重新运行第 2 节中的搜索模式。
  2. 确认热模拟 System 只查询非托管 component。
  3. 检查 Burst Inspector，针对预期使用 Burst 编译的 System/Job。
  4. 在代表性场景中进行性能分析，确认托管分配已从热路径中移除。
  5. 从干净的 Editor 启动进入 Play 模式，验证 SubScene Baking 仍能解析引用。

7. 故障排查

现象 原因 / 解决方案
转换后的 component 仍无法在 Burst 中查询 某个字段仍为托管类型。也检查嵌套结构体。
Play 模式下 UnityObjectRef<T>.Value 返回 null 引用的资源未被烘焙或加载。检查 Baker 和 SubScene/内容工作流程。
转换后视觉 System 仍然很慢 每帧逐 entity 解引用。按共享资源缓存，或烘焙紧凑的查找键。
ScriptableObject 变更不影响运行时数据 值已被烘焙。重新导入/重新烘焙 SubScene，或添加正确的 Baking 依赖。
仅编辑器的引用在 Player 构建中失败 将编辑器 API 置于 #if UNITY_EDITOR 后面，并烘焙运行时安全的数据。

Chapter 32. Entities 1.x → 6.5 Overview

1. 你即将面对的工作

从 Entities 1.x 升级到 6.5，主要是在一次包分发模式转变之上完成三项 DOTS 迁移：

  1. Editor/包分发升级。 迁移到 Unity 6000.4+（com.unity.entities 6.4）或 6000.5+（Entities 6.5），其中 Entities 是嵌入 Editor 的 Core Package。
  2. ECS 数据清理。 审查面向 ECS 的数据中的托管 Unity 对象引用，并将热路径 component 数据尽量迁移为非托管 component、烘焙值、BlobAssetReference<T> 或 UnityObjectRef<T>。
  3. ECS API 迁移。 在 Entities 1.4 中标记为过时的旧版 API——Entities.ForEach、IAspect——在 6.5 中仍然过时。另外：ComponentLookup.GetRefRWOptional → TryGetRefRW。

每项迁移都在本文件夹的独立子文档中完成。本页是地图——何时做哪项、按什么顺序。

2. 推荐顺序

图：Entities 1.x 到 6.5 迁移顺序。

为何如此排序：

  • 先升级 Editor，因为在 Editor 识别 6.x API 之前，其他一切都无法编译。

  • foreach / IAspect 最后处理——这些是 ECS 内部的；不会阻碍代码库其余部分。

3. 什么保持不变

令人惊讶地多——这让迁移不像看起来那么可怕。

不变项 说明

Chapter 34. Managed Object References → UnityObjectRef

1. 概述

迁移旧版 ECS 代码时，一个常见问题是托管对象引用泄漏到运行时 component 数据中。在 Entities 6.5 中，尽可能保持游戏 component 为非托管类型，并在 ECS 数据必须指向 Unity 对象（如网格、材质、纹理、音频片段或 Authoring ScriptableObject）时使用 UnityObjectRef<T>。

本页范围限于 DOTS component 设计。不涵盖引擎范围的 Unity 对象标识符 API。

2. 需要查找的内容

搜索直接存储托管 Unity 对象的 component：

模式 需要审查的原因
实现 IComponentData 的 class component 托管 component 是允许的，但速度较慢且受主线程约束。
component 数据内部的 UnityEngine.Object 字段 通常表明是托管 component 或无效的非托管 component 设计。
GameObject、Transform、Material、Mesh、Texture、AudioClip 字段 判断这是否属于 Authoring、表现层或 UnityObjectRef<T>。
运行时 System 逐 entity 调用 Resources.Load 将资源解析移至 Baking 或中央内容系统。

3. 迁移前后

迁移前——托管 component

using Unity.Entities;
using UnityEngine;

public class ProjectileVisual : IComponentData
{
    public Mesh Mesh;
    public Material Material;
}

这样可以工作，但会使 Component 变为托管类型。托管 Component 会降低 Burst/Job 的灵活性，通常不适合逐 Entity 的游戏玩法数据。

迁移后——使用 UnityObjectRef<T> 的非托管 component

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

在 Baking 时一次性烘焙引用：

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

4. 运行时访问模式

保持模拟 Job 纯数据导向。只在允许托管 Unity API 的地方解引用 Unity 对象。

using Unity.Entities;
using UnityEngine;

public partial class ProjectileVisualUploadSystem : SystemBase
{
    protected override void OnUpdate()
    {
        foreach (var visual in SystemAPI.Query<RefRO<ProjectileVisual>>())
        {
            Mesh mesh = visual.ValueRO.Mesh.Value;
            Material material = visual.ValueRO.Material.Value;

            if (mesh == null || material == null)
                continue;

            // Upload to a rendering registry, material property block cache, etc.
        }
    }
}

对于大量渲染，避免每帧逐 entity 解引用。在表现层注册表中缓存解析后的对象，以紧凑的 ECS 值作为键。

6. 迁移检查清单

  • 找到仅用于存储 Unity 对象引用的托管 component 数据。
  • 在适合使用非托管 component 的地方，将这些字段替换为 UnityObjectRef<T>。
  • 将对象赋值移入 Baker。

Chapter 35. foreach → IJobEntity Migration

1. 状态——已过时，尚未删除

遗留 component 迭代 API 在 Entities 1.4+ / 6.5 上已标记为过时，但仍可编译：

遗留 API 6.5 中的状态 替代方案
Entities.ForEach 自 1.4 起过时；每次使用都有废弃警告；Unity 的 changelog 表示将在未来主版本中删除 SystemAPI.Query<...>（主线程）或 IJobEntity（Job）
Job.WithCode 与 ForEach 同步过时 纯 IJob 结构体，或在 OnUpdate 中内联
IJobForEach（1.0 之前） 在 1.4 之前早已删除 IJobEntity

你来到这里，要么是因为在 6000.5+ 中打开了 1.x 项目，编译器正在大量警告；要么是因为想在未来主版本删除该 API 之前清除这些警告。一旦理解了模式，重构就是机械性的。

2. 两种替代 API

  • SystemAPI.Query<T, U, ...>——在 OnUpdate 内部的主线程迭代。简单，无 Job 开销，适用于小工作量或需要托管 API 的场景。
  • IJobEntity——源代码生成的并行 Job。如果你已准备好多线程工作，这是直接替代——这也是 DOTS 价值所在。

背景资料：../DOTS Workflows/11_IJobEntity · SystemAPI.Query.md。

3. 迁移——Entities.ForEach → SystemAPI.Query

3.1 迁移前

public partial class HealthRegenSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float dt = Time.DeltaTime;

        Entities
            .WithAll<Alive>()
            .WithNone<Dead>()
            .ForEach((ref Health health, in Regen regen) =>
            {
                health.Value = math.min(health.Value + regen.Rate * dt, health.Max);
            })
            .ScheduleParallel();
    }
}

3.2 迁移后——SystemAPI.Query（主线程）

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>()
                          .WithAll<Alive>()
                          .WithNone<Dead>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRO.Max);
        }
    }
}

形式变化说明：

  • SystemBase → ISystem（更轻量，支持 Burst）。如果旧代码捕获了托管字段，可能需要保留 SystemBase。
  • ref Health → RefRW<Health> 元组元素。通过 .ValueRO 读取，通过 .ValueRW 写入。
  • in Regen → RefRO<Regen>。
  • Time.DeltaTime → SystemAPI.Time.DeltaTime。
  • 过滤器（WithAll/WithNone）移至 Query 构建器上。

4. 迁移——Entities.ForEach.ScheduleParallel → IJobEntity

当 ForEach 被并行调度时，Job 形式是直接的 1:1 迁移，并保留并行性。

4.1 迁移后——IJobEntity

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

过滤器以特性形式放在 Job 结构体上：

[WithAll(typeof(Alive))]
[WithNone(typeof(Dead))]
public partial struct HealthRegenJob : IJobEntity { ... }

或在调度时传入查询：

var q = SystemAPI.QueryBuilder()
    .WithAll<Health, Regen, Alive>()
    .WithNone<Dead>()
    .Build();
new HealthRegenJob { DeltaTime = dt }.ScheduleParallel(q);

5. 捕获变量及其替代

遗留 Entities.ForEach 允许通过 .WithReadOnly、.WithDisposeOnCompletion 等捕获局部变量。在 IJobEntity 中，捕获变量是 Job 结构体上的字段——Burst 清楚地知道传入了什么。

// BEFORE
var buffer = new NativeArray<int>(count, Allocator.TempJob);
Entities.ForEach((ref Foo f) => { /* use buffer */ })
    .WithDisposeOnCompletion(buffer)
    .ScheduleParallel();

// AFTER
[BurstCompile]
partial struct FooJob : IJobEntity
{
    [ReadOnly] public NativeArray<int> Buffer;

    void Execute(ref Foo f) { /* use Buffer */ }
}

var buffer = new NativeArray<int>(count, Allocator.TempJob);
state.Dependency = new FooJob { Buffer = buffer }.ScheduleParallel(state.Dependency);
buffer.Dispose(state.Dependency);   // chained dispose replaces WithDisposeOnCompletion

特性替换：

  • .WithReadOnly(x) → [ReadOnly] public NativeArray<T> X;
  • .WithDisposeOnCompletion(x) → 调度后执行 x.Dispose(handle)。
  • .WithoutBurst() → 在 Job 结构体上添加 [WithoutBurst]（但强烈建议消除需要它的原因）。
  • .WithEntityAccess() → 在 Execute 中添加 Entity entity 参数。

6. 何时仍需 SystemBase

如果原来的 System 确实有托管字段（Input System、UI Toolkit、外部服务），则无法机械地转换为 ISystem。两种选择：

  1. 保留为 SystemBase，在 OnUpdate 内使用 SystemAPI.Query。单线程。
  2. 拆分为一个桥接托管世界的 SystemBase 和一个执行 Burst 工作的 ISystem——参见 ../DOTS Workflows/08_System — ISystem vs SystemBase.md。

7. 验证

迁移 System 后：

  1. 对行为进行单元测试。 迁移不应改变逻辑；如果输出不同，说明重构引入了 bug。
  2. 进行性能分析。 等效工作下，并行 ForEach 和 IJobEntity.ScheduleParallel 的差异应在 5% 以内。大幅回退意味着 Burst 未生效——检查 Burst Inspector。
  3. 删除旧的 System 文件。 抵制保留 #if LEGACY 回退——它会腐化代码。

Chapter 36. IAspect Deprecation — Migrating Off Aspects

1. 状态——已废弃，尚未删除

IAspect 在 Entities 1.4 中被标记为过时，在 6.x 版本线上仍然过时。它仍可编译、运行，在 6.5 API 参考中仍被列为有效的 IJobEntity.Execute 参数类型——但编译器会发出废弃警告，Unity 已声明该类型将在未来主版本中删除。新代码不应使用它；现有代码应伺机迁移。

IAspect 的引入是为了让 System 能够声明一个覆盖一组 component 的"视图"，并在 Entities.ForEach 或 IJobEntity 中作为单一参数使用。实践中，它增加了一层源代码生成和并行类型系统，得不偿失：

  • 编写 Aspect 重复了 component 参数列表。
  • 调试器需要单步执行生成的包装代码。
  • IJobEntity 和 SystemAPI.Query 上的 RefRW<T> / RefRO<T> 无需额外类型即可实现相同的"将 component 作为捆绑传递"的人体工程学。

由于该接口仍然存在，代码不会因 Editor 升级而中断——此迁移主要是清理警告并为 2.0 做准备。

2. 机械式重构

2.1 起始代码

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public readonly partial struct MovementAspect : IAspect
{
    public readonly RefRW<LocalTransform> Transform;
    public readonly RefRO<Velocity>       Velocity;
    public readonly RefRO<Speed>          Speed;

    public void Step(float dt)
    {
        Transform.ValueRW.Position += Velocity.ValueRO.Direction * Speed.ValueRO.Value * dt;
    }
}

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var m in SystemAPI.Query<MovementAspect>())
            m.Step(dt);
    }
}

2.2 迁移后——直接 component 参数

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var (transform, velocity, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<Velocity>, RefRO<Speed>>())
        {
            transform.ValueRW.Position += velocity.ValueRO.Direction * speed.ValueRO.Value * dt;
        }
    }
}

两处变化：

  1. Aspect 类型消失了。其字段成为 SystemAPI.Query 的元组元素。
  2. Aspect 上的方法（Step）被内联。如果过大不适合内联，提取一个接受相同参数的静态辅助函数——这样可以保持 Burst 友好：
static void Step(ref LocalTransform t, in Velocity v, in Speed s, float dt)
{
    t.Position += v.Direction * s.Value * dt;
}

调用方式：

Step(ref transform.ValueRW, velocity.ValueRO, speed.ValueRO, dt);

3. IJobEntity 中的 Aspect

3.1 迁移前

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;
    void Execute(MovementAspect aspect) => aspect.Step(DeltaTime);
}

3.2 迁移后——component 作为 Execute 参数

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in Velocity velocity, in Speed speed)
    {
        transform.Position += velocity.Direction * speed.Value * DeltaTime;
    }
}

注意 IJobEntity 使用普通的 ref / in 参数，而非 RefRW<T> / RefRO<T>。（生成器在底层将 component 包装为 chunk 数组访问。）

4. 包装 DynamicBuffer 的 Aspect

// BEFORE
public readonly partial struct PathAspect : IAspect
{
    public readonly DynamicBuffer<Waypoint> Path;
    public void Advance(int n) => Path.RemoveRange(0, n);
}

// AFTER — access buffer directly
foreach (var (path, e) in SystemAPI.Query<DynamicBuffer<Waypoint>>().WithEntityAccess())
{
    path.RemoveRange(0, 1);
}

在 Job 中，DynamicBuffer<T> 是有效的 Execute 参数——无需额外处理。

5. 跨 entity 的 Aspect

有时 Aspect 包装了 ComponentLookup<T>，用于"我需要查看另一个 entity 的 component"。替代方案：

// BEFORE
public readonly partial struct AITargetAspect : IAspect
{
    public readonly RefRO<Target> Target;
    [ReadOnly] public readonly ComponentLookup<Health> HealthLookup;
    public float TargetHealth =>
        HealthLookup.TryGetComponent(Target.ValueRO.Entity, out var h) ? h.Value : 0f;
}

// AFTER — declare the lookup on the system / job directly
public partial struct AISystem : ISystem
{
    private ComponentLookup<Health> _healthLookup;

    public void OnCreate(ref SystemState state)
    {
        _healthLookup = state.GetComponentLookup<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthLookup.Update(ref state);

        foreach (var (target, e) in SystemAPI
                     .Query<RefRO<Target>>()
                     .WithEntityAccess())
        {
            float hp = _healthLookup.TryGetComponent(target.ValueRO.Entity, out var h)
                ? h.Value : 0f;
            // ...
        }
    }
}

此模式也比 Aspect 版本更轻量，因为查找在所有 entity 间共享。

6. Aspect 较复杂时

如果 Aspect 字段众多、封装了状态或参与了非平凡的继承——这表明应考虑进行真正的重构，而非仅仅迁移。通常，代码读起来更像是两个通过 component 传递数据的 System，而非一个拥有"神级 Aspect"的 System。

7. 故障排查

现象 原因 / 解决方案
编译警告 IAspect is obsolete 6.5 上的预期行为。按上述方式迁移以清除警告；在未来主版本删除之前代码仍可运行。
type or namespace IAspect not found 你在已删除 IAspect 的版本上。完成迁移——没有回退。
Query 元组类型过多导致编译失败 SystemAPI.Query 支持大型元组；将非常宽的查询拆分为两个查询，或改用 IJobChunk，通常可读性更好。
