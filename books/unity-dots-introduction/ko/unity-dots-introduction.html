Part I. 시작하기

Chapter 1. 환경 설정 — Unity 6000.5 + Entities 6.5.0

1. 개요

Unity 6000.4+부터 Entities는 Core Package로서 Editor 내부에 포함되어 제공되므로, Package Manager를 통해 추가할 필요가 없다. 이 가이드에서는 Editor 설치, DOTS 프로젝트 생성, Entities 활성화 확인을 단계별로 안내한다.

프로젝트 환경 참조: Unity 6000.5.0b2 · Entities 6.5.0 · Collections / Mathematics / Entities Graphics (모두 Core Package).

2. Unity 6000.5+ 설치

  1. Unity Hub를 연다. 없다면 먼저 Unity Hub를 설치한다 (https://unity.com/download).
  2. Hub에서 Installs → Install Editor로 이동한다.
  3. 6000.5.x 버전 중 하나를 선택한다(베타 b 시리즈 이상). Entities 6.5는 6000.5+ 에서만 사용 가능하다.
  4. 모듈: 기본값을 유지한다. ECS 자체를 위해 추가 모듈은 필요하지 않다.

"DOTS" 모듈을 별도로 선택할 필요는 없다 — Entities는 6000.4부터 Editor에 내장되어 있다.

3. 새 프로젝트 생성

Hub → Projects → New project에서:

필드 값
Editor Version 6000.5.x
Template Universal 3D (또는 3D (Built-in), HDRP — 모두 가능)
Project Name 임의

Create project를 클릭한다. Editor가 빈 씬으로 열린다.

6000.4+에서는 전용 "DOTS template"이 더 이상 필요하지 않다. 어떤 템플릿이든 Entities 워크플로를 호스팅할 수 있다.

4. Core Package 활성화 확인

Window → Package Manager를 열고 스코프를 In Project로 전환한다. 다음 패키지들이 "Built-in"으로 표시되어야 한다:

  • Entities
  • Collections
  • Mathematics
  • Entities Graphics

파일로 확인하고 싶다면 Packages/manifest.json을 열어 본다: Editor와 함께 제공되므로 com.unity.entities 항목이 없어야 한다.

다음으로, ECS 창을 연다:

  • Window → Entities → Hierarchy — 런타임의 Entity를 표시한다.
  • Window → Entities → Systems — 등록된 System을 표시한다.
  • Window → Entities → Archetypes — Chunk 레이아웃을 표시한다.

세 메뉴가 모두 존재하면 Entities가 올바르게 연결된 것이다.

5. 스모크 테스트 — SubScene 생성

  1. Hierarchy에서 우클릭 → New Sub Scene → Empty Scene…
  2. 이름을 MainSubScene.unity로 지정하고 저장한다.
  3. 새 SubScene 안에서 임의의 기본 도형(3D Object → Cube)을 추가한다.
  4. 씬을 저장한다. SubScene이 닫히면(노란 아이콘) 또는 빌드 시점에 Cube가 Entity로 Baking된다.
  5. Play를 누른다. Window → Entities → Hierarchy를 열면 Cube Entity가 보여야 한다. 스모크 테스트 통과.

SubScene은 Authoring 진입점이다: 내부의 GameObject는 저장 및 빌드 시 Entity로 Baking된다. 03_Hello DOTS — First Entity.md의 나머지 튜토리얼은 이를 기반으로 한다.

6. 트러블슈팅

증상 원인 / 해결
Window → Entities 메뉴가 없음 Editor가 6000.4보다 오래됨 — Entities가 아직 Core Package가 아님. Editor를 업그레이드한다.
Baker<T> / IComponentData를 찾을 수 없음 어셈블리 정의 파일이 Unity.Entities를 제외하고 있음. 참조로 추가하거나, 빠른 테스트를 위해 asmdef를 삭제한다.
SubScene이 GameObject로 남아 Baking되지 않음 Baking된 Entity가 나타나려면 SubScene이 닫혀 있어야 한다(노란 아이콘). Hierarchy에서 이름 옆 체크박스로 열기/닫기를 전환한다.
Entities Hierarchy 창이 Play 모드에서 아무것도 표시하지 않음 SubScene이 현재 씬에 포함되어 있는지, Window → Entities → Hierarchy의 "World" 드롭다운이 DefaultGameObjectInjectionWorld로 설정되어 있는지 확인한다.
빌드 오류: com.unity.entities not found manifest.json에 1.x 프로젝트에서 남은 com.unity.entities 항목이 있음 — 삭제한다. Migration/02_Package Manager → Core Package.md 참조.

Chapter 2. 핵심 패키지 설명

1. 개요

Entities 패키지 6.4.0 릴리스(Unity Editor 6000.4와 함께 출시)와 함께, 주요 DOTS 패키지들이 Package Manager에서 Editor 자체로 이전하여 Core Package가 되었다. 패키지 변경 이력의 정확한 표현은 다음과 같다:

"Package is now a core package embedded in Unity."

이 변경은 com.unity.entities, com.unity.collections, com.unity.mathematics, com.unity.entities.graphics, 그리고 (매칭되는 6.5 릴리스에서) Netcode for Entities에 적용되었다. 설치, 버전 관리, 추적 방식이 변경되지만, 이것은 배포 변경이지 API 변경이 아니다.

이 페이지에서 설명하는 내용:

  • 어떤 패키지가 이제 Core Package인지(그리고 어떤 것이 아닌지).
  • 버전 번호가 어디서 오는지.
  • manifest.json과 packages-lock.json에서 무엇이 변경되는지.
  • 이 전환이 일어난 이유.

2. Core Package란?

Core Package는 Editor 내부에 탑재되는 패키지이다. Package Manager를 통해 추가할 필요가 없고, Editor와 독립적으로 버전을 변경할 수 없으며, 릴리스 주기가 Editor의 릴리스 주기와 같다.

특성 Package Manager 패키지 Core Package
설치 단계 UPM을 통해 추가 없음 — Editor와 함께 제공
버전 선택 주체 사용자 (manifest.json에서) Unity, Editor 버전별
릴리스 주기 UPM 독립 Editor 릴리스
manifest.json에 표시 예 아니오
packages-lock.json에 표시 예 예, builtin으로 표시
Package Manager에 표시 "In Project" 아래 "In Project" 아래 Built-in 태그

3. Unity 6000.5+에서의 DOTS 패키지 목록

참고: Job System과 Burst는 UPM 패키지였던 적이 없다 — 엔진의 일부이다. 6000.4의 변경은 구체적으로 그 위에 있는 DOTS 패키지들에 대한 것이다.

4. 전환 이후의 버전 번호

Core Package 버전은 Editor에 맞춰 정렬된다:

Editor Core Entities
Unity 6000.4 Entities 6.4
Unity 6000.5 Entities 6.5

6000.4+로 아직 이동할 수 없는 프로젝트를 위해 1.x 라인은 여전히 존재한다. 두 트랙은 분리되어 있으며, Editor를 업그레이드함으로써 6.x로 전환한다.

Entities 전환 노트는 Changelog/Entities 1.4 → 6.5 Key Changes.md를, Netcode 전용 변경 사항은 Changelog/Netcode for Entities 1.4 → 6.5 Key Changes.md를 참조한다.

5. 프로젝트 파일에 미치는 영향

manifest.json

com.unity.entities, com.unity.collections, com.unity.mathematics, com.unity.entities.graphics 항목이 필요 없다. 1.x 프로젝트를 업그레이드한다면 이 항목들을 제거한다 — Migration/02_Package Manager → Core Package.md 참조.

packages-lock.json

Core Package는 버전 URL 대신 "source": "builtin"으로 표시된다.

어셈블리 정의 파일 (.asmdef)

.asmdef 파일은 여전히 어셈블리 이름(예: Unity.Entities, Unity.Collections)을 참조한다. 변경되는 것은 없다 — 패키지 배포 레이어만 변경되었다.

6. 전환 이유

공식적으로 밝힌 목표는 DOTS 업데이트를 독립적인 패키지 일정 대신 Editor 릴리스에 연결함으로써, ECS 팀이 더 빠르고 점진적으로 기능을 출시할 수 있도록 하는 것이다. 또한 엔진이 내부적으로 ECS에 의존할 수 있게 되는데, 패키지가 선택 사항이었을 때는 이것이 불편했다.

7. 트러블슈팅

증상 원인 / 해결
Package Manager에 오래된 버전의 Entities가 "Unity Registry" 아래 표시됨 6000.4 이전 Editor 사용 중. Editor를 업그레이드하거나 1.x 라인을 유지한다.
com.unity.entities 제거 후 manifest.json 해석 오류 packages-lock.json을 삭제하고 Editor가 재생성하도록 한다.
샘플이나 애셋 스토어 패키지가 com.unity.entities@1.x를 대상으로 함 6000.5+에서 1.x API는 대부분 대체되었다. 동등한 항목은 Migration 폴더를 확인한다.
두 Editor 설치가 서로 다른 Entities 버전을 보고함 예상되는 동작. 버전은 Editor 설치별로 다르다.

Chapter 3. Hello DOTS — 첫 번째 Entity

1. 만들 것

ECS System으로 Y축을 중심으로 회전하는 큐브. 한 번의 실습에서 모든 핵심 개념을 다룬다: SubScene, Authoring → Baker, IComponentData, ISystem, 그리고 Entities Hierarchy / Systems 창. 마지막에는 메인 스레드 System을 IJobEntity로 교체하여 병렬 처리가 어떻게 도입되는지 확인한다.

전제 조건: [01_Environment Setup.md](01_Environment Setup (Unity 6000.5 + Entities 6.5.0).md) — SubScene과 Entities 메뉴가 확인된 Unity 6000.5 프로젝트.

2. SubScene 생성

  1. 빈 씬을 연다.
  2. Hierarchy에서 우클릭 → New Sub Scene → Empty Scene…
  3. Assets/Scenes/ 내부에 HelloDots.unity로 저장한다.
  4. 새 SubScene 안에서 우클릭 → 3D Object → Cube. 이름을 RotatingCube로 변경한다.
  5. 씬을 저장한다(Ctrl/⌘ + S). Hierarchy에서 SubScene 이름 옆 체크박스를 클릭하여 SubScene을 닫는다 — 아이콘이 노란색으로 바뀌면 내용이 Entity로 Baking되고 있음을 의미한다.

이 시점부터 RotatingCube의 Component를 수정하면 SubScene을 다시 열고 닫을 때 자동으로 리베이킹이 트리거된다.

3. Component 정의 — RotationSpeed

Assets/Scripts/RotationSpeed.cs를 생성한다:

using Unity.Entities;

public struct RotationSpeed : IComponentData
{
    public float RadiansPerSecond;
}

참고:

  • 비관리형 struct에 IComponentData를 사용하는 것이 기본 선택이다 — Chunk에 직접 저장되며 Burst 친화적이다.
  • 필드는 블리터블(primitives, Unity.Mathematics 타입, 기타 비관리형 struct)로 유지한다.

4. Authoring + Baker 정의

Assets/Scripts/RotationSpeedAuthoring.cs를 생성한다:

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class RotationSpeedAuthoring : MonoBehaviour
{
    public float DegreesPerSecond = 90f;

    class Baker : Baker<RotationSpeedAuthoring>
    {
        public override void Bake(RotationSpeedAuthoring authoring)
        {
            // Dynamic — this entity will have its transform written each frame.
            var entity = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(entity, new RotationSpeed
            {
                RadiansPerSecond = math.radians(authoring.DegreesPerSecond)
            });
        }
    }
}

큐브에 Authoring 컴포넌트 부착

  1. SubScene에서 RotatingCube를 선택한다.
  2. Inspector → Add Component → Rotation Speed Authoring.
  3. DegreesPerSecond = 90으로 유지한다.

SubScene이 닫히는 즉시(노란 아이콘), Baker가 실행되어 큐브가 RotationSpeed Component를 가진 Entity로 변환된다.

Baking 결과 확인

  • Window → Entities → Hierarchy — RotatingCube가 목록에 표시되어야 한다.
  • 클릭한다. 오른쪽 Inspector에 Component가 표시된다 — RotationSpeed { RadiansPerSecond: ~1.5708 } (π/2)를 확인한다.

5. System 작성 — RotationSystem

Assets/Scripts/RotationSystem.cs를 생성한다:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<RotationSpeed>>())
        {
            transform.ValueRW = transform.ValueRO.RotateY(speed.ValueRO.RadiansPerSecond * dt);
        }
    }
}

참고:

  • partial struct + ISystem은 비관리형 System 형태이다. Burst로 컴파일할 수 있으며 GC 오버헤드 없이 실행된다.
  • SystemAPI.Query<RefRW<A>, RefRO<B>>()는 최신 반복 API이다 — 이는 deprecated된 Entities.ForEach를 대체한다.
  • LocalTransform.RotateY(radians)는 회전된 복사본을 반환한다; transform.ValueRW = …으로 다시 쓴다.

7. IJobEntity로 교체 (병렬 처리)

RotationSystem.cs를 Job 기반 버전으로 교체하여 병렬 처리가 어떻게 연결되는지 확인한다:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotateJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in RotationSpeed speed)
    {
        transform = transform.RotateY(speed.RadiansPerSecond * DeltaTime);
    }
}

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new RotateJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

참고:

  • IJobEntity.Execute(ref A, in B)는 소스 생성기에 의해 Chunk 레벨 Job으로 생성된다. 매개변수가 접근하는 Component들이다.
  • .ScheduleParallel()은 Chunk를 워커 스레드 전반에 분산한다. 단일 큐브에서는 속도 향상을 볼 수 없지만, 코드의 형태는 변경 없이 수천 개의 Entity로 확장된다.
  • System의 의존성은 자동으로 처리된다 — ISystem.OnUpdate에서 ScheduleParallel()을 호출할 때 state.Dependency가 자동으로 결합된다.

Chapter 4. Baker 패턴 & SubScene

1. 개요

Entity는 Hierarchy에서 직접 Authoring할 수 없다. 대신 SubScene 안에서 GameObject를 Authoring하면 Baker가 SubScene이 닫히거나 빌드될 때 이를 Entity로 변환한다.

Figure: Baker and SubScene Pipeline.

2. SubScene 라이프사이클

SubScene에는 세 가지 표시 상태가 있다:

Hierarchy 아이콘 상태 의미
⬜ (흰색 체크박스) 열림 GameObject가 편집 가능하며, 아직 GameObject 상태로 Baking되지 않음.
🟨 (노란색 체크박스) 닫힘 Baker가 실행됨; 내용이 Entity로만 존재함.
⏳ Baking 중 증분 Baking 진행 중 (잠깐).

SubScene 생성

  • Hierarchy → 우클릭 → New Sub Scene → Empty Scene… — .unity 애셋을 저장하고 부모 GameObject에 SubScene Component를 추가한다.
  • SubScene 이름 옆 체크박스로 열기/닫기 전환. 닫으면 Baking이 트리거된다.

증분 Baking

SubScene이 열린 상태에서 GameObject를 편집하면 수정된 Authoring에 닿은 Baker만 다시 실행된다. 이러한 이유로 Bake()에서 비용이 큰 작업을 피한다.

3. Baker<T> API

Baker는 하나의 Authoring MonoBehaviour를 하나 이상의 Entity + Component로 매핑한다.

public class EnemyAuthoring : MonoBehaviour
{
    public float MaxHealth = 100f;
    public GameObject ProjectilePrefab;

    class Baker : Baker<EnemyAuthoring>
    {
        public override void Bake(EnemyAuthoring authoring)
        {
            // Primary entity for this GameObject.
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Health { Value = authoring.MaxHealth });

            // Bake a GameObject reference into an Entity reference.
            AddComponent(self, new ProjectilePrefabRef
            {
                Value = GetEntity(authoring.ProjectilePrefab, TransformUsageFlags.Dynamic)
            });
        }
    }
}

주요 API:

API 목적
GetEntity(TransformUsageFlags) 현재 Authoring GameObject의 Entity.
GetEntity(Object, TransformUsageFlags) 다른 GameObject/Prefab에 해당하는 Entity(Prefab Entity 등록).
CreateAdditionalEntity(TransformUsageFlags) 같은 Authoring에 연결된 두 번째 Entity(예: 하위 부품).
AddComponent<T>(entity, value) Component 데이터 부착.
AddBuffer<T>(entity) / SetBuffer<T> DynamicBuffer<T> 부착.
AddSharedComponent<T> ISharedComponentData 부착.
DependsOn(asset) 애셋 변경 시 Baker가 다시 실행되도록 애셋 의존성 선언.

4. TransformUsageFlags

Baking은 TransformUsageFlags를 통해 요청할 때만 LocalTransform / LocalToWorld / Parent를 추가한다. 사용하지 않는 Transform Component는 이유 없이 Chunk를 크게 만들므로, 여전히 동작하는 가장 좁은 플래그를 선택한다.

플래그 추가 Component 일반적인 사용 예
None 없음 순수 데이터 Entity (설정, Singleton, 로직 전용).
ManualOverride 없음; Transform을 직접 관리 커스텀 Transform 레이아웃.
Renderable LocalToWorld 움직이지 않는 정적 메시.
Dynamic LocalTransform, LocalToWorld (부모가 있을 경우 Parent 포함) 이동하거나 System이 기록하는 모든 것.
WorldSpace LocalTransform, LocalToWorld; 부모 설정 방지 계층 구조를 무시하는 Entity.
NonUniformScale PostTransformMatrix 추가 비균일 스케일 (DOTS에서는 드묾).

System이 이동·회전할 모든 것에 대한 안전한 기본값은 Dynamic이다.

5. Baking System (고급)

Baker는 단일 Authoring에서 Component를 생성한다. Baking System은 모든 Baker가 실행된 이후에 실행되며 완전히 Bake된 World를 볼 수 있다 — Entity 간 후처리(참조 해결, 파생 데이터 계산)에 유용하다.

[WorldSystemFilter(WorldSystemFilterFlags.BakingSystem)]
public partial struct EnemyWaveLinkSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Runs during bake only. Query entities tagged earlier by Bakers
        // and wire them together.
    }
}

Baker가 필요한 모든 정보를 알 수 없는 경우(예: 카운트, 교차 참조)에 사용한다. 결정론적으로 유지할 것 — Baking은 매번 동일한 결과를 생성해야 한다.

7. 문제 해결

증상 원인 / 해결
Entity는 보이지만 Component가 없음 Baker가 실행되지 않음 (SubScene이 닫힌 후 Authoring이 추가됨) — SubScene을 토글하여 다시 Bake하라.
GetEntity(prefab, ...)가 Entity.Null을 반환 참조된 GameObject가 Prefab 에셋이 아니거나 Baker에서 도달할 수 없는 Scene 객체다. Prefab이거나 같은 SubScene 내에 있는지 확인하라.
Editor에서 Baker가 매 프레임 실행됨 Bake()에 비결정론적 로직이 있음 (예: Random, Time.time). Baker는 Authoring의 순수 함수여야 한다.
Entity에 LocalToWorld는 있지만 LocalTransform이 없음 TransformUsageFlags.Renderable이 사용됨. Entity가 이동할 경우 Dynamic으로 전환하라.
Authoring 필드 변경 시 다시 Bake되지 않음 Baker가 해당 필드를 읽지 않거나 외부 에셋에 대해 DependsOn을 호출하지 않음.
Baking이 느림 Bake() 내부의 무거운 작업. Entity 간 로직은 Baking System이나 런타임 일회성 초기화로 이전하라.

Chapter 5. Spawner 예제

1. 개요

스포너는 더 많은 Entity를 생성하는 단일 Entity다 — 시작 시, 타이머에 따라, 또는 이벤트 발생 시. 이 페이지에서는 다음 내용을 포함하여 처음부터 끝까지 하나를 만들어 본다:

  • SubScene에서 스포너 Authoring하기.
  • Prefab Entity 참조 저장 (GameObject Prefab에서 Bake됨).
  • 구조적 변경이 안전하게 이루어지도록 EntityCommandBuffer를 통해 런타임에 인스턴스화하기.

01_Baker Pattern & SubScene.md를 먼저 읽으면 이해가 더 쉬울 것이다.

2. 데이터 모델

using Unity.Entities;
using Unity.Mathematics;

public struct Spawner : IComponentData
{
    public Entity  Prefab;       // baked prefab entity
    public float3  SpawnPoint;
    public float   Interval;
    public float   TimeLeft;
    public int     RemainingCount;  // stop when 0
}

주목할 두 가지 설계 선택:

  • Prefab은 GameObject가 아닌 Entity다. Baker가 GameObject Prefab을 Prefab Entity로 변환하며, 런타임은 Entity만 다룬다.
  • TimeLeft와 RemainingCount는 스포너 Entity에 있어 System이 상태를 유지하지 않아도 된다.

3. Authoring + Baker

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float      Interval = 0.5f;
    public int        Count    = 100;

    class Baker : Baker<SpawnerAuthoring>
    {
        public override void Bake(SpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Spawner
            {
                Prefab        = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                SpawnPoint    = authoring.transform.position,
                Interval      = authoring.Interval,
                TimeLeft      = 0f,            // fires immediately on first update
                RemainingCount = authoring.Count
            });
        }
    }
}

SubScene의 빈 GameObject에 SpawnerAuthoring을 붙이고, Prefab을 할당하고, Count를 설정한다. SubScene을 닫으면 스포너가 Entity가 된다.

4. 런타임 System

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnerSystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<Spawner>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Use the EndSimulation ECB so structural changes flush at a safe point.
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged);

        foreach (var (spawnerRW, spawnerEntity) in
                 SystemAPI.Query<RefRW<Spawner>>().WithEntityAccess())
        {
            ref var spawner = ref spawnerRW.ValueRW;

            if (spawner.RemainingCount <= 0)
            {
                ecb.DestroyEntity(spawnerEntity);   // done — remove the spawner
                continue;
            }

            spawner.TimeLeft -= dt;
            if (spawner.TimeLeft > 0f) continue;

            spawner.TimeLeft = spawner.Interval;
            spawner.RemainingCount--;

            var instance = ecb.Instantiate(spawner.Prefab);
            ecb.SetComponent(instance, LocalTransform.FromPosition(spawner.SpawnPoint));
        }
    }
}

핵심 포인트:

  • RequireForUpdate<Spawner>()는 스포너가 없을 때 System을 완전히 건너뛴다.
  • EndSimulationEntityCommandBufferSystem Singleton은 프레임 끝에 플러시되는 공유 ECB를 제공한다 — 루프 중간에 EntityManager로 직접 변경하는 것보다 안전하다.
  • ecb.Instantiate(prefab)은 지연된 Entity를 생성하고, ecb.SetComponent(instance, …)는 해당 Entity에 Component 쓰기를 예약한다. instance 핸들은 이 ECB의 Playback 내에서만 유효하다. 14_EntityCommandBuffer · Deferred Entity.md를 참조하라.

5. 규모 확장 — 병렬 스포너

많은 스포너에서 프레임당 수천 개를 스폰해야 한다면 병렬 ECB 작성자를 사용하는 IJobEntity로 변환하라:

[BurstCompile]
public partial struct SpawnJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, entity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

System 내부에서:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

new SpawnJob { DeltaTime = SystemAPI.Time.DeltaTime, ECB = ecb }
    .ScheduleParallel();

chunkIndex 정렬 키는 병렬 Playback을 결정론적으로 유지하는 것이다. 15_ParallelWriter · Deterministic Playback.md를 참조하라.

6. 문제 해결

증상 원인 / 해결
아무것도 스폰되지 않음 Spawner Entity가 없음 — SubScene을 열고 SpawnerAuthoring이 붙어 있으며 SubScene이 닫혀 있는지(노란색 아이콘) 확인하라.
Prefab이 월드 원점에 스폰됨 ECB가 LocalTransform을 설정하지 않음. ecb.SetComponent(instance, LocalTransform.FromPosition(...))을 추가하거나, Prefab에 이미 원하는 위치의 Transform이 있는지 확인하라.
Prefab은 보이지만 이동하지 않음 Prefab이 Dynamic 대신 TransformUsageFlags.Renderable로 Bake됨.
SystemAPI.GetSingleton<...ECBS.Singleton>()이 예외를 던짐 ECB System이 등록되지 않음 — 매우 간소화된 World에서 발생. state.EntityManager + 수동 ECB로 폴백하거나, Bootstrap에서 ECB System을 등록하라.
스포너가 멈추지 않음 RemainingCount를 spawnerRW.ValueRW가 아니라 캐시된 Spawner에서 감소시킴. 항상 RefRW.ValueRW를 통해 쓰라.
지연된 인스턴스의 ecb.SetComponent가 무시됨 Instantiate와 SetComponent에 서로 다른 ECB를 사용함. 두 호출은 같은 ECB 인스턴스를 공유해야 한다.

Chapter 6. ECS 핵심 개념

1. 다섯 가지 명사

Entities ECS는 다섯 가지 명사로 귀결된다:

Figure: ECS Core Nouns.

이 다이어그램을 내면화하는 것이 기존 DOTS 코드를 읽는 데 필요한 대부분이다.

2. Entity

Entity는 단순한 64비트 핸들이다: struct Entity { int Index; int Version; }. 데이터도 없고 식별 이외의 메서드도 없다 — 모든 데이터는 Component에 있다.

Entity e = state.EntityManager.CreateEntity();   // fresh, no components
state.EntityManager.AddComponent<Health>(e);     // now it has data
state.EntityManager.DestroyEntity(e);            // handle becomes stale

Version 필드는 오래된 참조를 무효화한다: 같은 Index에서 파괴하고 재생성하면, 이전 Entity 값은 더 이상 일치하지 않는다.

3. Component

Component는 순수 데이터 컨테이너다. 동작을 담지 않는다. 코드는 System에 있다.

public struct Health : IComponentData
{
    public float Value;
    public float Max;
}

여러 Component 변형이 있다 (비관리형, 관리형, 버퍼, 공유, 정리, 태그, Chunk, 활성화 가능). 05_Component Types.md를 참조하라.

4. Archetype

Archetype은 일부 Entity가 공유하는 고유한 Component 타입의 집합이다.

  • { LocalTransform, Health, Enemy }를 가진 Entity는 Archetype A에 속한다.
  • { LocalTransform, Health, Enemy, Burning }을 가진 Entity는 Archetype B에 속한다 — Burning을 추가하면 Component 집합이 달라지기 때문에 다른 Archetype이다.

Archetype 간 이동 (Component 추가/제거)은 구조적 변경이며, Entity를 물리적으로 새 Chunk로 이동시킨다. 13_Structural Change & Safety.md를 참조하라.

5. Chunk

같은 Archetype의 Entity는 16 KB Chunk에 압축된다. Chunk 내에서 각 Component 타입은 연속 배열을 형성한다 — 캐시 지역성에 탁월하다.

사실 의미
Chunk 크기는 16 KB 각 Component가 슬라이스를 차지하므로, Archetype에 Component가 많거나 클수록 Chunk당 Entity 수가 줄어든다.
데이터는 Component별로 열 방향(SoA)으로 배치됨 IJobChunk가 한 번에 하나의 Component 배열을 순회하고 Burst로 벡터화할 수 있다.
Component 추가/제거 시 Entity가 이동됨 구조적 변경. Query 반복자는 순회 중 Archetype이 안정적이라고 가정한다.
Chunk는 Component 변경 버전을 추적함 System은 마지막 실행 이후 변경되지 않은 Chunk를 건너뛸 수 있다 (EntityQuery.SetChangedVersionFilter).

라이브 Chunk 레이아웃은 Window → Entities → Archetypes에서 검사할 수 있다.

6. System

System은 매 프레임 실행되는 로직의 단위로, 일반적으로 Entity Query를 통해 동작한다.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var health in SystemAPI.Query<RefRW<Health>>())
        {
            health.ValueRW.Value = math.min(health.ValueRW.Value + dt, health.ValueRW.Max);
        }
    }
}

System은 자동으로 등록되며 SystemGroup 내에 존재한다. 08_System — ISystem vs SystemBase.md와 09_System Group & Update Order.md를 참조하라.

7. World

World는 다음을 소유하는 컨테이너다:

  • EntityManager (Entity를 생성, 소멸, 쿼리).
  • System 및 SystemGroup 집합.
  • Chunk 자체 (EntityManager를 통해).

기본 런타임 World는 DefaultGameObjectInjectionWorld다. 대부분의 게임은 프로덕션에서 하나의 World를 가진다; 다중 World 설정은 Netcode(서버/클라이언트)나 Editor 도구에서 등장한다. Netcode World 분리에 대해서는 16_Netcode Client-Server World & Bootstrap.md를 참조하라.

World world = World.DefaultGameObjectInjectionWorld;
EntityManager em = world.EntityManager;

Entity e = em.CreateEntity(typeof(Health));
em.SetComponentData(e, new Health { Value = 100, Max = 100 });

8. EntityManager

EntityManager는 구조적 변경을 위한 메인 스레드 API다: Entity 생성, 소멸, Component 추가/제거, Prefab Entity 인스턴스화, World 복사 등. EntityManager를 통한 구조적 변경은 즉각적이며 진행 중인 Query 반복자나 Job 참조를 무효화할 수 있다.

Job이나 반복 도중의 지연된 구조적 변경에는 EntityCommandBuffer를 사용하라 — 14_EntityCommandBuffer · Deferred Entity.md를 참조하라.

9. 종합

World
└── EntityManager
    └── Archetype { LocalTransform, Health, Enemy }
        └── Chunk  (up to ~N entities)
            └── Entity { Index, Version }
                ├── LocalTransform
                ├── Health
                └── Enemy (tag)

{ LocalTransform, Health }를 쿼리하는 System은 Archetype이 두 Component를 모두 포함하는 모든 Chunk를 탐색한다. 위의 Archetype — 그리고 Enemy와 추가 Component를 가진 다른 모든 Archetype도 포함된다.

10. 문제 해결

증상 원인 / 해결
Entity 조회 시 오래된 데이터 반환 Entity가 소멸되고 재생성됨 — Version이 더 이상 일치하지 않는다. Entity를 다시 해결하라.
Query가 아무것도 매칭하지 않지만 Entity에 분명히 Component가 있음 Component가 다른 World에서 추가됐거나, 활성화 가능한 Component가 현재 비활성화 상태다.
Archetype 수가 폭발적으로 증가 고카디널리티의 Entity당 태그로 인한 고유 Component 조합 과다. Shared Component나 활성화 가능한 플래그로 통합하라.
Chunk에 Entity가 1~2개뿐임 Archetype이 지나치게 크다 — Entity에 필요 없는 Component를 제거하거나, 여러 Entity/Archetype으로 분할하라.
반복 중 InvalidOperationException: … invalidated … Query 도중 구조적 변경이 발생. ECB를 사용하거나 루프 이후로 변경을 일괄 처리하라.

Chapter 7. Entity 참조 — Entity · EntityPrefabReference · UnityObjectRef

1. 개요

DOTS 코드는 소수의 참조 타입을 사용한다. 모두 "무언가"를 가리키기 때문에 혼동하기 쉽지만, Authoring에서 런타임 파이프라인의 서로 다른 지점에 존재한다.

타입 네임스페이스 참조 대상 일반적인 사용 예
Entity Unity.Entities 하나의 ECS World 내의 Entity. 런타임 관계 및 Component 필드.
EntityPrefabReference Unity.Entities.Serialization 나중에 로드할 수 있는 Bake된 Entity Prefab 에셋. 스트리밍 및 콘텐츠 워크플로우.
UnityObjectRef<T> Unity.Entities 비관리형 래퍼를 통한 UnityEngine.Object 에셋/객체. ECS 데이터에서 메시, 머티리얼, 텍스처, 오디오 클립 및 Authored 에셋 참조.

이 페이지는 DOTS/Entities 참조 타입에 관한 것이다. 엔진 전반의 UnityEngine.Object 식별자 API는 이 매뉴얼의 범위를 벗어난다.

2. Entity

Entity는 런타임 ECS 핸들이다. 하나의 World 내에서 Component 데이터의 행을 식별하는 비관리형 값이다.

using Unity.Entities;

public struct Target : IComponentData
{
    public Entity Value;
}

중요한 규칙:

  • Entity 값은 그것을 생성한 World 내에서만 의미가 있다.
  • Entity를 소멸시키면 이전 핸들이 무효화된다.
  • Entity 인덱스는 재사용될 수 있다; 방금 쿼리하지 않은 저장된 핸들을 역참조하기 전에 EntityManager.Exists(entity)를 사용하라.
  • 원시 Entity 값을 디스크에 저장하거나 안정적인 ID로 독립적인 World 간에 전달하지 마라.
if (state.EntityManager.Exists(target.Value))
{
    var transform = state.EntityManager.GetComponentData<LocalTransform>(target.Value);
}

3. Baking에서의 Entity 참조

Baker가 Authoring 객체를 ECS 데이터로 변환할 때, GetEntity를 사용하여 참조된 Authoring 객체나 Prefab을 Entity 참조로 변환하라.

using Unity.Entities;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
}

public struct Spawner : IComponentData
{
    public Entity Prefab;
}

public class SpawnerBaker : Baker<SpawnerAuthoring>
{
    public override void Bake(SpawnerAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new Spawner
        {
            Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic)
        });
    }
}

결과는 런타임 ECS 참조다. 관리형 GameObject 참조가 아니다.

4. EntityPrefabReference

EntityPrefabReference는 현재 Scene에 Entity 참조로 직접 임베드되는 대신 Entities 콘텐츠 파이프라인을 통해 로드해야 하는 Prefab 콘텐츠에 사용한다.

using Unity.Entities;
using Unity.Entities.Serialization;
using UnityEngine;

public class ProjectileLibraryAuthoring : MonoBehaviour
{
    public GameObject ProjectilePrefab;
}

public struct ProjectileLibrary : IComponentData
{
    public EntityPrefabReference ProjectilePrefab;
}

public class ProjectileLibraryBaker : Baker<ProjectileLibraryAuthoring>
{
    public override void Bake(ProjectileLibraryAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new ProjectileLibrary
        {
            ProjectilePrefab = new EntityPrefabReference(authoring.ProjectilePrefab)
        });
    }
}

Prefab이 콘텐츠 관리 또는 스트리밍 워크플로우에 속하는 경우 EntityPrefabReference를 사용하라. Prefab이 같은 SubScene에 Bake되어 Scene과 함께 사용 가능한 경우 일반 Entity Prefab 참조를 사용하라.

5. UnityObjectRef<T>

UnityObjectRef<T>는 Component를 관리형으로 만들지 않고 비관리형 ECS 데이터가 UnityEngine.Object를 참조할 수 있게 한다.

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

Authoring 데이터에서 Bake하라:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

메인 스레드나 관리형 프레젠테이션 코드에서만 .Value를 역참조하라:

Mesh mesh = visual.Mesh.Value;
Material material = visual.Material.Value;

규칙:

  • 저장된 래퍼는 비관리형이므로 IComponentData 내에 존재할 수 있다.
  • 대상 객체는 여전히 관리형 Unity 객체다; .Value는 Burst Job 작업이 아니다.
  • 역참조할 때 참조된 객체가 여전히 로드되어 있어야 한다.

6. 올바른 참조 선택

필요 사항 사용
같은 World에서 한 Entity에서 다른 Entity를 가리킴 Entity
같은 SubScene에 Bake된 Prefab 저장 GetEntity(authoring.Prefab, ...)의 Entity Prefab 참조
콘텐츠 스트리밍/지연 로딩용 Prefab 저장 EntityPrefabReference
비관리형 ECS 데이터에 메시, 머티리얼, 텍스처, 오디오 클립, ScriptableObject 참조 저장 UnityObjectRef<T>
세션 간 Entity 유지 원시 Entity가 아닌 게임 소유의 안정적인 ID

7. 문제 해결

증상 원인 / 해결
저장된 Entity가 더 이상 존재하지 않음 Entity가 소멸됐거나 다른 World에 속함. 다시 해결하거나 EntityManager.Exists를 확인하라.
Baking 후 Prefab 참조가 Entity.Null임 Baker가 Prefab에 대해 GetEntity를 호출하지 않았거나, Prefab이 Bake된 Scene/콘텐츠 워크플로우에 포함되지 않았음.
UnityObjectRef<T>.Value가 null 반환 참조된 Unity 객체가 로드되지 않았거나 소멸됨. 역참조를 보호하고 에셋을 다시 로드/해결하라.
Burst Job이 메시/머티리얼 참조를 사용할 수 없음 UnityObjectRef<T> 저장은 비관리형이지만 .Value는 관리형 Unity 객체에 접근함. 역참조를 메인 스레드 프레젠테이션 코드로 이전하라.
저장 파일에 원시 Entity 값을 저장 원시 Entity 핸들은 World 지역적이고 안정적이지 않다. 게임 소유 ID나 콘텐츠 키를 저장하라.

Chapter 8. Component 타입

1. 개요

ECS에서 "Component"는 Entity에 붙은 모든 데이터 컨테이너다. Entities는 저장 방식, 수명, 비용 프로필이 다른 여러 종류를 제공한다. 올바른 것을 선택하는 것이 ECS System 설계 작업의 절반이다.

빠른 맵:

인터페이스 / 형태 저장 변경 시 구조적 변경? 일반적인 사용 예
IComponentData (비관리형 struct) Chunk 내, SoA 아니오 (값 변경 시) 기본 Component
IComponentData (관리형 class) 사이드 테이블 아니오 참조 타입 (메시, GameObject 참조)
IBufferElementData Chunk 내 + 힙 오버플로우 아니오 Entity당 동적 배열
ISharedComponentData Chunk당 예 (Entity를 새 Chunk로 이동) 설정을 공유하는 Entity 그룹화
ICleanupComponentData Chunk 내 일반 Entity 소멸 정리 프로토콜
ICleanupSharedComponentData Chunk당 일반 공유 리소스 정리 프로토콜
태그 (빈 IComponentData) Chunk 내 크기 0 아니오 필터 마커, 데이터 없음
Chunk Component Chunk당 예 Chunk당 한 번 계산되는 메타데이터
활성화 가능 (IEnableableComponent) Chunk 내 + 비트 마스크 아니오 Entity별 동작 켜기/끄기

이 페이지의 나머지 부분에서 각 행을 자세히 설명한다.

2. 비관리형 IComponentData

기본값. Chunk 내에 촘촘히 압축된 SoA 데이터로 직접 존재한다. Burst 친화적이다.

public struct Velocity : IComponentData
{
    public float3 Value;
}

규칙:

  • struct여야 한다 — 참조 필드 불가 (class, string, List<T> 없음).
  • 허용되는 필드 타입: 기본 타입, Unity.Mathematics, 기타 비관리형 struct, Entity, FixedString*, BlobAssetReference<T>, UnityObjectRef<T>.
  • SystemAPI.Query에서 RefRO<T>로 읽고, RefRW<T>로 쓴다.

3. 관리형 IComponentData

Entity로 인덱싱된 사이드 테이블에 존재한다. 참조 타입과 작동하지만 GC 할당 비용이 발생하고 해당 Component에 대한 Burst 호환성을 깨뜨린다.

public class EnemyAI : IComponentData
{
    public AIModel Brain;      // a managed class
    public List<Entity> Allies;
}

관리형 참조(메시, 텍스처, Authored ScriptableObject)가 실제로 필요한 경우에 사용하라. UnityEngine.Object에 대한 약한 참조만 필요하다면 비관리형 Component 내의 UnityObjectRef<T>를 선호하라.

4. IBufferElementData — 동적 버퍼

DynamicBuffer<T>는 Entity당 크기 조정 가능한 배열이다.

public struct Waypoint : IBufferElementData
{
    public float3 Position;
}

var buffer = state.EntityManager.AddBuffer<Waypoint>(entity);
buffer.Add(new Waypoint { Position = new float3(1, 0, 0) });

저장: InternalBufferCapacity까지의 요소는 Chunk 내에 존재하며, 그 이상은 힙으로 넘친다. [InternalBufferCapacity(N)]으로 용량을 조정하라.

5. ISharedComponentData — 그룹화

Shared Component는 Chunk를 그룹화하는 값을 전달한다. 같은 ISharedComponentData 값을 가진 두 Entity는 같은 Chunk에 들어가고, 값이 다르면 다른 Chunk가 생성된다.

public struct RenderMeshGroup : ISharedComponentData
{
    public int MaterialID;
}

결과:

  • 값을 변경하면 Entity가 다른 Chunk로 이동 → 구조적 변경, 비용이 크다.
  • 자주 변경되지 않지만 그룹별로 달라지는 데이터(머티리얼 세트, 팀 ID, Bake 영역)에 적합하다.
  • 타입에 참조가 포함된 경우 값은 관리형 테이블에 저장된다 (ISharedComponentData는 관리형일 수 있음).

6. 정리 Component — ICleanupComponentData

Entity가 "소멸"될 때 System이 처리할 때까지 정리 Component만 남기고 살아있게 Entity를 표시한다.

public struct ConnectionCleanup : ICleanupComponentData
{
    public int ConnectionHandle;
}

// DestroyEntity(e) removes all normal components but leaves ConnectionCleanup
// behind so you can close the socket in a dedicated system, then destroy the
// cleanup component explicitly.

형제 타입 ICleanupSharedComponentData는 공유 리소스에 동일한 프로토콜을 적용한다.

사용 사례: 네이티브 리소스 해제, 외부 시스템 해제 등록, 여러 프레임에 걸친 단계적 소멸.

7. 태그 Component

태그는 빈 IComponentData다:

public struct Enemy : IComponentData {}

Chunk에서 0바이트이며 다른 Component와 마찬가지로 쿼리 가능하다. "X의 타입" 마커에 이상적이다. 태그를 남용하면(수백 개의 고유한 것) Archetype이 분열된다 — 고카디널리티 상태에는 활성화 가능한 Component나 Shared Component를 사용하라.

8. Chunk Component

Chunk Component는 Entity당이 아닌 Chunk당 하나의 값을 전달한다.

state.EntityManager.AddChunkComponentData<RegionBounds>(query, new RegionBounds { ... });

일반적인 사용: Chunk의 모든 Entity에 적용되는 미리 계산된 데이터(경계, 지역 ID, 파티션 키). Chunk Component를 변경하면 구조적 변경이 발생한다.

9. 활성화 가능한 Component (IEnableableComponent)

자세한 내용은 전용 페이지 06_Enableable Component.md를 참조하라. 요약: 구조적 변경 없이 Chunk당 비트 마스크를 통해 Entity별로 켜기/끄기를 토글할 수 있는 Component다.

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

state.EntityManager.SetComponentEnabled<Stunned>(entity, false);  // no structural change

Query는 활성화 가능한 Component가 비활성화된 Entity를 자동으로 건너뛴다 — System에서 Entity를 일시적으로 제외하는 저렴하고 일반적인 방법이다.

프레임당 Component 켜기/끄기 토글 활성화 가능한 Component
단순히 타입 표시 태그 (빈 IComponentData)
Chunk 전체에 대한 데이터 미리 계산 Chunk Component
Entity 소멸 시 코드 실행 정리 Component

11. 문제 해결

증상 원인 / 해결
InvalidOperationException: The type T must be unmanaged 비관리형이 필요한 곳에 관리형 타입을 사용함. Component나 API를 바꾸라 (예: ComponentLookup<T> vs ManagedAPI.GetComponent<T>).
Shared Component 값 설정 시 끊김 변경 때마다 구조적 변경이 발생. ECB로 일괄 처리하거나 이 값이 드물게 변경됨을 수용하라.
DynamicBuffer<T>가 힙으로 넘침 [InternalBufferCapacity(N)]이 사용량에 비해 너무 작음. 늘리거나 크기를 제한하라.
DestroyEntity 후 Entity가 다시 나타남 정리 Component가 아직 붙어 있음. 정리가 완료되면 명시적으로 제거하라.
Archetype이 너무 많아짐 태그 폭발. Entity당 고유한 "상태"를 활성화 가능한 Component로 변환하라.
태그 Component가 Query를 필터링하지 않음 태그 Component는 데이터가 없으므로 SystemAPI.Query<...> 튜플에서 RefRW<Tag> / RefRO<Tag>를 요청할 수 없다. 대신 .WithAll<Tag>() / .WithNone<Tag>()를 통해 적용하라.

Chapter 9. 활성화 가능한 Component

1. 활성화 가능한 Component가 존재하는 이유

Component를 추가하거나 제거하는 것은 구조적 변경이다: Entity가 다른 Chunk로 이동하고, Query는 다시 해결해야 하며, 참조가 무효화될 수 있다. 이것을 Entity당 매 프레임 수행하면(예: "Stunned" 켜기/끄기 토글), 그 비용이 프레임을 지배하게 된다.

활성화 가능한 Component는 구조적 변경 없이 동작을 켜고 끄는 방법을 제공한다. Component는 항상 Chunk에 존재하며, Chunk당 마스크의 비트가 Query 관점에서 각 Entity가 그것을 "가지고 있는지"를 결정한다.

2. 인터페이스

IComponentData와 IEnableableComponent 모두를 구현한다:

using Unity.Entities;

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

IBufferElementData도 활성화 가능하게 만들 수 있다.

3. 토글

System에서:

// Turn Stunned off without removing the component
state.EntityManager.SetComponentEnabled<Stunned>(entity, false);

// From an ECB (deferred, safe mid-iteration)
ecb.SetComponentEnabled<Stunned>(entity, true);

// From a job with a ComponentLookup
var stunnedLookup = SystemAPI.GetComponentLookup<Stunned>();
stunnedLookup.SetComponentEnabled(entity, false);

현재 상태 읽기:

bool isStunned = state.EntityManager.IsComponentEnabled<Stunned>(entity);

세 가지 경로 모두 비트 마스크에서 동작하므로 구조적 변경이 발생하지 않는다.

4. Query 동작

SystemAPI.Query<...>와 EntityQuery 기반 반복은 활성화 가능한 Component가 현재 비활성화된 Entity를 건너뛴다 — 기본적으로 나열된 모든 Component가 활성화된 Entity만 볼 수 있다.

foreach (var stunned in SystemAPI.Query<RefRW<Stunned>>())
{
    // only fires for entities where Stunned is enabled
}

비활성화된 Entity도 볼 필요가 있다면 재정의하라:

foreach (var (stunnedRW, entity) in SystemAPI
             .Query<RefRW<Stunned>>()
             .WithEntityAccess()
             .WithOptions(EntityQueryOptions.IgnoreComponentEnabledState))
{
    // fires for ALL entities with the component, enabled or not
}

5. 예제 — 만료되는 기절

단일 IJobEntity는 같은 Component에 대해 ref Stunned (RefRW<T>) 와 EnabledRefRW<Stunned>를 동시에 받을 수 없다 — Entities는 하나의 Job에서 같은 Component를 두 형태로 래핑하는 것을 명시적으로 허용하지 않는다. 두 가지 유효한 패턴:

5.1 값 읽기 + 쓰기, ECB를 통해 활성화 비트 토글

[BurstCompile]
public partial struct StunTickJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Stunned stunned)
    {
        stunned.Duration -= DeltaTime;
        if (stunned.Duration <= 0f)
            ECB.SetComponentEnabled<Stunned>(chunkIndex, entity, false);
    }
}

Playback은 다음 ECB 경계에서 발생한다 (예: EndSimulationECBS) — 14_EntityCommandBuffer · Deferred Entity.md를 참조하라. 구조적 변경 없이 알려진 지점에서 비트 마스크 전환만 수행한다.

5.2 Component 값이 필요 없을 때 활성화 비트를 직접 토글

[BurstCompile]
public partial struct StunDisableJob : IJobEntity
{
    void Execute(EnabledRefRW<Stunned> stunnedEnabled)
    {
        stunnedEnabled.ValueRW = false;   // no structural change
    }
}

EnabledRefRW<T>는 이 Job에서 유일한 Stunned 참조이므로 컴파일된다. System이 순수한 스위처일 때 이 형태를 사용하라.

한 번에 값을 읽고 같은 Entity의 활성화 비트를 토글하고 싶은가? 메인 스레드에서 EntityManager.SetComponentEnabled<T>(entity, false)로 수행하거나, state.Dependency를 통해 연결된 두 Job으로 분리하라.

6. 활성화 가능 vs 추가/제거 선택 시

상황 선택
고빈도 토글 (프레임당, 이벤트당) 활성화 가능
항상 존재하지 않는 상태 추가 (가변 필드를 가진 Entity당 수정자) 활성화 가능
저빈도 Archetype 변경 (예: 영구적인 기능 추가) 일반적으로 Component 추가/제거
많은 System에 균일하게 필터링 활성화 가능 (Query가 자동으로 처리)

일반 규칙: "이 Component를 가지고 있음" 비트가 Entity 수명 동안 여러 번 전환된다면 활성화 가능하게 만들어라.

7. 다른 기능과의 상호작용

기능 동작
WithChangeFilter<T> 활성화 비트를 토글하면 Component의 변경 버전도 증가한다.
ICleanupComponentData 정리 Component는 활성화 가능하게 만들 수 없다.
Netcode for Entities 활성화 가능한 Component는 ghost 동기화될 수 있지만, 토글은 권위적인 측을 통해야 한다.
직렬화 / World 내보내기 활성화 비트가 보존된다.

8. 문제 해결

증상 원인 / 해결
Query가 "꺼짐" 상태여야 하는 Entity를 포함 Component가 IEnableableComponent를 구현하지 않았거나, WithOptions(EntityQueryOptions.IgnoreComponentEnabledState)가 설정됨.
SetComponentEnabled<T>가 "type T does not support being enableable" 오류를 던짐 struct가 IComponentData만 구현함 — IEnableableComponent를 추가하라.
EnabledRefRW<T> 매개변수가 소스 생성기에서 거부됨 Job이나 System에 partial이 없거나, Component가 활성화 가능하지 않음.
소스 생성기 오류 "cannot have both RefRW and EnabledRefRW for the same component" 로직을 분리하라 — Job당 ref T 또는 EnabledRefRW<T>를 사용하되, 같은 Component에 둘 다 사용하지 마라. §5의 두 패턴을 참조하라.
변경 필터가 산발적으로 발생 Enabled를 토글하면 WithChangeFilter 목적상 변경으로 간주된다. 예상된 동작 — 토글 대신 EnabledRefRO로 읽기와 결합하라.
ECB의 SetComponentEnabled가 무시됨 그것을 생성한 ECB가 플러시되기 전에 지연된 Entity에서 호출함. 둘 다 같은 ECB 사이클 내에 있어야 한다.

Chapter 10. Singleton Component

1. Singleton이란 (그리고 아닌 것)

Entities에서 Singleton은 World에서 정확히 하나의 Entity에만 존재하는 Component다. 특별한 ISingleton 인터페이스는 없다 — 모든 IComponentData는 관례적으로 Singleton이 되며, SystemAPI는 "정확히 하나" 규칙을 확인하는 편리한 접근자를 제공한다.

일반적인 사용: 게임 상태, 전역 설정, 시간/틱 소스, 네트워크 상태, 중앙 ECB 등록.

public struct GameState : IComponentData
{
    public int   Score;
    public float TimeScale;
    public bool  Paused;
}

2. 접근자

모든 ISystem / SystemBase 내부에서:

// Reads
GameState gs      = SystemAPI.GetSingleton<GameState>();
RefRW<GameState> gsRW  = SystemAPI.GetSingletonRW<GameState>();
bool has          = SystemAPI.HasSingleton<GameState>();
bool got          = SystemAPI.TryGetSingleton<GameState>(out var gs2);

// Writes
SystemAPI.SetSingleton(new GameState { Score = 10 });

// Entity handle, if you need it
Entity e          = SystemAPI.GetSingletonEntity<GameState>();

// Buffers work too
DynamicBuffer<Waypoint> buf = SystemAPI.GetSingletonBuffer<Waypoint>();

Get* 변형은 Singleton이 없거나 둘 이상 있을 경우 예외를 던진다. TryGet*는 그 경우 false를 반환한다. RW 변형은 변경 추적을 위해 Component를 변경됨으로 표시한다.

3. Singleton 생성

두 가지 일반적인 패턴:

3.1 SubScene에서 Bake

public class GameStateAuthoring : MonoBehaviour
{
    public float InitialTimeScale = 1f;

    class Baker : Baker<GameStateAuthoring>
    {
        public override void Bake(GameStateAuthoring authoring)
        {
            var e = GetEntity(TransformUsageFlags.None);
            AddComponent(e, new GameState
            {
                Score     = 0,
                TimeScale = authoring.InitialTimeScale,
                Paused    = false
            });
        }
    }
}

SubScene의 단일 GameObject에 GameStateAuthoring을 붙이면, Baker가 Singleton을 가진 정확히 하나의 Entity를 생성한다.

3.2 System 시작 시 생성

public void OnCreate(ref SystemState state)
{
    var e = state.EntityManager.CreateEntity(typeof(GameState));
    state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
}

Authoring에 있으면 안 되는 데이터(서버 주입 설정, 절차적으로 생성된 것 등)에 사용하라.

4. Singleton에 System 게이팅

Singleton이 존재한 후에만 System이 의미가 있다면:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<GameState>();
}

RequireForUpdate<T>()는 Component가 없을 때 OnUpdate를 완전히 건너뛰도록 System에 지시한다. OnUpdate 내부의 방어적 null 체크를 피할 수 있다.

5. 읽기 전용 vs 읽기-쓰기 성능

  • GetSingleton<T>() — 복사본을 읽는다. 변경 추적 없음.
  • GetSingletonRW<T>() — Chunk를 가리키는 RefRW<T>를 반환한다. .ValueRW에 할당하면 Component를 변경됨으로 표시한다. 다른 System이 WithChangeFilter<T>를 사용하는 경우에 원하는 동작이다.
  • SetSingleton — 전체 쓰기, 항상 버전을 올린다.

읽기 위주 System에서는 GetSingleton을 선호하라; 실제로 변경할 때만 GetSingletonRW로 전환하라.

6. 여러 후보 — "Singleton이 둘 이상" 함정

GetSingleton<T>는 같은 Singleton Component를 가진 Entity가 두 개 있을 때 InvalidOperationException: expected exactly one entity with component T, found N을 던진다. 일반적인 원인:

  • SubScene의 두 GameObject 모두에 Authoring 스크립트가 붙어 있음.
  • 테스트 픽스처가 런타임에 Singleton을 생성했고 SubScene도 그것을 Bake함.
  • System이 HasSingleton 가드 없이 OnCreate에서 Singleton을 생성함.

방어적 패턴:

public void OnCreate(ref SystemState state)
{
    if (!SystemAPI.HasSingleton<GameState>())
    {
        var e = state.EntityManager.CreateEntity(typeof(GameState));
        state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
    }
}

7. 버퍼 Singleton

DynamicBuffer<T>도 Singleton이 될 수 있다:

public struct QueuedEvent : IBufferElementData
{
    public int Type;
    public int Payload;
}

// Access:
DynamicBuffer<QueuedEvent> events = SystemAPI.GetSingletonBuffer<QueuedEvent>();
events.Add(new QueuedEvent { Type = 1 });

전역 이벤트 큐와 명령 스트림에 유용하다.

8. 문제 해결

증상 원인 / 해결
InvalidOperationException: expected exactly one entity with component T, found 0 Singleton Entity가 생성되지 않음. Baker를 추가하거나 HasSingleton 가드와 함께 OnCreate에서 생성하라.
... found 2 두 Entity가 Component를 가짐. Authoring에서 중복을 제거하거나 시작 시 정리하라.
GetSingletonRW가 Component를 변경됨으로 표시하지 않는 것 같음 .ValueRO에 쓰거나 RefRW를 무시함. .ValueRW만 변경 추적을 트리거한다.
System이 Singleton 생성 전에 실행됨 RequireForUpdate<T>를 사용하거나 생성자가 먼저 실행되도록 System을 순서화하라 ([UpdateBefore]).
순서 없이 두 System에서 Singleton이 수정됨 [UpdateAfter]/[UpdateBefore]를 추가하거나 로직을 합치라. Singleton의 경쟁 조건은 특히 디버그하기 어렵다.
Netcode: 클라이언트와 서버 간 Singleton 값이 달라짐 Singleton을 ghost 동기화하거나, 동기화된 입력에서 결정론적으로 재구성하라.

Chapter 11. System — ISystem vs SystemBase

1. 개요

Entities에는 System을 선언하는 두 가지 방법이 있다: ISystem (비관리형 struct)과 SystemBase (관리형 class). SystemBase가 필요한 특별한 이유가 없다면 ISystem을 선택하라.

ISystem SystemBase
선언 partial struct : ISystem partial class : SystemBase
메모리 비관리형, 스택 할당 가능 관리형, GC 소유
Burst 컴파일 가능 예 ([BurstCompile] 추가) 아니오
관리형 필드 아니오 예
관리형 API 접근 EntityManager + ManagedAPI를 통해 직접
기본 선택 ✅ 반드시 필요한 경우에만

2. ISystem — 기본값

using Unity.Burst;
using Unity.Entities;

[BurstCompile]
public partial struct EnemyAISystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        // Called once when the system is created.
        state.RequireForUpdate<EnemySpawnConfig>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var config = SystemAPI.GetSingleton<EnemySpawnConfig>();
        // ... logic
    }

    [BurstCompile]
    public void OnDestroy(ref SystemState state)
    {
        // Called once when the system is destroyed.
    }
}

필수 사항:

  • Struct는 partial이어야 한다 — 소스 생성기가 숨겨진 래퍼를 추가한다.
  • 세 메서드(OnCreate, OnUpdate, OnDestroy) 모두 선택 사항 — 필요 없는 것은 생략하라.
  • Struct와 Burst 컴파일하려는 각 메서드에 [BurstCompile].

3. SystemBase — 관리형 System이 불가피할 때

using Unity.Entities;

public partial class PlayerInputSystem : SystemBase
{
    private InputActions _input;   // managed field — the reason SystemBase exists

    protected override void OnCreate()
    {
        _input = new InputActions();
        _input.Enable();
    }

    protected override void OnUpdate()
    {
        var move = _input.Gameplay.Move.ReadValue<Vector2>();
        // ... dispatch to components
    }

    protected override void OnDestroy() => _input.Dispose();
}

SystemBase를 선택해야 할 때:

  • System과 함께 살아야 하는 관리형 필드를 보유할 때 (Input System 객체, UI Toolkit 핸들, 관리형 서비스 등).
  • EntityManager를 통해 마샬링하지 않고 OnUpdate 내부에서 관리형 API를 직접 호출해야 할 때.

SystemBase의 모든 것은 Burst 없이 메인 스레드에서 실행된다.

4. System 수명 주기

두 종류 모두 같은 훅 이름을 공유한다:

Figure: System Lifecycle.

훅 실행 시점 일반적인 사용 예
OnCreate System 생성 시 한 번 Query 캐시, RequireForUpdate<T>, 네이티브 컨테이너 할당
OnStartRunning Query가 비어있음 → 비어있지 않음으로 변경 시 세션별 상태 리셋
OnUpdate 부모 SystemGroup에서 매 프레임 주요 작업

RequireForUpdate<T>()

필요한 Component나 Singleton이 없을 때 OnUpdate를 완전히 건너뛰도록 한다:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<EnemySpawnConfig>();
    state.RequireForUpdate<GameRunning>();
}

이것이 "설정이 존재함" / "게임이 시작됨" 조건으로 System 실행을 제한하는 표준 방법이다.

5. 둘 중 선택

경험 법칙:

  1. ISystem으로 시작하라. Burst 친화적이고 GC 없다.
  2. 관리형 필드가 강제할 때만 SystemBase로 전환하라. 그래도 분리를 고려하라: 결과를 Component에 밀어넣는 얇은 SystemBase와 해당 Component를 읽는 무거운 로직을 처리하는 ISystem으로 분리하라.
  3. ISystem struct에 관리형 필드를 추가하여 혼합하지 마라 — 컴파일되지 않는다. 위의 분리가 관용적인 해결책이다.

예제 분리:

// Managed — bridges Input System into components.
public partial class InputBridgeSystem : SystemBase
{
    private InputActions _input;
    // ... fills a PlayerInput singleton each frame
}

// Unmanaged + Burst — reads the PlayerInput singleton and drives gameplay.
[BurstCompile]
public partial struct PlayerMoveSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) { /* ... */ }
}

6. SystemState 빠른 참조

ISystem 내부에서, ref SystemState state는 World로의 창이다.

멤버 목적
state.EntityManager 메인 스레드 구조적 변경.
state.World 소유 World.
state.Dependency 현재 Job 의존성 핸들 — 예약된 Job과 결합하라.
state.GetEntityQuery(...) 캐시된 Query 빌드 또는 검색.
state.RequireForUpdate<T>() Component의 존재 여부로 System을 게이팅하라.
state.Enabled false로 설정하면 System을 소멸시키지 않고 건너뛴다.

SystemBase에서는 이것들이 클래스 자체의 멤버다 (예: this.EntityManager).

Chapter 12. System Group & Update Order

1. 개요

모든 System은 SystemGroup 내에 존재한다. Group은 자식들이 매 프레임 언제 실행될지를 결정하고, 순서 지정 속성은 Group 내에서의 순서를 결정한다. 그룹화를 올바르게 설정하는 것이 프레임 지연 쓰기, 경쟁 조건, "왜 이 Entity가 떨리지" 버그를 피하는 방법이다.

2. 내장 Group

Figure: System Group and Update Order.

Group 실행 시점 일반적인 내용
InitializationSystemGroup 프레임당 한 번, 시뮬레이션 전 일회성 초기화, 부트스트랩 작업
SimulationSystemGroup 프레임당 한 번 대부분의 게임플레이 로직
FixedStepSimulationSystemGroup 고정 타임스텝 (기본 60 Hz); 프레임당 0, 1, 또는 N번 실행 가능 물리, 결정론적 스텝 로직
PresentationSystemGroup 프레임당 한 번, 시뮬레이션 후 렌더링 준비, 보간

Netcode for Entities는 더 추가한다 (PredictedSimulationSystemGroup, GhostSendSystem 등) — 위의 것들 내부에 중첩된다. 16_Netcode Client-Server World & Bootstrap.md와 19_Netcode Prediction & Rollback.md를 참조하라.

3. Group에 System 배치

using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial struct EnemyAISystem : ISystem { /* ... */ }

[UpdateInGroup]을 생략하면 System은 기본적으로 SimulationSystemGroup에 속한다.

중첩

Group은 다른 Group 내부에 존재할 수 있다:

[UpdateInGroup(typeof(SimulationSystemGroup), OrderFirst = true)]
public partial class GameplaySystemGroup : ComponentSystemGroup { }

[UpdateInGroup(typeof(GameplaySystemGroup))]
public partial struct MovementSystem : ISystem { /* ... */ }

ComponentSystemGroup은 커스텀 Group의 기본 클래스다. System과 컨테이너 역할을 모두 한다.

4. Group 내 순서 지정

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateAfter(typeof(EnemyAISystem))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct EnemyPathfindingSystem : ISystem { /* ... */ }
속성 의미
[UpdateBefore(typeof(X))] X보다 먼저 실행 (둘 다 같은 Group에 있어야 함).
[UpdateAfter(typeof(X))] X 이후에 실행.
OrderFirst = true ([UpdateInGroup]에서) Group에서 가능한 한 먼저 실행.
OrderLast = true 가능한 한 나중에 실행.

팁: OrderFirst/Last보다 [UpdateAfter] 체인을 선호하라 — 코드 리뷰에서 더 잘 읽히고 리팩토링에도 견고하다.

5. [RequireMatchingQueriesForUpdate]

System의 각 Query에 적어도 하나의 Entity가 매칭될 때만 OnUpdate를 실행하도록 지시하는 최신 속성 (Entities 1.x+):

[RequireMatchingQueriesForUpdate]
public partial struct EnemyAISystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Only runs when there is at least one entity matching every query the system uses.
    }
}

특정 Component/Singleton을 요구하는 state.RequireForUpdate<T>()와 비교하라. [RequireMatchingQueriesForUpdate]는 더 묵시적이다 — System이 가진 Query에 기반한다.

6. 커스텀 Group

생성 이유:

  • 응집력 있는 게임플레이 단계 (예: "애니메이션 전에 데미지 적용").
  • 다른 틱 레이트 (예: 30 Hz의 커스텀 고정 스텝).
  • 기능 토글 — 전체 Group 활성화/비활성화.
using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial class DamagePipelineGroup : ComponentSystemGroup
{
    // You can override OnUpdate to customise how the group ticks.
}

[UpdateInGroup(typeof(DamagePipelineGroup), OrderFirst = true)]
public partial struct ApplyDamageSystem : ISystem { /* ... */ }

[UpdateInGroup(typeof(DamagePipelineGroup))]
[UpdateAfter(typeof(ApplyDamageSystem))]
public partial struct DeathSystem : ISystem { /* ... */ }

커스텀 틱 레이트

public partial class SlowTickGroup : ComponentSystemGroup
{
    private float _accum;
    protected override void OnUpdate()
    {
        _accum += (float)SystemAPI.Time.DeltaTime;
        if (_accum < 0.1f) return;      // 10 Hz
        _accum = 0f;
        base.OnUpdate();                // tick children
    }
}

7. 고정 vs 가변 타임스텝

결정론이 중요한 경우 (물리, 네트워크 예측, 결정론적 Playback), System을 FixedStepSimulationSystemGroup에 배치하라:

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial struct PhysicsStepSystem : ISystem { /* ... */ }

주의 사항:

  • FixedStepSimulationSystemGroup은 실시간을 따라잡기 위해 프레임당 여러 번 (또는 0번) 실행될 수 있다.
  • SystemAPI.Time.DeltaTime을 사용하라 — 이 Group 내에서는 고정 스텝의 dt다.
  • 입력을 읽거나 프레젠테이션을 생성하는 것은 SimulationSystemGroup / PresentationSystemGroup에 있어야 한다.

8. 런타임에서 순서 검사

Window → Entities → Systems에서 표시되는 것:

  • 모든 Group과 자식들, 실제 실행 순서.
  • System당 타이밍.
  • 활성화/비활성화 상태 (런타임에 토글하는 System).

속성으로 선언한 순서가 실제와 다르다면, 거의 항상 순환 의존성이나 누락된 [UpdateInGroup] 때문이다.

9. 문제 해결

증상 원인 / 해결
System이 잘못된 프레임 단계에서 실행됨 [UpdateInGroup]이 없거나 잘못됨. 기본값은 SimulationSystemGroup이다.
[UpdateAfter]가 무시됨 대상 System이 다른 Group에 있음. 둘 다 같은 Group을 공유해야 한다.
시작 시 순환 의존성 오류 [UpdateBefore] / [UpdateAfter] 체인이 사이클을 형성함. Systems 창에서 사이클을 출력한다.

커스텀 Group이 계속 실행됨 OnUpdate 재정의에서 base.OnUpdate()를 호출하지 않음.
System이 실행되지만 아무것도 하지 않음 RequireForUpdate<T>가 충족되지 않았거나 모든 Query가 비어있음. Window → Entities → Query를 확인하라.

Chapter 13. JobSystem & Burst

1. 개요

DOTS 성능은 Entities 이전부터 존재했고 여전히 기반이 되는 두 가지 엔진 기능에 의존한다:

  • Job System — 의존성 그래프를 통해 워커 스레드에 작업을 예약한다.
  • Burst Compiler — C# Job을 최적화된 네이티브 코드(SIMD, GC 없음, 가상 디스패치 없음)로 컴파일한다.

둘 다 Entities 특정 개념이 아니다. IJobEntity와 IJobChunk는 IJob + Burst 위의 얇은 레이어다. 기본 레이어를 이해하면 모든 상위 수준 패턴이 명확히 읽힌다.

2. Job System — 핵심 타입

인터페이스 형태 사용 시기
IJob void Execute() 일회성 단일 스레드 작업.
IJobParallelFor void Execute(int index) 0에서 N까지 병렬 루프.
IJobParallelForBatch void Execute(int start, int count) 병렬 배치 루프 (인덱스당보다 캐시 친화적).
IJobChunk void Execute(ArchetypeChunk chunk, int chunkIndex, bool useEnabledMask, in v128 enabledMask) 병렬로 ECS Chunk 반복.
IJobEntity void Execute([component params]) ECS Entity 반복 — 소스 생성을 통해 Chunk Job으로 변환됨.

이 모두 struct이며 Burst를 지원한다.

최소 IJobParallelFor 예제

using Unity.Burst;
using Unity.Collections;
using Unity.Jobs;

[BurstCompile]
struct ScaleJob : IJobParallelFor
{
    public NativeArray<float> Data;
    public float Factor;

    public void Execute(int index) => Data[index] *= Factor;
}

// Scheduling:
var handle = new ScaleJob { Data = arr, Factor = 2f }.Schedule(arr.Length, 64);
handle.Complete();

참고:

  • 필드는 블리터블이어야 한다.
  • 여러 Job에서 같은 NativeArray에 쓰기 접근은 의존성 체인이 필요하다 (아래 참조).

3. 예약 및 의존성

JobHandle a = jobA.Schedule();
JobHandle b = jobB.Schedule(a);              // b runs after a
JobHandle c = jobC.Schedule(JobHandle.CombineDependencies(a, b));
c.Complete();

규칙:

  • 모든 Schedule()은 JobHandle을 반환한다. 새 Job이 공유 데이터를 읽거나 쓰는 경우 이전 핸들을 의존성으로 전달하라.
  • 메인 스레드가 결과를 읽기 전에 마지막 핸들에 .Complete()를 절대 건너뛰지 마라 (Safety System이 이를 강제한다).
  • ISystem에서 state.Dependency는 System 수준 핸들이다; .Schedule(state.Dependency)를 사용하거나 IJobEntity.ScheduleParallel(...)과 같은 소스 생성 헬퍼에 의존하면 예약된 Job이 자동으로 결합된다.

ISystem 내부에서

[BurstCompile]
public void OnUpdate(ref SystemState state)
{
    var job = new MyJob { Dt = SystemAPI.Time.DeltaTime };
    state.Dependency = job.ScheduleParallel(state.Dependency);
}

명시적 의존성 없이 IJobEntity.ScheduleParallel()을 사용하면 state.Dependency와 묵시적으로 결합된다. 여러 Job을 체인할 때는 명시적으로 하라.

4. Burst — 무엇이고 왜인가

Burst는 C#의 서브셋 ([BurstCompile]으로 표시된 메서드, GC 할당 없음, 블리터블 데이터)을 LLVM을 통해 최적화된 네이티브 코드로 컴파일한다.

Burst가 잘 컴파일하는 것

  • [BurstCompile]이 태그된 Struct IJob* / ISystem 메서드.
  • Unity.Mathematics 타입의 수학 (float3, quaternion, int4 등).
  • NativeArray<T>, NativeList<T>, DynamicBuffer<T>, ArchetypeChunk 열 배열에 대한 타이트한 루프.

Burst가 컴파일할 수 없는 것

  • 관리형 참조 (class, string, List<T>, 비관리형이 아닌 모든 것).
  • 예외 (잡을 수는 있지만 일반적으로 피하라).
  • 인터페이스를 통한 동적 디스패치 (명시적 제네릭 인스턴스화는 괜찮음).
  • System.IO / Reflection / 수학/기본 타입 이상의 BCL 대부분.

Burst 활성화

using Unity.Burst;

[BurstCompile]
public partial struct MyJob : IJobEntity
{
    // attribute on the struct marks the type for Burst
    // methods need [BurstCompile] too if you want each compiled
    [BurstCompile]
    void Execute(ref Velocity v) { /* ... */ }
}

System의 경우:

[BurstCompile]
public partial struct MySystem : ISystem
{
    [BurstCompile] public void OnCreate(ref SystemState state) { }
    [BurstCompile] public void OnUpdate(ref SystemState state) { }
}

타입의 속성만으로는 충분하지 않다 — 컴파일하려는 각 메서드에도 [BurstCompile]이 필요하다.

Burst Inspector

Window → Analysis → Burst Inspector에서 메서드당 LLVM IR, 어셈블리, 벡터 사용량을 보여준다. "왜 빠르지 않은가"가 질문일 때 사용하라.

5. 네이티브 컨테이너

Burst와 Safety System은 관리형 컬렉션 대신 네이티브 컨테이너를 기대한다:

관리형 네이티브 동등물
T[] NativeArray<T>
List<T> NativeList<T>
Dictionary<K,V> NativeHashMap<K,V>, NativeParallelHashMap<K,V>
HashSet<T> NativeHashSet<T>, NativeParallelHashSet<T>
Queue<T> NativeQueue<T>
string FixedString32Bytes / FixedString64Bytes / 등.

할당:

var arr = new NativeArray<int>(1024, Allocator.TempJob);
// ...
arr.Dispose();

수명이 중요하다: Burst + Safety는 컨테이너가 Dispose되고 Job의 수명 이후에 접근되지 않는지 확인한다.

6. Allocator

Allocator 수명 일반적인 사용 예
Allocator.Temp 현재 프레임 내, 메인 스레드 전용 OnUpdate의 임시 컨테이너
Allocator.TempJob 하나의 Job에 걸쳐, ≤4 프레임 Job에 전달되는 컨테이너
Allocator.Persistent 수동 Dispose() 필요 프레임 간 지속되는 System 소유 컨테이너
state.WorldUpdateAllocator System Group 업데이트 끝에 지워짐 System당 임시 데이터에 선호

System 내부에서 state.WorldUpdateAllocator (또는 Rewindable Allocator)를 선호하라 — 수동 Dispose() 관리를 피할 수 있다.

7. Safety System

"Jobs > Leak Detection"과 "Jobs > Safety Checks"가 켜진 Editor에서 Unity는 다음을 검증한다:

  • 두 Job이 의존성 엣지 없이 같은 네이티브 컨테이너에 쓰지 않는지.
  • 메인 스레드 읽기가 실행 중인 쓰기와 경쟁하지 않는지.
  • 네이티브 컨테이너가 Dispose() 후 또는 Allocator 수명 이후에 사용되지 않는지.

Safety 오류는 명확한 "job A and job B both access X" 메시지와 함께 예외로 나타난다. 개발 중에는 Safety Check를 비활성화하지 마라 — 성능 이득은 작고 디버깅 비용은 크다.

8. 문제 해결

증상 원인 / 해결
Burst 경고: "method not burst compiled" 메서드에 [BurstCompile]이 없음; 또는 Burst가 지원하지 않는 타입을 찾음 (실패한 줄에 대해 Burst Inspector를 확인하라).
InvalidOperationException: … writes to … without declaring dependency 의존성 엣지 누락 — 이전 JobHandle을 전달하거나 state.Dependency 체인을 사용하라.
NativeArray의 ObjectDisposedException Job이 완료되기 전에 Dispose됨. arr.Dispose(handle)을 사용하거나 먼저 핸들을 기다려라.
예약된 Job이 완료되지 않음 체인에서 .Complete() 누락, 또는 순환 의존성.
[BurstCompile] 추가 후 성능 저하 도메인 리로드마다 메서드가 재컴파일됨; Burst 캐시를 확인하라. 또는 Burst 비호환 타입이 슬며시 들어옴 — 메서드가 조용히 IL로 폴백됨.
NativeHashMap 열거 시 예외 발생 동시 쓰기와 열거가 잘 작동하지 않음 — 대신 NativeArray<K> 복사본을 열거하라.
Job을 기다리며 메인 스레드 정지 .Complete()를 너무 일찍 호출함. 완료 시점을 프레임에서 가능한 한 늦게 이동하라.

Chapter 14. IJobEntity · SystemAPI.Query

1. 개요

Entities 6.5에서 Component 반복의 95%를 커버하는 두 가지 API:

API 실행 위치 사용 시기
SystemAPI.Query<...>() 메인 스레드, OnUpdate 내부 작업이 간단한 경우 (수백 개 Entity), 또는 메인 스레드 API가 필요한 경우.
IJobEntity 워커 스레드, Schedule / ScheduleParallel을 통해 작업이 더 무겁거나 병렬 처리에서 이득이 있는 경우.

둘 다 레거시 Entities.ForEach의 권장 대체제다 (Entities 1.4에서 폐기 표시; 6.5에서는 여전히 폐기 경고와 함께 컴파일됨; Unity의 변경 로그에 따르면 향후 주요 릴리스에서 제거 예정).

2. SystemAPI.Query<T>

메인 스레드에서 매칭되는 Entity를 반복한다. Component는 RefRO<T> / RefRW<T>를 통해 접근된다.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRW.Max);
        }
    }
}

규칙:

  • System struct는 partial이어야 한다.
  • 읽기 전용에는 RefRO<T>, 읽기-쓰기에는 RefRW<T>. RefRO를 통해 쓰지 마라.
  • ID가 필요하다면 튜플에 Entity를 추가하라: SystemAPI.Query<RefRW<Health>>().WithEntityAccess().

필터

foreach (var health in SystemAPI.Query<RefRW<Health>>()
             .WithAll<Enemy>()
             .WithNone<Dead>()
             .WithAny<Burning, Poisoned>())
{
    // only entities that are Enemy AND not Dead AND (Burning OR Poisoned)
}
  • WithAll<T>() — 나열된 Component를 모두 가져야 함.
  • WithNone<T>() — 그 중 어느 것도 가지지 않아야 함.
  • WithAny<T...>() — 적어도 하나를 가져야 함.
  • WithChangeFilter<T>() — 마지막 실행 이후 Component T가 변경된 Chunk만.
  • WithEntityAccess() — Entity ID도 반환.

3. IJobEntity

소스 생성 병렬 반복. Execute 시그니처가 접근하는 Component를 선언한다.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

시그니처 규칙:

  • void Execute(...) — 매개변수가 Component와 1:1 매핑.
  • ref Component — 읽기-쓰기. in Component — 읽기 전용 (쓰기 불가).
  • Entity entity도 매개변수로 허용 — 소스 생성기가 채워준다.
  • Job struct는 partial이어야 한다.

Schedule 변형

호출 스레딩
.Run() 메인 스레드, 반환 전에 완료. 디버깅이나 소규모 작업에 유용.

결정론적 Playback을 위해 ParallelWriter ECB와 쌍으로 사용할 때 유용하다 — Chunk 인덱스를 sortKey로 전달하라. 15_ParallelWriter · Deterministic Playback.md를 참조하라.

5. 주의 사항

  • Job에서 this나 관리형 필드를 캡처하지 마라. 소스 생성기가 시도 시 오류를 낸다.
  • RefRW<T>.ValueRW는 변경 추적을 트리거한다. ValueRW를 통해 읽고 쓰지 않아도 Chunk를 dirty로 표시한다.
  • WithChangeFilter<T>는 Chunk 수준이다. Chunk의 어느 Entity든 T에 쓰면 전체 Chunk가 변경된 것으로 간주된다.
  • Aspect는 폐기됐다 (제거되지는 않음). IAspect 그룹화 인터페이스는 1.4에서 폐기됐으며 6.5 API 참조에서는 여전히 유효한 Execute 매개변수 종류로 나열됨. 새 코드는 일반 Component 매개변수를 사용하라; 기존 Aspect는 6.5에서 여전히 컴파일된다.

증상 원인 / 해결

Chapter 15. IJobEntity vs IJobChunk

1. 짧은 답변

  • IJobEntity: "각 Entity에 같은 작업을 수행한다."
  • IJobChunk: "단순한 Entity당 맵이 아닌 각 Chunk에 무언가를 수행한다."

IJobEntity는 IJobChunk 위에 생성된다 — 동일한 반복 메커니즘에서 실행된다. IJobChunk는 단순히 코드에 전체 Chunk를 노출한다.

2. IJobEntity — Entity당 편의성

[BurstCompile]
public partial struct DamageTickJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Burn burn)
    {
        health.Value -= burn.DamagePerSecond * DeltaTime;
    }
}

강점:

  • 하나의 메서드, 하나의 Entity 뷰. 작성하기 쉽고 읽기 쉽다.
  • 소스 생성 배선: Entity, [ChunkIndexInQuery], [EntityIndexInQuery] 매개변수가 자동으로 채워진다.
  • struct의 [WithAll] / [WithNone] / [WithAny] 속성이나 .Schedule*에 전달된 QueryBuilder를 통한 필터.

약점:

  • Execute 내부에서 Entity 간 가시성 없음. 추가 상태 없이 "이 Chunk에서 합산"을 쉽게 할 수 없다.
  • Execute 내부에서 Chunk 수준 메타데이터(변경 버전, Shared Component 값)에 접근 불가.

3. IJobChunk — 전체 Chunk 제어

[BurstCompile]
public partial struct HealthAggregateJob : IJobChunk
{
    [ReadOnly] public ComponentTypeHandle<Health> HealthHandle;
    public NativeArray<float> PerChunkTotal;  // size = chunk count

    public void Execute(in ArchetypeChunk chunk,
                        int unfilteredChunkIndex,
                        bool useEnabledMask,
                        in v128 chunkEnabledMask)
    {
        var healths = chunk.GetNativeArray(ref HealthHandle);
        float total = 0f;
        for (int i = 0; i < chunk.Count; i++)
            total += healths[i].Value;

        PerChunkTotal[unfilteredChunkIndex] = total;
    }
}

강점:

  • Chunk 배열에 직접 접근: chunk.GetNativeArray(ref handle), chunk.GetBufferAccessor(ref bufferHandle).
  • 조건부 작업을 위한 chunk.Has<T>(), chunk.GetSharedComponent<T>(...).
  • Chunk당 활성화 마스크와 변경 버전을 볼 수 있다.
  • 축약, 공간 분할, 배치 ECB 기록에 완벽하다.

비용:

  • 더 많은 보일러플레이트: System에서 ComponentTypeHandle<T>을 선언하고 매 프레임 handle.Update(ref state)로 업데이트해야 한다.
  • 내부 for 루프를 직접 관리 — 실수할 곳이 더 많다.

System에서 핸들 설정

[BurstCompile]
public partial struct HealthAggregateSystem : ISystem
{
    private EntityQuery _query;
    private ComponentTypeHandle<Health> _healthHandle;

    public void OnCreate(ref SystemState state)
    {
        _query        = state.GetEntityQuery(ComponentType.ReadOnly<Health>());
        _healthHandle = state.GetComponentTypeHandle<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthHandle.Update(ref state);

        var totals = CollectionHelper.CreateNativeArray<float>(
            _query.CalculateChunkCount(), state.WorldUpdateAllocator);

        state.Dependency = new HealthAggregateJob
        {
            HealthHandle   = _healthHandle,
            PerChunkTotal  = totals
        }.ScheduleParallel(_query, state.Dependency);
    }
}

handle.Update(ref state)는 매 프레임 캐시된 타입 핸들을 새로고침한다 — 이를 잊으면 Burst가 오래된 인덱스를 사용하게 된다.

4. 활성화 마스크 (useEnabledMask)

IEnableableComponent를 포함하는 Query의 경우, Execute는 useEnabledMask = true와 활성화 비트의 v128 마스크를 받는다. 마스크를 존중하지 않으면 비활성화된 Entity를 처리하게 된다.

헬퍼:

var enumerator = new ChunkEntityEnumerator(useEnabledMask, chunkEnabledMask, chunk.Count);
while (enumerator.NextEntityIndex(out int i))
{
    // i is the index of the next enabled entity in the chunk
}

IJobEntity는 마스크를 자동으로 적용한다 — Chunk 수준 가시성이 필요하지 않은 한 이를 선호하는 또 다른 이유다.

5. 결정 가이드

상황 선택
체력 틱, 위치 이동, 입력 적용 — 순수 Entity당 수학 IJobEntity
Chunk의 모든 Entity에 걸쳐 값 합산 IJobChunk
Entity당 대신 Chunk당 하나의 ECB 명령 기록 IJobChunk
루프 내부에서 Shared Component 값 접근 IJobChunk
Chunk의 변경 버전 필요 IJobChunk
각 Entity의 DynamicBuffer<T> 반복 둘 다 가능; IJobEntity가 더 깔끔함
Entity당 로직이지만 Entity의 Chunk 인덱스 필요 IJobEntity와 [ChunkIndexInQuery] int chunkIndex
예약 비용이 중요하고 작업이 작음 벤치마크 — 적은 Chunk가 적은 Entity보다 낫다. IJobChunk는 보통 수천 Chunk에서 이긴다.

6. 상호 운용 — 함께 사용하기

일반적인 패턴: IJobChunk로 Chunk당 데이터를 수집하고, IJobEntity로 Entity당 쓰기를 수행하며, 의존성을 통해 체인.

var gather = new HealthAggregateJob { ... }
    .ScheduleParallel(query, state.Dependency);

var act = new HealJob { PerChunkTotal = totals }
    .ScheduleParallel(gather);

state.Dependency = act;

7. 문제 해결

증상 원인 / 해결
IJobChunk.Execute가 쓰레기 값을 읽음 예약 전에 handle.Update(ref state)를 잊었거나, 잘못된 Query가 사용됨.
ComponentTypeHandle<T>를 찾을 수 없음 using Unity.Entities; 누락, 또는 핸들이 잘못된 접근 모드로 선언됨. 읽기 전용에는 GetComponentTypeHandle<T>(true)를 사용하라.
활성화된 Entity가 건너뜀 마스크가 존중되지 않음 — 보여진 대로 ChunkEntityEnumerator를 사용하라. IJobEntity는 이를 피한다.
Job이 두 Chunk에서 같은 NativeArray에 쓴다 Chunk당 슬롯을 제공하거나 (HealthAggregateJob처럼) Chunk 인덱스를 키로 NativeParallelHashMap을 사용하라.
query 인수 없이 IJobChunk 예약 ScheduleParallel(query, ...)는 Query를 명시적으로 요구한다. IJobEntity.ScheduleParallel()은 속성/소스 생성에서 가져온다.
v128 타입을 찾을 수 없음 using Unity.Burst.Intrinsics;가 누락됨.

Chapter 16. 구조적 변경 & Safety

1. 구조적 변경의 범주

구조적 변경은 Entity를 Chunk 간에 이동하거나 Chunk 인벤토리를 변경할 수 있는 모든 작업이다. 다음은 모두 구조적이다:

작업 구조적인 이유
CreateEntity / DestroyEntity Chunk 슬롯을 할당하거나 해제.
AddComponent<T> / RemoveComponent<T> Entity의 Archetype이 변경되어 물리적으로 다른 Chunk로 이동.
SetSharedComponent<T> 다른 Shared Component 값 → 다른 Chunk (값으로 그룹화).
AddChunkComponent<T> / RemoveChunkComponent<T> Chunk 자체의 메타데이터를 변경.
SetName(entity, ...) EntityManager의 Entity 메타데이터를 건드림.
MoveEntitiesFrom World 간 Entity 복사.
Instantiate(prefab) 새 Entity 할당 (Archetype 미리 채울 수 있음).

SetComponentData / RefRW<T>.ValueRW를 통한 Component 값 쓰기는 구조적이 아니다 — Component는 이미 Chunk에 있으며, 바이트를 변경하는 것뿐이다.

2. 구조적 변경이 위험한 이유

Archetype이 변경되면 Chunk 내부 위치를 기준으로 잡고 있던 모든 참조가 무효화된다:

  • 진행 중인 Query 반복자 (SystemAPI.Query<...>)가 Entity를 건너뛰거나 재방문할 수 있다.
  • 영향받는 Entity의 ComponentLookup<T> 항목이 무효화된다.
  • Job의 ArchetypeChunk 핸들이 이전 Chunk를 가리킨다.
  • 동시에 실행되는 다른 System이 불일치한 상태를 관찰한다.

Safety System은 Editor에서 InvalidOperationException으로 대부분을 잡는다. 빌드에서는 충돌이나 Entity 손상이 된다.

3. 경험 법칙

같은 System 업데이트에서 Query를 반복하는 동안 구조적 변경을 수행하지 마라.

선호 순서에 따른 두 가지 준수 패턴:

  1. EntityCommandBuffer를 사용하라. 반복 중에 작업을 기록하고, 이후에 재생하라. 14_EntityCommandBuffer · Deferred Entity.md를 참조하라.
  2. NativeList<Entity>에 반복하고, 루프 후에 변경하라. 소규모 경우에 더 간단하다; ECB 오버헤드 없음.

패턴 2의 예제:

var toKill = new NativeList<Entity>(state.WorldUpdateAllocator);

foreach (var (health, e) in SystemAPI
             .Query<RefRO<Health>>()
             .WithEntityAccess())
{
    if (health.ValueRO.Value <= 0f)
        toKill.Add(e);
}

state.EntityManager.DestroyEntity(toKill.AsArray());

4. 활성화 가능한 Component — 탈출구

매 프레임 Component를 추가하고 제거하는 자신을 발견한다면, 거의 확실히 활성화 가능한 Component여야 한다 — 활성화 비트 토글은 구조적이 아니다. 06_Enableable Component.md를 참조하라.

5. Job과 구조적 변경

구조적 변경은 메인 스레드 작업이다. Job 내부에서 수행할 수 없다.

Job에서 변경하는 관용적인 방법:

접근 방식 시기
Job에서 기록한 EntityCommandBuffer.ParallelWriter를 메인 스레드에서 재생 병렬 코드에서의 Entity당 구조적 변경.
IJob이나 IJobChunk에서 기록한 EntityCommandBuffer (단일 스레드)를 메인 스레드에서 재생 일괄/Chunk 범위 변경.
Job 내부에서 EnabledRefRW<T>를 통해 활성화 가능한 Component 토글 구조 없는 Entity당 "활성화/비활성화".

ECB는 작업을 명령으로 캡처한다; Playback은 메인 스레드의 알려진 지점, 보통 EntityCommandBufferSystem 경계에서 발생한다.

6. 구조적 변경과 읽기 결합

두 System이 연속으로 실행되고 System A가 System B가 의존하는 구조적 변경을 수행하는 경우, 순서를 명시적으로 만들어라:

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct SpawnSystem : ISystem { /* uses ECB */ }

ECB System Playback이 A와 B 사이에 있을 때 (예: 적절한 [UpdateAfter] 체인과 함께 EndSimulationECBS), 구조적 변경이 B가 읽기 전에 완료된다.

7. 비용 모델

단일 변경에 대한 대략적인 순위:

SetComponentData (value only)          ~ 1x
SetComponentEnabled (enableable bit)   ~ 1x (per-chunk bitmask write)
AddComponent (structural)              ~ 10x   (entity moves chunk)
SetSharedComponent (structural)        ~ 10x
DestroyEntity (structural)             ~ 10x
Per-frame add + remove pattern         pathological

크기 차수; 정확한 수치는 Archetype 크기, Chunk 점유율, Allocator 압력에 따라 다르다.

8. 구조적 변경 과부하 진단

증상:

  • Window → Entities → Systems에서 System이 로직에 비해 불균형한 타이밍을 보인다.
  • Profiler 마커 EntityManager.SetArchetype 또는 Move Entity가 프레임을 지배한다.
  • Archetype 수가 시간이 지남에 따라 증가한다.

전술:

  • AddComponent / RemoveComponent 호출을 감사하라 — 프레임당 발생하는 것이 있는가? 활성화 가능한 Component로 교체하라.
  • Shared Component 값 집합 검사 — 실수로 Entity당 키가 있는 것은 없는가?
  • Archetypes 창에서 단일 Entity Chunk를 확인하라; 그것은 너무 세분화된 Component 집합으로 인한 단편화다.

9. 문제 해결

증상 원인 / 해결
InvalidOperationException: EntityCommandBuffer has already been played back Playback 후 ECB에 기록. 각 ECB는 단일 사용 — 매 프레임 ECBS에서 새 것을 가져오라.
foreach 중 InvalidOperationException: … invalidated … 루프 내부에서 구조적 변경. ECB로 이전하거나 버퍼-후-적용 방식을 사용하라.
Entity가 조용히 사라짐 DestroyEntity가 발생했지만 다른 System이 여전히 이전 Entity 핸들을 보유. EntityManager.Exists로 핸들을 확인하라.
추가/제거 패턴이 프레임 시간을 소모 활성화 가능한 Component로 교체하라.
RemoveComponent가 "cleanup component present" 오류를 트리거 Entity에 정리 Component가 있음. 먼저 정리를 처리하라 (05_Component Types.md 참조).
Shared Component 값 집합이 폭발 (수백 개의 고유 값) 값이 너무 세분화됨. 버킷화하거나 (예: 가장 가까운 N으로 반올림) 가변 부분을 일반 Component로 저장하라.

Chapter 17. EntityCommandBuffer · 지연된 Entity

1. ECB가 존재하는 이유

EntityCommandBuffer (ECB)는 구조적 변경 — 생성, 소멸, 추가, 제거, 설정 — 을 기록하고 나중에 메인 스레드의 안전한 지점에서 재생한다. 다음에서 World를 변경할 수 있게 한다:

  • Query foreach 내부 (직접 EntityManager 호출이 반복자를 무효화하는 경우).
  • Job (메인 스레드 구조적 변경이 허용되지 않는 경우).
  • 많은 Chunk에 걸친 병렬 Job (ParallelWriter를 통해).

"왜 이것이 위험한가"에 대한 배경은 13_Structural Change & Safety.md를 참조하라.

2. ECB 수명 주기

Figure: EntityCommandBuffer Lifecycle.

ECB는 단일 사용이다: Playback 후 Dispose된다. 매 프레임 올바른 EntityCommandBufferSystem에 새 것을 요청한다.

3. 내장 ECB System

System Playback 시점 일반적인 사용 예
BeginInitializationEntityCommandBufferSystem Initialization Group 시작 이전 프레임의 초기 World 설정
EndInitializationEntityCommandBufferSystem Initialization Group 끝 —
BeginSimulationEntityCommandBufferSystem Simulation 시작 게임플레이 실행 전 대기 중인 변경 적용
EndSimulationEntityCommandBufferSystem Simulation 끝 게임플레이가 주도하는 스폰/소멸의 기본 선택
BeginPresentationEntityCommandBufferSystem Presentation 시작 드물게; 시각 전용 설정

올바른 것을 선택하면 다른 System이 변경 사항을 관찰하는 시점이 결정된다. SimulationSystemGroup에서 총알을 스폰하고 이 프레임에 이동하려면, 다음 프레임의 BeginSimulation에 기록하라 — 다음 프레임이 괜찮으면 EndSimulation에.

4. ECB 가져오기

모든 ISystem.OnUpdate 내부에서:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

.Singleton 중첩 타입은 소스 생성기가 각 내장 ECB System을 위해 노출하는 System당 토큰이다. CreateCommandBuffer는 해당 System이 틱할 때 재생될 새 ECB를 제공한다.

병렬 Job의 경우:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

병렬 Playback 계약에 대해서는 15_ParallelWriter · Deterministic Playback.md를 참조하라.

5. 기록 — Entity 생성

Entity e = ecb.CreateEntity();
ecb.AddComponent(e, new Health { Value = 100 });
ecb.AddComponent(e, LocalTransform.FromPosition(0, 0, 0));

또는 Prefab Entity에서 인스턴스화:

Entity inst = ecb.Instantiate(prefab);
ecb.SetComponent(inst, LocalTransform.FromPosition(spawnPos));

CreateEntity와 Instantiate 모두 지연된 Entity를 반환한다 (§6 참조).

6. 지연된 Entity

ecb.CreateEntity()나 ecb.Instantiate()가 반환하는 Entity는 아직 존재하지 않는다. 임시 음수 인덱스를 가진다. 다음을 할 수 있다:

  • 같은 ECB의 후속 ECB 호출에 전달 — AddComponent, SetComponent, AppendToBuffer.
  • ecb.SetComponent(other, new ReferenceTo { Entity = inst })를 통해 다른 Component에 저장.

다음은 할 수 없다:

  • ECB가 재생되기 전에 EntityManager를 통해 읽거나 쓰기.
  • 다른 ECB 간에 전달 (임시 객체는 소유 ECB 내에서만 해결됨).

Playback 시간에 ECB System은 임시 객체가 사용된 모든 곳에 실제 Entity 핸들을 대입한다.

7. 기록 — 구조적 변경

ecb.AddComponent(entity, new Stunned { Duration = 2f });
ecb.RemoveComponent<Stunned>(entity);
ecb.SetComponent(entity, new Health { Value = 50 });

ecb.SetComponentEnabled<Stunned>(entity, false);

var buf = ecb.AddBuffer<Waypoint>(entity);
buf.Add(new Waypoint { Position = new float3(1, 0, 0) });
ecb.AppendToBuffer(entity, new Waypoint { Position = new float3(2, 0, 0) });

ecb.DestroyEntity(entity);

이 모두 대기 명령이다; Playback까지 아무것도 적용되지 않는다.

8. IJobEntity에서의 ECB

[BurstCompile]
public partial struct KillOnHealthZeroJob : IJobEntity
{
    public EntityCommandBuffer ECB;

    void Execute(Entity entity, in Health health)
    {
        if (health.Value <= 0)
            ECB.DestroyEntity(entity);
    }
}

// In OnUpdate:
var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

new KillOnHealthZeroJob { ECB = ecb }.Schedule();

.ScheduleParallel()의 경우 .AsParallelWriter()로 전환하고 sortKey를 함께 전달하라 — 다음 페이지에서 다룬다.

9. 독립형 (수동) ECB

때로는 직접 제어하는 Playback이 필요하다 — 예: 단일 OnUpdate 내부에서:

var ecb = new EntityCommandBuffer(Allocator.Temp);
// ... record ...
ecb.Playback(state.EntityManager);
ecb.Dispose();

자체 포함된 일회성 변경 패턴에 사용하라. System 경계를 넘거나 Job에서 실행되는 것에는 위의 내장 ECBS를 사용하라.

10. 결정론

ECB 명령은 단일 작성자 ECB에서 기록 순서로 재생된다. ParallelWriter에서는 Playback 순서가 제공하는 sortKey에 따라 달라진다 — 15_ParallelWriter · Deterministic Playback.md를 참조하라.

Netcode/리플레이 System의 경우 결정론적 Playback은 협상 불가다. 항상 안정적인 정렬 키(Chunk 인덱스, Query 내 Entity 인덱스)를 전달하고, 비결정론적 순서로 실행되는 메인 스레드 코드 경로에서 기록을 피하라.

11. 문제 해결

증상 원인 / 해결
InvalidOperationException: ECB has already been played back 프레임 간 재사용됨. CreateCommandBuffer를 통해 OnUpdate당 새 ECB를 가져오라.
Playback 후 지연된 Entity 핸들이 Entity.Null 두 다른 ECB에서 참조됨. 생성과 후속 설정을 모두 같은 ECB에 유지하라.
이 프레임에 스폰된 Entity가 나타나지 않음 EndSimulation ECBS에 기록됨 — Playback이 프레임 끝. 대부분의 System 전에 관찰하려면 다음 프레임의 BeginSimulation으로 전환하라.
ecb.SetComponent가 "component does not exist" 오류를 던짐 대상 Entity에 Component가 아직 없음 (먼저 AddComponent) 또는 Playback이 발생하지 않음.
ECBS가 관리하는 ECB에서 Dispose() 호출 CreateCommandBuffer의 ECB에서 Dispose()를 호출하지 마라 — ECBS가 수명을 처리한다. 독립형 (new EntityCommandBuffer(Allocator.Temp))만 Dispose하라.
병렬 ECB Playback이 명령을 재정렬 sortKey가 결정론적이지 않음. [ChunkIndexInQuery]나 안정적인 Query 제공 인덱스로 전환하라.

Chapter 18. ParallelWriter · 결정론적 Playback

1. 별도의 병렬 작성자가 존재하는 이유

EntityCommandBuffer는 단일 작성자다. 두 스레드가 같은 ECB에 기록하면 내부 체인에서 경쟁 조건이 발생한다. EntityCommandBuffer.ParallelWriter는 각 호출자에게 스레드 로컬 체인을 제공하고 명령당 정렬 키를 요구하여 이를 해결한다.

Playback 시간에 ECB System은:

  1. 모든 스레드 로컬 체인을 평탄화한다.
  2. sortKey로 명령을 정렬한다 (안정 정렬).
  3. 메인 스레드에서 실행한다.

결과는 정렬 키가 결정론적인 한 결정론적이다.

2. ParallelWriter 가져오기

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

AsParallelWriter()는 IJobEntity, IJobChunk, 또는 IJobParallelFor에 복사할 수 있는 struct EntityCommandBuffer.ParallelWriter를 반환한다.

3. sortKey 계약

모든 병렬 ECB 호출은 첫 번째 인수로 sortKey를 받는다:

ecb.Instantiate(sortKey, prefab);
ecb.SetComponent(sortKey, entity, value);
ecb.DestroyEntity(sortKey, entity);
ecb.AddComponent(sortKey, entity, component);

좋은 정렬 키의 규칙:

  • 결정론적. 같은 입력 → 같은 키, 스케줄러가 작업을 어떻게 분산하든 상관없이.
  • 충분히 고유함. 같은 Entity의 두 명령은 Playback 순서가 모호해도 되지 않는 한 충돌해서는 안 된다.
  • 저렴함. 이미 가지고 있는 int, 해시가 아님.

실제로 기본 선택은:

소스 가져오는 방법 사용 시기
IJobEntity의 [ChunkIndexInQuery] 소스 생성기가 채워줌 99%의 경우
IJobChunk.Execute의 unfilteredChunkIndex API에서 제공 IJobChunk Job
IJobEntity의 [EntityIndexInQuery] 소스 생성기 명령이 Entity당 개별 순서화가 필요한 경우
수동으로 계산된 결정론적 인덱스 직접 계산 드물게; 커스텀 Job 형태

4. 전체 예제 — 병렬 스폰

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnParallelJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity spawnerEntity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, spawnerEntity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

[BurstCompile]
public partial struct SpawnerParallelSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged)
            .AsParallelWriter();

        new SpawnParallelJob
        {
            DeltaTime = SystemAPI.Time.DeltaTime,
            ECB       = ecb
        }.ScheduleParallel();
    }
}

Instantiate와 후속 SetComponent 모두에 chunkIndex를 전달하면 두 명령이 같은 위치로 정렬되어 Instantiate의 지연된 Entity 참조가 올바른 대상으로 해결된다.

5. 결정론 — 보장하는 것과 하지 않는 것

보장됨: 동일한 ECB 내용이 주어지면, 어느 워커 스레드가 어느 명령을 기록했는지에 상관없이 Playback은 동일한 결과 상태를 생성한다.

자동으로 보장되지 않음:

  • Entity ID. 이전 명령이 달랐다면 두 번 실행 시 다른 Index 값을 가진 Entity가 생성될 수 있다.
  • System 간 Chunk 순서. System A가 병렬로 Entity를 생성하고 System B가 ChunkIndexInQuery 순서로 읽으면, 반복 순서는 여전히 Archetype 생성 이력에 따라 달라질 수 있다.

실행 간 비트 수준 결정론이 필요하다면 (리플레이, Netcode 롤백), 다음을 결합하라:

  1. 결정론적 정렬 키를 가진 병렬 ECB.
  2. FixedStepSimulationSystemGroup 내의 고정 스텝 System.
  3. Entity.Index에 의존하지 않음; 직접 제어하는 게임플레이 ID로 키 설정.

6. 정렬 키 패턴

6.1 Chunk당 (기본)

[ChunkIndexInQuery] int chunkIndex를 사용하라. Chunk의 모든 Entity가 동일한 명령을 생성하는 경우 (예: 전체 Chunk가 Stunned 표시) 충돌은 괜찮다.

6.2 Entity당

엄격한 Entity당 순서화가 필요한 경우 [EntityIndexInQuery] int entityIndex를 사용하라 (예: Entity 순서로 재생해야 하는 Entity당 피해 이벤트 기록).

6.3 커스텀 결정론적 인덱스

매우 드물게, 입력에서 직접 평탄 인덱스를 만든다 — 예: int sortKey = baseOffset + localIndex;. 일반적인 두 가지가 맞지 않을 때만 사용하라; 커스텀 인덱스는 결정론 버그의 빈번한 원인이다.

7. 이런 패턴을 피하라

  • JobsUtility.ThreadIndex를 sortKey로 사용. 결정론적이지 않다.
  • 키를 만들기 위해 Random 사용. 결정론적이지 않다.
  • ECB 혼합. 두 다른 ECB에 기록하면 전체 순서를 잃는다.
  • 여러 System에서 System당 ECB에 기록. 하나의 ECB, 하나의 기록자 (또는 하나의 ECB를 래핑하는 하나의 병렬 작성자).

8. 문제 해결

증상 원인 / 해결
지연된 Entity가 잘못된 대상으로 해결됨 Instantiate와 후속 SetComponent가 다른 정렬 키를 사용함. 같게 유지하라.
명령이 매 실행마다 다른 순서로 발생 비결정론적 정렬 키. [ChunkIndexInQuery]로 전환하라.
"JobEntity must have partial" 컴파일 오류 Job struct에 partial을 추가하라.
병렬 Job의 ECB.SetComponent가 "wrong thread" 오류를 던짐 병렬 Job 내부에서 비병렬 EntityCommandBuffer를 사용. 먼저 .AsParallelWriter()를 호출하고 병렬 API를 사용하라.
Netcode 리플레이 비동기화 병렬 ECB 중 어딘가에서 결정론적 정렬 키를 사용하지 않거나, IJobEntity가 비결정론적 값을 캡처함. 체인을 감사하라.
Dispose된 ECB에서 AsParallelWriter() ECB가 이미 재생됨 — 이 프레임에 새 것을 가져오라.

Chapter 19. Netcode 클라이언트-서버 World & Bootstrap

1. 개요

Netcode for Entities는 클라이언트-서버 모델을 사용하며, 클라이언트와 서버 로직을 서로 다른 ECS World로 분리한다.

World 역할
Server World 권위적인 시뮬레이션.
Client World 로컬 플레이어 시뮬레이션, 예측, 보간, 프레젠테이션.

3. Bootstrap 프로세스

6. World 마이그레이션

World 전환 시 연결 상태를 보존해야 할 때 DriverMigrationSystem을 사용하라.

using Unity.Entities;
using Unity.NetCode;

public World MigrateServerWorld(World sourceWorld)
{
    DriverMigrationSystem migrationSystem = null;
    foreach (var world in World.All)
    {
        if ((migrationSystem = world.GetExistingSystemManaged<DriverMigrationSystem>()) != null)
            break;
    }

    var ticket = migrationSystem.StoreWorld(sourceWorld);
    sourceWorld.Dispose();

    var newWorld = ClientServerBootstrap.CreateServerWorld("NewServerWorld");
    var newMigrationSystem = newWorld.GetExistingSystemManaged<DriverMigrationSystem>();
    newMigrationSystem.LoadWorld(ticket);
    return newWorld;
}

Chapter 20. Netcode 네트워크 연결 & 승인

1. 개요

Netcode for Entities는 모든 네트워크 연결을 Entity로 표현한다. 연결 Entity는 NetworkStreamConnection을 가지며, 연결이 끝나면 Entity가 소멸된다.

양측이 NetworkStreamInGame을 추가하기 전까지 연결은 "게임 내"로 간주되지 않는다. 해당 Component 없이는 서버가 스냅샷을 보내지 않고 클라이언트가 명령을 보내지 않는다.

2. 핵심 연결 Component

Component 의미
NetworkStreamConnection 전송 핸들을 저장.
NetworkId 승인 후 서버가 할당하는 고유 ID.
NetworkStreamInGame 스냅샷과 명령 데이터 교환에 필요.
CommandTarget 플레이어 명령을 받는 Entity; AutoCommandTarget을 사용하지 않는 한 게임 코드가 유지.
NetworkStreamRequestDisconnect 연결 해제 요청; 드라이버 연결 해제를 직접 호출하지 마라.
NetworkStreamIsReconnected 1.6.1 라인에서 추가된 재연결 추적 Component.

4. 연결 상태 기계

4.1 클라이언트 상태

Connecting → Handshake → [Approval] → Connected → Disconnected
상태 의미
Connecting Connect 후 한 번 발생.
Handshake 전송 연결 수립; 프로토콜 핸드셰이크 대기 중.

Connected 서버가 NetworkId를 할당.

4.2 서버 상태

상태 의미
Handshake 클라이언트 수락; NetworkProtocolVersion 교환.
Approval 연결 승인이 활성화된 경우에만 존재.

5.3 승인 RPC

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;

public struct ApprovalRpc : IApprovalRpcCommand
{
    public FixedString512Bytes Token;
}

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation |
    WorldSystemFilterFlags.ThinClientSimulation)]
public partial struct ClientApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (connection, entity) in
            SystemAPI.Query<RefRW<NetworkStreamConnection>>()
                .WithNone<NetworkId>()
                .WithNone<ApprovalSent>()
                .WithEntityAccess())
        {
            var rpcEntity = ecb.CreateEntity();
            ecb.AddComponent(rpcEntity, new ApprovalRpc { Token = "MY_TOKEN" });
            ecb.AddComponent<SendRpcCommandRequest>(rpcEntity);
            ecb.AddComponent<ApprovalSent>(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ServerApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, rpc, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ApprovalRpc>>()
                .WithEntityAccess())
        {
            var connEntity = req.ValueRO.SourceConnection;

            if (rpc.ValueRO.Token.Equals("MY_TOKEN"))
                ecb.AddComponent<ConnectionApproved>(connEntity);
            else
                ecb.AddComponent<NetworkStreamRequestDisconnect>(connEntity);

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

승인 타임아웃은 ClientServerTickRate.HandshakeApprovalTimeoutMS로 제어된다; 기본값은 5000 ms다.

Chapter 21. Netcode Ghost 스냅샷 & 동기화

1. 개요

Ghost는 서버에서 시뮬레이션되고 클라이언트에 복제되는 네트워크 Entity다. 서버는 주기적으로 Ghost 상태를 스냅샷으로 전송하고, 각 클라이언트는 보간 또는 예측을 통해 스냅샷을 적용한다.

메커니즘 사용
Ghost 내장 복제 최적화를 통한 비신뢰 패킷에서의 최종 일관성.
RPC 일회성 신뢰할 수 있는 이벤트.

2. Ghost vs RPC

Ghost 사용 RPC 사용
클라이언트 근처의 공간 Entity 상태. 고수준 게임 흐름 이벤트.
클라이언트 예측이 필요한 데이터. 채팅이나 설정 변경 같은 일회성 명령.
연결/재연결 사이클 후에도 존재해야 하는 상태. 영속적인 복제 상태가 없는 이벤트.

3. Ghost Authoring

복제되어야 하는 Prefab에 GhostAuthoringComponent를 추가한다.

속성 의미
Name Ghost 식별자.
Importance 대역폭이 제한될 때의 스냅샷 우선순위.
Supported Ghost Mode All, Interpolated, 또는 Predicted.
Default Ghost Mode Interpolated, Predicted, 또는 OwnerPredicted.
Optimization Mode 기본적으로 Dynamic, 드물게 변경되는 객체에는 Static.

3.1 Ghost 모드

모드 의미
Interpolated 클라이언트가 서버 스냅샷 간을 보간한다. 시각적으로 부드럽지만 지연이 있다.
Predicted 클라이언트가 입력 지연을 줄이기 위해 앞서 시뮬레이션한다. CPU 비용이 높다.
OwnerPredicted 소유 클라이언트는 예측하고; 다른 클라이언트는 보간한다.

4. 직렬화 속성

4.1 GhostField

직렬화해야 하는 Component 필드에 GhostField를 사용한다.

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

public struct MyComponent : IComponentData
{
    [GhostField(Quantization = 1000, Smoothing = SmoothingAction.Interpolate)]
    public float3 Position;

    [GhostField(Quantization = 100)]
    public float Health;

    [GhostField]
    public int Score;

    [GhostField(SendData = false)]
    public float LocalOnly;
}
GhostField 속성 의미
Quantization Float에서 int로의 곱수; 1000은 소수점 3자리를 유지한다.
Smoothing Clamp, Interpolate, 또는 InterpolateAndExtrapolate.
MaxSmoothingDistance 텔레포트에서 보간을 비활성화하는 거리 임계값.
Composite struct를 하나의 변경 비트로 처리.
SendData 직렬화에서 필드를 제외하려면 false로 설정.
SubType 특수 직렬화 규칙을 선택.

4.2 GhostEnabledBit

활성화 가능한 Component의 활성화 상태를 복제하려면 GhostEnabledBit를 사용한다.

ClientOnlyVariant Component가 클라이언트에만 존재.
ServerOnlyVariant Component가 서버에만 존재.
DontSerializeVariant 직렬화를 완전히 비활성화.

7. 스냅샷 System

7.1 부분 스냅샷

스냅샷이 MTU를 초과하면 Netcode는 중요도가 높은 Ghost를 먼저 보낸다. 나머지 Ghost는 나중 틱에 전송되어 World 상태가 점진적으로 스트리밍된다.

7.2 NetworkTickRate

NetworkTickRate가 SimulationTickRate보다 낮으면 GhostSendSystem 작업이 여러 시뮬레이션 틱에 분산된다.

SimulationTickRate = 60, NetworkTickRate = 30
Tick 1: GhostSendSystem sends a snapshot
Tick 2: GhostSendSystem skips
Tick 3: GhostSendSystem sends a snapshot
...

8.1 Prespawn Ghost

SubScene에 배치된 Ghost는 Prespawn Ghost가 된다:

  • SubScene이 로드될 때 서버와 클라이언트가 자동으로 같은 Entity를 매칭한다.
  • 별도의 스폰 메시지가 필요 없다; 스냅샷 동기화로 충분하다.
  • Prespawn Ghost ID는 1.6.1+ 라인에서 호스트 마이그레이션 중에 보존된다.

건물이나 지형 같은 정적 World 객체는 일반적으로 Static Optimization Mode를 가진 Prespawn Ghost로 하는 것이 가장 좋다.

  • 스냅샷 교환이 시작될 수 있도록 NetworkStreamInGame 추가.
  • 대역폭과 정밀도를 위해 Quantization 조정.
  • 자식 Entity 복제가 필요한 경우에만 SendDataForChildEntity 활성화.

Chapter 22. Netcode 예측 & 롤백

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 개요

클라이언트 예측은 권위적인 서버 결과가 도착하기 전에 클라이언트에서 동일한 시뮬레이션 코드를 실행한다. 이를 통해 입력 지연이 줄어들고 플레이어 소유 이동이 즉각적으로 느껴진다.

예측에는 실제 CPU 비용이 있다: 권위적인 스냅샷이 도착하면 클라이언트는 더 오래된 틱으로 롤백하고 현재 예측 틱까지 다시 시뮬레이션할 수 있다.

2. 핵심 Component와 System

Component / System 의미
PredictedGhost 클라이언트의 예측 Ghost와 서버의 모든 Ghost에 추가됨.
Simulate 현재 예측 틱에 시뮬레이션해야 하는 Entity에 설정되는 활성화 가능한 태그.
PredictedSimulationSystemGroup 예측 루프를 실행하는 고정 타임스텝 Group.
NetworkTime 서버 틱, 예측 틱, 관련 타이밍 데이터를 보유.

3. 클라이언트 예측 흐름

1. Receive a server snapshot
   └─ Apply the latest authoritative state to predicted entities

2. Find the oldest applied tick
   └─ Use the oldest tick across affected predicted entities

3. Roll back and re-simulate
   └─ PredictedSimulationSystemGroup repeats from oldest tick to target tick
   └─ TimeData and NetworkTime update for each tick

4. Render
   └─ Present the predicted result

높은 지연은 비용을 곱한다. 300 ms 연결은 틱레이트와 예측 설정에 따라 약 20프레임 이상의 재시뮬레이션이 필요할 수 있다.

4. 예측 System 작성

4.1 Simulate 태그 사용

예측 루프 시작 시 Netcode는 예측 Ghost의 Simulate를 비활성화하고, 현재 틱에 실행해야 하는 Entity에 대해서만 활성화한다.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct MovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 5f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<Simulate>())
        {
            var move = new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical);
            transform.ValueRW.Position += move * speed * dt;
        }
    }
}

예측 Query에는 항상 .WithAll<Simulate>()를 포함하라. 없으면 System이 이미 완료된 틱에 대해 Entity를 재시뮬레이션할 수 있다.

4.2 서버 측 동작

서버에서 예측 루프는 서버 틱당 정확히 한 번 실행된다:

  • TimeData는 일반 서버 틱 데이터를 포함.
  • Simulate는 항상 활성화됨.
  • 동일한 System 코드가 클라이언트와 서버 모두에서 실행될 수 있음.

5. 원격 플레이어 예측

다른 플레이어의 명령 데이터가 직렬화되면, 그 원격 플레이어도 예측될 수 있다.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct RemotePlayerMovement : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<PredictedGhost, Simulate>())
        {
            transform.ValueRW.Position +=
                new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical)
                * SystemAPI.Time.DeltaTime;
        }
    }
}

IInputComponentData와 함께 입력 System은 현재 틱의 입력을 자동으로 제공한다.

6. 예측 스무딩

예측 스무딩은 서버 결과가 예측 결과와 다를 때 보이는 수정을 줄인다.

using Unity.NetCode;
using Unity.Transforms;

GhostPredictionSmoothing.RegisterSmoothingAction<LocalTransform>(
    state.EntityManager,
    CustomSmoothing.Action);

using Unity.Burst;
using Unity.Collections.LowLevel.Unsafe;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[BurstCompile]
public static unsafe class CustomSmoothing
{
    public static readonly PortableFunctionPointer<
        GhostPredictionSmoothing.SmoothingActionDelegate> Action =
        new(SmoothingAction);

    [BurstCompile(DisableDirectCall = true)]
    private static void SmoothingAction(
        void* currentData, void* previousData, void* userData)
    {
        ref var current = ref UnsafeUtility.AsRef<LocalTransform>(currentData);
        ref var backup = ref UnsafeUtility.AsRef<LocalTransform>(previousData);

        var dist = math.distance(current.Position, backup.Position);
        if (dist > 0)
        {
            current.Position = math.lerp(backup.Position, current.Position, 0.5f);
        }
    }
}

예측은 비용이 크므로 Ghost는 Predicted와 Interpolated 간에 동적으로 전환할 수 있다.

7.1 전환 전제 조건

  • Supported Ghost Modes가 All이어야 한다.

7.2 전환 대기열에 추가

using Unity.NetCode;

ref var switchQueues = ref SystemAPI
    .GetSingletonRW<GhostPredictionSwitchingQueues>()
    .ValueRW;

switchQueues.ConvertToPredictedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 1f,
});

switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,

1. 개요

클라이언트가 게임 내 상태가 되면 서버에 지속적인 명령 스트림을 보낸다. 스트림에는 입력 데이터와 스냅샷 ACK 데이터가 포함되며; 플레이어가 적극적으로 입력을 누르지 않아도 연결이 동기화된 상태를 유지하기 위해 계속된다.

2. ICommandData vs IInputComponentData

ICommandData IInputComponentData

3. IInputComponentData

3.1 입력 정의

using Unity.NetCode;

public struct PlayerInput : IInputComponentData
{
    public int Horizontal;
    public int Vertical;

InputEvent는 GetKeyDown 같은 단발 이벤트를 보존하여 패킷 손실로 중복 명령 패킷이 누락되더라도 서버에서 정확히 한 번 등록되도록 한다.

3.2 클라이언트에서 입력 수집

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[UpdateInGroup(typeof(GhostInputSystemGroup))]
public partial struct GatherInputSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<PlayerInput>();
    }

    public void OnUpdate(ref SystemState state)
    {
        bool jump = Input.GetKeyDown(KeyCode.Space);
        bool left = Input.GetKey(KeyCode.A);
        bool right = Input.GetKey(KeyCode.D);

        foreach (var input in
            SystemAPI.Query<RefRW<PlayerInput>>()
                .WithAll<GhostOwnerIsLocal>())
        {
            input.ValueRW = default;
            if (jump) input.ValueRW.Jump.Set();
            if (left) input.ValueRW.Horizontal -= 1;
            if (right) input.ValueRW.Horizontal += 1;
        }
    }
}

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

public partial struct ProcessInputSystem : ISystem

    {

서버 코드가 GhostInputSystemGroup에서 실행됨 해당 그룹은 클라이언트 전용; 서버 로직은 PredictedSimulationSystemGroup에 배치.

Chapter 24. Netcode RPC

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 개요

RPC는 클라이언트와 서버 사이의 신뢰성 있는 단발 메시지다. RPC 직렬화 및 역직렬화 코드는 자동으로 생성되며, RPC 처리 역시 ECS / Job System 프레임 안에 맞아 들어간다.

RPC는 Ghost 복제나 명령 스트림의 대체가 아니다. 이산적인 게임 흐름 이벤트에 사용하며, 고빈도 상태에는 사용하지 않는다.

2. RPC vs Ghost vs Command

RPC Ghost Command
전송 방식 신뢰성, 순서 보장. 복제 최적화를 통한 비신뢰성. 중복 전송을 통한 비신뢰성.
방향 양방향. 서버 → 클라이언트. 클라이언트 → 서버.
빈도 단발. 필요 시 매 네트워크 틱. 매 시뮬레이션 틱.
사용 목적 게임 흐름, 로비, 채팅. 복제된 Entity 상태. 플레이어 입력.
전송 중 한도 있음, 신뢰성 파이프라인 한도. 없음. 없음.

RPC는 순서가 보장되고 신뢰성이 있으므로, 빈번한 RPC 트래픽은 전송 중인 패킷 뒤에서 지연될 수 있다.

3. RPC 정의

3.1 빈 RPC

using Unity.NetCode;

public struct PingRpc : IRpcCommand
{
}

4. RPC 전송

4.1 클라이언트에서 서버로

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation)]
public partial struct SendChatSystem : ISystem
{

    {
        state.RequireForUpdate<NetworkId>();
    }

    public void OnUpdate(ref SystemState state)
    {
        if (!Input.GetKeyDown(KeyCode.Space))
            return;

        var ecb = new EntityCommandBuffer(Allocator.Temp);
        var entity = ecb.CreateEntity();

        ecb.AddComponent(entity, new ChatMessageRpc

5. RPC 수신

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ReceiveChatSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, chat, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ChatMessageRpc>>()
                .WithEntityAccess())
        {
            var sourceConnection = req.ValueRO.SourceConnection;
            Debug.Log($"Player {chat.ValueRO.SenderId}: {chat.ValueRO.Message}");

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

처리 후에는 반드시 수신된 RPC Entity를 파괴한다. 그렇지 않으면 같은 RPC가 매 프레임 다시 처리된다.

6. RpcQueue

수동 RPC 스케줄링이 필요한 경우에만 RpcQueue를 사용한다.

using Unity.Entities;
using Unity.NetCode;

var rpcQueue = SystemAPI.GetSingleton<RpcCollection>()
    .GetRpcQueue<MyRpc, MyRpc>();
var bufferLookup = SystemAPI.GetBufferLookup<OutgoingRpcDataStreamBuffer>();

if (bufferLookup.HasBuffer(connectionEntity))
{
    var buffer = bufferLookup[connectionEntity];
    rpcQueue.Schedule(buffer, new MyRpc());
}

using Unity.Networking.Transport;

    {
        writer.WriteInt(data.Data);
    }

    public void Deserialize(ref DataStreamReader reader, in RpcDeserializerState state,
        ref MyCustomRpc data)
    {
        data.Data = reader.ReadInt();
    }

    // Execute implementation omitted.
}

Chapter 25. Netcode Ghost 최적화 · 중요도 · 관련성

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 개요

대규모 멀티플레이어 월드는 모든 Ghost를 매 틱마다 모든 클라이언트에 전송할 수 없다. Netcode for Entities는 Optimization Mode, Importance Scaling, Relevancy, 그리고 Ghost별 전송 속도 제어를 사용하여 CPU 비용과 대역폭을 줄인다.

핵심 규칙은 단순하다: 서버는 지금 특정 연결에서 중요한 Ghost에 스냅샷 예산을 써야 한다.

2. Optimization Mode

GhostAuthoringComponent에서 Optimization Mode를 설정한다.

모드 의미 적합한 경우
Dynamic 기본값. 데이터가 변경될 때와 그렇지 않을 때 모두 스냅샷 크기를 최적화한다. 일반적인 움직이는 Ghost.
Static Ghost가 변경되지 않았을 때 데이터 재전송을 피한다. 건물, 지형, 정적 소품.

1.11.0 라인은 변경되지 않은 정적 Ghost에 대해 GhostUpdateSystem 타이밍을 크게 줄일 수 있는 정적 Ghost 최적화를 추가했다.

3. 스냅샷 크기 제한

스냅샷 예산은 여러 수준에서 제한할 수 있다.

설정 의미
NetworkStreamSnapshotTargetSize 연결별 소프트 바이트 목표.
GhostSendSystemData.MaxSendEntities 스냅샷당 최대 Entity 수.
GhostSendSystemData.MaxSendChunks 스냅샷당 최대 Chunk 수.

예산이 제한되면 중요도가 높은 Ghost가 먼저 전송된다.

4. Importance Scaling

Importance Scaling은 서버에서 Ghost별 전송 우선순위를 계산한다.

4.1 입력 데이터

GhostImportance는 세 종류의 데이터를 결합한다.

데이터 의미
연결별 데이터 플레이어 위치 등 연결 고유 데이터.
Singleton 데이터 타일 크기 등 전역 설정.
Chunk별 공유 데이터 Chunk 수준의 스케일링 파라미터.

4.2 거리 기반 중요도

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

var gridSingleton = state.EntityManager.CreateSingleton(
    new GhostDistanceData
    {
        TileSize = new int3(50, 50, 256),
        TileCenter = new int3(0, 0, 128),
        TileBorderWidth = new float3(1f, 1f, 1f),
    });

state.EntityManager.AddComponentData(gridSingleton, new GhostImportance
{
    ScaleImportanceFunction = GhostDistanceImportance.ScaleFunctionPointer,
    GhostConnectionComponentType = ComponentType.ReadOnly<GhostConnectionPosition>(),
    GhostImportanceDataType = ComponentType.ReadOnly<GhostDistanceData>(),
    GhostImportancePerChunkDataType = ComponentType.ReadOnly<GhostDistancePartitionShared>(),
});

1.8.0 라인의 주요 변경 사항: GhostDistanceImportance 스케일 함수는 더 이상 baseImportance에 1000을 곱하지 않는다. 이전 곱수를 가정한 기존 커스텀 중요도 코드는 검토가 필요하다.

4.3 GhostDistancePartitionShared 자동 추가

using Unity.NetCode;

GhostDistancePartitioningSystem.AutomaticallyAddGhostDistancePartitionSharedComponent = true;

4.4 Importance Visualizer

PlayMode Tool에는 1.8.0+ 라인부터 Importance Visualizer가 포함된다. 코드에서 추측하는 대신 이를 사용하여 런타임 중요도 분포를 검사한다.

5. Ghost Relevancy

Ghost Relevancy를 통해 서버는 연결별로 특정 Ghost를 숨기거나 노출할 수 있다.

모드 의미
Disabled 기본값. 필터링 없음; 모든 Ghost가 후보.
SetIsRelevant 목록에 있는 Ghost만 해당 연결에 복제됨.
SetIsIrrelevant 목록에 있는 Ghost는 해당 연결에서 제외됨.

5.1 사용 사례

시나리오 구현 아이디어
거리 컬링 멀리 있는 Ghost를 비관련으로 표시.
전장의 안개 시야 밖의 Entity 숨기기.
클라이언트 전용 Ghost 한 클라이언트에게만 Ghost를 관련으로 표시.
팀 필터링 적 전용 내부 데이터 숨기기.

5.2 Relevancy 예시

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
[UpdateInGroup(typeof(GhostSimulationSystemGroup))]
[UpdateBefore(typeof(GhostSendSystem))]
public partial struct RelevancySystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var relevancy = SystemAPI.GetSingletonRW<GhostRelevancy>();

        relevancy.ValueRW.GhostRelevancyMode = GhostRelevancyMode.SetIsRelevant;
        relevancy.ValueRW.GhostRelevancySet.Clear();

        foreach (var (ghostInstance, transform) in
            SystemAPI.Query<RefRO<GhostInstance>, RefRO<LocalTransform>>())
        {
            foreach (var (connId, connPos) in
                SystemAPI.Query<RefRO<NetworkId>, RefRO<GhostConnectionPosition>>())
            {
                float dist = math.distance(
                    transform.ValueRO.Position,
                    connPos.ValueRO.Position);

                if (dist < 100f)
                {
                    relevancy.ValueRW.GhostRelevancySet.TryAdd(
                        new RelevantGhostForConnection(
                            connId.ValueRO.Value,
                            ghostInstance.ValueRO.ghostId),
                        1);
                }
            }
        }
    }
}

이 단순화된 예시는 O(Ghost × 연결) 복잡도를 가진다. 대규모 월드에서는 거리 기반 중요도 파티셔닝을 선호하고, 전장의 안개나 팀 가시성처럼 파티셔닝으로 표현할 수 없는 규칙에만 Relevancy를 사용한다.

5.3 DefaultRelevancyQuery

DefaultRelevancyQuery는 전역 게임 상태 Ghost나 UI 관련 네트워크 상태처럼 모든 연결에 항상 관련된 Ghost Chunk를 정의한다.

5.4 Relevancy와 중요도 결합

1.6.1+ 라인은 PrioChunks.isRelevant를 통해 Ghost Relevancy와 Importance Scaling을 결합하는 기능을 지원하여 더 빠른 Relevancy 경로를 가능하게 한다.

6. 예측 CPU 최적화

6.1 Prediction Switching

멀리 있거나 중요하지 않은 Ghost를 Predicted에서 Interpolated로 전환한다.

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

float distance = math.distance(playerPos, ghostPos);
if (distance > predictionRange)
{
    switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
    {
        TargetEntity = ghostEntity,
        TransitionDurationSeconds = 0.5f,
    });
}

6.2 GhostField 쓰기 접근 최소화

GhostField Component에 쓰기 접근을 가진 Job은 직렬화 검사를 트리거할 수 있다. 예측 루프에서 변경되지 않는 필드에는 읽기 전용 접근을 사용한다.

6.3 물리 재시뮬레이션

20회 이상의 예측 재시뮬레이션 프레임이 필요한 고지연 상황:

  • 메인 스레드 통합이 더 저렴한 경우 PhysicsStep Singleton을 통해 멀티스레드 물리를 비활성화한다.
  • 반복되는 물리 틱을 묶기 위해 MaxPredictionStepBatchSizeRepeatedTick을 사용한다.

7. MaxIterateChunks

GhostSendSystemData.MaxIterateChunks는 단일 틱에서 처리하는 Chunk 수를 제한한다.

설정 의미
MaxIterateChunks 틱당 직렬화 CPU 비용을 제한함.
MaxSendChunks 스냅샷에 포함되는 Chunk 수를 제한하여 대역폭을 줄임.

이 설정들은 서로 다른 문제를 해결한다. MaxIterateChunks는 CPU 작업을 분산하고, MaxSendChunks는 스냅샷 크기를 제한한다.

8. MaxSendRate

GhostAuthoringComponent.MaxSendRate는 Ghost의 전송 빈도를 Hz 단위로 제한한다.

NetworkTickRate = 60, MaxSendRate = 10
→ This Ghost can be included once every 6 network ticks.

빈도가 낮거나 중요도가 낮은 Ghost에 사용한다.

9. UseSingleBaseline

GhostAuthoringComponent.UseSingleBaseline = true는 델타 압축에 단일 기준선을 사용한다. CPU 비용을 줄일 수 있지만 대역폭이 증가할 수 있다.

10. 최적화 체크리스트

  • 거의 변경되지 않는 Ghost에는 Static Optimization Mode를 사용.
  • 연결당 대역폭을 제한하기 위해 NetworkStreamSnapshotTargetSize를 설정.
  • 거리 기반 Importance Scaling을 구현.
  • 파티셔닝으로 표현할 수 없는 규칙에만 Relevancy를 적용.
  • 멀리 있는 Ghost를 Interpolated 모드로 전환.
  • 빈도가 낮은 Ghost에는 MaxSendRate를 사용.
  • 변경되지 않는 GhostField 데이터에 대한 쓰기 접근을 피함.
  • Importance Visualizer로 런타임 우선순위를 검사.

Chapter 26. Netcode 물리 통합 & 지연 보상

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. 개요

Netcode for Entities는 네트워크 물리 시뮬레이션을 위해 Unity Physics와 통합된다. 물리 동작은 Ghost가 Interpolated인지 Predicted인지에 따라 다르다.

예측 물리가 실행되려면 씬에 Predicted Ghost가 최소 하나 이상 있어야 한다.

2. Interpolated vs Predicted 물리

2.1 클라이언트의 Interpolated Ghost

  • 위치와 회전은 서버 스냅샷에서 가져온다.
  • Simulate가 비활성화되어 있으므로 물리 오브젝트는 키네마틱처럼 동작한다.
  • PhysicsVelocity는 무시된다.

2.2 클라이언트의 Predicted Ghost

  • 물리는 예측 루프 내에서 실행되며 렌더링된 프레임당 여러 번 실행될 수 있다.
  • PhysicsSystemGroup은 PredictedFixedStepSimulationSystemGroup 내에서 실행된다.
  • 클라이언트는 예측 물리에 대해 서버와 동일한 시뮬레이션 결과를 생성해야 한다.
Netcode initialization
├─ PhysicsSystemGroup moves under PredictedFixedStepSimulationSystemGroup
└─ FixedStepSimulationSystemGroup moves under PredictedFixedStepSimulationSystemGroup

3. 예측 물리 성능

예측 물리는 롤백이 물리를 반복적으로 실행할 수 있으므로 비용이 크다.

3.1 서버 배칭

using Unity.NetCode;

ClientServerTickRate tickRate = default;
tickRate.MaxSimulationStepBatchSize = 4;
tickRate.MaxSimulationStepsPerFrame = 4;

3.2 클라이언트 배칭

using Unity.NetCode;

ClientTickRate clientRate = default;
clientRate.MaxPredictionStepBatchSizeFirstTimeTick = 2;
clientRate.MaxPredictionStepBatchSizeRepeatedTick = 4;

배칭은 오예측 위험을 높일 수 있으므로 기본값이 아닌 성능 트레이드오프로 사용한다.

물리 Ghost의 기본 Transform / Velocity 양자화 값은 1000이다.

높은 양자화 낮은 양자화

더 높은 대역폭. 더 많은 정밀도 손실.

4. 지연 보상

5. 다중 물리 World

네트워크 시뮬레이션에 참여하지 않는 VFX나 파티클 등 클라이언트 전용 효과에는 별도의 물리 World를 사용한다.

5.1 설정

  1. SubScene에 NetCodePhysicsConfig를 추가한다.
  2. Client Non Ghost World를 1과 같이 0이 아닌 값으로 설정한다.
  3. 클라이언트 전용 물리 오브젝트에 같은 값으로 PhysicsWorldIndex를 추가한다.

using Unity.Entities;
using Unity.NetCode;
using Unity.Physics.Systems;

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]

{

    {
    }
}

  • CustomPhysicsProxyDriver는 프록시를 Predicted Ghost에 연결한다.
  • SyncCustomPhysicsProxySystem은 위치와 회전을 동기화한다.

7. PredictedFixedStepSimulationSystemGroup

PredictedFixedStepSimulationSystemGroup은 모든 필수 조건이 충족될 때만 업데이트된다.

요구 사항 의미
NetworkStreamInGame Singleton이 존재 연결이 게임 내 상태.
Predicted Ghost가 최소 하나 존재 예측 물리가 시뮬레이션할 대상이 있음.

7.1 연결 이전의 물리

네트워크 연결이 존재하기 전에 물리가 실행되어야 한다면, 예측 물리 설정을 비활성화하여 일반 FixedStepSimulationSystemGroup 물리가 실행될 수 있도록 한다.

using Unity.Entities;
using Unity.NetCode;

[UpdateInGroup(typeof(InitializationSystemGroup))]
[CreateAfter(typeof(PredictedPhysicsConfigSystem))]
public partial struct DisablePredictedPhysics : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.World.GetExistingSystemManaged<PredictedPhysicsConfigSystem>()
            .Enabled = false;
    }
}

Part III. 최적화와 디버깅

Chapter 28. Chunk 레이아웃 & TypeManager

1. 16 KB Chunk

모든 Archetype은 Entity를 16 KB Chunk에 저장한다. Chunk에는 다음이 포함된다:

  1. 작은 헤더 (메타데이터 — archetype id, 버전 번호, enable-bit 마스크, change version).
  2. 연속적인 SoA 영역: Component 타입별 하나의 열, 각 열은 값의 배열.
  3. 16 KB 경계까지의 패딩.

Chunk 용량 — 얼마나 많은 Entity가 들어가는지 — 은 다음 수식으로 계산된다:

capacity ≈ (16 KB − header) / sum(sizeof(component_i))

작은 Component가 몇 개인 좁은 Archetype은 수백 개의 Entity를 담는다. Component가 많고 큰 뚱뚱한 Archetype은 소수만 담는다. 점유율이 낮은 Chunk는 더 많은 Entity를 담을 수 있었던 바이트를 낭비한다.

2. TypeManager — Component 레지스트리

TypeManager는 World 시작 시 모든 Component, 버퍼, System을 등록하고 각각에 안정적인 타입 인덱스를 부여한다. 런타임은 레지스트리에서 다음을 읽는다:

필드 사용 목적
TypeIndex 쿼리와 Chunk 메타데이터에 사용되는 압축 식별자.
SizeInChunk Chunk에서 이 Component의 Entity당 바이트 수.
Alignment Chunk 내 열 정렬.
ElementSize 버퍼 요소 크기 (IBufferElementData용).
HasBlobReferences / HasEntityReferences 직렬화 및 이동 작업 중 유효성 검사를 가능하게 함.

TypeManager를 직접 건드릴 일은 거의 없다. 중요한 것은 새 Component 타입을 추가하거나, 이름을 변경하거나, 어셈블리 간에 이동하면 타입 레지스트리 재빌드가 트리거된다는 것이며, 이것이 대규모 프로젝트에서 시작 시간이 소요되는 이유다.

Entities 1.4는 불필요한 어셈블리 스캔을 건너뛰어 대규모 프로젝트에서 TypeManager.Initialize가 약 2배 빨라졌다고 보고했다. 6.5에서는 해당 최적화가 계승된다.

3. Chunk 레이아웃 검사 — Archetypes 창

Window → Entities → Archetypes는 World의 모든 Archetype을 나열한다. 각 행에서:

  • Entities — 현재 Entity 수.
  • Chunks — 사용 중인 Chunk 수.
  • Entities / Chunk — 평균 점유율.
  • Capacity — Chunk당 이론적 최대값.

주의해야 할 사항:

읽기 해석
Entity가 1개인 Archetype이 많음 지나치게 세분화된 태그로 인한 Archetype 폭발. 통합할 것.
Entity 수가 많고 점유율이 높은 단일 Archetype 양호 — 적, 발사체, 총알.
Entities / Chunk ≪ Capacity 단편화 — Entity가 다른 Archetype으로 계속 이동하여 점유가 불규칙함 (구조적 변경 남발).
Capacity = 1 Archetype이 너무 넓음 (Component가 너무 큼). 크기가 큰 managed/shared Component를 확인.

4. Archetype 친화적인 Component 설계

영향력 순서의 대략적인 경험 법칙:

  1. 비관리 struct는 비관리 상태로 유지. 단일 managed IComponentData는 Entity당 부가 테이블 저장을 강제하고 Burst를 깨뜨린다.
  2. struct에 패딩을 넣지 말 것. float 뒤가 아닌 앞에 float3를 배치 — 정렬 홀은 SizeInChunk를 부풀린다.
  3. 콜드 필드는 분리. 하나의 System만 필드를 읽는다면 형제 Component로 이동. 사용되지 않는 열도 Entity당 바이트를 소모한다.
  4. 프레임별 add/remove 대신 enableable Component 선호. 13_Structural Change & Safety.md 참조 — 비트 토글이 Archetype 재구성보다 낫다.
  5. Shared Component 값 집합을 제한. 새 값마다 새 Chunk 버킷이 생성된다.
  6. 태그 Component는 바이트 비용이 없지만 Archetype을 추가한다. 모든 Entity에 고유 마커를 태그로 붙이지 말 것.

5. Chunk 버전 번호

각 Component 열은 change version을 추적한다 — 어느 System이 RefRW<T>.ValueRW, SetComponentData, 또는 ECB SetComponent를 통해 쓸 때마다 증가하는 32비트 카운터다.

System은 마지막 실행 이후 버전이 변경되지 않은 Chunk를 건너뛸 수 있다:

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>());
query.SetChangedVersionFilter(typeof(Health));

change filter와 함께 쿼리를 반복하면 해당 System이 마지막 실행된 이후 변경된 Chunk만 방문한다. 이것이 "Health 변경 시 UI 업데이트"와 같은 패턴이 저렴해지는 방식이다.

주의사항:

  • 필터는 Chunk 수준이며 Entity 수준이 아니다. 하나의 Entity 쓰기가 전체 Chunk를 더티로 만든다.
  • RefRW<T>.ValueRW를 통한 읽기(실제로 변경하지 않더라도)도 버전을 올린다.
  • enableable Component 토글도 버전을 올린다.

6. BlobAsset 저장

BlobAssetReference<T>는 크고 읽기 전용인 데이터(메시, LOD 테이블, 내비게이션 필드)를 Chunk 외부에 저장한다. Chunk에는 포인터만 저장된다. 이를 통해 동일한 Baked 데이터를 공유하는 많은 Entity가 있을 때 Chunk 용량이 부풀어 오르는 것을 방지한다.

실용적인 팁: 한 종류의 모든 Entity에 걸쳐 동일한 FixedList 또는 DynamicBuffer가 지배적인 Component가 있다면, 큰 페이로드를 간결한 Component에서 참조하는 blob asset에 저장한다.

7. 예시

7.1 피해야 할 뚱뚱한 Archetype

public struct EnemyState : IComponentData
{
    public float3 Position;
    public quaternion Rotation;
    public float Health;
    public float HealthMax;
    public float Stamina;
    public float StaminaMax;
    public int AIState;
    public float3 LastKnownPlayerPos;
    public float TargetDistance;
    public Entity Target;
    public FixedList512Bytes<float3> PathBuffer;   // 512 bytes per entity!
}

필드 하나인 PathBuffer만으로 Entity당 512바이트를 소모한다. 16 KB Chunk당 용량이 낮은 두 자릿수로 떨어진다. 분리한다:

public struct EnemyStats : IComponentData
{
    public float Health, HealthMax, Stamina, StaminaMax;
    public int   AIState;
    public Entity Target;
}

public struct EnemyPathPoint : IBufferElementData
{
    public float3 Value;
}

이제 dynamic buffer는 자체 저장소에 있고 Chunk 밀도가 회복된다.

7.2 저렴한 UI 업데이트를 위한 change filter

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>(),
                                 ComponentType.ReadWrite<HealthBarLink>());
query.SetChangedVersionFilter(typeof(Health));

state.Dependency = new UpdateHealthBarJob { /* ... */ }
    .ScheduleParallel(query, state.Dependency);

Health가 변경된 Chunk만 방문한다.

8. 문제 해결

증상 원인 / 해결책
대규모 프로젝트에서 느린 시작 TypeManager.Initialize가 너무 많은 어셈블리를 스캔함. Profiler의 시작 추적을 확인하고 최신 Entities 버전을 사용하는지 확인.
Archetype 용량이 1 Component가 너무 큼 (버퍼 용량, 내장 blob). 대용량 데이터를 BlobAssetReference<T> 또는 IBufferElementData로 이동.
change filter가 "변경 없음"에도 매 프레임 실행됨 누군가 쓰지 않으면서 RefRW<T>.ValueRW로 읽고 있음. 가능한 경우 RefRO<T>로 전환.
Archetype 수가 계속 증가함 높은 카디널리티의 Entity별 태그, 또는 연속 값을 키로 하는 Shared Component. 통합할 것.
Entities / Chunk가 Capacity보다 훨씬 적음 구조적 변경 남발 — add/remove가 Entity를 여러 Archetype에 분산시킴. enableable Component로 전환.
managed Component가 CPU 프로파일을 지배함 managed 필드를 UnityObjectRef<T>로 이동하거나 Component를 분리; 비관리 경로가 훨씬 빠름.

Chapter 29. Systems · Entity Inspector · Query Window

1. 개요

주로 사용하는 세 가지 Entities 도구 창:

창 열기 답변
Entities Hierarchy Window → Entities → Hierarchy "어떤 Entity가 존재하는가? 어떤 Component를 가지고 있는가?"
Systems Window → Entities → Systems "어떤 System이 어떤 순서로 실행되며 얼마나 시간이 걸리는가?"
Query Window → Entities → Query "주어진 Component 집합에서 어떤 Entity가 일치하는가? 어떤 System이 이를 쿼리하는가?"

네 번째인 Archetypes (Window → Entities → Archetypes)는 01_Chunk Layout & TypeManager.md에서 다룬다.

2. Entities Hierarchy

GameObject가 아닌 Entity를 위한 실시간 Hierarchy 역할을 한다.

World 선택기 (창 상단)

DefaultGameObjectInjectionWorld (싱글플레이어), ClientWorld/ServerWorld (Netcode), 또는 테스트 픽스처 World 간에 전환한다. World를 변경하면 해당 World의 Entity에 대해 트리가 재실행된다.

트리 뷰

  • 각 행은 Entity; 자식 행은 Parent / LinkedEntityGroup을 통해 연결된 Entity다.
  • 오른쪽 열 수: Component 수, 자식 수.
  • 행 클릭 → 해당 Component가 Inspector에 표시됨.

검색

  • c:Health — Health Component를 가진 Entity로 필터링.
  • n:Enemy — 이름 필터.
  • w:Server — World 필터.
  • 조합: c:Health c:Enemy n:boss.

Inspector

Entity의 Inspector는 다음을 표시한다:

  • Components 섹션 — 각 IComponentData + 현재 값. Play 모드에서 실시간 편집 가능.
  • Relationships — 다른 Entity로의 링크 (Parent, Child, LinkedEntityGroup).
  • Enabled state — IEnableableComponent 토글별.

실시간 편집은 write-through다: 필드를 변경하면 즉시 해당 Chunk의 change version이 올라간다.

3. Systems

Systems 창은 등록된 모든 System을 실제 스케줄러가 틱하는 순서대로 SystemGroup별로 그룹화하여 나열한다.

먼저 읽어야 할 열

열 의미
Name System 타입.
Namespace 위치 (종종 패키지 vs 사용자 코드를 식별).
Time (ms) OnUpdate에 소요된 메인 스레드 시간. 약 매 프레임 업데이트됨.
Queries System이 접근한 쿼리 수.
Entity Count 기본 쿼리에 일치하는 Entity 수.

확인해야 할 사항

읽기 해석
System이 0 ms를 표시하지만 녹색 RequireForUpdate<T> 또는 [RequireMatchingQueriesForUpdate]로 건너뜀. 정상.
System이 일관되게 높은 ms를 표시 실제 작업. 세부 분석을 위해 Profiler를 살펴볼 것.
System이 entity count에 --를 표시 일치하는 쿼리 없음 (또는 쿼리가 아직 빌드되지 않음).
두 System이 예상 순서와 다르게 나타남 [UpdateBefore] / [UpdateAfter] 체인이 잘못됨, 또는 다른 그룹에 있음.

활성화 / 비활성화

각 System에는 체크박스가 있어 — 재컴파일 없이 Play 모드에서 건너뛸 수 있다. A/B 테스트에 유용하다.

System 세부 패널

System을 선택하면 다음이 포함된 사이드 패널이 열린다:

  • Queries — 각 쿼리의 Component 집합 (All/Any/None/Disabled/Present/Absent).
  • Dependencies — 이 System이 [UpdateBefore]/[UpdateAfter]로 지정한 System, 실제 스케줄로 확인됨.
  • Type Info — 클래스와 각 메서드에 [BurstCompile]이 있는지 여부.

Queries 탭에서는 이제 1.4+ / 6.x에서 Disabled / Present / Absent / None Component 상태를 표시하므로 enableable 쿼리가 더 이상 모호하게 보이지 않는다.

4. Query 창

"X에 일치하는 것은?"이라는 질문을 위해 특별히 제작된 창이다.

워크플로:

  1. 플러스 버튼을 통해 All Component를 추가한다.
  2. 필요한 경우 Any / None을 추가한다.
  3. 창의 왼쪽 패널에 일치하는 Entity가, 오른쪽 패널에 이 쿼리를 실행하는 System이 나열된다.

사용 사례:

  • "어떤 System이 Health를 읽는가?" → All에 Health 추가; 오른쪽에 나열된 모든 System이 이를 접근함.
  • "이 Entity가 내 System 쿼리에 왜 일치하지 않는가?" → System 쿼리를 붙여넣어 Hierarchy 창에서 Entity의 Component 존재 여부와 비교.
  • "이 쿼리에 Prefab이 들어오고 있는가?" → 결과 목록에서 Prefab은 아이콘으로 구분됨 (1.4+에서 추가).

Query 창에서는 세션 중 재사용을 위해 쿼리를 저장할 수도 있다.

5. Journaling — 최후 수단 진단

Window → Analysis → Entities Journaling (Project Settings → Entities → Journaling에서 활성화)은 모든 구조적 변경, Component 쓰기, System 업데이트를 링 버퍼에 기록한다. 재생 뷰:

  • Operations — 타임스탬프, 소스 System, 영향받은 Entity와 함께 기록된 각 변경.
  • Entities — Entity별 누가 언제 접근했는지 타임라인.

Journaling에는 런타임 비용이 있다 — 버그를 조사하는 동안에만 켜고 일반 개발 중에는 끈다.

6. 현장 워크플로 — 누락된 업데이트 디버깅

일반적인 버그: "내 System이 Entity를 처리했지만 Component가 변경되지 않았다." 과정:

  1. Entities Hierarchy → Entity를 찾아 Component가 있는지 확인.
  2. Inspector → Play 전에 값을 메모. Play를 누르고 지켜봄.
  3. 값이 변경되지 않으면: Systems 창 → 관련 System을 찾아 Time (ms)를 확인. 0이면 건너뛰어지고 있음.
  4. 시간이 > 0이지만 값이 여전히 변경되지 않으면: Query 창 → System 쿼리 + Entity Component를 붙여넣음. Entity가 일치하는지 확인.
  5. 여전히 막힌다면: 1~2 프레임 동안 Journaling 활성화. 해당 Component에 대한 다른 System의 쓰기를 확인하여 덮어쓰는지 찾아본다.

7. 문제 해결

증상 원인 / 해결책
Play 모드에서 Entities Hierarchy가 비어 있음 World 선택기가 잘못된 World로 설정됨 (예: Client만, Server 없음).
실시간 필드가 업데이트되지 않음 도메인 리로드 후 캐시된 Entity를 보고 있음. 다시 선택.
Systems 창은 0 ms로 표시하지만 System이 분명히 실행됨 실제 작업이 ScheduleParallel에 있으면 Profiler가 이를 Chunk Job 마커 아래에 표시할 수 있다. 이 경우 메인 스레드 OnUpdate는 실제로 0이다 — Profiler에서 Job 타이밍을 확인하라.
Query 창이 내 System이 찾지 못한 Entity를 찾음 System이 Query 창에 알려지지 않은 WithAll<T>()나 change filter를 추가함. 정확히 일치시킬 것.
Journaling 창이 비어 있음 Project Settings에서 Journaling이 활성화되지 않았거나 버퍼가 롤오버됨.
Entity와 GameObject가 동기화되지 않아 보임 SubScene이 열려 있음 (흰색 아이콘) — 닫으면 (노란 아이콘) Baker가 실행되어 Entity가 나타남.

Chapter 30. Profiler · 병목 분석

1. 개요

Unity Profiler (Window → Analysis → Profiler)는 DOTS에서 "이 프레임이 왜 느린가"라는 질문에 대한 기본 도구다. Entities는 특정 System, Job, Chunk에 시간을 귀속할 수 있는 충분한 마커를 발생시킨다 — 어디를 봐야 하는지 알면.

이 페이지는 프로파일 읽기, 일반적인 DOTS 병목 패턴, 그리고 이에 대한 대응 방법을 안내한다.

2. 프로파일링 전 준비

  1. Deep Profile은 절제해서 사용. 특정 managed 경로를 추적하는 데 유용하지만 타이밍을 왜곡한다 — 성능 수치에는 절대 Deep Profile을 신뢰하지 말 것.
  2. "Record in Editor & Standalone" 활성화하여 Editor 프레임뿐 아니라 빌드도 캡처한다.
  3. Profiler 대상을 player로 설정, Editor가 아닌, 릴리스 성능을 측정할 때. Editor 오버헤드는 상당하다.
  4. 읽기 전 몇 초간 실행; 첫 몇 프레임에는 도메인 리로드와 JIT 작업이 포함된다.

3. CPU Usage 모듈 읽기

DOTS 프레임에서 최상위 마커:

PlayerLoop
├── Initialization.*
├── Update.ScriptRunBehaviourUpdate              (MonoBehaviour.Update)
├── SimulationSystemGroup                        ← DOTS simulation lives here
│   ├── SystemA.OnUpdate
│   ├── SystemB.OnUpdate
│   └── EndSimulationEntityCommandBufferSystem.Playback
├── FixedStepSimulationSystemGroup
├── PresentationSystemGroup
├── Render
└── WaitForTargetFPS

SystemA.OnUpdate 마커를 드릴다운하여 서브콜을 확인한다 — ScheduleParallel, Job 완료 대기, EntityManager 호출.

4. 시간이 실제로 소요되는 곳 — 일반적인 패턴

4.1 메인 스레드 OnUpdate 지배

단일 SystemX.OnUpdate가 10+ ms를 차지한다면, 작업이 메인 스레드에서 실행되고 있다. 확인:

  • IJobEntity.ScheduleParallel이었어야 하는 것을 타이트한 루프에서 SystemAPI.Query로 사용하고 있는가?
  • managed 필드로 인해 메인 스레드에 강제된 SystemBase인가?
  • Entity당 EntityManager.AddComponent/RemoveComponent가 호출되고 있는가?

4.2 Job 타이밍

Job은 Timeline 뷰 (또는 Hierarchy 뷰의 "Workers" 아래)에 나타난다. 확인할 사항:

  • 하나의 Job 레인이 완전히 포화되어 있고 다른 레인은 유휴 상태 → 작업이 Chunk에 걸쳐 분산되지 않고 있음.
  • 메인 스레드가 Job을 기다리고 있음 (JobHandle.Complete() 마커) → .Complete()가 너무 일찍 호출됨; 프레임 더 뒤로 이동.
  • 거대한 Burst 함수 이름 → 정상, 심볼이 전체 제네릭 인스턴스화임.

4.3 EntityManager 핫스팟

다음 이름의 마커를 찾는다:

  • EntityManager.SetArchetype / Move Entity — 구조적 변경 남발. 13_Structural Change & Safety.md 참조.
  • EntityCommandBufferSystem.OnUpdate (Playback) — 비용이 큰 ECB 재생. 보통 프레임당 너무 많은 명령이 기록되고 있음을 의미.
  • TypeManager.Initialize — 시작 시에만 나타남; 프레임당 나타난다면 무언가가 World를 다시 로드하고 있음.

4.4 GC.Alloc 마커

시뮬레이션에서 GC.Alloc이 있으면 DOTS에서의 회귀다. 출처:

  • OnUpdate에서 할당하는 SystemBase.
  • System 내의 managed 컬렉션에 대한 foreach.
  • 박싱 — 예: struct에 대한 object.ToString().

Burst System은 Profiler의 Memory 모듈에서 0 할당을 표시해야 한다.

5. 고정해야 할 Profiler 모듈

  • CPU Usage — 마커의 타임라인 및 계층.
  • Memory — 할당된 힙; 증가하는 GC.Used Memory를 주시.
  • Entities (사용 가능한 경우) — 시간에 따른 System별 타이밍을 보여주는 DOTS 전용 모듈.

Entities Profiler 모듈은 1.8.0 / 6.x와 함께 출시되어 타임라인에서 Systems 창과 정렬된다. 멀티 프레임 스파이크를 확인하면 이것이 가장 빠르게 귀속할 수 있는 방법이다.

네트워크 프로젝트에서는 Ghost 스냅샷, 예측 틱, Netcode 전용 대역폭을 검사하기 위해 ../DOTS Workflows/24_Netcode Profiler & Debugging.md와 함께 사용한다.

6. 일반적인 병목과 해결책

패턴 Profiler의 증상 해결책
Entity별 Entities.ForEach (레거시) 작은 샘플이 많은 메인 스레드 중심 OnUpdate IJobEntity.ScheduleParallel로 포팅.
프레임별 AddComponent/RemoveComponent Move Entity 마커가 지배 상태를 enableable Component로 변환.
ECB 과부하 ...ECBS.Playback이 ms 소요 명령 배치 (Entity당 하나 대신 Chunk당 하나), 또는 배치 크기를 늘림.
Shared Component 값 폭발 SetSharedComponent 지배 값을 버킷으로 나눔 (더 적은 고유 키로 양자화).
프레임 중간에 .Complete() 호출 메인 스레드에서 긴 JobHandle.Complete 대기 완료를 프레임 끝으로 연기하거나, 메인 스레드가 아직 결과가 필요하지 않도록 구조를 재편.
Burst 폴백 핫 메서드가 디맹글되지 않은 C# 이름 표시 캡처된 비블리터블 타입이 IL로 폴백됨. Burst Inspector 확인.
Archetype 단편화 작은 Chunk 다수; EntityManager.*Query 마커가 높음 Archetype이 너무 세분화됨 — 태그를 통합하거나 enableable Component를 사용.
Bake 시간 회귀 Play 중 Bake 마커 도메인 리로드가 Baking을 재트리거함. 보통 Editor 전용; 빌드에서 확인.

7. 프레임 예산 작업 예시

16.6 ms 예산(60 Hz)을 가정한다. Render를 제외하면:

  • SimulationSystemGroup: 6 ms
    • EnemyAISystem.OnUpdate: 2.1 ms (Burst, 4 워커 병렬 Job)
    • MovementSystem.OnUpdate: 1.4 ms (Burst, 병렬)
    • EndSimulationECBS.Playback: 1.8 ms ← 의심스러움
    • 나머지: 0.7 ms

소수의 구조적 변경에 대한 1.8 ms의 ECB 재생은 보통 가장 큰 개선 지점이다. 명령 수를 확인한다:

  1. 1 프레임 동안 Entities Journaling → System별 Instantiate/AddComponent/DestroyEntity 수.
  2. System이 매 프레임 수백 개의 명령을 기록하면 배치 처리 (Chunk당 하나, 또는 청크 인덱스 정렬 키를 사용한 병렬 레코더로 전환).

8. 독립 실행형 프로파일링

실제 빌드의 경우:

  3. 결과가 플레이어에게 보이는 것에 대해 더 정직하다. Editor 측정은 코드와 관련 없는 이유로 2~5배 느릴 수 있다.

Chapter 31. Managed Object Reference 감사

1. 개요

managed Unity object 참조는 ECS System을 실수로 빠른 경로에서 벗어나게 하는 가장 쉬운 방법 중 하나다. 이 감사는 Component 데이터, System, 또는 렌더링 글루에서 Burst, Job, 또는 깔끔한 SubScene Baking을 차단하는 방식으로 UnityEngine.Object 참조를 보유하는 곳을 찾는 데 도움이 된다.

ECS 핵심 마이그레이션 후, 또는 Profiler 추적이 데이터 지향 코드를 예상했던 곳에서 managed 작업을 보여줄 때마다 사용한다.

2. 검색할 항목

패턴 중요한 이유
class .*: IComponentData Managed Component 데이터; 유효하지만 핫 경로에는 보통 이상적이지 않음.
UnityEngine.Object managed Unity object 참조의 광범위한 마커.
GameObject, Transform, Component 런타임 씬 오브젝트 참조는 보통 시뮬레이션 데이터에 속하지 않음.
Mesh, Material, Texture, AudioClip 종종 UnityObjectRef<T>이거나 프레젠테이션 측 캐시된 데이터여야 함.
Resources.Load 런타임 에셋 조회; Baking/콘텐츠 System으로 처리하는 것이 더 나음.
ScriptableObject 유용한 Authoring 소스이지만, 런타임 Job은 Baked 값이나 blob을 읽어야 함.

감사는 ECS와 접하는 코드에 한정한다. MonoBehaviour 전용 UI나 Editor 도구는 DOTS 패턴으로 강제할 필요가 없다.

3. 분류 규칙

발견 가능한 조치
핫 시뮬레이션 쿼리의 managed Component 비관리 IComponentData, BlobAssetReference<T>, 또는 UnityObjectRef<T>로 변환.
프레젠테이션에서만 사용되는 managed Component Burst/Job 시뮬레이션 경로에서 격리되어 있다면 유지.
매 프레임 읽히는 ScriptableObject 필요한 값을 Component나 blob asset에 Bake.
ECS 데이터의 Mesh/Material 참조 UnityObjectRef<Mesh> / UnityObjectRef<Material>를 사용하고 프레젠테이션 코드에서 역참조.
Component 데이터에 저장된 GameObject/Transform ECS 데이터로 교체하거나 참조를 하이브리드 프레젠테이션 브리지로 이동.

4. 감사 예시

// Suspicious: managed component data used as per-entity visual state.
public class EnemyVisual : IComponentData
{
    public Material Material;
}

더 나은 방법:

using Unity.Entities;
using UnityEngine;

public struct EnemyVisual : IComponentData
{
    public UnityObjectRef<Material> Material;
}

그런 다음 메인 스레드 프레젠테이션 System에서 역참조하거나 리졸빙된 Material을 렌더러 레지스트리에 캐시한다.

5. Profiler 징후

Profiler 증상 확인 사항
시뮬레이션 System이 매 프레임 할당 OnUpdate의 managed object 조회 또는 LINQ/컬렉션 할당.
데이터 전용이어야 할 System에서 Burst가 비활성화됨 쿼리 경로에서 managed Component 또는 managed API 접근.
메인 스레드에서 작은 렌더링 조회가 많음 Entity별 루프 외부에서 UnityObjectRef<T>.Value 결과를 캐시.
SubScene이 예상보다 더 자주 리베이크됨 Authoring 스크립트나 object 참조가 불필요한 의존성을 생성하고 있음.

6. 검증

  1. §2의 검색 패턴을 다시 실행한다.
  2. 핫 시뮬레이션 System이 비관리 Component만 쿼리하는지 확인한다.
  3. Burst로 컴파일될 것으로 예상되는 System/Job에 대해 Burst Inspector를 확인한다.
  4. 대표적인 씬을 프로파일링하여 managed 할당이 핫 경로에서 사라졌는지 확인한다.
  5. 깨끗한 Editor 시작에서 Play 모드에 진입하여 SubScene Baking이 여전히 참조를 확인하는지 검증한다.

7. 문제 해결

증상 원인 / 해결책
변환된 Component가 여전히 Burst에서 쿼리될 수 없음 남은 필드가 managed임. 중첩 struct도 확인.
UnityObjectRef<T>.Value가 Play 모드에서 null 반환 참조된 에셋이 Bake되거나 로드되지 않음. Baker와 SubScene/콘텐츠 워크플로를 확인.
변환 후에도 시각화 System이 느림 매 프레임 Entity당 역참조하고 있음. 공유 에셋으로 캐시하거나 컴팩트 조회 키를 Bake.
ScriptableObject 변경이 런타임 데이터에 반영되지 않음 값이 Bake되었음. SubScene을 다시 가져오거나 적절한 Baking 의존성을 추가.
Editor 전용 참조가 player 빌드에서 실패 Editor API를 #if UNITY_EDITOR 뒤에 두고 런타임 안전한 데이터를 Bake.

Part IV. 마이그레이션

Chapter 32. Entities 1.x → 6.5 개요

1. 무엇을 해야 하는가

Entities 1.x에서 6.5로 업그레이드하는 것은 주로 하나의 패키지 배포 전환에서 세 가지 DOTS 마이그레이션이다:

  1. Editor/패키지 배포 업그레이드. Unity 6000.4+ (com.unity.entities 6.4) 또는 6000.5+ (Entities 6.5)로 이동하며, 여기서 Entities는 Editor에 내장된 Core Package가 된다.
  2. ECS 데이터 정리. ECS와 접하는 데이터에서 managed Unity object 참조를 검토하고 핫 경로 Component 데이터를 비관리 Component, Baked 값, BlobAssetReference<T>, 또는 UnityObjectRef<T>로 마이그레이션한다.
  3. ECS API 마이그레이션. Entities 1.4에서 폐기된 레거시 API — Entities.ForEach, IAspect — 는 6.5에서도 여전히 폐기됨. 또한: ComponentLookup.GetRefRWOptional → TryGetRefRW.

각 마이그레이션은 이 폴더의 자체 하위 문서에서 다룬다. 이 페이지는 지도 — 어떤 것을 언제, 어떤 순서로 해야 하는지.

명명 주의: Entities 패키지 버전 (1.4 / 6.4 / 6.5)은 Unity Editor 버전 (6000.3 / 6000.4 / 6000.5)과 다르다. 6.4부터 관행적으로 정렬되지만 별도의 릴리스 노트에서 추적된다. 이 문서는 두 축으로 처리하며, Changelog 페이지 (../Changelog/Entities 1.4 → 6.5 Key Changes.md)도 마찬가지다.

2. 권장 순서

  • Editor 먼저 — 6.x API에 대해 컴파일되는 것이 없기 때문에.
  • Package Manager → Core Package 다음 — 이것 없이는 중복/충돌하는 패키지 해석이 생긴다.

  4. 저장/로드 확인. raw Entity 핸들은 World 로컬이며 안정적이지 않다. 게임 소유 ID, 콘텐츠 키, 또는 Authored 데이터를 저장한다.

  • Editor가 프로젝트를 전혀 열지 못함 (드물다; 보통 ProjectVersion.txt 불일치).
  • packages-lock.json 해석이 잘못된 그래프를 생성하고 삭제해도 해결되지 않음.
  • 의존하는 중요 패키지 (서드파티)가 아직 6.5와 호환되지 않음.

계획: git checkout pre-entities-6.5, 이전 Unity 버전을 다시 설치하고, 게임이 이전 버전에서 여전히 실행되는지 확인한다.

부분적으로 완료되었지만 막힌 마이그레이션은 깔끔하게 중단하고 태그에서 재시작해야 한다 — 반만 마이그레이션된 코드를 배포하지 말 것.

Chapter 33. Package Manager → Core Package

1. 변경 사항

Unity 6000.4+에서 Entities (및 동반 DOTS 패키지)는 Package Manager를 통해 설치되지 않는다. Editor 내부에 Core Package로 포함된다. 프로젝트 파일은 세 곳에서 변경된다:

  • Packages/manifest.json — com.unity.entities* 항목이 제거된다.
  • Packages/packages-lock.json — 해석된 그래프가 줄어들고; 영향받는 패키지는 "source": "builtin"으로 나타난다.
  • Package Manager 창 — 패키지가 이제 "In Project" 아래에 Built-in으로 표시된다.

C# 코드는 변경되지 않는다. 어셈블리 이름 (Unity.Entities, Unity.Collections 등)은 동일하며; .asmdef 참조도 변경되지 않는다.

Chapter 34. Managed Object 참조 → UnityObjectRef

1. 개요

오래된 ECS 코드를 마이그레이션할 때 일반적인 문제는 managed object 참조가 런타임 Component 데이터로 누출되는 것이다. Entities 6.5에서는 가능한 경우 게임플레이 Component를 비관리 상태로 유지하고, ECS 데이터가 메시, 머티리얼, 텍스처, 오디오 클립, 또는 Authored ScriptableObject와 같은 Unity object를 가리켜야 할 때 UnityObjectRef<T>를 사용한다.

이 페이지는 DOTS Component 설계로 범위가 제한된다. 엔진 전반의 Unity object 식별자 API는 다루지 않는다.

2. 찾아야 할 것

managed Unity object를 직접 저장하는 Component를 검색한다:

패턴 검토 이유
IComponentData를 구현하는 class Component Managed Component는 허용되지만 더 느리고 메인 스레드로 제한됨.
Component 데이터 내 UnityEngine.Object 필드 보통 managed Component나 잘못된 비관리 Component 설계를 나타냄.
GameObject, Transform, Material, Mesh, Texture, AudioClip 필드 이것이 Authoring, 프레젠테이션, 또는 UnityObjectRef<T>에 속하는지 결정.
Entity당 Resources.Load를 호출하는 런타임 System 에셋 해석을 Baking 또는 중앙 콘텐츠 System으로 이동.

3. 이전과 이후

이전 — managed Component

using Unity.Entities;
using UnityEngine;

public class ProjectileVisual : IComponentData
{
    public Mesh Mesh;
    public Material Material;
}

이것은 작동하지만 Component를 managed로 만든다. Managed Component는 Burst/Job 유연성을 줄이고 보통 Entity별 게임플레이 데이터에는 적합하지 않다.

이후 — UnityObjectRef<T>를 사용한 비관리 Component

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

참조를 한 번 Bake한다:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

4. 런타임 접근 패턴

시뮬레이션 Job을 순수하게 데이터 지향으로 유지한다. Managed Unity API가 허용되는 곳에서만 Unity object를 역참조한다.

using Unity.Entities;
using UnityEngine;

public partial class ProjectileVisualUploadSystem : SystemBase
{
    protected override void OnUpdate()
    {
        foreach (var visual in SystemAPI.Query<RefRO<ProjectileVisual>>())
        {
            Mesh mesh = visual.ValueRO.Mesh.Value;
            Material material = visual.ValueRO.Material.Value;

            if (mesh == null || material == null)
                continue;

            // Upload to a rendering registry, material property block cache, etc.
        }
    }
}

고용량 렌더링의 경우 Entity당 프레임별 역참조를 피한다. 컴팩트한 ECS 값을 키로 하는 프레젠테이션 측 레지스트리에 리졸빙된 오브젝트를 캐시한다.

5. UnityObjectRef<T>를 사용하지 않는 경우

경우 더 나은 선택
순수 게임플레이 데이터 기본형 / 수학 / blob 데이터를 IComponentData에 직접 저장.
대형 읽기 전용 테이블 BlobAssetReference<T>.
같은 SubScene에 Bake된 Entity Prefab Entity Prefab 참조.
콘텐츠 워크플로를 통해 스트리밍되는 Prefab EntityPrefabReference.
Entity당 복잡한 managed 동작 시뮬레이션 경로 외부에 유지하거나 managed 프레젠테이션 System에 격리.

6. 마이그레이션 체크리스트

  • Unity object 참조만 저장하기 위해 사용되는 managed Component 데이터를 찾는다.
  • 비관리 Component가 적절한 경우 해당 필드를 UnityObjectRef<T>로 교체한다.
  • 오브젝트 할당을 Baker로 이동한다.
  • .Value 역참조를 Burst Job 외부에 유지한다.
  • 많은 Entity가 같은 에셋을 공유할 때 역참조된 오브젝트를 캐시한다.
  • 런타임에 ScriptableObject를 읽는 대신 대형 불변 게임플레이 데이터에는 BlobAssetReference<T>를 사용한다.

7. 문제 해결

증상 원인 / 해결책
Component를 Burst Job에서 사용할 수 없음 여전히 managed 상태임. Unity object 필드를 UnityObjectRef<T>로 변환하거나 프레젠테이션 데이터를 분리.
.Value가 null 참조된 에셋이 로드되지 않았거나 파괴됨. Authoring 참조와 콘텐츠 로딩 경로를 확인.
System이 매 프레임 할당 Entity당 에셋을 리졸빙하거나 로딩하고 있음. 참조를 Bake하고 리졸빙된 managed 오브젝트를 캐시.
게임플레이 System이 런타임에 ScriptableObject에 의존 필요한 값을 Component 데이터나 blob asset에 Bake.
빌드가 에셋을 제거함 에셋이 런타임 문자열 경로가 아닌 Baking/콘텐츠 워크플로를 통해 참조되는지 확인.

Chapter 35. foreach → IJobEntity 마이그레이션

1. 상태 — 폐기됨, 아직 제거되지 않음

레거시 Component 반복 API는 Entities 1.4+ / 6.5에서 obsolete 상태지만 여전히 컴파일된다:

레거시 6.5에서의 상태 대체
Entities.ForEach 1.4부터 폐기됨; 매 사용마다 폐기 경고; Unity 변경 로그는 향후 주요 릴리스에서 제거 예정이라고 명시 SystemAPI.Query<...> (메인 스레드) 또는 IJobEntity (Job)
Job.WithCode ForEach와 함께 폐기됨 일반 IJob struct, 또는 OnUpdate에 인라인
IJobForEach (1.0 이전) 1.4 훨씬 이전에 제거됨 IJobEntity

여기에 오게 된 것은 1.x 프로젝트를 6000.5+에서 열어 컴파일러가 경고를 많이 내고 있거나, 향후 주요 패키지 릴리스에서 API가 제거되기 전에 경고를 지우고 싶기 때문이다. 패턴을 이해하면 리팩터링은 기계적이다.

2. 두 가지 대체 API

  • SystemAPI.Query<T, U, ...> — OnUpdate 내부의 메인 스레드 반복. 단순하고 Job 오버헤드 없이 소규모 워크로드나 managed API가 필요한 경우에 이상적.
  • IJobEntity — 소스 생성 병렬 Job. DOTS의 성능이 나오는 멀티스레드 작업에 준비되면 드롭인 교체.

배경: ../DOTS Workflows/11_IJobEntity · SystemAPI.Query.md.

3. 포팅 — Entities.ForEach → SystemAPI.Query

3.1 이전

public partial class HealthRegenSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float dt = Time.DeltaTime;

        Entities
            .WithAll<Alive>()
            .WithNone<Dead>()
            .ForEach((ref Health health, in Regen regen) =>
            {
                health.Value = math.min(health.Value + regen.Rate * dt, health.Max);
            })
            .ScheduleParallel();
    }
}

3.2 이후 — SystemAPI.Query (메인 스레드)

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>()
                          .WithAll<Alive>()
                          .WithNone<Dead>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRO.Max);
        }
    }
}

형태 변경에 대한 참고 사항:

  • SystemBase → ISystem (더 저렴하고 Burst 가능). 이전 코드가 managed 필드를 캡처한 경우 SystemBase를 유지해야 할 수 있다.
  • ref Health → RefRW<Health> 튜플 요소. .ValueRO로 읽고 .ValueRW로 쓴다.
  • in Regen → RefRO<Regen>.
  • Time.DeltaTime → SystemAPI.Time.DeltaTime.
  • 필터 (WithAll/WithNone)는 Query 빌더로 이동.

4. 포팅 — Entities.ForEach.ScheduleParallel → IJobEntity

ForEach가 병렬로 스케줄되었을 때 Job 형태는 직접 1:1 포팅이며 병렬성을 유지한다.

4.1 이후 — IJobEntity

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

필터는 Job struct에 어트리뷰트로 붙인다:

[WithAll(typeof(Alive))]
[WithNone(typeof(Dead))]
public partial struct HealthRegenJob : IJobEntity { ... }

또는 스케줄 시 쿼리를 전달한다:

var q = SystemAPI.QueryBuilder()
    .WithAll<Health, Regen, Alive>()
    .WithNone<Dead>()
    .Build();
new HealthRegenJob { DeltaTime = dt }.ScheduleParallel(q);

5. 캡처와 대체

레거시 Entities.ForEach는 .WithReadOnly, .WithDisposeOnCompletion 등으로 로컬을 캡처할 수 있었다. IJobEntity에서 캡처는 Job struct의 필드다 — Burst는 정확히 무엇이 전달되는지 안다.

// BEFORE
var buffer = new NativeArray<int>(count, Allocator.TempJob);
Entities.ForEach((ref Foo f) => { /* use buffer */ })
    .WithDisposeOnCompletion(buffer)
    .ScheduleParallel();

// AFTER
[BurstCompile]
partial struct FooJob : IJobEntity
{
    [ReadOnly] public NativeArray<int> Buffer;

    void Execute(ref Foo f) { /* use Buffer */ }
}

var buffer = new NativeArray<int>(count, Allocator.TempJob);
state.Dependency = new FooJob { Buffer = buffer }.ScheduleParallel(state.Dependency);
buffer.Dispose(state.Dependency);   // chained dispose replaces WithDisposeOnCompletion

어트리뷰트 대체:

  • .WithReadOnly(x) → [ReadOnly] public NativeArray<T> X;
  • .WithDisposeOnCompletion(x) → 스케줄링 후 x.Dispose(handle).
  • .WithoutBurst() → Job struct에 [WithoutBurst] 적용. 단, 애초에 그것이 필요했던 이유를 제거할 수 있는지 강하게 검토하라.
  • .WithEntityAccess() → Execute에 Entity entity 매개변수 추가.

6. SystemBase가 여전히 필요한 경우

이전 System에 진짜 managed 필드가 있다면 (Input System, UI Toolkit, 외부 서비스) ISystem으로 기계적으로 변환할 수 없다. 두 가지 옵션:

  1. SystemBase로 유지하고 OnUpdate 내부에서 SystemAPI.Query를 사용. 단일 스레드.
  2. 분리 — managed 세계를 연결하는 SystemBase와 Burst 작업을 하는 ISystem으로 나눔 — ../DOTS Workflows/08_System — ISystem vs SystemBase.md 참조.

7. 검증

System을 포팅한 후:

  1. 동작을 단위 테스트. 포팅은 로직을 변경해서는 안 된다; 출력이 다르면 리팩터링에서 버그가 발생한 것이다.
  2. 프로파일링. 병렬 ForEach와 IJobEntity.ScheduleParallel은 동등한 작업에 대해 5% 이내여야 한다. 큰 회귀는 Burst가 동작하지 않음을 의미 — Burst Inspector를 검사.
  3. 이전 System 파일 삭제. #if LEGACY 폴백을 유지하려 하지 말 것 — 썩는다.

8. 문제 해결

증상 원인 / 해결책
warning: Entities.ForEach is obsolete 6.5에서 예상됨 — 코드는 여전히 컴파일됨. 향후 주요 릴리스에서 제거되기 전에 경고를 지우기 위해 위와 같이 포팅.
error: 'SystemAPI' does not contain 'Query' System이 partial이 아니거나, struct가 ISystem을 올바르게 구현하지 않음.
IJobEntity가 컴파일되지만 실행되지 않음 struct와 OnUpdate에 [BurstCompile]이 없음; 소스 생성기가 비partial 타입을 스케줄하지 않음.
Job이 오래된 값을 읽음 .Schedule*가 호출되기 전에 로컬을 캡처함. 스케줄링 전에 Job struct에 필드를 설정해야 함.
포팅 후 성능 회귀 Burst 폴백 확인 (Burst Inspector), 또는 managed 타입이 필드에 몰래 들어옴.
포팅 후 컴파일 타임 문제만 발생 ForEach API에만 적용되는 람다 클로저 문법이 아닌 어트리뷰트를 통해 캡처된 로컬을 제거했는지 교차 확인.

Chapter 36. IAspect 폐기 — Aspect에서 마이그레이션

1. 상태 — 폐기됨, 아직 제거되지 않음

IAspect는 Entities 1.4에서 폐기됨으로 표시되었고 6.x 라인에서도 폐기됨 상태를 유지한다. 여전히 컴파일되고 실행되며, 6.5 API 참조에서 IJobEntity.Execute의 유효한 매개변수 종류로 여전히 나열되지만 — 컴파일러는 폐기 경고를 발생시키며, Unity는 향후 주요 릴리스에서 제거될 것이라고 밝혔다. 새 코드는 사용하지 말고; 기존 코드는 적절한 시기에 마이그레이션해야 한다.

IAspect는 System이 Component 집합에 대한 "뷰"를 선언하고 Entities.ForEach나 IJobEntity에서 단일 인수로 사용하도록 도입되었다. 실제로는 스스로의 비용을 정당화하지 못하는 소스 생성과 병렬 타입 시스템 레이어를 추가했다:

  • Aspect 작성은 Component 매개변수 목록을 중복했다.
  • 디버거가 생성된 래퍼 코드를 단계적으로 실행했다.
  • IJobEntity와 SystemAPI.Query에서 RefRW<T> / RefRO<T>는 추가 타입 없이 동일한 "Component를 묶음으로 전달"하는 편의성을 달성한다.

API 표면이 여전히 남아 있으므로 Editor 업그레이드만으로 코드는 깨지지 않는다 — 이 마이그레이션은 경고를 정리하고 2.0을 준비하기 위한 것이다.

2. 기계적 리팩터링

2.1 시작점

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public readonly partial struct MovementAspect : IAspect
{
    public readonly RefRW<LocalTransform> Transform;
    public readonly RefRO<Velocity>       Velocity;
    public readonly RefRO<Speed>          Speed;

    public void Step(float dt)
    {
        Transform.ValueRW.Position += Velocity.ValueRO.Direction * Speed.ValueRO.Value * dt;
    }
}

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var m in SystemAPI.Query<MovementAspect>())
            m.Step(dt);
    }
}

2.2 이후 — 직접 Component 매개변수

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var (transform, velocity, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<Velocity>, RefRO<Speed>>())
        {
            transform.ValueRW.Position += velocity.ValueRO.Direction * speed.ValueRO.Value * dt;
        }
    }
}

두 가지 변경:

  1. Aspect 타입이 사라진다. 필드가 SystemAPI.Query의 튜플 요소가 된다.
  2. Aspect에 있던 메서드 (Step)가 인라인화된다. 인라인하기 너무 크면 동일한 매개변수를 받는 static 헬퍼를 추출한다 — Burst 친화적으로 유지된다:
static void Step(ref LocalTransform t, in Velocity v, in Speed s, float dt)
{
    t.Position += v.Direction * s.Value * dt;
}

호출:

Step(ref transform.ValueRW, velocity.ValueRO, speed.ValueRO, dt);

3. IJobEntity에서의 Aspect

3.1 이전

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;
    void Execute(MovementAspect aspect) => aspect.Step(DeltaTime);
}

3.2 이후 — Execute 매개변수로서의 Component

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in Velocity velocity, in Speed speed)
    {
        transform.Position += velocity.Direction * speed.Value * DeltaTime;
    }
}

IJobEntity는 RefRW<T> / RefRO<T>가 아닌 일반 ref / in 매개변수를 사용한다는 점에 주의. (생성기가 내부적으로 Component를 청크 배열 접근으로 래핑한다.)

4. DynamicBuffer를 래핑한 Aspect

// BEFORE
public readonly partial struct PathAspect : IAspect
{
    public readonly DynamicBuffer<Waypoint> Path;
    public void Advance(int n) => Path.RemoveRange(0, n);
}

// AFTER — access buffer directly
foreach (var (path, e) in SystemAPI.Query<DynamicBuffer<Waypoint>>().WithEntityAccess())
{
    path.RemoveRange(0, 1);
}

Job에서 DynamicBuffer<T>는 유효한 Execute 매개변수다 — 추가 작업이 필요 없다.

5. 크로스 Entity Aspect

때로 Aspect는 "다른 Entity의 Component를 들여다봐야 한다"는 목적으로 ComponentLookup<T>를 래핑했다. 대체:

// BEFORE
public readonly partial struct AITargetAspect : IAspect
{
    public readonly RefRO<Target> Target;
    [ReadOnly] public readonly ComponentLookup<Health> HealthLookup;
    public float TargetHealth =>
        HealthLookup.TryGetComponent(Target.ValueRO.Entity, out var h) ? h.Value : 0f;
}

// AFTER — declare the lookup on the system / job directly
public partial struct AISystem : ISystem
{
    private ComponentLookup<Health> _healthLookup;

    public void OnCreate(ref SystemState state)
    {
        _healthLookup = state.GetComponentLookup<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthLookup.Update(ref state);

        foreach (var (target, e) in SystemAPI
                     .Query<RefRO<Target>>()
                     .WithEntityAccess())
        {
            float hp = _healthLookup.TryGetComponent(target.ValueRO.Entity, out var h)
                ? h.Value : 0f;
            // ...
        }
    }
}

이 패턴은 조회가 Entity 간에 공유되므로 Aspect 버전보다 가볍다.

6. Aspect가 복잡한 경우

Aspect에 필드가 많고, 상태를 캡슐화하거나, 비트리비얼한 상속에 참여했다면 — 단순 포팅이 아닌 진정한 리팩터링을 고려해야 한다는 신호다. 종종 코드가 하나의 god-aspect를 가진 System보다 Component를 통해 데이터를 전달하는 두 System으로 더 잘 읽힌다.

7. 문제 해결

증상 원인 / 해결책
컴파일 경고 IAspect is obsolete 6.5에서 예상됨. 경고를 지우기 위해 위와 같이 포팅; 향후 주요 버전에서 제거될 때까지 코드는 여전히 실행됨.
type or namespace IAspect not found IAspect가 제거된 버전에 있음. 포팅을 완료 — 폴백이 없다.
쿼리 튜플에 타입이 너무 많아 컴파일되지 않음 SystemAPI.Query는 큰 튜플을 지원; 매우 넓은 쿼리를 두 쿼리로 나누거나 IJobChunk로 이동하는 것이 보통 더 잘 읽힌다.
Job이 더 이상 Burst 컴파일되지 않음 Aspect 메서드가 Burst가 좋아하지 않는 타입을 캡처했을 수 있음. Execute 매개변수를 직접 다시 확인.
크로스 Entity Aspect를 단순하게 포팅하면 느려짐 사용 전 OnUpdate에서 ComponentLookup<T>.Update(ref state)를 호출해야 함; 잊으면 조회가 무효화됨.
DynamicBuffer Aspect가 포팅되지 않음 DynamicBuffer<T>는 SystemAPI.Query 타입 및 IJobEntity.Execute 매개변수로 직접 유효함 — 래퍼 불필요.
Aspect 타입에 대해 assert하는 테스트가 실패 새 서명에 대해 해당 테스트를 삭제하거나 재작성; 유지할 것이 없다.

Appendix A. Entities 1.4 → 6.5 주요 변경 사항

ECS 프로젝트를 1.4 라인에서 6.5 Core Package 라인으로 이동할 때 중요한 com.unity.entities 패키지 변경 로그 항목의 범위 요약. 이 페이지는 의도적으로 Entities 패키지 변경 사항만 다룬다.

1. 버전 트랙

Entities 6.x는 Unity Editor 6000.x에 맞춰 제공되며 Core Package로 배포된다. 이전 Entities 1.x 버전은 Package Manager 패키지였다.

트랙 배포 예시
Entities 1.x Package Manager (com.unity.entities) 공식 6.5 변경 로그 히스토리의 1.4.2.
Entities 6.x Editor에 내장된 Core Package 6.4.0 / 6.5.0.

이 구분은 패키지 배포와 버전 관리에 관한 것이다. 관련 없는 UnityEngine 또는 Editor 전반의 마이그레이션을 DOTS 프로그래밍 문서로 가져올 권한이 아니다.

2. Entities 6.5.0

Entities 6.5.0의 공식 패키지 변경 로그 항목은:

"Placeholder for 6.5.0"

6.5.0을 Unity 6000.5 정렬 패키지 버전으로 처리한다. 패키지 변경 로그는 이 릴리스의 새로운 Entities API 표면을 나열하지 않는다.

3. Entities 6.4.0

Entities 6.4.0의 공식 패키지 변경 로그 항목은:

"Package is now a core package embedded in Unity."

실질적 효과:

영역 변경
설치 Unity 6000.4+에서 com.unity.entities는 더 이상 manifest.json에 수동으로 추가되지 않는다.
버전 선택 Entities 버전은 Editor 설치에서 온다.

9. 지원 개념

용어 설명
Burst Job과 [BurstCompile]으로 표시된 struct를 위한 Unity의 네이티브 컴파일러. 네이티브 SIMD 코드를 생성한다.
Change version System이 Chunk의 Component 열에 쓸 때 증가하는 32비트 카운터. SetChangedVersionFilter가 변경되지 않은 Chunk를 건너뛸 수 있게 한다.
SoA (Structure of Arrays) 각 필드가 자체 배열이 되는 저장 레이아웃 — 스칼라 반복에 캐시 친화적. Chunk는 SoA를 사용한다.
RefRW<T> / RefRO<T> 읽기-쓰기 / 읽기 전용 의도로 SystemAPI.Query에서 Component에 접근하기 위한 래퍼 타입.
EnabledRefRW<T> / EnabledRefRO<T> 구조적 변경을 트리거하지 않고 enableable Component의 토글 비트를 읽고 쓰기 위한 래퍼.
TypeManager 각 Component에 안정적인 타입 인덱스를 할당하는 런타임 레지스트리. 시작 시 재빌드됨; 대규모 프로젝트는 이를 느끼게 된다.
BlobAssetReference<T> Chunk 외부에 저장된 크고 읽기 전용인 데이터 blob에 대한 포인터. Chunk 용량 부풀림을 방지한다.
