Parte I. Primeros pasos

Capítulo 1. Configuración del entorno — Unity 6000.5 + Entities 6.5.0

1. Introducción

A partir de Unity 6000.4+, Entities es un paquete core — se distribuye dentro del Editor y no es necesario añadirlo a través del Package Manager. Esta guía explica cómo instalar el Editor, crear un proyecto listo para DOTS y verificar que Entities esté activo.

Entorno de referencia del proyecto: Unity 6000.5.0b2 · Entities 6.5.0 · Collections / Mathematics / Entities Graphics (todos Core Packages).

2. Instalar Unity 6000.5+

  1. Abrir Unity Hub. Instalar Unity Hub primero si aún no se dispone de él (https://unity.com/download).
  2. En Hub, ir a Installs → Install Editor.
  3. Seleccionar cualquier versión 6000.5.x (beta de la serie b o posterior). Entities 6.5 solo está disponible en 6000.5+.
  4. Módulos: dejar los valores predeterminados. No se requiere ningún módulo adicional para ECS.

No es necesario seleccionar ningún módulo "DOTS" — Entities está integrado en el Editor a partir de 6000.4.

3. Crear un nuevo proyecto

Desde Hub → Projects → New project:

Campo Valor
Editor Version 6000.5.x
Template Universal 3D (o 3D (Built-in), HDRP — cualquiera sirve)
Project Name cualquier nombre

Hacer clic en Create project. El Editor se abre con una escena vacía.

La antigua "plantilla DOTS" dedicada ya no es necesaria en 6000.4+. Cualquier plantilla puede albergar un flujo de trabajo con Entities.

4. Verificar que los paquetes core están activos

Abrir Window → Package Manager y cambiar el alcance a In Project. Deben aparecer paquetes marcados como "Built-in" para:

  • Entities
  • Collections
  • Mathematics
  • Entities Graphics

Si se prefiere verificar mediante archivos, revisar Packages/manifest.json: no debe haber ninguna entrada com.unity.entities porque se distribuye con el Editor.

A continuación, abrir las ventanas de ECS:

  • Window → Entities → Hierarchy — muestra las entidades en tiempo de ejecución.
  • Window → Entities → Systems — muestra los sistemas registrados.
  • Window → Entities → Archetypes — muestra la disposición de chunks.

Si los tres menús existen, Entities está correctamente configurado.

5. Prueba de humo — crear una SubScene

  1. En la Hierarchy, clic derecho → New Sub Scene → Empty Scene…
  2. Nombrarla MainSubScene.unity y guardar.
  3. Dentro de la nueva SubScene, añadir cualquier primitiva (3D Object → Cube).
  4. Guardar la escena. El Cube se hornea como entidad en cuanto la SubScene se cierra (ícono amarillo) o en el momento de la compilación.
  5. Pulsar Play. Abrir Window → Entities → Hierarchy — debe aparecer una entidad Cube. La prueba de humo ha pasado.

Una SubScene es el punto de entrada para authoring: sus GameObjects se convierten en Entities al guardar y durante el build. El resto del tutorial en 03_Hello DOTS — First Entity.md se basa en esto.

6. Solución de problemas

Síntoma Causa / Solución
El menú Window → Entities no aparece El Editor es anterior a 6000.4 — Entities aún no es un paquete core. Actualizar el Editor.
Baker<T> / IComponentData no se encuentra El archivo de definición de ensamblado excluye Unity.Entities. Añadirlo como referencia, o eliminar el asmdef para una prueba rápida.
La SubScene permanece como GameObject y nunca se hornea La SubScene debe estar cerrada (ícono amarillo) para que aparezcan las entidades horneadas. Abrir/cerrar mediante la casilla de verificación junto a su nombre en la Hierarchy.
La ventana Entities Hierarchy no muestra nada en modo Play Verificar que la SubScene está incluida en la escena actual y que el menú desplegable "World" de Window → Entities → Hierarchy está configurado en DefaultGameObjectInjectionWorld.
Error de compilación: com.unity.entities not found El archivo manifest.json todavía contiene una entrada residual com.unity.entities de un proyecto 1.x — eliminarla. Ver Migration/02_Package Manager → Core Package.md.

Capítulo 2. Los paquetes core explicados

1. Introducción

Con el lanzamiento del paquete Entities 6.4.0 (incluido con Unity Editor 6000.4), los principales paquetes DOTS pasaron del Package Manager al propio Editor como paquetes core. La redacción exacta del registro de cambios del paquete es:

"Package is now a core package embedded in Unity."

Esto afectó a com.unity.entities, com.unity.collections, com.unity.mathematics, com.unity.entities.graphics y (en su correspondiente versión 6.5) a Netcode for Entities. Cambia la forma de instalarlos, versionarlos y gestionarlos, pero es un cambio de distribución, no de API.

Esta página explica:

  • Qué paquetes son ahora paquetes core (y cuáles no).
  • De dónde provienen sus números de versión.
  • Qué cambia en manifest.json y packages-lock.json.
  • Por qué se produjo el cambio.

2. ¿Qué es un paquete core?

Un paquete core es un paquete que se distribuye dentro del Editor. No se añade a través del Package Manager, no es posible cambiar su versión de forma independiente al Editor, y su cadencia de versiones es la del Editor.

Característica Paquete de Package Manager Paquete core
Paso de instalación Añadir mediante UPM Ninguno — presente con el Editor
Versión elegida por El usuario, en manifest.json Unity, por versión del Editor
Cadencia de versiones Independiente de UPM Versiones del Editor
Aparece en manifest.json Sí No
Aparece en packages-lock.json Sí Sí, marcado como builtin
Se muestra en Package Manager Bajo "In Project" Bajo "In Project" etiquetado como Built-in

3. El mapa de paquetes DOTS en Unity 6000.5+

Nota: Job System y Burst nunca fueron paquetes de UPM — forman parte del motor. El cambio en 6000.4 se refiere específicamente a los paquetes DOTS que se apoyan en ellos.

4. Números de versión tras la transición

Las versiones de los paquetes core se alinean con el Editor:

Editor Entities core
Unity 6000.4 Entities 6.4
Unity 6000.5 Entities 6.5

La línea 1.x antigua sigue existiendo para proyectos que aún no pueden migrar a 6000.4+. Las dos líneas son independientes; se actualiza a la 6.x actualizando el Editor.

Ver Changelog/Entities 1.4 → 6.5 Key Changes.md para las notas de transición de Entities y Changelog/Netcode for Entities 1.4 → 6.5 Key Changes.md para los cambios específicos de Netcode.

5. Efecto en los archivos del proyecto

manifest.json

No se necesitan entradas para com.unity.entities, com.unity.collections, com.unity.mathematics ni com.unity.entities.graphics. Si se actualiza un proyecto 1.x, estas entradas se eliminan — ver Migration/02_Package Manager → Core Package.md.

packages-lock.json

Los paquetes core aparecen con "source": "builtin" en lugar de una URL de versión.

Definiciones de ensamblado (.asmdef)

Los archivos .asmdef siguen referenciando los mismos nombres de ensamblado (por ejemplo, Unity.Entities, Unity.Collections). Aquí no cambia nada — solo cambió la capa de distribución del paquete.

6. ¿Por qué el cambio?

El objetivo declarado es permitir que el equipo de ECS publique características de forma más rápida e incremental vinculando las actualizaciones de DOTS a la versión del Editor en lugar de a un calendario de paquetes independiente. También permite que el motor dependa internamente de ECS, lo cual era difícil mientras los paquetes eran opcionales.

Capítulo 3. Hola DOTS — Primera Entity

1. Qué se va a construir

Un cubo que rota sobre su eje Y, impulsado por un sistema ECS. En una sola sesión se tocarán todos los conceptos clave: SubScene, authoring → baker, IComponentData, ISystem y las ventanas Entities Hierarchy / Systems. Al final se sustituirá el sistema del hilo principal por un IJobEntity para ver cómo se introduce el paralelismo.

Prerrequisito: [01_Environment Setup.md](01_Environment Setup (Unity 6000.5 + Entities 6.5.0).md) — un proyecto Unity 6000.5 con una SubScene confirmada y el menú Entities.

2. Crear la SubScene

  1. Abrir la escena vacía.
  2. En la Hierarchy, clic derecho → New Sub Scene → Empty Scene…
  3. Guardarla como HelloDots.unity dentro de Assets/Scenes/.
  4. Dentro de la nueva SubScene, clic derecho → 3D Object → Cube. Renombrarlo como RotatingCube.
  5. Guardar la escena (Ctrl/⌘ + S). Cerrar la SubScene haciendo clic en la casilla de verificación junto a su nombre en la Hierarchy — el ícono se vuelve amarillo, lo que indica que el contenido se está horneando como entidades.

A partir de este momento, las modificaciones en los componentes de RotatingCube desencadenan un rehorneado automático al volver a abrir/cerrar la SubScene.

3. Definir el componente — RotationSpeed

Crear Assets/Scripts/RotationSpeed.cs:

using Unity.Entities;

public struct RotationSpeed : IComponentData
{
    public float RadiansPerSecond;
}

Notas:

  • IComponentData en un struct no administrado es la elección predeterminada — vive directamente en los chunks y es compatible con Burst.
  • Los campos deben ser blittables (primitivos, tipos de Unity.Mathematics, otros structs no administrados).

4. Definir el authoring + baker

Crear Assets/Scripts/RotationSpeedAuthoring.cs:

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class RotationSpeedAuthoring : MonoBehaviour
{
    public float DegreesPerSecond = 90f;

    class Baker : Baker<RotationSpeedAuthoring>
    {
        public override void Bake(RotationSpeedAuthoring authoring)
        {
            // Dynamic — this entity will have its transform written each frame.
            var entity = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(entity, new RotationSpeed
            {
                RadiansPerSecond = math.radians(authoring.DegreesPerSecond)
            });
        }
    }
}

Adjuntar el authoring al cubo

  1. Seleccionar RotatingCube en la SubScene.
  2. Inspector → Add Component → Rotation Speed Authoring.
  3. Dejar DegreesPerSecond = 90.

En cuanto la SubScene está cerrada (ícono amarillo), el baker se ejecuta y el cubo se convierte en una entidad con un componente RotationSpeed.

Verificar el horneado

  • Window → Entities → Hierarchy — debe aparecer RotatingCube en la lista.
  • Hacer clic en él. El Inspector de la derecha muestra sus componentes — buscar RotationSpeed { RadiansPerSecond: ~1.5708 } (π/2).

5. Escribir el sistema — RotationSystem

Crear Assets/Scripts/RotationSystem.cs:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<RotationSpeed>>())
        {
            transform.ValueRW = transform.ValueRO.RotateY(speed.ValueRO.RadiansPerSecond * dt);
        }
    }
}

Notas:

  • partial struct + ISystem es la forma de sistema no administrado. Puede compilarse con Burst y funciona sin overhead de GC.
  • SystemAPI.Query<RefRW<A>, RefRO<B>>() es la API de iteración moderna — reemplaza al obsoleto Entities.ForEach.
  • LocalTransform.RotateY(radians) devuelve una copia rotada; se escribe de vuelta con transform.ValueRW = ….

Pulsar Play

El cubo rota. En Window → Entities → Systems se puede ver RotationSystem listado bajo el grupo de simulación con tiempos por fotograma.

6. Inspeccionar lo que se ha construido

Ventanas útiles una vez en modo Play:

Ventana Qué muestra

Systems Todos los sistemas en ejecución, agrupados por su SystemGroup. Aquí se ve el tiempo por sistema.

7. Cambiar a IJobEntity (paralelismo)

Sustituir RotationSystem.cs por una versión basada en jobs para ver cómo se introduce el paralelismo:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotateJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in RotationSpeed speed)
    {
        transform = transform.RotateY(speed.RadiansPerSecond * DeltaTime);
    }
}

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new RotateJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Notas:

  • IJobEntity.Execute(ref A, in B) se genera mediante código fuente en un job a nivel de chunk. Los parámetros son los componentes que se manipulan.
  • .ScheduleParallel() distribuye los chunks entre los hilos de trabajo. Con un solo cubo no se percibirá mejora, pero el diseño del código escala a miles de entidades sin cambios.
  • La dependencia del sistema se gestiona automáticamente — state.Dependency se combina de forma automática al llamar a ScheduleParallel() desde ISystem.OnUpdate.

Parte II. Flujos de trabajo DOTS

Capítulo 4. Patrón Baker & SubScene

1. Introducción

Las entidades no pueden crearse directamente en la Hierarchy. En su lugar, se crean GameObjects dentro de una SubScene y un Baker los convierte en entidades cuando la SubScene se cierra o se compila.

Figure: Baker and SubScene Pipeline.

2. Ciclo de vida de la SubScene

Una SubScene tiene tres estados visibles:

Ícono en Hierarchy Estado Qué significa
⬜ (casilla blanca) Abierta Sus GameObjects son editables; siguen siendo GameObjects, sin horneado.
🟨 (casilla amarilla) Cerrada Los Bakers se han ejecutado; el contenido existe solo como entidades.
⏳ Horneando Horneado incremental en progreso (breve).

Crear una SubScene

  • Hierarchy → clic derecho → New Sub Scene → Empty Scene… — guarda un asset .unity y añade un componente SubScene al GameObject padre.
  • Abrir/cerrar mediante la casilla de verificación junto al nombre de la SubScene. Al cerrarla se desencadena un horneado.

Horneado incremental

Cuando la SubScene está abierta y se edita un GameObject, solo se vuelven a ejecutar los Bakers que tocaron el authoring modificado. Por esta razón, conviene evitar trabajo costoso dentro de Bake().

3. API de Baker<T>

Un Baker mapea un MonoBehaviour de authoring a una o más entidades + componentes.

public class EnemyAuthoring : MonoBehaviour
{
    public float MaxHealth = 100f;
    public GameObject ProjectilePrefab;

    class Baker : Baker<EnemyAuthoring>
    {
        public override void Bake(EnemyAuthoring authoring)
        {
            // Primary entity for this GameObject.
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Health { Value = authoring.MaxHealth });

            // Bake a GameObject reference into an Entity reference.
            AddComponent(self, new ProjectilePrefabRef
            {
                Value = GetEntity(authoring.ProjectilePrefab, TransformUsageFlags.Dynamic)
            });
        }
    }
}

APIs principales:

API Propósito
GetEntity(TransformUsageFlags) Entity para el GameObject de authoring actual.
GetEntity(Object, TransformUsageFlags) Entity correspondiente a otro GameObject/prefab (registra una entidad prefab).
CreateAdditionalEntity(TransformUsageFlags) Una segunda entidad vinculada al mismo authoring (p. ej., subpartes).
AddComponent<T>(entity, value) Adjuntar datos de componente.
AddBuffer<T>(entity) / SetBuffer<T> Adjuntar un DynamicBuffer<T>.
AddSharedComponent<T> Adjuntar un ISharedComponentData.
DependsOn(asset) Declarar una dependencia de asset para que el Baker se vuelva a ejecutar cuando el asset cambie.

4. TransformUsageFlags

El horneado solo añade LocalTransform / LocalToWorld / Parent cuando se solicitan mediante TransformUsageFlags. Los componentes de transformación no utilizados aumentan el tamaño de los chunks sin razón, por lo que conviene elegir el flag más restrictivo que siga funcionando.

Flag Añade Uso típico
None Nada Entidades de datos puros (configuración, singletons, solo lógica).
ManualOverride Nada; las transformaciones se gestionan manualmente Layouts de transformación personalizados.
Renderable LocalToWorld Malla estática que nunca se mueve.
Dynamic LocalTransform, LocalToWorld (+ Parent si tiene padre) Todo lo que se mueve o es escrito por un system.
WorldSpace LocalTransform, LocalToWorld; impide el parentado Entidades que ignoran la jerarquía.
NonUniformScale Añade PostTransformMatrix Escala no uniforme (poco común en DOTS).

Dynamic es el valor predeterminado seguro para todo lo que un system vaya a mover o rotar.

5. Baking systems (avanzado)

Un Baker produce componentes a partir de un único authoring. Un Baking System se ejecuta después de todos los Bakers y ve el mundo horneado completo — útil para el postprocesado entre entidades (resolución de referencias, cálculo de datos derivados).

[WorldSystemFilter(WorldSystemFilterFlags.BakingSystem)]
public partial struct EnemyWaveLinkSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Runs during bake only. Query entities tagged earlier by Bakers
        // and wire them together.
    }
}

Se recomienda usar esto cuando un Baker no dispone de toda la información necesaria (p. ej., conteos, referencias cruzadas). Debe ser determinista: el horneado debe producir el mismo resultado en cada ejecución.

6. Ejemplo — authoring de un spawner de proyectiles

Authoring:

using Unity.Entities;
using UnityEngine;

public class ProjectileSpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float Interval = 0.5f;

    class Baker : Baker<ProjectileSpawnerAuthoring>
    {
        public override void Bake(ProjectileSpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new ProjectileSpawner
            {
                Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                Interval = authoring.Interval,
                TimeLeft = authoring.Interval
            });
        }
    }
}

Component:

{
    public Entity Prefab;
    public float Interval;

}

Un system en tiempo de ejecución puede instanciar Prefab con un temporizador. Véase 02_Spawner Example.md.

7. Solución de problemas

Síntoma Causa / Solución

GetEntity(prefab, ...) devuelve Entity.Null El GameObject referenciado no es un prefab asset ni un objeto de escena accesible desde el Baker. Asegurarse de que sea un prefab o esté en el mismo SubScene.

Los cambios en un campo de authoring no vuelven a hornear El Baker no lee ese campo o no llamó a DependsOn para un asset externo.
El horneado es lento Trabajo pesado dentro de Bake(). Mueva la lógica entre entidades a un Baking System o a un inicializador de ejecución única en runtime.

Chapter 5. Spawner Example

1. Descripción general

Un spawner es una entidad única que crea más entidades — al inicio, con un temporizador o ante un evento. Esta sección lo construye de principio a fin, cubriendo:

  • Authoring del spawner en un SubScene.
  • Almacenar una referencia a una entidad prefab (horneada desde un prefab de GameObject).
  • Instanciarla en tiempo de ejecución mediante un EntityCommandBuffer para que los cambios estructurales ocurran de forma segura.

Leer 01_Baker Pattern & SubScene.md antes facilitará la comprensión.

2. Modelo de datos

using Unity.Entities;
using Unity.Mathematics;

public struct Spawner : IComponentData
{
    public Entity  Prefab;       // baked prefab entity
    public float3  SpawnPoint;
    public float   Interval;
    public float   TimeLeft;
    public int     RemainingCount;  // stop when 0
}

Dos decisiones de diseño a destacar:

  • Prefab es una Entity, no un GameObject. Los Bakers convierten el prefab de GameObject en una entidad prefab; el tiempo de ejecución solo ve entidades.
  • TimeLeft y RemainingCount viven en la entidad spawner para que el system permanezca sin estado.

3. Authoring + Baker

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float      Interval = 0.5f;
    public int        Count    = 100;

    class Baker : Baker<SpawnerAuthoring>
    {
        public override void Bake(SpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Spawner
            {
                Prefab        = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                SpawnPoint    = authoring.transform.position,
                Interval      = authoring.Interval,
                TimeLeft      = 0f,            // fires immediately on first update
                RemainingCount = authoring.Count
            });
        }
    }
}

Colocar SpawnerAuthoring en un GameObject vacío dentro del SubScene, asignar un Prefab y establecer un conteo. Cerrar el SubScene — el spawner ya es una entidad.

4. System en tiempo de ejecución

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnerSystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<Spawner>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Use the EndSimulation ECB so structural changes flush at a safe point.
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged);

        foreach (var (spawnerRW, spawnerEntity) in
                 SystemAPI.Query<RefRW<Spawner>>().WithEntityAccess())
        {
            ref var spawner = ref spawnerRW.ValueRW;

            if (spawner.RemainingCount <= 0)
            {
                ecb.DestroyEntity(spawnerEntity);   // done — remove the spawner
                continue;
            }

            spawner.TimeLeft -= dt;
            if (spawner.TimeLeft > 0f) continue;

            spawner.TimeLeft = spawner.Interval;
            spawner.RemainingCount--;

            var instance = ecb.Instantiate(spawner.Prefab);
            ecb.SetComponent(instance, LocalTransform.FromPosition(spawner.SpawnPoint));
        }
    }
}

Lo importante:

  • RequireForUpdate<Spawner>() omite el system por completo cuando no existe ningún spawner.
  • El singleton EndSimulationEntityCommandBufferSystem proporciona un ECB compartido que se vacía al final del fotograma — más seguro que mutar mediante EntityManager dentro del bucle.
  • ecb.Instantiate(prefab) crea una entidad diferida; ecb.SetComponent(instance, …) programa escrituras de componentes en ella. El handle instance solo es válido dentro de la reproducción de este ECB. Véase 14_EntityCommandBuffer · Deferred Entity.md.

5. Escalado — spawner paralelo

Si es necesario spawnear miles por fotograma desde muchos spawners, se puede convertir a un IJobEntity con un writer paralelo de ECB:

[BurstCompile]
public partial struct SpawnJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, entity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

Y en el system:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

new SpawnJob { DeltaTime = SystemAPI.Time.DeltaTime, ECB = ecb }
    .ScheduleParallel();

La clave de ordenación chunkIndex es lo que mantiene la reproducción paralela determinista. Véase 15_ParallelWriter · Deterministic Playback.md.

6. Solución de problemas

Síntoma Causa / Solución
No aparece nada La entidad Spawner no existe — abrir el SubScene, verificar que SpawnerAuthoring esté adjunto y que el SubScene esté cerrado (icono amarillo).
El prefab aparece en el origen del mundo El ECB nunca estableció LocalTransform. Añadir ecb.SetComponent(instance, LocalTransform.FromPosition(...)) o asegurarse de que el prefab ya tenga una transformación con la posición deseada.
El prefab es visible pero no se mueve El prefab se horneó con TransformUsageFlags.Renderable en lugar de Dynamic.
SystemAPI.GetSingleton<...ECBS.Singleton>() lanza excepción El ECB system no está registrado — ocurre en worlds muy reducidos. Usar state.EntityManager + ECB manual como alternativa, o registrar el ECB system en el bootstrap.
El spawner nunca se detiene RemainingCount se decrementó en una copia en caché de Spawner en lugar de en spawnerRW.ValueRW. Escribir siempre mediante RefRW.ValueRW.

Chapter 6. ECS Core Concepts

1. Los cinco sustantivos

El ECS de Entities se reduce a cinco sustantivos:

Figure: ECS Core Nouns.

Interiorizar este diagrama es gran parte de lo necesario para leer código DOTS existente.

2. Entity

Una entity es solo un handle de 64 bits: struct Entity { int Index; int Version; }. No tiene datos ni métodos más allá de la identidad — todos los datos viven en los componentes.

Entity e = state.EntityManager.CreateEntity();   // fresh, no components
state.EntityManager.AddComponent<Health>(e);     // now it has data
state.EntityManager.DestroyEntity(e);            // handle becomes stale

El campo Version invalida referencias antiguas: si se destruye y se vuelve a crear en el mismo Index, el valor antiguo de Entity ya no coincide.

3. Component

Un component es un contenedor de datos plano. No lleva comportamiento. El código vive en los systems.

public struct Health : IComponentData
{
    public float Value;
    public float Max;
}

Existen varias variantes de componente (no gestionados, gestionados, buffer, compartidos, cleanup, tag, chunk, habilitables). Véase 05_Component Types.md.

4. Archetype

Un archetype es un conjunto único de tipos de componente que comparten algunas entidades.

  • Una entidad con { LocalTransform, Health, Enemy } pertenece al archetype A.
  • Una entidad con { LocalTransform, Health, Enemy, Burning } pertenece al archetype B — uno distinto, porque añadir Burning cambió el conjunto de componentes.

Moverse entre archetypes (añadir o quitar un componente) es un cambio estructural y mueve físicamente la entidad a un nuevo chunk. Véase 13_Structural Change & Safety.md.

5. Chunk

Las entidades del mismo archetype se empaquetan en chunks de 16 KB. Dentro de un chunk, cada tipo de componente forma un array contiguo — excelente para la localidad de caché.

Hecho Implicación
Los chunks son de 16 KB Cada componente ocupa una porción; cuantos más componentes (o más grandes) tenga el archetype, menos entidades caben por chunk.
Los datos se disponen por columnas (SoA) por componente IJobChunk puede iterar un array de componentes a la vez y vectorizarse con Burst.
Añadir o quitar un componente mueve la entidad Cambio estructural. Los iteradores de query asumen que el archetype es estable durante la iteración.
Los chunks registran las versiones de cambio de componentes Los systems pueden omitir chunks que no cambiaron desde la última ejecución (EntityQuery.SetChangedVersionFilter).

Se puede inspeccionar el layout de chunks en vivo en Window → Entities → Archetypes.

6. System

Un system es una pieza de lógica que se ejecuta en cada fotograma, típicamente sobre una query de entidades.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var health in SystemAPI.Query<RefRW<Health>>())
        {
            health.ValueRW.Value = math.min(health.ValueRW.Value + dt, health.ValueRW.Max);
        }
    }
}

Los systems se registran automáticamente y viven dentro de un SystemGroup. Véase 08_System — ISystem vs SystemBase.md y 09_System Group & Update Order.md.

7. World

Un World es un contenedor que posee:

  • Un EntityManager (crea, destruye, consulta entidades).
  • Un conjunto de systems y sus SystemGroups.
  • Los propios chunks (a través del EntityManager).

El world de tiempo de ejecución predeterminado es DefaultGameObjectInjectionWorld. La mayoría de los juegos tienen un solo world en producción; las configuraciones multi-world aparecen en Netcode (servidor / cliente) o en herramientas del Editor. Véase 16_Netcode Client-Server World & Bootstrap.md para la división de worlds en Netcode.

World world = World.DefaultGameObjectInjectionWorld;
EntityManager em = world.EntityManager;

Entity e = em.CreateEntity(typeof(Health));
em.SetComponentData(e, new Health { Value = 100, Max = 100 });

8. EntityManager

El EntityManager es la API del hilo principal para cambios estructurales: crear, destruir, añadir o quitar componentes, instanciar entidades prefab, copiar worlds, etc. Los cambios estructurales mediante EntityManager son inmediatos y pueden invalidar cualquier iterador de query en vuelo o referencia de job.

Para cambios estructurales diferidos desde jobs o durante la iteración, se debe usar un EntityCommandBuffer — véase 14_EntityCommandBuffer · Deferred Entity.md.

9. Integrando todo

World
└── EntityManager
    └── Archetype { LocalTransform, Health, Enemy }
        └── Chunk  (up to ~N entities)
            └── Entity { Index, Version }
                ├── LocalTransform
                ├── Health
                └── Enemy (tag)

Un system que consulta { LocalTransform, Health } toca cada chunk cuyo archetype contiene ambos. Eso incluye el archetype anterior — más cualquier otro archetype que también tenga Enemy más componentes adicionales.

10. Solución de problemas

Síntoma Causa / Solución
Las búsquedas de Entity devuelven datos obsoletos La entidad fue destruida y recreada — el Version ya no coincide. Volver a resolver la entidad.
La query no encuentra nada aunque la entidad tiene el componente El componente se añadió en un world distinto, o el componente es habilitable y actualmente está desactivado.
El recuento de Archetype explota Demasiadas combinaciones únicas de componentes (a menudo por tags por entidad con alta cardinalidad). Consolidar usando componentes compartidos o flags habilitables.
El chunk contiene solo 1-2 entidades El archetype es demasiado grande — eliminar componentes que la entidad no necesita, o dividir en múltiples entidades o archetypes.
InvalidOperationException: … invalidated … durante la iteración Ocurrió un cambio estructural en medio de la query. Usar un ECB, o agrupar los cambios después del bucle.

Chapter 7. Entity References — Entity · EntityPrefabReference · UnityObjectRef

1. Descripción general

El código DOTS usa un pequeño conjunto de tipos de referencia. Es fácil confundirlos porque todos apuntan a "algo", pero pertenecen a etapas distintas del pipeline de authoring a runtime.

Tipo Espacio de nombres Qué referencia Uso típico
Entity Unity.Entities Una entidad dentro de un World ECS. Relaciones en tiempo de ejecución y campos de componente.
EntityPrefabReference Unity.Entities.Serialization Un prefab de entidad horneado que puede cargarse más tarde. Flujos de trabajo de streaming y contenido.
UnityObjectRef<T> Unity.Entities Un asset o objeto UnityEngine.Object a través de un wrapper no gestionado. Referenciar meshes, materiales, texturas, audio clips y assets de authoring desde datos ECS.

Esta sección trata únicamente los tipos de referencia de DOTS / Entities. Las API de identificador de UnityEngine.Object para todo el motor quedan fuera del alcance de este manual.

2. Entity

Entity es el handle ECS en tiempo de ejecución. Es un valor no gestionado que identifica una fila de datos de componentes dentro de un World.

using Unity.Entities;

public struct Target : IComponentData
{
    public Entity Value;
}

Reglas importantes:

  • Un valor de Entity solo tiene significado dentro del World que lo creó.
  • Destruir una entidad invalida el handle antiguo.
  • Los índices de entidad pueden reutilizarse; usar EntityManager.Exists(entity) antes de desreferenciar un handle almacenado que no se acaba de consultar.
  • No guardar valores de Entity en disco ni enviarlos entre worlds independientes como IDs estables.
if (state.EntityManager.Exists(target.Value))
{
    var transform = state.EntityManager.GetComponentData<LocalTransform>(target.Value);
}

3. Referencias de entidad en el baking

Cuando un Baker convierte objetos de authoring en datos ECS, se usa GetEntity para convertir un objeto de authoring o prefab referenciado en una referencia de Entity.

using Unity.Entities;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
}

public struct Spawner : IComponentData
{
    public Entity Prefab;
}

public class SpawnerBaker : Baker<SpawnerAuthoring>
{
    public override void Bake(SpawnerAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new Spawner
        {
            Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic)
        });
    }
}

El resultado es una referencia ECS en tiempo de ejecución. No es una referencia gestionada de GameObject.

4. EntityPrefabReference

EntityPrefabReference es para contenido de prefabs que debe cargarse mediante el pipeline de contenido de Entities en lugar de estar incrustado directamente como una referencia de Entity en la escena actual.

using Unity.Entities;
using Unity.Entities.Serialization;
using UnityEngine;

public class ProjectileLibraryAuthoring : MonoBehaviour
{
    public GameObject ProjectilePrefab;
}

public struct ProjectileLibrary : IComponentData
{
    public EntityPrefabReference ProjectilePrefab;
}

public class ProjectileLibraryBaker : Baker<ProjectileLibraryAuthoring>
{
    public override void Bake(ProjectileLibraryAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new ProjectileLibrary
        {
            ProjectilePrefab = new EntityPrefabReference(authoring.ProjectilePrefab)
        });
    }
}

Se debe usar EntityPrefabReference cuando el prefab pertenece a un flujo de trabajo de gestión de contenido o streaming. Usar una referencia de prefab Entity simple cuando el prefab está horneado en el mismo SubScene y está disponible con la escena.

5. UnityObjectRef<T>

UnityObjectRef<T> permite que los datos ECS no gestionados referencien un UnityEngine.Object sin convertir el componente en uno gestionado.

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

Hornearlo desde datos de authoring:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

Desreferenciar .Value solo en el hilo principal o en código de presentación gestionado:

Mesh mesh = visual.Mesh.Value;
Material material = visual.Material.Value;

Reglas:

  • El wrapper almacenado no está gestionado y puede vivir dentro de IComponentData.
  • El objeto de destino sigue siendo un objeto Unity gestionado; .Value no es una operación de Burst job.
  • El objeto referenciado debe seguir cargado cuando se desreferencia.

6. Elegir la referencia correcta

Necesidad Usar
Apuntar de una entidad a otra en el mismo world Entity
Almacenar un prefab horneado en el mismo SubScene Referencia de prefab Entity desde GetEntity(authoring.Prefab, ...)
Almacenar un prefab para streaming de contenido / carga diferida EntityPrefabReference
Almacenar una referencia a mesh, material, textura, audio clip o ScriptableObject en datos ECS no gestionados UnityObjectRef<T>
Persistir una entidad entre sesiones Un ID estable propio del juego, no un Entity en bruto

7. Solución de problemas

Síntoma Causa / Solución
La Entity almacenada ya no existe La entidad fue destruida o pertenece a otro world. Volverla a resolver o comprobar con EntityManager.Exists.
La referencia de prefab es Entity.Null tras el baking El Baker no llamó a GetEntity en el prefab, o el prefab no fue incluido en un workflow de baked scene o contenido.
UnityObjectRef<T>.Value devuelve null El objeto Unity referenciado no está cargado o fue destruido. Proteger la desreferencia y volver a cargar o resolver el asset.
Un Burst job no puede usar una referencia de mesh o material El almacenamiento de UnityObjectRef<T> no está gestionado, pero .Value toca objetos Unity gestionados. Mover la desreferencia al código de presentación del hilo principal.
El archivo de guardado almacena valores de Entity en bruto Los handles de entidad son locales al world y no son estables. Almacenar en su lugar un ID propio del juego o una clave de contenido.

Chapter 8. Component Types

1. Descripción general

Un "componente" en ECS es cualquier contenedor de datos adjunto a una entidad. Entities expone varias variantes con diferentes perfiles de almacenamiento, vida útil y coste. Elegir la correcta es la mitad del trabajo al diseñar un system ECS.

Mapa rápido:

Interfaz / Forma Almacenamiento ¿Cambio estructural al mutar? Uso típico
IComponentData (struct no gestionado) En chunk, SoA No (al cambiar el valor) Componente predeterminado
IComponentData (class gestionada) Tabla lateral No Tipos de referencia (meshes, refs de GameObject)
IBufferElementData En chunk + desbordamiento al heap No Arrays dinámicos por entidad
ISharedComponentData Por chunk Sí (mueve la entidad a un nuevo chunk) Agrupar entidades que comparten configuración
ICleanupComponentData En chunk Normal Protocolo de cleanup al destruir entidades
ICleanupSharedComponentData Por chunk Normal Protocolo de cleanup de recursos compartidos
Tag (empty IComponentData) Tamaño cero en chunk No Marcador de filtro, sin datos
Chunk component Por chunk Sí Metadatos calculados una vez por chunk
Enableable (IEnableableComponent) En chunk + máscara de bits No Activar o desactivar comportamiento por entidad

El resto de esta sección amplía cada fila.

2. IComponentData no gestionado

El predeterminado. Vive directamente en el chunk como datos SoA compactados. Compatible con Burst.

public struct Velocity : IComponentData
{
    public float3 Value;
}

Reglas:

  • Debe ser un struct — sin campos de referencia (sin class, sin string, sin List<T>).
  • Tipos de campo permitidos: primitivos, Unity.Mathematics, otros structs no gestionados, Entity, FixedString*, BlobAssetReference<T>, UnityObjectRef<T>.
  • Leer mediante RefRO<T>, escribir mediante RefRW<T> desde SystemAPI.Query.

3. IComponentData gestionado

Vive en una tabla lateral indexada por entidad. Funciona con tipos de referencia pero requiere una asignación GC y rompe la compatibilidad con Burst para ese componente.

public class EnemyAI : IComponentData
{
    public AIModel Brain;      // a managed class
    public List<Entity> Allies;
}

Se debe usar cuando genuinamente se necesita una referencia gestionada (mesh, textura, ScriptableObject de authoring). Preferir UnityObjectRef<T> dentro de un componente no gestionado cuando solo se necesita una referencia débil a un UnityEngine.Object.

4. IBufferElementData — buffers dinámicos

Un DynamicBuffer<T> es un array redimensionable por entidad.

public struct Waypoint : IBufferElementData
{
    public float3 Position;
}

var buffer = state.EntityManager.AddBuffer<Waypoint>(entity);
buffer.Add(new Waypoint { Position = new float3(1, 0, 0) });

Almacenamiento: hasta InternalBufferCapacity elementos viven en el chunk; más allá de eso el buffer se desborda al heap. Ajustar la capacidad con [InternalBufferCapacity(N)].

5. ISharedComponentData — agrupación

Los componentes compartidos llevan un valor que agrupa chunks. Dos entidades con el mismo valor de ISharedComponentData terminan en el mismo chunk; valores distintos producen chunks distintos.

public struct RenderMeshGroup : ISharedComponentData
{
    public int MaterialID;
}

Consecuencias:

  • Cambiar el valor mueve la entidad a otro chunk → cambio estructural, costoso.
  • Ideal para datos que no cambian frecuentemente pero varían por grupo (conjuntos de materiales, IDs de equipo, regiones de bake).
  • Los valores se almacenan en una tabla gestionada si el tipo contiene referencias (ISharedComponentData puede ser gestionado).

6. Componentes cleanup — ICleanupComponentData

Marca una entidad de modo que, cuando se "destruye", permanece viva solo con sus componentes cleanup hasta que el system los procese.

public struct ConnectionCleanup : ICleanupComponentData
{
    public int ConnectionHandle;
}

// DestroyEntity(e) removes all normal components but leaves ConnectionCleanup
// behind so you can close the socket in a dedicated system, then destroy the
// cleanup component explicitly.

El hermano ICleanupSharedComponentData aplica el mismo protocolo a recursos compartidos.

Casos de uso: liberar recursos nativos, desregistrarse de systems externos, destrucción gradual a lo largo de fotogramas.

7. Componentes tag

Un tag es un IComponentData vacío:

public struct Enemy : IComponentData {}

Cero bytes en un chunk, consultable como cualquier otro componente. Ideal para marcadores de "es un tipo de X". Abusar de los tags (cientos de únicos) fragmenta los archetypes — usar componentes habilitables o compartidos para estados de alta cardinalidad.

8. Chunk components

Un chunk component lleva un valor por chunk, no por entidad.

state.EntityManager.AddChunkComponentData<RegionBounds>(query, new RegionBounds { ... });

Uso típico: datos precalculados que se aplican a todas las entidades del chunk (bounds, ID de región, clave de partición). Mutar un chunk component provoca un cambio estructural.

9. Enableable components (IEnableableComponent)

Véase la página dedicada 06_Enableable Component.md para más detalle. Resumen aquí: un componente que puede activarse o desactivarse por entidad sin un cambio estructural, mediante una máscara de bits por chunk.

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

state.EntityManager.SetComponentEnabled<Stunned>(entity, false);  // no structural change

Las queries omiten automáticamente las entidades cuyos enableable components están desactivados — una forma barata y común de excluir temporalmente entidades de un system.

10. Guía de decisión

Quiero… Usar
Almacenar datos numéricos por entidad IComponentData no gestionado
Mantener una referencia gestionada por entidad IComponentData gestionado o UnityObjectRef<T>
Almacenar una lista de longitud variable por entidad IBufferElementData
Agrupar entidades para procesamiento por lotes ISharedComponentData

InvalidOperationException: The type T must be unmanaged Se usó un tipo gestionado donde se requiere uno no gestionado. Cambiar el componente o la API (p. ej., ComponentLookup<T> frente a ManagedAPI.GetComponent<T>).
Cambiar el valor de un componente compartido provoca tartamudeos Cada cambio es un cambio estructural. Agrupar con un ECB o aceptar que este valor se muta raramente.
DynamicBuffer<T> se desborda al heap [InternalBufferCapacity(N)] es demasiado pequeño para el uso previsto. Aumentarlo o mantener los tamaños acotados.
La entidad reaparece tras DestroyEntity Hay un cleanup component todavía adjunto. Eliminarlo explícitamente una vez completado el cleanup.
Demasiados archetypes Explosión de tags. Convertir los "estados" únicos por entidad en enableable components.
El componente tag no filtra una query Los tag components no tienen datos, por lo que no se puede solicitar un RefRW<Tag> / RefRO<Tag> en la tupla de SystemAPI.Query<...>. Aplicarlos mediante .WithAll<Tag>() / .WithNone<Tag>().

Chapter 9. Enableable Component

1. Por qué existen los enableable components

Añadir o quitar un componente es un cambio estructural: la entidad se mueve a un chunk diferente, las queries deben resolverse de nuevo y las referencias pueden invalidarse. Si esto ocurre cada fotograma por entidad (p. ej., activando y desactivando "Stunned"), el coste domina el fotograma.

Un enableable component permite activar y desactivar comportamiento sin un cambio estructural. El componente siempre está presente en el chunk; un bit en una máscara por chunk decide si cada entidad lo "tiene" desde el punto de vista de una query.

2. La interfaz

Implementar tanto IComponentData como IEnableableComponent:

using Unity.Entities;

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

También es posible hacer habilitable un IBufferElementData.

3. Activación y desactivación

Desde un system:

// Turn Stunned off without removing the component
state.EntityManager.SetComponentEnabled<Stunned>(entity, false);

// From an ECB (deferred, safe mid-iteration)
ecb.SetComponentEnabled<Stunned>(entity, true);

// From a job with a ComponentLookup
var stunnedLookup = SystemAPI.GetComponentLookup<Stunned>();
stunnedLookup.SetComponentEnabled(entity, false);

Leer el estado actual:

bool isStunned = state.EntityManager.IsComponentEnabled<Stunned>(entity);

Las tres rutas operan sobre una máscara de bits; ninguna desencadena un cambio estructural.

4. Comportamiento de las queries

SystemAPI.Query<...> y la iteración basada en EntityQuery omiten las entidades cuyos enableable components están actualmente desactivados — por defecto, solo se ven las entidades donde cada componente listado está habilitado.

foreach (var stunned in SystemAPI.Query<RefRW<Stunned>>())
{
    // only fires for entities where Stunned is enabled
}

Para ver también las entidades desactivadas, se puede sobrescribir este comportamiento:

foreach (var (stunnedRW, entity) in SystemAPI
             .Query<RefRW<Stunned>>()
             .WithEntityAccess()
             .WithOptions(EntityQueryOptions.IgnoreComponentEnabledState))
{
    // fires for ALL entities with the component, enabled or not
}

5. Ejemplo — stun que expira

Un único IJobEntity no puede recibir tanto ref Stunned (un RefRW<T>) como EnabledRefRW<Stunned> para el mismo componente — Entities lo prohíbe explícitamente. Dos patrones válidos:

5.1 Leer + escribir el valor, activar el bit habilitado mediante ECB

[BurstCompile]
public partial struct StunTickJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Stunned stunned)
    {
        stunned.Duration -= DeltaTime;
        if (stunned.Duration <= 0f)
            ECB.SetComponentEnabled<Stunned>(chunkIndex, entity, false);
    }
}

La reproducción ocurre en el siguiente límite de ECB (p. ej., EndSimulationECBS) — véase 14_EntityCommandBuffer · Deferred Entity.md. Sin cambio estructural; solo un flip de bit en un punto conocido.

5.2 Activar el bit directamente cuando no se necesita el valor del componente

[BurstCompile]
public partial struct StunDisableJob : IJobEntity
{
    void Execute(EnabledRefRW<Stunned> stunnedEnabled)
    {
        stunnedEnabled.ValueRW = false;   // no structural change
    }
}

EnabledRefRW<T> es la única referencia a Stunned en este job, por lo que compila. Usar esta forma cuando un system es puramente un conmutador.

¿Quiere leer el valor y activar el bit en la misma entidad en un solo paso? Hacerlo desde el hilo principal mediante EntityManager.SetComponentEnabled<T>(entity, false), o dividir en dos jobs encadenados mediante state.Dependency.

6. Cuándo preferir enableable frente a añadir/quitar

Situación Elegir
Conmutación de alta frecuencia (por fotograma, por evento) Enableable
Añade estado que no siempre está presente (modificador por entidad con campos variables) Enableable
Cambio de archetype de baja frecuencia (p. ej., añadir una capacidad permanente) Añadir o quitar el componente normalmente
Filtrado uniforme en muchos systems Enableable (las queries lo recogen automáticamente)

Regla general: si el bit "tiene este componente" cambia varias veces en la vida de una entidad, hacerlo habilitable.

7. Interacción con otras características

Característica Comportamiento
WithChangeFilter<T> Activar el bit también incrementa la versión de cambio del componente.
ICleanupComponentData Un cleanup component no puede ser también habilitable.
Netcode for Entities Los enableable components pueden sincronizarse como ghosts, pero los cambios de activación deben pasar por el lado autoritativo.
Serialización / exportación de world Los bits de habilitación se preservan.

8. Solución de problemas

Síntoma Causa / Solución
La query incluye entidades que deberían estar "desactivadas" El componente no implementa IEnableableComponent, o está establecido WithOptions(EntityQueryOptions.IgnoreComponentEnabledState).
SetComponentEnabled<T> lanza "type T does not support being enableable" El struct solo implementa IComponentData — añadir IEnableableComponent.
El parámetro EnabledRefRW<T> es rechazado por el source generator Al job o system le falta partial, o el componente no es habilitable.
Error del source generator "cannot have both RefRW and EnabledRefRW for the same component" Dividir la lógica — usar ref T o EnabledRefRW<T> por job, no ambos en el mismo componente. Véase §5 para los dos patrones.
El filtro de cambio se dispara de forma espuria Activar Enabled cuenta como un cambio para WithChangeFilter. Es el comportamiento esperado — combinar con lecturas de EnabledRefRO en lugar de activar/desactivar.
El ECB SetComponentEnabled se ignora Se llamó en una entidad diferida antes de que el ECB que la creó se vaciara. Ambas deben estar en el mismo ciclo de ECB.

Chapter 10. Singleton Component

1. Qué es (y qué no es) un singleton

Un singleton en Entities es simplemente un componente que se espera que exista en exactamente una entidad en el world. No existe una interfaz especial ISingleton — cualquier IComponentData se convierte en singleton por convención, y SystemAPI proporciona accesores ergonómicos que verifican la regla de "exactamente uno".

Usos típicos: estado del juego, configuración global, fuentes de tiempo y tick, estado de red, registros centrales de ECB.

public struct GameState : IComponentData
{
    public int   Score;
    public float TimeScale;
    public bool  Paused;
}

2. Los accesores

Dentro de cualquier ISystem / SystemBase:

// Reads
GameState gs      = SystemAPI.GetSingleton<GameState>();
RefRW<GameState> gsRW  = SystemAPI.GetSingletonRW<GameState>();
bool has          = SystemAPI.HasSingleton<GameState>();
bool got          = SystemAPI.TryGetSingleton<GameState>(out var gs2);

// Writes
SystemAPI.SetSingleton(new GameState { Score = 10 });

// Entity handle, if you need it
Entity e          = SystemAPI.GetSingletonEntity<GameState>();

// Buffers work too
DynamicBuffer<Waypoint> buf = SystemAPI.GetSingletonBuffer<Waypoint>();

Las variantes Get* lanzan excepción si existen cero o más de un singleton. Las variantes TryGet* devuelven false en esos casos. Las variantes RW marcan el componente como modificado para el rastreo de cambios.

3. Crear el singleton

Dos patrones comunes:

3.1 Hornearlo desde un SubScene

public class GameStateAuthoring : MonoBehaviour
{
    public float InitialTimeScale = 1f;

    class Baker : Baker<GameStateAuthoring>
    {
        public override void Bake(GameStateAuthoring authoring)
        {
            var e = GetEntity(TransformUsageFlags.None);
            AddComponent(e, new GameState
            {
                Score     = 0,
                TimeScale = authoring.InitialTimeScale,
                Paused    = false
            });
        }
    }
}

Adjuntar GameStateAuthoring a un único GameObject en el SubScene; el Baker produce exactamente una entidad con el singleton.

3.2 Crear al inicio del system

public void OnCreate(ref SystemState state)
{
    var e = state.EntityManager.CreateEntity(typeof(GameState));
    state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
}

Se usa esto para datos que no deben vivir en el authoring (configuración inyectada por servidor, generada proceduralmente, etc.).

4. Condicionando un system al singleton

Si un system solo tiene sentido cuando existe el singleton:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<GameState>();
}

RequireForUpdate<T>() indica al system que omita OnUpdate por completo cuando el componente no está presente. Evita comprobaciones nulas defensivas dentro de OnUpdate.

5. Rendimiento de solo lectura frente a lectura-escritura

  • GetSingleton<T>() — lee una copia. Sin rastreo de cambios.
  • GetSingletonRW<T>() — devuelve un RefRW<T> apuntando al chunk. Asignar .ValueRW marca el componente como modificado, lo que se desea si otros systems usan WithChangeFilter<T>.
  • SetSingleton — escritura completa, siempre incrementa la versión.

Preferir GetSingleton en systems de solo lectura intensiva; cambiar a GetSingletonRW solo cuando se muta realmente.

6. Múltiples candidatos — la trampa del "singleton con más de uno"

GetSingleton<T> lanza InvalidOperationException: expected exactly one entity with component T, found N cuando dos entidades acaban con el mismo componente singleton. Causas comunes:

  • Dos GameObjects en el SubScene tienen el script de authoring adjunto.
  • Un test fixture creó el singleton en tiempo de ejecución y el SubScene también lo hornea.
  • Un system crea el singleton en OnCreate sin guardar con HasSingleton.

Patrón defensivo:

public void OnCreate(ref SystemState state)
{
    if (!SystemAPI.HasSingleton<GameState>())
    {
        var e = state.EntityManager.CreateEntity(typeof(GameState));
        state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
    }
}

7. Buffer singletons

Un DynamicBuffer<T> también puede ser un singleton:

public struct QueuedEvent : IBufferElementData
{
    public int Type;
    public int Payload;
}

// Access:
DynamicBuffer<QueuedEvent> events = SystemAPI.GetSingletonBuffer<QueuedEvent>();
events.Add(new QueuedEvent { Type = 1 });

Útil para colas de eventos globales y flujos de comandos.

8. Solución de problemas

Síntoma Causa / Solución
InvalidOperationException: expected exactly one entity with component T, found 0 La entidad singleton nunca fue creada. Añadir un baker o crear en OnCreate con un guard de HasSingleton.
... found 2 Dos entidades tienen el componente. Deduplicar en el authoring o limpiar al inicio.
GetSingletonRW aparentemente no marca el componente como modificado Se escribió en .ValueRO o se ignoró el RefRW. Solo .ValueRW desencadena el rastreo de cambios.
El system se ejecuta antes de que se cree el singleton Usar RequireForUpdate<T> u ordenar los systems para que el creador se ejecute primero ([UpdateBefore]).
El singleton es modificado desde dos systems sin ordenación Añadir [UpdateAfter]/[UpdateBefore] o fusionar la lógica. Las condiciones de carrera en singletons son especialmente difíciles de depurar.
Netcode: el valor del singleton diverge entre cliente y servidor Sincronizar el singleton como ghost, o reconstruirlo determinísticamente desde entradas sincronizadas.

Chapter 11. System — ISystem vs SystemBase

1. Descripción general

Entities tiene dos formas de declarar un system: ISystem (struct no gestionado) y SystemBase (class gestionada). Se debe elegir ISystem salvo que se necesite específicamente lo que SystemBase ofrece.

ISystem SystemBase
Se declara como partial struct : ISystem partial class : SystemBase
Memoria No gestionada, asignable en pila Gestionada, propiedad del GC
Compilable con Burst Sí (añadir [BurstCompile]) No
Campos gestionados No Sí
Acceso a API gestionada Mediante EntityManager + ManagedAPI Directamente
Opción predeterminada ✅ Solo cuando sea necesario

2. ISystem — la opción predeterminada

using Unity.Burst;
using Unity.Entities;

[BurstCompile]
public partial struct EnemyAISystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        // Called once when the system is created.
        state.RequireForUpdate<EnemySpawnConfig>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var config = SystemAPI.GetSingleton<EnemySpawnConfig>();
        // ... logic
    }

    [BurstCompile]
    public void OnDestroy(ref SystemState state)
    {
        // Called once when the system is destroyed.
    }
}

Requisitos obligatorios:

  • El struct debe ser partial — el source generator añade un wrapper oculto.
  • Los tres métodos (OnCreate, OnUpdate, OnDestroy) son opcionales — omitir los que no se necesiten.
  • [BurstCompile] en el struct y en cada método que se quiera compilar con Burst.

3. SystemBase — cuando un system gestionado es inevitable

using Unity.Entities;

public partial class PlayerInputSystem : SystemBase
{
    private InputActions _input;   // managed field — the reason SystemBase exists

    protected override void OnCreate()
    {
        _input = new InputActions();
        _input.Enable();
    }

    protected override void OnUpdate()
    {
        var move = _input.Gameplay.Move.ReadValue<Vector2>();
        // ... dispatch to components
    }

    protected override void OnDestroy() => _input.Dispose();
}

Se debe usar SystemBase cuando:

  • Se necesita un campo gestionado que debe vivir con el system (objetos del Input System, handles de UI Toolkit, servicios gestionados, etc.).
  • Es necesario llamar a APIs gestionadas directamente dentro de OnUpdate sin usar el EntityManager.

Todo en SystemBase se ejecuta en el hilo principal sin Burst.

4. Ciclo de vida del system

Ambas variantes comparten los mismos nombres de hook:

Figure: System Lifecycle.

Hook Cuándo se dispara Uso típico
OnCreate Una vez, al crear el system Cachear queries, RequireForUpdate<T>, asignar contenedores nativos
OnStartRunning La(s) query(s) cambian de vacía(s) → no vacía(s) Reiniciar estado por sesión
OnUpdate Cada fotograma en el SystemGroup padre Trabajo principal
OnStopRunning La(s) query(s) cambian de no vacía(s) → vacía(s) Vaciar estado en buffer
OnDestroy Una vez, al destruir el world Liberar recursos nativos o gestionados

RequireForUpdate<T>()

Omitir OnUpdate por completo cuando un componente o singleton requerido está ausente:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<EnemySpawnConfig>();
    state.RequireForUpdate<GameRunning>();
}

Esta es la forma estándar de condicionar un system a "la configuración existe" / "el juego ha comenzado".

5. Elegir entre los dos

Reglas prácticas:

  1. Empezar con ISystem. Es compatible con Burst y no genera GC.
  2. Cambiar a SystemBase solo cuando un campo gestionado lo exija. Incluso entonces, considerar dividir: mantener el trabajo gestionado en un SystemBase delgado que escriba los resultados en componentes, y ejecutar la lógica pesada en un ISystem que lea esos componentes.
  3. No mezclar añadiendo un campo gestionado a un struct ISystem — no compilará. La división anterior es la solución idiomática.

Ejemplo de división:

// Managed — bridges Input System into components.
public partial class InputBridgeSystem : SystemBase
{
    private InputActions _input;
    // ... fills a PlayerInput singleton each frame
}

// Unmanaged + Burst — reads the PlayerInput singleton and drives gameplay.
[BurstCompile]
public partial struct PlayerMoveSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) { /* ... */ }
}

6. Referencia rápida de SystemState

Dentro de un ISystem, ref SystemState state es la ventana al world.

Miembro Propósito
state.EntityManager Cambios estructurales en el hilo principal.
state.World El World propietario.
state.Dependency El handle de dependencia de job actual — combinar con los jobs programados.
state.GetEntityQuery(...) Construir o recuperar una query en caché.
state.RequireForUpdate<T>() Condicionar el system a la presencia de un componente.
state.Enabled Establecer en false para omitir el system sin destruirlo.

En un SystemBase, estos son miembros de la propia clase (p. ej., this.EntityManager).

7. Solución de problemas

Síntoma Causa / Solución
error CS0246: The type or namespace name 'Entities' could not be found Falta using Unity.Entities;.
OnUpdate nunca se dispara RequireForUpdate<T> condiciona a un componente que aún no existe — verificar que el componente se cree en otro lugar.
Error de compilación: "partial struct must be declared partial" ISystem requiere partial para que el source generator pueda extender el struct.
Advertencia de Burst: "method is not burst compiled" Falta [BurstCompile] en el método. El atributo en el struct solo no es suficiente.
El system se ejecuta en el world incorrecto (cliente frente a servidor con Netcode) Añadir [WorldSystemFilter(...)] para elegir ServerSimulation / ClientSimulation / LocalSimulation.
No se puede añadir un campo gestionado a ISystem Es el comportamiento esperado — dividir la parte gestionada en un SystemBase como se muestra arriba.
El system no aparece en Window → Entities → Systems Falta partial, está fuera del world predeterminado, o excluido por [DisableAutoCreation].

Chapter 12. System Group & Update Order

1. Descripción general

Cada system vive dentro de un SystemGroup. Los grupos deciden cuándo se ejecutan sus hijos en cada fotograma, y los atributos de ordenación determinan la secuencia dentro de un grupo. Configurar correctamente la agrupación es la forma de evitar escrituras tardías por fotograma, condiciones de carrera y errores del tipo "por qué esta entidad tiembla".

2. Grupos predefinidos

Figure: System Group and Update Order.

Grupo Cuándo se ejecuta Contenido típico
InitializationSystemGroup Una vez por fotograma, antes de la simulación Inicializadores de un solo disparo, tareas de bootstrap
SimulationSystemGroup Una vez por fotograma La mayor parte de la lógica de juego
FixedStepSimulationSystemGroup Paso de tiempo fijo (predeterminado 60 Hz); puede ejecutarse 0, 1 o N veces por fotograma Física, lógica de paso determinista
PresentationSystemGroup Una vez por fotograma, después de la simulación Preparación de render, interpolación
BeginSimulationEntityCommandBufferSystem Primera cosa en Simulation Reproducción de ECB ordenada (temprana)
EndSimulationEntityCommandBufferSystem Última cosa en Simulation Reproducción de ECB ordenada (tardía)

Netcode for Entities añade más (PredictedSimulationSystemGroup, GhostSendSystem, etc.) — se anidan dentro de los anteriores. Véase 16_Netcode Client-Server World & Bootstrap.md y 19_Netcode Prediction & Rollback.md.

3. Colocar un system en un grupo

using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial struct EnemyAISystem : ISystem { /* ... */ }

Si se omite [UpdateInGroup], el system tiene por defecto SimulationSystemGroup.

Anidamiento

Los grupos pueden vivir dentro de otros grupos:

[UpdateInGroup(typeof(SimulationSystemGroup), OrderFirst = true)]
public partial class GameplaySystemGroup : ComponentSystemGroup { }

[UpdateInGroup(typeof(GameplaySystemGroup))]
public partial struct MovementSystem : ISystem { /* ... */ }

ComponentSystemGroup es la clase base para grupos personalizados. Actúan como systems y como contenedores a la vez.

4. Ordenación dentro de un grupo

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateAfter(typeof(EnemyAISystem))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct EnemyPathfindingSystem : ISystem { /* ... */ }
Atributo Significado
[UpdateBefore(typeof(X))] Ejecutar antes que X (ambos deben estar en el mismo grupo).
[UpdateAfter(typeof(X))] Ejecutar después de X.
OrderFirst = true (en [UpdateInGroup]) Ejecutar lo antes posible en el grupo.
OrderLast = true Ejecutar lo más tarde posible.

Consejo: preferir cadenas de [UpdateAfter] sobre OrderFirst/Last — se leen mejor en revisiones de código y sobreviven a refactorizaciones.

5. [RequireMatchingQueriesForUpdate]

Un atributo más reciente (Entities 1.x+) que indica a un system que omita OnUpdate a menos que al menos una entidad coincida con cada una de sus queries:

[RequireMatchingQueriesForUpdate]
public partial struct EnemyAISystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Only runs when there is at least one entity matching every query the system uses.
    }
}

Comparar con state.RequireForUpdate<T>(), que requiere un componente o singleton específico. [RequireMatchingQueriesForUpdate] es más implícito — se activa según las queries que tenga el system.

6. Grupos personalizados

Razones para crear uno:

  • Fase de juego cohesiva (p. ej., "aplicar daño antes de la animación").
  • Una frecuencia de tick diferente (p. ej., un paso fijo personalizado a 30 Hz).
  • Un interruptor de característica — activar o desactivar el grupo completo.
using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial class DamagePipelineGroup : ComponentSystemGroup
{
    // You can override OnUpdate to customise how the group ticks.
}

[UpdateInGroup(typeof(DamagePipelineGroup), OrderFirst = true)]
public partial struct ApplyDamageSystem : ISystem { /* ... */ }

[UpdateInGroup(typeof(DamagePipelineGroup))]
[UpdateAfter(typeof(ApplyDamageSystem))]
public partial struct DeathSystem : ISystem { /* ... */ }

Frecuencia de tick personalizada

public partial class SlowTickGroup : ComponentSystemGroup
{
    private float _accum;
    protected override void OnUpdate()
    {
        _accum += (float)SystemAPI.Time.DeltaTime;
        if (_accum < 0.1f) return;      // 10 Hz
        _accum = 0f;
        base.OnUpdate();                // tick children
    }
}

7. Paso de tiempo fijo frente a variable

Si el determinismo es importante (física, predicción en red, reproducción determinista), colocar el system en FixedStepSimulationSystemGroup:

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial struct PhysicsStepSystem : ISystem { /* ... */ }

Puntos a tener en cuenta:

  • FixedStepSimulationSystemGroup puede ejecutarse múltiples veces por fotograma (o cero veces) para ponerse al día con el tiempo real.
  • Usar SystemAPI.Time.DeltaTime — es el dt del paso fijo dentro de este grupo.
  • Todo lo que lee entrada o produce presentación debe permanecer en SimulationSystemGroup / PresentationSystemGroup.

8. Inspeccionar el orden en tiempo de ejecución

Window → Entities → Systems muestra:

  • Cada grupo y sus hijos, en el orden de ejecución real.
  • Tiempo por system.
  • Estado habilitado / deshabilitado (systems que se alternan en tiempo de ejecución).

Si el orden declarado mediante atributos no coincide con lo que se observa, casi siempre hay una dependencia circular o un [UpdateInGroup] olvidado.

9. Solución de problemas

Síntoma Causa / Solución
El system se ejecuta en la fase del fotograma incorrecta Falta o es incorrecto [UpdateInGroup]. El predeterminado es SimulationSystemGroup.
[UpdateAfter] se ignora El system objetivo está en un grupo diferente. Ambos deben compartir un grupo.
Error de dependencia circular al inicio Una cadena de [UpdateBefore] / [UpdateAfter] forma un ciclo. La ventana Systems imprime el ciclo.
El system se ejecuta varias veces por fotograma de forma inesperada Se colocó en FixedStepSimulationSystemGroup — ese grupo recupera el tiempo. Moverlo a SimulationSystemGroup si no debe hacerlo.
El grupo personalizado se ejecuta indefinidamente El override de OnUpdate nunca llamó a base.OnUpdate().
El system se ejecuta pero no hace nada RequireForUpdate<T> no está satisfecho, o cada query está vacía. Comprobar en Window → Entities → Query.

Chapter 13. JobSystem & Burst

1. Descripción general

El rendimiento de DOTS se apoya en dos características del motor que existían antes de Entities y que siguen siendo su base:

  • Job System — programar trabajo en hilos de trabajo con grafos de dependencias.
  • Burst Compiler — compilar jobs de C# a código nativo ajustado (SIMD, sin GC, sin despacho virtual).

Ninguno de los dos es un concepto específico de Entities. IJobEntity e IJobChunk son capas delgadas sobre IJob + Burst. Entender la capa base hace que todos los patrones de nivel superior sean fáciles de leer.

2. Job System — los tipos fundamentales

Interfaz Forma Usar cuando…
IJob void Execute() Trabajo de un solo uso en un único hilo.
IJobParallelFor void Execute(int index) Bucle de 0 a N en paralelo.
IJobParallelForBatch void Execute(int start, int count) Bucle en lotes paralelos (más amigable con la caché que por índice).
IJobChunk void Execute(ArchetypeChunk chunk, int chunkIndex, bool useEnabledMask, in v128 enabledMask) Iterar chunks ECS en paralelo.
IJobEntity void Execute([component params]) Iteración de entidades ECS — generado por fuente en un chunk job.

Todos son structs y todos son compatibles con Burst.

Ejemplo mínimo de IJobParallelFor

using Unity.Burst;
using Unity.Collections;
using Unity.Jobs;

[BurstCompile]
struct ScaleJob : IJobParallelFor
{
    public NativeArray<float> Data;
    public float Factor;

    public void Execute(int index) => Data[index] *= Factor;
}

// Scheduling:
var handle = new ScaleJob { Data = arr, Factor = 2f }.Schedule(arr.Length, 64);
handle.Complete();

Notas:

  • Los campos deben ser blittable.
  • El acceso de escritura al mismo NativeArray desde múltiples jobs requiere encadenamiento de dependencias (véase abajo).

3. Programación y dependencias

JobHandle a = jobA.Schedule();
JobHandle b = jobB.Schedule(a);              // b runs after a
JobHandle c = jobC.Schedule(JobHandle.CombineDependencies(a, b));
c.Complete();

Reglas:

  • Cada Schedule() devuelve un JobHandle. Pasar el handle anterior como dependencia si el nuevo job lee o escribe datos compartidos.
  • Nunca omitir .Complete() en el handle final antes de que el hilo principal lea los resultados (el Safety System lo impone).
  • En un ISystem, state.Dependency es el handle a nivel de system; los jobs programados se combinan automáticamente con él si se usa .Schedule(state.Dependency) o se usan ayudantes generados por fuente como IJobEntity.ScheduleParallel(...).

Dentro de un ISystem

[BurstCompile]
public void OnUpdate(ref SystemState state)
{
    var job = new MyJob { Dt = SystemAPI.Time.DeltaTime };
    state.Dependency = job.ScheduleParallel(state.Dependency);
}

IJobEntity.ScheduleParallel() sin una dependencia explícita se combinará con state.Dependency implícitamente. Ser explícito al encadenar múltiples jobs.

4. Burst — qué es y por qué

Burst compila un subconjunto de C# (métodos marcados con [BurstCompile], sin asignaciones GC, datos blittable) a código nativo altamente optimizado mediante LLVM.

Qué compila bien Burst

  • Métodos de struct IJob* / ISystem etiquetados con [BurstCompile].
  • Matemáticas en tipos de Unity.Mathematics (float3, quaternion, int4, etc.).
  • Bucles ajustados sobre NativeArray<T>, NativeList<T>, DynamicBuffer<T>, arrays de columna de ArchetypeChunk.

Qué no puede compilar Burst

  • Referencias gestionadas (class, string, List<T>, cualquier cosa no unmanaged).
  • Excepciones (se pueden capturar, pero conviene evitarlas).
  • Despacho dinámico mediante interfaces (la instanciación genérica explícita está bien).
  • System.IO / Reflection / la mayor parte del BCL más allá de matemáticas y primitivos.

Activar Burst

using Unity.Burst;

[BurstCompile]
public partial struct MyJob : IJobEntity
{
    // attribute on the struct marks the type for Burst
    // methods need [BurstCompile] too if you want each compiled
    [BurstCompile]
    void Execute(ref Velocity v) { /* ... */ }
}

Para systems:

[BurstCompile]
public partial struct MySystem : ISystem
{
    [BurstCompile] public void OnCreate(ref SystemState state) { }
    [BurstCompile] public void OnUpdate(ref SystemState state) { }
}

El atributo en el tipo no es suficiente — cada método que se quiera compilar también necesita [BurstCompile].

Burst Inspector

Window → Analysis → Burst Inspector muestra el IR de LLVM, el ensamblador y el uso de vectores por método. Usar cuando la pregunta es "por qué esto no es rápido".

5. Contenedores nativos

Burst y el Safety System esperan contenedores nativos en lugar de colecciones gestionadas:

Gestionado Equivalente nativo
T[] NativeArray<T>
List<T> NativeList<T>
Dictionary<K,V> NativeHashMap<K,V>, NativeParallelHashMap<K,V>
HashSet<T> NativeHashSet<T>, NativeParallelHashSet<T>
Queue<T> NativeQueue<T>
string FixedString32Bytes / FixedString64Bytes / etc.

Asignación:

var arr = new NativeArray<int>(1024, Allocator.TempJob);
// ...
arr.Dispose();

La vida útil importa: Burst + Safety verifica que los contenedores se eliminen y no se acceda a ellos tras la vida útil de un job.

6. Allocators

Allocator Vida útil Uso típico
Allocator.Temp Dentro del fotograma actual, solo hilo principal Contenedores de trabajo en OnUpdate
Allocator.TempJob A través de un job, ≤4 fotogramas Contenedores pasados a jobs
Allocator.Persistent Requiere Dispose() manual Contenedores propiedad del system que viven entre fotogramas
state.WorldUpdateAllocator Se limpia al final de la actualización del system group Preferido para datos transitorios por system

Preferir state.WorldUpdateAllocator (o su allocator Rewindable) dentro de los systems — evita la gestión manual de Dispose().

7. Safety System

En el Editor con "Jobs > Leak Detection" y "Jobs > Safety Checks" activados, Unity valida que:

  • Dos jobs no escriban en el mismo contenedor nativo sin una arista de dependencia.
  • Una lectura del hilo principal no compita con una escritura en curso.
  • Los contenedores nativos no se usen tras Dispose() o más allá de la vida útil de su allocator.

Los errores de seguridad se manifiestan como excepciones con un mensaje claro "job A y job B acceden ambos a X". No deshabilitar las comprobaciones de seguridad en desarrollo — las ganancias de rendimiento son pequeñas y el coste de depuración es alto.

8. Solución de problemas

Síntoma Causa / Solución
Advertencia de Burst: "method not burst compiled" Falta [BurstCompile] en el método, o Burst encontró un tipo no compatible (revise en el Burst Inspector la línea que falla).
InvalidOperationException: … writes to … without declaring dependency Falta una arista de dependencia — pasar el JobHandle anterior o usar el encadenamiento de state.Dependency.
ObjectDisposedException en un NativeArray Se eliminó antes de que terminara el job. Usar arr.Dispose(handle) o esperar primero en el handle.
El job programado nunca termina Falta .Complete() en una cadena, o hay una dependencia circular.
Regresión de rendimiento tras añadir [BurstCompile] El método se recompila en cada recarga de dominio; comprobar la caché de Burst. O un tipo incompatible con Burst se coló — el método volvió silenciosamente a IL.
La enumeración de NativeHashMap lanza excepción La enumeración no funciona bien con escrituras concurrentes — enumerar en su lugar una copia NativeArray<K>.
Bloqueo del hilo principal esperando un job Se llamó a .Complete() demasiado pronto. Mover el punto de finalización lo más tarde posible en el fotograma.

Chapter 14. IJobEntity · SystemAPI.Query

1. Descripción general

Dos APIs cubren el 95% de la iteración de componentes en Entities 6.5:

API Dónde se ejecuta Usar cuando…
SystemAPI.Query<...>() Hilo principal, dentro de OnUpdate El trabajo es trivial (unos pocos cientos de entidades), o se necesitan APIs del hilo principal.
IJobEntity Hilos de trabajo, mediante Schedule / ScheduleParallel El trabajo es más pesado o se beneficia del paralelismo.

Son el reemplazo recomendado del legacy Entities.ForEach (marcado como obsoleto en Entities 1.4; sigue compilando en 6.5 con advertencias de deprecación; el changelog de Unity indica que su eliminación está prevista para una futura versión principal).

2. SystemAPI.Query<T>

Iterar las entidades que coinciden en el hilo principal. Los componentes se acceden mediante RefRO<T> / RefRW<T>.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRW.Max);
        }
    }
}

Reglas:

  • El struct del system debe ser partial.
  • RefRO<T> para solo lectura, RefRW<T> para lectura-escritura. No escribir mediante RefRO.
  • Añadir una Entity a la tupla si se necesita el id: SystemAPI.Query<RefRW<Health>>().WithEntityAccess().

Filtros

foreach (var health in SystemAPI.Query<RefRW<Health>>()
             .WithAll<Enemy>()
             .WithNone<Dead>()
             .WithAny<Burning, Poisoned>())
{
    // only entities that are Enemy AND not Dead AND (Burning OR Poisoned)
}
  • WithAll<T>() — debe tener todos los componentes listados.
  • WithNone<T>() — no debe tener ninguno de ellos.
  • WithAny<T...>() — debe tener al menos uno de ellos.
  • WithChangeFilter<T>() — solo chunks donde el componente T cambió desde la última ejecución.
  • WithEntityAccess() — también devuelve el id de Entity.

3. IJobEntity

Una iteración paralela generada por source generation. La firma de Execute declara qué componentes se tocan.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Reglas de la firma:

  • void Execute(...) — los parámetros se mapean 1:1 con los componentes.
  • ref Component — lectura-escritura. in Component — solo lectura (no se puede escribir).
  • Entity entity como parámetro está permitido — el source generator lo rellena.
  • El struct del job debe ser partial.

Variantes de Schedule

Llamada Hilos
.Run() Hilo principal, completa antes de retornar. Útil para depuración o cargas de trabajo pequeñas.
.Schedule() Un único hilo de trabajo.
.ScheduleParallel() Distribuye los chunks entre los hilos de trabajo.

Los tres actualizan state.Dependency automáticamente dentro de ISystem.OnUpdate.

Filtros en jobs

new HealthRegenJob { DeltaTime = dt }
    .ScheduleParallel(SystemAPI.QueryBuilder()
        .WithAll<Enemy, Regen>()
        .WithNone<Dead>()
        .Build());

O mediante atributos [WithAll]/[WithNone]/[WithAny] en el propio struct del job.

Acceso a entidad en un job

void Execute(Entity entity, ref Health health) { /* ... */ }

Índice de chunk / [ChunkIndexInQuery]

Útil al combinar con un ECB de ParallelWriter para reproducción determinista — pasar el índice de chunk como sortKey. Véase 15_ParallelWriter · Deterministic Playback.md.

void Execute([ChunkIndexInQuery] int chunkIndex, in Health health, Entity entity)
{
    if (health.Value <= 0)
        ecb.DestroyEntity(chunkIndex, entity);
}

4. Cuándo usar cada uno

Situación Elegir
Decenas a cientos de entidades, matemáticas triviales SystemAPI.Query
Miles o más de entidades, trabajo independiente por entidad IJobEntity.ScheduleParallel
Trabajo por chunk (bounds, sumas, particionamiento espacial) IJobChunk — véase 12_IJobEntity vs IJobChunk.md
Se necesita API gestionada dentro del bucle SystemAPI.Query desde un SystemBase
Se necesitan cambios estructurales diferidos Cualquier forma, pero escribir mediante EntityCommandBuffer.ParallelWriter — véase 14_EntityCommandBuffer · Deferred Entity.md

5. Puntos a tener en cuenta

  • No capturar this ni campos gestionados en un job. El source generator dará error si se intenta.
  • RefRW<T>.ValueRW desencadena el rastreo de cambios. Leer mediante ValueRW sin escribir sigue marcando el chunk como sucio.
  • WithChangeFilter<T> opera a nivel de chunk. Si cualquier entidad del chunk escribe en T, todo el chunk se considera modificado.
  • Los Aspects están obsoletos (pero no eliminados). La interfaz de agrupación IAspect fue deprecada en 1.4 y sigue apareciendo como tipo de parámetro válido de Execute en la referencia de API 6.5. El código nuevo debe usar parámetros de componente simples; los aspects existentes siguen compilando en 6.5.

6. Solución de problemas

Síntoma Causa / Solución
Error de compilación: IJobEntity must be partial Añadir partial al struct del job.
ScheduleParallel lanza un error de condición de carrera Dos jobs escriben en el mismo componente sin una arista de ordenación. Asegurarse de que state.Dependency se combine correctamente, o añadir [UpdateAfter] entre systems.
El job no se ejecuta con Burst Falta [BurstCompile] en el struct del job, o se captura un tipo incompatible con Burst (referencia gestionada, string sin FixedString).
SystemAPI.Query no devuelve nada La combinación de filtros está vacía. Inspeccionar la query en Window → Entities → Query para ver qué coincide realmente.
Escribir mediante RefRO (bug silencioso) El compilador ahora produce un error con refRO.ValueRW = …. Si pasa con versiones antiguas de source-gen, cambie a RefRW.
Discrepancia en el recuento de entidades entre SystemAPI.Query y una EntityQuery construida manualmente Conjunto de filtros diferente. SystemAPI.Query incluye el filtrado de enableable components por defecto; EntityQueryBuilder requiere configuración explícita.

Chapter 15. IJobEntity vs IJobChunk

1. La respuesta corta

  • IJobEntity para "hacer lo mismo a cada entidad".
  • IJobChunk para "hacer algo en cada chunk que no sea un mapa simple por entidad".

IJobEntity se genera sobre IJobChunk — ambos usan la misma maquinaria de iteración. IJobChunk simplemente expone el chunk completo al código.

2. IJobEntity — ergonomía por entidad

[BurstCompile]
public partial struct DamageTickJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Burn burn)
    {
        health.Value -= burn.DamagePerSecond * DeltaTime;
    }
}

Ventajas:

  • Un método, una vista por entidad. Fácil de escribir, fácil de leer.
  • Fontanería generada por fuente: los parámetros Entity, [ChunkIndexInQuery], [EntityIndexInQuery] se rellenan automáticamente.
  • Filtros mediante atributos [WithAll] / [WithNone] / [WithAny] en el struct, o mediante un QueryBuilder pasado a .Schedule*.

Desventajas:

  • Sin visibilidad entre entidades dentro de Execute. No es fácil "sumar sobre este chunk" sin estado adicional.
  • Sin acceso a metadatos a nivel de chunk (versión de cambio, valor del componente compartido) desde dentro de Execute.

3. IJobChunk — control completo del chunk

[BurstCompile]
public partial struct HealthAggregateJob : IJobChunk
{
    [ReadOnly] public ComponentTypeHandle<Health> HealthHandle;
    public NativeArray<float> PerChunkTotal;  // size = chunk count

    public void Execute(in ArchetypeChunk chunk,
                        int unfilteredChunkIndex,
                        bool useEnabledMask,
                        in v128 chunkEnabledMask)
    {
        var healths = chunk.GetNativeArray(ref HealthHandle);
        float total = 0f;
        for (int i = 0; i < chunk.Count; i++)
            total += healths[i].Value;

        PerChunkTotal[unfilteredChunkIndex] = total;
    }
}

Ventajas:

  • Acceso directo a arrays de chunk: chunk.GetNativeArray(ref handle), chunk.GetBufferAccessor(ref bufferHandle).
  • chunk.Has<T>(), chunk.GetSharedComponent<T>(...) para trabajo condicional.
  • Ve la máscara de habilitados y la versión de cambio por chunk.
  • Ideal para reducciones, particionamiento espacial, registro de ECB por lotes.

Coste:

  • Más boilerplate: declarar ComponentTypeHandle<T> en el system, actualizarlos cada fotograma con handle.Update(ref state).
  • Se gestiona el bucle for interno — por lo que hay más lugares donde equivocarse.

Configurar los handles en un system

[BurstCompile]
public partial struct HealthAggregateSystem : ISystem
{
    private EntityQuery _query;
    private ComponentTypeHandle<Health> _healthHandle;

    public void OnCreate(ref SystemState state)
    {
        _query        = state.GetEntityQuery(ComponentType.ReadOnly<Health>());
        _healthHandle = state.GetComponentTypeHandle<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthHandle.Update(ref state);

        var totals = CollectionHelper.CreateNativeArray<float>(
            _query.CalculateChunkCount(), state.WorldUpdateAllocator);

        state.Dependency = new HealthAggregateJob
        {
            HealthHandle   = _healthHandle,
            PerChunkTotal  = totals
        }.ScheduleParallel(_query, state.Dependency);
    }
}

handle.Update(ref state) refresca el handle de tipo en caché cada fotograma — olvidarlo provoca que Burst use índices obsoletos.

4. Máscara de habilitados (useEnabledMask)

Para queries que incluyen un IEnableableComponent, Execute recibe useEnabledMask = true y una máscara v128 de bits habilitados. Se debe respetar la máscara o se procesarán entidades desactivadas.

Ayudante:

var enumerator = new ChunkEntityEnumerator(useEnabledMask, chunkEnabledMask, chunk.Count);
while (enumerator.NextEntityIndex(out int i))
{
    // i is the index of the next enabled entity in the chunk
}

IJobEntity aplica la máscara automáticamente — otra razón para preferirlo salvo que se necesite visibilidad a nivel de chunk.

5. Guía de decisión

Situación Elegir
Tick de salud, traslación de posición, aplicar entrada — matemáticas puras por entidad IJobEntity
Sumar valores de todas las entidades de un chunk IJobChunk
Registrar un comando ECB por chunk en lugar de por entidad IJobChunk
Acceder a un valor de componente compartido dentro del bucle IJobChunk
Necesitar la versión de cambio del chunk IJobChunk
Iterar un DynamicBuffer<T> en cada entidad Cualquiera funciona; IJobEntity es más limpio
Lógica por entidad pero se necesita el índice de chunk de la entidad IJobEntity con [ChunkIndexInQuery] int chunkIndex
El coste de programación importa y el trabajo es pequeño Hacer benchmark — menos chunks gana frente a menos entidades. IJobChunk suele ganar con miles de chunks.

6. Interoperabilidad — usar ambos juntos

Un patrón común: recopilar datos por chunk con IJobChunk, luego hacer escrituras por entidad con IJobEntity, encadenados mediante dependencias.

var gather = new HealthAggregateJob { ... }
    .ScheduleParallel(query, state.Dependency);

var act = new HealJob { PerChunkTotal = totals }
    .ScheduleParallel(gather);

state.Dependency = act;

7. Solución de problemas

Síntoma Causa / Solución
IJobChunk.Execute lee valores basura Se olvidó handle.Update(ref state) antes de programar, o se usó la query incorrecta.
ComponentTypeHandle<T> no encontrado Falta using Unity.Entities;, o el handle se declaró con el modo de acceso incorrecto. Usar GetComponentTypeHandle<T>(true) para solo lectura.
Entidades habilitadas omitidas No se respetó la máscara — usar ChunkEntityEnumerator como se muestra. IJobEntity evita esto.
El job escribe en el mismo NativeArray desde dos chunks Proporcionar un slot por chunk (como en HealthAggregateJob) o usar NativeParallelHashMap con el índice de chunk como clave.
Programar IJobChunk sin argumento de query ScheduleParallel(query, ...) requiere la query explícitamente. IJobEntity.ScheduleParallel() la obtiene de los atributos o del source generator.
Tipo v128 no encontrado Falta using Unity.Burst.Intrinsics;.

Chapter 16. Structural Change & Safety

1. Qué cuenta como cambio estructural

Un cambio estructural es cualquier operación que puede mover entidades entre chunks o alterar el inventario de chunks. Todas las siguientes son estructurales:

Operación Por qué es estructural
CreateEntity / DestroyEntity Asigna o libera un slot de chunk.
AddComponent<T> / RemoveComponent<T> El archetype de la entidad cambia, por lo que se mueve físicamente a un chunk diferente.
SetSharedComponent<T> Valor de componente compartido diferente → chunk diferente (agrupa por valor).
AddChunkComponent<T> / RemoveChunkComponent<T> Cambia los metadatos propios del chunk.
SetName(entity, ...) Toca los metadatos de entidad en el EntityManager.
MoveEntitiesFrom Copia entidades entre worlds.
Instantiate(prefab) Asigna una nueva entidad (puede pre-rellenar un archetype).

Escribir el valor de un componente mediante SetComponentData / RefRW<T>.ValueRW no es estructural — el componente ya vive en el chunk; solo se cambian bytes.

2. Por qué los cambios estructurales son peligrosos

Cuando el archetype cambia, todo lo que tiene una referencia basada en posición dentro de un chunk queda obsoleto:

  • Los iteradores de query en vuelo (SystemAPI.Query<...>) pueden omitir o revisitar entidades.
  • Las entradas de ComponentLookup<T> se invalidan para las entidades afectadas.
  • Los handles de ArchetypeChunk en un job apuntan al chunk antiguo.
  • Otros systems ejecutándose simultáneamente observan un estado inconsistente.

El Safety System detecta la mayoría de estos casos con una InvalidOperationException en el Editor. En una build, se convierten en crashes o entidades corruptas.

3. Regla general

No realizar cambios estructurales mientras se itera una query en la misma actualización del system.

Dos patrones conformes, en orden de preferencia:

  1. Usar un EntityCommandBuffer. Registrar la operación durante la iteración, reproducirla después. Véase 14_EntityCommandBuffer · Deferred Entity.md.
  2. Iterar en una NativeList<Entity>, luego mutar después del bucle. Más sencillo para casos pequeños; sin sobrecarga de ECB.

Ejemplo del patrón 2:

var toKill = new NativeList<Entity>(state.WorldUpdateAllocator);

foreach (var (health, e) in SystemAPI
             .Query<RefRO<Health>>()
             .WithEntityAccess())
{
    if (health.ValueRO.Value <= 0f)
        toKill.Add(e);
}

state.EntityManager.DestroyEntity(toKill.AsArray());

4. Enableable components — la vía de escape

Si se añade y quita un componente cada fotograma, casi con certeza debería ser un enableable component — activar el bit de habilitación no es estructural. Véase 06_Enableable Component.md.

5. Jobs y cambios estructurales

Los cambios estructurales son operaciones del hilo principal. No se pueden realizar dentro de un job.

Las formas idiomáticas de mutar desde un job:

Enfoque Cuándo
EntityCommandBuffer.ParallelWriter registrado en el job, reproducido en el hilo principal Cambios estructurales por entidad desde código paralelo.
EntityCommandBuffer (hilo único) registrado en un IJob o IJobChunk, reproducido en el hilo principal Cambios por lotes o con alcance de chunk.
Activar un enableable component mediante EnabledRefRW<T> dentro de un job "Activar/desactivar" por entidad sin estructura.

Un ECB captura la operación como un comando; la reproducción ocurre en un punto conocido del hilo principal, generalmente en un límite de EntityCommandBufferSystem.

6. Combinar cambios estructurales y lecturas

Si dos systems se ejecutan uno tras otro y el system A realiza un cambio estructural del que depende el system B, hacer el orden explícito:

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct SpawnSystem : ISystem { /* uses ECB */ }

Cuando la reproducción del ECB system se sitúa entre A y B (p. ej., EndSimulationECBS con una cadena adecuada de [UpdateAfter]), el cambio estructural ocurre antes de que B lea.

7. Modelo de coste

Clasificación aproximada para un único cambio:

SetComponentData (value only)          ~ 1x
SetComponentEnabled (enableable bit)   ~ 1x (per-chunk bitmask write)
AddComponent (structural)              ~ 10x   (entity moves chunk)
SetSharedComponent (structural)        ~ 10x
DestroyEntity (structural)             ~ 10x
Per-frame add + remove pattern         pathological

Órdenes de magnitud; los números exactos dependen del tamaño del archetype, la ocupación del chunk y la presión del allocator.

8. Diagnosticar el desgaste estructural

Síntomas:

  • Window → Entities → Systems muestra un system con un tiempo desproporcionado para su lógica.
  • El marcador del Profiler EntityManager.SetArchetype o Move Entity domina los fotogramas.
  • El recuento de archetypes crece con el tiempo.

Tácticas:

  • Auditar llamadas a AddComponent / RemoveComponent — ¿alguna es por fotograma? Reemplazarlas con enableable components.
  • Inspeccionar los conjuntos de valores de componentes compartidos — ¿alguna clave es accidentalmente por entidad?
  • Comprobar la ventana Archetypes en busca de chunks con una sola entidad; eso es fragmentación por conjuntos de componentes demasiado granulares.

9. Solución de problemas

Síntoma Causa / Solución
InvalidOperationException: EntityCommandBuffer has already been played back Se grabó en un ECB después de su reproducción. Cada ECB es de un único uso — obtener uno nuevo del ECBS en cada fotograma.
InvalidOperationException: … invalidated … durante un foreach Cambio estructural dentro del bucle. Mover a un ECB o usar el patrón buffer-luego-aplicar.
Las entidades desaparecen silenciosamente Ocurrió un DestroyEntity pero otros systems siguen teniendo el handle antiguo de Entity. Verificar con EntityManager.Exists.
El patrón de añadir/quitar consume tiempo de fotograma Reemplazar con enableable components.
RemoveComponent desencadena error "cleanup component present" La entidad tiene un cleanup component. Gestionar primero el cleanup (véase 05_Component Types.md).
El conjunto de valores del componente compartido explota (cientos de valores distintos) El valor es demasiado granular. Agrupar por cubos (p. ej., redondear al N más cercano) o almacenar la parte variable como componente normal.

Chapter 17. EntityCommandBuffer · Deferred Entity

1. Por qué existe el ECB

EntityCommandBuffer (ECB) registra cambios estructurales — crear, destruir, añadir, quitar, establecer — y los reproduce posteriormente en el hilo principal en un punto seguro. Permite mutar el world desde:

  • Dentro de un foreach de query (donde las llamadas directas a EntityManager invalidarían el iterador).
  • Un job (donde los cambios estructurales en el hilo principal no están permitidos).
  • Un job paralelo a través de muchos chunks (mediante ParallelWriter).

Véase 13_Structural Change & Safety.md para el contexto sobre "por qué esto es peligroso".

2. El ciclo de vida del ECB

Figure: EntityCommandBuffer Lifecycle.

Un ECB es de un único uso: tras la reproducción se elimina. Se solicita uno nuevo al EntityCommandBufferSystem correcto en cada fotograma.

3. ECB systems predefinidos

System Cuándo reproduce Uso típico
BeginInitializationEntityCommandBufferSystem Inicio del grupo Initialization Configuración temprana del world desde el fotograma anterior
EndInitializationEntityCommandBufferSystem Fin del grupo Initialization —
BeginSimulationEntityCommandBufferSystem Inicio de Simulation Aplicar cambios pendientes antes de que se ejecute el juego
EndSimulationEntityCommandBufferSystem Fin de Simulation Opción predeterminada para spawns/destroys dirigidos por el juego
BeginPresentationEntityCommandBufferSystem Inicio de Presentation Poco común; configuración solo visual

Elegir el correcto determina cuándo otros systems observan los cambios. Si se spawnea un proyectil en SimulationSystemGroup y se quiere que se mueva en ese mismo fotograma, grabar en BeginSimulation al siguiente fotograma — o en EndSimulation si el siguiente fotograma está bien.

4. Obtener un ECB

Dentro de cualquier ISystem.OnUpdate:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

El tipo anidado .Singleton es un token por system que el source generator expone para cada ECB system predefinido. CreateCommandBuffer proporciona un ECB nuevo que se reproducirá cuando ese system haga tick.

Para jobs paralelos:

var ecb = SystemAPI

5. Grabación — creación de entidades

Entity e = ecb.CreateEntity();
ecb.AddComponent(e, new Health { Value = 100 });

O instanciar desde una entidad prefab:

ecb.SetComponent(inst, LocalTransform.FromPosition(spawnPos));

Tanto CreateEntity como Instantiate devuelven una entidad diferida (véase §6).

6. Entidades diferidas

Una entidad devuelta por ecb.CreateEntity() o ecb.Instantiate() aún no existe. Tiene un índice de marcador de posición negativo. Se puede:

  • Pasarla a llamadas ECB posteriores en el mismo ECB — AddComponent, SetComponent, AppendToBuffer.
  • Almacenarla en otro componente mediante ecb.SetComponent(other, new ReferenceTo { Entity = inst }).

No se puede:

  • Leer o escribir en ella mediante EntityManager hasta que el ECB se reproduzca.
  • Pasarla entre diferentes ECBs (el marcador de posición solo se resuelve dentro del ECB propietario).

En el momento de la reproducción, el ECB system sustituye el handle real de Entity en todos los lugares donde se usó el marcador de posición.

10. Determinismo

Los comandos ECB se reproducen en el orden de grabación en ECBs de escritor único. En un ParallelWriter, el orden de reproducción depende del sortKey proporcionado — véase 15_ParallelWriter · Deterministic Playback.md.

Para sistemas Netcode / replay, la reproducción determinista es innegociable. Pasar siempre una clave de ordenación estable (índice de chunk, índice en la query) y evitar grabar en rutas de código del hilo principal que se ejecuten en orden no determinista.

Chapter 18. ParallelWriter · Deterministic Playback

1. Por qué existe un writer paralelo separado

EntityCommandBuffer es de escritor único. Dos hilos grabando en el mismo ECB competerían en su cadena interna. EntityCommandBuffer.ParallelWriter soluciona esto dando a cada llamador una cadena local por hilo y requiriendo una clave de ordenación por comando.

En el momento de la reproducción, el ECB system:

  1. Aplana todas las cadenas locales por hilo.
  2. Ordena los comandos por sortKey (ordenación estable).
  3. Los ejecuta en el hilo principal.

El resultado es determinista siempre que las claves de ordenación sean deterministas.

2. Obtener un ParallelWriter

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

AsParallelWriter() devuelve un struct EntityCommandBuffer.ParallelWriter que se puede copiar en un IJobEntity, IJobChunk o IJobParallelFor.

3. El contrato de sortKey

Cada llamada de ECB paralela toma un sortKey como primer argumento:

ecb.Instantiate(sortKey, prefab);
ecb.SetComponent(sortKey, entity, value);
ecb.DestroyEntity(sortKey, entity);
ecb.AddComponent(sortKey, entity, component);

Reglas para una buena clave de ordenación:

  • Determinista. La misma entrada → la misma clave, sin importar cómo distribuya el trabajo el planificador.
  • Suficientemente única. Dos comandos en la misma entidad no deben colisionar a menos que se quiera que el orden de reproducción sea ambiguo.
  • Barata. Un int que ya se tiene, no un hash.

En la práctica, las opciones preferidas son:

Fuente Cómo obtenerla Cuándo usar
[ChunkIndexInQuery] en IJobEntity El source generator la rellena 99% de los casos
unfilteredChunkIndex en IJobChunk.Execute Proporcionado por la API Jobs de IJobChunk
[EntityIndexInQuery] en IJobEntity Source generator Cuando los comandos por entidad deben ordenarse individualmente
Un índice determinista calculado manualmente Se calcula Poco común; formas de job personalizadas

4. Ejemplo completo — spawn paralelo

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnParallelJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity spawnerEntity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, spawnerEntity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

[BurstCompile]
public partial struct SpawnerParallelSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged)
            .AsParallelWriter();

        new SpawnParallelJob
        {
            DeltaTime = SystemAPI.Time.DeltaTime,
            ECB       = ecb
        }.ScheduleParallel();
    }
}

Pasar chunkIndex tanto a Instantiate como al posterior SetComponent garantiza que los dos comandos se ordenen en la misma posición y que la referencia a la entidad diferida de Instantiate se resuelva al objetivo correcto.

5. Determinismo — qué garantiza y qué no

Garantizado: Dado el mismo contenido de ECB, la reproducción produce un estado resultante idéntico — independientemente de qué hilo de trabajo grabó qué comando.

No garantizado automáticamente:

  • IDs de entidad. Dos ejecuciones pueden producir entidades con valores de Index distintos si los comandos anteriores difirieron.
  • Orden de chunks entre systems. Si el system A produce entidades en paralelo y el system B las lee en orden de ChunkIndexInQuery, el orden de iteración puede seguir dependiendo del historial de creación de archetypes.

Si se necesita determinismo bit a bit entre ejecuciones (replays, rollback de Netcode), combinar:

  1. ECB paralelo con claves de ordenación deterministas.
  2. Systems de paso fijo dentro de FixedStepSimulationSystemGroup.
  3. Sin dependencia de Entity.Index; usar IDs de juego que se controlen.

6. Patrones de clave de ordenación

6.1 Por chunk (el predeterminado)

Usar [ChunkIndexInQuery] int chunkIndex. Las colisiones están bien si cada entidad del chunk genera un comando idéntico (p. ej., un chunk completo marcado como aturdido).

6.2 Por entidad

Usar [EntityIndexInQuery] int entityIndex cuando se necesita un orden estricto por entidad (p. ej., grabar eventos de daño por entidad que deben reproducirse en orden de entidad).

6.3 Índice determinista personalizado

Muy raramente, se construye un índice plano desde las entradas — p. ej., int sortKey = baseOffset + localIndex;. Solo hacer esto cuando las dos opciones comunes no encajan; los índices personalizados son una fuente frecuente de bugs de determinismo.

7. Evitar estos patrones

  • Usar JobsUtility.ThreadIndex como sortKey. No es determinista.
  • Usar Random para construir la clave. No es determinista.
  • Mezclar ECBs. Grabar en dos ECBs distintos y se pierde el orden total.
  • Grabar en un ECB por system desde múltiples systems. Un ECB, un grabador (o un parallel-writer envolviendo un único ECB).

Síntoma Causa / Solución

└─ Fixed timestep, 60 Hz default

      ├─ GhostSendSystem

├─ Dynamic timestep

6. Migración de world

Usar DriverMigrationSystem cuando una transición de world debe preservar el estado de la conexión.

using Unity.Entities;
using Unity.NetCode;

public World MigrateServerWorld(World sourceWorld)
{
    DriverMigrationSystem migrationSystem = null;
    foreach (var world in World.All)
    {
        if ((migrationSystem = world.GetExistingSystemManaged<DriverMigrationSystem>()) != null)
            break;
    }

    var ticket = migrationSystem.StoreWorld(sourceWorld);
    sourceWorld.Dispose();

    var newWorld = ClientServerBootstrap.CreateServerWorld("NewServerWorld");
    var newMigrationSystem = newWorld.GetExistingSystemManaged<DriverMigrationSystem>();
    newMigrationSystem.LoadWorld(ticket);
    return newWorld;
}

Chapter 20. Netcode Network Connection & Approval

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Netcode for Entities representa cada conexión de red como una entidad. La entidad de conexión lleva NetworkStreamConnection; cuando la conexión termina, la entidad se destruye.

La conexión no se considera "in game" hasta que ambos lados añadan NetworkStreamInGame. Sin ese componente, el servidor no envía snapshots y el cliente no envía comandos.

2. Componentes de conexión fundamentales

NetworkStreamConnection Almacena el handle de transporte.
NetworkId ID único asignado por el servidor tras la aprobación.
NetworkStreamInGame Necesario para intercambiar snapshots y datos de comandos.

NetworkStreamRequestDisconnect Solicitud de desconexión; no llamar directamente al driver disconnect.
NetworkStreamIsReconnected Componente de seguimiento de reconexión añadido en la línea 1.6.1.

5.3 RPC de aprobación

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;

public struct ApprovalRpc : IApprovalRpcCommand
{
    public FixedString512Bytes Token;
}

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation |
    WorldSystemFilterFlags.ThinClientSimulation)]
public partial struct ClientApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (connection, entity) in
            SystemAPI.Query<RefRW<NetworkStreamConnection>>()
                .WithNone<NetworkId>()
                .WithNone<ApprovalSent>()
                .WithEntityAccess())
        {
            var rpcEntity = ecb.CreateEntity();
            ecb.AddComponent(rpcEntity, new ApprovalRpc { Token = "MY_TOKEN" });
            ecb.AddComponent<SendRpcCommandRequest>(rpcEntity);
            ecb.AddComponent<ApprovalSent>(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ServerApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, rpc, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ApprovalRpc>>()
                .WithEntityAccess())
        {
            var connEntity = req.ValueRO.SourceConnection;

            if (rpc.ValueRO.Token.Equals("MY_TOKEN"))
                ecb.AddComponent<ConnectionApproved>(connEntity);
            else
                ecb.AddComponent<NetworkStreamRequestDisconnect>(connEntity);

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

El tiempo de espera de aprobación está controlado por ClientServerTickRate.HandshakeApprovalTimeoutMS; el valor predeterminado es 5000 ms.

Chapter 21. Netcode Ghost Snapshot & Synchronization

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Un Ghost es una entidad en red simulada en el servidor y replicada en los clientes. El servidor envía periódicamente el estado del Ghost como snapshots, y cada cliente aplica esos snapshots mediante interpolación o predicción.

Mecanismo Uso
Ghost Consistencia eventual sobre paquetes no fiables con optimización de replicación integrada.
RPC Evento fiable de un único disparo.

2. Ghost vs RPC

Usar Ghost para Usar RPC para
Estado de entidad espacial cerca de un cliente. Eventos de flujo de juego de alto nivel.
Datos que necesitan predicción en el cliente. Comandos de un único disparo como chat o cambios de configuración.
Estado que debe persistir tras un ciclo de desconexión/reconexión. Eventos sin estado replicado persistente.

3. Authoring de Ghost

Añadir GhostAuthoringComponent al prefab que debe replicarse.

Propiedad Significado
Name Identificador del Ghost.
Importance Prioridad de snapshot cuando el ancho de banda es limitado.
Supported Ghost Mode All, Interpolated o Predicted.
Default Ghost Mode Interpolated, Predicted o OwnerPredicted.
Optimization Mode Dynamic por defecto, o Static para objetos que raramente cambian.
MaxSendRate Tasa máxima de replicación en Hz, limitada por NetworkTickRate.
UseSingleBaseline Opción de compresión delta de una sola línea base de la línea 1.5.0.

3.1 Modos de Ghost

Modo Significado
Interpolated El cliente interpola entre snapshots del servidor. Los visuales son suaves pero con retraso.
Predicted El cliente simula por adelantado para reducir la latencia de entrada. El coste de CPU es mayor.
OwnerPredicted El cliente propietario predice; otros clientes interpolan.

7. Sistema de snapshots

7.1 Snapshots parciales

Si un snapshot excedería el MTU, Netcode envía primero los Ghosts de mayor importancia. Los Ghosts restantes se envían en ticks posteriores, por lo que el estado del world se transmite progresivamente.

7.2 NetworkTickRate

Tick 2: GhostSendSystem skips
Tick 3: GhostSendSystem sends a snapshot

Chapter 22. Netcode Prediction & Rollback

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

La predicción en el cliente ejecuta el mismo código de simulación en el cliente antes de que llegue el resultado autoritativo del servidor. Esto reduce la latencia de entrada y hace que el movimiento del jugador responda de inmediato.

La predicción tiene un coste real de CPU: cuando llega un snapshot autoritativo, el cliente puede retroceder a un tick anterior y re-simular hacia el tick predicho actual.

2. Componentes y systems fundamentales

Componente / system Significado
PredictedGhost Añadido a los Ghosts predichos en los clientes y a todos los Ghosts en el servidor.
Simulate Tag habilitable establecido en las entidades que deben simular para el tick de predicción actual.
PredictedSimulationSystemGroup Grupo de paso de tiempo fijo que ejecuta el bucle de predicción.
NetworkTime Contiene el tick del servidor, el tick de predicción y datos de temporización relacionados.

3. Flujo de predicción en el cliente

1. Receive a server snapshot
   └─ Apply the latest authoritative state to predicted entities

2. Find the oldest applied tick
   └─ Use the oldest tick across affected predicted entities

3. Roll back and re-simulate
   └─ PredictedSimulationSystemGroup repeats from oldest tick to target tick
   └─ TimeData and NetworkTime update for each tick

4. Render
   └─ Present the predicted result

La alta latencia multiplica el coste. Una conexión de 300 ms puede requerir aproximadamente 20+ fotogramas de re-simulación según la frecuencia de tick y la configuración de predicción.

4. Escribir systems de predicción

4.1 Usar el tag Simulate

Al inicio del bucle de predicción, Netcode deshabilita Simulate en los Ghosts predichos y luego lo activa solo para las entidades que deben ejecutarse en el tick actual.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct MovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 5f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<Simulate>())
        {
            var move = new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical);
            transform.ValueRW.Position += move * speed * dt;
        }
    }
}

Incluir siempre .WithAll<Simulate>() en las queries de predicción. Sin él, el system puede re-simular entidades para ticks que ya están completados.

4.2 Comportamiento en el servidor

En el servidor, el bucle de predicción se ejecuta exactamente una vez por tick de servidor:

  • TimeData contiene datos normales de tick del servidor.
  • Simulate siempre está activado.
  • El mismo código del system puede ejecutarse tanto en el cliente como en el servidor.

5. Predicción de jugadores remotos

Si los datos de comandos de otro jugador están serializados, ese jugador remoto también puede ser predicho.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct RemotePlayerMovement : ISystem
{

    public void OnUpdate(ref SystemState state)

            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()

                * SystemAPI.Time.DeltaTime;
        }
    }
}

Con IInputComponentData, el sistema de entrada proporciona la entrada para el tick actual automáticamente.

6. Suavizado de predicción

El suavizado de predicción reduce la corrección visible cuando el resultado del servidor difiere del resultado predicho.

using Unity.NetCode;
using Unity.Transforms;

GhostPredictionSmoothing.RegisterSmoothingAction<LocalTransform>(
    state.EntityManager,
    CustomSmoothing.Action);

using Unity.Burst;
using Unity.Collections.LowLevel.Unsafe;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[BurstCompile]
public static unsafe class CustomSmoothing
{
    public static readonly PortableFunctionPointer<
        GhostPredictionSmoothing.SmoothingActionDelegate> Action =
        new(SmoothingAction);

    [BurstCompile(DisableDirectCall = true)]
    private static void SmoothingAction(
        void* currentData, void* previousData, void* userData)
    {
        ref var current = ref UnsafeUtility.AsRef<LocalTransform>(currentData);
        ref var backup = ref UnsafeUtility.AsRef<LocalTransform>(previousData);

        var dist = math.distance(current.Position, backup.Position);
        if (dist > 0)
        {
            current.Position = math.lerp(backup.Position, current.Position, 0.5f);
        }
    }
}

7. Cambio de predicción

La predicción es costosa, por lo que un Ghost puede cambiar dinámicamente entre Predicted e Interpolated.

7.1 Requisitos previos para el cambio

  • Los Supported Ghost Modes deben ser All.
  • El modo actual no debe ser OwnerPredicted.
  • El Ghost no debe estar ya en proceso de cambio.

7.2 Poner en cola los cambios

using Unity.NetCode;

ref var switchQueues = ref SystemAPI
    .GetSingletonRW<GhostPredictionSwitchingQueues>()
    .ValueRW;

switchQueues.ConvertToPredictedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 1f,
});

switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 0f,
});

Los Ghosts interpolados y predichos se ejecutan en líneas de tiempo distintas, a menudo separadas por aproximadamente dos intervalos de ping. SwitchPredictionSmoothing interpola posición y rotación durante la duración configurada para evitar teletransportes visibles.

8. Controles de rendimiento de predicción

Técnica Efecto
Prediction Switching Cambiar Ghosts lejanos o de baja importancia a Interpolated.
Restringir el acceso de escritura a GhostField Usar acceso de solo lectura para campos que no cambian en el bucle de predicción.
Agrupación de física Usar MaxPredictionStepBatchSizeRepeatedTick para la re-simulación repetida de física.
Ajuste de tick-rate Reducir SimulationTickRate disminuye el recuento de re-simulaciones, a costa de la capacidad de respuesta.

1. Descripción general

Una vez que un cliente está in-game, envía un flujo de comandos continuo al servidor. El flujo incluye datos de entrada y datos de ACK de snapshot; continúa incluso cuando el jugador no está presionando activamente ninguna entrada para que la conexión permanezca sincronizada.

3. IInputComponentData

3.1 Definir la entrada

using Unity.NetCode;

public struct PlayerInput : IInputComponentData
{
    public int Horizontal;
    public int Vertical;
    public InputEvent Jump;
}

3.2 Recoger entrada en el cliente

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[UpdateInGroup(typeof(GhostInputSystemGroup))]
public partial struct GatherInputSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<PlayerInput>();
    }

    public void OnUpdate(ref SystemState state)
    {
        bool jump = Input.GetKeyDown(KeyCode.Space);
        bool left = Input.GetKey(KeyCode.A);
        bool right = Input.GetKey(KeyCode.D);

        foreach (var input in
            SystemAPI.Query<RefRW<PlayerInput>>()
                .WithAll<GhostOwnerIsLocal>())
        {
            input.ValueRW = default;
            if (jump) input.ValueRW.Jump.Set();
            if (left) input.ValueRW.Horizontal -= 1;
            if (right) input.ValueRW.Horizontal += 1;
        }
    }
}

3.3 Procesar la entrada en el servidor y el bucle de predicción

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct ProcessInputSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<PlayerInput>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 3f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (input, transform) in
            SystemAPI.Query<RefRO<PlayerInput>, RefRW<LocalTransform>>()
                .WithAll<Simulate>())
        {
            if (input.ValueRO.Jump.IsSet)
            {
                // Jump logic; runs once on the input tick.

            transform.ValueRW.Position +=
                new float3(input.ValueRO.Horizontal, 0, 0) * speed * dt;
        }
    }
}

Chapter 24. Netcode RPC

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Un RPC es un mensaje fiable de un único disparo entre cliente y servidor. El código de serialización y deserialización de RPC se genera automáticamente, y el manejo de RPC sigue encajando en el marco ECS / Job System.

Los RPCs no son un sustituto de la replicación de Ghosts ni de los flujos de comandos. Usarlos para eventos discretos de flujo de juego, no para estado de alta frecuencia.

2. RPC vs Ghost vs Command

RPC Ghost Command
Comportamiento de transporte Fiable, ordenado. No fiable con optimización de replicación. No fiable con envíos redundantes.
Dirección Bidireccional. Servidor → Cliente. Cliente → Servidor.
Frecuencia Un único disparo. Cada tick de red según sea necesario. Cada tick de simulación.
Uso Flujo de juego, lobby, chat. Estado de entidad replicada. Entrada del jugador.
Límite en vuelo Sí, límite del pipeline fiable. No. No.

Como los RPCs son ordenados y fiables, el tráfico de RPC frecuente puede paralizarse tras paquetes en vuelo.

Chapter 25. Netcode Ghost Optimization · Importance · Relevancy

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Los worlds multijugador grandes no pueden enviar cada Ghost a cada cliente en cada tick. Netcode for Entities usa Optimization Mode, Importance Scaling, Relevancy y controles de tasa de envío por Ghost para reducir el coste de CPU y el ancho de banda.

La regla fundamental es sencilla: el servidor debe gastar el presupuesto de snapshot en los Ghosts que importan a una conexión específica en este momento.

2. Optimization Mode

Establecer Optimization Mode en GhostAuthoringComponent.

Modo Significado Adecuado para
Dynamic Predeterminado. Optimiza el tamaño del snapshot tanto cuando los datos cambian como cuando no. Ghosts en movimiento normal.
Static Evita reenviar datos cuando el Ghost no ha cambiado. Edificios, terreno, props estáticos.

La línea 1.11.0 añadió una optimización de Ghost estático que puede reducir sustancialmente el tiempo de GhostUpdateSystem para Ghosts estáticos sin cambios.

3. Límites de tamaño de snapshot

El presupuesto de snapshot puede limitarse en varios niveles.

Configuración Significado
NetworkStreamSnapshotTargetSize Objetivo de bytes suave por conexión.
GhostSendSystemData.MaxSendEntities Máximo de entidades por snapshot.
GhostSendSystemData.MaxSendChunks Máximo de chunks por snapshot.

Cuando el presupuesto es limitado, primero se envían los Ghosts de mayor importancia.

4. Importance Scaling

Importance Scaling calcula la prioridad de envío por Ghost en el servidor.

4.1 Entradas

GhostImportance combina tres tipos de datos.

Datos Significado
Por conexión Datos específicos de la conexión como la posición del jugador.
Datos singleton Configuración global como el tamaño del tile.
Datos compartidos por chunk Parámetros de escala a nivel de chunk.

4.2 Importancia basada en distancia

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

var gridSingleton = state.EntityManager.CreateSingleton(
    new GhostDistanceData
    {
        TileSize = new int3(50, 50, 256),
        TileCenter = new int3(0, 0, 128),
        TileBorderWidth = new float3(1f, 1f, 1f),
    });

state.EntityManager.AddComponentData(gridSingleton, new GhostImportance
{
    ScaleImportanceFunction = GhostDistanceImportance.ScaleFunctionPointer,
    GhostConnectionComponentType = ComponentType.ReadOnly<GhostConnectionPosition>(),
    GhostImportanceDataType = ComponentType.ReadOnly<GhostDistanceData>(),
    GhostImportancePerChunkDataType = ComponentType.ReadOnly<GhostDistancePartitionShared>(),
});

Cambio disruptivo de la línea 1.8.0: las funciones de escala de GhostDistanceImportance ya no multiplican baseImportance por 1000. El código de importancia personalizada existente que asumía el multiplicador antiguo necesita revisión.

4.3 GhostDistancePartitionShared automático

using Unity.NetCode;

GhostDistancePartitioningSystem.AutomaticallyAddGhostDistancePartitionSharedComponent = true;

4.4 Importance Visualizer

La PlayMode Tool incluye un Importance Visualizer en la línea 1.8.0+. Usarlo para inspeccionar la distribución de importancia en tiempo de ejecución en lugar de deducirla del código.

5. Ghost Relevancy

Ghost Relevancy permite al servidor ocultar o exponer Ghosts específicos por conexión.

Modo Significado
Disabled Predeterminado. Sin filtrado; todos los Ghosts son candidatos.
SetIsRelevant Solo los Ghosts listados se replican a esa conexión.
SetIsIrrelevant Los Ghosts listados se excluyen de esa conexión.

5.1 Casos de uso

Escenario Idea de implementación
Culling por distancia Marcar como irrelevantes los Ghosts lejanos.
Niebla de guerra Ocultar entidades fuera del campo de visión.
Ghosts específicos por cliente Marcar un Ghost como relevante solo para un cliente.
Filtrado por equipo Ocultar datos internos solo del enemigo.

5.2 Ejemplo de relevancia

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
[UpdateInGroup(typeof(GhostSimulationSystemGroup))]
[UpdateBefore(typeof(GhostSendSystem))]
public partial struct RelevancySystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var relevancy = SystemAPI.GetSingletonRW<GhostRelevancy>();

        relevancy.ValueRW.GhostRelevancyMode = GhostRelevancyMode.SetIsRelevant;
        relevancy.ValueRW.GhostRelevancySet.Clear();

        foreach (var (ghostInstance, transform) in
            SystemAPI.Query<RefRO<GhostInstance>, RefRO<LocalTransform>>())
        {
            foreach (var (connId, connPos) in
                SystemAPI.Query<RefRO<NetworkId>, RefRO<GhostConnectionPosition>>())
            {
                float dist = math.distance(
                    transform.ValueRO.Position,
                    connPos.ValueRO.Position);

                if (dist < 100f)
                {
                    relevancy.ValueRW.GhostRelevancySet.TryAdd(
                        new RelevantGhostForConnection(
                            connId.ValueRO.Value,
                            ghostInstance.ValueRO.ghostId),
                        1);
                }
            }
        }
    }
}

Este ejemplo simplificado es O(Ghosts × conexiones). Para worlds grandes, preferir el particionamiento de importancia basado en distancia y reservar la relevancia para reglas que el particionamiento no puede expresar, como la niebla de guerra o la visibilidad por equipos.

5.3 DefaultRelevancyQuery

DefaultRelevancyQuery define los chunks de Ghost que son siempre relevantes para cada conexión, como Ghosts de estado global del juego o estado de red relacionado con la UI.

5.4 Relevancia más importancia

La línea 1.6.1+ soporta combinar Ghost Relevancy e Importance Scaling mediante PrioChunks.isRelevant, lo que permite un camino de relevancia más rápido.

6. Optimización de CPU de predicción

6.1 Prediction Switching

Cambiar Ghosts lejanos o poco importantes de Predicted a Interpolated.

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

float distance = math.distance(playerPos, ghostPos);
if (distance > predictionRange)
{
    switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
    {
        TargetEntity = ghostEntity,
        TransitionDurationSeconds = 0.5f,
    });
}

6.2 Minimizar el acceso de escritura a GhostField

Los jobs con acceso de escritura a componentes GhostField pueden desencadenar comprobaciones de serialización. Usar acceso de solo lectura para campos que no cambian en el bucle de predicción.

6.3 Re-simulación de física

Para casos de alta latencia que requieren 20+ fotogramas de re-simulación predicha:

  • Deshabilitar la física multihilo mediante el singleton PhysicsStep cuando la consolidación en el hilo principal es más barata.
  • Usar MaxPredictionStepBatchSizeRepeatedTick para agrupar ticks de física repetidos.

7. MaxIterateChunks

GhostSendSystemData.MaxIterateChunks limita el número de chunks procesados en un único tick.

Configuración Significado
MaxIterateChunks Limita el coste de CPU de serialización por tick.
MaxSendChunks Limita el ancho de banda al acotar los chunks incluidos en un snapshot.

Estos parámetros resuelven problemas distintos. MaxIterateChunks distribuye el trabajo de CPU; MaxSendChunks restringe el tamaño del snapshot.

8. MaxSendRate

GhostAuthoringComponent.MaxSendRate limita la frecuencia de envío de un Ghost en Hz.

NetworkTickRate = 60, MaxSendRate = 10
→ This Ghost can be included once every 6 network ticks.

Se utiliza para Ghosts de baja frecuencia o baja importancia.

9. UseSingleBaseline

GhostAuthoringComponent.UseSingleBaseline = true utiliza una única línea base para la compresión delta. Esto puede reducir el coste de CPU con un posible incremento del ancho de banda.

10. Lista de verificación de optimización

  • Usar el modo de optimización Static para Ghosts que raramente cambian.
  • Establecer NetworkStreamSnapshotTargetSize para limitar el ancho de banda por conexión.
  • Implementar Importance Scaling basado en distancia.
  • Aplicar Relevancy solo donde la partición no pueda expresar la regla.
  • Cambiar Ghosts lejanos al modo Interpolated.
  • Usar MaxSendRate para Ghosts de baja frecuencia.
  • Evitar acceso de escritura a datos GhostField que no han cambiado.
  • Inspeccionar la prioridad en tiempo de ejecución con el Importance Visualizer.

Síntoma Causa / Solución

Capítulo 26. Integración de Física Netcode y Compensación de Lag

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Netcode for Entities se integra con Unity Physics para simulación de física en red. El comportamiento físico depende de si el Ghost es Interpolated o Predicted.

Es necesario que al menos un Ghost Predicted exista en la escena para que se ejecute la física predicha.

2. Física interpolada vs. predicha

2.1 Ghosts Interpolated en clientes

  • La posición y la rotación provienen de snapshots del servidor.
  • Simulate está desactivado, por lo que el objeto físico se comporta como kinematic.
  • PhysicsVelocity se ignora.

2.2 Ghosts Predicted en clientes

  • La física se ejecuta dentro del bucle de predicción y puede ejecutarse varias veces por frame renderizado.
  • PhysicsSystemGroup se ejecuta dentro de PredictedFixedStepSimulationSystemGroup.
  • Se espera que el cliente produzca el mismo resultado de simulación que el servidor para la física predicha.
Netcode initialization
├─ PhysicsSystemGroup moves under PredictedFixedStepSimulationSystemGroup
└─ FixedStepSimulationSystemGroup moves under PredictedFixedStepSimulationSystemGroup

3. Rendimiento de la física predicha

La física predicha es costosa porque el rollback puede ejecutar la física repetidamente.

3.1 Agrupamiento en el servidor

using Unity.NetCode;

ClientServerTickRate tickRate = default;
tickRate.MaxSimulationStepBatchSize = 4;
tickRate.MaxSimulationStepsPerFrame = 4;

3.2 Agrupamiento en el cliente

using Unity.NetCode;

ClientTickRate clientRate = default;
clientRate.MaxPredictionStepBatchSizeFirstTimeTick = 2;
clientRate.MaxPredictionStepBatchSizeRepeatedTick = 4;

El agrupamiento puede incrementar el riesgo de mala predicción, por lo que debe usarse como ajuste de rendimiento y no como valor predeterminado.

3.3 Cuantización

La cuantización predeterminada de Transform / Velocity para Ghosts físicos es 1000.

Mayor cuantización Menor cuantización
Simulación más precisa. Menor ancho de banda.
Mayor ancho de banda. Mayor pérdida de precisión.

4. Compensación de lag

Debido a que los clientes predicen por delante del servidor, el servidor autoritativo a veces necesita consultar un mundo de colisiones más antiguo que coincida con el tiempo de acción del cliente.

4.1 Habilitar la compensación de lag

Agregar NetCodePhysicsConfig a un SubScene y establecer EnableLagCompensation = true.

4.2 Consultar un mundo de colisiones antiguo

using Unity.NetCode;
using Unity.Physics;

var physicsHistory = SystemAPI.GetSingleton<PhysicsWorldHistorySingleton>();

physicsHistory.GetCollisionWorldFromTick(
    serverTick,
    interpolationDelay,
    ref physicsWorld,
    out var collisionWorld);

// Use collisionWorld for raycasts or other physics queries.

4.3 Clonación profunda de Collider

Parámetro Valor predeterminado Significado
DeepCopyDynamicColliders true Clonación profunda de colliders dinámicos; predeterminado en 6.5.0.
DeepCopyStaticColliders false Clonación profunda de colliders estáticos.

La línea 6.5.0 habilita la clonación profunda de colliders de Ghost dinámicos de forma predeterminada para evitar aserciones de blob asset.

Versión Tamaño del historial
Hasta 1.4.x 16 ticks.
1.5.0+ 32 ticks.
Personalizado GhostSystemConstants.SnapshotHistorySize mediante definición de compilador.

5. Multi Physics World

Se utiliza un mundo de física separado para efectos exclusivos del cliente, como VFX o partículas que no participan en la simulación en red.

5.1 Configuración

  1. Agregar NetCodePhysicsConfig al SubScene.
  2. Establecer Client Non Ghost World en un valor distinto de cero, como 1.
  3. Agregar PhysicsWorldIndex con el mismo valor a los objetos físicos exclusivos del cliente.

5.2 Grupo de sistemas de física personalizado

using Unity.Entities;
using Unity.NetCode;
using Unity.Physics.Systems;

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation)]
[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial class VfxPhysicsGroup : CustomPhysicsSystemGroup
{
    public const int WorldIndex = 1;

    public VfxPhysicsGroup() : base(WorldIndex, shareStaticColliders: true)
    {
    }
}

shareStaticColliders: true permite que el mundo de física personalizado comparta los colliders estáticos con el mundo de física principal.

6. Physics Proxy

Se utiliza un Physics Proxy cuando un Ghost Predicted debe interactuar con una entidad de física exclusiva del cliente.

  • CustomPhysicsProxyAuthoring crea la entidad proxy.
  • CustomPhysicsProxyDriver vincula el proxy al Ghost Predicted.
  • SyncCustomPhysicsProxySystem sincroniza posición y rotación.

7. PredictedFixedStepSimulationSystemGroup

PredictedFixedStepSimulationSystemGroup se actualiza solo cuando se cumplen todas las condiciones requeridas.

Requisito Significado
El singleton NetworkStreamInGame existe La conexión está en juego.
Existe al menos un Ghost Predicted La física predicha tiene algo que simular.
Se ejecuta la tasa de ticks fija La simulación de paso de tiempo fijo está activa.

Antes de que una conexión de red esté en juego o de que los Ghosts Predicted se transmitan, la física predicha no se ejecuta.

7.1 Física antes de la conexión

Si la física debe ejecutarse antes de que exista una conexión de red, se debe deshabilitar la configuración de física predicha para que la física de FixedStepSimulationSystemGroup normal pueda ejecutarse.

using Unity.NetCode;

[UpdateInGroup(typeof(InitializationSystemGroup))]
[CreateAfter(typeof(PredictedPhysicsConfigSystem))]
public partial struct DisablePredictedPhysics : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.World.GetExistingSystemManaged<PredictedPhysicsConfigSystem>()
            .Enabled = false;
    }
}

Capítulo 28. Diseño de Chunk y TypeManager

1. El chunk de 16 KB

Cada Archetype almacena sus entidades en chunks de 16 KB. El chunk contiene:

  1. Una pequeña cabecera (metadatos: ID de archetype, números de versión, máscaras de bits de habilitación, versiones de cambio).
  2. Una región SoA contigua: una columna por tipo de componente, cada columna siendo un array de valores.
  3. Relleno hasta el límite de 16 KB.

La capacidad del chunk — cuántas entidades caben — se deriva de la siguiente fórmula:

capacity ≈ (16 KB − header) / sum(sizeof(component_i))

Un Archetype ajustado con pocos componentes pequeños puede albergar cientos de entidades. Un Archetype amplio con muchos componentes grandes alberga unos pocos. Los chunks con baja ocupación desperdician los bytes que podrían haber contenido más entidades.

2. TypeManager — el registro de componentes

TypeManager registra cada componente, buffer y sistema al iniciar el world y asigna a cada uno un índice de tipo estable. Del registro, el runtime lee:

Campo Para qué se usa
TypeIndex Identificador compacto usado en queries y metadatos de chunk.
SizeInChunk Bytes por entidad para este componente en un chunk.
Alignment Alineación de columna dentro del chunk.
ElementSize Tamaño de elemento del buffer (para IBufferElementData).
HasBlobReferences / HasEntityReferences Habilita la validación durante la serialización y las operaciones de movimiento.

Rara vez se accede directamente a TypeManager. Lo importante es que agregar nuevos tipos de componentes, cambiarles el nombre o moverlos entre ensamblados desencadena una reconstrucción del registro de tipos, que es donde se invierte el tiempo de inicio en proyectos grandes.

En Entities 1.4 se informó que TypeManager.Initialize se volvió ~2× más rápido en proyectos grandes al omitir el escaneo innecesario de ensamblados. En la versión 6.5 esa optimización se hereda.

3. Inspeccionar el diseño de chunk — la ventana Archetypes

Window → Entities → Archetypes lista cada Archetype del world. Para cada fila:

  • Entities — conteo en vivo.
  • Chunks — número de chunks en uso.
  • Entities / Chunk — ocupación promedio.
  • Capacity — máximo teórico por chunk.

Qué vigilar:

Lectura Interpretación
Muchos Archetypes con 1 entidad / chunk Explosión de archetypes por etiquetas demasiado granulares. Consolide.
Un Archetype con un gran número de entidades y alta ocupación Saludable — enemigos, proyectiles, balas.
Entities / Chunk ≪ Capacity Fragmentación — la ocupación es irregular porque las entidades siguen pasando a otros Archetypes (cambio estructural).
Capacity = 1 El Archetype es demasiado amplio (componentes demasiado grandes). Revisar componentes managed/shared sobredimensionados.

4. Diseño de componentes compatibles con Archetype

Heurísticas, en orden aproximado de impacto:

  1. Mantener los structs no administrados sin gestionar. Un único IComponentData managed fuerza el almacenamiento en tabla lateral por entidad y rompe Burst.
  2. No rellenar los structs. Poner float3 antes que float, no después: los huecos de alineación inflan SizeInChunk.
  3. Separar los campos de acceso poco frecuente. Si solo un sistema lee un campo, moverlo a un componente hermano. Las columnas no usadas siguen costando bytes por entidad.
  4. Preferir componentes habilitables en lugar de add/remove por frame. Ver 13_Structural Change & Safety.md — alternar un bit supera remodelar el Archetype.
  5. Limitar los conjuntos de valores de componentes shared. Cada nuevo valor genera un nuevo bucket de chunk.
  6. Los componentes de etiqueta cuestan cero bytes pero añaden un Archetype. No etiquetar cada entidad con un marcador único.

5. Números de versión del chunk

Cada columna de componente rastrea una versión de cambio — un contador de 32 bits que se incrementa cada vez que un sistema escribe mediante RefRW<T>.ValueRW, SetComponentData o ECB SetComponent.

Los sistemas pueden omitir chunks cuya versión no ha cambiado desde la última ejecución:

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>());
query.SetChangedVersionFilter(typeof(Health));

Cuando una query se itera con un filtro de cambio, solo se visitan los chunks actualizados desde la última ejecución del sistema. Así es como patrones como "al cambiar Health, actualizar UI" se vuelven económicos.

Advertencias:

  • El filtro es a nivel de chunk, no de entidad. La escritura de una entidad ensucia el chunk completo.
  • Leer mediante RefRW<T>.ValueRW (aunque no se mute) incrementa la versión.
  • Alternar un componente habilitable también incrementa la versión.

6. Almacenamiento de BlobAsset

BlobAssetReference<T> almacena datos grandes y de solo lectura (mallas, tablas de LOD, campos de navegación) fuera del chunk. El chunk solo almacena un puntero. Esto evita que la capacidad del chunk se reduzca cuando muchas entidades comparten los mismos datos baked.

Consejo práctico: si se tiene un componente dominado por un FixedList o un DynamicBuffer idéntico para todas las entidades de un tipo, conviene almacenar la carga grande en un blob asset referenciado desde un componente ligero.

7. Ejemplos

7.1 Un Archetype amplio que conviene evitar

public struct EnemyState : IComponentData
{
    public float3 Position;
    public quaternion Rotation;
    public float Health;
    public float HealthMax;
    public float Stamina;
    public float StaminaMax;
    public int AIState;
    public float3 LastKnownPlayerPos;
    public float TargetDistance;
    public Entity Target;
    public FixedList512Bytes<float3> PathBuffer;   // 512 bytes per entity!
}

Solo el campo PathBuffer consume 512 bytes por entidad. La capacidad por chunk de 16 KB cae a dos dígitos bajos. Se recomienda dividirlo:

public struct EnemyStats : IComponentData
{
    public float Health, HealthMax, Stamina, StaminaMax;
    public int   AIState;
    public Entity Target;
}

public struct EnemyPathPoint : IBufferElementData
{
    public float3 Value;
}

Ahora el buffer dinámico vive en su propio almacenamiento y la densidad del chunk se recupera.

7.2 Filtro de cambio para actualizaciones de UI económicas

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>(),
                                 ComponentType.ReadWrite<HealthBarLink>());
query.SetChangedVersionFilter(typeof(Health));

state.Dependency = new UpdateHealthBarJob { /* ... */ }
    .ScheduleParallel(query, state.Dependency);

Solo se visitan los chunks donde Health ha cambiado.

Síntoma Causa / Solución

Capacidad de 1 para un Archetype El componente es demasiado grande (capacidad del buffer, blob embebido). Mover los datos voluminosos a un BlobAssetReference<T> o IBufferElementData.

Actúa como la Hierarchy en vivo, pero para entidades en lugar de GameObjects.

  • Columnas de la derecha: número de componentes, número de hijos.

3. Systems

La ventana Systems lista todos los sistemas registrados, agrupados por su SystemGroup, en el orden en que el programador los ejecuta realmente.

Columnas que conviene leer primero

Columna Significado
Name El tipo de sistema.
Namespace Dónde reside (a menudo identifica un paquete frente al código propio).

Queries Número de queries a las que ha accedido el sistema.

Qué buscar

Lectura Interpretación
Un sistema muestra 0 ms pero está en verde Omitido por RequireForUpdate<T> o [RequireMatchingQueriesForUpdate]. Correcto.
Un sistema muestra ms alto de forma constante Trabajo real. Profundizar en el Profiler para obtener un desglose.
Un sistema muestra -- en el conteo de entidades Sin query coincidente (o la query aún no se ha construido).

  • Queries — el conjunto de componentes de cada query (All/Any/None/Disabled/Present/Absent).

  • Type Info — indica si [BurstCompile] está aplicado a la clase y a cada método.

La pestaña Queries ahora muestra los estados de componente Disabled / Present / Absent / None en 1.4+ / 6.x — de modo que las queries de componentes habilitables ya no resultan ambiguas.

4. Ventana Query

Diseñada específicamente para responder preguntas del tipo "¿quién coincide con X?".

Flujo de trabajo:

  1. Agregar componentes All mediante el botón de más.
  2. Agregar Any / None si es necesario.
  3. La ventana lista las entidades coincidentes en el panel izquierdo y los sistemas que ejecutan esta query en el panel derecho.

Casos de uso:

  • "¿Qué sistemas leen Health?" → agregar Health a All; todos los sistemas listados a la derecha la tocan.
  • "¿Por qué esta entidad no coincide con la query de mi sistema?" → pegar la query del sistema y comparar la presencia de componentes con la entidad en la ventana Hierarchy.
  • "¿Se cuelan Prefabs en esta query?" → los Prefabs se distinguen por icono en la lista de resultados (añadido en 1.4+).

La ventana Query también permite guardar una query para reutilizarla durante la sesión.

5. Journaling — el diagnóstico de último recurso

Window → Analysis → Entities Journaling (habilitado desde Project Settings → Entities → Journaling) registra cada cambio estructural, escritura de componente y actualización de sistema en un búfer circular. Vistas de reproducción:

  • Operations — cada cambio registrado con marca de tiempo, sistema de origen y entidad afectada.
  • Entities — línea de tiempo por entidad de quién la tocó y cuándo.

El Journaling tiene un coste en tiempo de ejecución — activarlo solo durante la investigación de un error, deshabilitarlo en el desarrollo normal.

6. Flujo de trabajo de campo — depuración de una actualización ausente

Error habitual: "Mi sistema procesó la entidad pero el componente no cambió". Procedimiento:

  1. Entities Hierarchy → encontrar la entidad, confirmar que tiene el componente.
  2. Inspector → antes de Play, anotar el valor. Presionar Play y observarlo.
  3. Si el valor no cambia: ventana Systems → encontrar el sistema relevante, comprobar Time (ms). Si es 0, está siendo omitido.
  4. Si el tiempo es > 0 pero el valor sigue sin cambiar: ventana Query → pegar la query del sistema y los componentes de la entidad. Verificar que la entidad coincide.
  5. Aún bloqueado: habilitar Journaling durante 1-2 frames. Buscar escrituras en ese componente desde otros sistemas que sobrescriban el resultado.

7. Solución de problemas

Síntoma Causa / Solución
Entities Hierarchy está vacía en modo Play El selector de world apunta al world incorrecto (p. ej., solo Client, sin Server).
Un campo en vivo no se actualiza Se está viendo una entidad en caché después de una recarga de dominio. Volver a seleccionarla.
La ventana Systems indica 0 ms pero el sistema definitivamente se ejecuta El Profiler puede mostrarlo bajo el marcador de un chunk job si el trabajo real está en ScheduleParallel. El OnUpdate del hilo principal es genuinamente 0 en ese caso — consultar los tiempos de los jobs en el Profiler.

Capítulo 30. Profiler · Análisis de Cuellos de Botella

1. Descripción general

El Unity Profiler (Window → Analysis → Profiler) es la herramienta principal para responder preguntas del tipo "¿por qué este frame es lento?" en DOTS. Entities emite suficientes marcadores como para atribuir el tiempo a sistemas, jobs y chunks específicos — una vez que se sabe dónde buscar.

Esta página explica cómo interpretar el perfil, los patrones típicos de cuello de botella en DOTS y cómo actuar sobre ellos.

2. Antes de analizar el rendimiento

  1. Usar Deep Profile con moderación. Es útil para rastrear una ruta managed específica, pero distorsiona los tiempos — nunca confiar en Deep Profile para cifras de rendimiento.
  2. Habilitar "Record in Editor & Standalone" para capturar compilaciones, no solo frames del Editor.
  3. Configurar el Profiler para apuntar al player, no al Editor, al medir el rendimiento de producción. La sobrecarga del Editor es significativa.
  4. Ejecutar el juego varios segundos antes de leer; los primeros frames incluyen trabajo de recarga de dominio y JIT.

3. Interpretación del módulo CPU Usage

Marcadores de nivel superior bajo un frame de DOTS:

PlayerLoop
├── Initialization.*
├── Update.ScriptRunBehaviourUpdate              (MonoBehaviour.Update)
├── SimulationSystemGroup                        ← DOTS simulation lives here
│   ├── SystemA.OnUpdate
│   ├── SystemB.OnUpdate
│   └── EndSimulationEntityCommandBufferSystem.Playback
├── FixedStepSimulationSystemGroup
├── PresentationSystemGroup
├── Render
└── WaitForTargetFPS

Profundizar en un marcador SystemA.OnUpdate para ver sus subllamadas — ScheduleParallel, esperas de finalización de jobs, llamadas a EntityManager.

4. Dónde va realmente el tiempo — formas habituales

4.1 OnUpdate dominante en el hilo principal

Si un único SystemX.OnUpdate acumula 10+ ms, el trabajo se ejecuta en el hilo principal. Verificar:

  • ¿Usa SystemAPI.Query en un bucle ajustado que debería ser IJobEntity.ScheduleParallel?
  • ¿Es un SystemBase forzado al hilo principal por un campo managed?
  • ¿Se llama a EntityManager.AddComponent/RemoveComponent por entidad?

4.2 Tiempos de jobs

Los jobs aparecen en la vista Timeline (o bajo "Workers" en la vista Hierarchy). Buscar:

  • Una lane de job completamente saturada mientras otras están inactivas → el trabajo no se distribuye entre chunks.
  • El hilo principal esperando un job (marcador JobHandle.Complete()) → .Complete() llamado demasiado pronto; moverlo más tarde en el frame.
  • Nombres de función Burst muy largos → es normal; el símbolo es la instanciación genérica completa.

4.3 Puntos calientes de EntityManager

Buscar marcadores con los nombres:

  • EntityManager.SetArchetype / Move Entity — cambio estructural excesivo. Ver 13_Structural Change & Safety.md.
  • EntityCommandBufferSystem.OnUpdate (Playback) — playback de ECB costoso. Normalmente indica demasiados comandos registrados por frame.
  • TypeManager.Initialize — solo al inicio; si aparece por frame, algo está recargando el world.

4.4 Marcadores GC.Alloc

Cualquier GC.Alloc en la simulación es una regresión en DOTS. Fuentes habituales:

  • Un SystemBase que asigna memoria en su OnUpdate.
  • Un foreach sobre una colección managed dentro de un sistema.
  • Boxing — p. ej. object.ToString() en un struct.

Los sistemas Burst deben mostrar cero asignaciones en el módulo Memory del Profiler.

5. Módulos del Profiler que conviene fijar

  • CPU Usage — timeline y jerarquía de marcadores.
  • Memory — heap asignado; vigilar el aumento de GC.Used Memory.
  • Entities (si está disponible) — módulo específico de DOTS que muestra los tiempos por sistema a lo largo del tiempo.

El módulo Entities Profiler llegó con 1.8.0 / 6.x y se alinea con la ventana Systems en una línea de tiempo. Si se observan picos de varios frames, esta es la forma más rápida de atribuirlos.

Para proyectos en red, combinar esto con ../DOTS Workflows/24_Netcode Profiler & Debugging.md para inspeccionar snapshots de Ghost, ticks de predicción y ancho de banda específico de Netcode.

6. Cuellos de botella habituales y soluciones

7. Ejemplo de presupuesto de frame

Suponer un presupuesto de 16,6 ms (60 Hz). Tras descartar Render:

  • SimulationSystemGroup: 6 ms

    • MovementSystem.OnUpdate: 1,4 ms (Burst, paralelo)
    • EndSimulationECBS.Playback: 1,8 ms ← sospechoso
    • todo lo demás: 0,7 ms

1,8 ms de playback de ECB para lo que deberían ser un pequeño número de cambios estructurales es generalmente el mayor punto de palanca. Verificar el conteo de comandos:

  1. Entities Journaling durante 1 frame → conteos de Instantiate/AddComponent/DestroyEntity por sistema.
  2. Si un sistema registra cientos de comandos en cada frame, agruparlos (un comando por chunk, o usar un registrador paralelo con claves de ordenación por índice de chunk).

8. Análisis de rendimiento en build independiente

Para una compilación real:

  1. File → Build Profiles → Development Build + Autoconnect Profiler.
  2. El player se ejecuta y el Profiler captura desde el proceso remoto.
  3. Los resultados son más representativos de lo que verán los jugadores. Las mediciones del Editor pueden ser 2-5× más lentas por razones ajenas al código.

Capítulo 31. Auditoría de Referencias a Objetos Managed

1. Descripción general

Las referencias a objetos Unity managed son una de las formas más fáciles de sacar accidentalmente a un sistema ECS de la ruta rápida. Esta auditoría ayuda a encontrar lugares donde los datos de componentes, sistemas o el código de renderizado contienen referencias a UnityEngine.Object de maneras que bloquean Burst, los jobs o el baking limpio de SubScene.

Conviene usarla después de la migración central a ECS, o siempre que una traza del Profiler muestre trabajo managed donde se esperaba código orientado a datos.

2. Qué buscar

Patrón Por qué importa
class .*: IComponentData Datos de componentes managed; válidos pero generalmente no ideales para rutas calientes.
UnityEngine.Object Marcador amplio de referencias a objetos Unity managed.
GameObject, Transform, Component Las referencias a objetos de escena en tiempo de ejecución normalmente no pertenecen a los datos de simulación.
Mesh, Material, Texture, AudioClip A menudo debería ser UnityObjectRef<T> o datos en caché del lado de presentación.
Resources.Load Búsqueda de assets en tiempo de ejecución; a menudo mejor gestionada por sistemas de baking/contenido.
ScriptableObject Útil como fuente de Authoring, pero los jobs en tiempo de ejecución deben leer valores baked o blobs.

Conviene mantener la auditoría acotada al código orientado a ECS. Los MonoBehaviours de UI o las herramientas de Editor no necesitan seguir patrones DOTS.

3. Reglas de clasificación

Hallazgo Acción probable
Componente managed en una query de simulación caliente Convertir a IComponentData no administrado, BlobAssetReference<T>, o UnityObjectRef<T>.
Componente managed usado solo por presentación Mantenerlo si está aislado de las rutas de simulación Burst/job.
ScriptableObject leído en cada frame Bakear los valores necesarios en un componente o blob asset.
Referencia a Mesh/Material en datos ECS Usar UnityObjectRef<Mesh> / UnityObjectRef<Material> y desreferenciar en el código de presentación.
GameObject/Transform almacenado en datos de componente Reemplazar con datos ECS o mover la referencia a un puente de presentación híbrido.

4. Ejemplo de auditoría

// Suspicious: managed component data used as per-entity visual state.
public class EnemyVisual : IComponentData
{
    public Material Material;
}

Preferir:

using Unity.Entities;
using UnityEngine;

public struct EnemyVisual : IComponentData
{
    public UnityObjectRef<Material> Material;
}

Luego desreferenciar en un sistema de presentación del hilo principal o almacenar en caché el material resuelto en un registro de renderizadores.

5. Indicios en el Profiler

Síntoma en el Profiler Qué revisar

Capítulo 32. Descripción General de la Migración Entities 1.x → 6.5

1. En qué consiste

Actualizar de Entities 1.x a 6.5 implica principalmente tres migraciones de DOTS sobre una transición de distribución de paquetes:

  1. Actualización de distribución del Editor/paquete. Se pasa a Unity 6000.4+ (com.unity.entities 6.4) o 6000.5+ (Entities 6.5), donde Entities es un Core Package integrado en el Editor.
  2. Limpieza de datos ECS. Revisar las referencias a objetos Unity managed en el código orientado a ECS y migrar los datos de componentes de ruta caliente hacia componentes no administrados, valores baked, BlobAssetReference<T> o UnityObjectRef<T>.
  3. Migración de API de ECS. Las APIs legacy marcadas como obsoletas en Entities 1.4 — Entities.ForEach, IAspect — siguen siendo obsoletas en 6.5. Además: ComponentLookup.GetRefRWOptional → TryGetRefRW.

Cada migración se documenta en su propio subdocumento de esta carpeta. Esta página es el mapa — cuándo hacer cada una y en qué orden.

Atención con el nomenclatura: La versión del paquete Entities (1.4 / 6.4 / 6.5) es distinta de la versión del Editor Unity (6000.3 / 6000.4 / 6000.5). Se alinean por convención a partir de 6.4, pero se rastrean en notas de versión separadas. Este documento las trata como dos ejes; la página del Changelog (../Changelog/Entities 1.4 → 6.5 Key Changes.md) hace lo mismo.

2. Orden recomendado

Figure: Entities 1.x to 6.5 Migration Order.

Por qué este orden:

  • Editor primero porque nada más compila contra APIs de 6.x hasta que el Editor las conozca.
  • Package Manager → Core Package a continuación — sin esto, la resolución de paquetes es duplicada o conflictiva.
  • Limpieza de referencias managed antes de refactorizar iteración — mantiene el modelo de datos claro antes de reescribir sistemas y jobs.
  • foreach / IAspect al final — son internos a ECS; no bloquean el resto del código.

3. Qué permanece igual

Sorprendentemente mucho — lo que hace que la migración sea menos intimidante de lo que parece.

Sin cambios Notas
IComponentData, IBufferElementData, ISharedComponentData Las mismas interfaces, la misma semántica.
ISystem / SystemBase Los mismos hooks de ciclo de vida, los mismos atributos.
Superficie de la API de EntityManager CreateEntity, AddComponent, etc. — idénticos.
SystemAPI.Query Introducida en 1.x y sigue siendo la API central de iteración.
IJobEntity, IJobChunk La misma estructura; IJobEntity es el predeterminado recomendado.
EntityCommandBuffer + ECBS Igual.
Burst, Jobs, Collections, Mathematics Igual.
Conceptos de Netcode for Entities (Ghost, RPC, Prediction) La misma arquitectura; el paquete simplemente se distribuye de forma diferente.

  3. Ejecutar las pruebas de regresión de gameplay existentes. Si el rendimiento importa, capturar una traza de referencia del Profiler antes de la migración y compararla.
  4. Verificar guardado/carga. Los handles de Entity son locales al world y no son estables. Guardar IDs propios del juego, claves de contenido o datos de Authoring en su lugar.

  • Un paquete crítico del que se depende (de terceros) aún no es compatible con 6.5.

Capítulo 33. Package Manager → Core Package

1. Qué cambia

En Unity 6000.4+, Entities (y sus paquetes DOTS complementarios) dejan de instalarse a través del Package Manager. Se distribuyen dentro del Editor como Core Packages. Los archivos del proyecto cambian en tres lugares:

Capítulo 34. Referencias a Objetos Managed → UnityObjectRef

1. Descripción general

Al migrar código ECS antiguo, un problema habitual es que las referencias a objetos managed se filtren en los datos de componentes en tiempo de ejecución. En Entities 6.5, se deben mantener los componentes de gameplay sin gestionar donde sea posible y usar UnityObjectRef<T> cuando los datos ECS deban apuntar a un objeto Unity como una malla, material, textura, clip de audio o ScriptableObject de Authoring.

Esta página está acotada al diseño de componentes DOTS. No cubre las APIs de identificador de objetos Unity de ámbito general del motor.

2. Qué buscar

Buscar componentes que almacenen objetos Unity managed directamente:

Patrón Por qué revisarlo
Componentes class que implementan IComponentData Los componentes managed están permitidos pero son más lentos y están limitados al hilo principal.
Campos UnityEngine.Object dentro de datos de componente Generalmente indica un componente managed o un diseño de componente no administrado inválido.
Campos GameObject, Transform, Material, Mesh, Texture, AudioClip Decidir si pertenecen a Authoring, presentación o UnityObjectRef<T>.
Sistemas en tiempo de ejecución que llaman a Resources.Load por entidad Mover la resolución de assets al baking o a un sistema de contenido central.

3. Antes y después

Antes — componente managed

using Unity.Entities;
using UnityEngine;

public class ProjectileVisual : IComponentData
{
    public Mesh Mesh;
    public Material Material;
}

Esto funciona, pero convierte el componente en gestionado. Los componentes gestionados reducen la flexibilidad de Burst/jobs y normalmente no son adecuados para datos de gameplay por entidad.

Después — componente no administrado con UnityObjectRef<T>

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

Bakear las referencias una sola vez:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

4. Patrón de acceso en tiempo de ejecución

Mantener los jobs de simulación puramente orientados a datos. Desreferenciar objetos Unity solo donde las APIs Unity managed están permitidas.

using Unity.Entities;
using UnityEngine;

public partial class ProjectileVisualUploadSystem : SystemBase
{
    protected override void OnUpdate()
    {
        foreach (var visual in SystemAPI.Query<RefRO<ProjectileVisual>>())
        {
            Mesh mesh = visual.ValueRO.Mesh.Value;
            Material material = visual.ValueRO.Material.Value;

            if (mesh == null || material == null)
                continue;

            // Upload to a rendering registry, material property block cache, etc.
        }
    }
}

Para renderizado de alto volumen, evitar la desreferencia por entidad en cada frame. Almacenar en caché los objetos resueltos en un registro del lado de presentación indexado por un valor ECS compacto.

Capítulo 35. Migración de foreach → IJobEntity

1. Estado — obsoleto, aún no eliminado

Las APIs legacy de iteración de componentes están obsoletas en Entities 1.4+ / 6.5, pero aún compilan:

Legacy Estado en 6.5 Reemplazo
Entities.ForEach Obsoleto desde 1.4; advertencias de obsolescencia en cada uso; el changelog de Unity indica que se eliminará en una versión mayor futura SystemAPI.Query<...> (hilo principal) o IJobEntity (jobs)
Job.WithCode Obsoleto junto con ForEach Struct IJob simple, o inline en OnUpdate
IJobForEach (anterior a 1.0) Eliminado mucho antes de 1.4 IJobEntity

Se llega aquí porque se abrió un proyecto 1.x en 6000.5+ y el compilador emite muchas advertencias, o porque se quieren eliminar estas advertencias antes de que una versión mayor futura elimine la API. La refactorización es mecánica una vez que se identifica el patrón.

2. Las dos APIs de reemplazo

  • SystemAPI.Query<T, U, ...> — iteración en el hilo principal dentro de OnUpdate. Simple, sin sobrecarga de jobs, ideal para cargas de trabajo pequeñas o cuando se necesitan APIs managed.
  • IJobEntity — job paralelo generado por código. Reemplazo directo si se está listo para el trabajo multi-hilo, que es donde DOTS marca la diferencia.

Referencia: ../DOTS Workflows/11_IJobEntity · SystemAPI.Query.md.

3. Migración — Entities.ForEach → SystemAPI.Query

3.1 Antes

public partial class HealthRegenSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float dt = Time.DeltaTime;

        Entities
            .WithAll<Alive>()
            .WithNone<Dead>()
            .ForEach((ref Health health, in Regen regen) =>
            {
                health.Value = math.min(health.Value + regen.Rate * dt, health.Max);
            })
            .ScheduleParallel();
    }
}

3.2 Después — SystemAPI.Query (hilo principal)

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>()
                          .WithAll<Alive>()
                          .WithNone<Dead>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRO.Max);
        }
    }
}

Notas sobre los cambios de forma:

  • SystemBase → ISystem (más económico, compatible con Burst). Puede ser necesario mantener SystemBase si el código antiguo capturaba campos managed.
  • ref Health → elemento de tupla RefRW<Health>. Lectura mediante .ValueRO, escritura mediante .ValueRW.
  • in Regen → RefRO<Regen>.
  • Time.DeltaTime → SystemAPI.Time.DeltaTime.
  • Los filtros (WithAll/WithNone) se pasan al constructor de Query.

4. Migración — Entities.ForEach.ScheduleParallel → IJobEntity

Cuando el ForEach se planificaba en paralelo, la forma de job es una migración directa 1:1 que mantiene el paralelismo.

4.1 Después — IJobEntity

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Los filtros se aplican al struct del job como atributos:

[WithAll(typeof(Alive))]
[WithNone(typeof(Dead))]
public partial struct HealthRegenJob : IJobEntity { ... }

O se pasa una query en el momento de planificación:

var q = SystemAPI.QueryBuilder()
    .WithAll<Health, Regen, Alive>()
    .WithNone<Dead>()
    .Build();
new HealthRegenJob { DeltaTime = dt }.ScheduleParallel(q);

5. Capturas y sus reemplazos

El legacy Entities.ForEach permitía capturar variables locales con .WithReadOnly, .WithDisposeOnCompletion, etc. En IJobEntity, las capturas son campos del struct del job — Burst sabe exactamente qué se está pasando.

// BEFORE
var buffer = new NativeArray<int>(count, Allocator.TempJob);
Entities.ForEach((ref Foo f) => { /* use buffer */ })
    .WithDisposeOnCompletion(buffer)
    .ScheduleParallel();

// AFTER
[BurstCompile]
partial struct FooJob : IJobEntity
{
    [ReadOnly] public NativeArray<int> Buffer;

    void Execute(ref Foo f) { /* use Buffer */ }
}

var buffer = new NativeArray<int>(count, Allocator.TempJob);
state.Dependency = new FooJob { Buffer = buffer }.ScheduleParallel(state.Dependency);
buffer.Dispose(state.Dependency);   // chained dispose replaces WithDisposeOnCompletion

Reemplazos de atributos:

  • .WithReadOnly(x) → [ReadOnly] public NativeArray<T> X;
  • .WithDisposeOnCompletion(x) → x.Dispose(handle) después de planificar.
  • .WithoutBurst() → [WithoutBurst] en el struct del job (pero considerar seriamente eliminar la razón por la que fue necesario).
  • .WithEntityAccess() → añadir el parámetro Entity entity a Execute.

6. Cuándo sigue siendo necesario un SystemBase

Si el sistema original tenía un campo managed genuino (Input System, UI Toolkit, servicio externo), no es posible convertirlo mecánicamente a ISystem. Dos opciones:

  1. Mantenerlo como SystemBase y usar SystemAPI.Query dentro de OnUpdate. Un solo hilo.
  2. Dividirlo en un SystemBase que haga de puente con el mundo managed y un ISystem que realice el trabajo Burst — ver ../DOTS Workflows/08_System — ISystem vs SystemBase.md.

7. Validación

Después de migrar un sistema:

  1. Probar el comportamiento con pruebas unitarias. La migración no debe cambiar la lógica; si la salida difiere, la refactorización introdujo un error.
  2. Analizar el rendimiento. El ForEach paralelo y IJobEntity.ScheduleParallel deben estar dentro del 5% de diferencia para trabajo equivalente. Una regresión grande significa que Burst no está actuando — inspeccionar el Burst Inspector.
  3. Eliminar el archivo del sistema antiguo. Resistir la tentación de mantener un fallback #if LEGACY — se deteriora.

Capítulo 36. Obsolescencia de IAspect — Migración Fuera de Aspects

1. Estado — obsoleto, aún no eliminado

IAspect fue marcado como obsoleto en Entities 1.4 y sigue siendo obsoleto en la línea 6.x. Aún compila, aún se ejecuta y aún está listado como un tipo de parámetro válido de IJobEntity.Execute en la referencia de la API 6.5 — pero el compilador emite una advertencia de obsolescencia, y Unity ha indicado que el tipo se eliminará en una versión mayor futura. El nuevo código no debe usarlo; el código existente debe migrarlo a un ritmo progresivo.

IAspect se introdujo para permitir que los sistemas declararan una "vista" sobre un conjunto de componentes y la usaran como un único argumento en Entities.ForEach o IJobEntity. En la práctica, añadió una capa de generación de código y un sistema de tipos paralelo que no se justificó:

  • Escribir un aspect duplicaba la lista de parámetros de componentes.
  • Los depuradores avanzaban por el código wrapper generado.
  • RefRW<T> / RefRO<T> en IJobEntity y SystemAPI.Query logran la misma ergonomía de "pasar componentes como un conjunto" sin el tipo extra.

Dado que la superficie de API aún está presente, el código no se rompe al actualizar el Editor; esta migración consiste en limpiar advertencias y prepararse para la versión 2.0.

2. Refactorización mecánica

2.1 Punto de partida

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public readonly partial struct MovementAspect : IAspect
{
    public readonly RefRW<LocalTransform> Transform;
    public readonly RefRO<Velocity>       Velocity;
    public readonly RefRO<Speed>          Speed;

    public void Step(float dt)
    {
        Transform.ValueRW.Position += Velocity.ValueRO.Direction * Speed.ValueRO.Value * dt;
    }
}

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var m in SystemAPI.Query<MovementAspect>())
            m.Step(dt);
    }
}

2.2 Después — parámetros de componentes directos

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var (transform, velocity, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<Velocity>, RefRO<Speed>>())
        {
            transform.ValueRW.Position += velocity.ValueRO.Direction * speed.ValueRO.Value * dt;
        }
    }
}

Dos cambios:

  1. El tipo aspect desaparece. Sus campos se convierten en los elementos de la tupla de SystemAPI.Query.
  2. El método que estaba en el aspect (Step) se incorpora en línea. Si es demasiado grande para incorporarlo, se puede extraer como un helper estático que tome los mismos parámetros — esto lo mantiene compatible con Burst:
static void Step(ref LocalTransform t, in Velocity v, in Speed s, float dt)
{
    t.Position += v.Direction * s.Value * dt;
}

Para llamarlo:

Step(ref transform.ValueRW, velocity.ValueRO, speed.ValueRO, dt);

3. Aspects en IJobEntity

3.1 Antes

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;
    void Execute(MovementAspect aspect) => aspect.Step(DeltaTime);
}

3.2 Después — componentes como parámetros de Execute

[BurstCompile]
public partial struct MoveJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in Velocity velocity, in Speed speed)
    {
        transform.Position += velocity.Direction * speed.Value * DeltaTime;
    }
}

Nótese que IJobEntity usa parámetros simples ref / in, no RefRW<T> / RefRO<T>. (El generador envuelve los componentes en accesos a arrays de chunk por debajo.)

4. Aspects que envolvían un DynamicBuffer

// BEFORE
public readonly partial struct PathAspect : IAspect
{
    public readonly DynamicBuffer<Waypoint> Path;
    public void Advance(int n) => Path.RemoveRange(0, n);
}

// AFTER — access buffer directly
foreach (var (path, e) in SystemAPI.Query<DynamicBuffer<Waypoint>>().WithEntityAccess())
{
    path.RemoveRange(0, 1);
}

En los jobs, DynamicBuffer<T> es un parámetro válido de Execute — no se necesita nada adicional.

5. Aspects entre entidades

Ocasionalmente un aspect envolvía un ComponentLookup<T> para "necesito ver el componente de otra entidad". Reemplazo:

// BEFORE
public readonly partial struct AITargetAspect : IAspect
{
    public readonly RefRO<Target> Target;
    [ReadOnly] public readonly ComponentLookup<Health> HealthLookup;
    public float TargetHealth =>
        HealthLookup.TryGetComponent(Target.ValueRO.Entity, out var h) ? h.Value : 0f;
}

// AFTER — declare the lookup on the system / job directly
public partial struct AISystem : ISystem
{
    private ComponentLookup<Health> _healthLookup;

    public void OnCreate(ref SystemState state)
    {
        _healthLookup = state.GetComponentLookup<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthLookup.Update(ref state);

        foreach (var (target, e) in SystemAPI
                     .Query<RefRO<Target>>()
                     .WithEntityAccess())
        {
            float hp = _healthLookup.TryGetComponent(target.ValueRO.Entity, out var h)
                ? h.Value : 0f;
            // ...
        }
    }
}

Este patrón también es más ligero que la versión con aspect porque la búsqueda se comparte entre entidades.

6. Cuándo un aspect era complejo

Si el aspect tenía muchos campos, encapsulaba estado o participaba en herencia no trivial — eso es señal de considerar una refactorización genuina, no solo una migración. A menudo el código queda mejor como dos sistemas que pasan datos mediante componentes, en lugar de un sistema con un aspect dios.

9. Conceptos de apoyo

Término Qué es
Burst El compilador nativo de Unity para jobs y structs marcados con [BurstCompile]. Produce código SIMD nativo.
Change version Contador de 32 bits que se incrementa cuando un sistema escribe en una columna de componente de un chunk. Habilita SetChangedVersionFilter para omitir chunks sin cambios.
SoA (Structure of Arrays) Disposición de almacenamiento donde cada campo se convierte en su propio array — eficiente para la caché en iteración escalar. Los chunks usan SoA.
RefRW<T> / RefRO<T> Tipos wrapper para acceder a componentes en SystemAPI.Query con intención de lectura-escritura / solo lectura.
EnabledRefRW<T> / EnabledRefRO<T> Wrapper para leer/escribir el bit de alternancia de un componente habilitable sin desencadenar un cambio estructural.
TypeManager El registro del runtime que asigna a cada componente un índice de tipo estable. Se reconstruye al inicio; los proyectos grandes lo notan.
BlobAssetReference<T> Puntero a un blob de datos grande y de solo lectura almacenado fuera del chunk. Evita que la capacidad del chunk se reduzca.
