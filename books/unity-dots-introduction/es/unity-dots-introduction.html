Parte I. Primeros pasos

Capítulo 1. Configuración del entorno — Unity 6000.5 + Entities 6.5.0

1. Introducción

Desde Unity 6000.4+, Entities es un paquete core: viene dentro del Editor y no hace falta añadirlo desde el Package Manager. Esta guía explica cómo instalar el Editor, crear un proyecto preparado para DOTS y comprobar que Entities está activo.

Entorno de referencia del proyecto: Unity 6000.5.0b2 · Entities 6.5.0 · Collections / Mathematics / Entities Graphics (todos ellos Core Packages).

2. Instalar Unity 6000.5+

  1. Abrir Unity Hub. Si aún no se tiene, instalar primero Unity Hub (https://unity.com/download).
  2. En Hub, ir a Installs → Install Editor.
  3. Elegir cualquier versión 6000.5.x (beta de la serie b o posterior). Entities 6.5 solo existe en 6000.5+.
  4. Módulos: dejar los valores por defecto. ECS no necesita ningún módulo adicional.

No hace falta seleccionar ningún módulo "DOTS": Entities viene integrado en el Editor desde la versión 6000.4.

3. Crear un nuevo proyecto

Desde Hub → Projects → New project:

Campo Valor
Editor Version 6000.5.x
Template Universal 3D (o 3D (Built-in), HDRP — cualquiera sirve)
Project Name cualquier nombre

Hacer clic en Create project. El Editor se abre con una escena vacía.

La antigua plantilla específica para DOTS ya no es necesaria en 6000.4+. Cualquier plantilla sirve como base para un flujo de trabajo con Entities.

4. Verificar que los paquetes core están activos

Abrir Window → Package Manager y cambiar el filtro a In Project. Deben aparecer marcados como "Built-in" los paquetes:

  • Entities
  • Collections
  • Mathematics
  • Entities Graphics

Para comprobarlo desde los archivos, revisar Packages/manifest.json: no debe figurar ninguna entrada com.unity.entities, ya que se distribuye con el Editor.

A continuación, abrir las ventanas de ECS:

  • Window → Entities → Hierarchy — muestra las entidades en tiempo de ejecución.
  • Window → Entities → Systems — muestra los sistemas registrados.
  • Window → Entities → Archetypes — muestra la organización de los chunks.

Si los tres menús existen, Entities está correctamente configurado.

5. Prueba de humo — crear una SubScene

  1. En la Hierarchy, clic derecho → New Sub Scene → Empty Scene…
  2. Nombrarla MainSubScene.unity y guardar.
  3. Dentro de la nueva SubScene, añadir cualquier primitiva (3D Object → Cube).
  4. Guardar la escena. El Cube se hornea como entidad en cuanto la SubScene se cierra (ícono amarillo) o al compilar.
  5. Pulsar Play y abrir Window → Entities → Hierarchy: debe aparecer una entidad Cube. La prueba de humo se supera.

La SubScene es el punto de entrada del authoring: sus GameObjects se convierten en Entities al guardar y al compilar. El resto del tutorial, en 03_Hello DOTS — First Entity.md, parte de aquí.

Síntoma Causa / Solución
El menú Window → Entities no aparece El Editor es anterior a 6000.4, donde Entities todavía no es un paquete core. Actualizar el Editor.
No se encuentra Baker<T> / IComponentData El archivo de definición de ensamblado no incluye Unity.Entities. Añadirlo como referencia o, para una prueba rápida, eliminar el asmdef.
La SubScene sigue siendo un GameObject y nunca se hornea La SubScene debe estar cerrada (ícono amarillo) para que aparezcan las entidades horneadas. Se abre y se cierra con la casilla situada junto a su nombre en la Hierarchy.
La ventana Entities Hierarchy no muestra nada en modo Play Comprobar que la SubScene forma parte de la escena actual y que el desplegable "World" de Window → Entities → Hierarchy apunta a DefaultGameObjectInjectionWorld.

Capítulo 2. Los paquetes core explicados

1. Introducción

Con la llegada del paquete Entities 6.4.0 (que acompaña a Unity Editor 6000.4), los principales paquetes DOTS dejaron el Package Manager y pasaron al propio Editor como paquetes core. El registro de cambios del paquete lo expresa así:

"Package is now a core package embedded in Unity."

El cambio alcanzó a com.unity.entities, com.unity.collections, com.unity.mathematics, com.unity.entities.graphics y, en su versión 6.5 equivalente, a Netcode for Entities. Modifica cómo se instalan, se versionan y se gestionan, pero se trata de un cambio de distribución, no de API.

Esta página explica:

  • Qué paquetes son ahora paquetes core (y cuáles no).
  • De dónde salen sus números de versión.
  • Qué cambia en manifest.json y en packages-lock.json.
  • Por qué se hizo este cambio.

2. ¿Qué es un paquete core?

Un paquete core es un paquete que viene dentro del Editor. No se añade desde el Package Manager, su versión no se puede cambiar al margen del Editor y su ritmo de publicación es el del propio Editor.

Característica Paquete de Package Manager Paquete core
Paso de instalación Añadir mediante UPM Ninguno — presente con el Editor
Versión elegida por El usuario, en manifest.json Unity, por versión del Editor
Cadencia de versiones Independiente de UPM Versiones del Editor
Aparece en manifest.json Sí No
Aparece en packages-lock.json Sí Sí, marcado como builtin
Se muestra en Package Manager Bajo "In Project" Bajo "In Project" etiquetado como Built-in

Nota: Job System y Burst nunca fueron paquetes de UPM; forman parte del motor. El cambio de la 6000.4 afecta en concreto a los paquetes DOTS que se apoyan en ellos.

4. Números de versión tras la transición

Las versiones de los paquetes core se alinean con el Editor:

Editor Entities core
Unity 6000.4 Entities 6.4
Unity 6000.5 Entities 6.5

La antigua línea 1.x sigue disponible para los proyectos que todavía no pueden migrar a 6000.4+. Ambas líneas son independientes: para pasar a la 6.x basta con actualizar el Editor.

Las notas de transición de Entities están en Changelog/Entities 1.4 → 6.5 Key Changes.md y los cambios propios de Netcode en Changelog/Netcode for Entities 1.4 → 6.5 Key Changes.md.

5. Efecto en los archivos del proyecto

manifest.json

Ya no hacen falta las entradas com.unity.entities, com.unity.collections, com.unity.mathematics ni com.unity.entities.graphics. Al actualizar un proyecto 1.x conviene eliminarlas; ver Migration/02_Package Manager → Core Package.md.

packages-lock.json

Los paquetes core aparecen con "source": "builtin" en lugar de una URL de versión.

Definiciones de ensamblado (.asmdef)

Los archivos .asmdef siguen referenciando los mismos nombres de ensamblado (por ejemplo, Unity.Entities o Unity.Collections). Aquí no cambia nada: lo único que cambió es la capa de distribución del paquete.

6. ¿Por qué el cambio?

El objetivo, según Unity, es que el equipo de ECS publique novedades de forma más rápida y gradual al ligar las actualizaciones de DOTS a la versión del Editor en vez de a un calendario de paquetes aparte. Además, permite que el motor dependa internamente de ECS, algo incómodo mientras los paquetes eran opcionales.

Capítulo 3. Hola DOTS — Primera Entity

1. Qué se va a construir

Un cubo que gira sobre su eje Y, movido por un sistema ECS. En una sola sesión se recorren todos los conceptos clave: SubScene, authoring → baker, IComponentData, ISystem y las ventanas Entities Hierarchy / Systems. Al final se sustituye el sistema del hilo principal por un IJobEntity para ver cómo entra en juego el paralelismo.

Requisito previo: [01_Environment Setup.md](01_Environment Setup (Unity 6000.5 + Entities 6.5.0).md) — un proyecto Unity 6000.5 con una SubScene comprobada y el menú Entities.

2. Crear la SubScene

  1. Abrir la escena vacía.
  2. En la Hierarchy, clic derecho → New Sub Scene → Empty Scene…
  3. Guardarla como HelloDots.unity dentro de Assets/Scenes/.
  4. Dentro de la nueva SubScene, clic derecho → 3D Object → Cube. Renombrarlo como RotatingCube.
  5. Guardar la escena (Ctrl/⌘ + S) y cerrar la SubScene con la casilla situada junto a su nombre en la Hierarchy: el ícono se pone amarillo, señal de que su contenido se está horneando como entidades.

A partir de aquí, cualquier cambio en los componentes de RotatingCube provoca un rehorneado automático al volver a abrir y cerrar la SubScene.

3. Definir el componente — RotationSpeed

Crear Assets/Scripts/RotationSpeed.cs:

using Unity.Entities;

public struct RotationSpeed : IComponentData
{
    public float RadiansPerSecond;
}

Notas:

  • IComponentData sobre un struct no administrado es la opción por defecto: reside directamente en los chunks y es compatible con Burst.
  • Conviene que los campos sean blittables (tipos primitivos, tipos de Unity.Mathematics u otros structs no administrados).

4. Definir el authoring + baker

Crear Assets/Scripts/RotationSpeedAuthoring.cs:

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class RotationSpeedAuthoring : MonoBehaviour
{
    public float DegreesPerSecond = 90f;

    class Baker : Baker<RotationSpeedAuthoring>
    {
        public override void Bake(RotationSpeedAuthoring authoring)
        {
            // Dynamic — this entity will have its transform written each frame.
            var entity = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(entity, new RotationSpeed
            {
                RadiansPerSecond = math.radians(authoring.DegreesPerSecond)
            });
        }
    }
}

Adjuntar el authoring al cubo

  1. Seleccionar RotatingCube en la SubScene.
  2. Inspector → Add Component → Rotation Speed Authoring.
  3. Dejar DegreesPerSecond = 90.

En cuanto la SubScene queda cerrada (ícono amarillo), el baker se ejecuta y el cubo pasa a ser una entidad con un componente RotationSpeed.

Verificar el horneado

  • Window → Entities → Hierarchy — RotatingCube debe figurar en la lista.
  • Al seleccionarlo, el Inspector de la derecha muestra sus componentes; buscar RotationSpeed { RadiansPerSecond: ~1.5708 } (π/2).

5. Escribir el sistema — RotationSystem

Crear Assets/Scripts/RotationSystem.cs:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<RotationSpeed>>())
        {
            transform.ValueRW = transform.ValueRO.RotateY(speed.ValueRO.RadiansPerSecond * dt);
        }
    }
}

Notas:

  • partial struct + ISystem es la forma de sistema no administrado. Se puede compilar con Burst y se ejecuta sin sobrecarga del GC.
  • SystemAPI.Query<RefRW<A>, RefRO<B>>() es la API de iteración moderna, que sustituye al obsoleto Entities.ForEach.
  • LocalTransform.RotateY(radians) devuelve una copia ya rotada, que se reasigna con transform.ValueRW = ….

Pulsar Play

El cubo gira. En Window → Entities → Systems aparece RotationSystem dentro del grupo de simulación, con su tiempo por fotograma.

6. Inspeccionar lo que se ha construido

Ventanas útiles una vez en modo Play:

Ventana Qué muestra
Entities Hierarchy Lista en vivo de entidades. Al seleccionar una entidad se ven su chunk y sus componentes.
Systems Todos los sistemas en ejecución, agrupados por su SystemGroup. Aquí se ve el tiempo de cada sistema.

Conviene confirmar:

  1. La entidad RotatingCube existe y tiene LocalTransform y RotationSpeed.
  2. RotationSystem aparece dentro de SimulationSystemGroup y acumula tiempo de fotograma.

7. Cambiar a IJobEntity (paralelismo)

Sustituir RotationSystem.cs por una versión basada en jobs para ver cómo encaja el paralelismo:

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct RotateJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref LocalTransform transform, in RotationSpeed speed)
    {
        transform = transform.RotateY(speed.RadiansPerSecond * DeltaTime);
    }
}

[BurstCompile]
public partial struct RotationSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new RotateJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Notas:

  • IJobEntity.Execute(ref A, in B) se transforma, mediante generación de código, en un job a nivel de chunk. Los parámetros son los componentes con los que se trabaja.
  • .ScheduleParallel() reparte los chunks entre los hilos de trabajo. Con un solo cubo no se nota ninguna mejora, pero el código escala a miles de entidades sin tocar nada.
  • La dependencia del sistema se resuelve sola: state.Dependency se combina de forma automática al llamar a ScheduleParallel() desde ISystem.OnUpdate.

Parte II. Flujos de trabajo DOTS

Capítulo 4. Patrón Baker & SubScene

1. Introducción

Las entidades no se crean directamente en la Hierarchy. Lo que se hace es crear GameObjects dentro de una SubScene, y un Baker los convierte en entidades al cerrar o compilar la SubScene.

Figure: Baker and SubScene Pipeline.

2. Ciclo de vida de la SubScene

Una SubScene tiene tres estados visibles:

Ícono en Hierarchy Estado Qué significa
⬜ (casilla blanca) Abierta Sus GameObjects son editables; aún son GameObjects y no hay horneado.
🟨 (casilla amarilla) Cerrada Los Bakers ya se han ejecutado; el contenido solo existe como entidades.
⏳ Horneando Horneado incremental en progreso (breve).

Crear una SubScene

  • Hierarchy → clic derecho → New Sub Scene → Empty Scene… — guarda un asset .unity y añade un componente SubScene al GameObject padre.
  • Se abre y se cierra con la casilla situada junto al nombre de la SubScene. Al cerrarla se dispara un horneado.

Horneado incremental

Cuando la SubScene está abierta y se edita un GameObject, solo se reejecutan los Bakers afectados por el authoring modificado. Por eso conviene evitar operaciones costosas dentro de Bake().

3. API de Baker<T>

Un Baker traduce un MonoBehaviour de authoring a una o varias entidades con sus componentes.

public class EnemyAuthoring : MonoBehaviour
{
    public float MaxHealth = 100f;
    public GameObject ProjectilePrefab;

    class Baker : Baker<EnemyAuthoring>
    {
        public override void Bake(EnemyAuthoring authoring)
        {
            // Primary entity for this GameObject.
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Health { Value = authoring.MaxHealth });

            // Bake a GameObject reference into an Entity reference.
            AddComponent(self, new ProjectilePrefabRef
            {
                Value = GetEntity(authoring.ProjectilePrefab, TransformUsageFlags.Dynamic)
            });
        }
    }
}

APIs principales:

API Propósito
GetEntity(TransformUsageFlags) Entity del GameObject de authoring actual.
GetEntity(Object, TransformUsageFlags) Entity que corresponde a otro GameObject o prefab (registra una entidad prefab).
CreateAdditionalEntity(TransformUsageFlags) Una segunda entidad vinculada al mismo authoring (p. ej., subpartes).
AddComponent<T>(entity, value) Asigna datos de componente.
AddBuffer<T>(entity) / SetBuffer<T> Asigna un DynamicBuffer<T>.
AddSharedComponent<T> Asigna un ISharedComponentData.
DependsOn(asset) Declara una dependencia de un asset para que el Baker se reejecute cuando ese asset cambie.

4. TransformUsageFlags

El horneado solo añade LocalTransform, LocalToWorld o Parent cuando se piden explícitamente con TransformUsageFlags. Los componentes de transformación que no se usan agrandan los chunks sin motivo, así que conviene elegir el flag más restrictivo que aún funcione.

Flag Añade Uso típico
None Nada Entidades de datos puros (configuración, singletons, solo lógica).
ManualOverride Nada; las transformaciones las gestiona el propio código Disposiciones de transformación personalizadas.
Renderable LocalToWorld Malla estática que nunca se mueve.
Dynamic LocalTransform, LocalToWorld (+ Parent si tiene padre) Todo lo que se mueve o lo modifica un system.
WorldSpace LocalTransform, LocalToWorld; impide el parentado Entidades que ignoran la jerarquía.
NonUniformScale Añade PostTransformMatrix Escala no uniforme (poco común en DOTS).

Dynamic es el valor por defecto más seguro para todo lo que un system vaya a mover o girar.

5. Baking systems (avanzado)

Un Baker produce componentes a partir de un solo authoring. Un Baking System, en cambio, se ejecuta después de todos los Bakers y ve el mundo horneado completo: resulta útil para el posprocesado entre entidades (resolver referencias o calcular datos derivados).

[WorldSystemFilter(WorldSystemFilterFlags.BakingSystem)]
public partial struct EnemyWaveLinkSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Runs during bake only. Query entities tagged earlier by Bakers
        // and wire them together.
    }
}

Conviene recurrir a esto cuando un Baker no conoce todo lo que necesita (por ejemplo, recuentos o referencias cruzadas). Debe ser determinista: el horneado tiene que dar el mismo resultado en cada ejecución.

6. Ejemplo — authoring de un spawner de proyectiles

Authoring:

using Unity.Entities;
using UnityEngine;

public class ProjectileSpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float Interval = 0.5f;

    class Baker : Baker<ProjectileSpawnerAuthoring>
    {
        public override void Bake(ProjectileSpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new ProjectileSpawner
            {
                Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                Interval = authoring.Interval,
                TimeLeft = authoring.Interval
            });
        }
    }
}

Component:

using Unity.Entities;

public struct ProjectileSpawner : IComponentData
{
    public Entity Prefab;
    public float Interval;
    public float TimeLeft;
}

Un system en tiempo de ejecución puede crear instancias de Prefab a intervalos regulares. Véase 02_Spawner Example.md.

7. Solución de problemas

Síntoma Causa / Solución
La entidad aparece pero le falta un componente El Baker no se ejecutó (el authoring se añadió después de cerrar la SubScene). Desmarcar y volver a marcar la SubScene para rehornear.
GetEntity(prefab, ...) devuelve Entity.Null El GameObject referenciado no es un asset de prefab ni un objeto de escena accesible desde el Baker. Comprobar que sea un prefab o que esté en la misma SubScene.
El Baker se ejecuta en cada fotograma en el Editor Bake() contiene lógica no determinista (por ejemplo, Random o Time.time). Los Bakers deben ser funciones puras del authoring.
La entidad tiene LocalToWorld pero no LocalTransform Se usó TransformUsageFlags.Renderable. Cambiar a Dynamic si la entidad se va a mover.
Los cambios en un campo de authoring no provocan rehorneado El Baker no lee ese campo, o no llamó a DependsOn para un asset externo.
El horneado es lento Hay operaciones pesadas dentro de Bake(). Trasladar la lógica entre entidades a un Baking System o a un inicializador de una sola ejecución en runtime.

Capítulo 5. Ejemplo de Spawner

1. Descripción general

Un spawner es una sola entidad que crea otras entidades, ya sea al inicio, a intervalos regulares o en respuesta a un evento. Esta sección lo desarrolla de principio a fin y abarca:

  • Crear el authoring del spawner en una SubScene.
  • Guardar una referencia a una entidad prefab (horneada a partir de un prefab de GameObject).
  • Crear instancias en tiempo de ejecución mediante un EntityCommandBuffer, para que los cambios estructurales se apliquen de forma segura.

Leer antes 01_Baker Pattern & SubScene.md facilitará seguir esta sección.

2. Modelo de datos

using Unity.Entities;
using Unity.Mathematics;

public struct Spawner : IComponentData
{
    public Entity  Prefab;       // baked prefab entity
    public float3  SpawnPoint;
    public float   Interval;
    public float   TimeLeft;
    public int     RemainingCount;  // stop when 0
}

Conviene resaltar dos decisiones de diseño:

  • Prefab es una Entity, no un GameObject. Los Bakers convierten el prefab de GameObject en una entidad prefab; en tiempo de ejecución solo hay entidades.
  • TimeLeft y RemainingCount residen en la entidad spawner para que el system no tenga estado propio.

3. Authoring + Baker

using Unity.Entities;
using Unity.Mathematics;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
    public float      Interval = 0.5f;
    public int        Count    = 100;

    class Baker : Baker<SpawnerAuthoring>
    {
        public override void Bake(SpawnerAuthoring authoring)
        {
            var self = GetEntity(TransformUsageFlags.Dynamic);

            AddComponent(self, new Spawner
            {
                Prefab        = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic),
                SpawnPoint    = authoring.transform.position,
                Interval      = authoring.Interval,
                TimeLeft      = 0f,            // fires immediately on first update
                RemainingCount = authoring.Count
            });
        }
    }
}

Colocar SpawnerAuthoring en un GameObject vacío dentro de la SubScene, asignar un Prefab y fijar un número de instancias. Al cerrar la SubScene, el spawner ya es una entidad.

4. System en tiempo de ejecución

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnerSystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<Spawner>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Use the EndSimulation ECB so structural changes flush at a safe point.
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged);

        foreach (var (spawnerRW, spawnerEntity) in
                 SystemAPI.Query<RefRW<Spawner>>().WithEntityAccess())
        {
            ref var spawner = ref spawnerRW.ValueRW;

            if (spawner.RemainingCount <= 0)
            {
                ecb.DestroyEntity(spawnerEntity);   // done — remove the spawner
                continue;
            }

            spawner.TimeLeft -= dt;
            if (spawner.TimeLeft > 0f) continue;

            spawner.TimeLeft = spawner.Interval;
            spawner.RemainingCount--;

            var instance = ecb.Instantiate(spawner.Prefab);
            ecb.SetComponent(instance, LocalTransform.FromPosition(spawner.SpawnPoint));
        }
    }
}

Lo importante:

  • RequireForUpdate<Spawner>() hace que el system se omita por completo mientras no exista ningún spawner.
  • El singleton EndSimulationEntityCommandBufferSystem ofrece un ECB compartido que se vuelca al final del fotograma; es más seguro que modificar el estado con EntityManager dentro del bucle.
  • ecb.Instantiate(prefab) crea una entidad diferida, y ecb.SetComponent(instance, …) programa la escritura de sus componentes. El handle instance solo es válido durante la reproducción de este ECB. Véase 14_EntityCommandBuffer · Deferred Entity.md.

5. Escalado — spawner paralelo

Si hace falta generar miles de entidades por fotograma desde muchos spawners, conviene pasar a un IJobEntity con un escritor paralelo de ECB:

[BurstCompile]
public partial struct SpawnJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, entity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

Y en el system:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

new SpawnJob { DeltaTime = SystemAPI.Time.DeltaTime, ECB = ecb }
    .ScheduleParallel();

La clave de ordenación chunkIndex es lo que garantiza que la reproducción paralela sea determinista. Véase 15_ParallelWriter · Deterministic Playback.md.

6. Solución de problemas

Síntoma Causa / Solución
No aparece nada La entidad Spawner no existe. Abrir la SubScene, comprobar que SpawnerAuthoring esté adjunto y que la SubScene esté cerrada (ícono amarillo).
El prefab aparece en el origen del mundo El ECB nunca asignó LocalTransform. Añadir ecb.SetComponent(instance, LocalTransform.FromPosition(...)) o comprobar que el prefab ya tenga una transformación con la posición deseada.
El prefab es visible pero no se mueve El prefab se horneó con TransformUsageFlags.Renderable en lugar de Dynamic.
SystemAPI.GetSingleton<...ECBS.Singleton>() lanza una excepción El ECB system no está registrado; suele pasar en worlds muy recortados. Como alternativa, usar state.EntityManager con un ECB manual, o registrar el ECB system en el bootstrap.
El spawner nunca se detiene RemainingCount se decrementó sobre una copia en caché de Spawner en lugar de sobre spawnerRW.ValueRW. Escribir siempre a través de RefRW.ValueRW.
ecb.SetComponent sobre una instancia diferida no surte efecto Se usaron ECB distintos para Instantiate y SetComponent. Ambas llamadas deben usar la misma instancia de ECB.

Capítulo 6. Conceptos fundamentales de ECS

1. Los cinco sustantivos

El ECS de Entities se resume en cinco sustantivos:

Figure: ECS Core Nouns.

Asimilar este diagrama es buena parte de lo que hace falta para leer código DOTS ya escrito.

2. Entity

Una entity no es más que un handle de 64 bits: struct Entity { int Index; int Version; }. No tiene datos ni más métodos que los de identidad; todos los datos residen en los componentes.

Entity e = state.EntityManager.CreateEntity();   // fresh, no components
state.EntityManager.AddComponent<Health>(e);     // now it has data
state.EntityManager.DestroyEntity(e);            // handle becomes stale

El campo Version invalida las referencias antiguas: si se destruye una entidad y se vuelve a crear en el mismo Index, el valor antiguo de Entity deja de coincidir.

3. Component

Un component es un contenedor de datos sin más. No incluye comportamiento: el código vive en los systems.

public struct Health : IComponentData
{
    public float Value;
    public float Max;
}

Hay varias variantes de componente: no gestionados, gestionados, de buffer, compartidos, de cleanup, de tag, de chunk y habilitables. Véase 05_Component Types.md.

4. Archetype

Un archetype es un conjunto único de tipos de componente que comparten algunas entidades.

  • Una entidad con { LocalTransform, Health, Enemy } pertenece al archetype A.
  • Una entidad con { LocalTransform, Health, Enemy, Burning } pertenece al archetype B, uno distinto, porque añadir Burning cambia el conjunto de componentes.

Pasar de un archetype a otro (añadir o quitar un componente) es un cambio estructural y traslada físicamente la entidad a otro chunk. Véase 13_Structural Change & Safety.md.

5. Chunk

Las entidades del mismo archetype se empaquetan en chunks de 16 KB. Dentro de un chunk, cada tipo de componente forma un array contiguo, algo excelente para la localidad de caché.

Dato Implicación
Los chunks son de 16 KB Cada componente ocupa una parte; cuantos más componentes tenga el archetype, o más grandes sean, menos entidades caben por chunk.
Los datos se organizan por columnas (SoA), una por componente IJobChunk puede recorrer un array de componentes cada vez y vectorizarlo con Burst.
Añadir o quitar un componente mueve la entidad Es un cambio estructural. Los iteradores de query dan por hecho que el archetype no cambia durante la iteración.
Los chunks registran las versiones de cambio de cada componente Los systems pueden saltarse los chunks que no han cambiado desde la última ejecución (EntityQuery.SetChangedVersionFilter).

La organización de los chunks en vivo se puede inspeccionar en Window → Entities → Archetypes.

6. System

Un system es una unidad de lógica que se ejecuta en cada fotograma, normalmente sobre una query de entidades.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var health in SystemAPI.Query<RefRW<Health>>())
        {
            health.ValueRW.Value = math.min(health.ValueRW.Value + dt, health.ValueRW.Max);
        }
    }
}

Los systems se registran de forma automática y residen dentro de un SystemGroup. Véanse 08_System — ISystem vs SystemBase.md y 09_System Group & Update Order.md.

7. World

Un World es un contenedor que posee:

  • Un EntityManager (crea, destruye, consulta entidades).
  • Un conjunto de systems y sus SystemGroups.
  • Los propios chunks (a través del EntityManager).

El world de tiempo de ejecución por defecto es DefaultGameObjectInjectionWorld. La mayoría de los juegos usan un único world en producción; las configuraciones con varios worlds aparecen en Netcode (servidor y cliente) o en las herramientas del Editor. La división de worlds en Netcode se explica en 16_Netcode Client-Server World & Bootstrap.md.

World world = World.DefaultGameObjectInjectionWorld;
EntityManager em = world.EntityManager;

Entity e = em.CreateEntity(typeof(Health));
em.SetComponentData(e, new Health { Value = 100, Max = 100 });

8. EntityManager

El EntityManager es la API del hilo principal para los cambios estructurales: crear, destruir, añadir o quitar componentes, crear instancias de entidades prefab, copiar worlds, etc. Los cambios estructurales hechos con EntityManager son inmediatos y pueden invalidar cualquier iterador de query en curso o referencia de job activa.

Para cambios estructurales diferidos desde jobs o en mitad de una iteración, se usa un EntityCommandBuffer; véase 14_EntityCommandBuffer · Deferred Entity.md.

9. Todo junto

World
└── EntityManager
    └── Archetype { LocalTransform, Health, Enemy }
        └── Chunk  (up to ~N entities)
            └── Entity { Index, Version }
                ├── LocalTransform
                ├── Health
                └── Enemy (tag)

Un system que consulta { LocalTransform, Health } recorre todos los chunks cuyo archetype contiene ambos componentes. Eso incluye el archetype anterior y cualquier otro que, además de Enemy, tenga componentes adicionales.

10. Solución de problemas

Síntoma Causa / Solución
Las búsquedas de Entity devuelven datos obsoletos La entidad se destruyó y se recreó, así que el Version ya no coincide. Volver a resolver la entidad.
La query no encuentra nada aunque la entidad tiene el componente El componente se añadió en un world distinto, o el componente es habilitable y actualmente está desactivado.
El número de Archetype se dispara Hay demasiadas combinaciones únicas de componentes, a menudo por tags por entidad de alta cardinalidad. Consolidar con componentes compartidos o flags habilitables.
El chunk contiene solo 1 o 2 entidades El archetype es demasiado grande. Quitar los componentes que la entidad no necesita, o repartirla en varias entidades o archetypes.
InvalidOperationException: … invalidated … durante la iteración Se produjo un cambio estructural en mitad de la query. Usar un ECB o agrupar los cambios después del bucle.

Capítulo 7. Referencias de Entity — Entity · EntityPrefabReference · UnityObjectRef

1. Descripción general

El código DOTS usa un conjunto reducido de tipos de referencia. Es fácil confundirlos, porque todos apuntan a "algo", pero corresponden a etapas distintas del pipeline que va del authoring al runtime.

Tipo Espacio de nombres Qué referencia Uso típico
Entity Unity.Entities Una entidad dentro de un World ECS. Relaciones en tiempo de ejecución y campos de componente.
EntityPrefabReference Unity.Entities.Serialization Un prefab de entidad ya horneado que se puede cargar más adelante. Flujos de trabajo de streaming y contenido.
UnityObjectRef<T> Unity.Entities Un asset u objeto UnityEngine.Object mediante un wrapper no gestionado. Referenciar mallas, materiales, texturas, clips de audio y assets de authoring desde datos ECS.

Esta sección trata solo los tipos de referencia de DOTS / Entities. Las API de identificación de UnityEngine.Object que abarcan todo el motor quedan fuera del alcance de este manual.

2. Entity

Entity es el handle de ECS en tiempo de ejecución. Es un valor no gestionado que identifica una fila de datos de componentes dentro de un World.

using Unity.Entities;

public struct Target : IComponentData
{
    public Entity Value;
}

Reglas importantes:

  • Un valor de Entity solo tiene sentido dentro del World que lo creó.
  • Destruir una entidad invalida su handle anterior.
  • Los índices de entidad se pueden reutilizar; antes de desreferenciar un handle guardado que no se acaba de consultar, comprobarlo con EntityManager.Exists(entity).
  • No conviene guardar valores de Entity en disco ni pasarlos entre worlds independientes como si fueran IDs estables.
if (state.EntityManager.Exists(target.Value))
{
    var transform = state.EntityManager.GetComponentData<LocalTransform>(target.Value);
}

3. Referencias de entidad en el baking

Cuando un Baker convierte objetos de authoring en datos ECS, GetEntity transforma un objeto de authoring o un prefab referenciado en una referencia de Entity.

using Unity.Entities;
using UnityEngine;

public class SpawnerAuthoring : MonoBehaviour
{
    public GameObject Prefab;
}

public struct Spawner : IComponentData
{
    public Entity Prefab;
}

public class SpawnerBaker : Baker<SpawnerAuthoring>
{
    public override void Bake(SpawnerAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new Spawner
        {
            Prefab = GetEntity(authoring.Prefab, TransformUsageFlags.Dynamic)
        });
    }
}

El resultado es una referencia ECS de tiempo de ejecución, no una referencia gestionada a un GameObject.

4. EntityPrefabReference

EntityPrefabReference sirve para contenido de prefabs que debe cargarse a través del pipeline de contenido de Entities, en lugar de incrustarse directamente como referencia de Entity en la escena actual.

using Unity.Entities;
using Unity.Entities.Serialization;
using UnityEngine;

public class ProjectileLibraryAuthoring : MonoBehaviour
{
    public GameObject ProjectilePrefab;
}

public struct ProjectileLibrary : IComponentData
{
    public EntityPrefabReference ProjectilePrefab;
}

public class ProjectileLibraryBaker : Baker<ProjectileLibraryAuthoring>
{
    public override void Bake(ProjectileLibraryAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.None);
        AddComponent(entity, new ProjectileLibrary
        {
            ProjectilePrefab = new EntityPrefabReference(authoring.ProjectilePrefab)
        });
    }
}

Conviene usar EntityPrefabReference cuando el prefab forma parte de un flujo de gestión de contenido o de streaming. Si el prefab está horneado en la misma SubScene y se carga junto con la escena, basta con una referencia de prefab Entity simple.

5. UnityObjectRef<T>

UnityObjectRef<T> permite que los datos ECS no gestionados referencien un UnityEngine.Object sin que el componente pase a ser gestionado.

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

Hornearlo a partir de datos de authoring:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

Desreferenciar .Value únicamente en el hilo principal o en código de presentación gestionado:

Mesh mesh = visual.Mesh.Value;
Material material = visual.Material.Value;

Reglas:

  • El wrapper almacenado no es gestionado y puede residir dentro de un IComponentData.
  • El objeto de destino sigue siendo un objeto Unity gestionado; .Value no es una operación apta para un Burst job.
  • El objeto referenciado debe seguir cargado en el momento de desreferenciarlo.

6. Elegir la referencia correcta

Necesidad Usar
Apuntar desde una entidad a otra dentro del mismo world Entity
Guardar un prefab horneado en la misma SubScene Referencia de prefab Entity desde GetEntity(authoring.Prefab, ...)
Guardar un prefab para streaming de contenido o carga diferida EntityPrefabReference
Guardar en datos ECS no gestionados una referencia a una malla, material, textura, clip de audio o ScriptableObject UnityObjectRef<T>
Conservar una entidad entre sesiones Un ID estable propio del juego, no un Entity en crudo

7. Solución de problemas

Síntoma Causa / Solución
La Entity guardada ya no existe La entidad se destruyó o pertenece a otro world. Volver a resolverla o comprobarla con EntityManager.Exists.
La referencia de prefab es Entity.Null tras el baking El Baker no llamó a GetEntity sobre el prefab, o el prefab no se incluyó en un flujo de baked scene o de contenido.
UnityObjectRef<T>.Value devuelve null El objeto Unity referenciado no está cargado o se destruyó. Proteger la desreferencia y volver a cargar o resolver el asset.
Un Burst job no puede usar una referencia a una malla o un material El almacenamiento de UnityObjectRef<T> no es gestionado, pero .Value accede a objetos Unity gestionados. Trasladar la desreferencia al código de presentación del hilo principal.
El archivo de guardado almacena valores de Entity en crudo Los handles de entidad son locales al world y no son estables. Guardar en su lugar un ID propio del juego o una clave de contenido.

Capítulo 8. Tipos de Component

1. Descripción general

En ECS, un "componente" es cualquier contenedor de datos asociado a una entidad. Entities ofrece varias variantes, con distintos perfiles de almacenamiento, vida útil y coste. Elegir la adecuada es la mitad del trabajo al diseñar un system ECS.

Mapa rápido:

Interfaz / Forma Almacenamiento ¿Cambio estructural al mutar? Uso típico
IComponentData (struct no gestionado) En chunk, SoA No (al cambiar el valor) Componente predeterminado
IComponentData (class gestionada) Tabla aparte No Tipos de referencia (mallas, referencias a GameObject)
IBufferElementData En chunk, con desbordamiento al heap No Arrays dinámicos por entidad
ISharedComponentData Por chunk Sí (mueve la entidad a un nuevo chunk) Agrupar entidades que comparten configuración
ICleanupComponentData En chunk Normal Protocolo de limpieza al destruir entidades
ICleanupSharedComponentData Por chunk Normal Protocolo de limpieza de recursos compartidos
Tag (IComponentData vacío) Ocupa cero bytes en el chunk No Marcador de filtro, sin datos
Chunk component Por chunk Sí Metadatos calculados una vez por chunk
Enableable (IEnableableComponent) En chunk + máscara de bits No Activar o desactivar el comportamiento por entidad

El resto de la sección desarrolla cada fila.

2. IComponentData no gestionado

La opción por defecto. Reside directamente en el chunk como datos SoA bien empaquetados. Compatible con Burst.

public struct Velocity : IComponentData
{
    public float3 Value;
}

Reglas:

  • Debe ser un struct, sin campos de referencia (nada de class, string ni List<T>).
  • Tipos de campo permitidos: primitivos, tipos de Unity.Mathematics, otros structs no gestionados, Entity, FixedString*, BlobAssetReference<T> y UnityObjectRef<T>.
  • Se lee con RefRO<T> y se escribe con RefRW<T> desde SystemAPI.Query.

3. IComponentData gestionado

Reside en una tabla aparte indexada por entidad. Admite tipos de referencia, pero conlleva una asignación de GC y rompe la compatibilidad con Burst de ese componente.

public class EnemyAI : IComponentData
{
    public AIModel Brain;      // a managed class
    public List<Entity> Allies;
}

Conviene usarlo solo cuando se necesita de verdad una referencia gestionada (una malla, una textura o un ScriptableObject de authoring). Cuando basta con una referencia débil a un UnityEngine.Object, es preferible UnityObjectRef<T> dentro de un componente no gestionado.

4. IBufferElementData — buffers dinámicos

Un DynamicBuffer<T> es un array de tamaño variable, uno por entidad.

public struct Waypoint : IBufferElementData
{
    public float3 Position;
}

var buffer = state.EntityManager.AddBuffer<Waypoint>(entity);
buffer.Add(new Waypoint { Position = new float3(1, 0, 0) });

Almacenamiento: hasta InternalBufferCapacity elementos caben en el chunk; a partir de ahí el buffer se desborda al heap. La capacidad se ajusta con [InternalBufferCapacity(N)].

5. ISharedComponentData — agrupación

Los componentes compartidos llevan un valor que agrupa chunks. Dos entidades con el mismo valor de ISharedComponentData acaban en el mismo chunk; con valores distintos van a chunks distintos.

public struct RenderMeshGroup : ISharedComponentData
{
    public int MaterialID;
}

Consecuencias:

  • Cambiar el valor traslada la entidad a otro chunk: es un cambio estructural y resulta costoso.
  • Ideal para datos que cambian poco pero varían según el grupo (juegos de materiales, IDs de equipo, regiones de bake).
  • Si el tipo contiene referencias, los valores se guardan en una tabla gestionada (un ISharedComponentData puede ser gestionado).

6. Componentes de limpieza — ICleanupComponentData

Marca una entidad de modo que, al "destruirse", siga viva solo con sus componentes de limpieza hasta que el system los procese.

public struct ConnectionCleanup : ICleanupComponentData
{
    public int ConnectionHandle;
}

// DestroyEntity(e) removes all normal components but leaves ConnectionCleanup
// behind so you can close the socket in a dedicated system, then destroy the
// cleanup component explicitly.

Su equivalente ICleanupSharedComponentData aplica el mismo protocolo a los recursos compartidos.

Casos de uso: liberar recursos nativos, darse de baja de systems externos o destruir una entidad por etapas a lo largo de varios fotogramas.

7. Componentes tag

Un tag es un IComponentData vacío:

public struct Enemy : IComponentData {}

Ocupa cero bytes en el chunk y se puede consultar como cualquier otro componente. Ideal como marcador del tipo "es un X". Abusar de los tags (cientos de tipos distintos) fragmenta los archetypes; para estados de alta cardinalidad conviene usar componentes habilitables o compartidos.

8. Chunk components

Un chunk component lleva un valor por chunk, no por entidad.

state.EntityManager.AddChunkComponentData<RegionBounds>(query, new RegionBounds { ... });

Uso típico: datos precalculados que valen para todas las entidades del chunk (bounds, ID de región, clave de partición). Modificar un chunk component provoca un cambio estructural.

9. Enableable components (IEnableableComponent)

Véase la página dedicada 06_Enableable Component.md para más detalle. Resumen aquí: un componente que puede activarse o desactivarse por entidad sin un cambio estructural, mediante una máscara de bits por chunk.

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

state.EntityManager.SetComponentEnabled<Stunned>(entity, false);  // no structural change

Las queries descartan de forma automática las entidades cuyos enableable components están desactivados; es una manera barata y habitual de excluir entidades de un system de forma temporal.

10. Guía de decisión

Quiero… Usar

Mantener una referencia gestionada por entidad IComponentData gestionado o UnityObjectRef<T>
Guardar una lista de longitud variable por entidad IBufferElementData

InvalidOperationException: The type T must be unmanaged Se usó un tipo gestionado donde hace falta uno no gestionado. Cambiar el componente o la API (por ejemplo, ComponentLookup<T> en vez de ManagedAPI.GetComponent<T>).
Cambiar el valor de un componente compartido provoca tirones Cada cambio es un cambio estructural. Agruparlos con un ECB, o asumir que este valor se modifica pocas veces.
DynamicBuffer<T> se desborda al heap [InternalBufferCapacity(N)] se queda corto para el uso real. Aumentarlo o mantener los tamaños acotados.
La entidad reaparece tras DestroyEntity Todavía hay un cleanup component adjunto. Eliminarlo de forma explícita una vez terminada la limpieza.
Demasiados archetypes Explosión de tags. Convertir los "estados" únicos por entidad en enableable components.
El componente tag no filtra una query Los tag components no tienen datos, así que no se puede pedir un RefRW<Tag> ni un RefRO<Tag> en la tupla de SystemAPI.Query<...>. Hay que aplicarlos con .WithAll<Tag>() o .WithNone<Tag>().

Capítulo 9. Enableable Component

1. Por qué existen los enableable components

Añadir o quitar un componente es un cambio estructural: la entidad pasa a otro chunk, las queries tienen que volver a resolverse y las referencias pueden invalidarse. Si esto ocurre en cada fotograma y por entidad (por ejemplo, al activar y desactivar "Stunned"), ese coste acapara el fotograma.

Un enableable component permite activar y desactivar comportamiento sin ningún cambio estructural. El componente siempre está presente en el chunk; un bit de una máscara por chunk determina si, para una query, cada entidad lo "tiene" o no.

2. La interfaz

Implementar tanto IComponentData como IEnableableComponent:

using Unity.Entities;

public struct Stunned : IComponentData, IEnableableComponent
{
    public float Duration;
}

También es posible hacer habilitable un IBufferElementData.

3. Activación y desactivación

Desde un system:

// Turn Stunned off without removing the component
state.EntityManager.SetComponentEnabled<Stunned>(entity, false);

// From an ECB (deferred, safe mid-iteration)
ecb.SetComponentEnabled<Stunned>(entity, true);

// From a job with a ComponentLookup
var stunnedLookup = SystemAPI.GetComponentLookup<Stunned>();
stunnedLookup.SetComponentEnabled(entity, false);

Leer el estado actual:

bool isStunned = state.EntityManager.IsComponentEnabled<Stunned>(entity);

Las tres vías actúan sobre una máscara de bits; ninguna provoca un cambio estructural.

4. Comportamiento de las queries

SystemAPI.Query<...> y la iteración basada en EntityQuery descartan las entidades cuyos enableable components estén desactivados en ese momento; por defecto, solo se ven las entidades en las que todos los componentes indicados están habilitados.

foreach (var stunned in SystemAPI.Query<RefRW<Stunned>>())
{
    // only fires for entities where Stunned is enabled
}

Para ver también las entidades desactivadas, se puede sobrescribir este comportamiento:

foreach (var (stunnedRW, entity) in SystemAPI
             .Query<RefRW<Stunned>>()
             .WithEntityAccess()
             .WithOptions(EntityQueryOptions.IgnoreComponentEnabledState))
{
    // fires for ALL entities with the component, enabled or not
}

5. Ejemplo — stun que expira

Un mismo IJobEntity no puede recibir a la vez ref Stunned (un RefRW<T>) y EnabledRefRW<Stunned> para el mismo componente: Entities prohíbe expresamente envolver el mismo componente en ambas formas dentro de un job. Hay dos patrones válidos:

5.1 Leer y escribir el valor, y conmutar el bit de habilitado mediante ECB

[BurstCompile]
public partial struct StunTickJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity entity,
                 ref Stunned stunned)
    {
        stunned.Duration -= DeltaTime;
        if (stunned.Duration <= 0f)
            ECB.SetComponentEnabled<Stunned>(chunkIndex, entity, false);
    }
}

La reproducción se produce en el siguiente límite de ECB (por ejemplo, EndSimulationECBS); véase 14_EntityCommandBuffer · Deferred Entity.md. No hay cambio estructural: solo se invierte un bit en un punto bien definido.

5.2 Conmutar el bit directamente cuando no hace falta el valor del componente

[BurstCompile]
public partial struct StunDisableJob : IJobEntity
{
    void Execute(EnabledRefRW<Stunned> stunnedEnabled)
    {
        stunnedEnabled.ValueRW = false;   // no structural change
    }
}

EnabledRefRW<T> es la única referencia a Stunned en este job, por lo que compila. Conviene usar esta forma cuando un system se limita a conmutar.

¿Hace falta leer el valor y conmutar el bit de habilitado de la misma entidad en una sola pasada? Conviene hacerlo desde el hilo principal con EntityManager.SetComponentEnabled<T>(entity, false), o repartirlo en dos jobs encadenados mediante state.Dependency.

6. Cuándo preferir enableable frente a añadir/quitar

Situación Elegir
Conmutación frecuente (por fotograma o por evento) Enableable
Estado que no siempre está presente (un modificador por entidad con campos variables) Enableable
Cambio de archetype poco frecuente (por ejemplo, añadir una capacidad permanente) Añadir o quitar el componente de la forma habitual
Filtrado uniforme en muchos systems Enableable (las queries lo recogen automáticamente)

Regla general: si el bit de "tiene este componente" cambia varias veces durante la vida de una entidad, conviene hacerlo habilitable.

7. Interacción con otras características

Característica Comportamiento
WithChangeFilter<T> Conmutar el bit también incrementa la versión de cambio del componente.
ICleanupComponentData Un cleanup component no puede ser también habilitable.
Netcode for Entities Los enableable components pueden sincronizarse como ghosts, pero los cambios de estado deben pasar por el lado autoritativo.
Serialización / exportación de world Los bits de habilitación se preservan.

8. Solución de problemas

Síntoma Causa / Solución
La query incluye entidades que deberían estar "desactivadas" El componente no implementa IEnableableComponent, o se ha aplicado WithOptions(EntityQueryOptions.IgnoreComponentEnabledState).
SetComponentEnabled<T> lanza "type T does not support being enableable" El struct solo implementa IComponentData. Añadir también IEnableableComponent.
El parámetro EnabledRefRW<T> es rechazado por el source generator Al job o system le falta partial, o el componente no es habilitable.
Error del source generator "cannot have both RefRW and EnabledRefRW for the same component" Repartir la lógica: usar ref T o EnabledRefRW<T> por job, no ambos sobre el mismo componente. Los dos patrones están en §5.
El filtro de cambio se dispara sin motivo aparente Conmutar Enabled cuenta como cambio para WithChangeFilter. Es lo esperado: conviene combinarlo con lecturas de EnabledRefRO en lugar de conmutar.
El SetComponentEnabled del ECB no surte efecto Se llamó sobre una entidad diferida antes de que se volcara el ECB que la creó. Ambas operaciones deben estar en el mismo ciclo de ECB.

Capítulo 10. Singleton Component

1. Qué es (y qué no es) un singleton

En Entities, un singleton es sin más un componente que se espera que exista en una sola entidad del world. No hay una interfaz especial ISingleton: cualquier IComponentData es un singleton por convención, y SystemAPI ofrece accesores cómodos que verifican la regla de "exactamente uno".

Usos típicos: estado del juego, configuración global, fuentes de tiempo y de tick, estado de red o registros centralizados de ECB.

public struct GameState : IComponentData
{
    public int   Score;
    public float TimeScale;
    public bool  Paused;
}

2. Los accesores

Dentro de cualquier ISystem / SystemBase:

// Reads
GameState gs      = SystemAPI.GetSingleton<GameState>();
RefRW<GameState> gsRW  = SystemAPI.GetSingletonRW<GameState>();
bool has          = SystemAPI.HasSingleton<GameState>();
bool got          = SystemAPI.TryGetSingleton<GameState>(out var gs2);

// Writes
SystemAPI.SetSingleton(new GameState { Score = 10 });

// Entity handle, if you need it
Entity e          = SystemAPI.GetSingletonEntity<GameState>();

// Buffers work too
DynamicBuffer<Waypoint> buf = SystemAPI.GetSingletonBuffer<Waypoint>();

Las variantes Get* lanzan una excepción si hay cero singletons o más de uno. Las variantes TryGet* devuelven false en esos casos. Las variantes RW marcan el componente como modificado para el seguimiento de cambios.

3. Crear el singleton

Dos patrones comunes:

3.1 Hornearlo desde una SubScene

public class GameStateAuthoring : MonoBehaviour
{
    public float InitialTimeScale = 1f;

    class Baker : Baker<GameStateAuthoring>
    {
        public override void Bake(GameStateAuthoring authoring)
        {
            var e = GetEntity(TransformUsageFlags.None);
            AddComponent(e, new GameState
            {
                Score     = 0,
                TimeScale = authoring.InitialTimeScale,
                Paused    = false
            });
        }
    }
}

Adjuntar GameStateAuthoring a un solo GameObject de la SubScene; el Baker produce exactamente una entidad con el singleton.

3.2 Crearlo al arrancar el system

public void OnCreate(ref SystemState state)
{
    var e = state.EntityManager.CreateEntity(typeof(GameState));
    state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
}

Es la opción para datos que no deben residir en el authoring (configuración inyectada por el servidor, generada de forma procedural, etc.).

4. Condicionar un system al singleton

Si un system solo tiene sentido cuando existe el singleton:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<GameState>();
}

RequireForUpdate<T>() indica al system que se salte OnUpdate por completo mientras el componente no esté presente. Así se evitan comprobaciones de nulidad defensivas dentro de OnUpdate.

5. Rendimiento de solo lectura frente a lectura-escritura

  • GetSingleton<T>() — lee una copia, sin seguimiento de cambios.
  • GetSingletonRW<T>() — devuelve un RefRW<T> que apunta al chunk. Asignar .ValueRW marca el componente como modificado, que es justo lo que interesa si otros systems usan WithChangeFilter<T>.
  • SetSingleton — escritura completa; siempre incrementa la versión.

En systems con muchas lecturas conviene GetSingleton; usar GetSingletonRW solo cuando de verdad se modifica el valor.

6. Múltiples candidatos — la trampa del "singleton con más de uno"

GetSingleton<T> lanza InvalidOperationException: expected exactly one entity with component T, found N cuando dos entidades terminan con el mismo componente singleton. Causas habituales:

  • Dos GameObjects de la SubScene llevan el script de authoring.
  • Un fixture de prueba creó el singleton en tiempo de ejecución y la SubScene también lo hornea.
  • Un system crea el singleton en OnCreate sin protegerlo con HasSingleton.

Patrón defensivo:

public void OnCreate(ref SystemState state)
{
    if (!SystemAPI.HasSingleton<GameState>())
    {
        var e = state.EntityManager.CreateEntity(typeof(GameState));
        state.EntityManager.SetComponentData(e, new GameState { TimeScale = 1f });
    }
}

7. Buffer singletons

Un DynamicBuffer<T> también puede ser un singleton:

public struct QueuedEvent : IBufferElementData
{
    public int Type;
    public int Payload;
}

// Access:
DynamicBuffer<QueuedEvent> events = SystemAPI.GetSingletonBuffer<QueuedEvent>();
events.Add(new QueuedEvent { Type = 1 });

Resulta útil para colas de eventos globales y flujos de comandos.

8. Solución de problemas

Síntoma Causa / Solución
InvalidOperationException: expected exactly one entity with component T, found 0 La entidad singleton nunca se creó. Añadir un baker, o crearla en OnCreate protegida por HasSingleton.
... found 2 Dos entidades llevan el componente. Eliminar el duplicado en el authoring o limpiarlo al arrancar.
GetSingletonRW no parece marcar el componente como modificado Se escribió sobre .ValueRO o se ignoró el RefRW. Solo .ValueRW activa el seguimiento de cambios.
El system se ejecuta antes de que se cree el singleton Usar RequireForUpdate<T>, u ordenar los systems para que el creador se ejecute antes ([UpdateBefore]).
Dos systems modifican el singleton sin un orden definido Añadir [UpdateAfter]/[UpdateBefore] o unificar la lógica. Las condiciones de carrera en singletons son especialmente difíciles de depurar.
Netcode: el valor del singleton diverge entre cliente y servidor Sincronizar el singleton como ghost, o reconstruirlo de forma determinista a partir de inputs sincronizados.

Capítulo 11. System — ISystem vs SystemBase

1. Descripción general

Entities ofrece dos maneras de declarar un system: ISystem (un struct no gestionado) y SystemBase (una class gestionada). Conviene elegir ISystem salvo que se necesite en concreto lo que aporta SystemBase.

ISystem SystemBase
Se declara como partial struct : ISystem partial class : SystemBase
Memoria No gestionada, ubicable en la pila Gestionada, propiedad del GC
Compilable con Burst Sí (añadir [BurstCompile]) No
Campos gestionados No Sí
Acceso a API gestionada Mediante EntityManager + ManagedAPI Directamente
Opción predeterminada ✅ Solo si es imprescindible

2. ISystem — la opción por defecto

using Unity.Burst;
using Unity.Entities;

[BurstCompile]
public partial struct EnemyAISystem : ISystem
{
    [BurstCompile]
    public void OnCreate(ref SystemState state)
    {
        // Called once when the system is created.
        state.RequireForUpdate<EnemySpawnConfig>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var config = SystemAPI.GetSingleton<EnemySpawnConfig>();
        // ... logic
    }

    [BurstCompile]
    public void OnDestroy(ref SystemState state)
    {
        // Called once when the system is destroyed.
    }
}

Requisitos obligatorios:

  • El struct debe ser partial: el source generator le añade un wrapper oculto.
  • Los tres métodos (OnCreate, OnUpdate y OnDestroy) son opcionales; basta con omitir los que no se necesiten.
  • Hay que poner [BurstCompile] en el struct y en cada método que se quiera compilar con Burst.

3. SystemBase — cuando un system gestionado es inevitable

using Unity.Entities;

public partial class PlayerInputSystem : SystemBase
{
    private InputActions _input;   // managed field — the reason SystemBase exists

    protected override void OnCreate()
    {
        _input = new InputActions();
        _input.Enable();
    }

    protected override void OnUpdate()
    {
        var move = _input.Gameplay.Move.ReadValue<Vector2>();
        // ... dispatch to components
    }

    protected override void OnDestroy() => _input.Dispose();
}

Conviene recurrir a SystemBase cuando:

  • Hay un campo gestionado que debe residir junto al system (objetos del Input System, handles de UI Toolkit, servicios gestionados, etc.).
  • Hace falta llamar a APIs gestionadas directamente dentro de OnUpdate sin pasar por el EntityManager.

Todo lo de un SystemBase se ejecuta en el hilo principal y sin Burst.

4. Ciclo de vida del system

Las dos variantes comparten los mismos nombres de hook:

Figure: System Lifecycle.

Hook Cuándo se dispara Uso típico
OnCreate Una vez, al crear el system Cachear queries, RequireForUpdate<T>, asignar contenedores nativos
OnStartRunning Una query pasa de vacía a no vacía Reiniciar el estado de la sesión
OnUpdate En cada fotograma, dentro del SystemGroup padre Trabajo principal
OnStopRunning Una query pasa de no vacía a vacía Volcar el estado almacenado en buffer
OnDestroy Una vez, al destruir el world Liberar recursos nativos o gestionados

RequireForUpdate<T>()

Saltarse OnUpdate por completo cuando falta un componente o singleton requerido:

public void OnCreate(ref SystemState state)
{
    state.RequireForUpdate<EnemySpawnConfig>();
    state.RequireForUpdate<GameRunning>();
}

Es la forma habitual de condicionar un system a que exista la configuración o a que haya empezado el juego.

5. Elegir entre los dos

Reglas prácticas:

  1. Empezar con ISystem. Es compatible con Burst y no genera basura para el GC.
  2. Pasar a SystemBase solo cuando un campo gestionado lo obligue. Y aun así, conviene plantearse dividirlo: dejar el trabajo gestionado en un SystemBase ligero que escriba los resultados en componentes, y ejecutar la lógica pesada en un ISystem que lea esos componentes.
  3. No mezclar añadiendo un campo gestionado a un struct ISystem: no compila. La división anterior es la solución idiomática.

Ejemplo de división:

// Managed — bridges Input System into components.
public partial class InputBridgeSystem : SystemBase
{
    private InputActions _input;
    // ... fills a PlayerInput singleton each frame
}

// Unmanaged + Burst — reads the PlayerInput singleton and drives gameplay.
[BurstCompile]
public partial struct PlayerMoveSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state) { /* ... */ }
}

6. Referencia rápida de SystemState

Dentro de un ISystem, ref SystemState state es la vía de acceso al world.

Miembro Propósito
state.EntityManager Cambios estructurales en el hilo principal.
state.World El World propietario.
state.Dependency El handle de dependencia de jobs actual; debe combinarse con los jobs programados.
state.GetEntityQuery(...) Crear o recuperar una query cacheada.
state.RequireForUpdate<T>() Condicionar el system a la presencia de un componente.
state.Enabled Ponerlo en false para saltarse el system sin destruirlo.

En un SystemBase, estos son miembros de la propia clase (por ejemplo, this.EntityManager).

7. Solución de problemas

Síntoma Causa / Solución
error CS0246: The type or namespace name 'Entities' could not be found Falta using Unity.Entities;.
OnUpdate nunca se dispara RequireForUpdate<T> está condicionando a un componente que aún no existe. Comprobar que el componente se cree en algún otro punto.
Error de compilación: "partial struct must be declared partial" ISystem exige partial para que el source generator pueda ampliar el struct.
Advertencia de Burst: "method is not burst compiled" Falta [BurstCompile] en el método. No basta con tenerlo solo en el struct.
El system se ejecuta en el world incorrecto (cliente frente a servidor con Netcode) Añadir [WorldSystemFilter(...)] para elegir entre ServerSimulation, ClientSimulation o LocalSimulation.
No se puede añadir un campo gestionado a ISystem Es lo esperado: separar la parte gestionada en un SystemBase, como se muestra arriba.
El system no aparece en Window → Entities → Systems Falta partial, está fuera del world por defecto, o lo excluye [DisableAutoCreation].

Capítulo 12. System Group & orden de actualización

1. Descripción general

Cada system reside dentro de un SystemGroup. Los grupos deciden cuándo se ejecutan sus hijos en cada fotograma, y los atributos de ordenación determinan la secuencia dentro de cada grupo. Acertar con la agrupación es lo que evita escrituras que llegan un fotograma tarde, condiciones de carrera y esos errores del tipo "por qué tiembla esta entidad".

2. Grupos predefinidos

Figure: System Group and Update Order.

Grupo Cuándo se ejecuta Contenido típico
InitializationSystemGroup Una vez por fotograma, antes de la simulación Inicializadores de una sola ejecución, tareas de bootstrap
SimulationSystemGroup Una vez por fotograma La mayor parte de la lógica de juego
FixedStepSimulationSystemGroup Paso de tiempo fijo (60 Hz por defecto); puede ejecutarse 0, 1 o N veces por fotograma Física, lógica de paso determinista
PresentationSystemGroup Una vez por fotograma, después de la simulación Preparación de render, interpolación
BeginSimulationEntityCommandBufferSystem Lo primero dentro de Simulation Reproducción ordenada de ECB (temprana)
EndSimulationEntityCommandBufferSystem Lo último dentro de Simulation Reproducción ordenada de ECB (tardía)

Netcode for Entities añade otros (PredictedSimulationSystemGroup, GhostSendSystem, etc.) que se anidan dentro de los anteriores. Véanse 16_Netcode Client-Server World & Bootstrap.md y 19_Netcode Prediction & Rollback.md.

3. Colocar un system en un grupo

using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial struct EnemyAISystem : ISystem { /* ... */ }

Si se omite [UpdateInGroup], el system se asigna por defecto a SimulationSystemGroup.

Anidamiento

Un grupo puede estar dentro de otro grupo:

[UpdateInGroup(typeof(SimulationSystemGroup), OrderFirst = true)]
public partial class GameplaySystemGroup : ComponentSystemGroup { }

[UpdateInGroup(typeof(GameplaySystemGroup))]
public partial struct MovementSystem : ISystem { /* ... */ }

ComponentSystemGroup es la clase base de los grupos personalizados. Funcionan a la vez como systems y como contenedores.

4. Ordenación dentro de un grupo

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateAfter(typeof(EnemyAISystem))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct EnemyPathfindingSystem : ISystem { /* ... */ }
Atributo Significado
[UpdateBefore(typeof(X))] Ejecutarse antes que X (ambos deben estar en el mismo grupo).
[UpdateAfter(typeof(X))] Ejecutarse después de X.
OrderFirst = true (en [UpdateInGroup]) Ejecutarse lo antes posible dentro del grupo.
OrderLast = true Ejecutarse lo más tarde posible.

Consejo: es preferible encadenar [UpdateAfter] que usar OrderFirst/Last; se lee mejor en las revisiones de código y aguanta mejor las refactorizaciones.

5. [RequireMatchingQueriesForUpdate]

Un atributo más reciente (Entities 1.x+) que indica a un system que se salte OnUpdate salvo que al menos una entidad coincida con cada una de sus queries:

[RequireMatchingQueriesForUpdate]
public partial struct EnemyAISystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Only runs when there is at least one entity matching every query the system uses.
    }
}

Conviene compararlo con state.RequireForUpdate<T>(), que exige un componente o singleton concreto. [RequireMatchingQueriesForUpdate] es más implícito: depende de las queries que acabe teniendo el system.

6. Grupos personalizados

Razones para crear uno:

  • Una fase de juego cohesionada (por ejemplo, "aplicar el daño antes de la animación").
  • Una frecuencia de tick distinta (por ejemplo, un paso fijo propio a 30 Hz).
  • Un interruptor de funcionalidad: activar o desactivar el grupo entero.
using Unity.Entities;

[UpdateInGroup(typeof(SimulationSystemGroup))]
public partial class DamagePipelineGroup : ComponentSystemGroup
{
    // You can override OnUpdate to customise how the group ticks.
}

[UpdateInGroup(typeof(DamagePipelineGroup), OrderFirst = true)]
public partial struct ApplyDamageSystem : ISystem { /* ... */ }

[UpdateInGroup(typeof(DamagePipelineGroup))]
[UpdateAfter(typeof(ApplyDamageSystem))]
public partial struct DeathSystem : ISystem { /* ... */ }

Frecuencia de tick personalizada

public partial class SlowTickGroup : ComponentSystemGroup
{
    private float _accum;
    protected override void OnUpdate()
    {
        _accum += (float)SystemAPI.Time.DeltaTime;
        if (_accum < 0.1f) return;      // 10 Hz
        _accum = 0f;
        base.OnUpdate();                // tick children
    }
}

7. Paso de tiempo fijo frente a variable

Si importa el determinismo (física, predicción en red, reproducción determinista), conviene poner el system en FixedStepSimulationSystemGroup:

[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial struct PhysicsStepSystem : ISystem { /* ... */ }

Puntos a tener en cuenta:

  • FixedStepSimulationSystemGroup puede ejecutarse varias veces por fotograma (o ninguna) para ponerse al día con el tiempo real.
  • Usar SystemAPI.Time.DeltaTime: dentro de este grupo es el dt del paso fijo.
  • Todo lo que lea input o genere presentación debe quedarse en SimulationSystemGroup o PresentationSystemGroup.

8. Inspeccionar el orden en tiempo de ejecución

Window → Entities → Systems muestra:

  • Cada grupo y sus hijos, en el orden de ejecución real.
  • Tiempo por system.
  • El estado habilitado o deshabilitado (de los systems que se conmutan en tiempo de ejecución).

Si el orden declarado con atributos no coincide con lo que se ve, casi siempre hay una dependencia circular o un [UpdateInGroup] que se ha olvidado.

9. Solución de problemas

Síntoma Causa / Solución
El system se ejecuta en la fase del fotograma equivocada Falta [UpdateInGroup] o es incorrecto. Por defecto es SimulationSystemGroup.
Se ignora [UpdateAfter] El system de destino está en otro grupo. Los dos deben estar en el mismo grupo.
Error de dependencia circular al inicio Una cadena de [UpdateBefore] / [UpdateAfter] forma un ciclo. La ventana Systems imprime el ciclo.
El system se ejecuta varias veces por fotograma sin esperarlo Está en FixedStepSimulationSystemGroup, un grupo que recupera el tiempo acumulado. Si no debe hacerlo, moverlo a SimulationSystemGroup.
El grupo personalizado se ejecuta indefinidamente El override de OnUpdate nunca llamó a base.OnUpdate().
El system se ejecuta pero no hace nada No se cumple RequireForUpdate<T>, o todas las queries están vacías. Comprobarlo en Window → Entities → Query.

Capítulo 13. JobSystem & Burst

1. Descripción general

El rendimiento de DOTS se asienta sobre dos funciones del motor que ya existían antes de Entities y que siguen siendo su base:

  • Job System: reparte trabajo entre los hilos de trabajo mediante grafos de dependencias.
  • Burst Compiler: compila jobs de C# a código nativo muy compacto (SIMD, sin GC, sin despacho virtual).

Ninguno de los dos es un concepto propio de Entities. IJobEntity e IJobChunk son capas finas sobre IJob y Burst. Entender la capa base hace que todos los patrones de más alto nivel se lean con claridad.

2. Job System — los tipos fundamentales

Interfaz Forma Usar cuando…
IJob void Execute() Trabajo puntual en un solo hilo.
IJobParallelFor void Execute(int index) Bucle de 0 a N en paralelo.
IJobParallelForBatch void Execute(int start, int count) Bucle en lotes paralelos (aprovecha mejor la caché que el modo por índice).
IJobChunk void Execute(ArchetypeChunk chunk, int chunkIndex, bool useEnabledMask, in v128 enabledMask) Iterar chunks ECS en paralelo.
IJobEntity void Execute([component params]) Iteración de entidades ECS; el source generator lo convierte en un chunk job.

Todos son structs y todos son compatibles con Burst.

Ejemplo mínimo de IJobParallelFor

using Unity.Burst;
using Unity.Collections;
using Unity.Jobs;

[BurstCompile]
struct ScaleJob : IJobParallelFor
{
    public NativeArray<float> Data;
    public float Factor;

    public void Execute(int index) => Data[index] *= Factor;
}

// Scheduling:
var handle = new ScaleJob { Data = arr, Factor = 2f }.Schedule(arr.Length, 64);
handle.Complete();

Notas:

  • Los campos deben ser blittables.
  • Escribir en el mismo NativeArray desde varios jobs exige encadenar las dependencias (véase más abajo).

3. Programación y dependencias

JobHandle a = jobA.Schedule();
JobHandle b = jobB.Schedule(a);              // b runs after a
JobHandle c = jobC.Schedule(JobHandle.CombineDependencies(a, b));
c.Complete();

Reglas:

  • Cada Schedule() devuelve un JobHandle. Si el nuevo job lee o escribe datos compartidos, hay que pasarle el handle anterior como dependencia.
  • Nunca hay que omitir .Complete() sobre el handle final antes de que el hilo principal lea los resultados (el Safety System lo exige).
  • En un ISystem, state.Dependency es el handle del system; los jobs programados se combinan con él de forma automática al usar .Schedule(state.Dependency) o helpers generados por código como IJobEntity.ScheduleParallel(...).

Dentro de un ISystem

[BurstCompile]
public void OnUpdate(ref SystemState state)
{
    var job = new MyJob { Dt = SystemAPI.Time.DeltaTime };
    state.Dependency = job.ScheduleParallel(state.Dependency);
}

IJobEntity.ScheduleParallel(), sin dependencia explícita, se combina de forma implícita con state.Dependency. Al encadenar varios jobs conviene indicar la dependencia de forma explícita.

4. Burst — qué es y por qué

Burst compila un subconjunto de C# (métodos marcados con [BurstCompile], sin asignaciones de GC, con datos blittables) a código nativo muy optimizado mediante LLVM.

Qué compila bien Burst

  • Métodos de struct IJob* / ISystem etiquetados con [BurstCompile].
  • Cálculos con tipos de Unity.Mathematics (float3, quaternion, int4, etc.).
  • Bucles compactos sobre NativeArray<T>, NativeList<T>, DynamicBuffer<T> o los arrays de columna de ArchetypeChunk.

Qué no puede compilar Burst

  • Referencias gestionadas (class, string, List<T> o cualquier cosa que no sea unmanaged).
  • Excepciones (se pueden capturar, pero en general conviene evitarlas).
  • Despacho dinámico a través de interfaces (la instanciación genérica explícita sí es válida).
  • System.IO, Reflection y la mayor parte del BCL más allá de las matemáticas y los tipos primitivos.

Activar Burst

using Unity.Burst;

[BurstCompile]
public partial struct MyJob : IJobEntity
{
    // attribute on the struct marks the type for Burst
    // methods need [BurstCompile] too if you want each compiled
    [BurstCompile]
    void Execute(ref Velocity v) { /* ... */ }
}

Para systems:

[BurstCompile]
public partial struct MySystem : ISystem
{
    [BurstCompile] public void OnCreate(ref SystemState state) { }
    [BurstCompile] public void OnUpdate(ref SystemState state) { }
}

El atributo en el tipo no basta: cada método que se quiera compilar necesita además su propio [BurstCompile].

Burst Inspector

Window → Analysis → Burst Inspector muestra, por método, el IR de LLVM, el ensamblador y el uso de vectores. Es la herramienta para responder a la pregunta de "por qué esto no va rápido".

5. Contenedores nativos

Burst y el Safety System esperan contenedores nativos, no colecciones gestionadas:

Gestionado Equivalente nativo
T[] NativeArray<T>
List<T> NativeList<T>
Dictionary<K,V> NativeHashMap<K,V>, NativeParallelHashMap<K,V>
HashSet<T> NativeHashSet<T>, NativeParallelHashSet<T>
Queue<T> NativeQueue<T>
string FixedString32Bytes / FixedString64Bytes / etc.

Reserva de memoria:

var arr = new NativeArray<int>(1024, Allocator.TempJob);
// ...
arr.Dispose();

La vida útil importa: Burst y el Safety System verifican que los contenedores se liberen y que no se acceda a ellos una vez terminado el job.

6. Allocators

Allocator Vida útil Uso típico
Allocator.Temp Dentro del fotograma actual y solo en el hilo principal Contenedores temporales en OnUpdate
Allocator.TempJob Lo que dura un job, hasta 4 fotogramas Contenedores que se pasan a los jobs
Allocator.Persistent Requiere Dispose() manual Contenedores del propio system que persisten entre fotogramas

Dentro de los systems es preferible state.WorldUpdateAllocator (o su allocator Rewindable): evita tener que gestionar los Dispose() a mano.

7. Safety System

En el Editor con "Jobs > Leak Detection" y "Jobs > Safety Checks" activados, Unity valida que:

  • Dos jobs no escriben en el mismo contenedor nativo sin una arista de dependencia.
  • Una lectura del hilo principal no entra en conflicto con una escritura en curso.
  • Los contenedores nativos no se usan tras un Dispose() ni más allá de la vida útil de su allocator.

Los errores de seguridad aparecen como excepciones con un mensaje claro del tipo "el job A y el job B acceden los dos a X". No conviene desactivar las comprobaciones de seguridad durante el desarrollo: la ganancia de rendimiento es pequeña y el coste de depuración, alto.

8. Solución de problemas

Síntoma Causa / Solución
Advertencia de Burst: "method not burst compiled" Falta [BurstCompile] en el método, o Burst se topó con un tipo no compatible (revisar en el Burst Inspector la línea que falla).
InvalidOperationException: … writes to … without declaring dependency Falta una arista de dependencia: pasar el JobHandle anterior o encadenar mediante state.Dependency.
ObjectDisposedException en un NativeArray Se liberó antes de que terminara el job. Usar arr.Dispose(handle) o esperar antes a que el handle termine.
El job programado nunca termina Falta .Complete() en una cadena, o hay una dependencia circular.
Regresión de rendimiento tras añadir [BurstCompile] El método se recompila en cada recarga de dominio; comprobar la caché de Burst. O se ha colado un tipo incompatible con Burst y el método ha vuelto en silencio a ejecutarse como IL.
La enumeración de NativeHashMap lanza excepción La enumeración no se lleva bien con las escrituras concurrentes; en su lugar, enumerar una copia en un NativeArray<K>.
El hilo principal se queda bloqueado esperando un job Se llamó a .Complete() demasiado pronto. Llevar el punto de finalización lo más tarde posible dentro del fotograma.

Capítulo 14. IJobEntity · SystemAPI.Query

1. Descripción general

Dos APIs cubren el 95% de la iteración de componentes en Entities 6.5:

API Dónde se ejecuta Usar cuando…
SystemAPI.Query<...>() Hilo principal, dentro de OnUpdate El trabajo es trivial (unos cientos de entidades) o requiere APIs del hilo principal.
IJobEntity Hilos de trabajo, mediante Schedule / ScheduleParallel El trabajo es más pesado o se beneficia del paralelismo.

Son la alternativa recomendada al antiguo Entities.ForEach (marcado como obsoleto en Entities 1.4; en 6.5 todavía compila, con advertencias de deprecación; el changelog de Unity indica que su eliminación está prevista para una futura versión mayor).

2. SystemAPI.Query<T>

Recorre en el hilo principal las entidades que coinciden. A los componentes se accede mediante RefRO<T> o RefRW<T>.

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRW.Max);
        }
    }
}

Reglas:

  • El struct del system debe ser partial.
  • RefRO<T> para solo lectura y RefRW<T> para lectura-escritura. No se debe escribir a través de RefRO.
  • Si se necesita el id, basta con añadir una Entity a la tupla: SystemAPI.Query<RefRW<Health>>().WithEntityAccess().

Filtros

foreach (var health in SystemAPI.Query<RefRW<Health>>()
             .WithAll<Enemy>()
             .WithNone<Dead>()
             .WithAny<Burning, Poisoned>())
{
    // only entities that are Enemy AND not Dead AND (Burning OR Poisoned)
}
  • WithAll<T>(): debe tener todos los componentes indicados.
  • WithNone<T>(): no debe tener ninguno de ellos.
  • WithAny<T...>(): debe tener al menos uno de ellos.
  • WithChangeFilter<T>(): solo los chunks donde el componente T ha cambiado desde la última ejecución.
  • WithEntityAccess(): devuelve también el id de Entity.

3. IJobEntity

Una iteración paralela creada por el source generator. La firma de Execute declara con qué componentes se trabaja.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Reglas de la firma:

  • void Execute(...): cada parámetro se corresponde uno a uno con un componente.
  • ref Component: lectura-escritura. in Component: solo lectura (no se puede escribir).
  • Se permite Entity entity como parámetro: el source generator lo rellena.
  • El struct del job debe ser partial.

Variantes de Schedule

Llamada Hilos
.Run() Hilo principal; termina antes de devolver el control. Útil para depurar o para cargas de trabajo pequeñas.
.Schedule() Un solo hilo de trabajo.
.ScheduleParallel() Reparte los chunks entre los hilos de trabajo.

Los tres actualizan state.Dependency de forma automática dentro de ISystem.OnUpdate.

Filtros en jobs

new HealthRegenJob { DeltaTime = dt }
    .ScheduleParallel(SystemAPI.QueryBuilder()
        .WithAll<Enemy, Regen>()
        .WithNone<Dead>()
        .Build());

O con los atributos [WithAll], [WithNone] y [WithAny] sobre el propio struct del job.

Acceso a entidad en un job

void Execute(Entity entity, ref Health health) { /* ... */ }

Índice de chunk / [ChunkIndexInQuery]

Resulta útil al combinarlo con un ECB de ParallelWriter para una reproducción determinista: se pasa el índice de chunk como sortKey. Véase 15_ParallelWriter · Deterministic Playback.md.

void Execute([ChunkIndexInQuery] int chunkIndex, in Health health, Entity entity)
{
    if (health.Value <= 0)
        ecb.DestroyEntity(chunkIndex, entity);
}

4. Cuándo usar cada uno

Situación Elegir
De decenas a cientos de entidades, con cálculos triviales SystemAPI.Query
Miles de entidades o más, con trabajo independiente por entidad IJobEntity.ScheduleParallel
Trabajo por chunk (bounds, sumas, partición espacial) IJobChunk — véase 12_IJobEntity vs IJobChunk.md
Hace falta usar una API gestionada dentro del bucle SystemAPI.Query desde un SystemBase
Hacen falta cambios estructurales diferidos Cualquiera de las dos formas, pero escribiendo mediante EntityCommandBuffer.ParallelWriter; véase 14_EntityCommandBuffer · Deferred Entity.md

5. Puntos a tener en cuenta

  • No capturar this ni campos gestionados dentro de un job. El source generator da error si se intenta.
  • RefRW<T>.ValueRW activa el seguimiento de cambios. Leer a través de ValueRW sin escribir igualmente marca el chunk como sucio.
  • WithChangeFilter<T> trabaja a nivel de chunk. Si cualquier entidad del chunk escribe en T, el chunk entero se considera modificado.
  • Los Aspects están obsoletos (aunque no eliminados). La interfaz de agrupación IAspect se deprecó en la 1.4 y todavía figura como tipo de parámetro válido de Execute en la referencia de la API 6.5. El código nuevo debe usar parámetros de componente normales; los aspects ya existentes siguen compilando en 6.5.

6. Solución de problemas

Síntoma Causa / Solución
Error de compilación: IJobEntity must be partial Añadir partial al struct del job.
ScheduleParallel lanza un error de condición de carrera Dos jobs escriben en el mismo componente sin una arista de orden. Comprobar que state.Dependency se combine bien, o añadir [UpdateAfter] entre los systems.
El job no se ejecuta con Burst Falta [BurstCompile] en el struct del job, o se captura un tipo incompatible con Burst (una referencia gestionada, o un string en lugar de un FixedString).
SystemAPI.Query no devuelve nada La combinación de filtros está vacía. Inspeccionar la query en Window → Entities → Query para ver qué coincide realmente.
Escribir mediante RefRO (bug silencioso) El compilador ya da error ante refRO.ValueRW = …. Si se cuela con versiones antiguas del source generator, cambiar a RefRW.
El recuento de entidades no coincide entre SystemAPI.Query y una EntityQuery creada a mano Los filtros son distintos. SystemAPI.Query incluye por defecto el filtrado de enableable components; EntityQueryBuilder hay que configurarlo de forma explícita.

Capítulo 15. IJobEntity vs IJobChunk

1. La respuesta corta

  • IJobEntity: para "hacer lo mismo con cada entidad".
  • IJobChunk: para "hacer en cada chunk algo que no sea un simple mapeo por entidad".

IJobEntity se genera sobre IJobChunk: ambos usan la misma maquinaria de iteración. IJobChunk se limita a exponer el chunk completo al código.

2. IJobEntity — ergonomía por entidad

[BurstCompile]
public partial struct DamageTickJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Burn burn)
    {
        health.Value -= burn.DamagePerSecond * DeltaTime;
    }
}

Ventajas:

  • Un método y una vista por entidad. Fácil de escribir y fácil de leer.
  • Toda la infraestructura la genera el source generator: los parámetros Entity, [ChunkIndexInQuery] y [EntityIndexInQuery] se rellenan solos.
  • Filtros mediante los atributos [WithAll], [WithNone] y [WithAny] sobre el struct, o con un QueryBuilder que se pasa a .Schedule*.

Desventajas:

  • No hay visibilidad entre entidades dentro de Execute. Sin estado adicional, no es fácil "sumar todo el chunk".
  • Desde Execute no se accede a los metadatos del chunk (la versión de cambio, el valor del componente compartido).

3. IJobChunk — control completo del chunk

[BurstCompile]
public partial struct HealthAggregateJob : IJobChunk
{
    [ReadOnly] public ComponentTypeHandle<Health> HealthHandle;
    public NativeArray<float> PerChunkTotal;  // size = chunk count

    public void Execute(in ArchetypeChunk chunk,
                        int unfilteredChunkIndex,
                        bool useEnabledMask,
                        in v128 chunkEnabledMask)
    {
        var healths = chunk.GetNativeArray(ref HealthHandle);
        float total = 0f;
        for (int i = 0; i < chunk.Count; i++)
            total += healths[i].Value;

        PerChunkTotal[unfilteredChunkIndex] = total;
    }
}

Ventajas:

  • Acceso directo a los arrays del chunk: chunk.GetNativeArray(ref handle), chunk.GetBufferAccessor(ref bufferHandle).
  • chunk.Has<T>() y chunk.GetSharedComponent<T>(...) para trabajo condicional.
  • Ve la máscara de habilitados y la versión de cambio de cada chunk.
  • Ideal para reducciones, partición espacial y registro de ECB por lotes.

Coste:

  • Más código repetitivo: hay que declarar los ComponentTypeHandle<T> en el system y actualizarlos cada fotograma con handle.Update(ref state).
  • El bucle for interno lo gestiona el propio código, así que hay más sitios donde equivocarse.

Configurar los handles en un system

[BurstCompile]
public partial struct HealthAggregateSystem : ISystem
{
    private EntityQuery _query;
    private ComponentTypeHandle<Health> _healthHandle;

    public void OnCreate(ref SystemState state)
    {
        _query        = state.GetEntityQuery(ComponentType.ReadOnly<Health>());
        _healthHandle = state.GetComponentTypeHandle<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthHandle.Update(ref state);

        var totals = CollectionHelper.CreateNativeArray<float>(
            _query.CalculateChunkCount(), state.WorldUpdateAllocator);

        state.Dependency = new HealthAggregateJob
        {
            HealthHandle   = _healthHandle,
            PerChunkTotal  = totals
        }.ScheduleParallel(_query, state.Dependency);
    }
}

handle.Update(ref state) refresca cada fotograma el handle de tipo cacheado; si se olvida, Burst acaba usando índices obsoletos.

4. Máscara de habilitados (useEnabledMask)

En las queries que incluyen un IEnableableComponent, Execute recibe useEnabledMask = true y una máscara v128 con los bits de habilitado. Hay que respetar la máscara o se procesarán entidades desactivadas.

Hay un helper para ello:

var enumerator = new ChunkEntityEnumerator(useEnabledMask, chunkEnabledMask, chunk.Count);
while (enumerator.NextEntityIndex(out int i))
{
    // i is the index of the next enabled entity in the chunk
}

IJobEntity aplica la máscara de forma automática; es otra razón para preferirlo salvo que se necesite visibilidad a nivel de chunk.

5. Guía de decisión

Situación Elegir
Actualizar la salud, trasladar la posición, aplicar el input: cálculos puros por entidad IJobEntity
Sumar los valores de todas las entidades de un chunk IJobChunk
Registrar un comando ECB por chunk en lugar de por entidad IJobChunk
Acceder a un valor de componente compartido dentro del bucle IJobChunk
Se necesita la versión de cambio del chunk IJobChunk
Iterar un DynamicBuffer<T> en cada entidad Cualquiera funciona; IJobEntity es más limpio
Lógica por entidad, pero se necesita el índice de chunk de la entidad IJobEntity con [ChunkIndexInQuery] int chunkIndex
El coste de programación importa y el trabajo es pequeño Conviene medir: reducir el número de chunks compensa más que reducir el de entidades. Con miles de chunks suele ganar IJobChunk.

6. Interoperabilidad — usar ambos juntos

Un patrón habitual: recopilar datos por chunk con IJobChunk y, después, escribir por entidad con IJobEntity, encadenando ambos mediante dependencias.

var gather = new HealthAggregateJob { ... }
    .ScheduleParallel(query, state.Dependency);

var act = new HealJob { PerChunkTotal = totals }
    .ScheduleParallel(gather);

state.Dependency = act;

7. Solución de problemas

Síntoma Causa / Solución
IJobChunk.Execute lee valores basura Se olvidó handle.Update(ref state) antes de programar el job, o se usó la query equivocada.
No se encuentra ComponentTypeHandle<T> Falta using Unity.Entities;, o el handle se declaró con el modo de acceso equivocado. Para solo lectura, usar GetComponentTypeHandle<T>(true).
Se omiten entidades habilitadas No se respetó la máscara; usar ChunkEntityEnumerator como se ha mostrado. IJobEntity evita este problema.
El job escribe en el mismo NativeArray desde dos chunks Reservar un slot por chunk (como en HealthAggregateJob) o usar un NativeParallelHashMap con el índice de chunk como clave.
Programar un IJobChunk sin el argumento de query ScheduleParallel(query, ...) exige indicar la query de forma explícita. IJobEntity.ScheduleParallel() la obtiene de los atributos o del source generator.
No se encuentra el tipo v128 Falta using Unity.Burst.Intrinsics;.

Capítulo 16. Cambio estructural y seguridad

1. Qué cuenta como cambio estructural

Un cambio estructural es cualquier operación capaz de mover entidades entre chunks o de alterar el inventario de chunks. Todas estas son estructurales:

Operación Por qué es estructural
CreateEntity / DestroyEntity Reserva o libera un slot dentro de un chunk.
AddComponent<T> / RemoveComponent<T> El archetype de la entidad cambia, así que se traslada físicamente a otro chunk.
SetSharedComponent<T> Un valor distinto de componente compartido implica otro chunk (se agrupa por valor).
AddChunkComponent<T> / RemoveChunkComponent<T> Cambia los metadatos propios del chunk.
SetName(entity, ...) Modifica los metadatos de la entidad en el EntityManager.
MoveEntitiesFrom Copia entidades entre worlds.
Instantiate(prefab) Reserva una entidad nueva (puede pre-rellenar un archetype).

Escribir el valor de un componente con SetComponentData o RefRW<T>.ValueRW no es estructural: el componente ya está en el chunk y solo se cambian unos bytes.

2. Por qué los cambios estructurales son peligrosos

Cuando el archetype cambia, todo lo que mantenga una referencia posicional dentro de un chunk queda obsoleto:

  • Los iteradores de query en curso (SystemAPI.Query<...>) pueden saltarse entidades o volver a visitarlas.
  • Las entradas de ComponentLookup<T> dejan de ser válidas para las entidades afectadas.
  • Los handles de ArchetypeChunk de un job apuntan al chunk antiguo.
  • Otros systems que se ejecutan a la vez ven un estado inconsistente.

El Safety System detecta la mayoría de estos casos con una InvalidOperationException en el Editor. En una build se traducen en crashes o entidades corruptas.

3. Regla general

No realizar cambios estructurales mientras se itera una query en la misma actualización del system.

Dos patrones válidos, en orden de preferencia:

  1. Usar un EntityCommandBuffer. Se registra la operación durante la iteración y se reproduce después. Véase 14_EntityCommandBuffer · Deferred Entity.md.
  2. Volcar las entidades en una NativeList<Entity> y aplicar los cambios tras el bucle. Más sencillo para casos pequeños y sin la sobrecarga de un ECB.

Ejemplo del patrón 2:

var toKill = new NativeList<Entity>(state.WorldUpdateAllocator);

foreach (var (health, e) in SystemAPI
             .Query<RefRO<Health>>()
             .WithEntityAccess())
{
    if (health.ValueRO.Value <= 0f)
        toKill.Add(e);
}

state.EntityManager.DestroyEntity(toKill.AsArray());

4. Enableable components — la vía de escape

Si se acaba añadiendo y quitando un componente en cada fotograma, casi con seguridad debería ser un enableable component: conmutar el bit de habilitado no es estructural. Véase 06_Enableable Component.md.

5. Jobs y cambios estructurales

Los cambios estructurales son operaciones del hilo principal: no se pueden hacer dentro de un job.

Las formas idiomáticas de mutar desde un job:

Enfoque Cuándo
Un EntityCommandBuffer.ParallelWriter que se registra en el job y se reproduce en el hilo principal Cambios estructurales por entidad desde código paralelo.
Un EntityCommandBuffer (de un solo hilo) que se registra en un IJob o IJobChunk y se reproduce en el hilo principal Cambios por lotes o con alcance de chunk.
Conmutar un enableable component mediante EnabledRefRW<T> dentro de un job Activar o desactivar por entidad sin cambios estructurales.

Un ECB guarda la operación como un comando; la reproducción se produce en un punto bien definido del hilo principal, normalmente en un límite de EntityCommandBufferSystem.

6. Combinar cambios estructurales y lecturas

Si dos systems se ejecutan uno tras otro y el system A hace un cambio estructural del que depende el system B, conviene fijar el orden de forma explícita:

[UpdateInGroup(typeof(SimulationSystemGroup))]
[UpdateBefore(typeof(MovementSystem))]
public partial struct SpawnSystem : ISystem { /* uses ECB */ }

Cuando la reproducción del ECB system queda entre A y B (por ejemplo, EndSimulationECBS con una cadena de [UpdateAfter] bien planteada), el cambio estructural se aplica antes de que B lea.

7. Modelo de coste

Orden de magnitud aproximado para un único cambio:

SetComponentData (value only)          ~ 1x
SetComponentEnabled (enableable bit)   ~ 1x (per-chunk bitmask write)
AddComponent (structural)              ~ 10x   (entity moves chunk)
SetSharedComponent (structural)        ~ 10x
DestroyEntity (structural)             ~ 10x
Per-frame add + remove pattern         pathological

Son órdenes de magnitud; las cifras exactas dependen del tamaño del archetype, de la ocupación del chunk y de la presión sobre el allocator.

8. Diagnosticar el desgaste estructural

Síntomas:

  • En Window → Entities → Systems, un system consume un tiempo desproporcionado para la lógica que tiene.
  • El marcador del Profiler EntityManager.SetArchetype o Move Entity acapara los fotogramas.
  • El número de archetypes crece con el tiempo.

Tácticas:

  • Revisar las llamadas a AddComponent y RemoveComponent: ¿hay alguna por fotograma? Sustituirlas por enableable components.
  • Examinar los conjuntos de valores de los componentes compartidos: ¿hay alguna clave que sea por entidad sin querer?
  • Buscar en la ventana Archetypes chunks con una sola entidad; eso es fragmentación por conjuntos de componentes demasiado granulares.

9. Solución de problemas

Síntoma Causa / Solución
InvalidOperationException: EntityCommandBuffer has already been played back Se grabó en un ECB después de reproducirlo. Cada ECB es de un solo uso: hay que pedir uno nuevo al ECBS en cada fotograma.
InvalidOperationException: … invalidated … durante un foreach Hay un cambio estructural dentro del bucle. Pasarlo a un ECB o usar el patrón de almacenar primero y aplicar después.
Las entidades desaparecen sin avisar Se ejecutó un DestroyEntity, pero otros systems siguen guardando el handle antiguo de Entity. Comprobar los handles con EntityManager.Exists.
El patrón de añadir y quitar consume tiempo de fotograma Sustituirlo por enableable components.
RemoveComponent provoca el error "cleanup component present" La entidad tiene un cleanup component. Procesar primero la limpieza (véase 05_Component Types.md).
El conjunto de valores del componente compartido se dispara (cientos de valores distintos) El valor es demasiado granular. Agruparlo en intervalos (por ejemplo, redondear al N más cercano) o guardar la parte variable como un componente normal.

Capítulo 17. EntityCommandBuffer · Entidad diferida

1. Por qué existe el ECB

EntityCommandBuffer (ECB) registra cambios estructurales (crear, destruir, añadir, quitar, asignar) y los reproduce más tarde en el hilo principal, en un punto seguro. Permite modificar el world desde:

  • Dentro de un foreach de una query (donde una llamada directa a EntityManager invalidaría el iterador).
  • Un job (donde no se permiten cambios estructurales del hilo principal).
  • Un job paralelo que abarca muchos chunks (mediante ParallelWriter).

El contexto de por qué esto es peligroso está en 13_Structural Change & Safety.md.

2. El ciclo de vida del ECB

Figure: EntityCommandBuffer Lifecycle.

Un ECB es de un solo uso: tras la reproducción se libera. En cada fotograma se pide uno nuevo al EntityCommandBufferSystem adecuado.

3. ECB systems predefinidos

System Cuándo reproduce Uso típico

4. Obtener un ECB

Dentro de cualquier ISystem.OnUpdate:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged);

El tipo anidado .Singleton es un token, uno por system, que el source generator expone para cada ECB system predefinido. CreateCommandBuffer devuelve un ECB nuevo que se reproducirá cuando ese system se ejecute.

Para jobs paralelos:

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

El contrato de la reproducción paralela está en 15_ParallelWriter · Deterministic Playback.md.

5. Grabación — creación de entidades

Entity e = ecb.CreateEntity();
ecb.AddComponent(e, new Health { Value = 100 });
ecb.AddComponent(e, LocalTransform.FromPosition(0, 0, 0));

O crear una instancia a partir de una entidad prefab:

Entity inst = ecb.Instantiate(prefab);
ecb.SetComponent(inst, LocalTransform.FromPosition(spawnPos));

Tanto CreateEntity como Instantiate devuelven una entidad diferida (véase §6).

6. Entidades diferidas

La entidad que devuelven ecb.CreateEntity() o ecb.Instantiate() aún no existe: tiene un índice negativo provisional. Con ella se puede:

  • Pasarla a llamadas posteriores del mismo ECB: AddComponent, SetComponent o AppendToBuffer.
  • Guardarla en otro componente mediante ecb.SetComponent(other, new ReferenceTo { Entity = inst }).

No se puede:

  • Leerla ni escribir en ella mediante EntityManager antes de que el ECB se reproduzca.
  • Pasarla entre ECBs distintos (el valor provisional solo se resuelve dentro del ECB que la creó).

Durante la reproducción, el ECB system sustituye el valor provisional por el handle real de Entity en todos los puntos donde se utilizó.

7. Grabación — cambios estructurales

ecb.AddComponent(entity, new Stunned { Duration = 2f });
ecb.RemoveComponent<Stunned>(entity);
ecb.SetComponent(entity, new Health { Value = 50 });

ecb.SetComponentEnabled<Stunned>(entity, false);

var buf = ecb.AddBuffer<Waypoint>(entity);
buf.Add(new Waypoint { Position = new float3(1, 0, 0) });
ecb.AppendToBuffer(entity, new Waypoint { Position = new float3(2, 0, 0) });

ecb.DestroyEntity(entity);

Todos quedan en cola como comandos; ninguno surte efecto hasta la reproducción.

9. ECBs independientes (manuales)

A veces hace falta controlar la reproducción a mano, por ejemplo dentro de un solo OnUpdate:

var ecb = new EntityCommandBuffer(Allocator.Temp);
// ... record ...
ecb.Playback(state.EntityManager);
ecb.Dispose();

Es la opción para patrones de modificación autónomos y puntuales. Para todo lo que cruce los límites de un system o se ejecute en un job, usar los ECBS predefinidos anteriores.

10. Determinismo

En los ECB de escritor único, los comandos se reproducen en el orden en que se grabaron. En un ParallelWriter, el orden de reproducción depende del sortKey que se indique; véase 15_ParallelWriter · Deterministic Playback.md.

En sistemas de Netcode o de replay, la reproducción determinista es innegociable. Hay que pasar siempre una clave de ordenación estable (índice de chunk, índice dentro de la query) y evitar grabar en rutas de código del hilo principal que se ejecuten en un orden no determinista.

Capítulo 18. ParallelWriter · Reproducción determinista

1. Por qué existe un writer paralelo separado

EntityCommandBuffer es de escritor único. Si dos hilos grabaran en el mismo ECB, competirían por su cadena interna. EntityCommandBuffer.ParallelWriter lo resuelve dando a cada llamador una cadena propia por hilo y exigiendo una clave de ordenación en cada comando.

En el momento de la reproducción, el ECB system:

  1. Aplana todas las cadenas por hilo en una sola.
  2. Ordena los comandos por sortKey (con una ordenación estable).
  3. Los ejecuta en el hilo principal.

El resultado es determinista mientras las claves de ordenación lo sean.

2. Obtener un ParallelWriter

var ecb = SystemAPI
    .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
    .CreateCommandBuffer(state.WorldUnmanaged)
    .AsParallelWriter();

AsParallelWriter() devuelve un struct EntityCommandBuffer.ParallelWriter que se puede copiar dentro de un IJobEntity, un IJobChunk o un IJobParallelFor.

3. El contrato de sortKey

Cada llamada de un ECB paralelo recibe un sortKey como primer argumento:

ecb.Instantiate(sortKey, prefab);
ecb.SetComponent(sortKey, entity, value);
ecb.DestroyEntity(sortKey, entity);
ecb.AddComponent(sortKey, entity, component);

Reglas para una buena clave de ordenación:

  • Determinista. La misma entrada produce la misma clave, sin importar cómo reparta el trabajo el planificador.
  • Suficientemente única. Dos comandos sobre la misma entidad no deberían colisionar, salvo que se quiera un orden de reproducción ambiguo.
  • Barata. Un int del que ya se dispone, no un hash.

En la práctica, las opciones preferidas son:

Fuente Cómo obtenerla Cuándo usar
[ChunkIndexInQuery] en IJobEntity La rellena el source generator El 99% de los casos
unfilteredChunkIndex en IJobChunk.Execute Lo proporciona la API Jobs de IJobChunk
[EntityIndexInQuery] en IJobEntity Source generator Cuando los comandos de cada entidad deben ordenarse uno a uno
Un índice determinista calculado manualmente Lo calcula el propio código Poco habitual; estructuras de job a medida

4. Ejemplo completo — spawn paralelo

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

[BurstCompile]
public partial struct SpawnParallelJob : IJobEntity
{
    public float                               DeltaTime;
    public EntityCommandBuffer.ParallelWriter  ECB;

    void Execute([ChunkIndexInQuery] int chunkIndex,
                 Entity spawnerEntity,
                 ref Spawner spawner)
    {
        if (spawner.RemainingCount <= 0)
        {
            ECB.DestroyEntity(chunkIndex, spawnerEntity);
            return;
        }

        spawner.TimeLeft -= DeltaTime;
        if (spawner.TimeLeft > 0f) return;

        spawner.TimeLeft = spawner.Interval;
        spawner.RemainingCount--;

        var inst = ECB.Instantiate(chunkIndex, spawner.Prefab);
        ECB.SetComponent(chunkIndex, inst, LocalTransform.FromPosition(spawner.SpawnPoint));
    }
}

[BurstCompile]
public partial struct SpawnerParallelSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var ecb = SystemAPI
            .GetSingleton<EndSimulationEntityCommandBufferSystem.Singleton>()
            .CreateCommandBuffer(state.WorldUnmanaged)
            .AsParallelWriter();

        new SpawnParallelJob
        {
            DeltaTime = SystemAPI.Time.DeltaTime,
            ECB       = ecb
        }.ScheduleParallel();
    }
}

Pasar chunkIndex tanto a Instantiate como al SetComponent posterior garantiza que ambos comandos se ordenen en la misma posición y que la referencia a la entidad diferida de Instantiate se resuelva al destino correcto.

5. Determinismo — qué garantiza y qué no

Garantizado: con el mismo contenido de ECB, la reproducción produce un estado final idéntico, sin importar qué hilo de trabajo grabó cada comando.

No garantizado automáticamente:

  • Los IDs de entidad. Dos ejecuciones pueden generar entidades con valores de Index distintos si los comandos previos fueron diferentes.
  • El orden de los chunks entre systems. Si el system A genera entidades en paralelo y el system B las lee en orden de ChunkIndexInQuery, el orden de iteración aún puede depender del historial de creación de archetypes.

Si hace falta determinismo bit a bit entre ejecuciones (replays, rollback de Netcode), conviene combinar:

  1. Un ECB paralelo con claves de ordenación deterministas.
  2. Systems de paso fijo dentro de FixedStepSimulationSystemGroup.
  3. No depender de Entity.Index; usar IDs de juego propios y controlados.

6. Patrones de clave de ordenación

6.1 Por chunk (la opción por defecto)

Usar [ChunkIndexInQuery] int chunkIndex. Las colisiones no son problema si todas las entidades del chunk generan un comando idéntico (por ejemplo, un chunk entero marcado como aturdido).

6.2 Por entidad

Usar [EntityIndexInQuery] int entityIndex cuando hace falta un orden estricto por entidad (por ejemplo, registrar eventos de daño por entidad que deben reproducirse en orden de entidad).

6.3 Índice determinista personalizado

En contadas ocasiones se construye un índice plano a partir de las entradas, por ejemplo int sortKey = baseOffset + localIndex;. Solo conviene hacerlo cuando las dos opciones habituales no encajan; los índices a medida son una fuente frecuente de bugs de determinismo.

7. Evitar estos patrones

  • Usar JobsUtility.ThreadIndex como sortKey. No es determinista.
  • Construir la clave con Random. No es determinista.
  • Mezclar ECBs. Grabar en dos ECBs distintos hace perder el orden total.
  • Grabar en un ECB de un system desde varios systems. Un ECB, un solo grabador (o un parallel-writer que envuelva un único ECB).

6. Migración de world

Conviene usar DriverMigrationSystem cuando una transición de world debe conservar el estado de la conexión.

using Unity.Entities;
using Unity.NetCode;

public World MigrateServerWorld(World sourceWorld)
{
    DriverMigrationSystem migrationSystem = null;
    foreach (var world in World.All)
    {
        if ((migrationSystem = world.GetExistingSystemManaged<DriverMigrationSystem>()) != null)
            break;
    }

    var ticket = migrationSystem.StoreWorld(sourceWorld);
    sourceWorld.Dispose();

    var newWorld = ClientServerBootstrap.CreateServerWorld("NewServerWorld");
    var newMigrationSystem = newWorld.GetExistingSystemManaged<DriverMigrationSystem>();
    newMigrationSystem.LoadWorld(ticket);
    return newWorld;
}

Capítulo 20. Conexión de red y aprobación en Netcode

1. Descripción general

Netcode for Entities representa cada conexión de red como una entidad. La entidad de conexión lleva NetworkStreamConnection; al terminar la conexión, la entidad se destruye.

5.3 RPC de aprobación

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;

public struct ApprovalRpc : IApprovalRpcCommand
{
    public FixedString512Bytes Token;
}

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation |
    WorldSystemFilterFlags.ThinClientSimulation)]
public partial struct ClientApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (connection, entity) in
            SystemAPI.Query<RefRW<NetworkStreamConnection>>()
                .WithNone<NetworkId>()
                .WithNone<ApprovalSent>()
                .WithEntityAccess())
        {
            var rpcEntity = ecb.CreateEntity();
            ecb.AddComponent(rpcEntity, new ApprovalRpc { Token = "MY_TOKEN" });
            ecb.AddComponent<SendRpcCommandRequest>(rpcEntity);
            ecb.AddComponent<ApprovalSent>(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ServerApprovalSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, rpc, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ApprovalRpc>>()
                .WithEntityAccess())
        {
            var connEntity = req.ValueRO.SourceConnection;

            if (rpc.ValueRO.Token.Equals("MY_TOKEN"))
                ecb.AddComponent<ConnectionApproved>(connEntity);
            else
                ecb.AddComponent<NetworkStreamRequestDisconnect>(connEntity);

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

El tiempo de espera de la aprobación lo controla ClientServerTickRate.HandshakeApprovalTimeoutMS; el valor por defecto es 5000 ms.

Capítulo 21. Snapshot de Ghost y sincronización en Netcode

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Un Ghost es una entidad de red que se simula en el servidor y se replica en los clientes. El servidor envía periódicamente el estado del Ghost en forma de snapshots, y cada cliente los aplica mediante interpolación o predicción.

Mecanismo Uso
Ghost Consistencia eventual sobre paquetes no fiables, con optimización de replicación integrada.
RPC Evento fiable y puntual.

2. Ghost vs RPC

Usar Ghost para Usar RPC para
El estado de entidades espaciales cercanas a un cliente. Eventos de alto nivel del flujo de juego.
Datos que requieren predicción en el cliente. Comandos puntuales, como el chat o los cambios de configuración.
Estado que debe seguir existiendo tras un ciclo de desconexión y reconexión. Eventos sin estado replicado persistente.

3. Authoring de Ghost

Añadir GhostAuthoringComponent al prefab que se quiere replicar.

Propiedad Significado
Name Identificador del Ghost.
Importance Prioridad en el snapshot cuando el ancho de banda es limitado.
Supported Ghost Mode All, Interpolated o Predicted.
Default Ghost Mode Interpolated, Predicted o OwnerPredicted.
Optimization Mode Dynamic por defecto, o Static para objetos que apenas cambian.
MaxSendRate Tasa máxima de replicación en Hz, limitada por NetworkTickRate.
UseSingleBaseline Opción de compresión delta con una sola línea base, introducida en la línea 1.5.0.

3.1 Modos de Ghost

Modo Significado
Interpolated El cliente interpola entre los snapshots del servidor. La imagen es fluida, pero llega con retraso.
Predicted El cliente simula por adelantado para reducir la latencia del input. El coste de CPU es mayor.
OwnerPredicted El cliente propietario predice; los demás clientes interpolan.

7. Sistema de snapshots

7.1 Snapshots parciales

Si un snapshot superara el MTU, Netcode envía primero los Ghosts de mayor importancia. El resto se envía en ticks posteriores, de modo que el estado del world se transmite de forma progresiva.

7.2 NetworkTickRate

Cuando NetworkTickRate es menor que SimulationTickRate, el trabajo de GhostSendSystem se reparte entre varios ticks de simulación.

SimulationTickRate = 60, NetworkTickRate = 30
Tick 1: GhostSendSystem sends a snapshot
Tick 2: GhostSendSystem skips
Tick 3: GhostSendSystem sends a snapshot
...

   ├─ Put the prefab inside the SubScene, or
   └─ Let GhostCollection register the prefab

   └─ Source Generator emits serialization code

4. Runtime
   ├─ Server instantiates the Ghost entity
   ├─ GhostSendSystem sends snapshots
   └─ Client automatically spawns the Ghost

8.1 Prespawned Ghosts

Los Ghosts colocados en una SubScene se convierten en Prespawned Ghosts:

  • El servidor y el cliente emparejan de forma automática la misma entidad al cargar la SubScene.
  • No hace falta un mensaje de spawn aparte; basta con la sincronización de snapshots.
  • A partir de la línea 1.6.1, los IDs de Prespawn Ghost se conservan durante la Host Migration.

Los objetos estáticos del mundo, como edificios y terreno, suelen funcionar mejor como Prespawned Ghosts con Static Optimization Mode.

Capítulo 22. Predicción y rollback en Netcode

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

La predicción en el cliente ejecuta el mismo código de simulación en el propio cliente, antes de que llegue el resultado autoritativo del servidor. Esto reduce la latencia del input y hace que el movimiento del jugador responda al instante.

La predicción tiene un coste real de CPU: cuando llega un snapshot autoritativo, el cliente puede retroceder a un tick anterior y volver a simular hasta el tick predicho actual.

2. Componentes y systems fundamentales

Componente / system Significado
PredictedGhost Se añade a los Ghosts predichos en los clientes y a todos los Ghosts en el servidor.
Simulate Tag habilitable que se activa en las entidades que deben simularse en el tick de predicción actual.
PredictedSimulationSystemGroup Grupo de paso de tiempo fijo que ejecuta el bucle de predicción.
NetworkTime Contiene el tick del servidor, el tick de predicción y otros datos de temporización relacionados.

3. Flujo de predicción en el cliente

1. Receive a server snapshot
   └─ Apply the latest authoritative state to predicted entities

2. Find the oldest applied tick
   └─ Use the oldest tick across affected predicted entities

3. Roll back and re-simulate
   └─ PredictedSimulationSystemGroup repeats from oldest tick to target tick
   └─ TimeData and NetworkTime update for each tick

4. Render
   └─ Present the predicted result

Una latencia alta multiplica el coste. Una conexión de 300 ms puede llegar a necesitar más de 20 fotogramas de resimulación, según la frecuencia de tick y la configuración de predicción.

4. Escribir systems de predicción

4.1 Usar el tag Simulate

Al comienzo del bucle de predicción, Netcode desactiva Simulate en los Ghosts predichos y luego lo vuelve a activar solo en las entidades que deben ejecutarse en el tick actual.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct MovementSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 5f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (transform, input) in
            SystemAPI.Query<RefRW<LocalTransform>, RefRO<PlayerInput>>()
                .WithAll<Simulate>())
        {
            var move = new float3(input.ValueRO.Horizontal, 0, input.ValueRO.Vertical);
            transform.ValueRW.Position += move * speed * dt;
        }
    }
}

Hay que incluir siempre .WithAll<Simulate>() en las queries de predicción. Sin esa cláusula, el system puede volver a simular entidades en ticks que ya están terminados.

4.2 Comportamiento en el servidor

En el servidor, el bucle de predicción se ejecuta exactamente una vez por cada tick del servidor:

  • TimeData contiene los datos normales del tick del servidor.
  • Simulate está siempre activado.
  • El mismo código del system sirve tanto para el cliente como para el servidor.

5. Predicción de jugadores remotos

Si los datos de comandos de otro jugador se serializan, también es posible predecir a ese jugador remoto.

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;

using Unity.Transforms;

6. Suavizado de predicción

El suavizado de predicción atenúa la corrección visible cuando el resultado del servidor no coincide con el predicho.

using Unity.NetCode;
using Unity.Transforms;

GhostPredictionSmoothing.RegisterSmoothingAction<LocalTransform>(
    state.EntityManager,
    CustomSmoothing.Action);

using Unity.Burst;
using Unity.Collections.LowLevel.Unsafe;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[BurstCompile]
public static unsafe class CustomSmoothing
{
    public static readonly PortableFunctionPointer<
        GhostPredictionSmoothing.SmoothingActionDelegate> Action =
        new(SmoothingAction);

    [BurstCompile(DisableDirectCall = true)]
    private static void SmoothingAction(
        void* currentData, void* previousData, void* userData)
    {
        ref var current = ref UnsafeUtility.AsRef<LocalTransform>(currentData);
        ref var backup = ref UnsafeUtility.AsRef<LocalTransform>(previousData);

        var dist = math.distance(current.Position, backup.Position);
        if (dist > 0)
        {
            current.Position = math.lerp(backup.Position, current.Position, 0.5f);
        }
    }
}

7. Cambio de predicción

La predicción es costosa, así que un Ghost puede alternar de forma dinámica entre Predicted e Interpolated.

7.1 Requisitos previos para el cambio

  • Los Supported Ghost Modes deben ser All.
  • El modo actual no debe ser OwnerPredicted.
  • El Ghost no debe estar ya en proceso de cambio.

7.2 Poner en cola los cambios

using Unity.NetCode;

ref var switchQueues = ref SystemAPI
    .GetSingletonRW<GhostPredictionSwitchingQueues>()
    .ValueRW;

switchQueues.ConvertToPredictedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 1f,
});

switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
{
    TargetEntity = entity,
    TransitionDurationSeconds = 0f,
});

Los Ghosts interpolados y los predichos corren en líneas de tiempo distintas, a menudo separadas por unos dos intervalos de ping. SwitchPredictionSmoothing interpola la posición y la rotación durante la duración configurada para evitar teletransportes visibles.

8. Controles de rendimiento de predicción

Técnica Efecto
Prediction Switching Pasar a Interpolated los Ghosts lejanos o de baja importancia.
Restringir el acceso de escritura a GhostField Usar acceso de solo lectura para los campos que no cambian dentro del bucle de predicción.
Agrupación de física Usar MaxPredictionStepBatchSizeRepeatedTick para la resimulación repetida de la física.
Ajuste de tick-rate Bajar SimulationTickRate reduce el número de resimulaciones, a costa de la capacidad de respuesta.

1. Descripción general

Cuando un cliente está in-game, envía al servidor un flujo de comandos continuo. Ese flujo incluye los datos de input y los datos de ACK de snapshot, y sigue activo aunque el jugador no esté pulsando ninguna tecla, para que la conexión se mantenga sincronizada.

3. IInputComponentData

3.1 Definir el input

using Unity.NetCode;

public struct PlayerInput : IInputComponentData
{
    public int Horizontal;
    public int Vertical;
    public InputEvent Jump;
}

InputEvent conserva los eventos puntuales como GetKeyDown para que se registren exactamente una vez en el servidor, aunque la red descarte un paquete de comando redundante.

3.2 Recoger el input en el cliente

using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[UpdateInGroup(typeof(GhostInputSystemGroup))]
public partial struct GatherInputSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<PlayerInput>();
    }

    public void OnUpdate(ref SystemState state)
    {
        bool jump = Input.GetKeyDown(KeyCode.Space);
        bool left = Input.GetKey(KeyCode.A);
        bool right = Input.GetKey(KeyCode.D);

        foreach (var input in
            SystemAPI.Query<RefRW<PlayerInput>>()
                .WithAll<GhostOwnerIsLocal>())
        {
            input.ValueRW = default;
            if (jump) input.ValueRW.Jump.Set();
            if (left) input.ValueRW.Horizontal -= 1;
            if (right) input.ValueRW.Horizontal += 1;
        }
    }
}

3.3 Procesar el input en el servidor y en el bucle de predicción

using Unity.Burst;
using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[UpdateInGroup(typeof(PredictedSimulationSystemGroup))]
public partial struct ProcessInputSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<PlayerInput>();
    }

    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        var speed = 3f;
        var dt = SystemAPI.Time.DeltaTime;

        foreach (var (input, transform) in
            SystemAPI.Query<RefRO<PlayerInput>, RefRW<LocalTransform>>()
                .WithAll<Simulate>())
        {
            if (input.ValueRO.Jump.IsSet)
            {
                // Jump logic; runs once on the input tick.
            }

            transform.ValueRW.Position +=
                new float3(input.ValueRO.Horizontal, 0, 0) * speed * dt;
        }
    }
}

1. Descripción general

Un RPC es un mensaje fiable y puntual entre cliente y servidor. El código de serialización y deserialización del RPC se genera de forma automática, y su gestión sigue encajando en el marco de ECS y el Job System.

Los RPCs no sustituyen a la replicación de Ghosts ni a los flujos de comandos. Conviene usarlos para eventos puntuales del flujo de juego, no para estado de alta frecuencia.

2. RPC vs Ghost vs Command

RPC Ghost Command
Comportamiento de transporte Fiable, ordenado. No fiable con optimización de replicación. No fiable con envíos redundantes.
Dirección Bidireccional. Servidor → Cliente. Cliente → Servidor.
Frecuencia Puntual. En cada tick de red, según haga falta. En cada tick de simulación.
Uso Flujo de juego, lobby, chat. Estado replicado de entidades. Input del jugador.
Límite en vuelo Sí, el límite del pipeline fiable. No. No.

Como los RPCs son ordenados y fiables, un tráfico de RPC frecuente puede quedar atascado detrás de los paquetes en vuelo.

4. Enviar RPCs

4.1 Cliente a servidor

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation)]
public partial struct SendChatSystem : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.RequireForUpdate<NetworkId>();
    }

    public void OnUpdate(ref SystemState state)
    {
        if (!Input.GetKeyDown(KeyCode.Space))
            return;

        var ecb = new EntityCommandBuffer(Allocator.Temp);
        var entity = ecb.CreateEntity();

        ecb.AddComponent(entity, new ChatMessageRpc
        {
            Message = "Hello!",
            SenderId = SystemAPI.GetSingleton<NetworkId>().Value,
        });

        ecb.AddComponent<SendRpcCommandRequest>(entity);
        ecb.Playback(state.EntityManager);
    }
}

ISystem es un struct. El estado que persiste de un fotograma a otro no debe guardarse en campos de instancia del system, sino en componentes o singletons.

4.2 Servidor a un cliente

using Unity.Entities;
using Unity.NetCode;

Entity targetConnectionEntity = default;

var entity = ecb.CreateEntity();
ecb.AddComponent(entity, new NotifyRpc());
ecb.AddComponent(entity, new SendRpcCommandRequest
{
    TargetConnection = targetConnectionEntity,
});

4.3 Difusión del servidor

using Unity.Entities;
using Unity.NetCode;

var entity = ecb.CreateEntity();
ecb.AddComponent(entity, new GameStartRpc());
ecb.AddComponent(entity, new SendRpcCommandRequest
{
    TargetConnection = Entity.Null,
});

Entity.Null difunde el RPC.

5. Recibir RPCs

using Unity.Collections;
using Unity.Entities;
using Unity.NetCode;
using UnityEngine;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
public partial struct ReceiveChatSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        foreach (var (req, chat, entity) in
            SystemAPI.Query<RefRO<ReceiveRpcCommandRequest>, RefRO<ChatMessageRpc>>()
                .WithEntityAccess())
        {
            var sourceConnection = req.ValueRO.SourceConnection;
            Debug.Log($"Player {chat.ValueRO.SenderId}: {chat.ValueRO.Message}");

            ecb.DestroyEntity(entity);
        }

        ecb.Playback(state.EntityManager);
    }
}

Hay que destruir siempre la entidad del RPC recibido tras procesarlo. Si no, el mismo RPC vuelve a procesarse en cada fotograma.

6. RpcQueue

RpcQueue solo se usa cuando hace falta programar los RPC manualmente.

using Unity.Entities;
using Unity.NetCode;

var rpcQueue = SystemAPI.GetSingleton<RpcCollection>()
    .GetRpcQueue<MyRpc, MyRpc>();
var bufferLookup = SystemAPI.GetBufferLookup<OutgoingRpcDataStreamBuffer>();

if (bufferLookup.HasBuffer(connectionEntity))
{
    var buffer = bufferLookup[connectionEntity];
    rpcQueue.Schedule(buffer, new MyRpc());
}

7. Serialización manual

La generación automática es la opción por defecto. Si se desactiva, la serialización y la deserialización deben ser simétricas.

using Unity.Burst;
using Unity.Entities;
using Unity.NetCode;
using Unity.Networking.Transport;

[BurstCompile]
public struct MyCustomRpc : IComponentData, IRpcCommandSerializer<MyCustomRpc>
{
    public int Data;

    public void Serialize(ref DataStreamWriter writer, in RpcSerializerState state,
        in MyCustomRpc data)
    {
        writer.WriteInt(data.Data);
    }

    public void Deserialize(ref DataStreamReader reader, in RpcDeserializerState state,
        ref MyCustomRpc data)
    {
        data.Data = reader.ReadInt();
    }

    // Execute implementation omitted.
}

public struct LoginApproval : IApprovalRpcCommand

Capítulo 25. Optimización de Ghost, importancia y relevancia en Netcode

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Los worlds multijugador grandes no pueden enviar todos los Ghosts a todos los clientes en cada tick. Netcode for Entities recurre a Optimization Mode, Importance Scaling, Relevancy y a controles de tasa de envío por Ghost para reducir el coste de CPU y el ancho de banda.

La regla básica es sencilla: el servidor debe gastar el presupuesto de snapshot en los Ghosts que, en ese momento, le importan a una conexión concreta.

2. Optimization Mode

Optimization Mode se configura en GhostAuthoringComponent.

Modo Significado Adecuado para
Dynamic El modo por defecto. Optimiza el tamaño del snapshot tanto si los datos cambian como si no. Ghosts que se mueven con normalidad.
Static Evita reenviar datos cuando el Ghost no ha cambiado. Edificios, terreno, props estáticos.

La línea 1.11.0 incorporó una optimización para Ghosts estáticos que puede reducir bastante el tiempo de GhostUpdateSystem en los Ghosts estáticos que no han cambiado.

3. Límites de tamaño de snapshot

El presupuesto de snapshot se puede limitar en varios niveles.

Configuración Significado
NetworkStreamSnapshotTargetSize Objetivo de bytes orientativo por conexión.
GhostSendSystemData.MaxSendEntities Máximo de entidades por snapshot.
GhostSendSystemData.MaxSendChunks Máximo de chunks por snapshot.

Cuando el presupuesto es limitado, se envían primero los Ghosts de mayor importancia.

4. Importance Scaling

Importance Scaling calcula en el servidor la prioridad de envío de cada Ghost.

4.1 Entradas

GhostImportance combina tres tipos de datos.

Datos Significado
Por conexión Datos propios de la conexión, como la posición del jugador.
Datos singleton Configuración global, como el tamaño del tile.
Datos compartidos por chunk Parámetros de escala a nivel de chunk.

4.2 Importancia basada en distancia

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

var gridSingleton = state.EntityManager.CreateSingleton(
    new GhostDistanceData
    {
        TileSize = new int3(50, 50, 256),
        TileCenter = new int3(0, 0, 128),
        TileBorderWidth = new float3(1f, 1f, 1f),
    });

state.EntityManager.AddComponentData(gridSingleton, new GhostImportance
{
    ScaleImportanceFunction = GhostDistanceImportance.ScaleFunctionPointer,
    GhostConnectionComponentType = ComponentType.ReadOnly<GhostConnectionPosition>(),
    GhostImportanceDataType = ComponentType.ReadOnly<GhostDistanceData>(),
    GhostImportancePerChunkDataType = ComponentType.ReadOnly<GhostDistancePartitionShared>(),
});

Cambio que rompe compatibilidad en la línea 1.8.0: las funciones de escala de GhostDistanceImportance ya no multiplican baseImportance por 1000. El código de importancia personalizada que daba por hecho el multiplicador antiguo hay que revisarlo.

4.3 GhostDistancePartitionShared automático

using Unity.NetCode;

GhostDistancePartitioningSystem.AutomaticallyAddGhostDistancePartitionSharedComponent = true;

4.4 Importance Visualizer

Desde la línea 1.8.0, la PlayMode Tool incluye un Importance Visualizer. Conviene usarlo para examinar la distribución de importancia en tiempo de ejecución en vez de deducirla del código.

5. Ghost Relevancy

Ghost Relevancy permite que el servidor oculte o muestre Ghosts concretos según la conexión.

Modo Significado
Disabled El modo por defecto. Sin filtrado; todos los Ghosts son candidatos.
SetIsRelevant Solo se replican a esa conexión los Ghosts de la lista.
SetIsIrrelevant Se excluyen de esa conexión los Ghosts de la lista.

5.1 Casos de uso

Escenario Idea de implementación
Descarte por distancia Marcar como irrelevantes los Ghosts lejanos.
Niebla de guerra Ocultar entidades fuera del campo de visión.
Ghosts propios de cada cliente Marcar un Ghost como relevante solo para un cliente.
Filtrado por equipo Ocultar datos internos que solo conoce el enemigo.

5.2 Ejemplo de relevancia

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;
using Unity.Transforms;

[WorldSystemFilter(WorldSystemFilterFlags.ServerSimulation)]
[UpdateInGroup(typeof(GhostSimulationSystemGroup))]
[UpdateBefore(typeof(GhostSendSystem))]
public partial struct RelevancySystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var relevancy = SystemAPI.GetSingletonRW<GhostRelevancy>();

        relevancy.ValueRW.GhostRelevancyMode = GhostRelevancyMode.SetIsRelevant;
        relevancy.ValueRW.GhostRelevancySet.Clear();

        foreach (var (ghostInstance, transform) in
            SystemAPI.Query<RefRO<GhostInstance>, RefRO<LocalTransform>>())
        {
            foreach (var (connId, connPos) in
                SystemAPI.Query<RefRO<NetworkId>, RefRO<GhostConnectionPosition>>())
            {
                float dist = math.distance(
                    transform.ValueRO.Position,
                    connPos.ValueRO.Position);

                if (dist < 100f)
                {
                    relevancy.ValueRW.GhostRelevancySet.TryAdd(
                        new RelevantGhostForConnection(
                            connId.ValueRO.Value,
                            ghostInstance.ValueRO.ghostId),
                        1);
                }
            }
        }
    }
}

Este ejemplo simplificado es O(Ghosts × conexiones). En worlds grandes es preferible la partición de importancia por distancia y reservar la relevancia para reglas que la partición no puede expresar, como la niebla de guerra o la visibilidad por equipos.

5.3 DefaultRelevancyQuery

DefaultRelevancyQuery define los chunks de Ghost que siempre son relevantes para todas las conexiones, como los Ghosts de estado global del juego o el estado de red de la UI.

5.4 Relevancia más importancia

Desde la línea 1.6.1 se pueden combinar Ghost Relevancy e Importance Scaling mediante PrioChunks.isRelevant, lo que habilita una ruta de relevancia más rápida.

6. Optimización de la CPU de predicción

6.1 Prediction Switching

Pasar de Predicted a Interpolated los Ghosts lejanos o poco importantes.

using Unity.Entities;
using Unity.Mathematics;
using Unity.NetCode;

float distance = math.distance(playerPos, ghostPos);
if (distance > predictionRange)
{
    switchQueues.ConvertToInterpolatedQueue.Enqueue(new ConvertPredictionEntry
    {
        TargetEntity = ghostEntity,
        TransitionDurationSeconds = 0.5f,
    });
}

6.2 Minimizar el acceso de escritura a GhostField

Los jobs con acceso de escritura a componentes GhostField pueden disparar comprobaciones de serialización. Conviene usar acceso de solo lectura para los campos que no cambian dentro del bucle de predicción.

6.3 Resimulación de física

Para casos de alta latencia que necesitan más de 20 fotogramas de resimulación predicha:

  • Desactivar la física multihilo mediante el singleton PhysicsStep cuando salga más barato consolidar en el hilo principal.
  • Usar MaxPredictionStepBatchSizeRepeatedTick para agrupar los ticks de física repetidos.

7. MaxIterateChunks

GhostSendSystemData.MaxIterateChunks limita el número de chunks que se procesan en un solo tick.

Configuración Significado
MaxIterateChunks Limita el coste de CPU de la serialización por tick.
MaxSendChunks Limita el ancho de banda al acotar cuántos chunks entran en un snapshot.

Estos parámetros resuelven problemas distintos: MaxIterateChunks reparte el trabajo de CPU, mientras que MaxSendChunks limita el tamaño del snapshot.

8. MaxSendRate

GhostAuthoringComponent.MaxSendRate limita la frecuencia de envío de un Ghost en Hz.

NetworkTickRate = 60, MaxSendRate = 10
→ This Ghost can be included once every 6 network ticks.

Es la opción para Ghosts de baja frecuencia o de poca importancia.

9. UseSingleBaseline

GhostAuthoringComponent.UseSingleBaseline = true usa una sola línea base para la compresión delta. Esto puede reducir el coste de CPU, posiblemente a cambio de más ancho de banda.

10. Lista de verificación de optimización

  • Usar el modo de optimización Static para Ghosts que apenas cambian.
  • Definir NetworkStreamSnapshotTargetSize para limitar el ancho de banda por conexión.
  • Implementar Importance Scaling basado en distancia.
  • Aplicar Relevancy solo donde la partición no pueda expresar la regla.
  • Pasar los Ghosts lejanos al modo Interpolated.
  • Usar MaxSendRate para Ghosts de baja frecuencia.
  • Evitar el acceso de escritura a datos de GhostField que no han cambiado.
  • Inspeccionar la prioridad en tiempo de ejecución con el Importance Visualizer.

Síntoma Causa / Solución

Capítulo 26. Integración de física y compensación de lag en Netcode

Unity 6000.5 · Entities 6.5.0 · Netcode for Entities 6.5.0

1. Descripción general

Netcode for Entities se integra con Unity Physics para la simulación de física en red. El comportamiento físico depende de si el Ghost es Interpolated o Predicted.

Para que se ejecute la física predicha, en la escena debe existir al menos un Ghost Predicted.

2. Física interpolada vs. predicha

2.1 Ghosts Interpolated en clientes

  • La posición y la rotación provienen de los snapshots del servidor.
  • Simulate está desactivado, así que el objeto físico se comporta como kinematic.
  • PhysicsVelocity se ignora.

2.2 Ghosts Predicted en clientes

  • La física corre dentro del bucle de predicción y puede ejecutarse varias veces por fotograma renderizado.
  • PhysicsSystemGroup se ejecuta dentro de PredictedFixedStepSimulationSystemGroup.
  • En la física predicha, se espera que el cliente obtenga el mismo resultado de simulación que el servidor.
Netcode initialization
├─ PhysicsSystemGroup moves under PredictedFixedStepSimulationSystemGroup
└─ FixedStepSimulationSystemGroup moves under PredictedFixedStepSimulationSystemGroup

3. Rendimiento de la física predicha

La física predicha es costosa porque el rollback puede ejecutarla varias veces seguidas.

3.1 Agrupamiento en el servidor

using Unity.NetCode;

ClientServerTickRate tickRate = default;
tickRate.MaxSimulationStepBatchSize = 4;
tickRate.MaxSimulationStepsPerFrame = 4;

3.2 Agrupamiento en el cliente

using Unity.NetCode;

ClientTickRate clientRate = default;
clientRate.MaxPredictionStepBatchSizeFirstTimeTick = 2;
clientRate.MaxPredictionStepBatchSizeRepeatedTick = 4;

El agrupamiento puede aumentar el riesgo de errores de predicción, así que conviene usarlo como ajuste de rendimiento puntual, no como valor por defecto.

3.3 Cuantización

La cuantización por defecto de Transform y Velocity para los Ghosts físicos es 1000.

Mayor cuantización Menor cuantización
Simulación más precisa. Menor ancho de banda.
Mayor ancho de banda. Mayor pérdida de precisión.

4. Compensación de lag

Como los clientes predicen por delante del servidor, el servidor autoritativo a veces necesita consultar un mundo de colisiones más antiguo, el que corresponde al momento en que el cliente actuó.

4.1 Habilitar la compensación de lag

Añadir NetCodePhysicsConfig a una SubScene y poner EnableLagCompensation = true.

4.2 Consultar un mundo de colisiones antiguo

using Unity.NetCode;
using Unity.Physics;

var physicsHistory = SystemAPI.GetSingleton<PhysicsWorldHistorySingleton>();

physicsHistory.GetCollisionWorldFromTick(
    serverTick,
    interpolationDelay,
    ref physicsWorld,
    out var collisionWorld);

// Use collisionWorld for raycasts or other physics queries.

4.3 Clonación profunda de Collider

Parámetro Valor predeterminado Significado
DeepCopyDynamicColliders true Clonación profunda de los colliders dinámicos; activada por defecto en 6.5.0.
DeepCopyStaticColliders false Clonación profunda de los colliders estáticos.

La línea 6.5.0 activa por defecto la clonación profunda de los colliders de Ghosts dinámicos para evitar aserciones de blob asset.

4.4 Búfer de historial de física

Versión Tamaño del historial
Hasta 1.4.x 16 ticks.
1.5.0+ 32 ticks.
Personalizado GhostSystemConstants.SnapshotHistorySize mediante una define del compilador.

5. Multi Physics World

Se usa un mundo de física aparte para los efectos exclusivos del cliente, como los VFX o las partículas, que no participan en la simulación en red.

5.1 Configuración

  1. Añadir NetCodePhysicsConfig a la SubScene.
  2. Poner Client Non Ghost World en un valor distinto de cero, como 1.
  3. Añadir PhysicsWorldIndex con ese mismo valor a los objetos físicos exclusivos del cliente.

5.2 Grupo de systems de física personalizado

using Unity.Entities;
using Unity.NetCode;
using Unity.Physics.Systems;

[WorldSystemFilter(WorldSystemFilterFlags.ClientSimulation)]
[UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
public partial class VfxPhysicsGroup : CustomPhysicsSystemGroup
{
    public const int WorldIndex = 1;

    public VfxPhysicsGroup() : base(WorldIndex, shareStaticColliders: true)
    {
    }
}

Con shareStaticColliders: true, el mundo de física personalizado comparte los colliders estáticos con el mundo de física principal.

6. Physics Proxy

El Physics Proxy se usa cuando un Ghost Predicted debe interactuar con una entidad de física exclusiva del cliente.

  • CustomPhysicsProxyAuthoring crea la entidad proxy.
  • CustomPhysicsProxyDriver vincula el proxy al Ghost Predicted.
  • SyncCustomPhysicsProxySystem sincroniza posición y rotación.

7. PredictedFixedStepSimulationSystemGroup

PredictedFixedStepSimulationSystemGroup solo se actualiza cuando se cumplen todas las condiciones necesarias.

Requisito Significado
El singleton NetworkStreamInGame existe La conexión está en juego.
Existe al menos un Ghost Predicted La física predicha tiene algo que simular.
La tasa de ticks fija está en marcha La simulación de paso de tiempo fijo está activa.

Mientras una conexión de red no esté in-game o no hayan llegado los Ghosts Predicted, la física predicha no se ejecuta.

7.1 Física antes de la conexión

Si la física debe ejecutarse antes de que exista una conexión de red, hay que desactivar la configuración de física predicha para que pueda correr la física normal de FixedStepSimulationSystemGroup.

using Unity.Entities;
using Unity.NetCode;

[UpdateInGroup(typeof(InitializationSystemGroup))]
[CreateAfter(typeof(PredictedPhysicsConfigSystem))]
public partial struct DisablePredictedPhysics : ISystem
{
    public void OnCreate(ref SystemState state)
    {
        state.World.GetExistingSystemManaged<PredictedPhysicsConfigSystem>()
            .Enabled = false;
    }
}

Capítulo 28. Disposición de chunks y TypeManager

1. El chunk de 16 KB

Cada archetype guarda sus entidades en chunks de 16 KB. El chunk contiene:

  1. Una pequeña cabecera (metadatos: ID del archetype, números de versión, máscaras de bits de habilitación y versiones de cambio).
  2. Una región SoA contigua: una columna por cada tipo de componente, y cada columna es un array de valores.
  3. Relleno hasta el límite de los 16 KB.

La capacidad del chunk, es decir, cuántas entidades caben, se deduce de esta fórmula:

capacity ≈ (16 KB − header) / sum(sizeof(component_i))

Un archetype ajustado, con pocos componentes pequeños, puede albergar cientos de entidades. Uno ancho, con muchos componentes grandes, solo unas pocas. Los chunks poco ocupados desperdician los bytes que podrían haber alojado más entidades.

2. TypeManager — el registro de componentes

TypeManager registra cada componente, buffer y system al arrancar el world y le asigna un índice de tipo estable. De ese registro, el runtime lee:

Campo Para qué se usa
TypeIndex Identificador compacto que se usa en las queries y en los metadatos del chunk.
SizeInChunk Bytes que ocupa este componente por entidad dentro de un chunk.
Alignment Alineación de la columna dentro del chunk.
ElementSize Tamaño de cada elemento del buffer (para IBufferElementData).
HasBlobReferences / HasEntityReferences Activa la validación durante la serialización y las operaciones de movimiento.

Rara vez se accede directamente a TypeManager. Lo importante es que añadir nuevos tipos de componente, renombrarlos o moverlos entre ensamblados dispara una reconstrucción del registro de tipos, y ahí se va el tiempo de arranque en los proyectos grandes.

En Entities 1.4 se anunció que TypeManager.Initialize pasó a ser unas 2 veces más rápido en proyectos grandes al evitar escanear ensamblados innecesarios. La versión 6.5 hereda esa optimización.

3. Inspeccionar la disposición de los chunks: la ventana Archetypes

Window → Entities → Archetypes lista todos los archetypes del world. Para cada fila:

  • Entities — número de entidades en vivo.
  • Chunks — número de chunks en uso.
  • Entities / Chunk — ocupación media.
  • Capacity — máximo teórico por chunk.

Qué vigilar:

Lectura Interpretación
Muchos archetypes con 1 entidad por chunk Explosión de archetypes por tags demasiado granulares. Conviene consolidarlos.
Un archetype con muchas entidades y alta ocupación Es lo deseable: enemigos, proyectiles, balas.
Entities / Chunk ≪ Capacity Fragmentación: la ocupación es irregular porque las entidades no dejan de migrar a otros archetypes (cambios estructurales).
Capacity = 1 El archetype es demasiado ancho (componentes demasiado grandes). Revisar si hay componentes managed o shared sobredimensionados.

4. Diseñar componentes que respeten los archetypes

Reglas prácticas, en orden aproximado de impacto:

  1. Mantener no gestionados los structs no gestionados. Un solo IComponentData gestionado obliga a almacenar los datos en una tabla aparte por entidad y rompe Burst.
  2. No dejar relleno en los structs. Poner float3 antes que float, no después: los huecos de alineación inflan SizeInChunk.
  3. Separar los campos de uso poco frecuente. Si solo un system lee un campo, conviene moverlo a un componente aparte. Las columnas que no se usan siguen costando bytes por entidad.
  4. Preferir componentes habilitables a añadir y quitar componentes en cada fotograma. Ver 13_Structural Change & Safety.md: conmutar un bit es mejor que reorganizar el archetype.
  5. Limitar el conjunto de valores de los componentes shared. Cada valor nuevo crea un bucket de chunk nuevo.
  6. Los tag components cuestan cero bytes, pero aun así añaden un archetype. No conviene etiquetar cada entidad con un marcador único.

5. Números de versión del chunk

Cada columna de componente lleva una versión de cambio: un contador de 32 bits que se incrementa cada vez que un system escribe mediante RefRW<T>.ValueRW, SetComponentData o el SetComponent de un ECB.

Los systems pueden saltarse los chunks cuya versión no ha cambiado desde la última ejecución:

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>());
query.SetChangedVersionFilter(typeof(Health));

Cuando una query se recorre con un filtro de cambio, solo se visitan los chunks modificados desde la última ejecución del system. Así es como un patrón del tipo "al cambiar Health, actualizar la UI" sale barato.

Advertencias:

  • El filtro trabaja a nivel de chunk, no de entidad. La escritura de una sola entidad ensucia el chunk entero.
  • Leer mediante RefRW<T>.ValueRW, aunque no se modifique nada, incrementa la versión.
  • Conmutar un componente habilitable también incrementa la versión.

6. Almacenamiento de BlobAsset

BlobAssetReference<T> guarda datos grandes y de solo lectura (mallas, tablas de LOD, campos de navegación) fuera del chunk. El chunk solo guarda un puntero. Así se evita reducir la capacidad del chunk cuando muchas entidades comparten los mismos datos horneados.

Consejo práctico: si un componente está dominado por un FixedList o un DynamicBuffer que es idéntico en todas las entidades de un mismo tipo, conviene guardar esa carga grande en un blob asset al que apunte un componente ligero.

7. Ejemplos

7.1 Un archetype ancho que conviene evitar

public struct EnemyState : IComponentData
{
    public float3 Position;
    public quaternion Rotation;
    public float Health;
    public float HealthMax;
    public float Stamina;
    public float StaminaMax;
    public int AIState;
    public float3 LastKnownPlayerPos;
    public float TargetDistance;
    public Entity Target;
    public FixedList512Bytes<float3> PathBuffer;   // 512 bytes per entity!
}

El campo PathBuffer consume él solo 512 bytes por entidad. La capacidad de cada chunk de 16 KB cae a unas pocas decenas. Conviene dividirlo:

public struct EnemyStats : IComponentData
{
    public float Health, HealthMax, Stamina, StaminaMax;
    public int   AIState;
    public Entity Target;
}

public struct EnemyPathPoint : IBufferElementData
{
    public float3 Value;
}

Ahora el buffer dinámico vive en su propio almacenamiento y la densidad del chunk se recupera.

7.2 Filtro de cambio para actualizar la UI a bajo coste

var query = state.GetEntityQuery(ComponentType.ReadWrite<Health>(),
                                 ComponentType.ReadWrite<HealthBarLink>());
query.SetChangedVersionFilter(typeof(Health));

state.Dependency = new UpdateHealthBarJob { /* ... */ }
    .ScheduleParallel(query, state.Dependency);

Solo se recorren los chunks en los que Health ha cambiado.

8. Solución de problemas

Síntoma Causa / Solución
Inicio lento en proyecto grande TypeManager.Initialize escanea demasiados ensamblados. Revisar la traza de arranque del Profiler y comprobar que se usa una versión reciente de Entities.
Capacidad de 1 en un archetype El componente es demasiado grande (capacidad del buffer, blob embebido). Trasladar los datos voluminosos a un BlobAssetReference<T> o a un IBufferElementData.
El filtro de cambio se dispara en cada fotograma aunque "no haya cambiado nada" Algún código lee mediante RefRW<T>.ValueRW sin escribir. Cambiar a RefRO<T> donde se pueda.
El número de archetypes no deja de crecer Tags por entidad de alta cardinalidad, o componentes shared con valores continuos. Consolidarlos.
Entities / Chunk muy por debajo de Capacity Cambios estructurales: las adiciones y eliminaciones dispersan las entidades entre archetypes. Cambiar a componentes habilitables.
Un componente managed acapara el perfil de CPU Trasladar el campo managed a un UnityObjectRef<T> o dividir el componente; las rutas no gestionadas son mucho más rápidas.

Funciona como la Hierarchy en vivo, pero con entidades en lugar de GameObjects.

Inspector

La edición en vivo es de escritura directa: cambiar un campo incrementa al instante la versión de cambio de ese chunk.

3. Systems

La ventana Systems lista todos los systems registrados, agrupados por su SystemGroup, en el orden real en que los ejecuta el planificador.

Columnas que conviene leer primero

Columna Significado
Name El tipo del system.

Qué buscar

Lectura Interpretación
Un system muestra 0 ms pero aparece en verde Lo omite RequireForUpdate<T> o [RequireMatchingQueriesForUpdate]. Es correcto.
Un system muestra siempre un valor de ms alto Es trabajo real. Profundizar en el Profiler para ver el desglose.

  • Dependencies — los systems sobre los que este tiene [UpdateBefore]/[UpdateAfter], ya resueltos al orden real de ejecución.

4. Ventana Query

Diseñada a medida para responder preguntas del tipo "¿qué coincide con X?".

Flujo de trabajo:

  1. Añadir componentes All con el botón de más.
  2. Añadir Any / None si hace falta.
  3. La ventana lista las entidades coincidentes en el panel izquierdo y los systems que ejecutan esta query en el panel derecho.

Casos de uso:

  • "¿Qué systems leen Health?" → añadir Health a All; todos los systems listados a la derecha lo usan.
  • "¿Por qué esta entidad no coincide con la query de mi system?" → pegar la query del system y comparar los componentes presentes con los de la entidad en la ventana Hierarchy.
  • "¿Se cuelan prefabs en esta query?" → los prefabs se distinguen por un icono en la lista de resultados (añadido en 1.4+).

La ventana Query también permite guardar una query y reutilizarla durante la sesión.

5. Journaling — el diagnóstico de último recurso

Window → Analysis → Entities Journaling (que se activa en Project Settings → Entities → Journaling) registra en un buffer circular cada cambio estructural, cada escritura de componente y cada actualización de system. Vistas de reproducción:

  • Operations — cada cambio registrado, con su marca de tiempo, el system de origen y la entidad afectada.
  • Entities — una línea de tiempo por entidad de quién la modificó y cuándo.

El Journaling tiene un coste en tiempo de ejecución: conviene activarlo solo mientras se investiga un error y dejarlo desactivado durante el desarrollo normal.

6. Flujo de trabajo práctico — depurar una actualización que no llega

Error habitual: "Mi system procesó la entidad pero el componente no cambió". Procedimiento:

  1. Entities Hierarchy → encontrar la entidad, confirmar que tiene el componente.
  2. Inspector → antes de Play, anotar el valor. Pulsar Play y observarlo.
  3. Si el valor no cambia: ventana Systems → localizar el system implicado y comprobar Time (ms). Si es 0, se está omitiendo.
  4. Si el tiempo es mayor que 0 pero el valor sigue sin cambiar: ventana Query → pegar la query del system y los componentes de la entidad. Comprobar que la entidad coincide.
  5. Si aún no se resuelve: activar el Journaling durante 1 o 2 fotogramas. Buscar escrituras en ese componente desde otros systems que pisen el resultado.

Síntoma Causa / Solución
Entities Hierarchy está vacía en modo Play El selector de world apunta a un world equivocado (por ejemplo, solo Client, sin Server).
Un campo en vivo no se actualiza Se está viendo una entidad cacheada tras una recarga de dominio. Volver a seleccionarla.

Capítulo 30. Profiler · Análisis de cuellos de botella

1. Descripción general

El Unity Profiler (Window → Analysis → Profiler) es la herramienta principal para responder, en DOTS, a la pregunta de "¿por qué va lento este fotograma?". Entities emite suficientes marcadores como para atribuir el tiempo a systems, jobs y chunks concretos, una vez que se sabe dónde mirar.

Esta página explica cómo leer el perfil, cuáles son los patrones típicos de cuello de botella en DOTS y cómo actuar ante ellos.

2. Antes de analizar el rendimiento

  1. Usar Deep Profile con moderación. Es útil para seguir una ruta gestionada concreta, pero distorsiona los tiempos: nunca hay que fiarse de Deep Profile para las cifras de rendimiento.
  2. Activar "Record in Editor & Standalone" para capturar también las builds, no solo los fotogramas del Editor.
  3. Apuntar el Profiler al player, no al Editor, al medir el rendimiento de producción. La sobrecarga del Editor es notable.
  4. Dejar correr el juego unos segundos antes de leer los datos; los primeros fotogramas incluyen la recarga de dominio y el trabajo de JIT.

3. Interpretación del módulo CPU Usage

Marcadores de nivel superior dentro de un fotograma de DOTS:

PlayerLoop
├── Initialization.*
├── Update.ScriptRunBehaviourUpdate              (MonoBehaviour.Update)
├── SimulationSystemGroup                        ← DOTS simulation lives here
│   ├── SystemA.OnUpdate
│   ├── SystemB.OnUpdate
│   └── EndSimulationEntityCommandBufferSystem.Playback
├── FixedStepSimulationSystemGroup
├── PresentationSystemGroup
├── Render
└── WaitForTargetFPS

Al desplegar un marcador SystemA.OnUpdate se ven sus subllamadas: ScheduleParallel, esperas de finalización de jobs, llamadas a EntityManager.

4. Dónde se va realmente el tiempo: patrones habituales

4.1 Un OnUpdate que acapara el hilo principal

Si un solo SystemX.OnUpdate acumula más de 10 ms, el trabajo se está ejecutando en el hilo principal. Conviene comprobar:

  • ¿Usa SystemAPI.Query en un bucle compacto que debería ser un IJobEntity.ScheduleParallel?
  • ¿Es un SystemBase que un campo gestionado obliga a ejecutar en el hilo principal?
  • ¿Se está llamando a EntityManager.AddComponent/RemoveComponent entidad por entidad?

4.2 Tiempos de jobs

Los jobs aparecen en la vista Timeline (o bajo "Workers" en la vista Hierarchy). Conviene buscar:

  • Una lane de job totalmente saturada mientras otras están inactivas: el trabajo no se reparte entre los chunks.
  • El hilo principal esperando un job (marcador JobHandle.Complete()): se ha llamado a .Complete() demasiado pronto; conviene retrasarlo dentro del fotograma.
  • Nombres de función de Burst muy largos: es normal; el símbolo es la instanciación genérica completa.

4.3 Puntos calientes de EntityManager

Conviene buscar marcadores con estos nombres:

  • EntityManager.SetArchetype / Move Entity: exceso de cambios estructurales. Ver 13_Structural Change & Safety.md.
  • EntityCommandBufferSystem.OnUpdate (Playback): una reproducción de ECB costosa. Suele significar que se registran demasiados comandos por fotograma.
  • TypeManager.Initialize: solo al arrancar; si aparece en cada fotograma, algo está recargando el world.

4.4 Marcadores GC.Alloc

Cualquier GC.Alloc dentro de la simulación es una regresión en DOTS. Causas habituales:

  • Un SystemBase que reserva memoria en su OnUpdate.
  • Un foreach sobre una colección gestionada dentro de un system.
  • Boxing, por ejemplo object.ToString() sobre un struct.

Los systems con Burst deben mostrar cero asignaciones en el módulo Memory del Profiler.

5. Módulos del Profiler que conviene fijar

  • CPU Usage: la línea de tiempo y la jerarquía de marcadores.
  • Memory: el heap reservado; vigilar que GC.Used Memory no suba.
  • Entities (si está disponible): un módulo específico de DOTS que muestra los tiempos de cada system a lo largo del tiempo.

El módulo Entities Profiler llegó con la 1.8.0 / 6.x y se alinea con la ventana Systems sobre una línea de tiempo. Ante picos que abarcan varios fotogramas, es la forma más rápida de atribuirlos.

En proyectos en red, conviene combinarlo con ../DOTS Workflows/24_Netcode Profiler & Debugging.md para examinar los snapshots de Ghost, los ticks de predicción y el ancho de banda propio de Netcode.

6. Cuellos de botella habituales y soluciones

Patrón Síntoma en el Profiler Solución
Entities.ForEach por entidad (legacy) Un OnUpdate que carga mucho el hilo principal, con muchas muestras pequeñas Migrar a IJobEntity.ScheduleParallel.

7. Ejemplo práctico de presupuesto de fotograma

Imaginemos un presupuesto de 16,6 ms (60 Hz). Tras descartar Render:

  • SimulationSystemGroup: 6 ms
    • EnemyAISystem.OnUpdate: 2,1 ms (Burst, jobs paralelos en 4 workers)
    • MovementSystem.OnUpdate: 1,4 ms (Burst, paralelo)
    • EndSimulationECBS.Playback: 1,8 ms ← sospechoso
    • todo lo demás: 0,7 ms

1,8 ms de reproducción de ECB para lo que deberían ser unos pocos cambios estructurales suele ser la mayor palanca de mejora. Conviene revisar cuántos comandos hay:

  1. Entities Journaling durante 1 fotograma → recuento de Instantiate/AddComponent/DestroyEntity por system.
  2. Si un system registra cientos de comandos en cada fotograma, agruparlos (un comando por chunk, o un grabador paralelo con claves de ordenación por índice de chunk).

8. Perfilar una build independiente

Para una build real:

  1. File → Build Profiles → Development Build + Autoconnect Profiler.
  2. El player se ejecuta y el Profiler captura desde el proceso remoto.
  3. Los resultados reflejan mejor lo que verán los jugadores. Las mediciones del Editor pueden ser de 2 a 5 veces más lentas por motivos ajenos al código.

Capítulo 31. Auditoría de referencias a objetos administrados

1. Descripción general

Las referencias a objetos Unity gestionados son una de las maneras más fáciles de sacar sin querer a un system ECS de la ruta rápida. Esta auditoría ayuda a encontrar los puntos donde los datos de componentes, los systems o el código de renderizado guardan referencias a UnityEngine.Object de formas que bloquean Burst, los jobs o un baking limpio de la SubScene.

Conviene aplicarla tras la migración principal a ECS, o siempre que una traza del Profiler revele trabajo gestionado donde se esperaba código orientado a datos.

2. Qué buscar

Patrón Por qué importa
class .*: IComponentData Datos de componente gestionados; válidos, pero por lo general poco idóneos para las rutas calientes.
UnityEngine.Object Indicio general de referencias a objetos Unity gestionados.
GameObject, Transform, Component Las referencias a objetos de la escena en tiempo de ejecución no suelen tener cabida en los datos de simulación.
Mesh, Material, Texture, AudioClip A menudo deberían ser UnityObjectRef<T> o datos cacheados del lado de presentación.
Resources.Load Búsqueda de assets en tiempo de ejecución; suele ser mejor gestionarla con sistemas de baking o de contenido.
ScriptableObject Útil como fuente de authoring, pero los jobs de tiempo de ejecución deben leer valores horneados o blobs.

Conviene acotar la auditoría al código que da a ECS. Los MonoBehaviours de UI o las herramientas de Editor no tienen por qué seguir patrones DOTS.

3. Reglas de clasificación

Hallazgo Acción probable
Un componente gestionado en una query de simulación caliente Convertirlo a un IComponentData no gestionado, a un BlobAssetReference<T> o a un UnityObjectRef<T>.
Un componente gestionado que solo usa la presentación Conservarlo si está aislado de las rutas de simulación con Burst o jobs.
Un ScriptableObject que se lee en cada fotograma Hornear los valores necesarios en un componente o en un blob asset.
Una referencia a una malla o un material en datos ECS Usar UnityObjectRef<Mesh> / UnityObjectRef<Material> y desreferenciar en el código de presentación.
Un GameObject o Transform guardado en datos de componente Sustituirlo por datos ECS o llevar la referencia a un puente de presentación híbrido.

4. Ejemplo de auditoría

// Suspicious: managed component data used as per-entity visual state.
public class EnemyVisual : IComponentData
{
    public Material Material;
}

Preferir:

using Unity.Entities;
using UnityEngine;

public struct EnemyVisual : IComponentData
{
    public UnityObjectRef<Material> Material;
}

Después, desreferenciar en un system de presentación del hilo principal o cachear el material ya resuelto en un registro de renderizadores.

5. Indicios en el Profiler

Síntoma en el Profiler Qué revisar
Un system de simulación reserva memoria en cada fotograma Una búsqueda de objeto gestionado, o una asignación de LINQ o de una colección en OnUpdate.
Burst está desactivado para un system que debería ser solo de datos Acceso a un componente gestionado o a una API gestionada en la ruta de la query.
El hilo principal hace muchas búsquedas de renderizado pequeñas Cachear los resultados de UnityObjectRef<T>.Value fuera del bucle por entidad.
La SubScene se rehornea con más frecuencia de la esperada Los scripts de authoring o las referencias a objetos están creando dependencias innecesarias.

6. Validación

  1. Volver a ejecutar los patrones de búsqueda del apartado 2.
  2. Comprobar que los systems de simulación calientes solo consultan componentes no gestionados.
  3. Revisar el Burst Inspector para los systems y jobs que deberían compilar con Burst.
  4. Perfilar una escena representativa y comprobar que las asignaciones gestionadas han desaparecido de la ruta caliente.
  5. Entrar en modo Play desde un arranque limpio del Editor para verificar que el baking de la SubScene sigue resolviendo las referencias.

7. Solución de problemas

Síntoma Causa / Solución
El componente convertido sigue sin poder consultarse en Burst Queda algún campo gestionado. Revisar también los structs anidados.
UnityObjectRef<T>.Value devuelve null en modo Play El asset referenciado no se horneó o no se cargó. Revisar el Baker y el flujo de SubScene o de contenido.
El system visual sigue yendo lento tras la conversión Se está desreferenciando por entidad en cada fotograma. Cachear por asset compartido u hornear una clave de búsqueda compacta.
Los cambios en el ScriptableObject no afectan a los datos en tiempo de ejecución Los valores ya se hornearon. Volver a importar u hornear la SubScene, o añadir las dependencias de baking adecuadas.
Las referencias exclusivas del Editor fallan en las builds del player Llevar las APIs del Editor dentro de #if UNITY_EDITOR y hornear datos válidos en tiempo de ejecución.

Parte IV. Migración

Capítulo 32. Entities 1.x → 6.5 Resumen general

1. En qué consiste

Actualizar de Entities 1.x a 6.5 consiste, sobre todo, en tres migraciones de DOTS apoyadas en una transición de distribución de paquetes:

  1. Actualización de la distribución del Editor y del paquete. Se pasa a Unity 6000.4+ (com.unity.entities 6.4) o 6000.5+ (Entities 6.5), donde Entities es un Core Package integrado en el Editor.
  2. Limpieza de los datos ECS. Revisar las referencias a objetos Unity gestionados en el código que da a ECS y migrar los datos de componente de la ruta caliente hacia componentes no gestionados, valores horneados, BlobAssetReference<T> o UnityObjectRef<T>.
  3. Migración de la API de ECS. Las APIs antiguas marcadas como obsoletas en Entities 1.4 (Entities.ForEach, IAspect) siguen siéndolo en 6.5. Además: ComponentLookup.GetRefRWOptional → TryGetRefRW.

Cada migración se documenta en su propio subdocumento de esta carpeta. Esta página es el mapa: cuándo hacer cada una y en qué orden.

Cuidado con la nomenclatura: la versión del paquete Entities (1.4 / 6.4 / 6.5) es distinta de la versión del Editor de Unity (6000.3 / 6000.4 / 6000.5). Desde la 6.4 se alinean por convención, pero se documentan en notas de versión separadas. Este documento las trata como dos ejes; la página del Changelog (../Changelog/Entities 1.4 → 6.5 Key Changes.md) hace lo mismo.

2. Orden recomendado

Figure: Entities 1.x to 6.5 Migration Order.

Por qué este orden:

  • El Editor primero, porque nada más compilará contra las APIs de 6.x hasta que el Editor las conozca.
  • Después, Package Manager → Core Package: sin este paso, la resolución de paquetes queda duplicada o entra en conflicto.
  • La limpieza de referencias gestionadas antes de refactorizar la iteración: deja claro el modelo de datos antes de reescribir los systems y los jobs.
  • foreach e IAspect al final: son internos de ECS y no bloquean el resto del código.

3. Qué permanece igual

Bastante más de lo que parece, lo que hace que la migración asuste menos de lo esperado.

Sin cambios Notas
IComponentData, IBufferElementData, ISharedComponentData Las mismas interfaces, la misma semántica.
ISystem / SystemBase Los mismos hooks de ciclo de vida, los mismos atributos.
Superficie de la API de EntityManager CreateEntity, AddComponent, etc.: idénticos.
SystemAPI.Query Introducida en 1.x y sigue siendo la API central de iteración.
IJobEntity, IJobChunk La misma estructura; IJobEntity es la opción recomendada por defecto.
EntityCommandBuffer + ECBS Igual.
Burst, Jobs, Collections, Mathematics Igual.
Conceptos de Netcode for Entities (Ghost, RPC, Prediction) La misma arquitectura; lo único que cambia es cómo se distribuye el paquete.

Si el código 1.x ya usa IJobEntity y SystemAPI.Query (y no el antiguo Entities.ForEach), los pasos 4 y 5 de la migración prácticamente desaparecen.

4. Qué cambia, en resumen

Área 1.x 6.5
Distribución Paquete UPM com.unity.entities Core Package (sin entrada en el manifest)
Número de versión 1.4.x 6.5.x (alineado con Unity 6000.5)

El changelog completo del periodo está en ../Changelog/Entities 1.4 → 6.5 Key Changes.md.

5. Estrategia de copia de seguridad

Obligatorio antes de comenzar:

  1. Crear una rama en git. Con un nombre descriptivo (migrate/entities-6.5). No migrar en main.
  2. Hacer commit de manifest.json y packages-lock.json en su estado previo a la migración. Son la referencia fiable para revertir si la resolución falla.
  3. Etiquetar el commit previo a la migración. Con git tag pre-entities-6.5, las reversiones son triviales.
  4. Mantener el proyecto abierto en la versión antigua del Editor hasta estar listos. Abrirlo una vez en 6000.5 cambia el ProjectVersion.txt y fuerza la ruta de migración.

6. Listas de verificación por paso

6.1 Actualización del Editor

  • Instalar Unity 6000.5.x desde Hub.
  • No abrir aún el proyecto. Primero, hacer commit de cualquier trabajo pendiente.
  • Abrir el proyecto en el nuevo Editor: ofrecerá actualizar ProjectVersion.txt. Aceptar.
  • Dejar que termine la importación inicial; aparecerán advertencias sobre APIs obsoletas y paquetes que faltan. Los pasos siguientes las resuelven.

6.2 Limpieza de paquetes → 02_Package Manager → Core Package.md

  • Quitar com.unity.entities, com.unity.collections, com.unity.mathematics y com.unity.entities.graphics de manifest.json.
  • Borrar packages-lock.json y dejar que el Editor lo regenere.
  • Quitar también Netcode for Entities si el proyecto lo usa (la 6.5 es una versión de Core Package).

6.3 Referencias a objetos managed → 03_Managed Object References → UnityObjectRef.md

  • Revisar el código que da a ECS en busca de campos de objetos Unity gestionados.
  • Convertir los datos de componente de la ruta caliente a datos ECS no gestionados siempre que se pueda.
  • Usar UnityObjectRef<T> para las referencias del lado de ECS a assets u objetos de Unity.
  • Usar BlobAssetReference<T> para datos de juego inmutables de gran tamaño.

6.4 ForEach legacy → 04_foreach → IJobEntity.md

  • Para los systems que aún usan Entities.ForEach o Job.WithCode, migrarlos a IJobEntity + SystemAPI.Query.

6.5 Eliminación de Aspect → 05_IAspect Removal.md

  • Incorporar los métodos del aspect en el punto de llamada con parámetros RefRW<T> / RefRO<T>.
  • Eliminar el struct del IAspect.

7. Pruebas tras la migración

Pruebas de humo que conviene ejecutar antes de dar la migración por terminada:

  1. Modo Play en una SubScene representativa. Las entidades existen, los systems se actualizan y nada lanza excepciones al arrancar.
  2. Compilar un player independiente. Los fallos que solo aparecen al compilar (normalmente de serialización) salen aquí.
  3. Ejecutar las pruebas de regresión de gameplay existentes. Si el rendimiento importa, capturar una traza de referencia del Profiler antes de migrar y compararla.
  4. Comprobar el guardado y la carga. Los handles de Entity son locales al world y no son estables. En su lugar, guardar IDs propios del juego, claves de contenido o datos de authoring.
  5. Probar en la plataforma de destino. Algunos backends específicos de plataforma (rarezas de IL2CPP, AOT) se manifiestan aquí.

8. Cuándo revertir

Motivos para una reversión limpia:

  • El Editor no consigue abrir el proyecto en absoluto (raro; suele ser una discrepancia en ProjectVersion.txt).
  • La resolución de packages-lock.json genera un grafo sin sentido y borrar el archivo no lo arregla.
  • Un paquete de terceros crítico del que se depende aún no es compatible con 6.5.

Plan: git checkout pre-entities-6.5, reinstalar la versión antigua de Unity y comprobar que el juego sigue funcionando con la versión anterior.

Las migraciones que se quedaron a medias y bloqueadas deben abandonarse de forma limpia y reiniciarse desde la etiqueta: nunca se debe distribuir código a medio migrar.

9. Solución de problemas

Síntoma Causa / Solución
El Editor no se abre — "El proyecto se guardó en una versión más reciente" Alguien lo abrió antes en 6000.5; revertir ProjectSettings/ProjectVersion.txt a la cadena de la versión anterior.
Tras la limpieza del manifest, error de compilación "Unity.Entities assembly not found" El Editor es anterior a 6000.4, donde el Core Package no está disponible. Actualizar el Editor o volver a añadir el paquete UPM.
Cientos de advertencias de obsolescencia Es lo esperado. Conviene tratarlas como la lista de tareas de la migración: cada una señala algo que corregir.
La compilación tiene éxito pero el juego falla al inicio Suele ser un problema de ECB o de serialización introducido al eliminar un componente. Comparar el tamaño del asset EntityScene horneado; una reducción grande sugiere que un componente se eliminó sin avisar.
El rendimiento empeora tras la migración Suele deberse a que se introdujo un SystemBase que no debería existir (un campo gestionado), o a que un IJobEntity dejó de usar Burst. Revisar el Profiler y el Burst Inspector.

Capítulo 33. Package Manager → Core Package

1. Qué cambia

En Unity 6000.4+, Entities (y sus paquetes DOTS complementarios) ya no se instalan desde el Package Manager. Se distribuyen dentro del Editor como Core Packages. Los archivos del proyecto cambian en tres sitios:

  • Packages/manifest.json: se eliminan las entradas com.unity.entities*.
  • Packages/packages-lock.json: el grafo resuelto se reduce y los paquetes afectados aparecen con "source": "builtin".
  • Ventana del Package Manager: los paquetes ahora figuran en "In Project" etiquetados como Built-in.

El código C# no cambia. Los nombres de ensamblado (Unity.Entities, Unity.Collections, etc.) siguen siendo los mismos y las referencias de los .asmdef no se tocan.

2. Qué paquetes se trasladan

Pasaron a ser Core Package en 6000.4+:

ID de UPM (antes) Después
com.unity.entities Core Package, se distribuye con el Editor
com.unity.collections Core Package
com.unity.mathematics Core Package
com.unity.entities.graphics Core Package
com.unity.netcode (Netcode for Entities) Core Package en 6.5

Permanecen en el Package Manager:

  • com.unity.physics (Unity Physics)
  • com.unity.charactercontroller (Character Controller)

Integrados en el motor (sin cambios):

  • Jobs system
  • Burst (com.unity.burst aún figura como entrada UPM en algunas configuraciones, pero en 6000.4+ está ligado a la versión del Editor; lo más seguro es dejar el manifest tal cual).

Capítulo 34. Referencias a objetos administrados → UnityObjectRef

1. Descripción general

Al migrar código ECS antiguo, un problema habitual es que las referencias a objetos gestionados se cuelen en los datos de componente de tiempo de ejecución. En Entities 6.5 conviene mantener no gestionados los componentes de gameplay siempre que se pueda y usar UnityObjectRef<T> cuando los datos ECS deban apuntar a un objeto de Unity como una malla, un material, una textura, un clip de audio o un ScriptableObject de authoring.

Esta página se ciñe al diseño de componentes DOTS. No cubre las APIs de identificación de objetos de Unity que abarcan todo el motor.

2. Qué buscar

Buscar los componentes que guardan objetos Unity gestionados de forma directa:

Patrón Por qué revisarlo
Componentes class que implementan IComponentData Los componentes gestionados están permitidos, pero son más lentos y quedan restringidos al hilo principal.
Campos UnityEngine.Object dentro de los datos de un componente Suele indicar un componente gestionado o un diseño de componente no gestionado inválido.
Campos GameObject, Transform, Material, Mesh, Texture, AudioClip Decidir si su sitio está en el authoring, en la presentación o en un UnityObjectRef<T>.
Systems de tiempo de ejecución que llaman a Resources.Load por entidad Llevar la resolución de assets al baking o a un sistema de contenido centralizado.

3. Antes y después

Antes — componente gestionado

using Unity.Entities;
using UnityEngine;

public class ProjectileVisual : IComponentData
{
    public Mesh Mesh;
    public Material Material;
}

Esto funciona, pero convierte el componente en gestionado. Los componentes gestionados restan flexibilidad a Burst y a los jobs, y por lo general no encajan bien con los datos de gameplay por entidad.

Después — componente no gestionado con UnityObjectRef<T>

using Unity.Entities;
using UnityEngine;

public struct ProjectileVisual : IComponentData
{
    public UnityObjectRef<Mesh> Mesh;
    public UnityObjectRef<Material> Material;
}

Hornear las referencias una sola vez:

using Unity.Entities;
using UnityEngine;

public class ProjectileVisualAuthoring : MonoBehaviour
{
    public Mesh Mesh;
    public Material Material;
}

public class ProjectileVisualBaker : Baker<ProjectileVisualAuthoring>
{
    public override void Bake(ProjectileVisualAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);
        AddComponent(entity, new ProjectileVisual
        {
            Mesh = new UnityObjectRef<Mesh>(authoring.Mesh),
            Material = new UnityObjectRef<Material>(authoring.Material)
        });
    }
}

4. Patrón de acceso en tiempo de ejecución

Mantener los jobs de simulación puramente orientados a datos. Desreferenciar objetos de Unity solo allí donde se permiten las APIs gestionadas de Unity.

using Unity.Entities;
using UnityEngine;

public partial class ProjectileVisualUploadSystem : SystemBase
{
    protected override void OnUpdate()
    {
        foreach (var visual in SystemAPI.Query<RefRO<ProjectileVisual>>())
        {
            Mesh mesh = visual.ValueRO.Mesh.Value;
            Material material = visual.ValueRO.Material.Value;

            if (mesh == null || material == null)
                continue;

            // Upload to a rendering registry, material property block cache, etc.
        }
    }
}

Para renderizado a gran escala, conviene evitar la desreferencia por entidad en cada fotograma. Cachear los objetos ya resueltos en un registro del lado de presentación, indexado por un valor ECS compacto.

5. Cuándo no usar UnityObjectRef<T>

Caso Mejor alternativa
Datos de gameplay puros Guardar datos primitivos, matemáticos o de blob directamente en un IComponentData.
Tablas grandes de solo lectura BlobAssetReference<T>.
Un prefab de entidad horneado en la misma SubScene Una referencia de prefab de tipo Entity.
Un prefab que se carga mediante flujos de contenido EntityPrefabReference.
Comportamiento gestionado complejo por entidad Mantenerlo fuera de la ruta de simulación o aislarlo en un system de presentación gestionado.

6. Lista de verificación de migración

  • Localizar datos de componente gestionados que solo guardan referencias a objetos de Unity.
  • Sustituir esos campos por UnityObjectRef<T> donde proceda un componente no gestionado.
  • Llevar la asignación de objetos a los Bakers.
  • Mantener la desreferencia de .Value fuera de los Burst jobs.
  • Cachear los objetos ya desreferenciados cuando muchas entidades comparten el mismo asset.
  • Usar BlobAssetReference<T> para datos de gameplay inmutables de gran tamaño, en vez de leer ScriptableObjects en tiempo de ejecución.

7. Solución de problemas

Síntoma Causa / Solución
El componente no se puede usar en un Burst job Sigue siendo gestionado. Convertir los campos de objetos de Unity a UnityObjectRef<T> o separar los datos de presentación.
.Value es null El asset referenciado no está cargado o se destruyó. Revisar la referencia de authoring y la ruta de carga de contenido.
El system reserva memoria en cada fotograma Se están resolviendo o cargando assets por entidad. Hornear las referencias y cachear los objetos gestionados ya resueltos.
Un system de gameplay depende de un ScriptableObject en tiempo de ejecución Hornear los valores necesarios en datos de componente o en un blob asset.
La build descarta un asset Comprobar que el asset se referencia mediante flujos de baking o de contenido, y no solo mediante una ruta de string en tiempo de ejecución.

Capítulo 35. Migración de foreach → IJobEntity

1. Estado — obsoleto, aún no eliminado

Las antiguas APIs de iteración de componentes están obsoletas en Entities 1.4+ / 6.5, pero aún compilan:

Legacy Estado en 6.5 Reemplazo
Entities.ForEach Obsoleto desde 1.4; advertencias de obsolescencia en cada uso; el changelog de Unity indica que se eliminará en una versión mayor futura SystemAPI.Query<...> (hilo principal) o IJobEntity (jobs)
Job.WithCode Obsoleto junto con ForEach Struct IJob simple, o inline en OnUpdate
IJobForEach (anterior a 1.0) Eliminado mucho antes de 1.4 IJobEntity

Se llega aquí porque se abrió un proyecto 1.x en 6000.5+ y el compilador emite muchas advertencias, o porque se quieren eliminar estas advertencias antes de que una versión mayor futura elimine la API. La refactorización es mecánica una vez que se identifica el patrón.

2. Las dos APIs de reemplazo

  • SystemAPI.Query<T, U, ...> — iteración en el hilo principal dentro de OnUpdate. Simple, sin sobrecarga de jobs, ideal para cargas de trabajo pequeñas o cuando se necesitan APIs managed.
  • IJobEntity — job paralelo generado por código. Reemplazo directo si se está listo para el trabajo multi-hilo, que es donde DOTS marca la diferencia.

Referencia: ../DOTS Workflows/11_IJobEntity · SystemAPI.Query.md.

3. Migración — Entities.ForEach → SystemAPI.Query

3.1 Antes

public partial class HealthRegenSystem : SystemBase
{
    protected override void OnUpdate()
    {
        float dt = Time.DeltaTime;

        Entities
            .WithAll<Alive>()
            .WithNone<Dead>()
            .ForEach((ref Health health, in Regen regen) =>
            {
                health.Value = math.min(health.Value + regen.Rate * dt, health.Max);
            })
            .ScheduleParallel();
    }
}

3.2 Después — SystemAPI.Query (hilo principal)

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        foreach (var (health, regen) in
                 SystemAPI.Query<RefRW<Health>, RefRO<Regen>>()
                          .WithAll<Alive>()
                          .WithNone<Dead>())
        {
            health.ValueRW.Value = math.min(
                health.ValueRO.Value + regen.ValueRO.Rate * dt,
                health.ValueRO.Max);
        }
    }
}

Notas sobre los cambios de forma:

  • SystemBase → ISystem (más económico, compatible con Burst). Puede ser necesario mantener SystemBase si el código antiguo capturaba campos managed.
  • ref Health → elemento de tupla RefRW<Health>. Lectura mediante .ValueRO, escritura mediante .ValueRW.
  • in Regen → RefRO<Regen>.
  • Time.DeltaTime → SystemAPI.Time.DeltaTime.
  • Los filtros (WithAll/WithNone) se pasan al constructor de Query.

4. Migración — Entities.ForEach.ScheduleParallel → IJobEntity

Cuando el ForEach se planificaba en paralelo, la forma de job es una migración directa 1:1 que mantiene el paralelismo.

4.1 Después — IJobEntity

[BurstCompile]
public partial struct HealthRegenJob : IJobEntity
{
    public float DeltaTime;

    void Execute(ref Health health, in Regen regen)
    {
        health.Value = math.min(health.Value + regen.Rate * DeltaTime, health.Max);
    }
}

[BurstCompile]
public partial struct HealthRegenSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        new HealthRegenJob { DeltaTime = SystemAPI.Time.DeltaTime }
            .ScheduleParallel();
    }
}

Los filtros se aplican al struct del job como atributos:

[WithAll(typeof(Alive))]
[WithNone(typeof(Dead))]
public partial struct HealthRegenJob : IJobEntity { ... }

O se pasa una query en el momento de planificación:

var q = SystemAPI.QueryBuilder()
    .WithAll<Health, Regen, Alive>()
    .WithNone<Dead>()
    .Build();
new HealthRegenJob { DeltaTime = dt }.ScheduleParallel(q);

5. Capturas y sus reemplazos

El legacy Entities.ForEach permitía capturar variables locales con .WithReadOnly, .WithDisposeOnCompletion, etc. En IJobEntity, las capturas son campos del struct del job — Burst sabe exactamente qué se está pasando.

// BEFORE
var buffer = new NativeArray<int>(count, Allocator.TempJob);
Entities.ForEach((ref Foo f) => { /* use buffer */ })
    .WithDisposeOnCompletion(buffer)
    .ScheduleParallel();

// AFTER
[BurstCompile]
partial struct FooJob : IJobEntity
{
    [ReadOnly] public NativeArray<int> Buffer;

    void Execute(ref Foo f) { /* use Buffer */ }
}

var buffer = new NativeArray<int>(count, Allocator.TempJob);
state.Dependency = new FooJob { Buffer = buffer }.ScheduleParallel(state.Dependency);
buffer.Dispose(state.Dependency);   // chained dispose replaces WithDisposeOnCompletion

Reemplazos de atributos:

  • .WithReadOnly(x) → [ReadOnly] public NativeArray<T> X;
  • .WithDisposeOnCompletion(x) → x.Dispose(handle) después de planificar.
  • .WithoutBurst() → [WithoutBurst] en el struct del job (pero considerar seriamente eliminar la razón por la que fue necesario).
  • .WithEntityAccess() → añadir el parámetro Entity entity a Execute.

6. Cuándo sigue siendo necesario un SystemBase

Si el sistema original tenía un campo managed genuino (Input System, UI Toolkit, servicio externo), no es posible convertirlo mecánicamente a ISystem. Dos opciones:

  1. Mantenerlo como SystemBase y usar SystemAPI.Query dentro de OnUpdate. Un solo hilo.
  2. Dividirlo en un SystemBase que haga de puente con el mundo managed y un ISystem que realice el trabajo Burst — ver ../DOTS Workflows/08_System — ISystem vs SystemBase.md.

7. Validación

Después de migrar un sistema:

  1. Probar el comportamiento con pruebas unitarias. La migración no debe cambiar la lógica; si la salida difiere, la refactorización introdujo un error.
  2. Analizar el rendimiento. El ForEach paralelo y IJobEntity.ScheduleParallel deben estar dentro del 5% de diferencia para trabajo equivalente. Una regresión grande significa que Burst no está actuando — inspeccionar el Burst Inspector.
  3. Eliminar el archivo del sistema antiguo. Resistir la tentación de mantener un fallback #if LEGACY — se deteriora.

Capítulo 36. Deprecación de IAspect — migrar fuera de los Aspects

1. Estado — obsoleto, aún no eliminado

IAspect fue marcado como obsoleto en Entities 1.4 y sigue siendo obsoleto en la línea 6.x. Aún compila, aún se ejecuta y aún está listado como un tipo de parámetro válido de IJobEntity.Execute en la referencia de la API 6.5 — pero el compilador emite una advertencia de obsolescencia, y Unity ha indicado que el tipo se eliminará en una versión mayor futura. El nuevo código no debe usarlo; el código existente debe migrarlo a un ritmo progresivo.

IAspect se introdujo para permitir que los sistemas declararan una "vista" sobre un conjunto de componentes y la usaran como un único argumento en Entities.ForEach o IJobEntity. En la práctica, añadió una capa de generación de código y un sistema de tipos paralelo que no se justificó:

  • Escribir un aspect duplicaba la lista de parámetros de componentes.
  • Los depuradores avanzaban por el código wrapper generado.
  • RefRW<T> / RefRO<T> en IJobEntity y SystemAPI.Query logran la misma ergonomía de "pasar componentes como un conjunto" sin el tipo extra.

Dado que la superficie de API aún está presente, el código no se rompe al actualizar el Editor; esta migración consiste en limpiar advertencias y prepararse para la versión 2.0.

2. Refactorización mecánica

2.1 Punto de partida

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public readonly partial struct MovementAspect : IAspect
{
    public readonly RefRW<LocalTransform> Transform;
    public readonly RefRO<Velocity>       Velocity;
    public readonly RefRO<Speed>          Speed;

    public void Step(float dt)
    {
        Transform.ValueRW.Position += Velocity.ValueRO.Direction * Speed.ValueRO.Value * dt;
    }
}

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var m in SystemAPI.Query<MovementAspect>())
            m.Step(dt);
    }
}

2.2 Después — parámetros de componentes directos

using Unity.Entities;
using Unity.Mathematics;
using Unity.Transforms;

public partial struct MovementSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;
        foreach (var (transform, velocity, speed) in
                 SystemAPI.Query<RefRW<LocalTransform>, RefRO<Velocity>, RefRO<Speed>>())
        {
            transform.ValueRW.Position += velocity.ValueRO.Direction * speed.ValueRO.Value * dt;
        }
    }
}

Dos cambios:

  1. El tipo aspect desaparece. Sus campos se convierten en los elementos de la tupla de SystemAPI.Query.
  2. El método que estaba en el aspect (Step) se incorpora en línea. Si es demasiado grande para incorporarlo, se puede extraer como un helper estático que tome los mismos parámetros — esto lo mantiene compatible con Burst:
static void Step(ref LocalTransform t, in Velocity v, in Speed s, float dt)
{
    t.Position += v.Direction * s.Value * dt;
}

Para llamarlo:

Step(ref transform.ValueRW, velocity.ValueRO, speed.ValueRO, dt);

3.1 Antes

[BurstCompile]

{

}

3.2 Después — componentes como parámetros de Execute

[BurstCompile]

{

Nótese que IJobEntity usa parámetros simples ref / in, no RefRW<T> / RefRO<T>. (El generador envuelve los componentes en accesos a arrays de chunk por debajo.)

4. Aspects que envolvían un DynamicBuffer

// BEFORE
public readonly partial struct PathAspect : IAspect
{
    public readonly DynamicBuffer<Waypoint> Path;
    public void Advance(int n) => Path.RemoveRange(0, n);
}

// AFTER — access buffer directly
foreach (var (path, e) in SystemAPI.Query<DynamicBuffer<Waypoint>>().WithEntityAccess())
{
    path.RemoveRange(0, 1);
}

En los jobs, DynamicBuffer<T> es un parámetro válido de Execute — no se necesita nada adicional.

5. Aspects entre entidades

Ocasionalmente un aspect envolvía un ComponentLookup<T> para "necesito ver el componente de otra entidad". Reemplazo:

// BEFORE
public readonly partial struct AITargetAspect : IAspect
{
    public readonly RefRO<Target> Target;
    [ReadOnly] public readonly ComponentLookup<Health> HealthLookup;
    public float TargetHealth =>
        HealthLookup.TryGetComponent(Target.ValueRO.Entity, out var h) ? h.Value : 0f;
}

// AFTER — declare the lookup on the system / job directly
public partial struct AISystem : ISystem
{
    private ComponentLookup<Health> _healthLookup;

    public void OnCreate(ref SystemState state)
    {
        _healthLookup = state.GetComponentLookup<Health>(true);
    }

    public void OnUpdate(ref SystemState state)
    {
        _healthLookup.Update(ref state);

        foreach (var (target, e) in SystemAPI
                     .Query<RefRO<Target>>()
                     .WithEntityAccess())
        {
            float hp = _healthLookup.TryGetComponent(target.ValueRO.Entity, out var h)
                ? h.Value : 0f;
            // ...
        }
    }
}

Este patrón también es más ligero que la versión con aspect porque la búsqueda se comparte entre entidades.

6. Cuándo un aspect era complejo

Si el aspect tenía muchos campos, encapsulaba estado o participaba en herencia no trivial — eso es señal de considerar una refactorización genuina, no solo una migración. A menudo el código queda mejor como dos sistemas que pasan datos mediante componentes, en lugar de un sistema con un aspect dios.

5. Seguridad ante cambios estructurales

Término Qué es
Cambio estructural Agrega o elimina un componente, o crea o destruye una entidad. Como mueve entidades entre chunks, resulta costoso y no es seguro a mitad de una iteración.
EntityCommandBuffer (ECB) Registra los cambios estructurales durante la iteración y los reproduce en un punto seguro.
EntityCommandBuffer.ParallelWriter Variante de ECB segura para hilos, pensada para jobs paralelos; cada llamada recibe un sortKey que garantiza una reproducción determinista.
Entidad diferida Entidad creada con ecb.CreateEntity() o Instantiate() que aún no existe: su índice provisional se resuelve al reproducir el ECB.
Sort key Entero que se pasa a cada comando de un ECB paralelo; el valor predeterminado es [ChunkIndexInQuery].

9. Conceptos de apoyo

Término Qué es
Burst El compilador nativo de Unity para jobs y structs marcados con [BurstCompile]. Genera código SIMD nativo.
Change version Contador de 32 bits que se incrementa cuando un sistema escribe en la columna de un componente de un chunk. Permite que SetChangedVersionFilter omita los chunks que no han cambiado.
SoA (Structure of Arrays) Disposición de almacenamiento en la que cada campo forma su propio array; favorece la caché en la iteración escalar. Los chunks usan SoA.
RefRW<T> / RefRO<T> Tipos wrapper para acceder a componentes en SystemAPI.Query con intención de lectura-escritura o de solo lectura.
EnabledRefRW<T> / EnabledRefRO<T> Wrapper para leer o escribir el bit de activación de un componente habilitable sin provocar un cambio estructural.
TypeManager El registro del runtime que asigna a cada componente un índice de tipo estable. Se reconstruye al arrancar; los proyectos grandes lo notan.
BlobAssetReference<T> Puntero a un blob de datos grande y de solo lectura que se almacena fuera del chunk. Evita inflar la capacidad del chunk.
